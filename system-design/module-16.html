1. Monolith to Microservices

The Evolution: Why Companies Migrate

Every successful distributed system started as a monolith. The question is never “should we build microservices from day one?” — the answer to that is almost always no. The real question is: at what point does the monolith start actively impeding your ability to ship?

A monolith is a single deployable unit where all components (UI, business logic, data access) live in one codebase and share one process. This is how most applications should begin because it minimizes coordination cost, simplifies debugging, and lets you iterate at maximum speed when the team is small and the domain is poorly understood.

Problems emerge at scale along three axes:

  • Team scale — When 50+ engineers commit to the same codebase, merge conflicts, long CI pipelines, and deployment queues dominate the workweek. At Amazon in the early 2000s, a single deploy could take hours and require dozens of engineers to coordinate.
  • Operational scale — Different features have different scaling profiles. A monolith forces you to scale everything uniformly. If your search index needs 10x the compute of your checkout flow, you pay 10x for both.
  • Domain complexity — As the domain grows, bounded contexts blur together. An “Order” object that started clean accumulates responsibilities from shipping, payments, inventory, and fraud detection until changes anywhere risk breaking everything.

Monolith vs. Microservices: Trade-Off Matrix

Dimension Monolith Microservices

When Microservices Are the WRONG Choice

Anti-pattern: Premature Decomposition. If your team is fewer than ~20 engineers, your domain is not well understood, or you do not have mature CI/CD and observability infrastructure, microservices will slow you down, not speed you up. You trade one set of problems (monolith complexity) for a harder set (distributed systems complexity) without the organizational scale to justify it.
  • Startups finding product-market fit — You need maximum iteration speed. Microservices introduce latency in both the code path and the development process.
  • Small teams (<10 engineers) — The overhead of managing service-to-service communication, independent deployments, and distributed debugging dwarfs any benefit.
  • Tightly coupled domains — If nearly every user action touches most services, you have a distributed monolith — all the downsides of both.
  • No platform engineering team — Without tooling for service scaffolding, CI/CD, observability, and networking, each team reinvents the wheel.

Real-World Examples

Shopify: The Modular Monolith. Shopify runs one of the largest Ruby on Rails monoliths in the world, serving millions of merchants. Rather than decomposing into microservices, they invested in a modular monolith architecture where the codebase is partitioned into components with enforced boundaries (using a tool called “Packwerk”). Each component has a well-defined public API. This gives them team autonomy and code isolation without the operational overhead of a distributed system. They deploy multiple times per day.

Netflix: 700+ Microservices. Netflix started as a monolithic Java app. As they scaled to 200M+ subscribers across 190 countries, they decomposed into over 700 microservices. Each service is independently deployable, and teams own their services end-to-end. This lets them handle diverse workloads: the recommendation engine can scale independently from the streaming pipeline, which can scale independently from the billing system.

The Strangler Fig Pattern

Named after the strangler fig tree that grows around a host tree and eventually replaces it, this is the safest migration strategy. You do not rewrite the monolith. Instead, you incrementally route new functionality to new services while the monolith continues to serve existing traffic. Over months or years, the monolith shrinks until it can be decommissioned.

graph LR subgraph "Phase 1: Identify" A[Monolith] --> B[Identify Bounded Context] end subgraph "Phase 2: Extract" C[Facade / Proxy] -->|new traffic| D[New Service] C -->|existing traffic| E[Monolith] end subgraph "Phase 3: Replace" F[Facade / Proxy] -->|all traffic| G[New Service] H[Monolith - shrunk] -.->|deprecated| F end B --> C E --> F

Key principles of the Strangler Fig approach:

  1. Start with a proxy/facade that sits in front of the monolith and can route requests to either the old or new code.
  2. Extract one bounded context at a time, beginning with the one that is least coupled to the rest of the monolith.
  3. Run both old and new in parallel for a period, comparing outputs (shadow traffic) to build confidence.
  4. Cut over incrementally — percentage-based routing lets you ramp from 1% to 100% while monitoring error rates.
  5. Delete the old code once the new service is fully serving traffic.
Staff-Level Signal: In an interview, do not jump to microservices as the default. Start with a monolith and explain under what conditions you would decompose. This shows maturity. Then describe the migration strategy (Strangler Fig) rather than a big-bang rewrite. Interviewers value pragmatism over architecture astronautics.

2. Service Decomposition

Finding the Right Service Boundaries

The hardest part of microservices is not building them — it is deciding where to draw the lines. Get the boundaries wrong and you end up with a “distributed monolith”: all the operational complexity of microservices with none of the benefits. The primary tool for getting this right is Domain-Driven Design (DDD).

Domain-Driven Design and Bounded Contexts

Eric Evans coined the term “Bounded Context” in his 2003 book Domain-Driven Design. A bounded context is a boundary within which a particular domain model is defined and applicable. Different bounded contexts can have different models for the same real-world concept.

For example, in an e-commerce system:

  • Order Context: A “Product” is an order line item with quantity, price, and SKU.
  • Catalog Context: A “Product” is a rich object with images, descriptions, categories, and reviews.
  • Warehouse Context: A “Product” is an inventory item with bin location, weight, and dimensions.

Each bounded context becomes a candidate for a service. The key insight is that these are different models that happen to share a name. Forcing them into a single “Product” class creates a god object that every team must coordinate on.

How to Identify Service Boundaries

  1. Event Storming: Gather domain experts and developers in a room. Map out every domain event (OrderPlaced, PaymentReceived, ItemShipped) on sticky notes. Cluster related events — each cluster often maps to a bounded context.
  2. Look for natural team boundaries: If two groups of engineers rarely need to coordinate, their code can likely be separate services.
  3. Identify data ownership: Which data is read and written by which business capabilities? Data that is primarily owned by one capability belongs in that service.
  4. Examine change frequency: Features that change together should live together. If the payment flow changes independently from the catalog, they should be separate services.

The Two-Pizza Team Heuristic

Jeff Bezos famously mandated that every team at Amazon should be small enough to feed with two pizzas (6-10 people). This is not arbitrary — it aligns with the cognitive limit on how many people can effectively coordinate without formal processes slowing them down. Each two-pizza team should own one or a few closely related services. If a service requires more than one team to maintain, it is probably too large.

Common Decomposition Mistakes

Mistake 1: Too Fine-Grained. Creating a service for every database table or every REST endpoint leads to “nanoservices” — dozens of tiny services that cannot do anything useful without calling five other services. The result is enormous network overhead, debugging nightmares, and operational toil. If two services are always deployed together and always call each other synchronously, they should be one service.
Mistake 2: Shared Databases. If multiple services read from and write to the same database tables, you have not actually decomposed — the database is the coupling. Any schema change requires coordinating across all consuming services. This is the single most common failure mode in microservice migrations.
Mistake 3: Synchronous Chains. Service A calls B, which calls C, which calls D. Latency compounds, availability drops multiplicatively (0.99^4 = 0.96), and a single slow service brings down the whole chain. Prefer asynchronous communication or at least keep synchronous call chains to a maximum depth of 2-3.
Staff-Level Signal: When decomposing in an interview, articulate the criteria you use to decide boundaries. Saying “I would use DDD bounded contexts, looking at data ownership and change frequency” is far stronger than “I would create a user service, an order service, and a payment service.” Show your reasoning, not just the result.

3. API Gateway Pattern

The Front Door to Your Microservices

Without an API gateway, clients must know the addresses of individual services, handle authentication themselves, and make multiple requests to compose a single page view. The API gateway is a reverse proxy that provides a single entry point, handling cross-cutting concerns so that individual services do not have to.

Core Responsibilities

  • Request routing — Maps public URLs to internal services (e.g., /api/orders/* routes to the Order service).
  • Authentication & authorization — Validates JWT tokens or API keys at the edge so services receive pre-authenticated requests.
  • Rate limiting — Protects backend services from traffic spikes (see Module 12: Rate Limiting).
  • Protocol translation — Accepts REST/HTTP from clients, translates to gRPC for internal communication.
  • Response aggregation — Combines responses from multiple services into a single response (API composition).
  • TLS termination — Offloads HTTPS so internal traffic can use plain HTTP or mTLS.
  • Caching — Caches responses for read-heavy endpoints.
  • Observability — Adds request IDs, logs, and metrics at the edge.

Backend for Frontend (BFF) Pattern

A single API gateway quickly becomes a bottleneck if different clients (web, iOS, Android, TV) need different data shapes and aggregations. The BFF pattern creates a dedicated gateway per client type. Each BFF is owned by the frontend team it serves, so they can evolve the API contract without coordinating with other client teams. 

Web Client  -->  Web BFF      -->  Backend Services
iOS Client  -->  Mobile BFF   -->  Backend Services
TV Client   -->  TV BFF       -->  Backend Services

Why BFF works: The mobile app might need a condensed payload (bandwidth matters), the web app might need rich data for a single-page app, and the TV app might need a completely different data format. A single gateway serving all three ends up with complex conditional logic that is hard to maintain.

API Composition / Aggregation

Consider a product detail page that needs data from Catalog (product info), Inventory (stock status), Reviews (ratings), and Pricing (current price). Without composition, the client makes four requests. With an API gateway that supports composition, one client request fans out to four services in parallel: 

# Pseudocode: API composition in the gateway
async def get_product_detail(product_id):
    # Fan out to services in parallel
    catalog, inventory, reviews, pricing = await asyncio.gather(
        catalog_client.get_product(product_id),
        inventory_client.get_stock(product_id),
        reviews_client.get_ratings(product_id),
        pricing_client.get_price(product_id),
    )
    # Compose unified response
    return {
        "product": catalog,
        "in_stock": inventory.available > 0,
        "rating": reviews.average,
        "price": pricing.current,
    }

4. Service Mesh

Infrastructure Layer for Service-to-Service Communication

An API gateway handles north-south traffic (client to service). A service mesh handles east-west traffic (service to service). It extracts networking concerns — retries, circuit breaking, mTLS, load balancing, observability — out of application code and into a dedicated infrastructure layer.

The Sidecar Proxy Pattern

In a service mesh, every service instance is paired with a sidecar proxy (typically Envoy). All inbound and outbound traffic passes through the sidecar. The application code makes a plain HTTP/gRPC call to localhost; the sidecar intercepts it, applies policies (mTLS, retry, timeout, circuit breaking), and forwards it to the destination service's sidecar.

graph TB subgraph "Service A Pod" A[Service A Code] PA[Envoy Sidecar] end subgraph "Service B Pod" B[Service B Code] PB[Envoy Sidecar] end subgraph "Control Plane" CP[Istiod] CP -->|config push| PA CP -->|config push| PB end A -->|localhost| PA PA -->|mTLS| PB PB -->|localhost| B

Istio / Envoy Architecture

Istio is the most widely adopted service mesh. It consists of:

  • Data plane (Envoy proxies) — Sidecar proxies deployed alongside every service. They intercept all network traffic and enforce policies.
  • Control plane (Istiod) — A single binary that manages configuration, certificate issuance (for mTLS), and service discovery. It pushes configuration to all Envoy sidecars via xDS APIs.

What a Service Mesh Gives You

Capability Without Mesh (library-based) With Mesh (sidecar-based)
mTLS Each service manages certificates Automatic cert rotation, zero-trust by default
Retries / timeouts Each service implements its own Centrally configured, language-agnostic
Circuit breaking Library (Hystrix, resilience4j) in each service Configured in sidecar; app code unaware
Observability Instrumentation per service per language Automatic metrics, traces, access logs from proxy
Traffic management Custom load balancer config Canary routing, traffic splitting, fault injection

When a Service Mesh Is Overkill

  • Fewer than ~10 services — The operational overhead of managing the mesh (control plane, sidecar injection, debugging proxy issues) exceeds the benefit.
  • Homogeneous tech stack — If all services use the same language, a shared library can provide retries, circuit breaking, and observability without the proxy overhead.
  • Latency-critical paths — Each sidecar hop adds ~1-3ms of latency. For ultra-low-latency systems, this matters.
  • Small team — Someone needs to be the mesh expert. If nobody on the team understands Envoy configuration, debugging mesh issues becomes a black hole.
Staff-Level Signal: A staff engineer distinguishes between what the team needs now and what it will need at scale. Start with a shared library approach. Plan for a mesh migration when you cross ~20 services or when polyglot services make library-based solutions impractical. Articulate this progression in interviews.

5.2 Asynchronous Communication

Asynchronous communication decouples services in time: the sender does not wait for the receiver to process the message. This improves availability (sender succeeds even if receiver is down) and absorbs traffic spikes (queue acts as a buffer).

  • Message Queues (point-to-point) — A producer sends a message to a queue; exactly one consumer processes it. Examples: SQS, RabbitMQ. Good for task distribution and work queues.
  • Event Streams (pub/sub) — A producer publishes an event to a topic; multiple consumers can subscribe independently. Examples: Kafka, AWS SNS+SQS. Good for event-driven architectures where multiple services need to react to the same event.
# Example: Publishing an order event to Kafka
from kafka import KafkaProducer
import json

producer = KafkaProducer(
    bootstrap_servers=['kafka:9092'],
    value_serializer=lambda v: json.dumps(v).encode('utf-8')
)

def publish_order_created(order):
    event = {
        "event_type": "OrderCreated",
        "order_id": order.id,
        "user_id": order.user_id,
        "items": [{"sku": i.sku, "qty": i.qty} for i in order.items],
        "total": str(order.total),
        "timestamp": order.created_at.isoformat()
    }
    producer.send("orders", key=order.id.encode(), value=event)
    producer.flush()

5.3 Service Discovery

In a dynamic environment where service instances are created and destroyed (autoscaling, deployments), hard-coding addresses is not viable. Service discovery solves “where is the Order service right now?”

  • Client-side discovery: The client queries a service registry (Consul, Eureka) and picks an instance using a load-balancing strategy. Pros: no extra hop. Cons: client must implement discovery logic per language.
  • Server-side discovery: The client sends requests to a load balancer (AWS ALB, Kubernetes Service) that queries the registry and routes. Pros: client is simple. Cons: extra network hop, potential bottleneck.
  • DNS-based: Kubernetes Services provide DNS-based discovery out of the box (order-service.default.svc.cluster.local).

5.4 Resilience Patterns

Circuit Breaker Pattern

When a downstream service is failing, continuing to send requests wastes resources and increases latency. The circuit breaker monitors failure rates and “opens” the circuit when a threshold is exceeded, immediately returning errors without making the call. After a timeout, it moves to “half-open” and allows a few test requests. If they succeed, the circuit closes; if they fail, it opens again.

Requests pass through. Failures counted. end note note right of Open Fail fast. Return error or fallback immediately. No calls to downstream. end note note right of HalfOpen Allow limited test traffic. If success, close circuit. If failure, reopen. end note 
# Circuit breaker implementation sketch
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30, test_requests=3):
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.test_requests = test_requests
        self.last_failure_time = None
        self.test_success_count = 0

    def call(self, func, *args, **kwargs):
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.test_success_count = 0
            else:
                raise CircuitOpenError("Circuit is open; failing fast")

        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_success(self):
        if self.state == CircuitState.HALF_OPEN:
            self.test_success_count += 1
            if self.test_success_count >= self.test_requests:
                self.state = CircuitState.CLOSED
                self.failure_count = 0
        else:
            self.failure_count = 0

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN

Retry with Exponential Backoff and Jitter

Retrying immediately after a failure often makes things worse — all clients retry at the same time, creating a thundering herd. Exponential backoff spaces retries out (1s, 2s, 4s, 8s...). Adding jitter (randomness) prevents synchronized retries across clients. 

import random
import time

def retry_with_backoff(func, max_retries=5, base_delay=1.0, max_delay=60.0):
    """Retry with exponential backoff and full jitter."""
    for attempt in range(max_retries):
        try:
            return func()
        except RetryableError:
            if attempt == max_retries - 1:
                raise  # Last attempt, propagate error

            # Exponential backoff: 2^attempt * base_delay
            exp_delay = min(base_delay * (2 ** attempt), max_delay)

            # Full jitter: random between 0 and exp_delay
            sleep_time = random.uniform(0, exp_delay)

            print(f"Attempt {attempt + 1} failed. Retrying in {sleep_time:.2f}s")
            time.sleep(sleep_time)

Timeout and Deadline Propagation

Every network call must have a timeout. Without one, a hung downstream service will consume threads/connections indefinitely, eventually cascading the failure upstream. But timeouts alone are not enough in a chain:

If Service A has a 5s timeout and calls Service B (which has a 5s timeout to Service C), the total could be 10s before A gets an error. Deadline propagation passes a single deadline through the chain. gRPC supports this natively: A sets a 5s deadline; B receives 4.8s remaining (accounting for network hop); C receives 4.5s remaining. Every service knows how much time is left.

Bulkhead Pattern

Named after the watertight compartments in a ship's hull, the bulkhead pattern isolates failures. If Service A communicates with Service B and Service C, separate thread pools (or connection pools) ensure that if Service B becomes slow and exhausts its pool, Service C is unaffected. 

// Bulkhead using separate thread pools (Java / Resilience4j)
BulkheadConfig config = BulkheadConfig.custom()
    .maxConcurrentCalls(25)       // Max 25 concurrent calls
    .maxWaitDuration(Duration.ofMillis(500))  // Wait max 500ms for a slot
    .build();

Bulkhead orderServiceBulkhead = Bulkhead.of("orderService", config);
Bulkhead paymentServiceBulkhead = Bulkhead.of("paymentService", config);

// Each downstream service has its own isolated thread pool
// If order service is slow, payment service calls are unaffected
Supplier<Order> decoratedCall = Bulkhead.decorateSupplier(
    orderServiceBulkhead,
    () -> orderClient.getOrder(orderId)
);
Staff-Level Signal: In interviews, demonstrate that you think about failure modes systematically. When you add a service call, immediately discuss: “What timeout do we set? What happens when it fails? Do we retry? Do we need a circuit breaker? How do we prevent cascading failures?” This signals operational maturity.

6. Data Management in Microservices

Database per Service, Sagas, and Event-Driven Sync

Database per Service Pattern

Each service owns its data store exclusively. No other service can access it directly — only through the service's API. This provides:

  • Schema independence — The Order service can change its schema without coordinating with the Catalog service.
  • Technology freedom — The Catalog service might use Elasticsearch for full-text search; the Order service uses PostgreSQL for ACID transactions; the Session service uses Redis for speed.
  • Independent scaling — The analytics database can be a columnar store with heavy read replicas, while the transaction database is a write-optimized OLTP system.

Shared Database Anti-Pattern

Why shared databases undermine microservices: If Service A and Service B both read/write the orders table, they are coupled at the data layer. Any schema migration requires coordinating both teams. A poorly indexed query from Service B can bring down Service A's performance. You have microservices in name only — it is a distributed monolith with a shared state.

The one legitimate exception is during a migration. Using the Strangler Fig pattern, both the old monolith and the new service may temporarily share a database. But this should be time-bounded with a clear plan to split.

Data Consistency Challenges

With database-per-service, you lose ACID transactions across service boundaries. Consider placing an order: you need to (1) create the order, (2) reserve inventory, and (3) charge the payment. Each step is in a different service with a different database. If the payment fails, you need to undo the inventory reservation and cancel the order.

Saga Pattern

A saga is a sequence of local transactions, each within a single service. If one step fails, compensating transactions undo the prior steps. See Module 15 for an in-depth treatment.

Two orchestration styles:

  • Choreography: Each service publishes events; the next service reacts. No central coordinator. Simple for 2-3 step sagas; becomes hard to follow with more steps.
  • Orchestration: A saga orchestrator (a separate service or workflow engine like Temporal) directs each step and handles compensation. Easier to understand and debug for complex flows.

Event-Driven Data Synchronization

When services need each other's data for queries (e.g., the Order service needs product names from the Catalog service), there are two approaches:

  1. Synchronous API call: Order service calls Catalog service at query time. Simple but creates runtime coupling (Catalog being down breaks Order queries).
  2. Event-driven materialized view: Catalog service publishes ProductUpdated events. Order service consumes these events and maintains a local read-only copy of the product data it needs. This is eventually consistent but highly available and performant.
# Event-driven materialized view in the Order service
class ProductCacheConsumer:
    """Consumes ProductUpdated events to maintain a local cache."""

    def handle_product_updated(self, event):
        self.local_db.upsert(
            table="product_cache",
            key=event["product_id"],
            data={
                "name": event["name"],
                "price": event["price"],
                "image_url": event["image_url"],
                # Only store what the Order service actually needs
            }
        )

    def handle_product_deleted(self, event):
        self.local_db.soft_delete(
            table="product_cache",
            key=event["product_id"]
        )
Interview Tip: When discussing data management, always clarify the consistency requirements. “Does the business require strong consistency here, or is eventual consistency acceptable?” For most read-heavy, non-financial operations, eventual consistency (with a few seconds of staleness) is acceptable and dramatically simplifies the architecture.

7. Deployment Patterns

Safe, Progressive Delivery Strategies

Independent deployability is a core benefit of microservices, but deploying services safely at scale requires deliberate strategies. The goal is always the same: minimize blast radius while maximizing deployment velocity.

Blue-Green Deployment

Maintain two identical production environments: Blue (current live) and Green (new version). Deploy the new version to Green, run smoke tests, then switch the router from Blue to Green. If something goes wrong, switch back instantly.

  • Pro: Near-instant rollback (just switch the router).
  • Con: Requires 2x infrastructure during deployment. Database schema changes are tricky (both versions must be compatible).

Canary Deployment

Route a small percentage of traffic (1-5%) to the new version while the rest continues on the old version. Monitor error rates, latency, and business metrics. Gradually increase traffic to 100% over minutes to hours.

  • Pro: Minimal blast radius; real production traffic validates the change.
  • Con: Requires sophisticated routing and monitoring. Must handle backward-compatible APIs during the transition.

Rolling Deployment

Update instances one at a time (or in small batches). Kubernetes does this by default with Deployment resources. At any point, some instances run the old version and some run the new.

  • Pro: No extra infrastructure needed. Built into Kubernetes.
  • Con: During rollout, two versions coexist — must handle backward compatibility. Rollback is slower (must roll forward or back).

Feature Flags

Decouple deployment from release. Deploy code that is hidden behind a flag; toggle the flag in a configuration system (LaunchDarkly, Unleash, Flagsmith) to enable it for specific users, percentages, or regions. 

# Feature flag usage
from feature_flags import FeatureFlags

flags = FeatureFlags(config_url="https://flags.internal/api")

def get_recommendations(user_id):
    if flags.is_enabled("new_recommendation_engine", user_id=user_id):
        # New ML-based recommendations (being tested on 10% of users)
        return new_recommendation_engine.recommend(user_id)
    else:
        # Existing collaborative filtering
        return legacy_recommendations.recommend(user_id)

Feature flags are particularly powerful because they let you ship incomplete features (hidden behind flags), run A/B tests, do targeted rollouts (e.g., enable for employees first), and instantly kill a feature without a deployment if something goes wrong.

Feature Flag Hygiene: Feature flags are technical debt if not cleaned up. Every flag should have an owner, a planned removal date, and an alert if it stays in the codebase past its expiry. Stale flags create confusing code paths and combinatorial testing nightmares.
Staff-Level Signal: In an interview, mention that you would use canary deployments with automated rollback based on metrics thresholds. Explain that you monitor both technical metrics (p99 latency, error rate) and business metrics (conversion rate, revenue) during canary. This shows you think about deployment as a production safety concern, not just a CI/CD step.

8. Observability

The Three Pillars + Correlation

In a monolith, you can debug with a single stack trace. In a microservices architecture, a single user request may touch 10+ services. Without proper observability, you are debugging blind. Observability has three pillars: logs, metrics, and traces.

Distributed Tracing

A trace follows a single request across all services it touches. Each service creates a span (a unit of work with start time, end time, and metadata). Spans are linked by a shared trace ID and a parent-child relationship.

  • OpenTelemetry (OTel) — The industry standard for instrumentation. Language-specific SDKs auto-instrument common frameworks (HTTP, gRPC, database drivers). Vendor-neutral: sends data to Jaeger, Zipkin, Datadog, or any OTLP-compatible backend.
  • Jaeger — Open-source distributed tracing backend by Uber. Visualizes traces as Gantt charts showing timing across services.
# OpenTelemetry auto-instrumentation (Python)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanExporter
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter
from opentelemetry.instrumentation.flask import FlaskInstrumentor
from opentelemetry.instrumentation.requests import RequestsInstrumentor

# Setup: runs once at application startup
provider = TracerProvider()
provider.add_span_processor(BatchSpanExporter(OTLPSpanExporter()))
trace.set_tracer_provider(provider)

# Auto-instrument Flask and outbound HTTP calls
FlaskInstrumentor().instrument()
RequestsInstrumentor().instrument()

# Now every incoming request gets a trace, and every outbound
# HTTP call creates a child span automatically.

Centralized Logging

With 50 services running 3 instances each, SSH-ing into individual containers to read logs is not viable. Centralized logging aggregates all logs into a searchable system.

  • ELK Stack — Elasticsearch (storage + search), Logstash (ingestion + transformation), Kibana (visualization).
  • EFK Stack — Replace Logstash with Fluentd (lower resource usage, Kubernetes-native).
  • Cloud-native — AWS CloudWatch Logs, GCP Cloud Logging, Azure Monitor Logs.

Structured logging is essential. Emit JSON logs with consistent fields so they are machine-parseable:

Metrics Aggregation

  • Prometheus — Pull-based metrics collection. Services expose /metrics endpoints; Prometheus scrapes them at intervals. Excellent for time-series data. PromQL for querying.
  • Grafana — Dashboards and alerting. Connects to Prometheus (and many other data sources) to visualize metrics.

Key metrics every service should expose (RED method):

  • Rate — Requests per second.
  • Errors — Error rate (5xx, timeouts, etc.).
  • Duration — Latency percentiles (p50, p95, p99).

Correlation IDs

A correlation ID (or request ID) is generated at the edge (API gateway) and propagated through every service in the request chain via HTTP headers (e.g., X-Request-ID). Every log line, metric, and trace span includes this ID. When a user reports a problem, you search for their correlation ID and see the entire request flow across all services.

Health Checks and Readiness Probes

  • Liveness probe (/health/live) — “Is the process alive?” If this fails, the orchestrator restarts the container.
  • Readiness probe (/health/ready) — “Can this instance serve traffic?” If this fails, the load balancer stops sending traffic but does not restart the container. This handles cases like warming up caches or waiting for database connections.
# Flask health check endpoints
@app.route("/health/live")
def liveness():
    return {"status": "alive"}, 200

@app.route("/health/ready")
def readiness():
    checks = {
        "database": check_db_connection(),
        "cache": check_redis_connection(),
        "downstream": check_critical_dependencies(),
    }
    all_healthy = all(checks.values())
    status_code = 200 if all_healthy else 503
    return {"status": "ready" if all_healthy else "not_ready", "checks": checks}, status_code
Interview Tip: When designing any distributed system, proactively mention observability. “I would add distributed tracing with OpenTelemetry, structured logging with correlation IDs, and RED metrics per service.” This signals that you build production-ready systems, not just whiteboard architectures.

9. Conway's Law

How Organization Structure Shapes Architecture

“Any organization that designs a system will produce a design whose structure is a copy of the organization's communication structure.” — Melvin Conway, 1967

Conway's Law is not just an observation — it is a force of nature in software engineering. If you have a frontend team, a backend team, and a database team, you will get a three-tier architecture regardless of what the architecture diagram says, because that is how the teams communicate.

The Inverse Conway Maneuver

If Conway's Law says your architecture mirrors your org structure, then to get the architecture you want, organize your teams to match it. This is the Inverse Conway Maneuver, popularized by the book Team Topologies by Skelton and Pais.

Want microservices? You need autonomous cross-functional teams, each owning a service end-to-end (frontend, backend, database, ops). If you try to adopt microservices with functional teams (frontend team, backend team, DBA team), every feature will require cross-team coordination, defeating the purpose.

Team Topologies in Practice

  • Stream-aligned teams — Aligned to a business domain (Order team, Search team, Payments team). They own their services and can deliver end-to-end without blocking on other teams.
  • Platform teams — Provide self-service infrastructure (CI/CD, observability, service mesh, databases-as-a-service) so that stream-aligned teams do not each reinvent the operational wheel.
  • Enabling teams — Help other teams adopt new practices or technologies (e.g., helping teams adopt Kubernetes or distributed tracing).
  • Complicated subsystem teams — Own technically complex components (ML models, video encoding engines) that require deep specialist knowledge.
Staff-Level Signal: Staff engineers are expected to think about organizational dynamics, not just code. Mentioning Conway's Law and team topology in a system design interview demonstrates that you understand the sociotechnical system — that architecture cannot be designed in isolation from the humans who build and operate it. This is a strong differentiator from senior-level candidates who focus purely on technical components.

10. Real-World Architecture Examples

Netflix, Amazon, and Uber

Netflix: 700+ Microservices and Chaos Engineering

Netflix serves 200M+ subscribers across 190+ countries. Their architecture includes 700+ microservices running on AWS. Key architectural decisions:

  • Zuul as the API gateway — handles routing, auth, canary testing, and load shedding at the edge.
  • Eureka for service discovery — a REST-based registry where services register themselves and discover others.
  • Hystrix (now Resilience4j) for circuit breaking — pioneered the circuit breaker pattern at scale to prevent cascading failures.
  • Chaos Engineering — Netflix runs Chaos Monkey (randomly kills instances), Chaos Kong (simulates entire region failures), and Chaos Gorilla (takes down an entire availability zone). The philosophy: if your system cannot handle random failures in production, it will fail in ways you do not expect. Better to cause controlled failures regularly.
  • EVCache — Distributed cache (based on Memcached) serving 30+ million requests per second.

Amazon: Service-Oriented Architecture

In 2002, Jeff Bezos sent the famous “API Mandate” memo:

  1. All teams will expose their data and functionality through service interfaces.
  2. Teams must communicate with each other through these interfaces.
  3. There will be no other form of inter-process communication allowed.
  4. It does not matter what technology they use.
  5. All service interfaces must be designed from the ground up to be externalizable.
  6. Anyone who does not do this will be fired.

This memo birthed Amazon's service-oriented architecture and ultimately AWS itself. Today, a single page load on amazon.com calls 200-300 services. Each team owns their services, including ops (the “you build it, you run it” philosophy). This forced teams to build reliable, well-documented APIs because they knew other teams (and eventually external customers via AWS) would depend on them.

Uber: Domain-Oriented Microservice Architecture (DOMA)

By 2020, Uber had ~2,200 microservices. They found that the “pure” microservice model created chaos: too many services, too many dependencies, too much operational overhead. Their response was DOMA:

  • Domains: Group related microservices into domains (e.g., “Marketplace” domain contains pricing, matching, dispatch, and surge services).
  • Domain Gateways: Each domain exposes a single gateway that acts as the public API for that domain. Other domains call the gateway, not individual services. This dramatically reduces cross-domain coupling.
  • Layers: Domains are organized into layers (infrastructure, business logic, product, presentation) with strict dependency rules — upper layers can depend on lower layers but not vice versa.

DOMA is a pragmatic middle ground: you get the benefits of microservices (team autonomy, independent scaling) with the organizational clarity of a structured architecture. It is particularly relevant for companies that have already decomposed too far and need to add structure back.

11. Interview Application

How to Discuss Microservices in System Design Interviews

The Decision Framework

When the interviewer asks “Would you use microservices here?”, do not answer yes or no immediately. Walk through a decision framework:

  1. What is the team size and org structure? — Fewer than 20 engineers? Probably stay monolithic. Multiple autonomous teams? Microservices may make sense.
  2. What are the scaling requirements? — Do different components have radically different scaling profiles? If the search engine needs 100x the compute of the admin panel, separate them.
  3. What is the deployment velocity requirement? — Does the team need to deploy multiple times per day? Do different features change at different rates? Independent deployability is a key microservice benefit.
  4. Do you have the infrastructure maturity? — CI/CD, container orchestration, observability, service mesh. Without these, microservices amplify operational pain.

Common Trade-Offs to Articulate

Benefit Cost
Independent deployment Distributed system complexity
Team autonomy Inter-service API contract management
Technology freedom Operational overhead of multiple tech stacks
Independent scaling Network latency and data consistency challenges
Fault isolation Requires sophisticated observability and resilience patterns

What Interviewers Look For

  • Nuanced thinking — Not “microservices are always better” but “given the scale and team size, here is what I would choose and why.”
  • Awareness of operational complexity — Mentioning observability, deployment strategies, and failure modes unprompted shows production experience.
  • Data management strategy — How do you handle cross-service data consistency? Sagas? Event-driven materialized views?
  • Evolutionary architecture — Starting with a monolith and describing when and how you would decompose shows pragmatism.
  • Organizational awareness — Mentioning Conway's Law and team topology shows you think beyond code.

Red Flags to Avoid

Red Flags:
  • Jumping to microservices without justification
  • Ignoring data consistency challenges across services
  • Not discussing failure modes (what happens when Service B is down?)
  • Creating a “distributed monolith” (shared database, synchronous chains, coordinated deployments)
  • Not mentioning observability or monitoring
  • Decomposing too finely (one service per REST endpoint)

Sample Interview Dialogue

Interviewer: “Design a food delivery system like DoorDash.”

Strong answer (excerpt): “For the initial architecture, I would start with a modular monolith with clear module boundaries: Order Management, Restaurant Catalog, Delivery Tracking, Payments, and Notifications. Each module has its own database schema and interacts with others through well-defined internal APIs.

As the system scales to support millions of orders per day with hundreds of engineers, I would extract services starting with the highest-value candidates: Delivery Tracking (different scaling profile — high-frequency GPS updates), Payments (regulatory requirements, separate PCI scope), and Notifications (can be fully asynchronous). I would use the Strangler Fig pattern, introducing a routing layer that gradually shifts traffic to new services.

For inter-service communication, Order Management would publish events (OrderCreated, OrderCanceled) to Kafka. Delivery Tracking and Notifications consume these asynchronously. For the payment flow, I would use a saga with an orchestrator because it is critical-path and needs reliable compensation on failure.

I would instrument everything with OpenTelemetry from day one, use canary deployments with automated rollback based on p99 latency and error rate, and require circuit breakers on all synchronous service calls.”

Suggested Flashcards

5 Key Flashcard Q&A Pairs (for Anki/CSV export)

  1. Q: What is the Strangler Fig pattern?
    A: An incremental migration strategy where new functionality is built as separate services while the monolith continues serving existing traffic. A proxy routes traffic to either old or new code. Over time, the monolith shrinks until it can be decommissioned.
  2. Q: What are the three states of a circuit breaker?
    A: Closed (normal operation, tracking failures), Open (fail fast, no downstream calls), Half-Open (allow limited test requests to check recovery).
  3. Q: What is the difference between an API Gateway and a Service Mesh?
    A: API Gateway handles north-south traffic (client-to-service): routing, auth, rate limiting. Service Mesh handles east-west traffic (service-to-service): mTLS, retries, circuit breaking, observability via sidecar proxies.
  4. Q: What is Conway's Law and the Inverse Conway Maneuver?
    A: Conway's Law: system architecture mirrors org communication structure. Inverse Conway Maneuver: organize teams to match the desired architecture (e.g., cross-functional teams for microservices).
  5. Q: Why is “database per service” important in microservices?
    A: It ensures services are decoupled at the data layer. Shared databases create hidden coupling (coordinated schema changes, performance interference), resulting in a distributed monolith.

tl;dr

Microservices trade development simplicity for operational complexity. They are justified when team scale, deployment velocity, or heterogeneous scaling requirements outgrow what a monolith can support. The keys are: decompose along DDD bounded contexts (not arbitrarily), use an API gateway for edge concerns and asynchronous communication to reduce coupling, invest heavily in observability (traces, structured logs, metrics), and deploy progressively with canary releases. Conway's Law means your architecture will mirror your org — so structure your teams to match the architecture you want.
