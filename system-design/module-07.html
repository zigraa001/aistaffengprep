Module 17: RESHADE Framework Mastery

Learning Objectives

  • Master the RESHADE framework for structuring 45-60 minute system design interviews
  • Learn time allocation, pacing, and transition strategies for each phase
  • Understand what distinguishes staff-level answers from senior-level answers
  • Practice with worked examples and template scripts
  • Identify and avoid the most common interview anti-patterns

The RESHADE Framework Overview

RESHADE is a structured approach to system design interviews that ensures you cover all the areas interviewers evaluate. The letters stand for: Requirements, Estimation, Storage Schema, High-Level Design, API Design, Deep Dive, Edge Cases.

flowchart LR R["R: Requirements\n5-8 min"] --> E["E: Estimation\n3-5 min"] E --> S["S: Storage Schema\n3-5 min"] S --> H["H: High-Level Design\n10-15 min"] H --> A["A: API Design\n3-5 min"] A --> D["D: Deep Dive\n15-20 min"] D --> E2["E: Edge Cases\n5-10 min"] style R fill:#4CAF50,color:#fff style E fill:#2196F3,color:#fff style S fill:#FF9800,color:#fff style H fill:#9C27B0,color:#fff style A fill:#00BCD4,color:#fff style D fill:#F44336,color:#fff style E2 fill:#607D8B,color:#fff
Why RESHADE works: It mirrors how interviewers at Google, Meta, Amazon, and Microsoft score system design interviews. Each phase maps to an evaluation dimension. Skipping any phase leaves points on the table. The order is deliberate — each phase builds on the previous one.

Time Management: The 45-Minute Timeline

Pacing tip: If you find yourself spending 15+ minutes on Requirements, you're going too deep too early. The interviewer wants breadth first, then depth. Say: "I have more questions, but let me sketch the high-level design first, and we can refine as we go."

Phase 1: Requirements (R) — 5-8 Minutes

Why This Phase Matters ▼

Requirements clarification is where staff candidates separate themselves immediately. A senior engineer might ask 2-3 generic questions. A staff engineer asks targeted questions that reveal they already understand the hard parts of the problem.

What interviewers evaluate:

  • Can you identify the core use cases vs. nice-to-haves?
  • Do you think about non-functional requirements unprompted?
  • Can you scope the problem to fit a 45-minute discussion?
  • Do you make the interviewer's implicit assumptions explicit?
Functional Requirements: Template ▼

Start by listing 3-5 core features. Say them out loud and confirm with the interviewer:

Script: "Let me make sure I understand the core requirements. The system needs to: [1], [2], [3]. Are there any other critical features, or should I focus on these for now?"

Worked Example: "Design Twitter"

Functional Requirements (confirmed with interviewer): 1. Post tweets (text, up to 280 chars, optional images) 2. Follow/unfollow users 3. View home timeline (tweets from people you follow) 4. View user profile & their tweets Out of scope (explicitly stated): - Search, trending, DMs, notifications, ads - "I'll focus on the core posting and timeline, happy to discuss search later if time permits."

Common Mistake: Boiling the Ocean

Candidates who try to design everything (search, notifications, ads, analytics) end up with a shallow design. It's much better to deeply design 3 core features than shallowly design 10.

Non-Functional Requirements: The Staff-Level Checklist ▼

Non-functional requirements (NFRs) are where staff candidates shine. Ask about these explicitly:

NFR Question to Ask Twitter Example
Scale"How many DAU? Read/write ratio?"500M DAU, 100:1 read-heavy
Latency"What's acceptable for timeline load?"p99 < 500ms for timeline
Availability"What's our target uptime?"99.99% (52 min downtime/year)
Consistency"Strong or eventual? Does timeline need real-time?"Eventual OK — seconds delay acceptable
Durability"Can we ever lose a tweet?"No data loss, replicated storage
Geography"Single region or global?"Global, multi-region
Staff-Level Signal: Don't just list NFRs — explain the tension between them. "We want high availability AND strong consistency, but CAP tells us we can't have both during a partition. For a social media timeline, I'd favor availability over consistency — it's acceptable if a tweet appears a few seconds late, but it's not acceptable if the app is down."

Phase 2: Estimation (E) — 3-5 Minutes

Back-of-Envelope Calculations ▼

The goal isn't perfect math — it's showing you can reason about scale and make informed design decisions based on the numbers.

Worked Example: Twitter Estimation

# Traffic DAU: 500M users Tweets per user per day: 2 (average, including power users) Daily tweets: 500M x 2 = 1B tweets/day Write QPS: 1B / 86,400 = ~12,000 tweets/sec Peak write QPS: 12,000 x 3 = ~36,000 tweets/sec Timeline reads per user: 10 per day Read QPS: 500M x 10 / 86,400 = ~58,000 reads/sec Peak read QPS: ~175,000 reads/sec # Storage Avg tweet size: 280 chars + metadata = ~500 bytes Daily tweet storage: 1B x 500B = 500 GB/day 5-year storage: 500 GB x 365 x 5 = ~900 TB # Media (10% of tweets have images) 100M images/day x 500KB avg = 50 TB/day

Quick Reference: Powers of 2

Value Approx Name
2^101 Thousand1 KB
2^201 Million1 MB
2^301 Billion1 GB
2^401 Trillion1 TB

2.5M~2.5 x 10^6Seconds/month

Common Mistake: Spending Too Long on Math

Don't optimize the arithmetic. Round aggressively (500M, not 487M). The interviewer cares that you know order of magnitude, not the exact number. If your estimate leads to "we need about 1 TB of storage per day", that's the signal — it tells you that you need distributed storage, not a single machine.

Script: "Based on these numbers, we're looking at ~175K peak read QPS and ~36K peak write QPS with roughly 500 GB of new data per day. This tells me we need: horizontal scaling for the read path (can't serve this from a single machine), a write-behind pattern to handle burst writes, and a distributed storage system for the growing dataset."

Phase 3: Storage Schema (S) — 3-5 Minutes

Data Model Design ▼

Key Decisions to Articulate

  • SQL vs NoSQL: Explain why based on your access patterns, not just "NoSQL scales better"
  • Partition key: Choose high cardinality, even distribution. Explain hot partition risk.
  • Denormalization: Explicitly state the read/write trade-off
  • Indexes: Design for your query patterns, not for completeness

Worked Example: Twitter Data Model

# Tweets table (Cassandra / DynamoDB - write-heavy, partition by user) tweets: tweet_id: UUID (snowflake ID for time-ordering) user_id: INT (partition key) content: TEXT media_urls: LIST<TEXT> created_at: TIMESTAMP like_count: INT retweet_count: INT # User table (PostgreSQL - relational, strong consistency for profiles) users: user_id: INT (PK) username: VARCHAR(50) display_name: VARCHAR(100) follower_count: INT following_count: INT # Follows table (graph relationship) follows: follower_id: INT followee_id: INT created_at: TIMESTAMP PRIMARY KEY (follower_id, followee_id) # Timeline cache (Redis sorted set) timeline:{user_id} = ZSET of tweet_ids, scored by timestamp
Staff-Level Signal: "I'm using Cassandra for tweets because our write volume (36K/sec peak) exceeds what a single PostgreSQL instance can handle, and tweets are naturally partitioned by user_id. I'm keeping user profiles in PostgreSQL because we need strong consistency for follower counts and profile updates, and the write volume is much lower. The timeline is materialized in Redis because the read path needs sub-100ms latency."

Phase 4: High-Level Design (H) — 10-15 Minutes

Component Architecture ▼

This is the centerpiece of your answer. Draw a clear architecture diagram with labeled components. Walk through the write path and read path separately.

flowchart TB C[Clients] --> LB[Load Balancer] LB --> API[API Gateway] API --> TS[Tweet Service] API --> TL[Timeline Service] API --> US[User Service] TS --> TDB[(Tweet Store\nCassandra)] TS --> MQ[Message Queue\nKafka] US --> UDB[(User DB\nPostgreSQL)] TL --> RC[Timeline Cache\nRedis] MQ --> FO[Fan-out Service] FO --> RC FO --> NS[Notification Service] API --> CDN[CDN\nMedia] TS --> S3[Object Store\nS3] style MQ fill:#FF9800,color:#fff style RC fill:#F44336,color:#fff style FO fill:#9C27B0,color:#fff

Walk Through the Write Path

Script: "When a user posts a tweet: (1) the request hits the API Gateway, (2) Tweet Service validates and persists to Cassandra, (3) an event is published to Kafka, (4) Fan-out Service reads the event and pushes the tweet ID to each follower's timeline cache in Redis. For users with millions of followers (celebrities), we use a hybrid approach — fan-out on write for normal users, fan-out on read for celebrities."

Walk Through the Read Path

Script: "When a user opens their timeline: (1) Timeline Service reads tweet IDs from Redis sorted set, (2) hydrates full tweet data from Cassandra in parallel, (3) merges in any celebrity tweets (fan-out on read), (4) returns the assembled timeline. Cache hit rate should be ~99% since we pre-compute timelines."

Component Checklist

For every system design, consider these building blocks:

  • ☐ Load Balancer — L7 for HTTP routing, health checks
  • ☐ API Gateway — Auth, rate limiting, request routing
  • ☐ Application Services — Stateless, horizontally scalable
  • ☐ Cache Layer — Redis/Memcached for read-heavy paths
  • ☐ Primary Database — SQL or NoSQL based on access patterns
  • ☐ Message Queue — Kafka/SQS for async processing
  • ☐ CDN — For static content and media
  • ☐ Object Storage — S3 for files, images, videos
  • ☐ Search Index — Elasticsearch for full-text search
  • ☐ Monitoring — Metrics, logging, tracing

Phase 5: API Design (A) — 3-5 Minutes

Defining the Interface ▼

Worked Example: Twitter APIs

# Post a tweet POST /api/v1/tweets Headers: Authorization: Bearer {token}, Idempotency-Key: {uuid} Body: { "content": "Hello world", "media_ids": ["abc123"] } Response: 201 Created { "tweet_id": "...", "created_at": "..." } # Get home timeline (cursor-based pagination) GET /api/v1/timeline?cursor={last_tweet_id}&limit=20 Response: 200 OK { "tweets": [...], "next_cursor": "def456", "has_more": true } # Follow a user POST /api/v1/users/{user_id}/follow Response: 200 OK { "following": true }

Key Points to Mention

  • Pagination: Use cursor-based for feeds (consistent with new data arriving)
  • Idempotency: Include idempotency keys for write operations
  • Versioning: URL path versioning (e.g., /v1/) for public APIs
  • Rate limiting: Return X-RateLimit-* headers
  • Auth: Bearer tokens, mention OAuth 2.0

Phase 6: Deep Dive (D) — 15-20 Minutes

The Most Important Phase for Staff Candidates ▼

This phase is where the interviewer goes deep on 1-2 areas. The key skill is proactively identifying the hardest problem and diving in before being asked.

Script: "The most interesting technical challenge here is the fan-out problem. Let me walk through how I'd handle it, because the naive approach breaks at our scale."

Deep Dive Topics to Prepare

For any system, be ready to go deep on:

Worked Example: Fan-out Deep Dive

flowchart TB subgraph "Fan-out on Write (normal users)" T1["User posts tweet"] --> K1["Kafka event"] K1 --> FO1["Fan-out workers"] FO1 --> R1["Push to each\nfollower's timeline\nin Redis"] end subgraph "Fan-out on Read (celebrities)" T2["Timeline request"] --> TLS["Timeline Service"] TLS --> R2["Read pre-computed\ntimeline from Redis"] TLS --> C2["Fetch celebrity tweets\non demand"] R2 --> M["Merge &\nRank"] C2 --> M end

Why hybrid? A user with 50M followers would generate 50M Redis writes per tweet. At 500 bytes per timeline entry, that's 25 GB of writes for a single tweet. Instead, we only fan-out on write for users with <10K followers (99% of users), and merge celebrity tweets at read time (adds ~10ms latency but saves enormous write amplification).

Phase 7: Edge Cases & Failure Modes (E) — 5-10 Minutes

Production Readiness & Resilience ▼

Staff candidates proactively discuss what can go wrong. This shows operational maturity.

Failure Scenario Checklist

  • Database failure: What happens when the primary DB goes down? (Failover to replica, read-only mode for writes)
  • Cache cold start: What if Redis is restarted and timelines are empty? (Rebuild from DB, serve stale data with warning)
  • Message queue backlog: What if Kafka consumers fall behind? (Monitor consumer lag, add workers, circuit break non-critical consumers)
  • Thundering herd: Celebrity posts a tweet to 50M followers simultaneously. (Rate-limit fan-out, stagger delivery, use TTL jitter)
  • Network partition: US-East can't reach US-West. (Serve from local region, eventual sync when partition heals)

Monitoring & Alerting

Key metrics to monitor: - Timeline latency: p50, p95, p99 (alert if p99 > 500ms) - Fan-out lag: time between tweet creation and timeline delivery - Cache hit rate: should be >99% (alert if drops below 95%) - Error rate: 5xx responses (alert if >0.1%) - Kafka consumer lag: messages behind (alert if >10K)

Cost Considerations

At staff level, mention cost trade-offs: "Storing pre-computed timelines in Redis for 500M users requires ~500 GB of memory (assuming 1KB per user). At AWS pricing, that's about $50K/month for Redis. This is justified because it reduces our read path latency from 200ms (DB query) to 5ms (Redis), which directly impacts user engagement."

Common Interviewer Follow-ups

15 Questions and How to Handle Them ▼
Question How to Handle
"What happens at 10x scale?"Identify the first bottleneck (usually DB or cache), propose sharding or architectural change
"What if the database goes down?"Discuss replication, failover, read from cache, queue writes for replay
"How would you handle a hot partition?"Key splitting, local caching, dedicated shard for hot keys
"Is this strongly consistent?"Name your consistency model, explain why it's appropriate for the use case
"How would you deploy this safely?"Canary deployment, feature flags, rollback plan, smoke tests
"What metrics would you track?"RED method (Rate, Errors, Duration), business metrics, SLOs
"Why not use [alternative technology]?"Acknowledge the alternative, compare trade-offs, justify your choice
"How does this handle network partitions?"Reference CAP, explain your availability/consistency choice per component

Anti-Patterns: What NOT to Do

Common Mistakes That Signal "Not Staff Level" ▼

Jumping to Solution

Starting to draw architecture before understanding requirements. Shows you can't scope a problem.

Buzzword Bingo

"We'll use Kafka, Redis, Kubernetes, and a microservice mesh." Without explaining why each component is needed.

Happy Path Only

Only discussing the success case. Never mentioning what happens during failures, partitions, or overload.

Monologue Mode

Talking for 10+ minutes without checking in. This is a conversation, not a presentation.

Ignoring Numbers

"We'll shard the database." Okay, but how many shards? What's the partition key? What's the data size per shard?

Over-Engineering

Designing for 1B users when the problem says 1M. Match your complexity to the actual requirements.

Communication Strategies

Think Aloud

Verbalize your thought process: "I'm considering two approaches here... Option A gives us X but costs Y. Option B..."

Draw Diagrams

Visual communication is clearer. Use boxes for services, cylinders for databases, arrows for data flow.

Check In Regularly

"Does this make sense so far? Should I go deeper on any of these components, or should I move on to [next phase]?"

Drive the Conversation

Don't wait to be led. Proactively identify the hardest problem and say "Let me dive into that."
