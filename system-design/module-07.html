Module 7: Interview Framework Mastery

Learning Objectives

  • Master the RESHADE framework for system design interviews
  • Learn capacity estimation techniques
  • Develop effective communication strategies
  • Practice structured problem-solving approach

The RESHADE Framework

flowchart LR R[Requirements
5-8 min] --> E[Estimation
3-5 min] E --> S[Storage Schema
3-5 min] S --> H[High-Level Design
10-15 min] H --> A[API Design
3-5 min] A --> D[Deep Dive
15-20 min] D --> E2[Edge Cases
5-10 min]

Phase 1: Requirements

Clarifying Requirements (5-8 min) ▼

Functional Requirements

  • What are the core features?
  • Who are the users?
  • What operations are supported?

Non-Functional Requirements

  • Scale: How many users? How much data?
  • Performance: Latency requirements?
  • Availability: What's acceptable downtime?
  • Consistency: Strong or eventual?
Template Questions:
  1. "What are the most important features to focus on?"
  2. "How many daily active users should we design for?"
  3. "What's the read/write ratio?"
  4. "Are there any latency requirements?"
  5. "Should we prioritize consistency or availability?"

Phase 2: Estimation

Capacity Planning (3-5 min) ▼

Storage Estimation

Users: X million Items per user: Y Size per item: Z bytes Total storage = X × Y × Z Growth rate = % per year 5-year storage = Total × (1 + growth)^5

Traffic Estimation

DAU: X million Actions per user: Y Total daily actions: X × Y QPS = Total / 86,400 Peak QPS = QPS × 2-3 (typical diurnal peak) # Note: special events (Black Friday, World Cup) can be 5-10x

Phase 3: Storage Schema

Data Model & Storage (3-5 min) ▼

Key Decisions

  • SQL vs NoSQL: Does the data have strong relational integrity needs (e.g., financial transactions)? Or is it denormalized / access-pattern-driven (e.g., user timelines)?
  • Partition key: Choose a key with high cardinality and even distribution. Avoid hot partitions (e.g., celebrity user IDs in a social network).
  • Indexing strategy: Define your access patterns first, then add indexes. Secondary indexes in distributed databases (Cassandra, DynamoDB) have significant cost and limitations.

Staff-Level Signal

Go beyond just listing tables. Explain why you chose a particular schema — e.g., “I’m denormalizing the user profile into the message table to avoid a cross-partition join on the read path, accepting the write amplification trade-off.”

Phase 4: High-Level Design

Component Architecture (10-15 min) ▼
flowchart TB C[Clients] --> LB[Load Balancer] LB --> API[API Servers] API --> Cache[Cache Layer] API --> DB[(Database)] API --> Queue[Message Queue] Queue --> Workers[Background Workers] API --> Storage[Object Storage]

Component Checklist

  • ☐ Load Balancer (for high availability)
  • ☐ API Servers (stateless, horizontally scalable)
  • ☐ Cache Layer (for read-heavy workloads)
  • ☐ Database (SQL vs NoSQL decision)
  • ☐ Message Queue (for async processing)
  • ☐ CDN (for static content)
  • ☐ Object Storage (for files, images)

Phase 5: API Design

Defining the Interface (3-5 min) ▼
  • Define the core API endpoints (REST or RPC) with request/response shapes.
  • Specify pagination strategy (cursor-based is usually better than offset for large datasets).
  • Include authentication, rate limiting headers, and idempotency keys where relevant.
  • Consider backward compatibility — how will you version the API?

Staff-Level Signal

Discuss API contracts as the system’s external promise. Mention how you’d handle breaking changes (e.g., URL versioning vs. content negotiation), and how the API design impacts client caching (ETags, Cache-Control).

Phase 6: Deep Dive

Drilling Into Critical Components (15-20 min) ▼

This is the most important phase for staff-level candidates. The interviewer will pick 1-2 areas and go deep. Prepare to discuss:

  • Scaling bottlenecks: Which component is the first to break at 10x scale? What’s your strategy (shard, cache, queue, or redesign)?
  • Consistency model: Walk through a specific read/write flow and explain what happens during a network partition or node failure.
  • Data pipeline: How does data flow from write to being queryable? What is the end-to-end latency? Where could data be lost?
  • Operational concerns: How do you deploy changes safely (canary, blue-green)? What metrics and alerts would you set up? How do you debug a latency spike in production?

Staff-Level Signal

Don’t wait for the interviewer to ask — proactively identify the hardest part of the design and say “I think the trickiest part here is X, let me walk through how I’d handle it.” This shows technical judgment.

Phase 7: Edge Cases & Failure Modes

Resilience & Production Readiness (5-10 min) ▼
  • Failure scenarios: What happens when the database is down? Cache goes cold? A downstream service times out?
  • Data integrity: How do you handle partial failures (e.g., payment charged but order not created)?
  • Hot spots: A viral post, a celebrity user, a flash sale — how does the system degrade gracefully?
  • Security: Input validation, rate limiting, access control, encryption at rest/in transit.
  • Cost: At scale, infrastructure cost matters. Mention cost-aware choices (e.g., tiered storage, reserved instances, caching to reduce DB load).

Communication Tips

Think Aloud

Verbalize your thought process. Interviewers want to understand HOW you think.

Draw Diagrams

Visual communication is clearer. Use boxes, arrows, and labels.

Discuss Trade-offs

There's no perfect solution. Explain why you chose option A over B.

Ask Questions

Clarify before diving in. Shows you understand requirements matter.
