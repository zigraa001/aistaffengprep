Module 7: Data Consistency Patterns

Understanding consistency models, consensus protocols, and conflict resolution strategies that underpin every distributed system — and how to reason about them in a staff-level interview.

What You'll Learn

  • The full spectrum of consistency models, from linearizability to eventual consistency
  • How consensus protocols (Paxos, Raft) achieve agreement across replicas
  • Vector clocks, Lamport timestamps, and logical time for ordering events
  • CRDTs: data structures that converge without coordination
  • Quorum systems, sloppy quorums, and tunable consistency
  • Gossip protocols and anti-entropy mechanisms
  • How Google Spanner, CockroachDB, and DynamoDB implement consistency in production
  • A decision framework for choosing the right consistency level in interviews

1. Why Consistency Matters

The Fundamental Problem

The moment you replicate data — for durability, availability, or latency — you face a question: what does a reader see while a write is in flight? The answer defines your consistency model, and every design decision you make in a distributed system flows from that choice.

Consistency is not a binary "strong vs. weak" toggle. It is a spectrum, and different parts of the same system can live at different points on that spectrum. A banking ledger demands linearizability; a social media "like" counter can tolerate seconds-old staleness without any user noticing. Understanding where on the spectrum each data path belongs — and articulating why — is one of the clearest staff-level signals an interviewer can observe.

Real-World Impact

Scenario Required Consistency Why
Bank account transfer Linearizable / Serializable Overdraft = real money lost. Double-spend = regulatory violation.
Flight seat reservation Serializable Double-booking a seat is unacceptable; but millisecond staleness on the seat map UI is fine.
Shopping cart (Amazon style) Eventual + CRDTs Must always be writable (availability first); conflicts merged on read.
Social media "like" count Eventual Showing 4,201 vs. 4,203 for a few seconds is invisible to users.
Collaborative document editing Causal + CRDTs / OT Users must see their own edits immediately, and see co-authors' edits in causal order.
DNS propagation Eventual (minutes/hours) TTL-based caching; staleness tolerable for hours.

Staff-Level Signal

When you discuss consistency in an interview, always anchor it to a user-visible consequence. "We need strong consistency here because a user who just transferred money must never see the old balance" is far stronger than "we pick CP because CAP theorem says so."

The Consistency Spectrum

Think of consistency as a spectrum from strongest (most expensive) to weakest (most available):

Linearizable Single-copy illusion
Sequential Global total order
Causal Respects causality
Session / RYW Per-client guarantees
Eventual Converges... eventually

Moving left increases coordination cost (more round trips, higher tail latency, lower availability during partitions). Moving right increases application complexity (developers must handle stale reads, conflicts, and anomalies). The art of system design is finding the sweet spot for each data path.

2. Consistency Models

Linearizability (Strongest)

Linearizability is the gold standard. It provides the illusion that there is only a single copy of the data and that every operation takes effect atomically at some point between its invocation and response. In other words, once a write completes, every subsequent read (by any client, on any replica) will see that write or something newer.

Formal Definition

A history of operations is linearizable if we can assign each operation a single point in real time (its linearization point) such that the result of every operation is the same as if they were executed sequentially in linearization-point order.

Why It's Expensive

  • Coordination required: Every write must be acknowledged by a majority of replicas before returning.
  • Read cost: Reads must either go to the leader or use a quorum read to avoid stale data.
  • Availability trade-off: During a network partition, a linearizable system must choose to reject some operations (CP in CAP terms).
  • Latency: Cross-datacenter linearizability means round trips measured in tens of milliseconds.

When You Need It

  • Leader election and distributed locks (e.g., ZooKeeper, etcd)
  • Unique constraint enforcement (username uniqueness)
  • Financial transactions where double-spend is unacceptable
  • Compare-and-swap operations

Systems That Provide It

Google Spanner (via TrueTime), etcd (Raft-based), ZooKeeper (ZAB protocol), CockroachDB (serializable with linearizable reads), FoundationDB.

Sequential Consistency

Sequential consistency relaxes linearizability slightly. It guarantees that all operations appear to execute in some total order, and that each client's operations appear in the order they were issued — but this total order need not correspond to real-time ordering.

Key Difference from Linearizability

With linearizability, if Client A's write completes before Client B's read begins (in real time), Client B must see that write. With sequential consistency, Client B might see an older value, as long as all clients agree on the same total order of operations.

Think of it like this: linearizability = real-time ordering respected. Sequential consistency = some total order exists that is consistent with each client's program order.

Where It Appears

  • ZooKeeper reads: By default, reads can be served from any replica (not linearizable) but all clients see the same order of updates.
  • Multi-core CPUs: Many CPU memory models provide sequential consistency (or weaker) rather than linearizability, for performance.

Interview Pitfall

Many candidates conflate linearizability and sequential consistency. If an interviewer asks "Is ZooKeeper linearizable?" the precise answer is: "Writes are linearizable (all go through the leader), but reads are sequentially consistent by default — you need a sync() call for a linearizable read."

Causal Consistency

Causal consistency guarantees that operations that are causally related are seen by all nodes in the same order. Operations that are concurrent (not causally related) may be seen in different orders on different nodes.

Happens-Before Relationship

Event A happens before event B (written A → B) if:

  1. A and B are on the same process and A occurred before B in program order.
  2. A is a send and B is the corresponding receive of a message.
  3. There exists C such that A → C and C → B (transitivity).

If neither A → B nor B → A, then A and B are concurrent. Causal consistency says: if A → B, then every node sees A before B. But two concurrent events may be seen in different orders by different nodes.

Example: Social Media Comment Thread

Alice posts a photo. Bob comments on the photo. Carol replies to Bob's comment. Causal consistency ensures every user sees: Photo → Bob's comment → Carol's reply, in that order. But if Dave and Eve both "like" the photo independently (concurrent events), some users might see Dave's like first and others might see Eve's first.

Why It's Attractive

  • Does not require a single leader — can be implemented in multi-leader or leaderless systems.
  • Available during partitions (unlike linearizability).
  • Captures the ordering guarantees that humans intuitively expect.

Systems

MongoDB (causal sessions since 3.6), COPS (research system), Eiger. Many collaborative editing systems rely on causal ordering.

Eventual Consistency

Eventual consistency provides the weakest useful guarantee: if no new writes occur, all replicas will eventually converge to the same value. The "eventually" might be milliseconds or minutes — there is no upper bound unless the system specifies one.

The Convergence Contract

Eventual consistency alone says nothing about what value replicas converge to when there are conflicting writes. Systems must layer a conflict resolution strategy on top:

  • Last-writer-wins (LWW): Highest timestamp wins. Simple but can lose updates.
  • Application-level merge: The application resolves conflicts (Amazon's shopping cart).
  • CRDTs: Data structures that merge automatically without conflicts (covered in section 5).

When It's Appropriate

  • Caches (stale data eventually refreshed)
  • DNS (TTL-based convergence)
  • Social media feeds, like counts, view counts
  • Product catalog (brief staleness rarely matters)
  • Session stores (user is typically sticky to one replica)

Systems

Amazon DynamoDB (default), Apache Cassandra (default), Riak, CouchDB, Amazon S3 (now strong consistency as of Dec 2020, but historically eventual).

Staff-Level Insight

"Eventual consistency" without specifying a conflict resolution strategy is incomplete. Always state how conflicts are resolved. In an interview, saying "we'll use eventual consistency with LWW based on Lamport timestamps" shows depth, whereas "we'll use eventual consistency" sounds hand-wavy.

Session Guarantees (Read-Your-Writes, Monotonic Reads, Monotonic Writes)

Between causal and eventual, there are per-session guarantees that are often sufficient for good user experience without the cost of full causal consistency:

Read-Your-Writes (RYW)

After a client writes a value, subsequent reads by that same client will see the written value or something newer. Other clients may still see stale data.

Example: You update your profile picture. RYW guarantees you see the new picture immediately, even if other users see the old one for a few seconds.

Implementation: Track the client's latest write timestamp. On read, ensure the replica has caught up to at least that timestamp (or route reads to the same replica that handled the write).

Monotonic Reads

If a client reads value v at time t, it will never subsequently read a value older than v. Time doesn't "go backwards." Without this guarantee, a user could refresh a page and see an older version of data — a confusing experience.

Implementation: Pin the client to a replica, or track the client's last-read timestamp and ensure subsequent reads come from a replica that is at least as up-to-date.

Monotonic Writes

Writes by the same client are applied in the order they were issued. Without this, a sequence of incremental updates could be applied out of order, producing an inconsistent state.

Guarantee What It Prevents Cost

3. Consensus Protocols

Why Consensus Is Hard

Consensus is the problem of getting N nodes to agree on a single value, despite crash failures and network delays. It is the backbone of linearizable replicated state machines, leader election, and distributed locks.

The FLP impossibility result (1985) proves that no deterministic algorithm can guarantee consensus in an asynchronous system with even one crash failure. In practice, we circumvent this by using timeouts, randomization, or partial synchrony assumptions.

Cost of Consensus

  • Latency: At minimum, 1 round trip to a majority (Raft/Paxos). In Spanner, cross-continent consensus adds 50-150ms.
  • Availability: Requires a majority of nodes to be reachable. 3 nodes tolerates 1 failure; 5 nodes tolerates 2.
  • Throughput: Single-leader consensus serializes writes through one node, limiting throughput.
Paxos (Simplified)

Paxos, invented by Leslie Lamport, is the foundational consensus algorithm. It is notoriously difficult to understand and implement, which is why Raft was created as an equivalent but more understandable alternative.

Basic Paxos (Single-Decree)

Three roles: Proposers, Acceptors, Learners.

  1. Prepare phase: Proposer picks a unique proposal number n and sends Prepare(n) to a majority of acceptors.
  2. Promise: Each acceptor, if n is the highest it has seen, responds with Promise(n) and any value it has already accepted.
  3. Accept phase: If the proposer gets promises from a majority, it sends Accept(n, v) where v is either the highest-numbered previously accepted value, or the proposer's own value.
  4. Accepted: Acceptors accept if they haven't promised a higher number. Once a majority accepts, the value is chosen.

Multi-Paxos

Single-decree Paxos decides one value. To build a replicated log, you run Paxos for each log entry — this is Multi-Paxos. In practice, you elect a stable leader to skip the Prepare phase for subsequent entries, reducing it to a single round trip. This is essentially what Google's Chubby and Spanner use internally.

Interview Tip

You rarely need to explain Paxos in detail during an interview. It's more important to know what it achieves (consensus despite failures), its limitations (liveness issues with competing proposers), and that Raft solves the same problem more understandably. If asked, saying "I'm more familiar with Raft, which is equivalent to Multi-Paxos" is perfectly fine.

Raft: Understandable Consensus

Raft (2014, Ongaro & Ousterhout) decomposes consensus into three sub-problems: leader election, log replication, and safety. It is used by etcd, CockroachDB, TiKV, Consul, and many other production systems.

Raft Roles

  • Leader: Handles all client requests. Replicates log entries to followers.
  • Follower: Passively replicates the leader's log. Redirects client requests to the leader.
  • Candidate: A follower that has timed out waiting for heartbeats and starts an election.

Leader Election

stateDiagram-v2 [*] --> Follower Follower --> Candidate: Election timeout\n(no heartbeat received) Candidate --> Candidate: Split vote\n(restart election) Candidate --> Leader: Receives majority votes Candidate --> Follower: Discovers higher term Leader --> Follower: Discovers higher term Leader --> Leader: Sends heartbeats\nReplicates log entries

Election process step by step:

  1. A follower's election timer expires (randomized 150-300ms to prevent split votes).
  2. It increments its currentTerm, transitions to Candidate, votes for itself, and sends RequestVote RPCs to all other nodes.
  3. A node grants its vote if: (a) it hasn't voted in this term yet, and (b) the candidate's log is at least as up-to-date as its own.
  4. If the candidate receives votes from a majority, it becomes Leader and immediately sends heartbeats to suppress new elections.
  5. If two candidates split the vote, both time out with new random delays and try again.

Log Replication

  1. Client sends a command to the leader.
  2. Leader appends the command to its log as a new entry with the current term.
  3. Leader sends AppendEntries RPCs to all followers in parallel.
  4. Once a majority of followers acknowledge, the entry is committed.
  5. Leader applies the command to its state machine and responds to the client.
  6. Followers learn about commits via the leader's next heartbeat and apply committed entries.

Safety Guarantee

Raft's key safety property: if a log entry is committed in a given term, that entry will be present in the logs of all leaders for all higher terms. This is enforced by the election restriction — a candidate cannot win an election unless its log contains all committed entries.

Raft in Code (Simplified State)

# Simplified Raft node state
class RaftNode:
    def __init__(self, node_id, peers):
        self.node_id = node_id
        self.peers = peers
        self.state = "follower"      # follower | candidate | leader
        self.current_term = 0
        self.voted_for = None
        self.log = []                # List of (term, command)
        self.commit_index = 0
        self.last_applied = 0

    def start_election(self):
        self.current_term += 1
        self.state = "candidate"
        self.voted_for = self.node_id
        votes_received = 1  # vote for self

        for peer in self.peers:
            # Send RequestVote RPC
            vote_granted = self.send_request_vote(
                peer,
                term=self.current_term,
                last_log_index=len(self.log) - 1,
                last_log_term=self.log[-1][0] if self.log else 0
            )
            if vote_granted:
                votes_received += 1

        if votes_received > (len(self.peers) + 1) // 2:
            self.become_leader()

    def on_append_entries(self, leader_term, entries, leader_commit):
        if leader_term < self.current_term:
            return False  # Reject stale leader
        self.state = "follower"
        self.current_term = leader_term
        # Append new entries, update commit_index
        self.log.extend(entries)
        self.commit_index = min(leader_commit, len(self.log) - 1)
        return True

Staff-Level Insight

In interviews, knowing how Raft handles leader failure is a strong signal: the new leader might have uncommitted entries from the old leader. Raft's "election restriction" ensures the new leader has all committed entries, and uncommitted entries from old terms are never directly committed — they are committed only as a side effect of committing a new entry in the current term.

4. Vector Clocks & Logical Time

Lamport Timestamps

Physical clocks in distributed systems are unreliable — clock skew, NTP drift, and leap seconds all introduce ordering errors. Lamport timestamps (1978) provide a logical clock that captures the happens-before relationship.

Rules

  1. Each process maintains a counter C.
  2. Before each event, increment C: C = C + 1
  3. When sending a message, attach the current C value.
  4. On receiving a message with timestamp T: C = max(C, T) + 1

Property

If event A happens-before event B, then L(A) < L(B). But the converse is not true: L(A) < L(B) does NOT imply A happened before B. Two concurrent events may have any ordering of Lamport timestamps.

Limitation

Lamport timestamps cannot detect concurrency. If L(A) < L(B), you cannot tell whether A → B or they are concurrent. This is why we need vector clocks.

Vector Clocks: Detecting Concurrency

A vector clock is an array of N counters, one per node. It captures the full happens-before relationship and can determine whether two events are causally related or concurrent.

Rules

  1. Each node i maintains a vector V of size N, initialized to all zeros.
  2. Before each event on node i: V[i] = V[i] + 1
  3. When node i sends a message, attach V.
  4. On receiving a message with vector W on node j: V[k] = max(V[k], W[k]) for all k, then V[j] = V[j] + 1

Comparison Rules

  • V1 = V2 iff V1[k] = V2[k] for all k
  • V1 ≤ V2 iff V1[k] ≤ V2[k] for all k
  • V1 < V2 (V1 happens-before V2) iff V1 ≤ V2 and V1 ≠ V2
  • Concurrent: if neither V1 < V2 nor V2 < V1

Amazon Dynamo and Vector Clocks

Amazon Dynamo (the internal system, not DynamoDB) uses vector clocks to detect conflicting writes. When a client reads a key, it may receive multiple versions with incomparable vector clocks. The client or application must merge these siblings.

In practice, Dynamo found that vector clocks grew too large (proportional to number of client nodes), so they truncated old entries — at the cost of sometimes failing to detect causality and producing unnecessary conflicts. DynamoDB (the AWS service) moved away from vector clocks to simpler LWW for most use cases.

Interview Application

If designing a leaderless replicated system, mention vector clocks as the mechanism for conflict detection. A strong answer explains: "We detect concurrent writes using vector clocks, and resolve conflicts via [LWW / application merge / CRDTs] depending on the data semantics."

5. CRDTs (Conflict-free Replicated Data Types)

What Are CRDTs?

CRDTs are data structures designed so that concurrent updates on different replicas can always be merged without conflicts. They guarantee strong eventual consistency: any two replicas that have received the same set of updates (in any order) have the same state.

This is achieved by making the merge operation commutative, associative, and idempotent, forming a mathematical join-semilattice.

Two Flavors

  • State-based (CvRDTs): Replicas ship their full state; merge via a join function. Simpler but more bandwidth.
  • Operation-based (CmRDTs): Replicas ship operations; must be delivered exactly once. Less bandwidth but requires reliable delivery.
Common CRDT Types

G-Counter (Grow-only Counter)

Each of N nodes maintains its own counter. The total value is the sum of all node counters. Merge is element-wise max. 

class GCounter:
    """Grow-only counter CRDT"""
    def __init__(self, node_id, n_nodes):
        self.node_id = node_id
        self.counts = [0] * n_nodes  # One slot per node

    def increment(self):
        self.counts[self.node_id] += 1

    def value(self):
        return sum(self.counts)

    def merge(self, other):
        """Merge is element-wise max - commutative, associative, idempotent"""
        for i in range(len(self.counts)):
            self.counts[i] = max(self.counts[i], other.counts[i])

# Example: 3-node cluster
a = GCounter(0, 3)  # [0, 0, 0]
b = GCounter(1, 3)  # [0, 0, 0]
a.increment()        # a: [1, 0, 0]
a.increment()        # a: [2, 0, 0]
b.increment()        # b: [0, 1, 0]
# Merge: a.merge(b) => [2, 1, 0], value = 3  ✓
# Merge: b.merge(a) => [2, 1, 0], value = 3  ✓ (commutative)

PN-Counter (Positive-Negative Counter)

Two G-Counters: one for increments (P), one for decrements (N). Value = P.value() - N.value(). Supports both increment and decrement. 

class PNCounter:
    def __init__(self, node_id, n_nodes):
        self.p = GCounter(node_id, n_nodes)  # positive
        self.n = GCounter(node_id, n_nodes)  # negative

    def increment(self):
        self.p.increment()

    def decrement(self):
        self.n.increment()

    def value(self):
        return self.p.value() - self.n.value()

    def merge(self, other):
        self.p.merge(other.p)
        self.n.merge(other.n)

G-Set (Grow-only Set)

Elements can only be added, never removed. Merge is set union. Simple and useful for append-only collections (e.g., set of users who liked a post).

OR-Set (Observed-Remove Set)

Allows both add and remove operations. Each add creates a unique tag. Remove deletes all currently observed tags for that element. Concurrent add+remove: the add wins (add-wins semantics). 

import uuid

class ORSet:
    """Observed-Remove Set CRDT (add-wins semantics)"""
    def __init__(self):
        self.elements = {}  # element -> set of unique tags

    def add(self, element):
        tag = str(uuid.uuid4())
        if element not in self.elements:
            self.elements[element] = set()
        self.elements[element].add(tag)

    def remove(self, element):
        """Remove all currently observed tags for this element"""
        if element in self.elements:
            self.elements[element] = set()

    def lookup(self, element):
        return element in self.elements and len(self.elements[element]) > 0

    def value(self):
        return {e for e, tags in self.elements.items() if len(tags) > 0}

    def merge(self, other):
        all_elements = set(self.elements.keys()) | set(other.elements.keys())
        for e in all_elements:
            mine = self.elements.get(e, set())
            theirs = other.elements.get(e, set())
            self.elements[e] = mine | theirs  # Union of tags

LWW-Register (Last-Writer-Wins Register)

Each write carries a timestamp. Merge picks the value with the highest timestamp. Simple but can silently lose concurrent writes. 

class LWWRegister:
    def __init__(self):
        self.value = None
        self.timestamp = 0

    def write(self, value, timestamp):
        if timestamp > self.timestamp:
            self.value = value
            self.timestamp = timestamp

    def merge(self, other):
        if other.timestamp > self.timestamp:
            self.value = other.value
            self.timestamp = other.timestamp

  • Redis (CRDTs): Redis Enterprise supports CRDT-based active-active geo-replication for counters, sets, and strings.
  • Figma: Uses CRDTs for real-time collaborative design editing.
  • Apple Notes / iCloud: Uses CRDTs for offline-first sync across devices.
  • Automerge / Yjs: Open-source CRDT libraries for collaborative applications.

Staff-Level Signal

CRDTs are a powerful tool to mention when the interviewer pushes on availability during partitions. "For the shopping cart, we could use an OR-Set CRDT so that items can be added and removed on any replica, even during partitions, and all replicas converge automatically without conflict resolution logic." This shows you know a concrete mechanism, not just "eventual consistency."

6. Quorum Systems

Read/Write Quorums

In a system with N replicas, a quorum system defines how many replicas must participate in reads (R) and writes (W) to guarantee consistency:

R + W > N ⇒ Reads always see the latest write

This works because any read quorum and write quorum must overlap by at least one node, and that overlapping node has the latest value.

Common Configurations (N=3)

Config W R Behavior
Strong consistency 2 2 Both reads and writes go to majority. Balanced latency.
Fast reads 3 1 Write to all replicas. Read from any one. Read-heavy workloads.
Fast writes 1 3 Write to one. Read from all. Write-heavy workloads.
Eventual (no quorum) 1 1 R + W = 2 ≤ N. No consistency guarantee. Maximum availability.

Read Repair

When a quorum read discovers that some replicas have stale data, it can perform read repair: send the latest value back to the stale replicas. This is an opportunistic anti-entropy mechanism used by Cassandra and Riak.

Sloppy Quorums and Hinted Handoff

A strict quorum requires responses from W (or R) of the N designated replicas for a given key. But what if some designated replicas are unreachable?

A sloppy quorum allows writes to go to any W reachable nodes, even if they are not the designated replicas. This preserves write availability during partitions.

Hinted Handoff

When a non-designated node accepts a write on behalf of a down replica, it stores a hint. Once the designated replica comes back, the hint is forwarded and the data is transferred to its correct home. This is used by DynamoDB and Cassandra.

Critical Trade-off

Sloppy quorums do not guarantee R + W > N consistency, because reads and writes might go to non-overlapping sets of nodes. They trade consistency for availability. Dynamo's original paper (2007) calls this out explicitly: sloppy quorums ensure durability, but reads may return stale data until hints are replayed.

Tunable Consistency: Cassandra Example

Apache Cassandra lets you specify consistency level per query: 

-- Write with quorum consistency
INSERT INTO users (id, name) VALUES (42, 'Alice')
USING CONSISTENCY QUORUM;

-- Read with ONE (fast, possibly stale)
SELECT * FROM users WHERE id = 42
USING CONSISTENCY ONE;

-- Read with ALL (slowest, strongest)
SELECT * FROM users WHERE id = 42
USING CONSISTENCY ALL;

-- Read with LOCAL_QUORUM (quorum within local datacenter)
SELECT * FROM users WHERE id = 42
USING CONSISTENCY LOCAL_QUORUM;
Level Meaning Use Case
ONE 1 replica responds Time-series, logs, analytics (stale OK)
QUORUM ⌊N/2⌋ + 1 replicas respond Most read/write paths
LOCAL_QUORUM Quorum in local DC only Multi-DC with low-latency reads
ALL All N replicas respond Critical reads (rare, low availability)
SERIAL Linearizable (Paxos-based LWT) Compare-and-set, unique constraints

Interview Tip

When discussing Cassandra in an interview, don't just say "eventual consistency." Say "Cassandra supports tunable consistency — we'd use QUORUM for writes and reads on our critical path (W + R = N+1 > N, giving us strong consistency), but ONE for analytics reads where staleness is acceptable." This shows you understand the knobs.

7. Gossip Protocols

How Gossip Works

Gossip protocols (also called epidemic protocols) spread information through a cluster the same way a rumor spreads in a social network: each node periodically picks a random peer and exchanges state. Information propagates exponentially, reaching all N nodes in O(log N) rounds.

Basic Gossip Algorithm

import random, time

class GossipNode:
    def __init__(self, node_id, peers, state):
        self.node_id = node_id
        self.peers = peers
        self.state = state       # Key-value state with version info
        self.heartbeat = 0

    def gossip_round(self):
        """Execute one gossip round - called periodically (e.g., every 1 second)"""
        self.heartbeat += 1
        self.state[self.node_id] = {
            'heartbeat': self.heartbeat,
            'timestamp': time.time(),
            'status': 'alive'
        }
        # Pick a random peer
        target = random.choice(self.peers)
        # Exchange state (push-pull)
        remote_state = target.receive_gossip(self.state)
        self.merge_state(remote_state)

    def receive_gossip(self, incoming_state):
        """Receive gossip from a peer, merge, and return our state"""
        self.merge_state(incoming_state)
        return self.state

    def merge_state(self, remote_state):
        """Keep the entry with the higher heartbeat for each node"""
        for node_id, info in remote_state.items():
            if node_id not in self.state:
                self.state[node_id] = info
            elif info['heartbeat'] > self.state[node_id]['heartbeat']:
                self.state[node_id] = info

Gossip for Failure Detection

Each node tracks the last heartbeat it received (via gossip) for every other node. If a node's heartbeat hasn't increased for a threshold period (e.g., 10 seconds), it is suspected to be down. After a longer period, it is marked as permanently failed.

This is how Cassandra detects node failures. The Phi accrual failure detector uses the distribution of heartbeat arrival times to compute a suspicion level rather than a hard threshold.

In each round, every node gossips with one random peer. After O(log N) rounds, information from any single node has reached all others with high probability.

SWIM Protocol

SWIM (Scalable Weakly-consistent Infection-style process group Membership) is a membership protocol that separates failure detection from dissemination:

  1. Failure detection: Each round, node i picks a random target j and sends a ping. If j doesn't respond, i asks k random other nodes to ping j on its behalf (indirect ping). If still no response, j is marked suspect.
  2. Dissemination: Membership changes (join, leave, suspect, confirm) are piggybacked on ping/ack messages, propagating via gossip.

SWIM detects failures in O(1) expected time per node and disseminates membership changes in O(log N) rounds. It is used by HashiCorp's Serf and Consul (via the memberlist library).

Anti-Entropy and Merkle Trees

Anti-entropy is the process of comparing replicas and reconciling differences. While gossip and read-repair catch many inconsistencies, a background anti-entropy process provides a safety net for data that is rarely read.

Merkle Trees for Efficient Comparison

Comparing all data between two replicas is expensive. Merkle trees (hash trees) make this efficient:

  1. Each replica maintains a Merkle tree over its data, where leaf nodes are hashes of individual key-value pairs.
  2. Parent nodes are hashes of their children.
  3. Two replicas compare their root hashes. If equal, all data matches.
  4. If different, they recurse down the tree, comparing child hashes, until they find the specific keys that differ.
  5. Only the differing keys are transferred — O(log N) comparisons to find differences in N keys.

Used by: Cassandra (for anti-entropy repair), DynamoDB, Riak. Amazon Dynamo's paper specifically describes Merkle tree-based anti-entropy.

Interview Mention

When designing any system with eventual consistency, mention anti-entropy as a background process: "For data that's rarely read and wouldn't be caught by read-repair, we run periodic Merkle tree comparisons between replicas to detect and fix drift."

8. Consistency in Practice

Google Spanner: TrueTime and External Consistency

Google Spanner is a globally distributed database that provides external consistency (equivalent to linearizability) using a unique approach: TrueTime, a globally synchronized clock with bounded uncertainty.

How TrueTime Works

Every Google datacenter has GPS receivers and atomic clocks. The TrueTime API returns an interval [earliest, latest] rather than a single timestamp, representing the uncertainty of the current time. Typical uncertainty is 1-7 milliseconds. 

// TrueTime API (simplified)
type TrueTime struct {
    Earliest time.Time
    Latest   time.Time
}

func Now() TrueTime {
    // Returns interval [earliest, latest] with bounded uncertainty
    // Typical epsilon: 1-7ms
}

// Spanner's commit protocol:
// 1. Acquire locks on all participants
// 2. Pick commit timestamp s = TT.Now().Latest
// 3. WAIT until TT.Now().Earliest > s (called "commit-wait")
// 4. Release locks and apply commit
//
// The commit-wait ensures that the commit timestamp is
// definitely in the past by the time any reader could see it,
// guaranteeing external consistency without consensus on every read.

Why Commit-Wait?

The commit-wait delay (typically 7-10ms worst case) ensures that if transaction T1 commits before T2 begins, then T1's timestamp is strictly less than T2's. This provides external consistency: the order of transactions as observed by the real world matches their timestamp order.

Key Insight

Spanner trades latency (commit-wait delay) for consistency. The tighter the clock synchronization, the shorter the wait. Google invested in physical infrastructure (GPS + atomic clocks) to make this delay small.

CockroachDB: Serializable Without Atomic Clocks

CockroachDB aims to provide similar guarantees to Spanner but without specialized hardware. It uses hybrid logical clocks (HLC) and an uncertainty interval approach:

  • Each node maintains an HLC that combines a physical timestamp with a logical counter.
  • Reads specify a timestamp. If a read encounters a value written within its uncertainty window, it retries at a higher timestamp (called a read restart).
  • The uncertainty interval is based on maximum clock offset (default 250ms for NTP-synced nodes, configurable).
  • This is more conservative than Spanner (larger uncertainty = more restarts) but works without GPS/atomic clocks.

CockroachDB uses Raft for replication within each range (partition), providing linearizable writes. Combined with HLC-based MVCC, it achieves serializable isolation.

DynamoDB: Tunable Consistency

Amazon DynamoDB offers two read consistency options:

  • Eventually consistent reads (default): May return stale data. Reads from any replica. Lower latency, higher throughput, 50% cheaper.
  • Strongly consistent reads: Returns the most recent data. Routes to the leader replica. Higher latency, may fail during leader failover.

For transactions, DynamoDB supports TransactWriteItems and TransactGetItems that provide serializable isolation across up to 100 items across tables. Internally, this uses a Paxos-like protocol. 

# DynamoDB: eventually consistent read (default)
response = table.get_item(Key={'id': '42'})

# DynamoDB: strongly consistent read
response = table.get_item(
    Key={'id': '42'},
    ConsistentRead=True
)

# DynamoDB: transactional write (serializable)
client.transact_write_items(
    TransactItems=[
        {
            'Update': {
                'TableName': 'Accounts',

                'UpdateExpression': 'SET balance = balance - :amt',
                'ExpressionAttributeValues': {':amt': {'N': '100'}},
                'ConditionExpression': 'balance >= :amt'
            }
        },
        {
            'Update': {
                'TableName': 'Accounts',
                'Key': {'id': {'S': 'bob'}},
                'UpdateExpression': 'SET balance = balance + :amt',

Decision Framework: How to Choose

Use this framework when an interviewer asks you about consistency for your design:

Step 1: Classify the Data Path

  • User-facing, writes money/inventory: Strong consistency (linearizable or serializable).
  • User-facing, reads own writes: Read-your-writes (session consistency).
  • User-facing, tolerates brief staleness: Eventual consistency with bounded staleness.
  • Internal, analytics/logging: Eventual consistency.

Step 2: Choose Mechanism

Requirement Mechanism Example Technology
Linearizable writes Consensus (Raft/Paxos) etcd, ZooKeeper, Spanner
Serializable transactions 2PC + consensus Spanner, CockroachDB
Read-your-writes Sticky sessions or version tracking DynamoDB (consistent read), any DB with session routing
High availability + convergence CRDTs or LWW Riak, Redis Enterprise
Tunable per-query Quorum with configurable R, W Cassandra, DynamoDB

Step 3: Articulate the Trade-off

Always state what you're giving up:

  • "We choose strong consistency for the payment path, accepting higher tail latency (~50ms for consensus) and reduced availability during partitions."
  • "The activity feed uses eventual consistency — users may see posts a few seconds late, but we get sub-10ms reads from any replica and full availability."
  • "For the shopping cart, we use CRDTs (OR-Set) so the cart is always writable, even during partitions. The trade-off is that we can't enforce max-item limits without coordination."

Trade-offs to Always Discuss

Latency vs. Consistency

Consensus adds at least one round trip to a majority. Cross-DC consensus adds 50-150ms. Strong consistency = higher p99 latency.

Availability vs. Consistency

During partitions, a CP system rejects writes. An AP system accepts writes but may have conflicts. State which your system chooses per data path.

Complexity vs. Consistency

Weak consistency pushes complexity to the application (conflict resolution, idempotency, compensation). Strong consistency simplifies app logic but constrains infrastructure.

Red Flags Interviewers Watch For

  • Saying "eventual consistency" without specifying conflict resolution.
  • Claiming "strong consistency" without explaining the cost or mechanism.
  • Treating all data paths as needing the same consistency level.
  • Not mentioning what happens during failures or partitions.
  • Conflating database isolation levels (serializable) with distributed consistency (linearizable).

10. Self-Check Quiz

1. What is the key difference between linearizability and sequential consistency?

Linearizability respects real-time ordering: if operation A completes before operation B begins, then A appears before B in the total order. Sequential consistency only requires that a total order exists consistent with each client's program order, but this order may not match real-time. A sequentially consistent system might re-order operations from different clients in ways that don't match wall-clock time.

2. In a system with N=5 replicas, what values of R and W guarantee that every read sees the latest write? Give two different valid configurations.

Any R and W where R + W > 5. Two examples:
  • R=3, W=3 (balanced, both require majority)
  • R=1, W=5 (write to all, read from any one — fast reads, slow writes)
Other valid: R=2, W=4 or R=4, W=2.

3. Two nodes have vector clocks V1 = [3, 1, 2] and V2 = [2, 3, 1]. Are these events causally related or concurrent? Why?

They are concurrent. V1[0]=3 > V2[0]=2 but V1[1]=1 < V2[1]=3. Neither V1 ≤ V2 nor V2 ≤ V1, so neither happened before the other. A system using vector clocks would detect this as a conflict requiring resolution (merge, LWW, or CRDT).

4. Why does Raft use randomized election timeouts?

To avoid split votes. If all followers had the same timeout, they would all start elections simultaneously and split the vote (each voting for itself), preventing any candidate from getting a majority. Randomized timeouts (e.g., 150-300ms) ensure that usually one node times out first and wins the election before others start.

5. A G-Counter on 3 nodes has state Node A=[5,0,0], Node B=[3,2,0], Node C=[3,0,4]. What is the counter value after all nodes merge?

Merge is element-wise max: max([5,3,3], [0,2,0], [0,0,4]) = [5, 2, 4]. The counter value is 5 + 2 + 4 = 11.

6. What is the fundamental trade-off of sloppy quorums compared to strict quorums?

Sloppy quorums improve write availability during partitions (writes can go to any reachable node, not just designated replicas), but they break the R + W > N consistency guarantee because reads and writes may hit non-overlapping sets of nodes. Hinted handoff eventually delivers data to the correct replicas, but until then, strongly consistent reads are not guaranteed.

7. How does Google Spanner achieve external consistency without requiring consensus on every read?

Spanner uses TrueTime, a globally synchronized clock with bounded uncertainty. Write transactions perform a commit-wait: they pick a commit timestamp and wait until TrueTime guarantees that timestamp is in the past. This ensures that any transaction starting after the commit will see it. Reads at a snapshot timestamp can be served from any replica that has caught up to that timestamp, without an additional consensus round.

8. You're designing a collaborative text editor that must work offline. Would you use consensus (Raft) or CRDTs? Why?

CRDTs (or a related approach like OT). Consensus requires a majority of nodes to be reachable, which is incompatible with offline operation. CRDTs allow each device to make edits independently and merge when connectivity is restored, guaranteeing convergence without coordination. Libraries like Automerge and Yjs implement text CRDTs for exactly this use case. The trade-off is that some editing semantics (like "undo") are harder to implement with CRDTs.

Key Takeaways

Summary

  1. Consistency is a spectrum — match the level to each data path's requirements, not one-size-fits-all.
  2. Linearizability is expensive — use it only for data that requires it (locks, money, uniqueness).
  3. Consensus (Raft/Paxos) is the foundation for strong consistency, but it limits availability and throughput.
  4. Vector clocks detect concurrency; CRDTs resolve it automatically.
  5. Quorums are tunable — R + W > N for consistency, lower for availability.
  6. Gossip + Merkle trees provide background anti-entropy for eventually consistent systems.
  7. Always articulate the trade-off: what consistency buys you and what it costs.
