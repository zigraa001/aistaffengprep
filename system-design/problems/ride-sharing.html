1. Problem Statement

Design a ride-sharing platform like Uber or Lyft that connects riders who need transportation with nearby drivers. The system must handle real-time location tracking, efficient driver-rider matching, dynamic pricing, trip management, and payments — all at massive scale across hundreds of cities worldwide.

Key challenges include: ingesting millions of location updates per second, performing sub-second geospatial queries to find nearby drivers, running a fair and efficient matching algorithm, and maintaining a consistent trip state machine across distributed services.

Functional Requirements ▼
  • Request a ride: Rider specifies pickup location, destination, and ride type (economy, premium, pool)

  • Dynamic pricing: Surge pricing based on supply-demand conditions in a geographic area

Geohash vs. H3 vs. Quadtree

Approach How It Works Pros Cons
Geohash Encodes lat/lng into a base-32 string; shared prefix = proximity Simple, works with any DB, prefix queries Edge effects at cell boundaries; rectangular cells distort near poles
H3 (Uber) Hexagonal hierarchical grid; each hex ~0.7 km² at resolution 9 Uniform cell sizes, no edge effects, built for ride-sharing More complex, requires library support
Quadtree Recursive spatial subdivision; denser areas get finer splits Adaptive density, efficient for uneven distribution Not natively supported by DBs, harder to distribute
Staff-Level Signal: Uber uses H3 in production because hexagonal cells have uniform distances to neighbors (unlike rectangular Geohash cells). This matters for "find drivers within 3 km" queries — with Geohash, you must search 9 cells (the target + 8 neighbors) to handle boundary effects. With H3, you search 7 cells (the target + 6 neighbors), and each neighbor is equidistant. At 1.25M updates/second, this 22% reduction in cell lookups is significant.

import h3

def find_nearby_drivers(lat, lng, radius_km=3, resolution=9):
    """Find drivers in nearby H3 hexagons."""
    center_hex = h3.latlng_to_cell(lat, lng, resolution)
    # k=1 ring gives 7 hexagons (center + 6 neighbors)
    # k=2 ring gives 19 hexagons for larger radius
    k_ring = 1 if radius_km <= 3 else 2
    hex_ring = h3.grid_disk(center_hex, k_ring)

    nearby_drivers = []
    for hex_id in hex_ring:
        # Each hex maps to a Redis set of driver IDs
        drivers = redis.smembers(f"hex:{hex_id}:drivers")
        for driver_id in drivers:
            meta = redis.hgetall(f"driver:{driver_id}:meta")
            if meta['status'] == 'available':
                nearby_drivers.append({
                    'driver_id': driver_id,
                    'lat': float(meta['lat']),
                    'lng': float(meta['lng']),
                    'vehicle_type': meta['vehicle_type'],
                    'rating': float(meta['rating'])
                })

    # Sort by distance and return top candidates
    nearby_drivers.sort(key=lambda d: haversine(lat, lng, d['lat'], d['lng']))
    return nearby_drivers[:20]

6b. Matching Algorithm ▼

Matching is the most business-critical algorithm. A naive approach (assign closest driver) leaves money and efficiency on the table. Staff-level candidates should discuss batch matching vs. greedy matching.

Naive vs. Batch Matching

Greedy (First-Come-First-Served)

Each ride request immediately grabs the closest available driver. Simple, but suboptimal:

  • Driver A is 2 min from Rider 1 and 3 min from Rider 2
  • Driver B is 10 min from Rider 1 and 1 min from Rider 2
  • Greedy assigns A→1, B→2 (total: 2+1=3 min if lucky, but if Rider 1 arrives first, it's A→1 and B→2 = 2+1=3 min; if Rider 2 first, it's A→2 and B→1 = 3+10=13 min)

Batch Matching (Uber's Approach)

Accumulate ride requests for a short window (2-5 seconds), then solve a global optimization:

  • Formulate as a bipartite matching problem
  • Minimize total wait time (or maximize total driver utilization)
  • Use the Hungarian algorithm or auction-based algorithms
  • Uber reported a 20% reduction in wait times after switching to batch matching

Driver Scoring Function

def score_driver(driver, ride_request):
    """
    Compute a match score for a driver-rider pair.
    Lower score = better match.
    """
    # Distance-based ETA (primary factor)
    eta_minutes = compute_eta(
        driver.lat, driver.lng,
        ride_request.pickup_lat, ride_request.pickup_lng
    )

    # Filter: skip if ETA > max threshold (e.g., 15 min)
    if eta_minutes > MAX_PICKUP_ETA:
        return float('inf')

    # Vehicle type match
    if ride_request.ride_type == 'premium' and driver.vehicle_type != 'premium':
        return float('inf')

    # Composite score (weights tuned via ML)
    score = (
        0.6 * eta_minutes +                    # Proximity
        0.2 * (5.0 - driver.rating) * 3 +     # Driver rating (lower = better)
        0.1 * driver.acceptance_rate_penalty +  # Penalize low acceptance
        0.1 * driver.idle_time_bonus            # Prioritize idle drivers
    )

    return score

Matching Flow

async def match_ride(ride_request):
    """Full matching pipeline."""
    # 1. Find candidate drivers
    candidates = location_service.find_nearby_drivers(
        ride_request.pickup_lat,
        ride_request.pickup_lng,
        radius_km=5
    )

    if not candidates:
        # Expand radius and retry
        candidates = location_service.find_nearby_drivers(
            ride_request.pickup_lat,
            ride_request.pickup_lng,
            radius_km=10
        )

    if not candidates:
        return MatchResult(status='NO_DRIVERS_AVAILABLE')

    # 2. Score and rank
    scored = [(score_driver(d, ride_request), d) for d in candidates]
    scored.sort(key=lambda x: x[0])

    # 3. Offer to best driver (with timeout)
    for score, driver in scored[:5]:  # Try top 5
        offer = await send_driver_offer(driver.driver_id, ride_request, timeout=15)
        if offer.accepted:
            # 4. Atomically mark driver as busy
            success = await claim_driver(driver.driver_id, ride_request.trip_id)
            if success:
                return MatchResult(status='MATCHED', driver=driver)
            # Another request claimed this driver; try next

    return MatchResult(status='NO_DRIVER_ACCEPTED')
Staff-Level Signal: Discuss the claim_driver atomicity problem. Two ride requests might try to claim the same driver simultaneously. Use Redis SETNX or a database compare-and-swap: UPDATE drivers SET status='busy', trip_id=? WHERE driver_id=? AND status='available'. Check affected rows = 1 to confirm the claim. This is a classic distributed locking problem.

WebSocket Connection Management

At peak, we have 8-10 million concurrent WebSocket connections. Key design considerations:

  • Connection affinity: Use consistent hashing to route a user's connection to the same gateway server
  • Heartbeats: Ping/pong every 30 seconds to detect dead connections
  • Reconnection: Client-side exponential backoff with jitter; server replays last known state on reconnect
  • Fan-out: A driver's location update must reach only the specific rider tracking that trip (1:1 fan-out, not broadcast)

6d. ETA Calculation ▼

ETA accuracy directly impacts user trust and pricing. Uber has published extensively on their approach, which combines graph algorithms with machine learning.

ETA Layers

  1. Road Graph (Dijkstra / A*): Base routing on the road network graph (OpenStreetMap or proprietary). Gives theoretical shortest-time path.
  2. Historical Traffic: Overlay average speed data per road segment per time-of-day and day-of-week. "This highway averages 25 mph on Friday at 5 PM."
  3. Real-time Traffic: Use live GPS data from active drivers to compute current segment speeds. If 50 drivers are crawling on I-405 at 10 mph, update that edge weight.
  4. ML Post-Processing: A gradient-boosted model adjusts the graph-based ETA using features like weather, events, pickup complexity (airport vs. street), and historical trip durations for similar routes.
class ETAService:
    def compute_eta(self, origin_lat, origin_lng, dest_lat, dest_lng):
        """Multi-layer ETA computation."""
        # Layer 1: Graph-based routing
        route = self.routing_engine.route(
            origin=(origin_lat, origin_lng),
            destination=(dest_lat, dest_lng),
            algorithm='a_star'
        )
        base_eta = route.duration_seconds

        # Layer 2: Historical traffic adjustment
        hour = datetime.now().hour
        day_of_week = datetime.now().weekday()
        historical_factor = self.traffic_db.get_speed_factor(
            route.segments, hour, day_of_week
        )
        adjusted_eta = base_eta * historical_factor

        # Layer 3: Real-time traffic from driver GPS
        realtime_factor = self.realtime_traffic.get_speed_factor(
            route.segments
        )
        adjusted_eta *= realtime_factor

        # Layer 4: ML correction
        features = {
            'base_eta': adjusted_eta,
            'distance_km': route.distance_km,
            'num_turns': route.num_turns,
            'hour': hour,
            'day_of_week': day_of_week,
            'weather': self.weather_api.current_conditions(),
            'is_airport': self.is_airport_pickup(origin_lat, origin_lng),
        }
        ml_eta = self.ml_model.predict(features)

        return int(ml_eta)
Staff-Level Signal: Discuss ETA decomposition. The pickup ETA and trip ETA have different error profiles. Pickup ETAs must account for the driver finding the exact pickup spot (parking, one-way streets), which adds 2-5 minutes of uncertainty that pure routing ignores. Uber adds a learned "last mile" correction for this.
6e. Surge Pricing ▼

Surge pricing is a marketplace balancing mechanism. When demand exceeds supply in an area, prices increase to (a) incentivize more drivers to come online and (b) reduce marginal demand.

How Surge Works

  1. Divide city into zones: Use H3 hexagons at resolution 7 (~5 km² each)
  2. Measure supply: Count available drivers per zone
  3. Measure demand: Count ride requests per zone in last N minutes
  4. Compute multiplier: demand/supply ratio mapped to a pricing curve
  5. Smooth: Apply temporal smoothing to avoid price whiplash
def compute_surge_multiplier(zone_id, city_id):
    """Compute surge pricing multiplier for a zone."""
    # Count supply: available drivers in this zone
    supply = redis.scard(f"zone:{zone_id}:available_drivers")

    # Count demand: ride requests in last 5 minutes
    now = time.time()
    demand = redis.zcount(
        f"zone:{zone_id}:requests",
        now - 300,  # 5 min window
        now
    )

    if supply == 0:
        return MAX_SURGE  # Cap at 3.0x

    # Raw ratio
    ratio = demand / max(supply, 1)

    # Map ratio to multiplier using a curve
    # ratio < 0.5: no surge (1.0x)
    # ratio 0.5-1.0: mild surge (1.0-1.5x)
    # ratio 1.0-2.0: moderate surge (1.5-2.5x)
    # ratio > 2.0: heavy surge (2.5-3.0x, capped)
    if ratio < 0.5:
        multiplier = 1.0
    elif ratio < 1.0:
        multiplier = 1.0 + (ratio - 0.5) * 1.0
    elif ratio < 2.0:
        multiplier = 1.5 + (ratio - 1.0) * 1.0
    else:
        multiplier = min(2.5 + (ratio - 2.0) * 0.5, MAX_SURGE)

    # Temporal smoothing: blend with previous value
    prev = redis.get(f"zone:{zone_id}:surge") or 1.0
    smoothed = 0.7 * multiplier + 0.3 * float(prev)

    redis.setex(f"zone:{zone_id}:surge", 60, round(smoothed, 2))
    return round(smoothed, 2)

Fare Calculation

def calculate_fare(trip):
    """Calculate total fare for a completed trip."""
    # City-specific rate card
    rates = get_rate_card(trip.city_id, trip.ride_type)

    base_fare = rates.base_fare                          # e.g., $2.50
    distance_fare = trip.distance_miles * rates.per_mile  # e.g., $1.75/mile
    time_fare = trip.duration_minutes * rates.per_minute  # e.g., $0.35/min

    subtotal = base_fare + distance_fare + time_fare

    # Apply surge
    surged = subtotal * trip.surge_multiplier

    # Add booking fee and tolls
    total = surged + rates.booking_fee + trip.tolls

    # Apply minimum fare
    total = max(total, rates.minimum_fare)

    return round(total, 2)
Fairness Considerations (Staff-Level):
  • Surge caps: Many cities regulate maximum surge (e.g., 3x) and prohibit surge during declared emergencies
  • Upfront pricing: Modern Uber shows the fare before booking — this uses ML to predict the actual fare, decoupling the rider's price from the metered trip
  • Anti-gaming: Detect and prevent drivers from colluding to create artificial scarcity (mass log-offs)
  • Equity: Ensure surge doesn't systematically disadvantage lower-income neighborhoods
6f. Trip State Machine ▼

A trip's lifecycle is modeled as a finite state machine. Every transition triggers side effects (notifications, billing, driver status changes). Getting this right is critical for consistency.

stateDiagram-v2 [*] --> REQUESTED: Rider requests ride REQUESTED --> MATCHING: System searches for drivers MATCHING --> MATCHED: Driver accepts MATCHING --> NO_DRIVERS: No drivers available NO_DRIVERS --> MATCHING: Retry with expanded radius NO_DRIVERS --> CANCELLED: Timeout / rider cancels MATCHED --> DRIVER_EN_ROUTE: Driver heading to pickup DRIVER_EN_ROUTE --> DRIVER_ARRIVED: Driver at pickup DRIVER_ARRIVED --> IN_PROGRESS: Rider picked up, trip starts IN_PROGRESS --> COMPLETED: Trip ends at destination COMPLETED --> [*] REQUESTED --> CANCELLED: Rider cancels MATCHED --> CANCELLED: Rider/Driver cancels DRIVER_EN_ROUTE --> CANCELLED: Rider/Driver cancels DRIVER_ARRIVED --> CANCELLED: No-show timeout CANCELLED --> [*]

State Transition Handler

class TripStateMachine:
    VALID_TRANSITIONS = {
        'REQUESTED':       ['MATCHING', 'CANCELLED'],
        'MATCHING':        ['MATCHED', 'NO_DRIVERS', 'CANCELLED'],
        'NO_DRIVERS':      ['MATCHING', 'CANCELLED'],
        'MATCHED':         ['DRIVER_EN_ROUTE', 'CANCELLED'],
        'DRIVER_EN_ROUTE': ['DRIVER_ARRIVED', 'CANCELLED'],
        'DRIVER_ARRIVED':  ['IN_PROGRESS', 'CANCELLED'],
        'IN_PROGRESS':     ['COMPLETED'],
        'COMPLETED':       [],
        'CANCELLED':       [],
    }

    def transition(self, trip_id, new_status, actor=None):
        """Atomically transition a trip's state."""
        # Optimistic locking with version check
        trip = db.execute("""
            SELECT trip_id, status, version
            FROM trips WHERE trip_id = %s FOR UPDATE
        """, [trip_id])

        if new_status not in self.VALID_TRANSITIONS[trip.status]:
            raise InvalidTransitionError(
                f"Cannot go from {trip.status} to {new_status}"
            )

        # Update with version bump
        rows = db.execute("""
            UPDATE trips
            SET status = %s, version = version + 1, updated_at = NOW()
            WHERE trip_id = %s AND version = %s
        """, [new_status, trip_id, trip.version])

        if rows != 1:
            raise ConcurrentModificationError()

        # Trigger side effects
        self._on_transition(trip_id, trip.status, new_status, actor)

    def _on_transition(self, trip_id, old_status, new_status, actor):
        """Side effects for each state transition."""
        if new_status == 'MATCHED':
            notify_rider("Driver is on the way!", trip_id)
            notify_driver("New ride request accepted", trip_id)
            update_driver_status(trip_id, 'busy')

        elif new_status == 'DRIVER_ARRIVED':
            notify_rider("Your driver has arrived!", trip_id)
            start_noshow_timer(trip_id, timeout=300)  # 5 min

        elif new_status == 'IN_PROGRESS':
            cancel_noshow_timer(trip_id)
            start_trip_metering(trip_id)

        elif new_status == 'COMPLETED':
            stop_trip_metering(trip_id)
            fare = calculate_fare(trip_id)
            charge_rider(trip_id, fare)
            update_driver_status(trip_id, 'available')
            request_ratings(trip_id)

        elif new_status == 'CANCELLED':
            handle_cancellation(trip_id, old_status, actor)
            update_driver_status(trip_id, 'available')
Staff-Level Signal: Use SELECT ... FOR UPDATE (pessimistic locking) or optimistic concurrency control (version column) to prevent race conditions. Two simultaneous requests (rider cancels while driver completes) must result in exactly one valid transition, not a corrupted state. In a distributed system, consider using a saga pattern with compensation actions for multi-service state changes.
6g. Geo-indexing with Geohash: How It Works ▼

Geohashing converts a 2D coordinate (latitude, longitude) into a 1D string by interleaving bits of the latitude and longitude, then encoding in base-32. Nearby points share a common prefix.

Geohash Precision Levels

Length Cell Width Cell Height Use Case
4 39.1 km 19.5 km Metro area / city-level surge zones
5 4.9 km 4.9 km Neighborhood-level driver search
6 1.2 km 0.61 km Precise nearby driver lookup
7 153 m 153 m Street-level precision
8 38 m 19 m Exact pickup location

The Boundary Problem

Two points right next to each other can have completely different geohash prefixes if they fall on opposite sides of a cell boundary. To handle this, always query the target cell + its 8 neighbors:

import geohash2

def find_drivers_geohash(lat, lng, precision=6):
    """Find nearby drivers using geohash with neighbor expansion."""
    center = geohash2.encode(lat, lng, precision=precision)
    neighbors = geohash2.neighbors(center)  # Returns 8 surrounding cells

    all_cells = [center] + neighbors  # 9 cells total

    drivers = []
    for cell in all_cells:
        # Each cell maps to a Redis set of driver IDs
        cell_drivers = redis.smembers(f"geohash:{cell}:drivers")
        drivers.extend(cell_drivers)

    # Filter by exact distance and availability
    return [d for d in drivers
            if is_available(d) and haversine_distance(lat, lng, d.lat, d.lng) < 5.0]
Staff-Level Signal: Compare Geohash precision 6 (1.2 km cells, 9 cells = ~10 km² search area) with H3 resolution 9 (0.1 km² cells, 7 cells = ~0.7 km²). H3 gives much finer-grained control, but requires k-ring expansion for larger radii. Discuss the trade-off: Geohash is simpler and natively supported by Redis and many databases; H3 requires a library but gives better spatial properties.

7. Scaling Considerations

Scaling to Global Operations ▼

City-Level Sharding

Ride-sharing has a natural sharding key: city. A ride in New York never needs to know about drivers in London. This gives us clean isolation:

  • Location data: Shard Redis by city — city:{city_id}:drivers
  • Matching: Run independent matching engines per city
  • Surge: Compute per-city (or per-zone within city)
  • Trip data: Shard PostgreSQL by city_id for active trips; archive to a global data warehouse

Location Data Volume

At 21.6 TB/day of raw location data, storage strategy is critical:

  • Hot tier (Redis): Only latest location per driver (~5M entries, ~1 GB). This is the real-time index.
  • Warm tier (TimescaleDB/InfluxDB): Last 24 hours of location history for active trip tracking and ETA refinement. TTL = 24h.
  • Cold tier (S3 + Parquet): Compressed location history for analytics, ML training, route optimization. Stored indefinitely but queried via Spark/Presto.

Handling Peak Load

Challenge Strategy
New Year's Eve (10x surge in ride requests) Pre-provision matching capacity; aggressive surge pricing activates early to attract drivers; queue ride requests with estimated wait times
Sports event end (sudden spike in one area) Predictive scaling based on event schedules; "scheduled rides" feature; geofence-aware surge
WebSocket connection storms Rate limiting on reconnects; connection draining during deployments; regional WebSocket gateway clusters
Kafka consumer lag Partition by city for parallelism; auto-scale consumer groups; drop stale location updates (>10s old)

Multi-Region Deployment

  • Regional independence: Each region (US, EU, APAC) runs its own full stack. A rider in Mumbai never hits US servers.
  • Data sovereignty: GDPR requires EU trip data stays in EU. No cross-region replication of PII.
  • Global services: User authentication, account management, and analytics can be centralized with read replicas.
  • Cross-region travel: When a user opens the app in a new region, route them to the nearest regional deployment. Account data syncs lazily.

8. Failure Scenarios

Handling Failures Gracefully ▼

Driver Goes Offline Mid-Trip

Detection: No location update received for >30 seconds (normally every 4s).

Response:

  1. Mark driver connection as "stale" after 15 seconds
  2. Show rider "driver connection lost" and last known position
  3. After 60 seconds, attempt phone call to driver
  4. After 120 seconds, offer rider option to cancel (no fee) or wait
  5. If driver reconnects, resume trip from last known state
  6. If driver never reconnects, cancel trip, refund any charge, assign new driver

Root causes: Tunnel/dead zone, phone crash, battery died, driver-side app crash. Design for all of these.

Payment Fails After Trip Completion

Strategy: Pre-authorize payment before matching a driver. Place a hold for the estimated fare amount.

  1. On ride request: pre-auth estimated fare + 20% buffer
  2. On trip completion: capture actual amount (must be ≤ pre-auth)
  3. If capture fails: retry 3 times with exponential backoff
  4. If still fails: queue for async retry, mark trip as "payment_pending"
  5. Driver gets paid regardless (platform absorbs risk)
  6. Rider account flagged; future rides require alternative payment method

GPS Drift / Inaccurate Location

Problem: Urban canyons, tunnels, and multi-story buildings cause GPS accuracy to degrade to 50-100m.

Mitigations:

  • Snap to road: Use map-matching algorithms to snap GPS points to the nearest road segment
  • Kalman filter: Smooth noisy GPS readings using speed, heading, and historical trajectory
  • Outlier rejection: If a GPS point implies >200 km/h movement, discard it
  • Pickup pin: Let riders manually adjust pickup pin; driver sees both GPS and pin location

Matching Service Failure

Fallback strategy:

  • Queue ride requests in Kafka; they survive service restarts
  • Degrade to simple nearest-driver matching (skip optimization)
  • Show riders "finding your ride" with extended timeout
  • Circuit breaker prevents cascading failures to location and pricing services

Redis Cluster Failure (Location Index)

Impact: Cannot find nearby drivers = cannot match rides.

Mitigations:

  • Redis Cluster with replicas (automatic failover in <10s)
  • Secondary index in Elasticsearch as hot standby
  • Drivers cache their own location locally; on Redis recovery, bulk re-register
  • During outage: use last known positions (stale by seconds, usually acceptable)

10. Common Follow-up Questions

Questions Interviewers Ask ▼

Q: How would you implement ride-sharing/pooling (UberPool)?

A: Pool introduces a fundamentally harder matching problem. Instead of 1:1 driver-rider matching, you must solve a vehicle routing problem (VRP): given N active pool requests, find routes that pick up and drop off multiple riders with minimal detour. This requires:

  • A route compatibility check: Can rider B be added to driver X's current route without adding >5 min to rider A's trip?
  • Dynamic re-optimization as new requests arrive
  • Shared fare splitting based on actual detour percentage
  • This is NP-hard; use heuristic algorithms (insertion heuristic, simulated annealing) with time limits

Q: How do you handle the "two riders request the same driver" race condition?

A: Use atomic claim operations. In Redis: SET driver:{id}:trip {trip_id} NX EX 30 (NX = only if not exists). In PostgreSQL: UPDATE drivers SET status='busy' WHERE id=? AND status='available' and check affected_rows == 1. The loser retries with the next-best driver. This is compare-and-swap at the application level.

Q: How would you handle a natural disaster or mass evacuation?

A: Disable surge pricing (regulatory requirement in many jurisdictions). Activate emergency mode: prioritize rides leaving the affected area, increase matching radius, enable all vehicle types for all requests, coordinate with city emergency services for road closures, and use push notifications to alert drivers of surge demand areas (without price incentive, use "community service" framing). Pre-compute evacuation routing.

Q: How do you prevent driver fraud (fake rides, GPS spoofing)?

A: Multi-layered detection:

  • GPS spoofing detection: Compare GPS with cell tower triangulation and Wi-Fi positioning. Flag inconsistencies.
  • Trip pattern analysis: ML model trained on legitimate trips detects anomalies (too-perfect routes, rides between the same two points repeatedly, rides that start and end at the same location).
  • Speed/acceleration validation: If GPS shows 0-to-60 in 0.5 seconds, the data is spoofed.
  • Photo verification: Periodic selfie checks for drivers; rider-driver photo match.

Q: How do you ensure driver safety?

A: Real-time trip monitoring: if a trip deviates significantly from the expected route or enters an unusual area, trigger an automated check-in. Provide an in-app SOS button that shares live location with emergency services. Record trip audio (with consent) as a deterrent. Driver identity verification at onboarding and periodic re-checks.

Q: How would you design scheduled rides (book a ride for tomorrow at 6 AM)?

A: Scheduled rides use a deferred matching approach: store the request with a target time, then run a matching job 10-15 minutes before the scheduled pickup. The challenge is reserving driver capacity without locking drivers too early (they might want to take other rides). Solutions: show the scheduled ride to nearby drivers as an "offer" they can pre-accept; use ML to predict driver supply at the target time and only confirm if confidence is high. If no pre-accepted driver, fall back to on-demand matching at T-10 minutes.
