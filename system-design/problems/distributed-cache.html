Design: Distributed Cache

Examples: Memcached, Redis Cluster, Amazon ElastiCache, Hazelcast

Difficulty: Hard | Time: 45-60 minutes

1. Problem Statement

Design a distributed in-memory caching system that can store hundreds of terabytes of data across thousands of nodes, serving millions of read/write operations per second with sub-millisecond latency.

The cache sits between your application tier and the persistent data store (database, object store, etc.), absorbing the vast majority of read traffic. At companies like Facebook, a single Memcached cluster serves over 1 billion requests per second and stores hundreds of terabytes of data. Getting caching right is one of the highest-leverage design decisions at scale.

Why is this a staff-level problem? A distributed cache touches nearly every core distributed systems concept: partitioning, replication, consistency, failure detection, memory management, and performance optimization. Your answer reveals how deeply you understand the infrastructure that powers modern web-scale systems.

2. Requirements Clarification

Functional Requirements ▼
  • GET(key): Retrieve a value by key; return null/miss on absence
  • SET(key, value, TTL): Store a key-value pair with an optional time-to-live
  • DELETE(key): Explicitly remove a key from the cache
  • TTL / Expiration: Keys automatically expire after a configurable duration
  • Eviction: When memory is full, evict entries based on a configurable policy (LRU, LFU, random)
  • Atomic operations: Compare-and-swap (CAS), increment/decrement for counters
  • Bulk operations: Multi-get / multi-set for reducing round trips
Non-Functional Requirements ▼
  • Sub-millisecond latency: P99 read latency < 1ms within a datacenter
  • High throughput: Support 1M+ operations per second per node
  • High availability: No single point of failure; survive node and rack failures
  • Horizontal scalability: Add nodes to increase capacity linearly
  • Consistency model: Eventual consistency is acceptable; strong consistency optional
  • Memory efficiency: Minimize per-key overhead to maximize useful data stored
  • Operational simplicity: Easy to add/remove nodes, monitor, and debug

4. High-Level Architecture

Architecture Overview ▼

The system has four main layers:

  1. Client Library -- embedded in app servers; handles key hashing, routing, connection pooling, retries
  2. Cache Proxy (optional) -- a thin proxy layer (like mcrouter or twemproxy) that handles routing, replication, and failover centrally
  3. Cache Nodes -- the actual in-memory data store instances, each owning a portion of the keyspace
  4. Configuration Service -- a coordination service (ZooKeeper/etcd) that maintains the cluster topology and consistent hash ring

Request Flow (Cache Read)

  1. Application calls cache.get("user:12345")
  2. Client library hashes the key: hash("user:12345") mod N to determine the target node
  3. Request is sent to that node (directly or via proxy)
  4. Node looks up key in its in-memory hash table
  5. On hit: return value directly (sub-millisecond)
  6. On miss: application fetches from database and populates cache (cache-aside pattern)

5. Data Model

Key-Value Storage Structure ▼

Each cache entry stores the following:

+------------------------------------------------------------------+
| Cache Entry                                                      |
+------------------------------------------------------------------+
| key_hash       : uint64    (8 bytes)  - hash of the key          |
| key_length     : uint16    (2 bytes)  - length of the raw key    |
| key            : byte[]    (variable) - the actual key string    |
| value_length   : uint32    (4 bytes)  - length of the value      |
| value          : byte[]    (variable) - serialized value (opaque)|
| flags          : uint32    (4 bytes)  - client-defined flags     |
| ttl_timestamp  : uint32    (4 bytes)  - absolute expiry (epoch)  |
| cas_token      : uint64    (8 bytes)  - compare-and-swap version |
| lru_prev       : pointer   (8 bytes)  - LRU doubly-linked list  |
| lru_next       : pointer   (8 bytes)  - LRU doubly-linked list  |
| slab_class     : uint8     (1 byte)   - slab allocator class ID |
+------------------------------------------------------------------+
| Metadata overhead per entry: ~56 bytes (excluding key/value)     |
+------------------------------------------------------------------+

Hash Table Design

The primary index is an open-addressing hash table (like Memcached) or a chained hash table. Key design decisions:

  • Hash function: Use a fast, well-distributed function like xxHash or MurmurHash3 (not cryptographic -- speed matters more than security here)
  • Collision resolution: Chaining with linked lists, or cuckoo hashing for better cache-line utilization
  • Load factor: Keep below 0.75 to maintain O(1) average lookup
  • Resizing: Incremental rehashing (Redis-style) to avoid latency spikes during expansion

6. Deep Dives

6a. Consistent Hashing with Virtual Nodes ▼

Why not simple modular hashing? With hash(key) % N, adding or removing a node remaps nearly all keys. For a 100TB cache, rehashing means a massive thundering-herd of cache misses hitting your database simultaneously -- a catastrophic scenario called a cache avalanche.

Consistent hashing maps both keys and nodes onto a virtual ring (0 to 2^32-1). A key is assigned to the first node encountered clockwise from its hash position. Adding/removing a node only affects the keys between it and its predecessor -- roughly 1/N of total keys.

Virtual Nodes

With few physical nodes, the ring can become unbalanced. Virtual nodes (vnodes) solve this by placing each physical node at multiple positions on the ring (typically 100-200 vnodes per physical node).

  • Better load distribution: With 150 vnodes, standard deviation of load drops to ~10% of mean (vs. ~50% with 1 vnode)
  • Heterogeneous hardware: Give more vnodes to nodes with more memory
  • Smoother rebalancing: When a node leaves, its load spreads evenly across many nodes, not just one neighbor

Implementation

# Consistent hash ring with virtual nodes
import hashlib
from bisect import bisect_right

class ConsistentHashRing:
    def __init__(self, nodes=None, vnodes=150):
        self.vnodes = vnodes
        self.ring = {}       # hash_value -> node
        self.sorted_keys = []
        if nodes:
            for node in nodes:
                self.add_node(node)

    def _hash(self, key: str) -> int:
        """Generate a 32-bit hash for a given key."""
        return int(hashlib.md5(key.encode()).hexdigest(), 16) % (2**32)

    def add_node(self, node: str):
        """Add a node with its virtual nodes to the ring."""
        for i in range(self.vnodes):
            vnode_key = f"{node}:vnode{i}"
            h = self._hash(vnode_key)
            self.ring[h] = node
            self.sorted_keys.append(h)
        self.sorted_keys.sort()

    def remove_node(self, node: str):
        """Remove a node and all its virtual nodes."""
        for i in range(self.vnodes):
            vnode_key = f"{node}:vnode{i}"
            h = self._hash(vnode_key)
            del self.ring[h]
            self.sorted_keys.remove(h)

    def get_node(self, key: str) -> str:
        """Find the node responsible for a given key."""
        if not self.ring:
            return None
        h = self._hash(key)
        idx = bisect_right(self.sorted_keys, h)
        if idx == len(self.sorted_keys):
            idx = 0  # Wrap around the ring
        return self.ring[self.sorted_keys[idx]]

# Usage
ring = ConsistentHashRing(["cache-1", "cache-2", "cache-3"])
print(ring.get_node("user:12345"))  # => "cache-2"
print(ring.get_node("session:abc"))  # => "cache-1"
6b. Eviction Policies: LRU Implementation ▼

When a cache node runs out of memory, it must evict entries to make room. The eviction policy determines which entries to remove.

PolicyMechanismProsCons
LRU (Least Recently Used) Evict the entry accessed longest ago Good general-purpose; recency is strong signal Scan pollution (one-time scans evict hot entries)
LFU (Least Frequently Used) Evict the entry with fewest accesses Keeps long-term popular items Slow to adapt to changing access patterns
Random Evict a random entry O(1), no bookkeeping overhead May evict hot entries
TTL-based Evict expired entries first Predictable; respects data freshness Doesn't help when nothing is expired
Approx LRU (Redis) Sample N random keys, evict the oldest Near-LRU quality with O(1) memory overhead Slightly worse than true LRU

LRU: Doubly-Linked List + HashMap (O(1) operations)

The classic LRU cache uses two data structures together: a hash map for O(1) lookup by key, and a doubly-linked list to track access order.

class LRUNode:
    """Node in the doubly-linked list."""
    def __init__(self, key, value):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None

class LRUCache:
    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> LRUNode

        # Sentinel nodes simplify edge cases
        self.head = LRUNode(None, None)  # Most recently used
        self.tail = LRUNode(None, None)  # Least recently used
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: LRUNode):
        """Remove a node from the linked list."""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node: LRUNode):
        """Add a node right after head (most recently used)."""
        node.prev = self.head
        node.next = self.head.next
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: str):
        """O(1) get -- move accessed node to front."""
        if key not in self.cache:
            return None  # Cache miss
        node = self.cache[key]
        self._remove(node)
        self._add_to_front(node)
        return node.value

    def put(self, key: str, value):
        """O(1) put -- evict LRU if at capacity."""
        if key in self.cache:
            self._remove(self.cache[key])
        node = LRUNode(key, value)
        self._add_to_front(node)
        self.cache[key] = node

        if len(self.cache) > self.capacity:
            # Evict from tail (least recently used)
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]

# Usage
cache = LRUCache(3)
cache.put("a", 1)
cache.put("b", 2)
cache.put("c", 3)
cache.get("a")       # Moves "a" to front
cache.put("d", 4)    # Evicts "b" (least recently used)

Redis Approximate LRU

Redis does not maintain a true linked list. Instead, each key stores a 24-bit timestamp of its last access. When eviction is needed, Redis samples maxmemory-samples (default 5) random keys and evicts the one with the oldest timestamp. With a sample size of 10, this approximation is nearly indistinguishable from true LRU -- and uses zero extra pointers per key.

6c. Hot Key Problem ▼

A hot key is a single key that receives a disproportionate share of traffic. Examples: a viral tweet, a celebrity's profile, a flash sale product. A single cache node handling that key becomes a bottleneck and can be overwhelmed.

Detection

  • Real-time monitoring: Track per-key QPS at the proxy layer. Flag keys exceeding a threshold (e.g., >10,000 QPS)
  • Sampling: Log a random 1% of requests and aggregate counts server-side
  • Redis: redis-cli --hotkeys (uses LFU approximation)

Solutions

ApproachHow It WorksTrade-off
Local (L1) cache Each app server caches hot keys in-process memory (e.g., Guava cache with 1-5s TTL) Risk of stale data; bounded by app server memory
Key replication Replicate hot key to N random nodes; client randomly picks one Adds write complexity; consistency delay
Key splitting Append a random suffix: hot_key_1, hot_key_2, ..., hot_key_N; client reads from random shard Application must know about splitting
Read-through proxy cache Proxy layer (mcrouter) caches hot responses locally for a short TTL Additional memory at proxy tier

Facebook's Approach

Facebook uses mcrouter as a cache proxy. For hot keys, mcrouter maintains a small local cache and batches requests to the same key into a single backend fetch, then fans out the response. This is called request coalescing -- if 1,000 concurrent requests arrive for the same key, only one is sent to the cache node, and the result is shared among all 1,000 waiters.

6d. Cache Coherency & Invalidation ▼

"There are only two hard things in Computer Science: cache invalidation and naming things." -- Phil Karlton

Caching Patterns

PatternRead PathWrite PathBest For
Cache-Aside (Lazy Loading) App checks cache; on miss, reads DB and populates cache App writes DB, then invalidates/deletes cache Read-heavy workloads; simplest model
Write-Through App reads cache (always populated) App writes to cache, cache synchronously writes to DB Strong consistency needed; write latency acceptable
Write-Behind (Write-Back) App reads cache App writes to cache; cache asynchronously flushes to DB Write-heavy; can tolerate brief data loss risk
Read-Through Cache itself fetches from DB on miss (transparent) Varies When you want the cache to manage DB interaction

Invalidation Strategies

  • Delete on write: After writing to DB, delete the cache key. Next read triggers a cache-aside fill. Simple and safe, but the next read has a cache miss penalty.
  • Lease-based invalidation (Facebook): On a cache miss, the server issues a "lease" (a token). The client must present this lease when filling the cache. If the key was invalidated between the miss and the fill, the lease is revoked, preventing stale writes. This solves the thundering herd and stale set problems simultaneously.
  • Version-based: Store a version counter. Cache entries include their version. On read, compare versions; re-fetch if stale.
  • Event-driven invalidation: Database change events (CDC / binlog) trigger cache invalidation asynchronously via a message queue.

6e. Replication for High Availability ▼

For a cache, replication is a nuanced decision. Unlike a database, lost cache data can be re-fetched from the source of truth. However, at massive scale, losing an entire node's data causes a cache stampede that can overwhelm the database.

Replication Strategies

StrategyDescriptionUsed By
No replication Accept cold-cache penalty on node failure; simple and memory-efficient Memcached (by default)
Primary-replica async Primary accepts writes, replicates to N replicas asynchronously Redis Cluster
Multi-region replication Regional clusters with cross-datacenter invalidation Facebook (regional pools)

Redis Cluster Replication

  • Each primary has 1+ replicas; replicas are read-only
  • Replication is asynchronous -- brief data loss is possible on failover
  • When a primary fails, the cluster promotes a replica via Raft-like consensus among the remaining primaries
  • Clients are redirected via MOVED and ASK redirections during resharding

Warm-up on Recovery

When a new or restarted node joins, its cache is empty. To prevent a stampede:

  • Gradual traffic shifting: Route only a small % of traffic to the new node initially, increasing over minutes
  • Snapshot loading: Load an RDB/AOF snapshot from a replica before serving traffic
  • Double-read: On miss, also check the old owner before going to the database
6f. Memory Management: Slab Allocation ▼

Naive memory allocation with malloc()/free() leads to severe memory fragmentation over time. After millions of allocations and frees of varying sizes, the heap becomes a Swiss cheese of unusable small gaps.

Memcached's Slab Allocator

Memcached pre-allocates memory into slab classes, each holding fixed-size chunks.

  • Slab class 1: 96-byte chunks
  • Slab class 2: 120-byte chunks (1.25x growth factor)
  • Slab class 3: 152-byte chunks
  • ... up to 1 MB chunks

When storing a 100-byte item, Memcached picks slab class 2 (120 bytes) -- wasting 20 bytes but eliminating fragmentation. Each slab class maintains its own LRU list and free list.

Slab Class 3 (152-byte chunks)
+--------+--------+--------+--------+--------+--------+
| Chunk1 | Chunk2 | Chunk3 | Chunk4 | Chunk5 | Chunk6 |
| [USED] | [USED] | [FREE] | [USED] | [FREE] | [USED] |
+--------+--------+--------+--------+--------+--------+
         ^                            ^
         |                            |
       LRU tail                    Free list head

Trade-offs

  • Pro: Zero fragmentation; O(1) allocation and deallocation
  • Con: Internal waste (items rarely perfectly fill a chunk)
  • Con: Slab calcification -- if workload changes and different slab classes are needed, memory is stuck in the old classes. Memcached added slab_reassign and slab automove to address this.

Redis Approach

Redis uses jemalloc as its memory allocator (a general-purpose allocator optimized for multi-threaded applications). jemalloc uses size classes and thread-local caches to minimize fragmentation and lock contention. Redis also provides MEMORY DOCTOR and MEMORY PURGE commands for diagnosis.

6g. Client-Side Optimizations ▼

Connection Pooling

Establishing a TCP connection is expensive (~1ms with TLS handshake). Each app server maintains a pool of persistent connections to each cache node. This avoids connection overhead per request.

  • Typical pool size: 5-20 connections per cache node per app server
  • Connections are health-checked and recycled periodically
  • Use consistent connection ordering to avoid deadlocks

Pipelining

Instead of waiting for each response before sending the next request (one RTT per operation), pipelining sends multiple commands without waiting, then reads all responses. This reduces N round trips to 1.

# Without pipelining: N round trips
for key in keys:
    value = cache.get(key)  # 1 RTT each = N * 0.5ms

# With pipelining: 1 round trip
pipe = cache.pipeline()
for key in keys:
    pipe.get(key)
results = pipe.execute()  # 1 RTT = 0.5ms total

# For 100 keys: 50ms vs 0.5ms -- a 100x improvement

Multi-Get Optimization

When fetching multiple keys that live on different nodes, the client library should:

  1. Group keys by their target node (using consistent hash)
  2. Send parallel requests to each node
  3. Aggregate results

This turns a serial fan-out into a parallel fan-out. Latency = max(individual node latencies) instead of sum.

Compression

For values larger than ~1 KB, compress before storing (zstd, lz4, or snappy). This reduces memory usage and network bandwidth at the cost of CPU. LZ4 compresses at ~4 GB/s and decompresses at ~8 GB/s -- negligible overhead for most workloads.

7. Failure Handling

Failure Scenarios & Mitigations ▼
FailureImpactMitigation
Single node crash ~1/N of keyspace becomes uncached; cache misses spike Replicas promote automatically; consistent hashing limits blast radius; gradual warm-up of replacement node
Network partition Some clients can't reach some nodes Client retries with fallback to replica or DB; proxy detects and reroutes
Thundering herd (cache stampede) Many clients simultaneously miss same key and hit DB Lease-based locking (only one client fills); stale-while-revalidate pattern; request coalescing at proxy
Cache avalanche Many keys expire at the same time Add random jitter to TTLs (e.g., TTL = base + random(0, 60s)); stagger expiration
Cache penetration Requests for non-existent keys always miss and hit DB Cache negative results (null values) with short TTL; Bloom filter to reject impossible keys
Split brain Partitioned replicas both accept writes, causing divergence Quorum reads/writes; fencing tokens; for cache, eventual merge is usually acceptable since data can be re-fetched
Full memory (OOM) Node can't accept new writes Eviction policy kicks in automatically; monitor memory usage; set maxmemory with eviction enabled

Gutter Pool (Facebook Pattern)

Facebook maintains a small "gutter" pool of idle cache servers. When a node fails, traffic is temporarily redirected to the gutter pool. This prevents the thundering herd from hitting the database. The gutter pool entries use a short TTL (e.g., 10 seconds) since they are purely a shock absorber while the primary node recovers.

When to Choose Which?

  • Choose Memcached when: you need a simple, fast, multi-threaded cache for string values; memory efficiency is paramount; you are caching database query results or rendered page fragments at extreme scale.
  • Choose Redis when: you need rich data structures (sorted sets for leaderboards, lists for queues); you want built-in replication and persistence; you need pub/sub or Lua scripting; your use case benefits from atomic operations on complex types.

9. Common Follow-up Questions

Interview Follow-ups ▼

Q: How would you handle multi-region caching?

A: Each region has its own cache cluster. Writes go to the local cluster and are replicated to remote clusters asynchronously. Use a message bus (Kafka) for cross-region invalidation. Accept eventual consistency (reads in remote regions may be stale for a few hundred milliseconds). Facebook's approach: writes go to the primary region; remote regions receive invalidation markers via a dedicated replication pipeline.

Q: How do you handle cache warming after a cold start?

A: (1) Load an RDB snapshot from a replica. (2) Use shadow traffic: replay production requests to the new node without serving responses, just to fill the cache. (3) Gradual traffic shifting: start with 1% of traffic, increase as hit ratio improves. (4) Pre-populate known hot keys from a database query.

Q: How would you implement rate limiting with a distributed cache?

A: Use Redis's atomic INCR with EXPIRE. For a sliding window, use a sorted set with timestamps as scores. For token bucket, use a Lua script that atomically checks and decrements a counter. The key is the rate limit identifier (user ID, IP, API key).

Q: What if your cache needs to handle objects larger than memory on a single node?

A: Split large values into chunks, store each chunk as a separate key with a prefix (e.g., blob:123:chunk:0, blob:123:chunk:1). Store metadata about chunk count. Use multi-get to fetch all chunks in parallel. Alternatively, use a tiered cache: in-memory cache for hot/small items, SSD-based cache (e.g., CacheLib by Facebook) for larger items.

Q: How do you monitor the health of a distributed cache?

A: Key metrics: (1) Hit ratio -- should be >95%; drops indicate working set changed or TTLs are wrong. (2) Latency percentiles (P50, P99, P99.9) -- P99 spikes indicate GC pauses or network issues. (3) Eviction rate -- high eviction means you need more memory. (4) Memory fragmentation ratio -- should be near 1.0. (5) Connection count -- sudden spikes may indicate connection leak. (6) Replication lag (Redis) -- high lag means replicas are falling behind.

Q: How would you handle data that should never be cached?

A: Implement a "no-cache" flag at the application layer. Sensitive data (PII, payment info) should bypass the cache or use encrypted cache entries with short TTLs. For compliance (GDPR), ensure cache invalidation is part of your data deletion pipeline -- when a user requests data deletion, purge all cache entries for that user across all regions.

Q: What is cache-aside vs read-through vs write-through? When would you use each?

A: See Section 6d above for the full comparison. In short: cache-aside is the simplest and most common -- the application manages cache population. Read-through hides the DB behind the cache (good for CDN-like patterns). Write-through ensures the cache is always up-to-date but adds write latency. Write-behind absorbs write spikes but risks data loss.

  • Structured approach: Start with requirements, then estimation, then high-level design, then deep dives. Don't jump into implementation details.
  • Trade-off awareness: Every design decision is a trade-off. Explain WHY you chose LRU over LFU, eventual consistency over strong consistency, replication over no replication.
  • Real-world grounding: Reference Memcached, Redis, and how companies like Facebook, Twitter, and Netflix use caching at scale.
  • Failure thinking: Proactively discuss what happens when things go wrong -- node failures, hot keys, thundering herds, split brains.

Red Flags to Avoid

  • Treating the cache as a single box instead of a distributed system
  • Ignoring the consistency problem between cache and database
  • Not discussing eviction until asked
  • Using hash(key) % N without understanding why consistent hashing is needed
  • Forgetting about hot keys and uneven load distribution
  • Not discussing memory management (fragmentation, slab allocation)
  • Assuming the cache is always correct (stale data issues)

11. Key Takeaways

Design Summary

  1. Partitioning: Consistent hashing with virtual nodes distributes keys evenly and minimizes redistribution on topology changes
  2. Eviction: LRU (or approximate LRU) is the default; use LFU for frequency-biased workloads
  3. Hot keys: Detect with sampling/monitoring; mitigate with local caching, key splitting, or request coalescing
  4. Coherency: Cache-aside with delete-on-write is simplest; use leases or CDC for stronger guarantees
  5. Availability: Replication trades memory for resilience; gutter pools absorb failure shock
  6. Memory: Slab allocation (Memcached) or jemalloc (Redis) prevent fragmentation
  7. Client: Connection pooling, pipelining, and parallel multi-get are essential for performance
