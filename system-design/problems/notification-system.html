Design: Notification System

Examples: Facebook Notifications, Uber Alerts, Amazon Order Updates, Slack Messages

Difficulty: Hard | Time: 45-60 minutes

1. Requirements Clarification

Functional Requirements ▼
  • Multi-channel delivery: Push notifications (iOS/Android), Email, SMS, In-app notifications
  • Template engine: Support parameterized templates for each channel (e.g., "Hi {{name}}, your order {{order_id}} has shipped")
  • Scheduling: Send immediately or schedule for a future time (e.g., "send at 9 AM user local time")
  • User preferences: Users can opt in/out per channel and per notification type (marketing, transactional, social)
  • Notification history: Users can view past notifications in an in-app feed
  • Read/unread tracking: Mark notifications as read; show unread count badge
  • Batch/bulk send: Send a notification to millions of users (e.g., product announcement)
  • Priority levels: Urgent (OTP, security alerts), High (order updates), Normal (social), Low (marketing)
Non-Functional Requirements ▼
  • Scale: 100M users, 1B notifications/day across all channels
  • Low latency: Real-time notifications delivered within seconds for urgent messages
  • At-least-once delivery: Every notification must be delivered at least once (prefer exactly-once for SMS/email to avoid user annoyance)
  • Deduplication: Prevent sending the same notification twice to the same user
  • High availability: 99.99% uptime -- notifications are critical for user engagement and transactional flows
  • Observability: Track delivery rates, open rates, click-through rates per channel
  • Extensibility: Easy to add new channels (WhatsApp, Slack, browser push) without changing core logic
Scope for Interview ▼
In Scope:
  • Core notification pipeline (ingest, route, deliver)
  • Push, Email, SMS, and In-app channels
  • User preferences and opt-out
  • Retry, deduplication, and rate limiting
  • Template rendering
Out of Scope:
  • Content moderation / spam filtering (mention briefly)
  • A/B testing of notification copy
  • Rich media rendering in push payloads

3. High-Level Design

Architecture Overview ▼

The notification system follows an event-driven, queue-based architecture. Internal services publish notification requests to the Notification Service, which validates, renders templates, checks user preferences, and fans out messages to per-channel queues. Channel-specific workers consume from these queues and deliver via third-party providers (APNs, FCM, Twilio, SendGrid).

Key Design Decisions

  • Queue per channel: Each delivery channel has its own message queue. This isolates failures -- if the SMS provider is down, push notifications keep flowing.
  • Preference check before enqueue: We filter out opted-out channels early to avoid wasted work in downstream workers.
  • Template rendering in the service: The Notification Service renders the template once and passes the final content to channel queues, avoiding duplicate rendering.
  • Dedup cache: A Redis-based idempotency check using a hash of (user_id, notification_type, dedup_key) with a short TTL prevents duplicate sends.

6. Deep Dive

Notification Pipeline Flow ▼

When a service sends a notification request, it passes through several stages before reaching the user. This sequence diagram shows the end-to-end flow:

sequenceDiagram participant P as Producer Service participant NS as Notification Service participant DD as Dedup Cache (Redis) participant PS as Preference Store participant TE as Template Engine participant MQ as Message Queues participant W as Channel Worker participant PR as Provider (FCM/Twilio/SES) participant AN as Analytics P->>NS: POST /notifications/send NS->>DD: Check dedup_key exists? alt Duplicate DD-->>NS: Key exists NS-->>P: 200 Already sent else New DD-->>NS: Key not found NS->>DD: SET dedup_key (TTL 24h) NS->>PS: Get user preferences PS-->>NS: Enabled channels + quiet hours NS->>NS: Filter channels by prefs NS->>TE: Render template per channel TE-->>NS: Rendered content NS->>NS: Persist notification record loop For each enabled channel NS->>MQ: Enqueue to channel queue end NS-->>P: 202 Accepted end MQ->>W: Dequeue message W->>PR: Send via provider API alt Success PR-->>W: 200 OK (message_id) W->>AN: Log delivery event else Failure PR-->>W: Error (429/500) W->>MQ: Re-enqueue with backoff end

Pipeline Stages Explained

  1. Validation: Check required fields, validate notification_type, verify the calling service has permission to send this type.
  2. Deduplication: Hash of (user_id + dedup_key) checked against Redis. Prevents double-sends from retried upstream calls.
  3. Preference Lookup: Load user's channel preferences from cache (Redis) or DB. Apply quiet hours check based on user's timezone.
  4. Template Rendering: Load the template for the notification_type + channel combination. Substitute variables. Produce channel-specific output (HTML for email, short text for SMS, JSON for push).
  5. Persistence: Write the notification record to the database. This enables the notification history feed and audit trail.
  6. Fan-out to Queues: For each enabled channel, enqueue a message containing the rendered content, user_id, device tokens (for push), etc.
  7. Channel Workers: Consume from channel-specific queues. Call the third-party provider API. Handle retries on failure.
  8. Analytics: Log delivery events (sent, delivered, opened, clicked) to the analytics pipeline for tracking.
Priority Queue and Rate Limiting ▼

Not all notifications are equal. An OTP code must arrive in seconds; a marketing email can wait hours. We use priority queues and per-user rate limiting to balance urgency with user experience.

Priority Queue Design

Each channel has multiple priority queues (or a single queue with priority lanes in Kafka). Workers always drain higher-priority queues first:

# Kafka topic partitioning by priority # Each channel has 4 priority topics: # push-notifications-p1 (urgent: OTP, security) # push-notifications-p2 (high: order updates) # push-notifications-p3 (normal: social) # push-notifications-p4 (low: marketing) # Worker consumer logic (Python pseudocode) class PriorityConsumer: def __init__(self, channel: str): self.consumers = { 1: KafkaConsumer(f'{channel}-notifications-p1'), 2: KafkaConsumer(f'{channel}-notifications-p2'), 3: KafkaConsumer(f'{channel}-notifications-p3'), 4: KafkaConsumer(f'{channel}-notifications-p4'), } async def consume(self): while True: # Always check higher priority first for priority in [1, 2, 3, 4]: messages = self.consumers[priority].poll(timeout_ms=100) if messages: for msg in messages: await self.process(msg) break # Go back to P1 after processing a batch else: await asyncio.sleep(0.1) # No messages in any queue async def process(self, message): # Rate limit check before sending user_id = message.value['user_id'] if await self.is_rate_limited(user_id): # Re-enqueue with delay or drop if low priority if message.value['priority'] >= 3: return # Drop low-priority if rate limited await self.delay_and_requeue(message) return await self.send(message)

Per-User Rate Limiting

Users should not receive more than N notifications per hour to prevent notification fatigue. We use a sliding window counter in Redis:

# Redis sliding window rate limiter import time import redis class NotificationRateLimiter: def __init__(self, redis_client: redis.Redis): self.redis = redis_client # Limits per channel per hour self.limits = { 'push': 10, # Max 10 push per hour 'email': 5, # Max 5 emails per hour 'sms': 3, # Max 3 SMS per hour 'in_app': 50, # More lenient for in-app } def is_allowed(self, user_id: int, channel: str) -> bool: key = f"rate:{user_id}:{channel}" now = time.time() window = 3600 # 1 hour pipe = self.redis.pipeline() pipe.zremrangebyscore(key, 0, now - window) # Remove old entries pipe.zcard(key) # Count recent pipe.zadd(key, {str(now): now}) # Add current pipe.expire(key, window) # Set TTL results = pipe.execute() current_count = results[1] limit = self.limits.get(channel, 10) return current_count < limit def get_remaining(self, user_id: int, channel: str) -> int: key = f"rate:{user_id}:{channel}" now = time.time() self.redis.zremrangebyscore(key, 0, now - 3600) count = self.redis.zcard(key) limit = self.limits.get(channel, 10) return max(0, limit - count)
Interview Tip: Mention that urgent notifications (priority 1) bypass rate limits. An OTP or security alert must always go through regardless of the user's recent notification volume.
Retry and Failure Handling ▼

Third-party providers (APNs, FCM, Twilio, SendGrid) can fail transiently. We need robust retry logic with exponential backoff and a dead-letter queue (DLQ) for messages that exhaust all retries.

Dead Letter Queue (DLQ) Strategy

Failure Type Action Example
Invalid device token Mark token inactive, no retry APNs returns 410 (unregistered)
Provider rate limit (429) Retry with backoff, respect Retry-After header Twilio rate limit exceeded
Provider outage (503) Retry with exponential backoff, circuit breaker SendGrid downtime
Max retries exhausted Move to DLQ, alert ops team, try fallback channel 5 failed attempts on push → try email
Malformed payload DLQ immediately, alert dev team Template rendered invalid JSON
Circuit Breaker Pattern: If a provider fails N consecutive times within a window, open the circuit breaker. Stop sending to that provider and either queue messages for later or fall back to an alternative provider (e.g., FCM → Huawei Push, SendGrid → AWS SES). Check provider health periodically and close the circuit when it recovers.

Multi-Device Handling

  • A user may have multiple devices (phone, tablet, watch). We send to all active device tokens for that user.
  • Store device tokens in a cache (Redis hash: user_id → {platform: [tokens]}) for fast lookups.
  • On APNs 410 (unregistered) or FCM "NotRegistered" response, immediately mark the token as inactive.
  • Periodic cleanup job removes tokens not seen in 90+ days.

Templates decouple notification content from code. Product teams can update copy without deploying code changes. Each template is stored per channel because push notifications have strict character limits while emails can be rich HTML.

Rendering Engine

# Template rendering with variable substitution and validation import re from typing import Dict, Optional class TemplateRenderer: VARIABLE_PATTERN = re.compile(r'\{\{(\w+)\}\}') def __init__(self, template_cache): self.cache = template_cache # Redis or in-memory cache of templates async def render(self, template_name: str, params: Dict[str, str]) -> Dict: template = await self.cache.get(template_name) if not template: raise ValueError(f"Template '{template_name}' not found") # Validate all required variables are provided required = set(template.get('variables', [])) provided = set(params.keys()) missing = required - provided if missing: raise ValueError(f"Missing template variables: {missing}") # Render subject (for email) subject = None if template.get('subject'): subject = self._substitute(template['subject'], params) # Render body body = self._substitute(template['body_template'], params) return { 'subject': subject, 'body': body, 'channel': template['channel'], } def _substitute(self, text: str, params: Dict[str, str]) -> str: def replace(match): key = match.group(1) value = params.get(key, '') return self._sanitize(value, key) return self.VARIABLE_PATTERN.sub(replace, text) def _sanitize(self, value: str, key: str) -> str: """Prevent injection attacks in rendered templates.""" # For URLs, validate format if 'url' in key.lower(): if not value.startswith(('http://', 'https://')): return '' # For HTML templates, escape HTML entities return value.replace('&', '&amp;').replace('<', '&lt;').replace('>', '&gt;')

Personalization Layers

  • User attributes: Name, language, timezone for localized content
  • i18n: Templates versioned per locale (en_US, es_MX, ja_JP). Locale stored in user profile.
  • Dynamic content: Product images, recommendation carousels in email (fetched at render time)
  • Channel adaptation: Same notification renders differently: rich HTML for email, short text for SMS, structured JSON for push
User Preference Management ▼

User preferences are critical for compliance (CAN-SPAM, GDPR) and user experience. The preference system must handle:

  • Global opt-out: User unsubscribes from all non-transactional notifications
  • Per-type opt-out: "I want order updates but not marketing emails"
  • Per-channel opt-out: "Send me push but not SMS for social notifications"
  • Quiet hours: "Don't notify me between 10 PM and 8 AM in my timezone"
  • Frequency caps: "Max 3 marketing notifications per week"

Preference Resolution Logic

# Preference resolution: most specific setting wins class PreferenceResolver: # Notification categories TRANSACTIONAL = {'otp', 'security_alert', 'password_reset', 'order_update'} async def get_allowed_channels(self, user_id: int, notification_type: str, priority: int) -> list: """Return list of channels this notification can be sent through.""" # Priority 1 (urgent/transactional) bypasses all preferences if priority == 1 and notification_type in self.TRANSACTIONAL: return ['push', 'sms', 'email', 'in_app'] prefs = await self.load_preferences(user_id) # Check global opt-out if prefs.get('global_optout', False): if notification_type not in self.TRANSACTIONAL: return [] allowed = [] for channel in ['push', 'email', 'sms', 'in_app']: # Check type+channel specific preference key = f"{notification_type}:{channel}" if key in prefs: if prefs[key]['enabled']: allowed.append(channel) continue # Fall back to channel-level default channel_key = f"default:{channel}" if channel_key in prefs: if prefs[channel_key]['enabled']: allowed.append(channel) continue # Default: enabled allowed.append(channel) # Apply quiet hours filter allowed = await self.filter_quiet_hours(user_id, allowed, prefs) return allowed async def filter_quiet_hours(self, user_id, channels, prefs): """Remove channels blocked by quiet hours. Queue for later instead.""" quiet = prefs.get('quiet_hours') if not quiet or not quiet.get('enabled'): return channels user_tz = quiet.get('timezone', 'UTC') user_now = get_user_local_time(user_tz) start = quiet['start'] # e.g., "22:00" end = quiet['end'] # e.g., "08:00" if is_in_quiet_hours(user_now, start, end): # During quiet hours, only allow in_app (silent) return [c for c in channels if c == 'in_app'] return channels
Compliance Note: Transactional notifications (OTP, order confirmations, security alerts) are legally required to be sent regardless of marketing opt-out preferences. Always separate transactional from promotional in your design. For SMS, you must honor TCPA stop-word requests ("STOP" replies).
Notification Deduplication ▼

Duplicate notifications annoy users and erode trust. Dedup must happen at multiple layers:

Layer Mechanism TTL What It Catches

Handling Bulk Sends (Campaigns)

A marketing campaign targeting 50M users cannot be sent all at once. This would overwhelm providers and violate rate limits. Instead:

  1. Segment expansion: A background job expands the user segment into a list of user IDs.
  2. Batched enqueue: Enqueue in batches of 10,000, with throttling between batches.
  3. Time-zone aware scheduling: Group users by timezone and send at 9 AM local time.
  4. Provider rate limiting: Workers respect per-provider rate limits (e.g., SendGrid 100K emails/sec, Twilio 10K SMS/sec).
  5. Progress tracking: Store batch progress in Redis for real-time monitoring of the campaign.

Email-Specific: Batching and Warm-up

  • IP warm-up: New sending IPs must be gradually ramped (start at 1K/day, double each day over 2 weeks) to build sender reputation.
  • Dedicated IPs: Separate IPs for transactional vs. marketing email to protect transactional deliverability.
  • Bounce handling: Hard bounces (invalid address) must immediately suppress the address. Soft bounces retry 3 times then suppress.
  • Unsubscribe handling: One-click unsubscribe header (List-Unsubscribe) required by Gmail/Yahoo since Feb 2024.

In-App Feed: Read Path Optimization

  • Redis Sorted Set: Per-user sorted set with notification_id as member and timestamp as score. Fast retrieval of latest N notifications.
  • Fan-out on write: When a notification is created, add to the user's sorted set. Trim to keep only the latest 200.
  • Unread count: Maintain a separate Redis counter for unread count, decremented on read. Avoids scanning the entire feed.
  • Pagination: Use cursor-based pagination (ZRANGEBYSCORE) for smooth infinite scroll.

8. Common Follow-up Questions

How would you handle notification ordering? ▼

Partition Kafka topics by user_id to ensure per-user ordering within a partition. This guarantees that a "payment received" notification arrives before a "order shipped" notification for the same user. Across different users, ordering is not required.

For the in-app feed, use the notification's created_at timestamp as the sort key rather than delivery time, so the feed always appears chronologically correct even if delivery was delayed.

How do you handle provider failover? ▼

Maintain multiple providers per channel (e.g., SendGrid + AWS SES for email, Twilio + Vonage for SMS). Use a circuit breaker pattern:

  • If provider A fails 5 consecutive requests within 30 seconds, open the circuit.
  • Route all traffic to provider B.
  • After 60 seconds, send a probe request to provider A (half-open state).
  • If the probe succeeds, close the circuit and resume using A.

For push, APNs and FCM are mandatory (no alternatives for their respective platforms), so retry with backoff is the only option. For web push, you can fall back to in-app notifications if push delivery fails.

How do you send a notification to 100M users? ▼

A broadcast to all users is essentially a large fan-out problem:

  1. Segment query: Run a query against the user table (or a pre-materialized segment) to get all eligible user IDs. Paginate through results using cursor.
  2. Batch producer: A dedicated "campaign worker" reads user IDs in batches of 10K and publishes individual notification messages to Kafka.
  3. Throttling: Insert artificial delays between batches to control the publishing rate (e.g., 500K/minute to complete 100M in ~3.3 hours).
  4. Timezone bucketing: Group users by timezone. Send to each timezone bucket at the optimal local time.
  5. Progress: Track completion (users_processed / total_users) in Redis and expose via admin dashboard.

At 30K notifications/sec peak capacity, sending to 100M users takes approximately 55 minutes if throughput is fully utilized, or can be spread over hours for lower priority campaigns.

How would you implement scheduled notifications? ▼

Two approaches depending on precision requirements:

  • Database polling (simple): Store scheduled_at in the notifications table. A cron job runs every minute, queries for notifications where scheduled_at <= NOW() and status = 'pending', and enqueues them. Works for minute-level precision.
  • Delayed queue (precise): Use Redis sorted sets with scheduled_at as the score. A worker continuously polls ZRANGEBYSCORE for entries whose score <= current_time. Achieves second-level precision.

For time-zone-aware scheduling ("send at 9 AM local time"), pre-compute the UTC send time for each user based on their timezone setting and insert into the delayed queue. Handle DST transitions by recomputing times for affected users during DST change windows.

How do you prevent notification fatigue? ▼

Notification fatigue leads to users disabling notifications or uninstalling the app. Prevention strategies:

  • Per-user rate limits: Max N notifications per channel per hour (as implemented in the rate limiter above).
  • Intelligent batching: Instead of 10 separate "new like" notifications, batch them: "Alice, Bob, and 8 others liked your post."
  • Smart suppression: If the user is currently active in the app, suppress push/email and only show in-app. Detected via a "last_active" heartbeat in Redis.
  • ML-based send time optimization: Learn the best time to send each user notifications based on historical engagement patterns.
  • Notification grouping: On iOS/Android, use thread_id / notification channels to group related notifications rather than flooding the notification shade.
  • Frequency caps per category: Max 1 marketing push per day, max 3 social notifications per hour.
What about multi-region deployment? ▼

For a global user base:

  • Regional notification services: Deploy the Notification Service in each region (US, EU, APAC). Route users to their closest region.
  • Shared Kafka clusters: Use MirrorMaker 2 or Confluent Cluster Linking to replicate across regions for disaster recovery.
  • Global preference store: User preferences replicated across regions with eventual consistency (CRDTs or last-write-wins).
  • Provider proximity: Use regional provider endpoints (e.g., SES in eu-west-1 for EU users) to reduce latency and comply with data residency requirements.
  • GDPR compliance: EU user data must be processed in EU region. Route EU user notifications through EU infrastructure only.

Key Talking Points for the Interview

  1. Queue-per-channel isolation prevents one channel's failure from affecting others.
  2. Deduplication at multiple layers (API, content hash, consumer) ensures exactly-once delivery semantics on top of at-least-once infrastructure.
  3. Priority queues guarantee urgent notifications (OTP, security) are never starved by bulk campaigns.
  4. User preferences and rate limiting are not just features -- they're compliance requirements (CAN-SPAM, GDPR, TCPA).
  5. Provider abstraction with circuit breakers enables failover and prevents vendor lock-in.
  6. Analytics pipeline is essential for measuring notification effectiveness and detecting delivery issues.
