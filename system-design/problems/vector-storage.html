Design: Multi-Tenant Vector Storage

Examples: Pinecone, Weaviate, Qdrant Cloud, Milvus, Amazon OpenSearch Serverless

Difficulty: Hard (Staff-Level) | Time: 45-60 minutes | Why it's asked: Tests knowledge of ANN algorithms, resource isolation, distributed storage, and the intersection of ML infrastructure with traditional systems design.

1. Requirements Clarification

Functional Requirements ▼
  • Upsert vectors: Insert or update vectors with associated metadata (e.g., 1536-d OpenAI embeddings with JSON payload)
  • k-NN search: Given a query vector, return the top-k most similar vectors within a tenant's namespace
  • Metadata filtering: Pre-filter or post-filter results by arbitrary metadata predicates (e.g., category = "tech" AND date > 2024-01-01)
  • Namespace/collection management: Each tenant can create multiple collections with different dimensionality and distance metrics
  • Delete vectors: By ID or by metadata filter
  • Multi-tenancy: Strict data isolation between tenants — Tenant A must never see Tenant B's vectors
  • Batch operations: Bulk upsert (100K+ vectors in a single call) for initial data loading
Non-Functional Requirements ▼
  • Low-latency queries: p50 < 20ms, p99 < 100ms for top-10 search over 10M vectors
  • High availability: 99.95% uptime SLA — RAG pipelines are in the critical path of user-facing LLM applications
  • Scalability: Support 10,000+ tenants, 1B+ total vectors, auto-scale with demand
  • Freshness: Newly upserted vectors should be searchable within seconds (near-real-time indexing)
  • Noisy-neighbor isolation: A single tenant running heavy queries must not degrade performance for others
  • Cost efficiency: Support tenants ranging from 1K to 100M vectors; don't over-provision small tenants
  • Durability: Zero data loss — vectors represent expensive embedding computation
Staff-Level Insight: The core tension in this design is between query performance (which wants purpose-built in-memory indexes) and cost efficiency (which wants dense packing of tenants onto shared infrastructure). A junior candidate treats this as "just a database with vectors" — a staff candidate designs around the unique properties of ANN indexes: they're large, must be memory-resident for performance, and expensive to rebuild.

2. Capacity Estimation

Scale Numbers ▼
Assumptions:
- 10,000 tenants, 1B total vectors
- Average dimension: 1536 (OpenAI text-embedding-3-large)
- Vector stored as float32 → 1536 × 4 bytes = 6 KB per vector
- Average metadata: 500 bytes per vector
- 80/20 distribution: 20% of tenants hold 80% of vectors

Raw Vector Storage:
1B × 6 KB = 6 TB (raw vectors alone)

HNSW Index Overhead:
~1.5× raw vector size (graph edges + metadata) ≈ 9 TB index memory
With M=16, ef_construction=200: each vector stores ~32 edges × 8 bytes = 256 bytes overhead
Total index: 6 TB + 256 GB (edges) ≈ 6.25 TB

Metadata Storage:
1B × 500 bytes = 500 GB

Total Working Set:
~7 TB must be memory-resident for low-latency queries

Query QPS:
- Peak: 50,000 queries/second across all tenants
- Average query scans ~200-500 nodes in HNSW graph
- Single core handles ~1,000 HNSW queries/sec on 1M vectors

Write QPS:
- 10,000 upserts/second steady state
- Burst: 100,000/sec during batch loads

Cluster Size (memory-optimized nodes):
Using r6g.8xlarge (256 GB RAM): 7 TB / 200 GB usable = ~35 nodes
With replication factor 2: ~70 nodes
Why this matters for staff interviews: Interviewers test whether you understand that vector search is fundamentally a memory-bound problem. Unlike traditional databases where data lives on disk with an in-memory index, vector indexes (HNSW) need the full graph in memory. This drives every architectural decision — from sharding to tenant placement to cost optimization.

Segment-Based Architecture

Vectors are organized into segments — immutable files containing vectors, the HNSW graph, and a metadata index. This is similar to LSM-tree segments in databases like Cassandra.

Segment Layout (on disk and in S3): +------------------------------------------------------+ | Segment Header | | - segment_id: uuid | | - tenant_id: string | | - collection_id: string | | - vector_count: uint64 | | - dimension: uint32 | | - metric: enum (cosine|euclidean|dot_product) | | - created_at: timestamp | +------------------------------------------------------+ | Raw Vectors (memory-mapped) | | - Contiguous float32 array: count x dimension | | - Offset table: vector_id -> byte_offset | +------------------------------------------------------+ | HNSW Graph | | - Layer 0: all nodes, M*2 neighbors each | | - Layer 1..L: subset of nodes, M neighbors each | | - Entry point node ID | +------------------------------------------------------+ | Metadata Inverted Index | | - field_name -> { value -> [vector_id_list] } | | - Supports: eq, range, in, prefix | +------------------------------------------------------+ | Bloom Filter (for point lookups by vector ID) | +------------------------------------------------------+ | Footer + Checksum | +------------------------------------------------------+

Write Path

  1. Append to WAL — Durably persist the upsert before acknowledging
  2. Insert into in-memory buffer — A mutable "active segment" (like memtable in LSM)
  3. Background flush — When the buffer reaches threshold (e.g., 50K vectors), seal it as an immutable segment and build HNSW index
  4. Segment compaction — Periodically merge small segments into larger ones (reduces segment count, improves query performance)
  5. Upload to object storage — For durability and recovery

Read Path

  1. Query hits all segments for the tenant's collection (both active buffer and sealed segments)
  2. Metadata filter applied first using inverted index to produce a bitmap of candidate vectors
  3. HNSW search constrained to candidate set — skip nodes not in bitmap during graph traversal
  4. Merge results from all segments using a min-heap, return global top-k

6. Vector Indexing Deep Dive

HNSW (Hierarchical Navigable Small World) ▼

HNSW is the dominant algorithm for production vector search. It builds a multi-layer graph where each layer is a "small world" network with progressively fewer nodes.

flowchart TB subgraph "Layer 2 - Skip layer (few nodes)" L2A((A)) --- L2D((D)) end subgraph "Layer 1 - Middle layer" L1A((A)) --- L1B((B)) L1B --- L1D((D)) L1D --- L1F((F)) L1A --- L1F end subgraph "Layer 0 - Base layer (all nodes)" L0A((A)) --- L0B((B)) L0B --- L0C((C)) L0C --- L0D((D)) L0D --- L0E((E)) L0E --- L0F((F)) L0A --- L0C L0B --- L0E L0D --- L0F L0A --- L0F end L2A -.-> L1A L2D -.-> L1D L1A -.-> L0A L1B -.-> L0B L1D -.-> L0D L1F -.-> L0F

How HNSW Search Works

  1. Start at the entry point on the highest layer
  2. Greedily traverse toward the query vector (each hop moves closer in distance)
  3. When no closer neighbor exists in the current layer, descend to the layer below
  4. At layer 0, perform a more thorough search using ef_search candidates
  5. Return top-k from the candidate list

Critical Parameters

Parameter Effect Typical Value Trade-off
M Max edges per node per layer 16 Higher = better recall, more memory (M×2 at layer 0)
ef_construction Candidate list size during index building 200 Higher = better graph quality, slower build
ef_search Candidate list size during query 100 Higher = better recall, higher latency. Must be ≥ top_k
Memory per vector (HNSW with M=16, d=1536):
Vector: 1536 × 4 = 6,144 bytes
Graph edges: ~32 edges × 8 bytes = 256 bytes
Overhead (node metadata): ~64 bytes
Total: ~6.5 KB per vector

For 10M vectors: ~65 GB of RAM per shard
Alternative: IVF-PQ (for cost optimization) ▼

Inverted File Index + Product Quantization compresses vectors by 10-50x, enabling disk-based search for cost-sensitive workloads.

How It Works

  1. IVF: K-means clusters vectors into nlist partitions (e.g., 1024). At query time, only the nearest nprobe partitions are searched.
  2. PQ: Each 1536-d vector is split into 192 sub-vectors of 8 dimensions. Each sub-vector is quantized to 1 byte (256 centroids). Result: 1536 × 4 bytes → 192 bytes (32x compression).
Metric HNSW IVF-PQ
Memory per 1M vectors (d=1536) ~6.5 GB ~200 MB
Recall@10 99%+ 90-95%
Query latency (10M vectors) 1-5ms 5-20ms
Build time Incremental (online) Requires training (offline)
Best for Latency-critical, <100M vectors Cost-sensitive, >100M vectors
Staff-Level Answer: Use a tiered approach. Hot tenants (paying for performance SLA) get HNSW on memory-optimized instances. Smaller or free-tier tenants get IVF-PQ on storage-optimized instances with NVMe. This is exactly what Pinecone does with "serverless" vs "pod" indexes.
Filtered Search: The Hard Problem ▼

Combining vector similarity with metadata filters is deceptively hard. It's the most common follow-up question in interviews.

Three Strategies

Strategy How It Works Pros Cons
Pre-filter Run metadata filter first, then search only matching vectors Exact filter, no wasted distance computations If filter is very selective (<1% pass), HNSW graph is too sparse to navigate efficiently
Post-filter Run HNSW search, then filter results Full graph quality, simple implementation May return fewer than k results; wastes compute on filtered-out vectors
In-traversal filter Check filter during HNSW traversal; skip non-matching nodes but continue navigation Best of both worlds; always returns k results Complex to implement; may visit many extra nodes
# In-traversal filtered HNSW search (pseudocode) def filtered_hnsw_search(query, k, filter_fn, ef_search=100): """HNSW search that respects metadata filters during traversal.""" candidates = MinHeap() # by distance (closest first) result = MaxHeap(k) # by distance (farthest first, bounded by k) visited = set() entry = graph.entry_point candidates.push(distance(query, entry), entry) while candidates: dist, node = candidates.pop() # Prune: if farthest result is closer than current candidate, stop if result.size == k and dist > result.peek_max(): break if node in visited: continue visited.add(node) # Check filter DURING traversal (the key insight) if filter_fn(node.metadata): result.push(dist, node) # ALWAYS expand neighbors (even if current node doesn't pass filter) # This maintains graph connectivity for neighbor in graph.neighbors(node): if neighbor not in visited: d = distance(query, neighbor) candidates.push(d, neighbor) return result.to_sorted_list()
Production systems use adaptive strategy selection: If the filter selectivity is >20%, use pre-filter. If <1%, use post-filter with over-fetching. For everything in between, use in-traversal filtering. Qdrant and Weaviate both implement this adaptive approach.

Tier 1 — Enterprise ($$$): Dedicated cluster (Model C). Full isolation, custom SLA, VPC peering.

Tenant Placement Algorithm

# Tenant placement: bin-packing with memory as the constraint class TenantScheduler: def place_collection(self, tenant_id, estimated_memory_gb): """Place a new collection onto a shard node using best-fit decreasing.""" nodes = self.get_nodes_sorted_by_available_memory() for node in nodes: available = node.total_memory - node.used_memory # Reserve 15% headroom for spikes and OS overhead if available * 0.85 >= estimated_memory_gb: # Check affinity: prefer nodes already hosting this tenant # (reduces cross-node queries for multi-collection tenants) if node.hosts_tenant(tenant_id): return self.assign(node, tenant_id, estimated_memory_gb) # No existing node has affinity + space; find best-fit for node in nodes: available = node.total_memory - node.used_memory if available * 0.85 >= estimated_memory_gb: return self.assign(node, tenant_id, estimated_memory_gb) # No node has capacity; trigger scale-out new_node = self.provision_node() return self.assign(new_node, tenant_id, estimated_memory_gb) def rebalance(self): """Periodically rebalance to fix memory skew.""" nodes = self.get_all_nodes() avg_utilization = mean(n.memory_utilization for n in nodes) for node in nodes: if node.memory_utilization > avg_utilization * 1.3: # 30% above avg # Find smallest tenant index to migrate victim = node.get_smallest_index() target = self.find_least_loaded_node() self.migrate_index(victim, node, target)

Hot/Cold Index Management

Not all tenant indexes can fit in memory simultaneously. We need a strategy for loading/unloading indexes.

  • Active tenants: Indexes stay memory-resident. Defined by query activity in last N minutes.
  • Idle tenants: Indexes are evicted from memory (but segments remain on S3). First query triggers a load (~2-5 seconds for a 1GB index from SSD, ~10-30 seconds from S3).
  • Serverless model: Charge per-query instead of per-provisioned-capacity. Automatically scale index memory up/down. This is Pinecone's "serverless" offering.
Cold-start latency budget:
- SSD read: 1 GB in ~300ms (NVMe, ~3.5 GB/s sequential)
- S3 read: 1 GB in ~1-2s (multi-part parallel download)
- HNSW deserialization: ~500ms for 1 GB index
Total cold-start: 1-3 seconds (SSD) or 2-5 seconds (S3)

Mitigate with: predictive pre-loading (load at first API call of the day), keep-warm API for premium tenants.

8. Sharding & Partitioning

Sharding Strategy for Large Collections ▼

Scatter-Gather Query Flow

  1. Query Router looks up shard map from etcd: collection "products" → [shard_1@nodeA, shard_2@nodeB, shard_3@nodeC, shard_4@nodeD]
  2. Sends the query vector to all shards in parallel
  3. Each shard runs HNSW search locally and returns its local top-k
  4. Router merges results using a min-heap: 4 shards × k results → global top-k
  5. Latency = max(shard latencies) + merge time (typically <1ms)

Shard Assignment Strategies

Strategy How Best For
Random assignment hash(vector_id) % num_shards Even distribution; works for most cases
Cluster-based K-means on vector space; each cluster = 1 shard Can prune shards at query time (skip distant clusters)
Metadata-based Partition by metadata field (e.g., date range) When queries always filter on partition key
Staff Insight: Random sharding is simpler and usually preferred. Cluster-based sharding sounds appealing (skip irrelevant shards) but in practice, high-dimensional vectors spread across many clusters, limiting pruning benefit. The extra complexity of maintaining cluster centroids and rebalancing rarely pays off. Save cluster-based for very large collections (>500M vectors) where reducing scatter fan-out from 20 shards to 5 materially impacts latency.

9. Noisy Neighbor & Resource Isolation

The Problem and Solutions ▼

When Tenant A runs a burst of heavy queries (large ef_search, complex filters, large top_k), it monopolizes CPU and memory bandwidth, causing latency spikes for Tenant B on the same node. This is the #1 operational challenge in multi-tenant vector databases.

Admission Control: Query Complexity Estimation

Before executing a query, estimate its cost to prevent expensive queries from starving others:

def estimate_query_cost(query, collection_stats): """Estimate CPU cost units for a query.""" base_cost = collection_stats.vector_count / 1_000_000 # scale with size # HNSW search cost scales with ef_search search_cost = base_cost * (query.ef_search / 100) # Filter cost depends on selectivity if query.filter: selectivity = estimate_filter_selectivity(query.filter, collection_stats) if selectivity < 0.01: # Very selective filter = expensive (sparse graph traversal) search_cost *= 3.0 elif selectivity < 0.1: search_cost *= 1.5 return search_cost # Reject if estimated cost exceeds tenant's budget if estimate_query_cost(query, stats) > tenant_quota.max_cost_per_query: raise QueryTooExpensive("Simplify filter or reduce ef_search")

10. Consistency & Freshness

Write-to-Read Visibility ▼

After a vector is upserted, how quickly is it visible in search results? This is uniquely challenging for vector databases because HNSW index updates are expensive.

sequenceDiagram participant Client participant Gateway participant IngestQueue as Kafka participant ShardNode participant WAL participant Buffer as Mutable Buffer participant HNSW as HNSW Index Client->>Gateway: Upsert vector Gateway->>IngestQueue: Enqueue Gateway-->>Client: ACK (202 Accepted) IngestQueue->>ShardNode: Deliver batch ShardNode->>WAL: Append (durable) ShardNode->>Buffer: Insert into mutable buffer Note over Buffer: Searchable immediately via brute-force scan Note over Buffer: Buffer holds up to 50K vectors ShardNode->>HNSW: Background: merge buffer into HNSW index Note over HNSW: ~1-5 seconds for incremental insertion

Two-Phase Searchability

  • Phase 1 (immediate, <1 second): Vector is in the mutable buffer. Search queries scan both the HNSW index AND the buffer. Buffer scan is brute-force but cheap for <50K vectors.
  • Phase 2 (<5 seconds): Background thread adds the vector to the HNSW graph. Now it participates in efficient approximate search.
Staff Insight: The key trade-off is between freshness and index quality. Inserting into HNSW immediately (with the graph's dynamic insert) gives ~95% of the recall of a rebuilt index. For truly latency-sensitive RAG applications, this is acceptable. For recommendation systems that need maximum recall, periodic full rebuilds are worth the cost.
Replication & Durability ▼

Replication Strategy

  • Synchronous WAL replication: Primary writes to WAL, replicates to 1+ followers before ACK. Ensures zero data loss.
  • Asynchronous index replication: HNSW index is not replicated synchronously (too expensive). Instead, replicas independently build indexes from the replicated WAL.
  • Read replicas serve queries: Distribute query load. Replicas may lag by a few seconds in HNSW index freshness — acceptable for most use cases.

Failure Recovery

Failure Recovery Strategy RTO
Shard node crash Promote replica to primary; replay WAL for unindexed entries <30 seconds
Disk failure Rebuild from S3 segments + replay WAL from Kafka Minutes (depends on index size)
AZ failure Replicas in other AZ serve queries; re-provision in surviving AZs <1 minute (for reads)
Corrupt HNSW index Rebuild from raw vectors in S3 + checksum verification Minutes to hours (offline rebuild)

12. Cost Optimization

Reducing Infrastructure Costs at Scale ▼
Technique How Savings Trade-off
Scalar Quantization float32 → int8 (per-dimension min/max scaling) 4x memory reduction <1% recall loss for most embeddings
Binary Quantization float32 → 1-bit (sign of each dimension) 32x memory reduction 5-10% recall loss; use as first-pass with re-ranking
Matryoshka Embeddings Use first N dimensions of embedding (e.g., 256 of 1536) 6x memory reduction 2-5% recall loss; model must support it (OpenAI does)
Disk-based ANN Keep PQ codes in memory, full vectors on NVMe SSD 10x memory reduction 5-10x latency increase; good for cost-tier
Auto-scaling to zero Evict idle tenant indexes; load on-demand Pay only when queried Cold-start penalty (2-5 seconds)
# Scalar quantization: float32 -> int8 with minimal recall loss import numpy as np class ScalarQuantizer: def __init__(self): self.mins = None self.maxs = None def fit(self, vectors: np.ndarray): """Compute per-dimension min/max from training data.""" self.mins = vectors.min(axis=0) self.maxs = vectors.max(axis=0) self.ranges = self.maxs - self.mins self.ranges[self.ranges == 0] = 1.0 # avoid div by zero def encode(self, vectors: np.ndarray) -> np.ndarray: """Quantize float32 vectors to uint8.""" normalized = (vectors - self.mins) / self.ranges return (normalized * 255).clip(0, 255).astype(np.uint8) def decode(self, quantized: np.ndarray) -> np.ndarray: """Reconstruct approximate float32 vectors from uint8.""" return (quantized.astype(np.float32) / 255) * self.ranges + self.mins # Usage: # Original: 1M vectors x 1536 dims x 4 bytes = 6.14 GB # Quantized: 1M vectors x 1536 dims x 1 byte = 1.54 GB (4x savings) # Recall@10 drop: typically < 1% for OpenAI/Cohere embeddings

Staff Insight: The unique metric for vector databases is recall quality. Traditional databases have deterministic correctness — a query either returns the right rows or it doesn't. ANN search is probabilistic — you can return results that are "close enough" but miss the true nearest neighbors. Running periodic canary queries against a brute-force ground truth is essential for detecting index degradation.

Common Interview Follow-Ups ▼
  • "How would you handle a tenant with 500M vectors?" → Shard across 20+ nodes; scatter-gather queries; consider IVF-PQ for memory savings; cluster-based sharding for pruning.
  • "What happens when a new embedding model comes out?" → Collections are tied to a model version. Tenants must create a new collection, re-embed, and backfill. Support shadow indexing to build new index while old one serves traffic.
  • "How do you ensure tenant A can't see tenant B's data?" → Separate HNSW indexes (no shared graph traversal); tenant_id validated at API gateway from auth token; access logs audited; for enterprise: dedicated nodes + network isolation.
  • "How does filtered search interact with HNSW?" → Three strategies (pre/post/in-traversal); adaptive selection based on filter selectivity. This is the hardest part — discuss the sparse graph traversal problem.
  • "How do you do zero-downtime index migration?" → Build new index on shadow shard nodes; when ready, atomically swap shard map in etcd; old nodes drain and decommission. Similar to blue-green deployment.
  • "What about hybrid search (vector + keyword)?" → Maintain a parallel inverted index (BM25) alongside HNSW. Run both searches, combine scores with reciprocal rank fusion (RRF). This is what Weaviate and Vespa do.
