Design: Airbnb Booking System

Examples: Airbnb, Vrbo, Booking.com, Agoda

Difficulty: Hard (Staff-Level) | Time: 45-60 minutes | Why it's asked: Tests two-sided marketplace dynamics, calendar-based locking, geo-spatial search, dynamic pricing, payment orchestration, and the critical tension between consistency (no double-booking) and availability (fast search).

1. Requirements Clarification

Functional Requirements ▼
  • Listing management (hosts): Create/edit listings with photos, amenities, house rules, and per-night pricing calendars
  • Search (guests): Search by location (geo), check-in/check-out dates, guest count, price range, amenities, property type — return ranked results
  • Booking flow: Instant Book (auto-confirm) or Request to Book (host approval within 24h)
  • Availability calendar: Per-day pricing with blocked dates, minimum stay requirements, and seasonal adjustments
  • Payment splitting: Guest pays total → platform holds → host paid out 24h after check-in (minus platform fee)
  • Reviews: Dual-blind review system — both guest and host write reviews, revealed simultaneously after 14 days
  • Cancellation policies: Flexible, Moderate, Strict — each with different refund rules and host payout timelines
  • Messaging: In-app messaging between guest and host pre- and post-booking
Non-Functional Requirements ▼
  • Strong consistency for bookings: No double-booking — if two guests try to book the same listing for overlapping dates, exactly one succeeds
  • Low-latency search: p50 < 200ms, p99 < 800ms for geo+date+filter queries
  • High availability: 99.99% for search; 99.95% for booking (degraded search is better than no search)
  • Global scale: 7M active listings, 150M users, 2M bookings/day peak (summer), 5K search QPS peak
  • Eventual consistency OK for: Search index (stale availability by minutes is acceptable), reviews, analytics
  • Multi-currency: Support 40+ currencies with real-time conversion at booking time
  • Idempotent payments: Network failures must not cause double charges
Staff-Level Insight: The core architectural tension is between search availability (eventual consistency, cache-friendly, read-optimized) and booking correctness (strong consistency, ACID transactions, write-optimized). A two-sided marketplace adds host availability dynamics — calendars change constantly, pricing is dynamic, and supply is heterogeneous — that are absent in single-sided systems like e-commerce.

2. Capacity Estimation

Scale Numbers ▼
Assumptions:
- 7M active listings, 150M registered users
- 2M bookings/day peak → ~23 bookings/sec avg, ~100 TPS peak
- 5,000 search QPS peak (50x more than booking)
- Average listing: 10 photos × 500KB = 5MB

Calendar Data:
7M listings × 365 days × ~100 bytes/day = ~250 GB
(each day row: listing_id, date, price, available, min_stay, booking_id)

Listing Metadata:
7M × 10 KB avg (title, description, amenities, rules, location) = ~70 GB

Photos:
7M listings × 5 MB avg = ~35 TB (S3 + CDN)

Search Index:
7M documents × 5 KB/doc (denormalized) + availability bitmaps = ~50 GB in Elasticsearch

Booking Records:
2M/day × 365 × 1 KB = ~700 GB/year
Staff Insight: The read/write asymmetry is extreme: search is 50,000× more frequent than booking. This drives a fundamental architectural split — optimize search for throughput and cache-friendliness (Elasticsearch + CDN), optimize booking for correctness and consistency (PostgreSQL + row-level locks). Don't try to use a single data store for both.

Database Choice by Component

Store Used For Why This Store
PostgreSQL Bookings, Calendar, Payments, Users ACID transactions, row-level locking for double-booking prevention, strong consistency
Elasticsearch Search (geo + text + filters + availability) geo_distance queries, faceted search, full-text, bitmap filters for availability
Redis Price cache, session, rate limiting, hot listing counts Sub-ms reads; precomputed price lookups during search
S3 + CDN Listing photos, user avatars 35 TB of images; CDN edge caching for global low-latency delivery
Kafka + ClickHouse Event streaming, analytics, CDC pipeline Async propagation from PG → ES; real-time analytics dashboards
Staff Insight: The calendar table with (listing_id, date) as the composite primary key is the heart of the entire system. Every booking operation is ultimately a transactional update to calendar rows. The booking_id column being NOT NULL for booked dates (plus the uniqueness constraint) is the database-level guard against double-booking. This is simpler and more reliable than application-level locks.

6. Deep Dive — Search (Geo + Availability + Pricing)

The Search Challenge ▼

Airbnb search must intersect three dimensions simultaneously: geographic proximity, temporal availability (specific dates), and attribute filters (price, amenities, property type). Doing this efficiently at 5K QPS over 7M listings is the hardest part of the system.

Naive Approach (and why it fails)

Query Elasticsearch for geo + filters, then for each result, query PostgreSQL calendar to check availability. At 5K QPS with 200 results each, that's 1M calendar lookups/second — PostgreSQL collapses.

Recommended: Availability Bitmap Approach

Precompute a 365-day availability bitmap per listing and store it in Elasticsearch alongside listing data. During search, filter by bitmap at query time. Re-check availability in PostgreSQL only at booking time.

sequenceDiagram participant Guest participant API as API Gateway participant Search as Search Service participant ES as Elasticsearch participant Redis participant PG as PostgreSQL Guest->>API: GET /search?lat,lng,dates,filters API->>Search: Forward query Search->>ES: geo_distance + bitmap_check(dates) + filters ES-->>Search: 200 matching listings (pre-filtered by availability) Search->>Redis: Batch fetch prices for date range Redis-->>Search: Per-night prices for top 20 Search-->>API: Ranked results with total prices API-->>Guest: Search results page Note over Guest,PG: Later, at booking time... Guest->>API: POST /bookings {listing_id, dates} API->>PG: SELECT FOR UPDATE calendar WHERE dates AND available=true PG-->>API: Confirmed available (real-time check)

Search Index Update Pipeline (CDC)

flowchart LR PG["PostgreSQL\n(calendar table)"] DEB["Debezium\n(CDC)"] KF["Kafka\n(calendar-changes topic)"] CONS["Bitmap Builder\n(Consumer)"] ES["Elasticsearch\n(update bitmap field)"] PG -->|"WAL stream"| DEB --> KF --> CONS --> ES style PG fill:#336791,color:#fff style ES fill:#f0db4f,color:#333 style KF fill:#231f20,color:#fff

When a host blocks dates or a booking is confirmed, the CDC pipeline captures the calendar change, the consumer recalculates the 365-day bitmap, and updates the ES document. Typical propagation latency: 2-10 seconds.

Staff Insight — "Optimistic Search, Pessimistic Booking": This is the key architectural pattern. Search uses eventually-consistent bitmaps (fast, scalable, but may show a listing as available when it was just booked). Booking uses a real-time PostgreSQL transaction with row-level locking (slow but correct). The worst case is a guest sees a listing in search results that turns out to be unavailable when they try to book — a minor UX annoyance, not a correctness violation.

7. Deep Dive — Double-Booking Prevention

Concurrency Control Strategies ▼

When two guests simultaneously try to book the same listing for overlapping dates, exactly one must succeed. This is the single most important correctness requirement.

Approach Comparison

Approach How It Works Pros Cons
SELECT FOR UPDATE (Recommended) Lock calendar rows for the date range within a transaction Simple, battle-tested, ACID guarantees Holds row locks during payment auth (~500ms)
Optimistic Concurrency (version check) UPDATE ... WHERE available=true AND version=X; check rowcount No blocking; good for low contention Retry storms under high contention
Distributed Lock (Redis Redlock) Acquire lock on listing_id before booking Works for hot listings with extreme contention Extra infrastructure; lock expiry edge cases
Serializable Isolation SET TRANSACTION ISOLATION LEVEL SERIALIZABLE Strongest guarantee High abort rate under contention; expensive

Recommended: Two-Phase Calendar Locking

sequenceDiagram participant G1 as Guest 1 participant G2 as Guest 2 participant BS as Booking Service participant PG as PostgreSQL G1->>BS: Book listing X, Aug 1-5 G2->>BS: Book listing X, Aug 3-7 BS->>PG: BEGIN; SELECT FROM calendar WHERE listing_id=X AND date IN (Aug 1-5) AND available=true FOR UPDATE Note over PG: Rows locked for Guest 1 BS->>PG: BEGIN; SELECT FROM calendar WHERE listing_id=X AND date IN (Aug 3-7) FOR UPDATE Note over PG: Guest 2 BLOCKS on rows Aug 3-5 (held by Guest 1) PG-->>BS: Guest 1: 5 rows returned (all available) BS->>PG: UPDATE calendar SET available=false, booking_id=bk_1 WHERE listing_id=X AND date IN (Aug 1-5) BS->>PG: COMMIT Note over PG: Guest 2 unblocks, reads updated rows PG-->>BS: Guest 2: only 2 of 5 rows available (Aug 3-5 already booked) BS->>PG: ROLLBACK BS-->>G2: 409 Conflict: dates no longer available

Implementation

# booking_service.py - Calendar locking with SELECT FOR UPDATE import asyncpg from datetime import date, timedelta from typing import List from uuid import uuid4 async def create_booking( pool: asyncpg.Pool, listing_id: str, guest_id: str, check_in: date, check_out: date, idempotency_key: str ) -> dict: """Atomically lock calendar dates and create booking.""" # Generate date range num_nights = (check_out - check_in).days dates = [check_in + timedelta(days=i) for i in range(num_nights)] async with pool.acquire() as conn: async with conn.transaction(): # 1. Check idempotency (replay protection) existing = await conn.fetchrow( "SELECT id, status FROM bookings WHERE idempotency_key = $1", idempotency_key ) if existing: return {"booking_id": existing["id"], "status": existing["status"]} # 2. Lock calendar rows (SELECT FOR UPDATE) # ORDER BY date prevents deadlocks between concurrent txns rows = await conn.fetch(""" SELECT date, price, available, booking_id FROM calendar WHERE listing_id = $1 AND date = ANY($2) ORDER BY date FOR UPDATE """, listing_id, dates) # 3. Validate all dates are available if len(rows) != num_nights: raise BookingError("Calendar not configured for all dates") unavailable = [r["date"] for r in rows if not r["available"]] if unavailable: raise BookingError( f"Dates unavailable: {unavailable}" ) # 4. Calculate total price from per-night prices nightly_total = sum(r["price"] for r in rows) service_fee = int(nightly_total * 0.142) # 14.2% guest fee total_price = nightly_total + service_fee # 5. Create booking record booking_id = str(uuid4()) await conn.execute(""" INSERT INTO bookings (id, listing_id, guest_id, check_in, check_out, guests, status, nightly_total, service_fee, total_price, currency, idempotency_key) VALUES ($1,$2,$3,$4,$5,$6,'confirmed',$7,$8,$9,'USD',$10) """, booking_id, listing_id, guest_id, check_in, check_out, 1, nightly_total, service_fee, total_price, idempotency_key) # 6. Mark calendar dates as booked (atomic with booking) await conn.execute(""" UPDATE calendar SET available = false, booking_id = $1 WHERE listing_id = $2 AND date = ANY($3) """, booking_id, listing_id, dates) return { "booking_id": booking_id, "status": "confirmed", "total_price": total_price }
Staff Insight: At Airbnb's scale (~100 booking TPS peak across 7M listings), the average contention per listing is ~0.01 bookings/sec/listing. This is extremely low — database-level row locking (SELECT FOR UPDATE) is more than sufficient. You don't need Redis distributed locks or complex conflict resolution. Always add the unique constraint on (listing_id, date, booking_id) as a safety net — even if the application logic has a bug, the DB won't allow double-booking.

8. Deep Dive — Booking Saga (Orchestration)

Multi-Step Booking Flow ▼

A booking involves multiple services (calendar, payment, notification) that must either all succeed or all roll back. We use an orchestrator-based saga pattern with compensation actions.

sequenceDiagram participant Guest participant BS as Booking Service
(Orchestrator) participant CS as Calendar Service participant PS as Payment Service participant NS as Notification Service participant Host Guest->>BS: POST /bookings BS->>BS: Validate request & idempotency check rect rgb(200, 230, 200) Note over BS,PS: Happy Path BS->>CS: Lock dates (SELECT FOR UPDATE) CS-->>BS: Dates locked OK BS->>PS: Authorize payment hold PS-->>BS: Auth hold OK (hold_id: hld_123) alt Instant Book BS->>BS: Set status = CONFIRMED else Request to Book BS->>BS: Set status = PENDING_HOST BS->>NS: Notify host (24h to respond) Host-->>BS: Host approves BS->>BS: Set status = CONFIRMED end BS->>NS: Confirm booking notification (guest + host) BS-->>Guest: 200 OK {booking_id, status: CONFIRMED} end rect rgb(255, 200, 200) Note over BS,PS: Compensation Path (if payment fails) BS->>CS: Lock dates OK BS->>PS: Authorize payment PS-->>BS: PAYMENT DECLINED BS->>CS: COMPENSATE: Unlock dates (set available=true) BS-->>Guest: 402 Payment Failed end

Compensation Actions

Step Forward Action Compensation Trigger
1 Lock calendar dates Unlock dates (set available=true, booking_id=NULL) Payment failure, host decline, timeout
2 Authorize payment hold Void authorization / release hold Host decline, timeout, guest cancel
3 Host approval (Request to Book) Auto-decline after 24h timeout Timer expiry
4 Send confirmation (no compensation needed — notification is idempotent) N/A

Saga Orchestrator State Machine

# booking_saga.py - State machine for booking orchestration from enum import Enum from dataclasses import dataclass from datetime import datetime, timedelta class SagaState(Enum): INITIATED = "initiated" DATES_LOCKED = "dates_locked" PAYMENT_HELD = "payment_held" PENDING_HOST = "pending_host" CONFIRMED = "confirmed" FAILED = "failed" COMPENSATING = "compensating" @dataclass class BookingSaga: booking_id: str state: SagaState listing_id: str dates: list hold_id: str = None failure_reason: str = None async def execute(self, services): """Drive the saga forward from current state.""" try: if self.state == SagaState.INITIATED: await services.calendar.lock_dates( self.listing_id, self.dates, self.booking_id ) self.state = SagaState.DATES_LOCKED if self.state == SagaState.DATES_LOCKED: self.hold_id = await services.payment.authorize( self.booking_id ) self.state = SagaState.PAYMENT_HELD if self.state == SagaState.PAYMENT_HELD: listing = await services.listing.get(self.listing_id) if listing.instant_book: self.state = SagaState.CONFIRMED else: self.state = SagaState.PENDING_HOST await services.notification.notify_host( self.listing_id, self.booking_id, deadline=datetime.utcnow() + timedelta(hours=24) ) if self.state == SagaState.CONFIRMED: await services.notification.confirm_booking( self.booking_id ) except Exception as e: self.failure_reason = str(e) await self.compensate(services) async def compensate(self, services): """Undo completed steps in reverse order.""" self.state = SagaState.COMPENSATING if self.hold_id: await services.payment.void_hold(self.hold_id) self.hold_id = None await services.calendar.unlock_dates( self.listing_id, self.dates, self.booking_id ) self.state = SagaState.FAILED

Edge Cases

  • Host timeout (24h): A scheduled job checks PENDING_HOST bookings older than 24h. Auto-declines and runs compensation (void payment hold, unlock dates). This penalizes the host's response rate.
  • Payment hold expiry: Stripe auth holds expire after 7 days. For Request-to-Book, if the host takes close to 24h, re-authorize if needed. Track hold expiry timestamps.
  • Saga orchestrator crash: Persist saga state to database. On restart, load incomplete sagas and resume from last completed step. This is why idempotent steps are critical — replaying a completed step must be safe.
  • Partial date lock: If only 3 of 5 dates can be locked (race condition), the transaction rolls back atomically since it's a single PG transaction.

9. Deep Dive — Dynamic Pricing

Pricing Engine Architecture ▼

Airbnb's Smart Pricing suggests prices to hosts based on multiple factors. The pricing engine must serve two use cases: real-time price suggestions (host dashboard) and batch price precomputation (search optimization).

Pricing Factors

  • Seasonality: Historical demand patterns by location and time of year (Christmas in NYC, summer in Barcelona)
  • Local events: Concerts, conferences, sports events drive demand spikes (Super Bowl, CES)
  • Supply/demand balance: Ratio of available listings to search volume for a location+date
  • Booking lead time: Last-minute bookings often command higher prices; far-future bookings lower
  • Listing attributes: Quality signals (reviews, Superhost status, photos quality score)
  • Competitive pricing: Similar listings in the area and their current pricing

Price Cache Design

# Redis key structure for price cache # Key: price:{listing_id}:{date} Value: price_cents # TTL: 24 hours (re-computed daily by batch job) # Batch precomputation (runs daily at 2 AM UTC) # 7M listings x 365 days = 2.555B keys # Each key: ~30 bytes key + 4 bytes value = ~34 bytes # Total: 2.555B * 34 bytes ≈ 87 GB # With Redis hash packing: ~20 GB (hash key = listing_id, field = date) # Hash structure (more memory efficient): # HSET price:lst_abc123 2025-08-01 18000 2025-08-02 18500 ... # Search flow: when returning results, batch-fetch prices # HMGET price:lst_abc123 2025-08-01 2025-08-02 2025-08-03 2025-08-04 # Returns: [18000, 18500, 19000, 18000] (cents per night) # Total for 4 nights: $735.00
Staff Insight: Precomputing prices into Redis means the search path never needs to call the pricing engine or query per-night calendar prices from PostgreSQL. At 5K search QPS with 20 results each, that's 100K Redis HMGET/sec — trivial for a Redis cluster. The daily batch job takes ~30 minutes on a Spark cluster. Hosts who manually set prices override the ML suggestion, and their prices go directly into the cache.

10. Deep Dive — Payment Splitting

Payment Flow & Split Calculation ▼
sequenceDiagram participant Guest participant BS as Booking Service participant PS as Payment Service participant Stripe participant Host Note over Guest,Host: At Booking Time Guest->>BS: POST /bookings BS->>PS: Authorize $920 (total) PS->>Stripe: Create PaymentIntent (auth only) Stripe-->>PS: pi_xxx (authorized, not captured) PS-->>BS: Hold confirmed Note over Guest,Host: At Check-in (T+0) BS->>PS: Capture $920 PS->>Stripe: Capture PaymentIntent pi_xxx Stripe-->>PS: Captured Note over Guest,Host: 24 Hours After Check-in (T+1) PS->>PS: Calculate split Note over PS: Guest paid: $920
Service fee (14.2%): $130.64
Host fee (3%): $23.69
Host payout: $920 - $130.64 - $23.69 = $765.67 PS->>Stripe: Transfer $765.67 to host Connected Account Stripe-->>Host: Payout initiated Note over Guest,Host: On Cancellation (if applicable) Guest->>BS: Cancel booking BS->>PS: Calculate refund per cancellation policy PS->>Stripe: Refund (full/partial based on policy)

Key Payment Design Decisions

  • Auth hold at booking, capture at check-in: Gives a cancellation window without immediately charging. Holds typically expire after 7 days, so for bookings far in advance, charge a deposit and hold the remainder closer to check-in.
  • 24h payout delay: If the guest reports issues on check-in day, the platform has time to intervene before the host is paid.
  • Idempotency keys on every payment call: Network retries must never cause double charges. Map booking_id to Stripe PaymentIntent ID.
  • Multi-currency: Guest is charged in their local currency. Host receives payout in their preferred currency. Platform records the exchange rate at booking time and absorbs FX risk (or passes it to the host).
  • Reconciliation: Daily batch job compares internal payment records with Stripe's records. Flag discrepancies (missed captures, orphaned holds) for manual review.

11. Deep Dive — Trust & Safety

Review System & Fraud Detection ▼

Blind Double-Review System

Both guest and host submit reviews independently. Reviews are hidden (revealed = false) until both submit or 14 days pass after checkout, whichever comes first. This prevents retaliatory reviews.

  • Both submit → both reviews revealed simultaneously
  • Only one submits within 14 days → that review is revealed; the other party's window closes
  • Neither submits → no reviews for this booking

Fraud Signals

  • New account + expensive first booking: High-risk signal for stolen credit cards
  • Host and guest in same geo + short stay: Possible fake booking for review farming

  • Message content analysis: Attempts to move communication off-platform (sharing phone numbers, emails) to avoid platform fees

Consistency Model per Component

Component Strategy Why
Search (ES) Replicate globally; query local cluster Low latency; eventual consistency is fine for search
Booking (PG) Partition by listing region; route booking to listing's home region Strong consistency for booking; avoid cross-region transactions
Price Cache (Redis) Replicate globally (read replicas per region) Prices are batch-computed; staleness by hours is acceptable
Photos (S3+CDN) S3 cross-region replication; CDN serves from edge Immutable content; replicate once, cache forever
Staff Insight — Geographic Affinity: The key insight is that a listing is physically located in one region, and all bookings for that listing should be routed to the PG primary in that region. A guest in the US searching for apartments in Paris will see search results from their local ES cluster (fast), but when they click "Book", the booking request is routed to EU-West where the listing's calendar lives (strong consistency, slightly higher latency from cross-region hop — acceptable since booking is a one-time action, not a repeated query).

14. Failure Scenarios & Mitigations

Complete Failure Matrix ▼
Scenario Impact Mitigation
PostgreSQL primary down No new bookings; search still works (ES unaffected) Automatic failover to synchronous replica (RTO <30s); queue booking requests during failover
Elasticsearch cluster down Search unavailable; existing bookings unaffected Multi-AZ ES deployment; fallback to cached search results or PG geo query (degraded)
Payment provider timeout Booking stuck in "payment pending" state Retry with idempotency key; circuit breaker; dates remain locked with 15-min TTL
CDC pipeline lag (>5 min) Search shows stale availability; guests see "available" but booking fails Monitor Kafka consumer lag; auto-scale consumers; graceful "just booked" message to user
Double charge (network retry) Guest charged twice for same booking Idempotency keys on all payment operations; daily reconciliation job
Booking service crash mid-saga Dates locked but booking incomplete; "ghost lock" Saga state persisted in DB; recovery job resumes or compensates incomplete sagas
Hot listing (celebrity property) Lock contention on calendar rows; booking latency spike Queue bookings per-listing (serialize via Redis lock); return "pending" status immediately
Redis price cache failure Search results missing total prices Fallback to base_price from ES document; show "estimated price" badge

15. Summary & Interview Tips

Architecture Summary

Scale 7M listings, 150M users, 2M bookings/day, 5K search QPS
Search Elasticsearch with availability bitmap + geo_distance + filters
Booking consistency PostgreSQL SELECT FOR UPDATE on calendar rows
Key pattern Optimistic search (eventual) + Pessimistic booking (strong)
Saga orchestration Lock dates → Auth payment → Host approval → Confirm
Pricing ML model + rule engine; precomputed into Redis (~20 GB)
Payment Auth hold at booking, capture at check-in, host payout at T+24h
Multi-region Geographic affinity: booking routes to listing's home region
Common Interview Follow-Ups ▼
  • "How do you prevent double-booking?" → Calendar table with (listing_id, date) PK; SELECT FOR UPDATE in a transaction; unique constraint as safety net. At 0.01 bookings/sec/listing, DB locking is sufficient.
  • "How does search handle availability checking efficiently?" → Precomputed 365-day availability bitmap in ES, updated via CDC pipeline. Check bitmap at query time; re-verify in PG only at booking time.
  • "What happens if the payment provider is down?" → Circuit breaker pattern; dates stay locked with a 15-min TTL; retry with idempotency key; saga compensates if payment ultimately fails.
  • "How do you handle a celebrity listing with thousands of concurrent booking attempts?" → Serialize booking attempts per-listing with a Redis queue; return "pending" immediately; process sequentially. First successful booking wins.
  • "How would you add Experiences (tours, cooking classes)?" → Same calendar model but with capacity (10 spots vs 1 for a house). Change calendar check from "available = true" to "booked_count < capacity". Decrement atomically.
  • "How do you handle timezone issues for check-in/check-out?" → All dates stored as listing-local dates (DATE type, not TIMESTAMP). A listing in Tokyo has its check-in at "2025-08-01" in JST. The guest in NYC sees "Aug 1" regardless of their timezone.
