1. Why Caching Matters

Latency Numbers Every Engineer Should Know

Caching exploits the locality principle — recently or frequently accessed data is likely to be accessed again. By placing data in a faster storage tier, we trade a small amount of memory for massive latency reductions.

A RAM-based cache (Redis/Memcached) serves data in ~100 µs, compared to a database query that may take 5–50 ms (50–500x slower). At 100k QPS, a 90% cache hit rate means only 10k queries reach the database — the difference between a $5,000/month DB bill and a $50,000/month one.

The 80/20 Rule (Pareto Principle)

In most production systems, 20% of the data serves 80% of requests. Twitter reported that their top 150 accounts generate a disproportionate share of timeline reads. Instagram found that photos less than 48 hours old receive the vast majority of views. This skew makes caching extraordinarily effective: you only need to cache the "hot" subset.

When an interviewer asks "should we add a cache?", don't immediately say yes. First ask: What is the read:write ratio? If writes dominate, a cache might hurt performance due to constant invalidation overhead. Show you understand the trade-off.

2. Cache Strategies

Cache-Aside (Lazy Loading)

Cache-Aside is the most widely used caching pattern. The application code directly manages the cache: it checks the cache first, and on a miss, loads from the database and populates the cache. The cache never communicates with the database directly.

How it works

sequenceDiagram participant App as Application participant Cache as Cache (Redis) participant DB as Database App->>Cache: GET user:123 alt Cache HIT Cache-->>App: Return cached data else Cache MISS Cache-->>App: null App->>DB: SELECT * FROM users WHERE id=123 DB-->>App: User data App->>Cache: SET user:123 (TTL 300s) Cache-->>App: OK end

Python Implementation

import redis
import json
from typing import Optional

r = redis.Redis(host='cache.internal', port=6379, decode_responses=True)

def get_user(user_id: int) -> Optional[dict]:
    cache_key = f"user:{user_id}"

    # 1. Check cache first
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)       # Cache HIT

    # 2. Cache MISS — fetch from database
    user = db.query("SELECT * FROM users WHERE id = %s", (user_id,))
    if user is None:
        # Cache the negative result to prevent repeated DB lookups
        r.setex(cache_key, 60, json.dumps(None))
        return None

    # 3. Populate cache with TTL
    r.setex(cache_key, 300, json.dumps(user))
    return user

Pros & Cons

  • Pro: Only requested data is cached (no wasted memory)
  • Pro: Cache failures are non-fatal — reads fall through to DB
  • Pro: Works with any database; cache is a separate layer
  • Con: First request is always a cache miss (cold start)
  • Con: Data can become stale between writes and TTL expiry
  • Con: Application must manage cache logic (read + write paths)
Cache-aside is the default recommendation in interviews unless you have a specific reason for another strategy. Always mention TTL as the primary staleness mitigation, and note that you would also cache negative results (null) to prevent "cache penetration."
Read-Through Cache

In a read-through cache, the cache library/layer itself is responsible for loading data from the database on a miss. The application only ever talks to the cache, never directly to the DB for reads. The cache acts as the primary data interface.

How it differs from Cache-Aside

AspectCache-AsideRead-Through
Who loads data on miss? Application code Cache layer (via a configured loader)
App ↔ DB coupling App reads from both cache and DB App only reads from cache
Implementation Manual in application Cache framework provides the pattern (e.g., Guava LoadingCache)

Java Example (Guava LoadingCache)

LoadingCache<Long, User> userCache = CacheBuilder.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .build(new CacheLoader<Long, User>() {
        @Override
        public User load(Long userId) {
            // This is called automatically on cache miss
            return userRepository.findById(userId);
        }
    });

// Application code — just call get(), never touches DB directly
User user = userCache.get(userId);

Use when: You want to simplify application code and ensure the cache is always the single source of truth for reads. Common with in-process caches (Guava, Caffeine) and some distributed caches (NCache, Oracle Coherence).

Write-Through Cache

Write-through writes data to the cache and the database in the same operation. The write is only acknowledged after both stores are updated, guaranteeing consistency between cache and DB.

sequenceDiagram participant App as Application participant Cache as Cache Layer participant DB as Database App->>Cache: PUT user:123 Cache->>DB: INSERT/UPDATE users SET ... DB-->>Cache: ACK Cache-->>App: ACK (both updated) Note over App,DB: Read always returns fresh data from cache App->>Cache: GET user:123 Cache-->>App: Fresh data (guaranteed consistent)

Trade-offs

  • Pro: Cache and DB are always consistent — no stale reads
  • Pro: Simplifies invalidation — no need for separate cache-busting
  • Con: Higher write latency (must wait for both cache + DB)
  • Con: Writes data to cache even if it is never read (wastes memory)
  • Con: If the cache goes down, writes can fail or be lost

Use when: Consistency is more important than write latency. Commonly paired with read-through for a "cache-as-system-of-record" architecture. DynamoDB Accelerator (DAX) uses write-through internally.

Write-Behind (Write-Back) Cache

Write-behind is the asynchronous cousin of write-through. Writes go to the cache immediately, and the cache asynchronously flushes changes to the database in the background. This dramatically reduces write latency at the cost of durability risk.

sequenceDiagram participant App as Application participant Cache as Cache Layer participant Queue as Async Queue participant DB as Database App->>Cache: PUT user:123 Cache-->>App: ACK (immediate) Cache->>Queue: Enqueue DB write Note over Queue,DB: Background process Queue->>DB: Batch INSERT/UPDATE DB-->>Queue: ACK

Key Design Decisions

  • Batching: Accumulate multiple writes and flush in a single batch (e.g., every 100ms or every 50 writes)
  • Coalescing: If the same key is written multiple times before flush, only the latest value is written to DB
  • Durability: If cache crashes before flush, data is lost. Mitigate with replication or WAL
  • Ordering: Must preserve write order for consistency, especially for keys with dependencies

Use when: Write latency matters more than durability guarantees. Great for analytics counters (page views, likes), event logging, and situations where the DB is a bottleneck. CPU L1/L2 caches use write-back by default for performance.

In an interview, mentioning write-behind shows depth. Discuss the durability trade-off: "If we lose the cache node before the async flush, we lose writes. We can mitigate this with Redis AOF persistence or by replicating to a secondary node before acknowledging."
Refresh-Ahead

Refresh-ahead proactively reloads cache entries before they expire. The cache predicts which entries will be needed soon (based on recent access patterns) and refreshes them in the background, so reads always hit warm cache.

  • Pro: Eliminates cache-miss latency spikes for popular keys
  • Pro: Reduces thundering herd risk since keys never fully expire
  • Con: Wasted work if predictions are wrong (refreshes unused keys)
  • Con: Increased backend load from proactive refreshes

Use when: You have predictable access patterns and need ultra-low latency. CDNs use refresh-ahead for popular assets. Amazon ElastiCache supports refresh-ahead via lazy TTL extensions on access.

3. Cache Invalidation

"There are only two hard things in computer science..."

Phil Karlton's famous quote: "There are only two hard things in Computer Science: cache invalidation and naming things." Cache invalidation is hard because it requires answering: When should cached data be removed or updated? Get it wrong and users see stale data; over-invalidate and you lose caching benefits entirely.

3.1 TTL-Based Expiration

The simplest approach: set a time-to-live on each cache entry. After the TTL expires, the next read triggers a fresh load from the database. 

# Redis: set key with 5-minute TTL
SET user:123 '{"name":"Alice","email":"alice@example.com"}' EX 300

# Check remaining TTL
TTL user:123
# => 287 (seconds remaining)

Choosing TTL values is an art:

  • Too short (5s): High cache miss rate, excessive DB load
  • Too long (24h): Users see very stale data after updates
  • Sweet spot: Depends on data change frequency and staleness tolerance. User profiles: 5–15 min. Product catalog: 1–6 hours. Config data: 30–60s.

3.2 Event-Driven Invalidation (Pub/Sub)

Instead of waiting for TTL to expire, actively invalidate the cache when the underlying data changes. This is typically done via a message queue or pub/sub system. 

# Producer: when user updates their profile
def update_user(user_id, new_data):
    db.execute("UPDATE users SET ... WHERE id = %s", (user_id,))
    # Publish invalidation event
    redis_pubsub.publish("cache-invalidation", json.dumps({
        "key": f"user:{user_id}",
        "action": "delete"
    }))

# Consumer: cache invalidation service
def on_invalidation_message(message):
    data = json.loads(message["data"])
    redis_cache.delete(data["key"])

Why this is better than TTL alone: Invalidation is near-instant (milliseconds) instead of waiting for TTL to expire. Combine both: use event-driven invalidation for correctness and TTL as a safety net in case events are lost.

3.3 Version-Based Invalidation

Embed a version number in the cache key. When data changes, increment the version. Old keys naturally become unreachable and are evicted by the cache's memory management. 

# Version is stored in a fast counter
def get_user_cache_key(user_id):
    version = redis.get(f"user:{user_id}:version") or "1"
    return f"user:{user_id}:v{version}"

def invalidate_user_cache(user_id):
    # Simply increment the version — old key becomes orphaned
    redis.incr(f"user:{user_id}:version")

This approach avoids explicit DELETE operations and race conditions where a stale value is written after a delete. Facebook's Memcache paper uses a variant of this technique called "lease" tokens.

3.4 Why Invalidation Is Hard: Race Conditions

Consider this classic race condition with cache-aside:

sequenceDiagram participant T1 as Thread 1 (Read) participant T2 as Thread 2 (Write) participant Cache participant DB T1->>Cache: GET user:123 (MISS) T1->>DB: SELECT user:123 Note over T1: Gets old data: {name: "Alice"} T2->>DB: UPDATE user:123 SET name="Bob" T2->>Cache: DELETE user:123 Note over T1: Thread 1 resumes with OLD data T1->>Cache: SET user:123 {name: "Alice"} Note over Cache: Cache now has STALE data! Note over Cache: "Alice" instead of "Bob"

This race is why Facebook's Memcache paper introduced lease tokens: when a cache miss occurs, the cache issues a lease (a unique token). If the key is invalidated before the lease holder writes back, the cache rejects the stale write.

Discussing the race condition between read-miss-populate and concurrent writes shows you understand distributed systems deeply. Mention Facebook's lease token solution, and contrast it with simpler approaches like "delete on write + short TTL" that accept brief staleness.

4. The Thundering Herd Problem

Cache Stampede: When Everyone Misses at Once

Imagine a popular cache key (e.g., the home page feed for millions of users) expires. In the instant after expiration, thousands of concurrent requests all see a cache miss and simultaneously hit the database with the same expensive query. This is the thundering herd (or cache stampede) problem.

graph TD subgraph "Normal Operation" A1[Request 1] -->|HIT| C1[Cache] A2[Request 2] -->|HIT| C1 A3[Request 3] -->|HIT| C1 end subgraph "Thundering Herd (Key Expires)" B1[Request 1] -->|MISS| C2[Cache - Empty] B2[Request 2] -->|MISS| C2 B3[Request 3] -->|MISS| C2 B4["... 1000s more"] -->|MISS| C2 C2 -.->|"All hit DB simultaneously"| DB[(Database)] DB -->|Overloaded!| X[Timeout / Crash] end style X fill:#e74c3c,stroke:#c0392b,color:#fff style DB fill:#f39c12,stroke:#e67e22,color:#fff

Solution 1: Cache Stampede Lock (Mutex)

Only one thread is allowed to regenerate the cache entry. All other threads either wait or receive the stale value. 

import time

def get_with_lock(key: str, ttl: int = 300, lock_ttl: int = 10) -> dict:
    # 1. Try cache
    value = redis.get(key)
    if value:
        return json.loads(value)

    # 2. Try to acquire lock
    lock_key = f"lock:{key}"
    acquired = redis.set(lock_key, "1", nx=True, ex=lock_ttl)

    if acquired:
        try:
            # 3. This thread regenerates the cache
            value = expensive_db_query(key)
            redis.setex(key, ttl, json.dumps(value))
            return value
        finally:
            redis.delete(lock_key)
    else:
        # 4. Another thread is regenerating — wait and retry
        time.sleep(0.05)  # 50ms backoff
        return get_with_lock(key, ttl, lock_ttl)  # retry

Solution 2: Probabilistic Early Expiration

Instead of all keys expiring at exactly the same time, each read has a small probability of triggering a background refresh before the TTL expires. The probability increases as the key approaches expiration. 

import random, math

def get_with_early_expiry(key: str, ttl: int = 300, beta: float = 1.0):
    value, expiry = redis.get_with_ttl(key)  # pseudocode

    if value is not None:
        remaining_ttl = expiry - time.time()
        # Probabilistic early refresh: higher chance as TTL approaches 0
        # XFetch algorithm from "Optimal Probabilistic Cache Stampede Prevention"
        if remaining_ttl - beta * math.log(random.random()) > 0:
            return json.loads(value)  # Still valid, return it

    # Refresh (either expired or probabilistic early refresh)
    value = expensive_db_query(key)
    redis.setex(key, ttl, json.dumps(value))
    return value

Solution 3: Stale-While-Revalidate

Serve the stale value immediately while refreshing in the background. This is the same concept as HTTP's stale-while-revalidate directive. 

def get_stale_while_revalidate(key: str, ttl: int = 300, stale_ttl: int = 60):
    value = redis.get(key)
    remaining = redis.ttl(key)

    if value and remaining > 0:
        if remaining < stale_ttl:
            # TTL is low — trigger background refresh
            background_refresh(key, ttl)
        return json.loads(value)  # Return (possibly stale) value immediately

    # Hard miss — must block
    return blocking_refresh(key, ttl)

Facebook's Solution (Memcache Paper)

Facebook's approach combines multiple techniques:

  • Lease tokens: On a cache miss, memcache issues a lease (64-bit token). Only the holder of the lease can SET the value. If a DELETE happens while the lease is outstanding, the lease is invalidated.
  • Stale values: When a key is deleted, the old value is moved to a "stale" store. Clients receiving an invalidated lease can optionally read the stale value instead of hitting the DB.
  • Rate limiting: If a key is deleted frequently (hot key), memcache limits lease issuance to once per 10 seconds, forcing other readers to use stale data.
In an interview, the thundering herd question separates senior from staff engineers. Know at least two solutions and their trade-offs. The lock approach is simple but adds latency for waiting threads. Probabilistic early expiration is elegant but harder to tune. Stale-while- revalidate is the most practical for most systems.

When to Choose Redis

  • You need rich data structures (sorted sets for leaderboards, streams for event sourcing)
  • You need persistence or replication for durability
  • You want pub/sub or Lua scripting for atomic operations
  • You need built-in clustering with automatic failover
  • You are using it as more than just a cache (session store, rate limiter, queue)

When to Choose Memcached

  • Pure caching with simple key-value access patterns
  • Maximum memory efficiency for millions of small keys
  • Multi-threaded performance on multi-core machines
  • You already have a large Memcached deployment (Facebook, Netflix EVCache)
  • You don't need persistence — the cache is purely ephemeral

Redis Cluster divides the keyspace into 16,384 hash slots. Each key is assigned to a slot via CRC16(key) mod 16384. Slots are distributed across primary nodes. Each primary has one or more replicas for failover. If a primary fails, its replica is automatically promoted by the cluster's gossip protocol.

In interviews, don't just say "use Redis." Show you understand the operational implications: single-threaded means one slow Lua script blocks all other commands. Redis Cluster doesn't support multi-key operations across slots unless you use hash tags {user:123}.name. These details signal production experience.

Multi-Tier Caching Architecture

Production systems use multiple cache layers. Each layer trades off latency, capacity, and freshness differently. Understanding this hierarchy is essential for staff-level interviews.

Layer 1: Client-Side / Browser Cache

HTTP cache headers control what the browser stores locally. This is the fastest cache — zero network requests. 

Cache-Control: public, max-age=3600, stale-while-revalidate=60
ETag: "abc123"
Last-Modified: Sat, 26 Apr 2026 10:00:00 GMT
  • Cache-Control: max-age=3600 — Browser caches for 1 hour without checking server
  • stale-while-revalidate=60 — After max-age, serve stale for 60s while revalidating in background
  • ETag — Server-generated hash. Browser sends If-None-Match on revalidation; server returns 304 Not Modified if unchanged
  • Last-Modified — Timestamp-based conditional request via If-Modified-Since

Layer 2: CDN Caching

CDNs cache static and dynamic content at edge locations close to users. Key concepts:

  • Edge caching: Content cached at 200+ PoPs globally. CloudFront has 400+ edge locations.
  • Origin shield: A single intermediate cache between edge and origin. Reduces origin load from N edge servers to 1 shield request.
  • Cache keys: URL + query params + headers (Vary). Incorrect cache keys cause cache pollution or privacy leaks (e.g., caching authenticated responses).
  • Purging: CDN invalidation is eventual (2–30 seconds globally). Design for this latency.

Layer 3: Application-Level (In-Process) Cache

In-memory caches within the application process. Zero network latency, but limited by heap size and not shared across instances.

  • Java: Caffeine (successor to Guava Cache) — near-optimal hit rate via W-TinyLFU
  • Python: functools.lru_cache or cachetools
  • Go: groupcache (distributed in-process cache by Brad Fitzpatrick)

Warning: In-process caches create cache coherency problems across instances. If Server A updates data, Server B's local cache still has the old value. Use short TTLs (5–30s) or pub/sub invalidation.

Layer 4: Distributed Cache (Redis/Memcached)

Shared across all application instances. Single source of truth for cached data. Adds ~0.5ms network latency but provides consistency and large capacity (hundreds of GB).

Layer 5: Database Query Cache

Some databases have built-in query caches (MySQL query cache, PostgreSQL's buffer cache). MySQL deprecated its query cache in 8.0 because invalidation overhead outweighed benefits for write-heavy workloads. PostgreSQL's shared_buffers and OS page cache are more effective.

When designing a system, walk through the caching hierarchy from client to database. Interviewers love seeing you consider CDN caching for static content, application-level caching for computed results, and distributed caching for shared state — each with different TTLs and invalidation strategies.

7. Cache Eviction Policies

Eviction Strategies Compared

When a cache reaches its memory limit, it must decide which entries to remove. The eviction policy directly impacts hit rate and therefore application performance.

Policy Evicts Complexity Hit Rate Best For
LRU (Least Recently Used) Entry not accessed for the longest time O(1) with hashmap + doubly-linked list High for temporal locality General purpose (most common)
LFU (Least Frequently Used) Entry with fewest accesses O(1) with hashmap + frequency buckets Higher than LRU for frequency-skewed workloads Data with stable popularity (CDN content)
FIFO (First In First Out) Oldest entry regardless of access pattern O(1) with queue Lower than LRU Simple implementations, event streams
Random Random entry O(1) Surprisingly competitive in some workloads When simplicity matters more than optimality
TTL Entry closest to expiration O(1) with min-heap Depends on TTL tuning Time-sensitive data (sessions, tokens)
W-TinyLFU Admission filter + LFU segmented O(1) Near-optimal Caffeine cache (Java), state-of-the-art

LRU Implementation: Doubly-Linked List + HashMap

This is a classic interview coding question. The key insight is that a hashmap provides O(1) lookup, and the doubly-linked list provides O(1) move-to-front and eviction from the tail. 

class Node:
    def __init__(self, key: int, value: int):
        self.key = key
        self.value = value
        self.prev = None
        self.next = None


class LRUCache:
    """O(1) get and put operations."""

    def __init__(self, capacity: int):
        self.capacity = capacity
        self.cache = {}  # key -> Node

        # Sentinel nodes to avoid null checks
        self.head = Node(0, 0)  # Most recently used
        self.tail = Node(0, 0)  # Least recently used
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: Node):
        """Remove node from its current position in the list."""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node: Node):
        """Add node right after head (most recently used position)."""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        if key in self.cache:
            node = self.cache[key]
            # Move to front (mark as recently used)
            self._remove(node)
            self._add_to_front(node)
            return node.value
        return -1

    def put(self, key: int, value: int):
        if key in self.cache:
            # Update existing: remove old position
            self._remove(self.cache[key])
            del self.cache[key]

        # Create new node at front
        node = Node(key, value)
        self._add_to_front(node)
        self.cache[key] = node

        # Evict if over capacity
        if len(self.cache) > self.capacity:
            # Remove from tail (least recently used)
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]


# Usage
cache = LRUCache(3)
cache.put(1, "a")
cache.put(2, "b")
cache.put(3, "c")
cache.get(1)        # Returns "a", moves key 1 to front
cache.put(4, "d")   # Evicts key 2 (least recently used)
cache.get(2)        # Returns -1 (evicted)

Redis Eviction Policies

Redis does not implement exact LRU (too memory-intensive to maintain a linked list for every key). Instead it uses approximated LRU: on eviction, Redis samples N random keys (default 5) and evicts the one with the oldest access time. Increasing the sample size improves accuracy but costs more CPU. 

# Redis configuration
maxmemory 4gb
maxmemory-policy allkeys-lru

# Available policies:
# noeviction      — return error on write when memory full
# allkeys-lru     — approximate LRU across all keys
# volatile-lru    — approximate LRU only among keys with TTL set
# allkeys-lfu     — approximate LFU across all keys (Redis 4.0+)
# volatile-lfu    — approximate LFU among keys with TTL
# allkeys-random  — random eviction
# volatile-random — random eviction among keys with TTL
# volatile-ttl    — evict keys closest to expiration
Know that Redis uses approximated LRU, not exact LRU. In Redis 4.0+, LFU (allkeys-lfu) often provides better hit rates for real workloads because it considers frequency, not just recency. Mentioning this distinction shows you have operated Redis at scale.

8. Distributed Caching Challenges

Consistent Hashing, Hot Keys, and Multi-Region

8.1 Consistent Hashing for Cache Distribution

When distributing cache keys across N nodes, naive modulo hashing (hash(key) % N) causes massive redistribution when nodes are added or removed. With 100 nodes and 100M keys, adding one node moves ~99% of keys.

Consistent hashing places both nodes and keys on a virtual ring (0 to 2^32). A key is assigned to the next node clockwise on the ring. Adding or removing a node only affects the keys between it and the previous node on the ring — roughly K/N keys (where K = total keys, N = nodes).

Virtual nodes: To avoid uneven distribution, each physical node is mapped to 100–200 virtual nodes on the ring. This provides much more uniform key distribution. Amazon Dynamo, Cassandra, and the Ketama Memcached client all use this approach.

8.2 Hot Key Problem

A single extremely popular key (e.g., a viral tweet, celebrity profile, flash sale product) can overwhelm a single cache shard, creating a bottleneck even when the rest of the cluster is idle.

Solutions:

  • Local (in-process) cache for hot keys: Keep a small L1 cache in each application server. Popular keys are served from local memory, reducing load on the distributed cache. Use short TTLs (1–5s) to limit staleness.
  • Key splitting / replication: Instead of one key product:123, create N replicas: product:123:r0 through product:123:r7. The client randomly picks a replica for reads, spreading load across 8 cache nodes instead of 1. 
    import random

REPLICA_COUNT = 8

def get_hot_key(key: str):
    # Spread reads across N replicas
    replica = random.randint(0, REPLICA_COUNT - 1)
    return redis.get(f"{key}:r{replica}")

def set_hot_key(key: str, value: str, ttl: int):
    # Write to ALL replicas
    pipe = redis.pipeline()
    for i in range(REPLICA_COUNT):
        pipe.setex(f"{key}:r{i}", ttl, value)
    pipe.execute()
  • Dedicated hot-key tracking: Monitor cache access patterns in real-time. When a key's QPS exceeds a threshold, automatically promote it to the local cache tier. Redis 7.0+ provides CLIENT NO-TOUCH and key access frequency stats for this.

8.3 Cache Coherency in Multi-Region Deployments

When your application runs in multiple regions (e.g., US-East, EU-West, AP-Southeast), each region typically has its own cache cluster. The challenge is keeping caches consistent when a write in one region must be visible in others.

Approaches:

  • Invalidation via cross-region messaging: When a write occurs in US-East, publish an invalidation event to a global message bus (Kafka, SNS). Consumers in other regions delete the key from their local cache. Facebook uses McRouter + McSqueal for this.
  • Remote marker (Facebook's approach): After an update, set a "remote marker" in the local cache that redirects reads to the primary region until the replication lag catches up. This prevents reading stale data from a local replica database after a write.
  • Accept staleness with bounded TTL: For non-critical data, simply accept that cross-region caches may be stale for up to the TTL duration. Simpler to implement and sufficient for many use cases.

8.4 Cache Warming Strategies

When deploying a new cache cluster or after a cache failure, the cache is completely cold. All requests hit the database, potentially causing an outage.

  • Snapshot-based warming: Periodically serialize the cache to disk (Redis RDB). On startup, load the snapshot. Data may be slightly stale but the cache starts warm.
  • Shadow traffic warming: Route a copy of production traffic to the new cache before making it primary. By the time you cut over, the cache is pre-populated.
  • Pre-computed warming: Run a batch job that loads the top-N most popular keys from the database into the cache before launch.
  • Gradual ramp: Slowly shift traffic to the new cache (1% → 10% → 50% → 100%), allowing it to warm up without overwhelming the database.
Cache warming is an often-overlooked topic that signals operational maturity. In an interview, mention that you would warm the cache before cutover and describe your strategy. Also discuss what happens during a cache failure: do you have enough database capacity to handle the full load temporarily? This is called the "cold cache" capacity plan.

9. Real-World Case Studies

Netflix: EVCache

Netflix's EVCache (Ephemeral Volatile Cache) is built on top of Memcached and serves 30+ million requests per second across their global infrastructure.

  • Architecture: Memcached instances in each AWS region, with data replicated across availability zones for durability. Writes go to all zone copies; reads prefer the local zone.
  • Client library: Netflix's EVCache client handles connection pooling, retry logic, zone-aware routing, and circuit breaking.
  • Use cases: User session data, personalization features, video metadata, A/B test assignments.
  • Scale numbers: Thousands of Memcached instances, petabytes of cached data, sub-millisecond p99 latency within a zone.
  • Why Memcached over Redis? Netflix chose Memcached for its simpler memory model, multi-threaded performance, and predictable latency. They don't need Redis's data structures for their caching use case.
Facebook: TAO (The Associations and Objects Cache)

Facebook's TAO is a graph-aware distributed cache that serves the social graph (friends, likes, comments, etc.) at billions of reads per second.

  • Data model: Objects (users, posts, photos) and Associations (edges: friend-of, liked-by, commented-on). Each has a type, ID, and properties.
  • Architecture: Two-tier: a leader cache in one region handles writes and a follower cache in each region handles reads. Followers receive invalidation events from the leader via an async replication stream.
  • Why not just Memcached? The social graph has complex query patterns (e.g., "all friends of user X who liked post Y"). TAO provides a semantic API (assoc_get, assoc_count) that understands these patterns and can batch/optimize queries that Memcached's simple GET/SET cannot.
  • Read-after-write consistency: TAO guarantees that a user who writes data will immediately see their own writes. Cross-user consistency is eventual (typically under 1 second).
  • Scale: Handles more read QPS than the entire Memcached layer. One of the largest caching systems ever built.
Twitter: Timeline Cache Architecture

Twitter uses a fan-out-on-write architecture where each user's home timeline is pre-computed and cached in Redis.

  • Write path: When user A posts a tweet, it is "fanned out" to the timeline cache of every follower. If A has 1M followers, 1M cache entries are updated.
  • Read path: When user B opens their timeline, it is a simple Redis read of their pre-built timeline — no complex query or join needed. Sub-millisecond latency.
  • Hybrid approach for celebrities: Users with millions of followers (e.g., @BarackObama) are too expensive to fan out. Their tweets are merged at read time using a "fan-out-on-read" approach for only these accounts.
  • Cache structure: Each user's timeline is a Redis sorted set (ZSET) scored by timestamp. New tweets are ZADD'd; old tweets fall off when the set exceeds ~800 entries.
  • Scale: 300M+ active users, each with a cached timeline. Petabytes of Redis storage across thousands of shards.
# Simplified Twitter timeline commands
# When user "alice" tweets:
ZADD timeline:bob 1714156800 tweet:98765
ZADD timeline:charlie 1714156800 tweet:98765
# ... repeated for all followers

# When bob opens their timeline:
ZREVRANGE timeline:bob 0 49  # Latest 50 tweets
# => ["tweet:98765", "tweet:98764", ...]
The Twitter timeline example is one of the most commonly asked system design questions. Know the fan-out-on-write vs fan-out-on-read trade-off cold. The hybrid approach (fan-out-on-write for normal users, fan-out-on-read for celebrities) is the key insight interviewers look for.

Key Takeaways

Caching Deep Dive - Summary

  • Cache-aside is the default strategy; use read-through/write-through when the cache framework supports it and consistency matters.
  • Cache invalidation is hard because of race conditions between concurrent reads and writes. Use TTL as a safety net, event-driven invalidation for freshness, and lease tokens for correctness.
  • Thundering herd is solved with stampede locks, probabilistic early expiration, or stale-while-revalidate.
  • Redis for rich data structures and persistence; Memcached for simple, high-throughput, multi-threaded caching.
  • Multi-tier caching (browser → CDN → app cache → distributed cache → DB) provides defense in depth.
  • LRU is the default eviction policy; LFU often performs better for frequency-skewed workloads. Redis uses approximated versions of both.
  • Distributed caching requires consistent hashing, hot key mitigation, and cache warming strategies.

Flashcard Q&A Pairs (for Anki/CSV export)

  1. Q: What is the cache-aside pattern? A: Application checks cache first; on miss, loads from DB and populates cache. Cache never talks to DB directly.
  2. Q: How does write-behind differ from write-through? A: Write-through writes cache + DB synchronously; write-behind writes cache immediately and DB asynchronously in batches.
  3. Q: What is the thundering herd problem? A: When a popular cache key expires, thousands of concurrent requests all miss and hit the DB simultaneously, potentially causing overload.
  4. Q: How does consistent hashing help cache distribution? A: Adding/removing a node only remaps ~K/N keys instead of almost all keys with modulo hashing.
  5. Q: Why does Redis use approximated LRU? A: Exact LRU requires a linked list for every key, which is too memory-intensive. Redis samples N random keys and evicts the oldest.
