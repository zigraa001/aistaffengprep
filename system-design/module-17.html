1. Why Search Is Hard

Full-Text Search vs. Database Queries

Relational databases excel at structured lookups: SELECT * FROM users WHERE id = 42 executes in microseconds using a B-tree index. But try searching for "best pizza near me open late" and you hit a wall: every word must be matched loosely (stemming, synonyms, typo tolerance), relevance must be ranked, and results must appear in <200 ms at enormous scale.

The Scale Challenge

Google processes over 8.5 billion searches per day (2024) against an index of 100+ billion web pages. Each query fans out to thousands of machines, each holding a shard of the index. Results are merged, re-ranked, and returned in under 500 ms including network round-trips. This requires:

  • Inverted indexes that compress billions of posting lists into memory
  • Tiered caching — result cache, posting list cache, filter cache
  • Scatter-gather parallelism across thousands of shards
  • Approximate scoring to avoid scoring every matching document
Staff-Level Signal

In interviews, demonstrate that you understand why search is a separate subsystem and not just a LIKE '%term%' query. A staff engineer chooses Elasticsearch when they need fuzzy matching, relevance ranking, or faceting — not for exact key lookups.

2. The Inverted Index

How It Works: From Documents to Postings

An inverted index is the foundational data structure behind every search engine. Instead of mapping document → words (forward index), it maps word → documents. This lets you answer "which documents contain the word 'distributed'?" in O(1) lookup time rather than scanning every document.

Worked Example

Consider three documents:

Doc 1: "distributed systems are scalable"
Doc 2: "search systems use inverted indexes"
Doc 3: "distributed search at scale"

Step 1 — Tokenization: Split each document into tokens (words), lowercased.

Step 2 — Stop-word removal: Remove common words like "are", "use", "at".

Step 3 — Stemming: Reduce words to root forms: "scalable" → "scalabl", "systems" → "system", "indexes" → "index".

Step 4 — Build the inverted index:

Why Inverted Indexes Are Fast

  • O(1) term lookup via hash map or sorted dictionary
  • Posting list intersection for AND queries: merge two sorted lists in O(n+m)
  • Compression: Delta-encoding + variable-byte encoding shrinks posting lists by 5-10x
  • Skip pointers: Allow jumping ahead in long posting lists during intersection

Key Concepts for Interviews

Tokenization

Breaking text into terms. Choices: whitespace, n-gram, word-piece (BERT-style). Affects recall vs. index size.

Stemming vs. Lemmatization

Stemming (Porter algorithm) is fast but crude: "running" → "run". Lemmatization uses dictionaries for accuracy: "better" → "good".

Stop Words

High-frequency words ("the", "is", "at") that add noise. Modern engines keep them but down-weight using IDF.

Positional Index

Stores word positions to support phrase queries ("distributed systems") and proximity scoring.

3. Elasticsearch / Lucene Deep Dive

Architecture: Nodes, Shards, and Replicas

Lucene Internals: Segments

Each Lucene shard is composed of immutable segments. When documents are indexed:

  1. New documents go into an in-memory buffer
  2. Every refresh_interval (default 1 second), the buffer is flushed to a new segment on disk — making documents searchable
  3. Periodically, segments are merged into larger segments (background thread)
  4. Deleted documents are marked in a .del file and purged during merges

This is why Elasticsearch is called near-real-time: there is a 1-second gap (configurable) between indexing and searchability.

Staff-Level Signal

Explain segment merging trade-offs: fewer segments = faster queries (less merging at read time), but aggressive merging burns I/O. Tuning merge_policy and refresh_interval is a key operational concern at scale.

Mappings, Analyzers, and Query DSL

Mappings

Mappings define the schema: field names, types (text, keyword, integer, geo_point), and per-field analyzers. Key distinction:

  • text — analyzed (tokenized, stemmed). Used for full-text search.

Relevance Scoring: TF-IDF and BM25

Elasticsearch uses BM25 (Best Matching 25) as its default scoring function, which evolved from TF-IDF. The intuition:

TF-IDF Intuition:
score = TF(t, d) × IDF(t)

TF (Term Frequency) = How often term t appears in document d
IDF (Inverse Document Frequency) = log(N / df) — rarer terms score higher

BM25 improves on TF-IDF with two key refinements:

  • Saturation: TF doesn't grow linearly forever. A word appearing 100 times isn't 100x more relevant than once. BM25 uses parameter k1 (default 1.2) to control saturation.
  • Document length normalization: Longer documents naturally have higher TF. Parameter b (default 0.75) penalizes longer documents proportionally.
BM25:
score(q, d) = Σ IDF(t) × [ TF(t,d) × (k1 + 1) ] / [ TF(t,d) + k1 × (1 - b + b × |d| / avgdl) ]
Staff-Level Signal

Being able to explain BM25 intuitively (not just recite the formula) shows deep understanding. Key talking points: "BM25 saturates term frequency so a document mentioning 'java' 50 times doesn't dominate over one mentioning it 5 times. The b parameter normalizes for document length so a 10-page article doesn't always beat a concise one."

4. Search Autocomplete / Typeahead

Trie Data Structure and Architecture

Autocomplete must return suggestions within 100 ms of each keystroke. At Google's scale, that means serving millions of prefix lookups per second with personalized, frequency-weighted results.

Trie (Prefix Tree)

A trie stores strings character by character along edges. To find all completions of a prefix, traverse to the prefix node and enumerate its subtree. Each node can store:

  • A flag indicating "this is a complete word"
  • Frequency / popularity count for ranking
  • Top-k precomputed suggestions at each node (amortized O(1) lookups)

Prefix Matching Algorithm

class TrieNode:
    def __init__(self):
        self.children = {}
        self.is_word = False
        self.frequency = 0
        self.top_k = []   # precomputed top-k suggestions

class AutocompleteTrie:
    def __init__(self):
        self.root = TrieNode()

    def insert(self, word: str, freq: int):
        node = self.root
        for ch in word:
            if ch not in node.children:
                node.children[ch] = TrieNode()
            node = node.children[ch]
        node.is_word = True
        node.frequency = freq

    def search_prefix(self, prefix: str, k: int = 5) -> list:
        """Return top-k suggestions for a prefix."""
        node = self.root
        for ch in prefix:
            if ch not in node.children:
                return []
            node = node.children[ch]

        # If top_k is precomputed, return immediately (O(1))
        if node.top_k:
            return node.top_k[:k]

        # Otherwise DFS to collect all words (expensive, offline only)
        results = []
        self._dfs(node, prefix, results)
        results.sort(key=lambda x: -x[1])
        return [word for word, freq in results[:k]]

    def _dfs(self, node, prefix, results):
        if node.is_word:
            results.append((prefix, node.frequency))
        for ch, child in node.children.items():
            self._dfs(child, prefix + ch, results)

Key Design Decisions

  • Precompute top-k at each trie node: Avoids DFS at query time. Trade-off: more memory, but O(1) lookups.
  • Shard by prefix range: "a-m" on shard 1, "n-z" on shard 2. Each keystroke routes to exactly one shard.
  • Cache aggressively: Popular prefixes ("how", "what", "best") are cached in Redis. Cache hit rate often exceeds 90%.
  • Rebuild periodically: Tries are rebuilt hourly/daily from aggregated query logs rather than updated in real-time.
  • Client-side debouncing: Only send requests after 100-200 ms of no typing to reduce QPS.
Staff-Level Signal

Discuss the freshness vs. cost trade-off: real-time trie updates are complex (concurrent writes to a trie); periodic rebuilds are simpler but lag behind trending queries. A hybrid approach uses a small real-time layer for trending queries merged with a bulk-built trie.

5. Geo-Search / Location-Based Search

Geohash, Quadtree, and Proximity Search

The Problem

"Find all restaurants within 5 km of my location" requires spatial indexing. A naive approach — computing the Haversine distance to every restaurant — is O(n) and impractical at scale (millions of points of interest).

Geohash

A geohash encodes a (latitude, longitude) pair into a short alphanumeric string. The key insight: nearby locations share a common prefix.

  1. Divide the world into a 2D grid
  2. Recursively subdivide each cell, alternating longitude and latitude bits
  3. Encode the bit sequence as base-32

Proximity Search Algorithm

def find_nearby(lat, lng, radius_km, precision=6):
    """Find nearby points using geohash prefix matching."""
    import geohash2

    # 1. Compute geohash of user's location
    center_hash = geohash2.encode(lat, lng, precision)

    # 2. Get neighboring geohash cells (handles edge cases)
    neighbors = geohash2.neighbors(center_hash)
    search_cells = [center_hash] + neighbors  # 9 cells total

    # 3. Query DB for all POIs in these cells
    candidates = []
    for cell in search_cells:
        candidates += db.query(
            "SELECT * FROM pois WHERE geohash LIKE ?",
            (cell + '%',)
        )

    # 4. Post-filter by exact distance (Haversine)
    results = []
    for poi in candidates:
        dist = haversine(lat, lng, poi.lat, poi.lng)
        if dist <= radius_km:
            results.append((poi, dist))

    return sorted(results, key=lambda x: x[1])

Quadtree

A quadtree recursively divides 2D space into four quadrants. Each leaf node contains up to a maximum number of points. It adapts to data density: dense areas (cities) have deep trees; sparse areas (oceans) have shallow trees.

Staff-Level Signal

The geohash boundary problem: two points 1 meter apart can have completely different geohashes if they straddle a cell boundary. Always query neighboring cells (8 neighbors + center = 9 cells) and post-filter with Haversine. Mention this unprompted in interviews.

6. Search Ranking & Relevance

From BM25 to Learning-to-Rank

Raw BM25 scores are a starting point, but modern search systems layer multiple ranking stages to deliver truly relevant results. This is the ranking funnel:

  1. Query Understanding — Before any retrieval happens
  2. Candidate Retrieval — BM25 / embedding ANN to get top ~1000 docs
  3. Scoring / Re-ranking — ML model scores top candidates
  4. Post-processing — Diversity, freshness, business rules

Query Understanding

  • Spell correction: "distribted systems" → "distributed systems" using edit distance, n-gram overlap, or neural models
  • Synonyms: "k8s" → "kubernetes"; "ML" → "machine learning"
  • Query expansion: "python web framework" expands to include "Django", "Flask", "FastAPI"
  • Intent classification: "apple" — fruit or company? Use context, user history, and trending signals.

Learning to Rank (L2R)

L2R treats ranking as a supervised ML problem. Training data comes from click logs, human judgments, or A/B tests.

ApproachHow It WorksExample
Pointwise Predict relevance score for each (query, doc) pair independently Linear regression, Random Forest
Pairwise Predict which of two documents is more relevant RankSVM, LambdaMART
Listwise Optimize the entire ranked list at once ListNet, LambdaRank

Common L2R Features

Text Relevance

BM25 score, TF-IDF, exact match in title, query term coverage

Document Quality

PageRank, domain authority, freshness, content length, spam score

User Engagement

Click-through rate, dwell time, bounce rate, conversion rate

Personalization

User's past searches, browsing history, location, language, device

A/B Testing Search Quality

Search quality is measured through offline and online metrics:

  • Offline: NDCG (Normalized Discounted Cumulative Gain), MAP (Mean Average Precision), MRR (Mean Reciprocal Rank)
  • Online: Click-through rate, zero-result rate, session success rate, time-to-click
  • Interleaving: Show results from two ranking models interleaved in one page; measure which model's results get more clicks (more sensitive than A/B)
Staff-Level Signal

A staff engineer understands that BM25 alone is insufficient for product search. Mentioning the ranking funnel (retrieve with BM25 → re-rank with ML → post-process for diversity) and metrics like NDCG demonstrates mature search thinking.

7. Scaling Search

Sharding, Scatter-Gather, and Index Pipelines

Sharding Strategies

StrategyHowProsCons
Document-based hash(doc_id) % N Even distribution, simple Every query hits all shards
Term-based Each shard owns a range of terms Query hits fewer shards Hot terms cause skew
Tenant-based Each tenant on dedicated shards Isolation, per-tenant tuning Uneven shard sizes
Time-based Index per day/week Easy retention, hot-warm-cold Cross-time queries span indexes

Scatter-Gather Pattern

In document-based sharding (the default for Elasticsearch), every search query is:

  1. Scatter: The coordinating node sends the query to all shards in parallel
  2. Local execution: Each shard returns its local top-k results
  3. Gather: The coordinating node merges results from all shards and returns global top-k

Tail latency is dominated by the slowest shard. Mitigation strategies:

  • Hedged requests: Send to both primary and replica; use whichever responds first
  • Adaptive replica selection: Route to the least-loaded replica
  • Early termination: Stop scoring after reaching a "good enough" threshold

Caching Search Results

Result Cache

Cache entire search result pages. Key = normalized query string. High hit rate for popular queries. Invalidate on index refresh.

Query Cache (Node Level)

Elasticsearch caches filter clause results as bitsets. Reused across queries sharing the same filter. Automatic; no configuration needed.

Field Data Cache

Caches field values for sorting and aggregations. Memory-hungry; prefer doc_values (column-oriented on-disk) in modern ES.

Cache Invalidation

Every index refresh (1s default) invalidates the request cache. For rarely-changing data, increase refresh_interval to improve cache hit rate.

8. Real-World Systems

Google, Uber, and Amazon Search Architectures

Google Web Search

  • Scale: 100+ billion web pages indexed, 8.5B queries/day
  • PageRank: Link analysis algorithm treating the web as a graph. A page's rank is proportional to the rank of pages linking to it. While now just one of hundreds of signals, it was the founding insight.
  • Caffeine (2010): Replaced batch indexing with a continuous pipeline that indexes new content within seconds.
  • BERT/MUM (2019-2021): Transformer models for query understanding, capturing semantic meaning beyond keyword matching.
  • Architecture: Web → Crawl → Parse → Index → Serve. Each stage is a separate distributed system with its own sharding and replication.

Elasticsearch at Uber

  • Scale: 150+ Elasticsearch clusters, some with 1000+ nodes
  • Use cases: Trip search, driver matching, fraud detection, logging
  • Challenge: Multi-tenant clusters with noisy neighbors. Solution: dedicated clusters for critical use cases, resource quotas for shared ones.
  • Operational tooling: Automated shard rebalancing, cluster health monitoring, automated index lifecycle management (ILM).

Amazon Product Search

  • Scale: 350+ million products, billions of queries/month
  • Challenge: Structured (category, price, brand) + unstructured (description, reviews) data in one search
  • Approach:
    • Structured filters (facets) use inverted indexes on keyword fields
    • Text search uses BM25 + semantic embeddings (DSSM - Deep Structured Semantic Model)
    • Re-ranking considers purchase probability, profit margin, Prime eligibility, reviews
    • Personalization: past purchases, browsing history, collaborative filtering
  • A9/Cosmo: Amazon's search engine evolved from A9 to Cosmo, incorporating more ML-driven retrieval and ranking.
Staff-Level Signal

Reference specific real-world architectures to show you've studied production search systems. "At Uber's scale with 150+ ES clusters, operational concerns like noisy neighbors and automated ILM become as important as the search algorithm itself."

System Design Problems Involving Search

Design Twitter Search

Inverted index on tweet text. Real-time indexing (<10s). Sharded by time (recent tweets are hot). Ranking by relevance + recency + engagement.

Design Yelp / Nearby Places

Geohash or quadtree for spatial indexing. Combine location filter with text search. Rank by distance + rating + relevance.

Design Google Autocomplete

Trie with precomputed top-k. Sharded by prefix. Offline rebuild from query logs. Client-side debouncing + caching.

Design E-Commerce Search

Structured + unstructured search. Facets (brand, price, category). Personalized ranking. Spell correction + synonyms.

Design a Log Search System

Time-based indexes (daily rollover). Hot-warm-cold architecture. Retention policies. Querying across time ranges (ELK stack).

Design a Code Search Engine

Trigram index for substring search. Language-aware tokenization. Repository-level ranking. Handle billions of lines (like Sourcegraph/grep.app).

Key Trade-Offs to Discuss

Relevance vs. Latency

More sophisticated ranking (ML re-ranking, semantic search) improves relevance but adds latency. Budget your time: 50 ms for retrieval, 50 ms for re-ranking, 100 ms total.

Freshness vs. Cost

Real-time indexing requires more resources (higher refresh rate, more segment merges). Many use cases tolerate seconds to minutes of lag.

Precision vs. Recall

Strict matching (precision) returns fewer but more accurate results. Loose matching (recall) finds more but includes noise. Tune based on use case: legal search needs precision; e-commerce needs recall.

Consistency vs. Availability

Search indexes are eventually consistent by nature (CDC pipeline lag). Accept stale reads for availability. Never use search as source of truth.

Interview Framework: Designing a Search System

  1. Clarify requirements: What is being searched? How fresh must results be? What ranking signals matter?
  2. Data model: Define the document schema, which fields are searchable (text vs. keyword), facets.
  3. Indexing pipeline: Source of truth → CDC → Kafka → Index writer → Elasticsearch. Never dual-write.
  4. Query path: Query understanding → BM25 retrieval → ML re-ranking → Post-processing.
  5. Scaling: Shard strategy, replica count, caching layers, scatter-gather optimization.
  6. Operational concerns: Monitoring (search latency, zero-result rate), re-indexing strategy, capacity planning.
