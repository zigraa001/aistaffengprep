URL Shortener (TinyURL/Bitly) Easy

1) Key Algorithms & Core Tradeoffs

  • Base62 encoding of auto-increment ID or random hash — simple, collision-free
  • MD5/SHA256 hash + truncation — collision risk requires DB check
  • Tradeoff: Sequential IDs are predictable but fast; random hashes hide info but need collision handling
  • Key insight: 7 chars of Base62 = 62^7 = 3.5 trillion URLs — more than enough

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 100:1 — heavily read-optimized
  • P50/P99 latency: Reads 5ms/50ms (cache hit), Writes 20ms/100ms
  • Optimization: Read path via cache; write path can be async

3) Core Data Stores

  • Redis/Memcached: Hot URL cache — 90%+ hit rate for popular links
  • NoSQL (DynamoDB/Cassandra): URL mappings — high write throughput, key-value access pattern
  • Distributed ID Generator (Snowflake): Unique IDs without coordination overhead

6) Best Reference

  • System Design Primer: URL Shortener — Classic reference with Base62, hashing tradeoffs
  • Instagram Engineering: Sharding & IDs — How Instagram generates unique IDs at scale

4) Failure Stories

What breaks: Cache cluster fails during traffic spike
Who feels pain: All users — redirects slow to 200ms+, DB gets hammered
Mitigation: Cache-aside pattern with TTL; circuit breaker to DB; pre-warm cache from DB on recovery
What breaks: Hash collision on popular URL — two different URLs map to same short code
Who feels pain: Users — clicking short link goes to wrong destination
Mitigation: Collision detection with retry; append random suffix; use DB unique constraint as safety net
What breaks: Distributed ID generator (Snowflake) clock skew — duplicate IDs generated across data centers
Who feels pain: System — silent data corruption, URL overwrites
Mitigation: NTP sync monitoring; include datacenter ID in key; idempotent writes with IF NOT EXISTS

News Feed / Timeline (Facebook/Twitter) Medium

1) Key Algorithms & Core Tradeoffs

  • Fan-out on write: Push posts to follower feeds at write time — fast reads, expensive writes
  • Fan-out on read: Pull from followees at read time — cheap writes, slow reads
  • Hybrid approach: Fan-out on write for normal users; fan-out on read for celebrities (>1M followers)
  • Ranking: ML model scores posts (engagement, recency, affinity) — adds 20-50ms latency

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 1000:1 — users scroll much more than post
  • P50/P99 latency: Feed fetch 100ms/500ms; Post creation 50ms/200ms (async fan-out)
  • Optimization: Pre-computed feeds in cache; ranking can be batched

3) Core Data Stores

  • Redis Sorted Sets: User timelines — O(log N) insert, O(log N + M) range queries by score
  • Graph DB / Social Graph Service: Follower relationships — who follows whom
  • Blob Storage + CDN: Media (images, videos) — posts reference media IDs

6) Best Reference

  • Meta Engineering: News Feed Ranking — ML-powered ranking at 2B+ users scale
  • Meta Engineering: Client-Side Ranking — Hybrid ranking architecture

4) Failure Stories

What breaks: Celebrity posts to 50M followers — write amplification explodes
Who feels pain: System — queue backlog grows, feed freshness degrades for everyone
Mitigation: Detect high-follower accounts; switch to pull model; rate limit fan-out workers
What breaks: Ranking model returns all zero scores after bad ML deployment — feed shows random/stale posts
Who feels pain: All users — engagement drops 40% before oncall notices
Mitigation: Shadow scoring with canary; minimum engagement threshold check; automatic rollback on metric regression
What breaks: Redis sorted set for user timeline exceeds memory limit — OOM kills cascade across cluster
Who feels pain: Users — empty feeds, 500 errors
Mitigation: Cap timeline size (e.g., 800 posts); TTL on cold timelines; memory alerts at 80%

Chat System (WhatsApp/Messenger) Medium

1) Key Algorithms & Core Tradeoffs

  • WebSocket connections: Persistent bi-directional — low latency, high server memory
  • Message ordering: Lamport timestamps or vector clocks — eventual consistency vs strict ordering
  • End-to-end encryption: Signal protocol — security vs features (search, server-side backup)
  • Group chat: Fan-out to members — complexity scales with group size

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 1:1 — balanced, real-time bidirectional
  • P50/P99 latency: Message delivery 50ms/200ms (same region); 150ms/500ms (cross-region)
  • Optimization: WebSocket keeps connection warm; messages queued for offline users

3) Core Data Stores

  • Cassandra/ScyllaDB: Message history — time-series data, partition by (chat_id, time_bucket)
  • Redis: Online presence + connection mapping — which server holds user's WebSocket
  • Message Queue (Kafka): Async delivery to offline users — durability guarantee

6) Best Reference

  • Erlang Solutions: WhatsApp Interview — Why Erlang powers 100B+ messages/day
  • Discord Engineering: Scaling Elixir — 12M+ concurrent users, WebSocket architecture

4) Failure Stories

What breaks: Chat server crashes — 100K WebSocket connections drop
Who feels pain: Users — messages appear undelivered, reconnection storm
Mitigation: Sticky sessions with failover; exponential backoff on reconnect; message queue buffers during outage
What breaks: Message ordering diverges between sender and recipient — group chat shows messages in wrong order
Who feels pain: Users — confusing conversations, replies appear before the message they reference
Mitigation: Server-assigned sequence numbers per chat; client-side reordering buffer; conflict resolution UI
What breaks: End-to-end encryption key rotation fails — new device can't decrypt old messages
Who feels pain: User — message history appears blank on new phone
Mitigation: Key escrow with user-held recovery code; signal protocol session renegotiation; "request key from other device" flow

Video Streaming (YouTube/Netflix) Hard

1) Key Algorithms & Core Tradeoffs

  • Adaptive Bitrate (HLS/DASH): Client switches quality based on bandwidth — smooth playback vs quality
  • Transcoding pipeline: Convert to multiple resolutions/codecs — CPU cost vs storage cost
  • CDN edge caching: Popular content at edge — latency vs cache cost vs origin load
  • Recommendation ML: Personalized content — engagement vs diversity vs filter bubbles

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 10000:1 — video uploads rare, views constant
  • P50/P99 latency: Video start 500ms/2s; Chunk fetch 50ms/200ms from CDN
  • Optimization: Pre-buffer next chunks; CDN handles 95%+ of traffic

3) Core Data Stores

  • Object Storage (S3): Video files — petabytes of content, lifecycle policies for cold storage
  • CDN (CloudFront/Akamai): Edge delivery — reduce origin bandwidth by 95%
  • Metadata DB (MySQL + Redis): Video info, user data — relational for complex queries

6) Best Reference

  • Netflix Tech Blog: Behind the Streams — Live streaming architecture and transcoding
  • Netflix Open Connect — How Netflix's CDN delivers 95%+ of traffic from edge

4) Failure Stories

What breaks: Origin storage region goes down — CDN cache expires
Who feels pain: Users globally — buffering, failed playback for cache-missed content
Mitigation: Multi-region origin replication; extend CDN TTLs during incidents; serve stale content
What breaks: Transcoding pipeline backlog — new uploads stuck in queue for hours during peak
Who feels pain: Creators — video appears as "processing" indefinitely, miss viral window
Mitigation: Priority lanes for popular creators; progressive quality (480p fast, 4K async); auto-scale transcoding fleet
What breaks: Adaptive bitrate manifest stale — client requests segment that was purged from CDN
Who feels pain: Users — playback stutters with 404 errors mid-stream
Mitigation: Versioned manifests; segment TTL > manifest TTL; graceful fallback to lower quality

Distributed Cache (Redis/Memcached) Medium

1) Key Algorithms & Core Tradeoffs

  • Consistent hashing: Minimize rebalancing on node add/remove — complexity vs even distribution
  • LRU eviction: Remove least recently used — memory efficiency vs hit rate
  • Cache-aside vs write-through: Consistency vs latency vs complexity
  • Replication: Async replicas for reads — availability vs consistency

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 100:1 — caches are read-optimized by design
  • P50/P99 latency: Reads 0.5ms/2ms; Writes 1ms/5ms
  • Optimization: Local L1 cache → distributed L2 cache → database fallback

3) Core Data Stores

  • Redis Cluster: Primary cache — hash slots for sharding, pub/sub for invalidation
  • Local in-process cache (Caffeine): Hot keys — avoid network round-trip
  • ZooKeeper/etcd: Cluster membership — track which nodes are alive

6) Best Reference

  • Meta Engineering: Scaling Memcache — Caching at Facebook's scale with consistency
  • Redis Official: Scaling Guide — Consistent hashing, replication, cluster mode

4) Failure Stories

What breaks: Hot key (viral post) — single shard overwhelmed
Who feels pain: All users accessing that key — timeouts, cascade to DB
Mitigation: Key replication across shards; local caching; request coalescing for same key
What breaks: Thundering herd after cache invalidation — thousands of requests hit DB simultaneously for same key
Who feels pain: Database — connection pool exhausted, cascading failures
Mitigation: Request coalescing (singleflight); stale-while-revalidate; probabilistic early expiration (cache stampede protection)
What breaks: Split-brain during network partition — two Redis masters accept conflicting writes
Who feels pain: Users — inconsistent data, last-write-wins causes silent data loss
Mitigation: Redis Sentinel with quorum; fencing tokens; reconciliation on partition heal

Rate Limiter Easy

1) Key Algorithms & Core Tradeoffs

  • Token bucket: Smooth rate + burst allowance — flexible but memory per user
  • Sliding window log: Exact counting — accurate but high memory
  • Sliding window counter: Weighted average of windows — memory-efficient approximation
  • Fixed window: Simple counters — edge case: 2x burst at window boundary

2) Read vs Write Pattern & Latency

  • Read:Write ratio: N/A — every request is both a read and increment
  • P50/P99 latency: 0.1ms/1ms — must be negligible overhead
  • Optimization: In-memory with async persistence; fail open vs fail closed tradeoff

3) Core Data Stores

  • Redis: Counters with TTL — INCR + EXPIRE atomic ops
  • Local memory: Per-instance rate limiting — no network latency
  • Config store (etcd): Rate limit rules — which endpoints, what limits

6) Best Reference

  • Cloudflare: Rate Limiting at Scale — Sliding window algorithm for millions of domains
  • Stripe Engineering: Rate Limiters — Token bucket implementation for API protection

4) Failure Stories

What breaks: Redis cluster fails — rate limiter loses state
Who feels pain: Ops — either over-limit legit users (fail closed) or let attacks through (fail open)
Mitigation: Local fallback with conservative limits; distributed rate limiting with sync intervals
What breaks: Fixed window boundary — attacker sends 100 requests at 11:59:59 and 100 more at 12:00:00, effectively 200/min
Who feels pain: Backend services — receive 2x expected load in a 1-second window
Mitigation: Sliding window counter; token bucket with gradual refill; combine global + per-window limits

Notification System (Push/Email/SMS) Medium

1) Key Algorithms & Core Tradeoffs

  • Priority queues: Urgent vs batch notifications — latency vs throughput
  • Deduplication: Idempotency keys prevent double-sends — complexity vs user trust
  • Rate limiting per user: Prevent notification spam — engagement vs annoyance
  • Template rendering: Pre-compiled vs dynamic — flexibility vs performance

2) Read vs Write Pattern & Latency

  • Read:Write ratio: N/A — write-only from system perspective
  • P50/P99 latency: Push 100ms/500ms; Email/SMS 1s/10s (third-party dependent)
  • Optimization: Batch non-urgent; priority lane for transactional (2FA, receipts)

3) Core Data Stores

  • Kafka: Event ingestion — handles burst of notification triggers
  • Redis: User preferences + device tokens — fast lookup per notification
  • PostgreSQL: Notification history + analytics — queryable delivery status

6) Best Reference

  • Meta Engineering: Push Notifications — Scaling push to billions of devices
  • Firebase Cloud Messaging Docs — FCM architecture and delivery options

4) Failure Stories

What breaks: APNs/FCM rate limit hit — push notifications queue up
Who feels pain: Users — delayed or missing notifications for time-sensitive events
Mitigation: Exponential backoff; fallback to email/SMS; surface delivery failures in app
What breaks: Deduplication service down — user receives same notification 5 times
Who feels pain: Users — annoyed, lose trust, disable notifications entirely
Mitigation: Client-side dedup by notification ID; idempotency at every layer; circuit breaker on dedup service with "skip" vs "block" policy
What breaks: Template rendering bug sends PII (account balance) to wrong user via email
Who feels pain: Company — GDPR violation, security incident, user trust destroyed
Mitigation: Template sandbox testing; PII access audit log; per-user template variable binding validation; staged rollout for template changes

Search Autocomplete (Typeahead) Medium

1) Key Algorithms & Core Tradeoffs

  • Trie: Prefix tree for suggestions — fast lookup, high memory
  • Inverted index: Prefix → documents — good for phrase completion
  • Frequency-weighted ranking: Popular queries first — relevance vs recency vs personalization
  • Debouncing: Wait 100-200ms between keystrokes — reduce backend load vs perceived latency

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 10000:1 — writes are aggregated search logs
  • P50/P99 latency: 10ms/50ms — must feel instant (under 100ms perceived)
  • Optimization: Edge cache popular prefixes; pre-compute top-K suggestions

3) Core Data Stores

  • Redis: Sorted sets for top suggestions per prefix — ZRANGE for top-K
  • Elasticsearch: Completion suggester — supports fuzzy matching, multi-field
  • Kafka + Flink: Aggregating search logs — build frequency counts offline

6) Best Reference

  • LinkedIn Engineering: Search Suggestions — Typeahead with Galene and trie structures
  • Elasticsearch: Completion Suggester — Production-ready autocomplete implementation

4) Failure Stories

What breaks: Trending topic causes cache miss storm for new prefix
Who feels pain: Users — slow suggestions during breaking news/viral moments
Mitigation: Request coalescing; real-time trending detection; fallback to cached stale data
What breaks: Offensive suggestion surfaces for common prefix — PR nightmare
Who feels pain: Company — brand damage, user complaints, media coverage
Mitigation: Blocklist of offensive terms; human review for top-K suggestions; real-time content moderation layer

Web Crawler (Googlebot) Hard

1) Key Algorithms & Core Tradeoffs

  • BFS vs DFS traversal: BFS for breadth coverage; DFS for depth — memory vs completeness
  • URL frontier prioritization: PageRank, freshness, domain importance — quality vs coverage
  • Politeness: Respect robots.txt, rate limit per domain — thoroughness vs being blocked
  • Deduplication: URL normalization + content fingerprinting — avoid redundant crawls

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 1:1 — fetch page (read) then store (write)
  • P50/P99 latency: Page fetch 200ms/2s (network bound); storage 10ms/100ms
  • Optimization: Async I/O; parallel fetches; DNS caching

3) Core Data Stores

  • Distributed Queue (Kafka): URL frontier — partitioned by domain for politeness
  • Bloom Filter: Seen URLs — memory-efficient deduplication (false positives OK)
  • Object Storage (S3): Raw HTML content — cheap, durable, append-only

6) Best Reference

  • Twitter Engineering: SpiderDuck — Real-time URL fetching with rate limiting per domain
  • Google Research: Web Search Architecture — Crawling and indexing at Google scale

4) Failure Stories

What breaks: Spider trap — infinite URL generation (calendars, session IDs)
Who feels pain: Crawler — wastes resources on one domain, starves others
Mitigation: Max depth per domain; URL pattern detection; domain-level crawl budgets
What breaks: DNS resolution bottleneck — crawler stalls waiting for DNS for millions of unique domains
Who feels pain: Crawl freshness — pages not re-crawled within SLA, stale index
Mitigation: Local DNS cache (TTL-aware); pre-resolve batch of URLs; multiple DNS resolvers; async DNS lookups

Cloud Storage (Dropbox/Google Drive) Hard

1) Key Algorithms & Core Tradeoffs

  • Block-level deduplication: Store unique 4MB blocks — storage cost vs upload latency
  • Delta sync: rsync-style — bandwidth vs CPU for diff computation
  • Content-addressable storage: Hash as key — natural deduplication, immutable
  • Conflict resolution: Last-write-wins vs branching — simplicity vs data safety

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 3:1 — more downloads/syncs than uploads
  • P50/P99 latency: Metadata ops 50ms/200ms; file transfer bandwidth-bound
  • Optimization: Local cache; sync queue for batch uploads; resume interrupted transfers

3) Core Data Stores

  • Object Storage (S3): File blocks — 11 9's durability, cross-region replication
  • Metadata DB (MySQL/Spanner): File tree, permissions, versions — ACID for consistency
  • Notification Service: Real-time sync events — push changes to other devices

6) Best Reference

  • Dropbox Tech: Magic Pocket — Exabyte-scale storage with block deduplication
  • Dropbox Tech: Sync Engine — Delta sync and conflict resolution at scale

4) Failure Stories

What breaks: Metadata DB leader failover during bulk upload
Who feels pain: Users — uploads stuck, file tree inconsistent across devices
Mitigation: Client-side journaling; idempotent uploads with block hashes; reconciliation on reconnect
What breaks: Conflict resolution picks wrong version — user's 3-hour work overwritten by stale version from another device
Who feels pain: User — lost work, may not notice until much later
Mitigation: Never auto-delete versions; show conflict UI; keep both versions with "conflicted copy" naming (like Dropbox); version history with 30-day retention

Ride Sharing (Uber/Lyft) Hard

1) Key Algorithms & Core Tradeoffs

  • Geospatial indexing: Quadtree/Geohash for nearby drivers — precision vs query speed
  • Dispatch optimization: Hungarian algorithm / ML matching — ETA accuracy vs fairness
  • Surge pricing: Dynamic pricing based on supply/demand — revenue vs user trust
  • ETA prediction: ML on historical + real-time traffic — accuracy vs computation cost

2) Read vs Write Pattern & Latency

  • Read:Write ratio: 1:1 — constant location updates (writes) and nearby queries (reads)
  • P50/P99 latency: Match 500ms/2s; Location update 50ms/200ms
  • Optimization: Batch location updates; pre-compute nearby candidates; geofenced caching

3) Core Data Stores

  • Redis with geospatial: Real-time driver locations — GEOADD/GEOSEARCH (GEORADIUS deprecated in 6.2)
  • Kafka: Location event stream — replay for analytics, ML training
  • PostgreSQL + PostGIS: Trip history, user data — spatial queries for analytics

6) Best Reference

  • Uber Engineering: H3 Geospatial Index — Hexagonal hierarchical spatial index for matching
  • Uber Engineering: Dispatch System — Real-time matching optimization (DISCO)

4) Failure Stories

What breaks: Location service lag during concert end — 50K people request rides simultaneously
Who feels pain: Riders — long match times, phantom drivers, ETA jumps
Mitigation: Pre-position drivers for events; queue with ETA; degrade to larger radius search
What breaks: Surge pricing algorithm feedback loop — high price reduces demand, price drops, demand spikes, price re-surges
Who feels pain: Riders — unpredictable pricing, frustration, switch to competitor
Mitigation: Surge price dampening (max change per interval); commitment pricing ("lock this price for 5 min"); transparent surge communication

  • P50/P99 latency: Queries 5ms/50ms (HNSW, 10M vectors); Upserts async via Kafka
