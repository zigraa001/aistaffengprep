REST Deep Dive

Resource Design Patterns ▼

Resource Naming Conventions

  • Use nouns, not verbs: /users not /getUsers
  • Use plural form: /users not /user
  • Use hierarchical structure: /users/{id}/posts
  • Use lowercase with hyphens: /user-profiles not /userProfiles

HTTP Methods

GET /users - List all users (collection)
GET /users/{id} - Get single user (resource)
POST /users - Create new user
PUT /users/{id} - Replace user (full update)
PATCH /users/{id} - Partial update user
DELETE /users/{id} - Delete user

Nested Resources & Complex Relationships

Nested resources express ownership. The rule of thumb: nest no more than 2 levels deep. Beyond that, promote the sub-resource to a top-level resource with filters.

# Good: clear ownership
GET /users/{userId}/orders # Orders belonging to a user
GET /users/{userId}/orders/{orderId} # Specific order for a user
POST /users/{userId}/orders # Create order for user

# Avoid: too deeply nested (3+ levels)
GET /users/{userId}/orders/{orderId}/items/{itemId}/reviews

# Better: promote with query filter
GET /reviews?orderId={orderId}&itemId={itemId}

# Relationships that are not ownership:
# Use sub-resource pattern for actions
POST /orders/{orderId}/cancel
POST /users/{userId}/deactivate

Filtering, Searching, and Sorting

# Filtering - use query parameters
GET /products?category=electronics&min_price=100&max_price=500&in_stock=true

# Sorting - support multiple fields, prefix with - for descending
GET /products?sort=-created_at,price

# Full-text search - use a dedicated query parameter
GET /products?q=wireless+headphones&category=electronics

# Field selection (sparse fieldsets) - reduce payload size
GET /users/123?fields=id,name,email

# Embedding related resources (avoid N+1 on client side)
GET /orders/456?include=items,shipping_address,customer

HATEOAS & Richardson Maturity Model ▼

The Richardson Maturity Model describes four levels of REST maturity. Most production APIs sit at Level 2; true RESTful APIs reach Level 3 with HATEOAS.

Level Name Description Example
0Swamp of POXSingle endpoint, RPC-stylePOST /api with action in body
1ResourcesIndividual URIs per resource/users/123, /orders/456
2HTTP VerbsProper use of GET, POST, PUT, DELETEGET /users, POST /users
3HATEOASHypermedia links in responses guide navigationLinks to next actions in response

HATEOAS Example

HATEOAS (Hypermedia as the Engine of Application State) means the API response tells the client what it can do next, via links. This decouples client logic from hardcoded URLs.

GET /orders/12345

{
  "id": 12345,
  "status": "pending_payment",
  "total": 89.99,
  "_links": {
    "self": { "href": "/orders/12345" },
    "pay": { "href": "/orders/12345/pay", "method": "POST" },
    "cancel": { "href": "/orders/12345/cancel", "method": "POST" },
    "customer":{ "href": "/users/789" }
  }
}

# After payment, the response changes - "pay" link disappears:
{
  "id": 12345,
  "status": "paid",
  "_links": {
    "self": { "href": "/orders/12345" },
    "refund": { "href": "/orders/12345/refund", "method": "POST" },
    "receipt": { "href": "/orders/12345/receipt" }
  }
}
Why This Matters in Interviews: HATEOAS is rarely implemented fully in practice (even APIs claiming to be "RESTful" usually stop at Level 2). But understanding it demonstrates architectural depth. In an interview, you can say: "We could add hypermedia links for discoverability, but for our use case, Level 2 with good documentation is sufficient." This shows you know the ideal but make pragmatic trade-offs.
Bulk Operations & Conditional Requests ▼

Bulk Operations Pattern

When clients need to create, update, or delete many resources at once, making individual HTTP requests is wasteful. Bulk operations solve this but introduce complexity around partial failures.

# Bulk create - POST to collection with array
POST /users/bulk
Content-Type: application/json

{
  "operations": [
    { "method": "create", "body": { "name": "Alice", "email": "alice@co.com" } },
    { "method": "create", "body": { "name": "Bob", "email": "bob@co.com" } },
    { "method": "update", "id": "user_789", "body": { "name": "Charlie Updated" } }
  ]
}

# Response: 207 Multi-Status (partial success)
{
  "results": [
    { "status": 201, "id": "user_101", "success": true },
    { "status": 409, "error": "Email already exists", "success": false },
    { "status": 200, "id": "user_789", "success": true }
  ],
  "summary": { "total": 3, "succeeded": 2, "failed": 1 }
}

Why 207 Multi-Status? Because some operations may succeed and others fail. Returning a flat 200 or 400 would be misleading. Google's Gmail API and Salesforce's Composite API use this pattern.

Conditional Requests (ETags)

Conditional requests prevent lost updates (two clients overwriting each other) and save bandwidth (skip transfer if unchanged). They use ETags as version fingerprints.

# Step 1: Client fetches the resource and receives an ETag
GET /users/123

HTTP/1.1 200 OK
ETag: "a1b2c3d4"
{
  "id": 123,
  "name": "Alice",
  "email": "alice@example.com"
}

# Step 2a: Optimistic concurrency - update only if unchanged
PUT /users/123
If-Match: "a1b2c3d4"
{ "name": "Alice Smith", "email": "alice@example.com" }

# If ETag matches current version: 200 OK (update applied)
# If someone else changed it: 412 Precondition Failed (retry needed)

# Step 2b: Cache validation - fetch only if changed
GET /users/123
If-None-Match: "a1b2c3d4"

# If unchanged: 304 Not Modified (no body, saves bandwidth)
# If changed: 200 OK with new ETag and full body
Staff-Level Insight: ETags and conditional requests are how systems like GitHub handle concurrent edits to the same pull request, and how CDNs decide whether to serve cached content. In an interview, mentioning ETags for optimistic concurrency control shows you understand distributed consistency without heavy locking.
API Versioning Strategies ▼

Versioning Approaches

1. URL Path Versioning (Most common - used by Stripe, Google, Twitter)

GET /api/v1/users
GET /api/v2/users

Pros: Highly visible, easy to route, cacheable. Cons: Can lead to URL proliferation; harder to deprecate gradually.

2. Header-based Versioning (Used by GitHub v3)

GET /api/users
Accept: application/vnd.myapp.v2+json

Pros: Clean URLs, supports content negotiation. Cons: Harder to test in browser, less discoverable.

3. Query Parameter Versioning

GET /api/users?version=2

Pros: Simple, optional default. Cons: Can interfere with caching, pollutes query string.

4. No Versioning - Additive Changes Only (GraphQL's approach)

Instead of versioning, you only make additive changes: add new fields, deprecate old ones with a timeline. This works when you have a strongly-typed schema that can evolve.

Recommendation: Use URL path versioning for public APIs (most visible and cacheable). For internal APIs, consider additive-only evolution with feature flags. Stripe's approach is excellent: they version by date (/v1/charges with a Stripe-Version: 2023-10-16 header) allowing gradual migration.
Error Response Design & Idempotency ▼

Consistent Error Format

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Request validation failed",
    "details": [
      { "field": "email", "message": "Invalid email format", "code": "INVALID_FORMAT" },
      { "field": "age", "message": "Must be between 0 and 150", "code": "OUT_OF_RANGE" }
    ],
    "request_id": "req_abc123",
    "documentation_url": "https://api.example.com/docs/errors#VALIDATION_ERROR"
  }
}

Error Design Best Practices

  • Use consistent structure across all endpoints
  • Include machine-readable error codes (for programmatic handling)
  • Include human-readable messages (for debugging)
  • Include a request ID for distributed tracing / support tickets
  • Link to documentation when helpful
  • Never expose internal stack traces in production
  • Return a Retry-After header for 429 and 503 responses

Idempotency in API Design

An operation is idempotent if performing it multiple times produces the same result as performing it once. This is critical for reliability when clients retry requests due to network failures.

Method Idempotent? Safe? Notes
GETYesYesRead-only, cacheable
PUTYesNoFull replacement is naturally idempotent
DELETEYesNoDeleting twice = same end state
POSTNoNoNeeds idempotency key for safe retries
PATCHNo*NoDepends on operation (set vs. increment)

Idempotency Keys for POST Requests

# Python: Implementing idempotency key handling
import hashlib, json, redis

class IdempotencyMiddleware:
    def __init__(self, redis_client: redis.Redis):
        self.redis = redis_client
        self.TTL = 86400 # 24 hours

    def process_request(self, request):
        key = request.headers.get('Idempotency-Key')
        if not key:
            return None # No idempotency key, proceed normally

        cache_key = f"idempotency:{request.path}:{key}"
        cached = self.redis.get(cache_key)

        if cached:
            # Return the stored response (replay)
            return json.loads(cached)

        # Use SETNX to prevent race condition between concurrent retries
        lock_key = f"idempotency_lock:{cache_key}"
        if not self.redis.set(lock_key, "1", nx=True, ex=30):
            return {"status": 409, "error": "Request in progress"}

        return None # Proceed with request

    def store_response(self, request, response):
        key = request.headers.get('Idempotency-Key')
        if key:
            cache_key = f"idempotency:{request.path}:{key}"
            self.redis.setex(cache_key, self.TTL, json.dumps(response))
Staff-Level Insight: Stripe, PayPal, and Square all require idempotency keys for payment creation. In interviews, always discuss idempotency for write endpoints -- it shows you understand real-world failure modes like network retries, at-least-once delivery, and the impossibility of exactly-once semantics in distributed systems.

GraphQL Deep Dive

Schema, Queries & Mutations ▼

GraphQL inverts the API contract: instead of the server defining which fields each endpoint returns, the client specifies exactly what it needs. This is powerful for mobile apps (limited bandwidth) and dashboards (varied data needs), but introduces complexity on the server side.

Key insight: The same backend serves both queries. With REST, you would need either two endpoints or a bloated universal response. This is exactly why Shopify migrated their Storefront API to GraphQL -- they had hundreds of different merchant apps each needing different subsets of product data.

The N+1 Problem & DataLoader ▼

The N+1 problem is GraphQL's most notorious performance pitfall. When resolving a list of N items where each item requires a separate database query for a related field, you end up with 1 query for the list + N queries for the relations.

The Problem

# This innocent-looking query triggers N+1:
query {
  posts(first: 50) { # 1 query: SELECT * FROM posts LIMIT 50
    title
    author { # 50 queries: SELECT * FROM users WHERE id = ?
      name
    }
  }
}
# Total: 51 database queries!

The Solution: DataLoader (Batching + Caching)

# Python example using aiodataloader
from aiodataloader import DataLoader

class UserLoader(DataLoader):
    async def batch_load_fn(self, user_ids):
        """
        Called once per "tick" with ALL requested IDs.
        Instead of 50 queries, we make 1:
        SELECT * FROM users WHERE id IN (1, 2, 3, ..., 50)
        """
        users = await db.fetch_all(
            "SELECT * FROM users WHERE id = ANY($1)", user_ids
        )
        # IMPORTANT: Return in the same order as requested IDs
        user_map = {u.id: u for u in users}
        return [user_map.get(uid) for uid in user_ids]

# In resolver:
async def resolve_author(post, info):
    # DataLoader batches these across the entire query
    return await info.context.user_loader.load(post.author_id)

# Result: 51 queries collapsed to 2 queries!
# 1: SELECT * FROM posts LIMIT 50
# 2: SELECT * FROM users WHERE id IN (1, 2, 3, ..., 50)

How DataLoader works under the hood:

  1. Resolvers call loader.load(id) which returns a Promise/Future
  2. DataLoader queues all IDs requested in the current execution "tick"
  3. At the end of the tick, it calls batch_load_fn with ALL queued IDs
  4. Results are cached for the duration of the request (per-request cache prevents stale reads)
Real-World Example: Facebook invented DataLoader to solve exactly this problem in their GraphQL API. GitHub's GraphQL API (v4) also uses DataLoader internally. The pattern is now standard in every GraphQL server framework: Apollo Server (JS), Strawberry (Python), gqlgen (Go), and Sangria (Scala).
GraphQL Subscriptions & When to Choose GraphQL ▼

Subscriptions for Real-Time Data

GraphQL Subscriptions provide a declarative way to push real-time updates to clients, typically over WebSocket connections.

# Client subscribes to new comments on a post
subscription OnNewComment($postId: ID!) {
  commentAdded(postId: $postId) {
    id
    body
    author { name avatar }
    createdAt
  }
}

// JavaScript client (Apollo)
const { data, loading } = useSubscription(ON_NEW_COMMENT, {
  variables: { postId: '123' },
  onData: ({ data }) => {
    // Automatically updates the Apollo cache
    console.log('New comment:', data.commentAdded);
  }
});

When to Choose GraphQL

Choose GraphQL When Avoid GraphQL When
Multiple client types (iOS, Android, web, TV) need different data shapesSimple CRUD with uniform clients
Complex, deeply nested data graphs (social networks, e-commerce)File upload / download heavy workflows
Rapid frontend iteration (no backend changes needed for new views)HTTP caching is critical (CDN-heavy workloads)

Real-World GraphQL Deployments

  • GitHub API v4: Migrated from REST v3 to GraphQL because integrators needed wildly different subsets of repository data. A single REST endpoint could not serve both a CI tool (needs commit status) and a project board (needs issue labels) efficiently.
  • Shopify Storefront API: Thousands of merchant storefronts, each with different product display requirements. GraphQL lets each storefront fetch only the fields it renders.
  • Airbnb: Uses GraphQL federation to unify data from hundreds of microservices behind a single graph, enabling the mobile team to build new features without coordinating backend API changes.
Security Consideration: GraphQL's flexibility is also a risk. Without safeguards, a malicious client can craft deeply nested queries that consume excessive resources. Always implement: (1) query depth limiting, (2) query complexity scoring, (3) rate limiting per client, and (4) persistent/approved queries in production. GitHub limits query depth to 10 levels and complexity to 500 points.

gRPC Deep Dive

Protobuf Schema & Streaming Patterns ▼

gRPC uses Protocol Buffers (Protobuf) as its interface definition language and serialization format. Protobuf messages are ~10x smaller than JSON and ~5-10x faster to serialize/deserialize, making gRPC ideal for high-throughput internal communication.

Protobuf Schema Example

// order_service.proto
syntax = "proto3";

package ecommerce;

import "google/protobuf/timestamp.proto";

service OrderService {
  // Unary RPC: single request, single response
  rpc GetOrder (GetOrderRequest) returns (Order);
  rpc CreateOrder (CreateOrderRequest) returns (Order);

  // Server streaming: client sends one request, server streams responses
  rpc WatchOrderStatus (GetOrderRequest) returns (stream OrderStatusUpdate);

  // Client streaming: client streams requests, server sends one response
  rpc UploadOrderItems (stream OrderItem) returns (UploadSummary);

  // Bidirectional streaming: both sides stream simultaneously
  rpc OrderChat (stream ChatMessage) returns (stream ChatMessage);
}

message GetOrderRequest {
  string order_id = 1;
}

message CreateOrderRequest {
  string customer_id = 1;
  repeated OrderItem items = 2;
  Address shipping_address = 3;
}

message Order {
  string id = 1;
  string customer_id = 2;
  repeated OrderItem items = 3;
  OrderStatus status = 4;
  int64 total_cents = 5; // Store money as cents to avoid float issues
  google.protobuf.Timestamp created_at = 6;
}

message OrderItem {
  string product_id = 1;
  int32 quantity = 2;
  int64 price_cents = 3;
}

message Address {
  string street = 1;
  string city = 2;
  string state = 3;
  string zip = 4;
  string country = 5;
}

enum OrderStatus {
  ORDER_STATUS_UNSPECIFIED = 0;
  ORDER_STATUS_PENDING = 1;
  ORDER_STATUS_CONFIRMED = 2;
  ORDER_STATUS_SHIPPED = 3;
  ORDER_STATUS_DELIVERED = 4;
  ORDER_STATUS_CANCELLED = 5;
}

message OrderStatusUpdate {
  OrderStatus status = 1;
  string message = 2;
  google.protobuf.Timestamp timestamp = 3;
}

Python gRPC Client Example

# Generated from protobuf: order_service_pb2, order_service_pb2_grpc
import grpc
import order_service_pb2 as pb2
import order_service_pb2_grpc as pb2_grpc

# Unary call
channel = grpc.insecure_channel('localhost:50051')
stub = pb2_grpc.OrderServiceStub(channel)
order = stub.GetOrder(pb2.GetOrderRequest(order_id="ord_123"))
print(f"Order status: {order.status}")

# Server streaming - watch order status in real time
for update in stub.WatchOrderStatus(pb2.GetOrderRequest(order_id="ord_123")):
    print(f"Status changed to: {update.status} at {update.timestamp}")

# With deadlines (timeouts) - critical for production
try:
    order = stub.GetOrder(
        pb2.GetOrderRequest(order_id="ord_123"),
        timeout=5.0 # 5 second deadline
    )
except grpc.RpcError as e:
    if e.code() == grpc.StatusCode.DEADLINE_EXCEEDED:
        print("Request timed out")
    elif e.code() == grpc.StatusCode.UNAVAILABLE:
        print("Service unavailable")

gRPC in Production: Service Mesh Integration

gRPC works exceptionally well with service meshes (Istio, Linkerd) because HTTP/2 supports multiplexing and proper load balancing. However, there is a subtlety:

  • Connection-level load balancing (L4) does not work well with gRPC because a single HTTP/2 connection multiplexes many streams. A traditional TCP load balancer sends all traffic to one backend.
  • Request-level load balancing (L7) is required. Service meshes solve this by acting as L7 proxies (Envoy sidecar) that inspect HTTP/2 frames and distribute individual RPCs across backends.
  • Health checking: gRPC has a standard health checking protocol (grpc.health.v1.Health) that load balancers can use.

When to Use gRPC

  • Internal microservice communication -- Google uses gRPC for virtually all internal services
  • Low-latency, high-throughput -- Protobuf is 5-10x faster than JSON parsing
  • Polyglot environments -- protoc generates type-safe clients for 10+ languages from one .proto file
  • Streaming use cases -- real-time data feeds, chat, telemetry
  • Netflix uses gRPC for inter-service communication across their ~1000 microservices
  • Google Cloud APIs offer gRPC as the primary interface (REST is auto-generated via transcoding)
Interview Tip: When discussing gRPC in an interview, always mention: (1) the L7 load balancing requirement, (2) the lack of native browser support (need gRPC-Web), and (3) the debugging trade-off -- binary Protobuf is not human-readable like JSON, making curl-based debugging impossible. These nuances show real production experience.

DNS in System Design

  • GeoDNS: Returns different IPs based on client location (Cloudflare, Route53 geolocation routing). Used to route users to the nearest data center.
  • DNS-based load balancing: Return multiple A records; clients pick one (round-robin). Simple but coarse-grained -- no health checking built in.
  • Failover: Health-checked DNS records automatically remove unhealthy endpoints. AWS Route53 failover routing can switch to a backup region in ~60 seconds.
  • TTL trade-offs: Low TTL (30s) = fast failover but more DNS queries and higher latency. High TTL (3600s) = fewer queries but slow failover. Many CDNs use TTL of 60-300 seconds as a balance.
  • DNS prefetching: Browsers resolve DNS for linked domains ahead of time (<link rel="dns-prefetch" href="//api.example.com">).
HTTP/2, HTTP/3 (QUIC), and Protocol Evolution ▼

HTTP/2 Key Features

HTTP/2 was designed to solve the performance limitations of HTTP/1.1, which required opening multiple TCP connections for parallelism (browsers limited to ~6 per domain).

HTTP/3 and QUIC

HTTP/3 replaces TCP with QUIC (a UDP-based transport protocol originally developed by Google). The key motivation: TCP head-of-line blocking.

  • The problem with HTTP/2 over TCP: If a single TCP packet is lost, ALL streams on that connection stall until retransmission completes. This is because TCP guarantees in-order delivery at the transport layer.
  • QUIC solves this: Each stream has independent loss recovery. A lost packet on stream 1 does not block stream 2. This reduces tail latency significantly on lossy networks (mobile, WiFi).
  • 0-RTT connection establishment: QUIC combines the TLS handshake with the transport handshake, achieving 1-RTT for new connections and 0-RTT for resumption. TCP+TLS requires 2-3 round trips.
  • Connection migration: QUIC identifies connections by a connection ID, not by IP:port tuple. This means a mobile device switching from WiFi to cellular does not need to re-establish the connection.
Real-World Impact: Google reports that QUIC reduces search latency by 8% on desktop and 3.6% on mobile. YouTube re-buffering decreased by 15-18% with QUIC. As of 2024, HTTP/3 serves ~30% of global web traffic (Cloudflare, Google, Meta all default to it).

CDN and Edge Computing

CDNs cache and serve content from edge locations close to users, reducing latency. In system design, CDNs are relevant for:

  • Static content: Images, CSS, JS files -- standard CDN use case. Akamai serves ~30% of global web traffic.
  • API responses: Cacheable GET responses can be served from CDN edge (with proper Cache-Control headers and cache invalidation).
  • Edge compute: Cloudflare Workers, AWS Lambda@Edge, Vercel Edge Functions run code at CDN nodes. Used for: A/B testing, authentication, request routing, personalization, geolocation-based content.
  • Cache invalidation: This is the hard part. Strategies include: TTL-based expiry, purge APIs (Cloudflare purge), stale-while-revalidate (serve stale, fetch fresh in background), and cache tags (invalidate all resources with a given tag).

Webhooks invert the API relationship: instead of the client polling the server for changes, the server pushes notifications to a client-specified URL when events occur. They are the standard pattern for integrations (Stripe payment events, GitHub repo events, Shopify order events).

Retry Strategy with Exponential Backoff

Webhook endpoints fail (downtime, network issues, rate limits). A robust retry strategy is essential.

# Retry schedule (exponential backoff with jitter)
# Attempt 1: Immediate
# Attempt 2: ~1 minute
# Attempt 3: ~5 minutes
# Attempt 4: ~30 minutes
# Attempt 5: ~2 hours
# Attempt 6: ~8 hours
# Attempt 7: ~24 hours
# After 7 failures: move to Dead Letter Queue (DLQ)

import random, math

def calculate_retry_delay(attempt: int, base_seconds: int = 60) -> int:
    """Exponential backoff with full jitter (AWS-recommended)"""
    max_delay = base_seconds * (2 ** attempt)
    return random.randint(0, min(max_delay, 86400)) # Cap at 24h

# Success = 2xx response within 30 seconds
# Failure = 4xx (except 410), 5xx, timeout, connection error
# 410 Gone = endpoint deliberately unsubscribed; stop retrying

Signature Verification

Webhook consumers MUST verify that the payload was sent by the expected source, not an attacker. The standard pattern uses HMAC-SHA256 signatures.

# Python: Verifying a Stripe-style webhook signature
import hmac, hashlib, time

def verify_webhook_signature(payload: bytes, signature_header: str, secret: str) -> bool:
    """
    Signature header format: t=timestamp,v1=signature
    """
    parts = dict(p.split('=', 1) for p in signature_header.split(','))
    timestamp = parts['t']
    provided_sig = parts['v1']

    # Prevent replay attacks: reject if timestamp is too old
    if abs(time.time() - int(timestamp)) > 300: # 5 minute tolerance
        return False

    # Compute expected signature
    signed_payload = f"{timestamp}.{payload.decode()}"
    expected_sig = hmac.new(
        secret.encode(), signed_payload.encode(), hashlib.sha256
    ).hexdigest()

    # Constant-time comparison to prevent timing attacks
    return hmac.compare_digest(expected_sig, provided_sig)

Dead Letter Queue (DLQ) Pattern

  • After exhausting all retry attempts, the event goes to a DLQ
  • DLQ events can be inspected, replayed manually, or used for alerting
  • Provide a dashboard for consumers to view failed deliveries and replay them
  • Stripe, GitHub, and Shopify all offer webhook event replay in their dashboards

Best Practices

  • Idempotent consumers: Webhooks may be delivered more than once. Use the event ID to deduplicate.
  • Respond quickly: Return 200 immediately, then process asynchronously. Long-running handlers cause timeouts and retries.
  • Include event type: Let consumers filter by event type without parsing the full payload.
  • Event ordering: Don't guarantee order. Include timestamps and sequence numbers so consumers can reorder if needed.
  • Fan-out: Support multiple webhook URLs per event type (Stripe allows up to 16 endpoints).

API Gateway Patterns

Gateway Architecture, BFF, & Composition ▼

An API Gateway is a reverse proxy that sits between clients and backend services. It handles cross-cutting concerns so individual services do not have to. Think of it as the "front door" of your microservice architecture.

Core Gateway Responsibilities

  • Request routing: Route /users/* to User Service, /orders/* to Order Service
  • Authentication & Authorization: Validate JWT tokens, API keys, OAuth scopes before requests reach backend services
  • Rate limiting: Per-client, per-endpoint, or per-tier limits with token bucket or sliding window algorithms
  • Request/Response transformation: Rename fields, aggregate responses, convert between protocols (REST to gRPC)
  • Observability: Centralized logging, distributed tracing (inject trace IDs), metrics collection
  • Circuit breaking: Stop sending requests to unhealthy backends, return fallback responses
  • TLS termination: Handle HTTPS at the edge, internal traffic can be plain HTTP (in a trusted network)

BFF (Backend for Frontend) Pattern

The BFF pattern creates a dedicated API gateway per client type. Instead of one generic gateway serving web, mobile, and TV, each client gets a tailored backend that assembles exactly the data shape it needs.

# Without BFF: One generic API, clients do the aggregation
# Mobile app makes 3 separate API calls to render a product page:
GET /api/products/123 -> Product details
GET /api/products/123/reviews -> Reviews
GET /api/inventory/123 -> Stock info
# Problem: 3 round trips on a slow mobile network

# With BFF: Mobile-specific backend does the aggregation
# Mobile app makes 1 call:
GET /mobile-api/product-page/123
# Response includes exactly the fields the mobile UI needs:
{
  "name": "Widget",
  "price": "$9.99",
  "thumbnail": "https://cdn.../sm/widget.jpg", // Small image for mobile
  "rating": 4.5,
  "reviewCount": 128,
  "inStock": true
}

# Web BFF returns different data:
GET /web-api/product-page/123
{
  "name": "Widget",
  "price": "$9.99",
  "heroImage": "https://cdn.../lg/widget.jpg", // Large image for web
  "description": "Full description...",
  "rating": 4.5,
  "topReviews": [...], // First 5 reviews inline
  "relatedProducts": [...],
  "inStock": true,
  "deliveryEstimate": "Mar 15-17"
}

When to use BFF:

  • Multiple client types with significantly different data needs
  • Different teams own different clients (mobile team, web team)
  • Need to optimize for client-specific constraints (mobile bandwidth, TV input limitations)

Trade-off: More BFFs means more code to maintain. Netflix, SoundCloud, and Airbnb use BFF. An alternative is GraphQL, which achieves similar flexibility without separate backends.

API Composition / Aggregation

When a single client request requires data from multiple microservices, the gateway (or a dedicated aggregation service) fans out requests and assembles the response.

# API Composition: Gateway aggregates from 3 services

# Client request:
GET /api/dashboard

# Gateway internally calls (in parallel):
# GET user-service:8080/users/current (50ms)
# GET order-service:8080/orders?user=123&limit=5 (80ms)
# GET notification-service:8080/unread?user=123 (30ms)

# Total latency: ~80ms (max of parallel calls) + gateway overhead
# vs. 50+80+30 = 160ms if client called sequentially

# Assembled response:
{
  "user": { "name": "Alice", "plan": "pro" },
  "recentOrders": [...],
  "unreadNotifications": 7
}

Rate Limiting at the Gateway

# Token Bucket Algorithm (most common for API rate limiting)
# - Bucket holds N tokens, refills at R tokens/second
# - Each request consumes 1 token
# - If bucket empty, request is rejected (429)

# Redis-based sliding window rate limiter (Python)
import time, redis

def is_rate_limited(redis_client, client_id, limit=100, window=60):
    """Sliding window counter: 100 requests per 60 seconds"""
    now = time.time()
    key = f"rate_limit:{client_id}"
    pipe = redis_client.pipeline()
    pipe.zremrangebyscore(key, 0, now - window) # Remove old entries
    pipe.zadd(key, {str(now): now}) # Add current request
    pipe.zcard(key) # Count requests in window
    pipe.expire(key, window) # TTL for cleanup
    _, _, count, _ = pipe.execute()
    return count > limit

# Response headers for rate-limited APIs:
# X-RateLimit-Limit: 100
# X-RateLimit-Remaining: 23
# X-RateLimit-Reset: 1614556860 (Unix timestamp)
# Retry-After: 42 (seconds, on 429 responses)

  • Netflix Zuul/Spring Cloud Gateway: Custom gateway for their ~1000 microservices
  • Envoy Proxy: L7 proxy used in service meshes (Istio); handles gRPC load balancing, circuit breaking, observability

Pagination Deep Dive

Offset, Cursor, Keyset & Time-Based Pagination ▼

1. Offset-Based Pagination

GET /users?offset=20&limit=10

-- SQL equivalent:
SELECT * FROM users ORDER BY id LIMIT 10 OFFSET 20;

Response:
{
  "data": [...],
  "pagination": { "offset": 20, "limit": 10, "total": 15000 }
}

Pros: Simple to implement, allows jumping to any page, total count available.

Cons:

  • Performance degrades with large offsets. OFFSET 100000 means the database must scan and discard 100,000 rows before returning 10. At offset 1M, PostgreSQL query time can exceed 5 seconds.
  • Inconsistent results when data changes. If a row is inserted between page requests, items can be duplicated or skipped.

2. Cursor-Based Pagination (Opaque Cursors)

GET /users?cursor=eyJpZCI6MTIzfQ==&limit=10

# Cursor is a base64-encoded opaque token (e.g., {"id": 123})
# Server decodes it and uses it as a WHERE clause:
SELECT * FROM users WHERE id > 123 ORDER BY id LIMIT 10;

Response:
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJpZCI6MTMzfQ==",
    "has_more": true
  }
}

Pros: Consistent results regardless of insertions/deletions. O(1) performance regardless of page depth.

Cons: Cannot jump to an arbitrary page. Cursor tokens are opaque (clients cannot construct them).

3. Keyset Pagination (Seek Method)

Like cursor-based but with transparent, composite keys. Essential when sorting by non-unique columns.

# Sorting by created_at (non-unique) + id (unique tiebreaker)
GET /posts?after_date=2024-01-15T10:30:00Z&after_id=456&limit=20

-- SQL: uses composite index on (created_at, id)
SELECT * FROM posts
WHERE (created_at, id) > ('2024-01-15T10:30:00Z', 456)
ORDER BY created_at ASC, id ASC
LIMIT 20;

-- This is a "row value comparison" - extremely efficient
-- Uses the index directly, no OFFSET scanning

Why keyset over cursor? Transparent keys let clients bookmark specific positions and resume later. Useful for batch processing, data exports, and CDC (Change Data Capture) patterns.

4. Time-Based Pagination for Event Streams

# Ideal for audit logs, activity feeds, event sourcing
GET /events?since=2024-01-15T00:00:00Z&until=2024-01-16T00:00:00Z&limit=100

Response:
{
  "data": [...],
  "pagination": {
    "next_since": "2024-01-15T12:34:56.789Z",
    "has_more": true
  },
  "meta": {
    "window_start": "2024-01-15T00:00:00Z",
    "window_end": "2024-01-16T00:00:00Z"
  }
}

Performance Comparison

Method Page 1 (10 rows) Page 1,000 (10 rows) Page 100,000 (10 rows) Jump to Page?
Offset~1ms~50ms~5,000msYes
Cursor/Keyset~1ms~1ms~1msNo
Time-based~1ms~1ms~1msYes (by time)

Numbers are approximate for a PostgreSQL table with 10M rows and proper indexes.

When to Use Which:
  • Offset: Small datasets (<10K rows), admin dashboards, page-number UIs
  • Cursor: Social feeds, infinite scroll, real-time streams (Twitter, Facebook)
  • Keyset: Data exports, batch processing, CDC, APIs where clients need to resume from a known point
  • Time-based: Audit logs, event streams, analytics, monitoring dashboards
