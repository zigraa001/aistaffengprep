Scenario 5: Review this PR — find all the issues

Context: A junior engineer submitted this PR for a payment webhook handler. Find all the issues.

import json
import hashlib
from flask import Flask, request

app = Flask(__name__)
DB_PASSWORD = "prod_p@ssw0rd_2024"

@app.route("/webhooks/payment", methods=["GET", "POST"])
def handle_payment_webhook():
    data = json.loads(request.data)
    event_type = data["type"]

    if event_type == "payment.completed":
        order_id = data["data"]["order_id"]
        amount = data["data"]["amount"]

        # Update order status
        db.execute(f"UPDATE orders SET status='paid', amount={amount} WHERE id='{order_id}'")

        # Send confirmation email
        email = db.query(f"SELECT email FROM users WHERE id=(SELECT user_id FROM orders WHERE id='{order_id}')")
        send_email(email, "Payment confirmed", f"Your payment of ${amount} was received")

        # Log for debugging
        print(f"Payment received: order={order_id}, amount={amount}, email={email}")

    return "OK", 200

Issues found (12+):

  1. Hardcoded credentials — DB_PASSWORD in source code. Use environment variable or secrets manager.
  2. No signature verification — No HMAC check. Anyone can send fake webhooks.
  3. Accepts GET — Webhooks should be POST only. GET allows CSRF-style attacks.
  4. SQL injection (x3) — String interpolation in all three SQL queries. Use parameterized queries.
  5. No idempotency — Processing the same webhook twice will send duplicate emails. Check event ID.
  6. No error handling — Any exception returns 500, triggering retries that hit all the same bugs.
  7. N+1 query — Subquery to get user email. Could be a JOIN.
  8. PII in logs — Logging email address. Violates privacy regulations.
  9. print() for logging — Use structured logging with proper levels.
  10. No transaction — If email send fails after DB update, state is inconsistent.
  11. Always returns 200 — Even for unknown event types or errors. Webhook sender thinks delivery succeeded.
  12. No rate limiting — Endpoint could be DDoS'd.
  13. No input validation — No check that required fields exist in the payload.

How to deliver this feedback: Prioritize the issues. Start with security (SQL injection, no auth), then correctness (idempotency, error handling), then operational concerns (logging, monitoring). Don't dump all 12 issues at once — group them by severity.

Scenario 6: Diagnose a production incident — cascading failure

Situation: At 2:15 PM, your monitoring shows P99 latency spiked from 200ms to 15 seconds across all API endpoints. Error rate jumped from 0.1% to 35%. Walk through your diagnosis.

Step 1: Triage (first 5 minutes)

  • Check: Is this affecting all services or specific ones? → All services through the API gateway
  • Check: Did a deployment happen recently? → Payment service deployed at 2:10 PM
  • Check: Any infrastructure changes? → No

Step 2: Identify the source (next 10 minutes)

  • Dashboard shows: Payment service response times went from 50ms to 30 seconds
  • API gateway thread pool is exhausted waiting for payment service responses
  • All other services are healthy but unreachable through the gateway (connection pool exhaustion)
graph TD GW[API Gateway - thread pool exhausted] --> PS[Payment Service - 30s response time] GW --> US[User Service - healthy but blocked] GW --> OS[Order Service - healthy but blocked] PS --> DB[(Database - missing index on new column)]

Step 3: Root cause

The payment service deployment added a new query that scans a table without an index. Under load, this query takes 30 seconds. The API gateway's shared connection pool fills up waiting for payment service, blocking all other services (no bulkhead).

Step 4: Mitigation

  1. Immediate: Roll back the payment service deployment (2 minutes)
  2. Short-term: Add the missing index, redeploy with the fix
  3. Long-term: Add bulkhead pattern to API gateway (separate connection pools per service), add query performance CI checks, add deployment canary that monitors latency

Key lesson: A single slow service took down the entire platform due to missing bulkheads. This is the #1 reason for cascading failures in microservices.

Scenario 7: Black Friday capacity planning

Problem: Your e-commerce platform handles 5,000 orders/minute on a normal day. Black Friday is expected to be 10x. Plan the capacity.

Capacity calculations:

Normal load:
  - Orders: 5,000/min = 83/sec
  - API requests: ~500/sec (10 API calls per order flow)
  - Database writes: ~250/sec
  - Message broker throughput: ~500 events/sec

Black Friday (10x):
  - Orders: 50,000/min = 833/sec
  - API requests: ~5,000/sec
  - Database writes: ~2,500/sec
  - Message broker throughput: ~5,000 events/sec

Current infrastructure:
  - API servers: 10 pods (can handle ~1,000 req/sec total)
  - Database: Single primary, 2 read replicas
  - Kafka: 3 brokers, 12 partitions on order topic

Scaling plan:

ComponentCurrentRequiredAction
API servers10 pods50 podsConfigure autoscaler: min=20, max=60, target CPU=60%
Database writes1 primaryRead/write split + connection poolingAdd PgBouncer, optimize hot queries, add read replicas to 5
Kafka12 partitions24+ partitionsIncrease partitions, add broker, increase consumer instances
CacheRedis 1 nodeRedis clusterMove to 3-node Redis cluster, pre-warm cache for popular products
CDNStandardAggressive cachingCache product pages for 5 min (accept slight staleness)

Testing plan:

  1. Load test at 12x expected peak (headroom) in staging
  2. Identify bottleneck: database connection limits hit at 8x → add PgBouncer
  3. Re-test: Kafka consumer lag at 10x → increase partitions and consumers
  4. Re-test: passes at 15x — ready
  5. Game day: simulate Black Friday on staging with full team on call

Graceful degradation plan: If load exceeds capacity despite scaling, prioritize checkout over browsing. Return cached product pages, disable recommendations, queue non-critical operations (email confirmations, analytics).

Scenario 8: Monolith to microservices migration

Problem: Your 5-year-old monolith has 500K lines of code. 8 teams work in it. Deployments take 2 hours and happen twice a week. Design the migration strategy.

Step 1: Don't migrate everything

Identify the 2-3 bounded contexts that cause the most friction. Common candidates:

  • The part with the highest change frequency (e.g., checkout flow)
  • The part with different scaling needs (e.g., notification sending)
  • The part owned by a different team (Conway's Law alignment)

Step 2: Strangler Fig pattern

graph TB subgraph "Phase 1" C1[Client] --> GW1[API Gateway] GW1 --> |/notifications/*| NS1[New: Notification Service] GW1 --> |Everything else| M1[Monolith] end subgraph "Phase 3" C3[Client] --> GW3[API Gateway] GW3 --> |/notifications/*| NS3[Notification Service] GW3 --> |/payments/*| PS3[Payment Service] GW3 --> |/search/*| SS3[Search Service] GW3 --> |Core| M3[Smaller Monolith] end

Migration order (risk-based):

  1. Notifications (low risk, stateless, easy to verify)
  2. Search (read-only, different scaling needs, can dual-run)
  3. Payments (higher risk, but highest value for independent deployment)
  4. Keep the rest as a well-modularized monolith (don't extract everything)

Anti-patterns to avoid:

  • Big bang rewrite — Never. Always strangler fig.
  • Distributed monolith — If all services must deploy together, you've made things worse.
  • Shared database — Each service must own its data. Use events for cross-service data needs.
Scenario 9: Handling a security vulnerability disclosure

Situation: A security researcher reports that your API has an IDOR vulnerability — any authenticated user can access any other user's data by changing the user ID in the URL. Walk through your response.

Immediate response (first 4 hours):

  1. Acknowledge the report — Thank the researcher, confirm you're investigating. Don't disclose details.
  2. Verify the vulnerability — Reproduce it. Confirm scope: which endpoints are affected?
  3. Assess impact — Was it exploited? Check access logs for suspicious patterns (single user accessing many other users' data).
  4. Classify severity — IDOR on user data is HIGH. If it includes payment info, it's CRITICAL.

Fix and deploy (24-48 hours):

  1. Write the fix — Add authorization checks: verify current_user.id == requested_user_id or current_user.is_admin
  2. Audit all endpoints — IDOR in one place likely means IDOR in others. Grep for all endpoints that take user IDs.
  3. Add regression tests — Test that user A cannot access user B's data for every affected endpoint
  4. Deploy with fast rollout — Skip canary, deploy immediately (security fix)

Post-fix (1-2 weeks):

  • Add automated IDOR checking to CI pipeline (DAST tool targeting auth-required endpoints)
  • Add authorization middleware that's applied by default to all endpoints
  • Conduct a broader security review
  • If data was accessed: notify affected users per privacy regulations (GDPR, CCPA)
  • Reward the researcher (bug bounty or acknowledgment)
Scenario 10: Design an internal developer platform from scratch

Context: Your company has 80 engineers across 10 teams. Each team manages their own CI/CD, infrastructure, and monitoring. You've been asked to build an IDP. How do you approach it?

Phase 0: Discovery (weeks 1-4)

  • Interview all 10 teams: What takes the most time? What's duplicated?
  • Survey results show: CI/CD (#1 pain), environment provisioning (#2), observability setup (#3)
  • Calculate ROI: 10 teams x 15% time on infra x $180K avg salary = $270K/year in waste

Phase 1: Thinnest viable platform (months 1-3)

  • Standardized CI/CD templates (GitHub Actions reusable workflows)
  • Terraform modules for common infrastructure (databases, queues, caches)
  • Documentation hub (Notion/Confluence with golden path guides)
  • Measure: 3 early-adopter teams onboarded, deployment time reduced 60%

Phase 2: Self-service (months 4-8)

  • Deploy Backstage with service catalog
  • Self-service environment creation via Backstage templates
  • Automated observability setup (OTel auto-instrumentation)
  • Measure: 7/10 teams onboarded, new service creation time from 2 weeks to 2 hours

Phase 3: Maturity (months 9-12)

  • Service scorecards (production readiness tracking)
  • Cost attribution per service
  • Security baseline enforcement via policy-as-code
  • Measure: 10/10 teams, 90%+ developer satisfaction, 3x deployment frequency

Team composition: Start with 3 engineers (1 senior frontend for Backstage, 1 senior backend for CI/CD and infra, 1 staff engineer for architecture and stakeholder management). Grow to 5 by Phase 2.
