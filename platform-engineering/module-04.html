Communication Pattern Tradeoffs

Q: How do polling, webhooks, SSE, and WebSockets compare?

Decision framework:

  • Need real-time bidirectional? → WebSockets

  • Need to notify external systems of events? → Webhooks
  • Simple, infrequent checks with no infra requirements? → Polling

Webhook Design

Q: What makes a well-designed webhook system?

A: A production-grade webhook system needs these components:

graph LR E[Event Occurs] --> Q[Event Queue] Q --> D[Delivery Worker] D --> |POST| EP[Consumer Endpoint] EP --> |2xx| S[Mark Delivered] EP --> |non-2xx/timeout| R[Retry Queue] R --> |exponential backoff| D R --> |max retries exceeded| DLQ[Dead Letter Queue] DLQ --> ALERT[Alert + Manual Review]

Essential features:

  1. Signature verification — HMAC-SHA256 so consumers can verify the payload is authentic
  2. Idempotency — Include a unique event ID so consumers can deduplicate
  3. Retry with backoff — Exponential backoff (e.g., 1m, 5m, 30m, 2h, 24h)
  4. Dead letter queue — Store failed deliveries for manual investigation
  5. Event types — Consumers subscribe to specific event types, not everything
  6. Delivery log — Searchable history of all delivery attempts with request/response details
  7. Replay capability — Allow consumers to re-request missed events
Q: How do you implement HMAC signature verification for webhooks?

A: The sender computes an HMAC of the payload using a shared secret. The receiver recomputes it and compares.

Sender (Python):

import hmac
import hashlib
import json
import time

def send_webhook(url: str, payload: dict, secret: str):
    body = json.dumps(payload, separators=(',', ':'))
    timestamp = str(int(time.time()))
    # Sign: timestamp + '.' + body
    signature_payload = f"{timestamp}.{body}"
    signature = hmac.new(
        secret.encode(),
        signature_payload.encode(),
        hashlib.sha256
    ).hexdigest()

    headers = {
        "Content-Type": "application/json",
        "X-Webhook-Signature": f"t={timestamp},v1={signature}",
        "X-Webhook-Id": str(uuid.uuid4()),
    }
    response = requests.post(url, data=body, headers=headers, timeout=30)
    return response

Receiver (Node.js):

const crypto = require('crypto');

function verifyWebhookSignature(req, secret) {
  const sigHeader = req.headers['x-webhook-signature'];
  const parts = Object.fromEntries(
    sigHeader.split(',').map(p => p.split('='))
  );

  const timestamp = parts.t;
  const expectedSig = parts.v1;

  // Prevent replay attacks: reject if timestamp is too old
  const age = Math.floor(Date.now() / 1000) - parseInt(timestamp);
  if (age > 300) { // 5 minutes
    throw new Error('Webhook timestamp too old - possible replay attack');
  }

  const signaturePayload = `${timestamp}.${req.rawBody}`;
  const computedSig = crypto
    .createHmac('sha256', secret)
    .update(signaturePayload)
    .digest('hex');

  // Constant-time comparison to prevent timing attacks
  if (!crypto.timingSafeEqual(
    Buffer.from(expectedSig),
    Buffer.from(computedSig)
  )) {
    throw new Error('Invalid webhook signature');
  }

  return true;
}

Critical details:

  • Use constant-time comparison (timingSafeEqual) to prevent timing attacks
  • Include timestamp in the signature to prevent replay attacks
  • Use the raw body for verification, not a re-serialized version (JSON key ordering may differ)

Q: What delivery guarantees can webhooks provide?

A: Webhooks inherently provide at-least-once delivery. Here's why:

  • At-most-once: Send once, never retry. Simple but you'll lose events when the consumer is temporarily down.
  • At-least-once: Retry on failure. The consumer may receive duplicates (e.g., if the 200 response was lost in transit). This is the standard for production webhooks.
  • Exactly-once: Impossible at the network level. Achieved through at-least-once delivery + consumer-side idempotency.
# Consumer-side idempotency handler
async def handle_webhook(request):
    event = await request.json()
    event_id = event["id"]

    # Check if we've already processed this event
    if await redis.sismember("processed_events", event_id):
        return Response(status_code=200)  # Acknowledge but don't reprocess

    # Process the event (inside a transaction)
    async with db.transaction():
        await process_event(event)
        await redis.sadd("processed_events", event_id)
        await redis.expire(f"processed_events", 86400 * 7)  # 7-day TTL

    return Response(status_code=200)

Key rule: Always design webhook consumers to be idempotent. You will receive duplicates — retries, replays, and network issues make it inevitable.

Q: How do you design webhook retry strategies?

A: Retries should use exponential backoff with jitter to avoid thundering herd:

import random

def calculate_retry_delay(attempt: int, base_delay: float = 60) -> float:
    """Exponential backoff with full jitter."""
    max_delay = base_delay * (2 ** attempt)  # 1m, 2m, 4m, 8m, 16m...
    max_delay = min(max_delay, 86400)        # Cap at 24 hours
    return random.uniform(0, max_delay)       # Full jitter

# Typical retry schedule:
# Attempt 1: ~0-1 minute after failure
# Attempt 2: ~0-2 minutes
# Attempt 3: ~0-4 minutes
# Attempt 4: ~0-8 minutes
# Attempt 5: ~0-16 minutes
# ... up to max retries (e.g., 8 attempts over ~24 hours)

Disable policy: After consecutive failures (e.g., 3 days of failures), automatically disable the webhook endpoint and notify the consumer via email. This prevents wasting resources on a permanently broken endpoint.

What counts as success: A 2xx response within the timeout (typically 30 seconds). A 3xx redirect should be followed (once). Any 4xx or 5xx or timeout triggers a retry.

Event-Driven Architecture

Q: What is the CloudEvents specification and why should platform teams care?

A: CloudEvents is a CNCF specification for describing event data in a common way. It standardizes event metadata across systems.

{
  "specversion": "1.0",
  "type": "com.example.order.created",
  "source": "/orders/service",
  "id": "A234-1234-1234",
  "time": "2024-01-15T10:30:00Z",
  "datacontenttype": "application/json",
  "subject": "order-12345",
  "data": {
    "orderId": "order-12345",
    "customerId": "cust-789",
    "total": 99.99
  }
}

Why platform teams should adopt it:

  • Interoperability — Events from different services have a common envelope format
  • Tooling — SDKs, middleware, and routing tools understand CloudEvents natively
  • Governance — Standard fields (type, source, id) enable consistent filtering, routing, and auditing
  • Event catalog — Standard type naming (com.company.domain.action) enables a searchable event registry
Q: What is the difference between event notification, event-carried state transfer, and event sourcing?

A: These are three distinct patterns that use events differently:

PatternEvent ContainsConsumer ActionCoupling
Event NotificationMinimal: "Order 123 was created"Calls back to source for detailsHigher (needs API call)
Event-Carried State TransferFull state: "Order 123 with all details"Uses event data directly, no callbackLower (self-contained)
Event SourcingState change: "Item X added to order 123"Rebuilds state by replaying eventsLowest (append-only log)

Staff-level decision: Most teams should start with event-carried state transfer. It's simpler than event sourcing and avoids the back-and-forth of event notification. Include enough data in the event that consumers don't need to make additional API calls. Reserve event sourcing for domains where auditability and state reconstruction are requirements (financial systems, compliance).

Q: What is the fan-out pattern for webhooks?

A: Fan-out delivers a single event to multiple consumer endpoints. This is critical for platforms where multiple teams subscribe to the same events.

graph LR E[Order Created Event] --> FO[Fan-Out Service] FO --> C1[Billing Webhook] FO --> C2[Shipping Webhook] FO --> C3[Analytics Webhook] FO --> C4[Notification Webhook]

Implementation considerations:

  • Independent delivery — Each consumer's delivery is independent. If billing's endpoint is down, shipping still receives the event.
  • Independent retry — Each consumer has its own retry queue. Don't delay all consumers because one is slow.
  • Subscription management — Self-service API for consumers to register endpoints and select event types.
  • Ordering — Fan-out typically does NOT guarantee ordering across consumers. If order matters, use a message broker with partitioned topics instead.

# SSRF prevention: validate webhook URLs
import ipaddress
from urllib.parse import urlparse

BLOCKED_RANGES = [
    ipaddress.ip_network('10.0.0.0/8'),
    ipaddress.ip_network('172.16.0.0/12'),
    ipaddress.ip_network('192.168.0.0/16'),
    ipaddress.ip_network('127.0.0.0/8'),
    ipaddress.ip_network('169.254.0.0/16'),  # link-local
]

def validate_webhook_url(url: str) -> bool:
    parsed = urlparse(url)
    if parsed.scheme not in ('https',):  # Require HTTPS
        return False
    try:
        ip = ipaddress.ip_address(parsed.hostname)
        for blocked in BLOCKED_RANGES:
            if ip in blocked:
                return False  # Block internal IPs
    except ValueError:
        pass  # Hostname, not IP - resolve and check
    return True
Q: How do you test webhooks during development?

A: Webhook testing is uniquely challenging because the sender pushes to your endpoint. Strategies:

  1. Local tunnel — Tools like ngrok or Cloudflare Tunnel expose your localhost to the internet for receiving webhooks during development
  2. Webhook test mode — Send test events from the provider's dashboard (Stripe, GitHub support this)
  3. Request bin — Use services like webhook.site to capture and inspect webhook payloads
  4. Mock webhook sender — Build a test harness that sends signed webhook payloads to your endpoint
  5. Record and replay — Capture production webhooks, anonymize data, replay in test environments
# Test helper: send a signed test webhook
def send_test_webhook(endpoint: str, event_type: str, data: dict, secret: str):
    """Send a properly signed test webhook for integration testing."""
    event = {
        "id": f"evt_test_{uuid.uuid4().hex[:8]}",
        "type": event_type,
        "created": int(time.time()),
        "data": data
    }
    body = json.dumps(event)
    timestamp = str(int(time.time()))
    sig = hmac.new(secret.encode(), f"{timestamp}.{body}".encode(), hashlib.sha256).hexdigest()

    response = requests.post(
        endpoint,
        data=body,
        headers={
            "Content-Type": "application/json",
            "X-Webhook-Signature": f"t={timestamp},v1={sig}",
            "X-Webhook-Id": event["id"],
        },
        timeout=30
    )
    return response
Q: How do you monitor webhook health in production?

A: Key metrics to track:

  • Delivery success rate — Per endpoint and overall. Alert if below 95%.
  • Delivery latency — P50, P95, P99 time from event to successful delivery.
  • Retry rate — High retry rate indicates consumer health issues.
  • DLQ depth — Growing DLQ means events are permanently failing.
  • Endpoint response times — Slow consumers can back up delivery queues.
  • Event volume by type — Unexpected spikes may indicate system issues.

Alerting rules:

  • Alert: Delivery success rate < 95% for 15 minutes
  • Alert: DLQ depth > 100 events
  • Alert: Specific endpoint failing for > 1 hour
  • Alert: Delivery latency P95 > 5 minutes

Dashboard: Build a webhook health dashboard showing per-endpoint delivery status, recent failures with response codes, and event volume trends. This is invaluable for debugging integration issues with partners.

Q: How should webhook consumers handle ordering and consistency?

A: Webhooks do not guarantee ordering. Events may arrive out of order due to retries, parallel delivery, and network variability.

Strategies:

  • Include a sequence number or timestamp — Let consumers detect out-of-order events
  • Include full state, not deltas — If the event carries the full current state, out-of-order delivery is less problematic (last-write-wins)
  • Version field — Include a monotonically increasing version on the resource. Consumer ignores events with a version older than what they've already seen.
# Consumer: handle out-of-order with version checking
async def handle_order_updated(event):
    order_data = event["data"]
    order_id = order_data["id"]
    event_version = order_data["version"]

    # Only process if this is a newer version
    current_version = await db.fetchval(
        "SELECT version FROM orders WHERE id = $1", order_id
    )
    if current_version and current_version >= event_version:
        return  # Skip stale event

    await db.execute(
        "UPDATE orders SET status = $1, version = $2 WHERE id = $3 AND version < $2",
        order_data["status"], event_version, order_id
    )

If strict ordering is required: Use a message broker with partitioned topics (Kafka) instead of webhooks. Webhooks are best for event notification where eventual consistency is acceptable.

Q: What is Server-Sent Events (SSE) and when should you use it over WebSockets?

A: SSE is a standard for server-to-client streaming over HTTP. The client opens a long-lived GET connection and the server sends events as they occur.

# Server (Python/FastAPI)
from fastapi import FastAPI
from fastapi.responses import StreamingResponse
import asyncio

app = FastAPI()

async def event_stream(user_id: str):
    while True:
        event = await get_next_event(user_id)
        if event:
            yield f"id: {event.id}\n"
            yield f"event: {event.type}\n"
            yield f"data: {json.dumps(event.data)}\n\n"
        await asyncio.sleep(1)

@app.get("/events/stream")
async def stream_events(user_id: str):
    return StreamingResponse(
        event_stream(user_id),
        media_type="text/event-stream",
        headers={"Cache-Control": "no-cache", "Connection": "keep-alive"}
    )
// Client (JavaScript)
const source = new EventSource('/events/stream?user_id=123');

source.addEventListener('order.updated', (e) => {
  const data = JSON.parse(e.data);
  console.log('Order updated:', data);
});

source.onerror = () => {
  // EventSource auto-reconnects with Last-Event-ID header
  console.log('Connection lost, reconnecting...');
};

SSE vs WebSockets:

  • SSE: Server-to-client only, auto-reconnect built-in, works with HTTP/2 multiplexing, simpler infrastructure
  • WebSockets: Bidirectional, more complex, better for chat/gaming, requires sticky sessions or a pub/sub layer

Rule of thumb: If you only need server → client push (dashboards, notifications, live feeds), use SSE. It's simpler and plays better with HTTP infrastructure (load balancers, proxies, CDNs).
