Pagination Patterns

Q: What are the main pagination strategies and their tradeoffs?

A:

StrategyHow It WorksProsConsBest For
Offset ?offset=20&limit=10 Simple, jump to any page Slow on large datasets (OFFSET scans rows), inconsistent with concurrent writes Small datasets, admin UIs
Cursor ?cursor=eyJpZCI6MTIzfQ&limit=10 Consistent results, efficient (index seek) Can't jump to arbitrary page, opaque cursor Infinite scroll, feeds, large datasets
Keyset ?after_id=123&limit=10 Transparent, efficient, stable Requires sortable unique column, can't jump pages API consumers who need transparency
Page number ?page=3&per_page=10 Intuitive for users Same problems as offset (it IS offset) Simple web UIs with page navigation

Why offset pagination breaks at scale:

-- Offset pagination: database must scan and discard 1,000,000 rows
SELECT * FROM events ORDER BY created_at DESC OFFSET 1000000 LIMIT 10;

-- Keyset pagination: database seeks directly via index
SELECT * FROM events
WHERE created_at < '2024-01-15T10:30:00Z'
ORDER BY created_at DESC
LIMIT 10;

The offset query gets linearly slower as the offset grows. At 1M offset, PostgreSQL must scan 1,000,010 rows to return 10. The keyset query always scans ~10 rows regardless of position.

Q: How do you implement cursor-based pagination?

A: A cursor encodes the position in the result set, typically as a base64-encoded combination of sort fields:

import base64
import json
from datetime import datetime

def encode_cursor(created_at: datetime, id: str) -> str:
    """Encode pagination position as opaque cursor."""
    payload = {"created_at": created_at.isoformat(), "id": id}
    return base64.urlsafe_b64encode(json.dumps(payload).encode()).decode()

def decode_cursor(cursor: str) -> dict:
    """Decode cursor back to pagination position."""
    return json.loads(base64.urlsafe_b64decode(cursor.encode()))

async def list_events(cursor: str = None, limit: int = 20):
    query = "SELECT * FROM events"
    params = []

    if cursor:
        pos = decode_cursor(cursor)
        # Use compound condition for stable sorting
        query += """
            WHERE (created_at, id) < (%(created_at)s, %(id)s)
        """
        params = {"created_at": pos["created_at"], "id": pos["id"]}

    query += " ORDER BY created_at DESC, id DESC LIMIT %(limit)s"
    params["limit"] = limit + 1  # Fetch one extra to check if more exist

    rows = await db.fetch(query, params)
    has_next = len(rows) > limit
    items = rows[:limit]

    return {
        "items": items,
        "pagination": {
            "next_cursor": encode_cursor(items[-1].created_at, items[-1].id) if has_next else None,
            "has_next": has_next
        }
    }

Key implementation details:

  • Always include a unique column (like id) in the cursor to handle ties in the sort field
  • Fetch limit + 1 rows to determine if there's a next page without a separate COUNT query
  • Make cursors opaque (base64) so clients don't try to construct them manually
  • Include a composite index on (created_at, id) for efficient seeks
Q: What is Relay-style pagination for GraphQL?

A: The Relay Connection specification is the standard pagination pattern for GraphQL. It uses edges, nodes, cursors, and pageInfo:

type Query {
  users(first: Int, after: String, last: Int, before: String): UserConnection!
}

type UserConnection {
  edges: [UserEdge!]!
  pageInfo: PageInfo!
  totalCount: Int
}

type UserEdge {
  node: User!
  cursor: String!
}

type PageInfo {
  hasNextPage: Boolean!
  hasPreviousPage: Boolean!
  startCursor: String
  endCursor: String
}

# Query example
query {
  users(first: 10, after: "cursor123") {
    edges {
      node {
        id
        name
        email
      }
      cursor
    }
    pageInfo {
      hasNextPage
      endCursor
    }
    totalCount
  }
}

Why this structure:

  • Edges wrap nodes so you can attach edge-specific metadata (e.g., relationship properties in a graph)
  • Cursors on each edge let you start from any item, not just the page boundary
  • pageInfo provides navigation without separate queries
  • first/after for forward pagination, last/before for backward

When to use Relay vs simple pagination: Use Relay-style for production GraphQL APIs that multiple clients consume. For internal tools or simple APIs, a simpler {items, nextCursor} pattern is fine.

Rate Limiting

Q: What are the main rate limiting algorithms and when should you use each?

A:

AlgorithmHow It WorksProsConsBest For
Token Bucket Tokens added at fixed rate, each request consumes a token Allows bursts, smooth average rate Burst size needs tuning Most APIs (default choice)
Leaky Bucket Requests queue and drain at fixed rate Perfectly smooth output rate No burst tolerance, adds latency Systems needing constant throughput
Fixed Window Count requests in fixed time windows Simple, low memory Burst at window boundaries (2x rate) Simple rate limiting needs
Sliding Window Log Track timestamp of each request, count in rolling window Accurate, no boundary issues Memory-intensive (stores all timestamps) Low-volume, high-accuracy needs
Sliding Window Counter Weighted average of current and previous window Memory-efficient, smooth Approximate (but close enough) High-volume production use
Q: How does the token bucket algorithm work? Show an implementation.

A: Token bucket maintains a bucket of tokens that refills at a constant rate. Each request consumes one token. If the bucket is empty, the request is rejected.

graph LR R[Refill: tokens added at rate R] --> B[Token Bucket - capacity C] B --> |token available| A[Request Allowed] B --> |bucket empty| D[Request Rejected - 429] 
import time
import redis

class TokenBucketRateLimiter:
    def __init__(self, redis_client: redis.Redis, capacity: int, refill_rate: float):
        """
        capacity: max tokens in bucket
        refill_rate: tokens added per second
        """
        self.redis = redis_client
        self.capacity = capacity
        self.refill_rate = refill_rate

    def allow_request(self, key: str) -> tuple[bool, dict]:
        """Check if request is allowed. Returns (allowed, metadata)."""
        now = time.time()
        bucket_key = f"ratelimit:{key}"

        # Atomic operation via Lua script
        lua_script = """
        local key = KEYS[1]
        local capacity = tonumber(ARGV[1])
        local refill_rate = tonumber(ARGV[2])
        local now = tonumber(ARGV[3])

        local bucket = redis.call('HMGET', key, 'tokens', 'last_refill')
        local tokens = tonumber(bucket[1]) or capacity
        local last_refill = tonumber(bucket[2]) or now

        -- Refill tokens based on elapsed time
        local elapsed = now - last_refill
        tokens = math.min(capacity, tokens + elapsed * refill_rate)

        local allowed = 0
        if tokens >= 1 then
            tokens = tokens - 1
            allowed = 1
        end

        redis.call('HMSET', key, 'tokens', tokens, 'last_refill', now)
        redis.call('EXPIRE', key, math.ceil(capacity / refill_rate) * 2)

        return {allowed, math.floor(tokens)}
        """
        allowed, remaining = self.redis.eval(
            lua_script, 1, bucket_key,
            self.capacity, self.refill_rate, now
        )

        return bool(allowed), {
            "X-RateLimit-Limit": self.capacity,
            "X-RateLimit-Remaining": remaining,
            "X-RateLimit-Reset": int(now + (self.capacity - remaining) / self.refill_rate)
        }

Why Lua script? The read-compute-write must be atomic. Without the Lua script, a race condition between two concurrent requests could both read the same token count and both succeed, exceeding the limit.

Q: What is the sliding window counter algorithm and why is it preferred in production?

A: The sliding window counter combines the simplicity of fixed windows with the accuracy of sliding window logs. It uses a weighted average of the current and previous window counts.

def sliding_window_counter(key: str, limit: int, window_seconds: int) -> bool:
    """
    Approximate sliding window using weighted average.
    Example: limit=100, window=60s
    """
    now = time.time()
    current_window = int(now // window_seconds)
    previous_window = current_window - 1
    elapsed_in_current = now % window_seconds
    weight_previous = 1 - (elapsed_in_current / window_seconds)

    prev_count = redis.get(f"rate:{key}:{previous_window}") or 0
    curr_count = redis.get(f"rate:{key}:{current_window}") or 0

    # Weighted estimate of requests in the sliding window
    estimated_count = int(prev_count) * weight_previous + int(curr_count)

    if estimated_count >= limit:
        return False  # Rate limited

    # Increment current window counter
    pipe = redis.pipeline()
    pipe.incr(f"rate:{key}:{current_window}")
    pipe.expire(f"rate:{key}:{current_window}", window_seconds * 2)
    pipe.execute()
    return True

Example: If the previous window (0:00-1:00) had 80 requests and the current window (1:00-2:00) has 30 requests at time 1:15 (25% into the window):

  • Weight of previous window: 75% (0.75)
  • Estimated count: 80 × 0.75 + 30 = 90
  • If limit is 100, this request is allowed (90 < 100)

Why production prefers this: Only two counters in memory per client (current + previous window), O(1) time and space, and 99.9% accurate compared to exact sliding window. Cloudflare uses this algorithm for their rate limiting.

Q: How should rate limit information be communicated to clients?

A: Use standard headers (draft RFC 6585 / RateLimit header fields):

HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 742
X-RateLimit-Reset: 1705312800
RateLimit-Policy: 1000;w=3600

---

HTTP/1.1 429 Too Many Requests
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1705312800
Retry-After: 45
Content-Type: application/problem+json

{
  "type": "https://api.example.com/errors/rate-limited",
  "title": "Rate Limit Exceeded",
  "status": 429,
  "detail": "You have exceeded the rate limit of 1000 requests per hour",
  "retryAfter": 45
}

Client-side best practices:

  • Always check X-RateLimit-Remaining and proactively slow down before hitting 0
  • On 429, respect Retry-After header — don't retry immediately
  • Implement exponential backoff with jitter for retries
  • Consider a client-side rate limiter that mirrors the server's limits
Q: How do you design rate limiting for a multi-tenant platform?

A: Multi-tenant rate limiting requires multiple layers:

LayerScopePurposeExample
GlobalEntire APIProtect infrastructure50,000 req/s total
Per-tenantOrganization/accountFair share, prevent noisy neighbor1,000 req/min per org
Per-userIndividual userPrevent abuse100 req/min per user
Per-endpointSpecific routesProtect expensive operations10 req/min for /reports/generate

Tiered limits: Different plans get different limits. Store limit configurations in a fast lookup (Redis hash or in-memory cache):

# Rate limit configuration per plan
PLAN_LIMITS = {
    "free":       {"requests_per_minute": 60,   "requests_per_day": 1000},
    "pro":        {"requests_per_minute": 600,  "requests_per_day": 50000},
    "enterprise": {"requests_per_minute": 6000, "requests_per_day": 500000},
}

async def rate_limit_middleware(request, call_next):
    tenant = get_tenant(request)
    plan = await get_tenant_plan(tenant.id)
    limits = PLAN_LIMITS[plan]

    # Check per-minute limit
    if not rate_limiter.allow(f"tenant:{tenant.id}:rpm", limits["requests_per_minute"], 60):
        return JSONResponse(status_code=429, content={"error": "Rate limit exceeded"})

    # Check per-day limit
    if not rate_limiter.allow(f"tenant:{tenant.id}:rpd", limits["requests_per_day"], 86400):
        return JSONResponse(status_code=429, content={"error": "Daily quota exceeded"})

    return await call_next(request)

API Gateways

Q: What is an API gateway and what responsibilities should it own?

A: An API gateway is a reverse proxy that sits between clients and backend services, handling cross-cutting concerns:

graph LR C[Clients] --> GW[API Gateway] GW --> |/users/*| US[User Service] GW --> |/orders/*| OS[Order Service] GW --> |/products/*| PS[Product Service] subgraph "Gateway Responsibilities" AUTH[Authentication] RL[Rate Limiting] LOG[Logging/Metrics] TLS[TLS Termination] ROUTE[Routing] TRANS[Request Transform] end

Should own:

  • TLS termination
  • Authentication (token validation, not issuance)
  • Rate limiting
  • Request/response logging
  • Request routing
  • CORS handling
  • Request ID injection

Should NOT own:

  • Business logic
  • Data transformation beyond simple header mapping
  • Complex orchestration (use a BFF or API composition layer instead)
  • Authorization beyond coarse-grained checks (fine-grained authz belongs in services)

Popular options:

GatewayTypeBest For
KongOpen-source / EnterprisePlugin ecosystem, Lua extensibility
EnvoyOpen-sourceService mesh integration, gRPC, high performance
AWS API GatewayManagedAWS-native, serverless integration
Azure API ManagementManagedAzure-native, developer portal
TraefikOpen-sourceDocker/Kubernetes, auto-discovery
Q: What is the Backend-for-Frontend (BFF) pattern and when should you use it?

A: BFF creates a dedicated backend service for each frontend client type (web, mobile, smart TV), tailoring the API to each client's needs.

graph TB Web[Web App] --> BFF_W[Web BFF] Mobile[Mobile App] --> BFF_M[Mobile BFF] TV[Smart TV] --> BFF_T[TV BFF] BFF_W --> US[User Service] BFF_W --> OS[Order Service] BFF_M --> US BFF_M --> OS BFF_T --> US

When to use BFF:

  • Different clients need different data shapes (mobile needs less data than web)
  • Clients need aggregated data from multiple services
  • Different auth flows per client (OAuth for web, API key for mobile)
  • Different performance requirements (mobile needs optimized payloads)

When NOT to use BFF:

  • Single client type — just build a good API
  • GraphQL already solves the over/under-fetching problem
  • Small team that can't maintain multiple BFFs

BFF vs GraphQL: GraphQL is often proposed as an alternative to BFF because it lets clients request exactly the data they need. In practice, many organizations use both — a BFF that exposes a GraphQL endpoint, handling service orchestration and data aggregation behind the scenes.

Q: How does a service mesh differ from an API gateway?

A:

DimensionAPI GatewayService Mesh
PositionEdge (north-south traffic)Between services (east-west traffic)
ScopeExternal client to internal servicesService-to-service communication
DeploymentCentralized proxySidecar proxy per service
FeaturesAuth, rate limiting, routing, TLSmTLS, retries, circuit breaking, observability
ExamplesKong, AWS API GatewayIstio, Linkerd, Consul Connect
Who configuresPlatform teamPlatform team + service teams
graph TB subgraph "External Traffic (API Gateway)" Client --> GW[API Gateway] end subgraph "Internal Traffic (Service Mesh)" GW --> SA[Service A + Sidecar] SA --> SB[Service B + Sidecar] SA --> SC[Service C + Sidecar] SB --> SC end

Staff-level take: Most organizations need an API gateway. Not all need a service mesh. A service mesh adds significant operational complexity (sidecar resource overhead, debugging through proxies, configuration management). Consider it only when you have 20+ services and need consistent mTLS, traffic policies, or observability across all service-to-service communication.

Q: What are API composition patterns and when are they needed?

A: API composition aggregates data from multiple backend services into a single response. This is needed when a client request spans multiple service boundaries.

Patterns:

  1. Gateway aggregation — The API gateway calls multiple services and merges responses. Simple but puts logic in the gateway.
  2. BFF orchestration — A dedicated BFF service orchestrates calls. Better separation of concerns.
  3. GraphQL resolution — GraphQL resolver calls multiple services per query. Client-driven composition.
  4. Async composition — For non-real-time needs, pre-compute aggregated views via event processing.
# BFF orchestration example - parallel service calls
import asyncio
import httpx

async def get_user_dashboard(user_id: str):
    async with httpx.AsyncClient() as client:
        # Parallel calls to multiple services
        profile_task = client.get(f"http://user-service/users/{user_id}")
        orders_task = client.get(f"http://order-service/users/{user_id}/orders?limit=5")
        notifications_task = client.get(f"http://notification-service/users/{user_id}/unread")

        profile, orders, notifications = await asyncio.gather(
            profile_task, orders_task, notifications_task,
            return_exceptions=True  # Don't fail if one service is down
        )

        return {
            "profile": profile.json() if not isinstance(profile, Exception) else None,
            "recent_orders": orders.json() if not isinstance(orders, Exception) else None,
            "unread_notifications": notifications.json() if not isinstance(notifications, Exception) else {"count": 0},
        }

Key consideration: Always use return_exceptions=True and handle partial failures gracefully. A dashboard should still render if the notifications service is down — just show "0 notifications" as a degraded fallback.

Q: How do you handle traffic shaping and canary routing at the gateway level?

A: Traffic shaping at the gateway enables safe deployments and A/B testing:

Canary routing: Route a percentage of traffic to the new version:

# Envoy route configuration for canary
routes:
  - match:
      prefix: "/api/users"
    route:
      weighted_clusters:
        clusters:
          - name: user-service-v1
            weight: 90
          - name: user-service-v2
            weight: 10

Header-based routing: Route specific users to new versions:

# Route beta users to v2
routes:
  - match:
      prefix: "/api/users"
      headers:
        - name: "X-Beta-User"
          exact_match: "true"
    route:
      cluster: user-service-v2
  - match:
      prefix: "/api/users"
    route:
      cluster: user-service-v1

Traffic shaping strategies:

  • Percentage-based — 1% → 5% → 25% → 100% progressive rollout
  • Header-based — Beta users, internal employees first
  • Region-based — Roll out to one region before global
  • Shadow traffic — Mirror traffic to new version without serving responses (for testing)

Staff-level insight: Traffic shaping at the gateway is a platform capability that enables safe deployments for all teams. Build it once, and every service can do canary deployments without implementing their own routing logic.

Q: What are the key anti-patterns in API gateway design?

A:

  1. God Gateway — Putting business logic, data transformation, and orchestration in the gateway. It becomes a monolith that's hard to test and deploy.
  2. Single point of failure — Not making the gateway highly available. Always deploy at least 2 instances with health checks.
  3. Gateway per team — Each team deploying their own gateway, defeating the purpose of centralized cross-cutting concerns.
  4. Tight coupling — Gateway configuration that hardcodes service locations instead of using service discovery.
  5. No circuit breaking — A slow backend service causes the gateway to hold connections, eventually exhausting resources and affecting all services.
  6. Over-transformation — Transforming requests/responses extensively in the gateway. Keep transformations minimal; if complex, use a BFF.

Rule of thumb: If you can't explain what the gateway does in one sentence ("It handles auth, rate limiting, and routing"), it's doing too much.
