E2E Testing

Q: When should you write E2E tests and what are their tradeoffs?

A: E2E tests verify the complete user journey through the system. They're the most realistic but also the most expensive.

ProsCons
Highest confidence (tests what users experience)Slow (minutes per test)
Catches integration issues across all layersFlaky (network, timing, external deps)
Tests real deployment configurationExpensive to maintain
Good for critical user journeysHard to debug when they fail

Rule: Write E2E tests only for critical user journeys — the flows where a failure means lost revenue or broken trust. For a platform: "developer creates a service", "developer deploys to production", "developer views service health". Aim for 5-15 E2E tests, not 500.

Q: How do you write maintainable E2E tests with the Page Object Model?
// Playwright with Page Object Model
// pages/LoginPage.js
class LoginPage {
  constructor(page) {
    this.page = page;
    this.emailInput = page.locator('[data-testid="email"]');
    this.passwordInput = page.locator('[data-testid="password"]');
    this.submitButton = page.locator('[data-testid="login-submit"]');
    this.errorMessage = page.locator('[data-testid="login-error"]');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email, password) {
    await this.emailInput.fill(email);
    await this.passwordInput.fill(password);
    await this.submitButton.click();
  }
}

// tests/login.spec.js
test('user can log in with valid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@test.com', 'password123');
  await expect(page).toHaveURL('/dashboard');
});

test('shows error for invalid credentials', async ({ page }) => {
  const loginPage = new LoginPage(page);
  await loginPage.goto();
  await loginPage.login('user@test.com', 'wrong');
  await expect(loginPage.errorMessage).toBeVisible();
});

Key principles: Use data-testid attributes (not CSS selectors). Encapsulate page interactions in page objects. Keep tests focused on user behavior, not implementation details.

Chaos Engineering

Q: What is chaos engineering and what are its core principles?

A: Chaos engineering is the discipline of experimenting on a system to build confidence in its ability to withstand turbulent conditions in production.

The chaos engineering cycle:

graph LR H[Define steady state] --> E[Form hypothesis] E --> R[Run experiment] R --> A[Analyze results] A --> F[Fix weaknesses] F --> H

Principles (from Principles of Chaos Engineering):

  1. Build a hypothesis around steady-state behavior — Define what "normal" looks like (latency < 200ms, error rate < 0.1%)
  2. Vary real-world events — Inject realistic failures (server crash, network partition, dependency latency)
  3. Run experiments in production — Testing in staging misses production-specific behaviors
  4. Automate experiments — Run continuously, not just once
  5. Minimize blast radius — Start small (1% of traffic), have a kill switch
Q: How do you use Toxiproxy for chaos testing in integration tests?

A: Toxiproxy is a TCP proxy that simulates network conditions (latency, disconnects, bandwidth limits) between your service and its dependencies.

# Toxiproxy setup for testing
from toxiproxy import Toxiproxy

toxiproxy = Toxiproxy()
proxy = toxiproxy.create(name="postgres", upstream="postgres:5432", listen="0.0.0.0:15432")

def test_service_handles_database_latency():
    """Service should timeout gracefully when DB is slow."""
    # Add 5 second latency to all DB traffic
    proxy.add_toxic(type="latency", attributes={"latency": 5000})
    try:
        result = service.get_user("user-123")
        # Should return cached result or graceful error, not hang
        assert result.source == "cache" or result.error == "timeout"
    finally:
        proxy.remove_toxic("latency_downstream")

def test_service_handles_database_disconnect():
    """Service should handle sudden DB connection loss."""
    proxy.add_toxic(type="reset_peer", attributes={"timeout": 0})
    try:
        result = service.create_order(order_data)
        assert result.status == "error"
        assert "database" in result.message.lower()
    finally:
        proxy.remove_toxic("reset_peer_downstream")

def test_service_handles_slow_responses():
    """Service circuit breaker should trip on slow downstream."""
    proxy.add_toxic(type="latency", attributes={"latency": 10000})
    # Make multiple calls to trip circuit breaker
    results = [service.get_user(f"user-{i}") for i in range(10)]
    # After threshold, circuit breaker should return fast failures
    assert any(r.error == "circuit_open" for r in results[-3:])

Safe Deployment Patterns

Q: How do canary deployments work and what metrics should you watch?

A: Canary deployment routes a small percentage of traffic to the new version while monitoring for regressions.

graph LR LB[Load Balancer] -->|95%| V1[v1.2.0 - 10 pods] LB -->|5%| V2[v1.3.0 - 1 pod - CANARY] V2 --> M[Monitor: latency, errors, business metrics] M -->|healthy for 30min| PROMOTE[Promote to 100%] M -->|regression detected| ROLLBACK[Rollback canary]

Metrics to monitor during canary:

  • Error rate — Compare canary vs baseline. Alert if canary error rate is 2x+ higher.
  • Latency — P50, P95, P99 comparison. Alert if P99 regresses > 20%.
  • Business metrics — Conversion rate, checkout completions (if applicable)
  • Resource usage — CPU, memory. A memory leak won't show in error rates initially.

Automated canary analysis: Tools like Kayenta (Netflix/Spinnaker) and Flagger (Kubernetes) automate the comparison and promotion/rollback decision. Manual canary analysis doesn't scale.

Q: How do feature flags enable safe testing in production?

A: Feature flags decouple deployment from release. You deploy code to production but activate it gradually.

# Feature flag patterns
from featureflags import FeatureClient

flags = FeatureClient(api_key="...")

async def get_recommendations(user_id: str):
    if flags.is_enabled("new-recommendation-engine", user_id=user_id):
        return await new_recommendation_engine.get(user_id)
    else:
        return await legacy_recommendation_engine.get(user_id)

# Rollout strategies:
# 1. Percentage: 5% of users see new feature
# 2. User targeting: specific user IDs (for internal testing)
# 3. Group targeting: all beta users, all employees
# 4. Geographic: enabled in US-West, not US-East

Feature flag lifecycle:

  1. Deploy code with flag defaulting to OFF
  2. Enable for internal users (dogfooding)
  3. Enable for 1% of production users, monitor
  4. Gradual rollout to 10%, 50%, 100%
  5. Remove the flag and the old code path (critical! tech debt otherwise)

Tools: LaunchDarkly, Split, Unleash, Flagsmith, or custom (Redis-backed flag store)

Performance Testing

Q: How do you design and run performance tests with k6?
// k6 load test script
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  stages: [
    { duration: '2m', target: 100 },   // Ramp up to 100 users
    { duration: '5m', target: 100 },   // Stay at 100 for 5 min
    { duration: '2m', target: 200 },   // Ramp to 200
    { duration: '5m', target: 200 },   // Stay at 200 for 5 min
    { duration: '2m', target: 0 },     // Ramp down
  ],
  thresholds: {
    http_req_duration: ['p(95)<500', 'p(99)<1000'],  // P95 < 500ms
    http_req_failed: ['rate<0.01'],                    // < 1% errors
  },
};

export default function () {
  const res = http.get('https://api.example.com/users/me');
  check(res, {
    'status is 200': (r) => r.status === 200,
    'response time < 500ms': (r) => r.timings.duration < 500,
  });
  sleep(1);
}

Types of performance tests:

TypePurposePattern
Load testVerify expected loadRamp to expected peak, hold, ramp down
Stress testFind breaking pointRamp continuously until failures
Soak testFind memory leaksModerate load for hours
Spike testVerify autoscalingSudden traffic spike, then drop

CI/CD Pipeline Design

Q: How should you design a CI/CD pipeline that balances speed and safety?
graph LR COMMIT[Code Commit] --> LINT[Lint + Format - 30s] LINT --> UNIT[Unit Tests - 2min] UNIT --> BUILD[Build + Security Scan - 3min] BUILD --> INT[Integration Tests - 5min] INT --> CONTRACT[Contract Tests - 2min] CONTRACT --> DEPLOY_STG[Deploy Staging] DEPLOY_STG --> E2E[E2E Tests - 10min] E2E --> CANARY[Canary Deploy 5%] CANARY --> |monitor 30min| PROMOTE[Full Deploy]

Key principles:

  • Fast feedback first: Lint and unit tests in < 3 minutes. Developers shouldn't wait.
  • Fail fast: If lint fails, don't run the rest. Saves CI resources.
  • Parallelize where possible: Unit, build, and security scan can run in parallel.
  • Gate on confidence: Each stage increases confidence. Don't skip stages.
  • Automate everything: No manual approval gates except for production deployment (and even that can be automated with canary).
Q: What is shadow traffic (dark launching) and when should you use it?

A: Shadow traffic mirrors production requests to a new version without serving the new version's responses to users. You compare the new version's behavior against the current version.

Use when:

  • Replacing a critical service (payments, search) and need confidence before switching
  • Testing performance under real production load patterns
  • Validating a new ML model against production queries

Caution: Shadow traffic should be read-only. Never shadow-test write operations (you'd create duplicate orders, send duplicate emails). For write operations, use feature flags with careful rollback instead.

Q: What does a staff-level QA strategy look like for a platform team?

A: A staff engineer's QA strategy covers the entire testing portfolio:

LayerResponsibilityFrequency
Unit testsIndividual teamsEvery commit
Integration testsIndividual teams (platform provides tooling)Every commit
Contract testsCross-team (platform enforces)Every commit
E2E testsPlatform team + critical path ownersPre-deploy
Performance testsPlatform teamWeekly + pre-release
Chaos experimentsPlatform team + SREMonthly (automated weekly)
Security scanningPlatform team (automated)Every commit + nightly

The platform team's role: Don't write all the tests — provide the tooling, templates, and CI infrastructure that makes it easy for every team to test well. Standardize on Testcontainers, provide Pact broker, maintain the E2E framework, and run performance/chaos tests centrally.
