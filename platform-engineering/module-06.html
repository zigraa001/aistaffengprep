Retry Patterns

Q: What are the different retry strategies and when should you use each?

A:

StrategyDelay PatternProsConsUse When
Immediate0, 0, 0Fastest recoveryCan overwhelm failing serviceTransient network blips
Fixed interval5s, 5s, 5sPredictableNo backoff pressureKnown recovery time
Exponential backoff1s, 2s, 4s, 8sGives service time to recoverSynchronized retries (thundering herd)Most cases
Exponential + jitter~0.7s, ~2.3s, ~5.1sSpreads load, prevents stormsLess predictableDefault choice
 
import random
import time

def retry_with_backoff(func, max_retries=5, base_delay=1.0, max_delay=60.0):
    """Exponential backoff with full jitter - the standard approach."""
    for attempt in range(max_retries + 1):
        try:
            return func()
        except RetryableError as e:
            if attempt == max_retries:
                raise  # Exhausted retries

            # Exponential backoff with full jitter
            exp_delay = min(base_delay * (2 ** attempt), max_delay)
            actual_delay = random.uniform(0, exp_delay)

            log.warning(f"Attempt {attempt + 1} failed: {e}. "
                        f"Retrying in {actual_delay:.1f}s")
            time.sleep(actual_delay)

AWS's three jitter algorithms:

  • Full jitter: random(0, min(cap, base * 2^attempt)) — Best for reducing contention
  • Equal jitter: min(cap, base * 2^attempt) / 2 + random(0, half) — Guarantees some minimum wait
  • Decorrelated jitter: min(cap, random(base, prev_delay * 3)) — Each retry independent of attempt count
Q: What are retry storms and thundering herd, and how do you prevent them?

A:

Retry storm: When a service fails, all its clients retry simultaneously. Each retry wave adds load, preventing recovery. The failing service gets MORE traffic when it's least able to handle it.

Thundering herd: A specific case where many clients simultaneously wake up and hit the same resource (e.g., cache expiry causes all requests to hit the database).

sequenceDiagram participant C1 as Client 1 participant C2 as Client 2 participant C3 as Client 3 participant S as Service (overloaded) Note over S: Service struggling C1->>S: Request (fails) C2->>S: Request (fails) C3->>S: Request (fails) Note over C1,C3: All retry at 1s C1->>S: Retry 1 (fails) C2->>S: Retry 1 (fails) C3->>S: Retry 1 (fails) Note over C1,C3: All retry at 2s Note over S: Load INCREASES, can't recover!

Mitigations:

  • Jitter — Randomize retry timing so clients don't retry in sync
  • Circuit breaker — Stop retrying entirely when failure rate is high
  • Retry budgets — Limit retries to X% of total requests (e.g., "retry at most 10% of traffic")
  • Server-side rate limiting — Shed excess load with 429 + Retry-After header
  • Client-side rate limiting — Client tracks recent failures and reduces request rate
class RetryBudget:
    """Limit retries to a percentage of total requests."""
    def __init__(self, budget_ratio=0.1, window_seconds=60):
        self.budget_ratio = budget_ratio
        self.window = window_seconds
        self.requests = []
        self.retries = []

    def can_retry(self) -> bool:
        now = time.time()
        cutoff = now - self.window
        self.requests = [t for t in self.requests if t > cutoff]
        self.retries = [t for t in self.retries if t > cutoff]

        if len(self.requests) == 0:
            return True
        return len(self.retries) / len(self.requests) < self.budget_ratio

    def record_request(self):
        self.requests.append(time.time())

    def record_retry(self):
        self.retries.append(time.time())

Circuit Breaker

Q: How does the circuit breaker pattern work?

A: A circuit breaker monitors failures and "trips" (opens) when the failure rate exceeds a threshold, preventing further requests to a failing service.

stateDiagram-v2 [*] --> Closed Closed --> Open: Failure threshold exceeded Open --> HalfOpen: Timeout expires HalfOpen --> Closed: Probe succeeds HalfOpen --> Open: Probe fails Closed --> Closed: Success / Failure below threshold

States:

  • Closed — Normal operation. Requests pass through. Failures are counted.
  • Open — Requests fail immediately without calling the downstream service. Returns fallback or error.
  • Half-Open — After a timeout, allow one probe request. If it succeeds, close the circuit. If it fails, reopen.
import time
from enum import Enum

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(self, failure_threshold=5, recovery_timeout=30, half_open_max=1):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.half_open_max = half_open_max
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.last_failure_time = 0
        self.half_open_calls = 0

    def call(self, func, fallback=None):
        if self.state == CircuitState.OPEN:
            if time.time() - self.last_failure_time > self.recovery_timeout:
                self.state = CircuitState.HALF_OPEN
                self.half_open_calls = 0
            else:
                if fallback:
                    return fallback()
                raise CircuitOpenError("Circuit is open")

        if self.state == CircuitState.HALF_OPEN:
            if self.half_open_calls >= self.half_open_max:
                raise CircuitOpenError("Circuit half-open, max probes reached")
            self.half_open_calls += 1

        try:
            result = func()
            self._on_success()
            return result
        except Exception as e:
            self._on_failure()
            raise

    def _on_success(self):
        self.failure_count = 0
        self.state = CircuitState.CLOSED

    def _on_failure(self):
        self.failure_count += 1
        self.last_failure_time = time.time()
        if self.failure_count >= self.failure_threshold:
            self.state = CircuitState.OPEN

Production libraries: Use resilience4j (Java), Polly (.NET), pybreaker (Python), or gobreaker (Go) instead of rolling your own. They handle edge cases like concurrent access, metrics, and configuration.

Bulkhead & Isolation

Q: What is the bulkhead pattern and how does it prevent cascading failures?

A: Named after ship compartments, the bulkhead pattern isolates resources so a failure in one area doesn't sink the whole system.

graph TB subgraph "Without Bulkhead" API1[All API calls] --> POOL[Shared Thread Pool - 100 threads] POOL --> SvcA[Service A - slow] POOL --> SvcB[Service B - healthy] POOL --> SvcC[Service C - healthy] end subgraph "With Bulkhead" API2[API calls] --> PA[Pool A - 30 threads] API2 --> PB[Pool B - 30 threads] API2 --> PC[Pool C - 30 threads] PA --> SvcA2[Service A - slow] PB --> SvcB2[Service B - healthy] PC --> SvcC2[Service C - healthy] end

Without bulkhead: Service A becomes slow, exhausts all 100 threads in the shared pool. Services B and C (which are healthy) can't be reached because there are no threads left. One slow dependency takes down everything.

With bulkhead: Service A's pool (30 threads) fills up, but Services B and C still have their dedicated pools. The failure is contained.

Implementation approaches:

  • Thread pool bulkhead — Separate thread pools per dependency (Java/C#)
  • Semaphore bulkhead — Limit concurrent calls per dependency with semaphores
  • Process bulkhead — Separate processes/containers per dependency
  • Connection pool bulkhead — Separate HTTP connection pools per downstream service
import asyncio

class SemaphoreBulkhead:
    """Limit concurrent calls to a dependency."""
    def __init__(self, name: str, max_concurrent: int):
        self.name = name
        self.semaphore = asyncio.Semaphore(max_concurrent)
        self.rejected = 0

    async def execute(self, func):
        if self.semaphore.locked():
            self.rejected += 1
            raise BulkheadFullError(f"Bulkhead '{self.name}' is full")
        async with self.semaphore:
            return await func()

# Usage
payment_bulkhead = SemaphoreBulkhead("payment-service", max_concurrent=20)
inventory_bulkhead = SemaphoreBulkhead("inventory-service", max_concurrent=50)

result = await payment_bulkhead.execute(lambda: call_payment_service(order))

Backpressure & Load Shedding

Q: What is backpressure and how do you implement it?

A: Backpressure is a mechanism for a consumer to signal to a producer that it can't keep up, causing the producer to slow down or buffer.

Without backpressure: Producer sends faster than consumer can process → memory grows unbounded → OOM crash or dropped messages.

Backpressure strategies:

  • Drop — Discard new messages when buffer is full (good for metrics/logs where latest matters most)
  • Buffer and bound — Queue messages up to a limit, reject new ones when full (HTTP 503)
  • Slow the producer — TCP flow control, Kafka consumer lag triggering producer throttling
  • Sample — Process every Nth message (good for analytics at high volume)

Example: bounded queue with backpressure:

import asyncio

class BoundedProcessor:
    def __init__(self, max_queue_size=1000, workers=10):
        self.queue = asyncio.Queue(maxsize=max_queue_size)
        self.workers = workers

    async def submit(self, item):
        try:
            self.queue.put_nowait(item)
        except asyncio.QueueFull:
            raise BackpressureError("Queue full, slow down")

    async def worker(self):
        while True:
            item = await self.queue.get()
            await self.process(item)
            self.queue.task_done()
Q: What is load shedding and how does it differ from rate limiting?

A:

AspectRate LimitingLoad Shedding
TriggerPer-client request countServer resource utilization
GoalFair access across clientsProtect server from overload
When activeAlways (proactive)Under stress (reactive)
Decision basisClient identity + request countCPU, memory, queue depth, latency

Load shedding implementation:

async def load_shedding_middleware(request, call_next):
    # Check server health
    cpu_usage = get_cpu_usage()
    queue_depth = get_request_queue_depth()
    p99_latency = get_p99_latency()

    if cpu_usage > 85 or queue_depth > 1000 or p99_latency > 5000:
        # Shed load: prioritize by request importance
        priority = get_request_priority(request)
        if priority == "low":
            return Response(status_code=503, headers={"Retry-After": "30"})
        # High-priority requests still get through

    return await call_next(request)

Priority-based shedding: Not all requests are equal. During overload, shed analytics requests before payment requests. Categorize requests into priority tiers and shed low-priority first.

Idempotency & Timeouts

Q: How do you implement idempotency keys for distributed operations?

A: Idempotency keys ensure that retrying a request doesn't cause duplicate side effects.

from fastapi import FastAPI, Request, HTTPException
import redis
import json

app = FastAPI()
redis_client = redis.Redis()

@app.post("/transfers")
async def create_transfer(request: Request, transfer: TransferRequest):
    idempotency_key = request.headers.get("Idempotency-Key")
    if not idempotency_key:
        raise HTTPException(400, "Idempotency-Key header required")

    cache_key = f"idempotency:{idempotency_key}"

    # Check for existing result
    cached = redis_client.get(cache_key)
    if cached:
        cached_data = json.loads(cached)
        # Verify request body matches (prevent key reuse for different ops)
        if cached_data["request_hash"] != hash_request(transfer):
            raise HTTPException(422, "Idempotency key already used for different request")
        return cached_data["response"]

    # Acquire lock to prevent concurrent execution of same key
    lock = redis_client.lock(f"lock:{cache_key}", timeout=30)
    if not lock.acquire(blocking=False):
        raise HTTPException(409, "Request with this idempotency key is in progress")

    try:
        result = await execute_transfer(transfer)
        # Cache result for 24 hours
        redis_client.setex(cache_key, 86400, json.dumps({
            "request_hash": hash_request(transfer),
            "response": result,
        }))
        return result
    finally:
        lock.release()

Key design decisions:

  • TTL: Cache results for 24-48 hours. Long enough for clients to retry, short enough to not bloat storage.
  • Locking: Prevent concurrent execution of the same idempotency key (race condition where two retries run simultaneously).
  • Request matching: Verify the request body matches the original. Reject reuse of keys for different operations.
  • Key generation: Client generates UUIDs. Don't use deterministic keys (like user_id + amount) — a user may legitimately make two identical transfers.
Q: How should you configure timeouts in a microservices architecture?

A: Timeouts have three layers, and they must be configured in the right order:

Timeout TypeWhat It ControlsTypical Value
Connect timeoutTime to establish TCP connection1-5 seconds
Read timeoutTime waiting for response after request sent5-30 seconds
Total timeoutEntire operation including retries30-120 seconds

The golden rule: Upstream timeouts must be LONGER than downstream timeouts. If Service A calls B calls C:

A's timeout (10s) > B's timeout (5s) > C's processing time (2s)

If A's timeout is shorter than B's, A may give up and retry while B is still waiting for C. Now B has two in-flight requests consuming resources for a response nobody wants.

Deadline propagation: Pass a deadline (absolute time) through the call chain rather than independent timeouts:

# Deadline propagation pattern
async def service_a_handler(request):
    deadline = time.time() + 10  # 10s from now
    result = await call_service_b(request, deadline=deadline)
    return result

async def service_b_handler(request, deadline):
    remaining = deadline - time.time()
    if remaining <= 0:
        raise TimeoutError("Deadline already passed")
    result = await call_service_c(request, timeout=remaining - 0.5)  # Reserve buffer
    return result

gRPC has built-in deadline propagation. For REST, pass it as a header (e.g., X-Request-Deadline).

Health Checks & Graceful Degradation

Q: What is the difference between liveness and readiness probes?

A:

ProbeQuestion It AnswersFailure ActionChecks
Liveness"Is the process running correctly?"Kill and restart the containerProcess is alive, not deadlocked
Readiness"Can this instance serve traffic?"Remove from load balancerDependencies connected, warmed up
Startup"Has the app finished starting?"Don't check liveness yetInitialization complete

Critical mistake: Making liveness probes check database connectivity. If the database goes down, ALL pods restart, making things worse. Liveness should check if the process is healthy (not deadlocked, not OOM). Readiness should check if the instance can serve traffic (database connected, cache warmed).

# Correct health check implementation
@app.get("/healthz")  # Liveness
async def liveness():
    return {"status": "alive"}  # Simple - just proves the process responds

@app.get("/readyz")  # Readiness
async def readiness():
    checks = {
        "database": await check_db_connection(),
        "cache": await check_redis_connection(),
    }
    all_healthy = all(checks.values())
    return JSONResponse(
        status_code=200 if all_healthy else 503,
        content={"ready": all_healthy, "checks": checks}
    )
Q: What does graceful degradation look like in practice?

A: Graceful degradation means providing reduced functionality rather than a complete failure when dependencies are unhealthy.

Examples:

Dependency DownHard FailureGraceful Degradation
Recommendation serviceProduct page fails to loadShow products without recommendations
User profile serviceDashboard blankShow cached profile, mark as "may be stale"
Search serviceSearch brokenFall back to browsing by category
Payment processorCheckout failsQueue the payment, confirm asynchronously

Implementation pattern:

async def get_product_page(product_id: str):
    product = await product_service.get(product_id)  # Required - fail if unavailable

    # Optional enrichments - degrade gracefully
    try:
        recommendations = await recommendation_service.get(product_id)
    except (TimeoutError, ServiceUnavailableError):
        recommendations = []  # Degrade: no recommendations

    try:
        reviews = await review_service.get(product_id)
    except (TimeoutError, ServiceUnavailableError):
        reviews = {"summary": "Reviews temporarily unavailable", "items": []}

    return ProductPage(
        product=product,
        recommendations=recommendations,
        reviews=reviews,
    )

Staff-level principle: Classify every dependency as critical (page can't render without it) or optional (page degrades without it). Design fallbacks for optional dependencies from the start, not as an afterthought.

Q: How do all these resilience patterns work together?

A: In a production system, you layer multiple patterns:

graph LR REQ[Request] --> TIMEOUT[Timeout 5s] TIMEOUT --> BULKHEAD[Bulkhead max=20] BULKHEAD --> CB[Circuit Breaker] CB --> RETRY[Retry 3x with jitter] RETRY --> SVC[Downstream Service] CB --> |Circuit Open| FALLBACK[Fallback Response]

Order matters:

  1. Timeout — Outermost: limits total time spent
  2. Bulkhead — Limits concurrent calls (prevents resource exhaustion)
  3. Circuit Breaker — Stops calling when failure rate is high
  4. Retry — Innermost: retries individual failed calls
# Composed resilience in Python
async def call_payment_service(request):
    async with timeout(5.0):                              # 1. Timeout
        async with payment_bulkhead.acquire():            # 2. Bulkhead
            if payment_circuit.is_open():                 # 3. Circuit breaker
                return fallback_response()
            return await retry(                           # 4. Retry
                lambda: http_client.post("/payments", request),
                max_retries=3,
                on_failure=payment_circuit.record_failure,
                on_success=payment_circuit.record_success,
            )

Platform team's role: Provide these as a composable middleware library so individual service teams don't reinvent them. Standardize defaults (3 retries, 5s timeout, 50% failure threshold for circuit breaker) and let teams override for their specific needs.
