Three Pillars of Observability

Q: What are the three pillars of observability and their limitations?
PillarWhat It CapturesBest ForLimitations
LogsDiscrete events with contextDebugging specific requests, audit trailsExpensive to store at scale, hard to correlate across services
MetricsAggregated numerical measurementsDashboards, alerting, trendsLoss of individual request detail, cardinality explosions
TracesRequest flow across servicesLatency analysis, dependency mappingSampling loses data, complex to set up

The limitation beyond three pillars: These pillars capture known-unknowns. For unknown-unknowns (novel failure modes you didn't instrument for), you need high-cardinality event data (wide events with many fields) that can be queried ad-hoc. Tools like Honeycomb focus on this approach — store rich events and query them after the fact, rather than pre-aggregating into metrics.

Q: How should you implement structured logging?
import structlog
import uuid

# Configure structured logging (JSON output)
structlog.configure(
    processors=[
        structlog.processors.TimeStamper(fmt="iso"),
        structlog.processors.JSONRenderer(),
    ]
)
logger = structlog.get_logger()

# Middleware: inject correlation ID
async def correlation_middleware(request, call_next):
    correlation_id = request.headers.get("X-Correlation-ID", str(uuid.uuid4()))
    # Bind to all log entries in this request
    structlog.contextvars.bind_contextvars(
        correlation_id=correlation_id,
        method=request.method,
        path=request.url.path,
    )
    response = await call_next(request)
    response.headers["X-Correlation-ID"] = correlation_id
    logger.info("request_completed", status=response.status_code)
    return response

# Usage in handlers
async def create_order(order_data):
    logger.info("creating_order", customer_id=order_data.customer_id)
    order = await db.create(order_data)
    logger.info("order_created", order_id=order.id, total=order.total)

Output:

{"event": "order_created", "correlation_id": "abc-123", "order_id": "ord-456", "total": 99.99, "method": "POST", "path": "/orders", "timestamp": "2024-01-15T10:30:00Z"}

Key rules:

  • Always JSON format (not free-text) for machine parsing
  • Always include correlation ID for tracing across services
  • Log events, not sentences ("order_created" not "An order was created successfully")
  • Include relevant context (IDs, amounts) but NEVER log secrets, PII, or credentials

SLIs, SLOs, and Error Budgets

Q: What are SLIs, SLOs, SLAs, and error budgets?
TermDefinitionExample
SLI (Service Level Indicator)A metric that measures service qualityRequest latency P99, error rate, availability
SLO (Service Level Objective)Target value for an SLI"P99 latency < 500ms" or "99.9% availability"
SLA (Service Level Agreement)Contractual commitment with consequences"99.9% uptime or customer gets credits"
Error BudgetAllowed amount of unreliability99.9% SLO = 0.1% error budget = 43.8 min/month

Error budget calculation:

SLO: 99.9% availability
Error budget per month: 30 days × 24 hours × 60 min × 0.1% = 43.2 minutes

SLO: 99.95% availability
Error budget per month: 30 × 24 × 60 × 0.05% = 21.6 minutes

SLO: 99.99% availability
Error budget per month: 30 × 24 × 60 × 0.01% = 4.32 minutes

How error budgets drive decisions:

  • Budget remaining > 50%: Ship features aggressively, take risks
  • Budget remaining 20-50%: Normal development, be careful with risky changes
  • Budget remaining < 20%: Slow down feature work, focus on reliability
  • Budget exhausted: Freeze deployments except reliability improvements

Distributed Tracing & OpenTelemetry

Q: How does OpenTelemetry work and how do you instrument a service?
# OpenTelemetry setup (Python)
from opentelemetry import trace
from opentelemetry.sdk.trace import TracerProvider
from opentelemetry.sdk.trace.export import BatchSpanProcessor
from opentelemetry.exporter.otlp.proto.grpc.trace_exporter import OTLPSpanExporter

# Setup
provider = TracerProvider()
processor = BatchSpanProcessor(OTLPSpanExporter(endpoint="http://otel-collector:4317"))
provider.add_span_processor(processor)
trace.set_tracer_provider(provider)
tracer = trace.get_tracer("order-service")

# Instrument a function
@tracer.start_as_current_span("create_order")
async def create_order(order_data):
    span = trace.get_current_span()
    span.set_attribute("order.customer_id", order_data.customer_id)
    span.set_attribute("order.item_count", len(order_data.items))

    # Child span for database operation
    with tracer.start_as_current_span("db.insert_order"):
        order = await db.insert(order_data)
        span.set_attribute("order.id", order.id)

    # Child span for event publishing
    with tracer.start_as_current_span("kafka.publish_event"):
        await kafka.publish("order-events", order.to_event())

    return order

Trace propagation: When Service A calls Service B, the trace context (trace ID, span ID) is propagated via HTTP headers (traceparent for W3C Trace Context). This links spans across services into a single trace.

Alerting Philosophy

Q: What makes a good alerting strategy vs a bad one?

Alert on symptoms, not causes:

Bad Alert (cause)Good Alert (symptom)
CPU usage > 80%P99 latency > 500ms for 5 minutes
Disk usage > 90%Error rate > 1% for 5 minutes
Pod restartedSLO burn rate > 10x for 1 hour

Alert fatigue is the #1 enemy: If on-call engineers get 50 alerts/day and 45 are noise, they'll ignore all of them. Every alert should require human action. If it doesn't, it's a log entry, not an alert.

Google's alerting principles (from SRE book):

  • Every alert should be actionable
  • Every alert should require intelligence (not automatable)
  • Every alert should be about a novel problem (not a known ongoing issue)

Multi-window, multi-burn-rate alerts: Instead of a simple threshold, alert when the error budget burn rate is unsustainable. "At the current error rate, we'll exhaust the monthly error budget in 6 hours" is more actionable than "error rate is 0.5%".

Q: How should incident management work at a platform team?

Severity levels:

SeverityDefinitionResponse TimeExample
SEV1Complete service outage, data loss15 minutesProduction database down
SEV2Major feature degraded, SLO breached1 hourPayments failing for 10% of users
SEV3Minor feature degraded, workaround exists4 hoursSearch results slightly stale
SEV4Cosmetic issue, no user impactNext business dayDashboard chart rendering incorrectly

Postmortem template (blameless):

  1. Summary: What happened, impact, duration
  2. Timeline: When detected, who responded, key actions taken
  3. Root cause: Technical root cause (what broke)
  4. Contributing factors: What made it worse or delayed recovery
  5. Action items: Concrete steps to prevent recurrence (assigned, with deadlines)
  6. Lessons learned: What worked well, what didn't

Staff-level incident role: During a SEV1, the staff engineer acts as the incident commander — coordinating response, making decisions about mitigations vs fixes, communicating to stakeholders, and ensuring the postmortem happens.

Q: What is the cost of observability at scale and how do you manage it?

A: Observability costs can exceed the infrastructure costs of the services being observed. At scale:

  • Logs: A service logging 1KB per request at 10K rps generates 864 GB/day. At $0.50/GB ingestion, that's $432/day per service.
  • Metrics: High-cardinality metrics (per-user, per-request-ID) can create millions of time series, overwhelming Prometheus.
  • Traces: Storing every trace at high-throughput services is prohibitively expensive.

Cost management strategies:

  • Sampling: Trace 1-10% of requests. Always trace errors and slow requests (head-based + tail-based sampling).
  • Log levels: INFO in production, DEBUG only when investigating. Use dynamic log levels to turn up verbosity temporarily.
  • Aggregation: Pre-aggregate metrics instead of storing raw events. P50/P95/P99 summaries instead of individual latency values.
  • Retention policies: Keep detailed data for 7 days, aggregated data for 30 days, summaries for 1 year.
  • Cardinality management: Don't use unbounded values (user IDs, request IDs) as metric labels.
