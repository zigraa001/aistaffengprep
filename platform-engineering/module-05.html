Kafka Deep Dive

Q: How do Kafka partitions, consumer groups, and offsets work together?

A: Kafka's core model is a distributed append-only log partitioned across brokers.

graph TB P[Producer] --> T[Topic: order-events] T --> P0[Partition 0] T --> P1[Partition 1] T --> P2[Partition 2] subgraph "Consumer Group A (Order Service)" C1[Consumer 1] --> P0 C2[Consumer 2] --> P1 C2 --> P2 end subgraph "Consumer Group B (Analytics)" C3[Consumer 3] --> P0 C3 --> P1 C3 --> P2 end

Key concepts:

  • Partitions — A topic is split into partitions. Each partition is an ordered, immutable log. Messages within a partition have a sequential offset.
  • Consumer groups — Each group gets all messages, but partitions are distributed among consumers in the group. Two groups reading the same topic each get all messages independently.
  • Offsets — Each consumer tracks its position (offset) in each partition. Committing offsets is how Kafka knows what's been processed.
  • Partition key — Determines which partition a message goes to. Messages with the same key always go to the same partition, guaranteeing ordering within that key.
# Kafka producer with partition key
from confluent_kafka import Producer

producer = Producer({'bootstrap.servers': 'kafka:9092'})

def publish_order_event(order_id: str, event_type: str, data: dict):
    event = {
        "type": event_type,
        "order_id": order_id,
        "data": data,
        "timestamp": datetime.utcnow().isoformat()
    }
    # Use order_id as key: all events for same order go to same partition
    producer.produce(
        topic='order-events',
        key=order_id.encode(),
        value=json.dumps(event).encode(),
    )
    producer.flush()

Scaling rule: Max parallelism = number of partitions. If you have 6 partitions, at most 6 consumers in a group can work simultaneously. The seventh consumer would be idle. Start with more partitions than you need (but not too many — each partition has overhead).

Q: What are the delivery semantics (at-most-once, at-least-once, exactly-once) and which should you use?

A:

SemanticHowRiskUse Case
At-most-onceCommit offset before processingMessage lost if processing failsMetrics, logs (loss tolerable)
At-least-onceCommit offset after processingDuplicates if crash after process but before commitMost production workloads
Exactly-onceKafka transactions or idempotent consumersHigher latency, complexityFinancial, billing systems
 
# At-least-once consumer pattern
from confluent_kafka import Consumer

consumer = Consumer({
    'bootstrap.servers': 'kafka:9092',
    'group.id': 'order-processor',
    'enable.auto.commit': False,  # Manual commit for at-least-once
})
consumer.subscribe(['order-events'])

while True:
    msg = consumer.poll(1.0)
    if msg is None:
        continue

    try:
        process_message(msg)           # Process first
        consumer.commit(message=msg)   # Then commit offset
    except Exception as e:
        log.error(f"Processing failed: {e}")
        # Don't commit - message will be redelivered

Staff-level recommendation: Use at-least-once for nearly everything, and make consumers idempotent. Exactly-once semantics in Kafka (via transactions) add significant complexity and latency. It's almost always simpler to handle duplicates in the consumer than to pay the cost of exactly-once in the broker.

Q: How does Kafka achieve exactly-once semantics when needed?

A: Kafka's exactly-once has two layers:

1. Idempotent producer: Prevents duplicate messages from producer retries. Each producer gets a Producer ID, and each message gets a sequence number. The broker deduplicates.

producer = Producer({
    'bootstrap.servers': 'kafka:9092',
    'enable.idempotence': True,  # Prevents duplicate produces
})

2. Transactions: For read-process-write patterns where you consume from one topic, process, and produce to another topic. The offset commit and the output write are atomic.

// Java - Kafka transactional consumer-producer
producer.initTransactions();

while (true) {
    ConsumerRecords records = consumer.poll(Duration.ofMillis(100));
    producer.beginTransaction();
    try {
        for (ConsumerRecord record : records) {
            ProducerRecord output = process(record);
            producer.send(output);
        }
        // Atomically commit consumer offsets + produced messages
        producer.sendOffsetsToTransaction(
            currentOffsets, consumer.groupMetadata()
        );
        producer.commitTransaction();
    } catch (Exception e) {
        producer.abortTransaction();
    }
}

Limitation: Exactly-once only works within Kafka. If your consumer writes to an external database, you still need application-level idempotency. Kafka can't make your database write atomic with a Kafka offset commit.

Advanced Patterns

Q: What is event sourcing and when should you use it?

A: Event sourcing stores every state change as an immutable event. Current state is derived by replaying all events from the beginning.

graph LR subgraph "Traditional: Store Current State" DB1[Orders Table: Order 123 - Status: Shipped, Total: $99] end subgraph "Event Sourcing: Store Events" E1[OrderCreated total=$99] --> E2[PaymentReceived $99] E2 --> E3[ItemPacked] E3 --> E4[OrderShipped tracking=XYZ] end

Benefits:

  • Complete audit trail — Every change is recorded and immutable
  • Time travel — Reconstruct state at any point in time
  • Event replay — Rebuild read models, fix bugs by replaying with corrected logic
  • Decoupling — Multiple consumers can build different views from the same events

Costs:

  • Complexity — Much harder to implement than CRUD
  • Event schema evolution — Old events must remain parseable as schemas change
  • Eventual consistency — Read models are updated asynchronously
  • Storage growth — Events accumulate forever (snapshots mitigate this)

Use when: Financial/banking (audit requirements), collaborative editing (Git, Google Docs), compliance-heavy domains. Don't use for: Simple CRUD apps, small teams, domains without audit requirements.

Q: What is CQRS and how does it relate to event sourcing?

A: CQRS (Command Query Responsibility Segregation) separates read and write models. Writes go to one model, reads come from a different, optimized model.

graph TB Client --> |Commands: CreateOrder, CancelOrder| WS[Write Service] Client --> |Queries: GetOrder, SearchOrders| RS[Read Service] WS --> |Events| EB[Event Bus / Kafka] WS --> WDB[(Write DB - normalized)] EB --> PROJ[Projection Service] PROJ --> RDB[(Read DB - denormalized)] RS --> RDB

CQRS without event sourcing: You can use CQRS with a traditional database. Write to a normalized DB, publish change events, and maintain a denormalized read DB (e.g., Elasticsearch for search, Redis for dashboards).

CQRS with event sourcing: Events ARE the write model. Projections consume events and build read models. This is the full pattern but adds significant complexity.

Staff-level take: CQRS is often over-applied. You don't need it for most services. It's valuable when read and write patterns are fundamentally different — e.g., writes are simple command processing but reads need complex aggregations, full-text search, or graph traversals. Start with a single model and split only when performance or complexity demands it.

Q: How do you design Kafka topics and choose partition keys?

A:

Topic design principles:

  • One topic per event type vs one topic per entity — Generally prefer per-entity (e.g., orders topic with event types in the payload) for simpler consumer management
  • Naming convention: <domain>.<entity>.<version> e.g., ecommerce.orders.v1
  • Avoid too many topics — Each topic has overhead (metadata, file handles). 1,000 topics is fine; 100,000 is problematic.

Partition key selection:

ScenarioGood Partition KeyWhy
Order eventsorder_idAll events for one order in sequence
User activityuser_idPer-user ordering for session replay
IoT sensor datadevice_idPer-device ordering for time-series
Multi-tenant SaaStenant_idPer-tenant isolation and ordering

Watch out for hot partitions: If one key has vastly more events than others (e.g., a single large tenant), that partition becomes a bottleneck. Solutions: composite keys (tenant_id + shard_key), or dedicated topics for high-volume tenants.

Q: How do dead letter queues (DLQ) work and what's the best practice for handling them?

A: A DLQ captures messages that can't be processed after exhausting retries. Without a DLQ, poison messages block the consumer forever.

graph LR T[Main Topic] --> C[Consumer] C --> |Success| COMMIT[Commit Offset] C --> |Failure| RETRY[Retry up to 3x] RETRY --> |Still failing| DLQ[Dead Letter Topic] DLQ --> ALERT[Alert Team] DLQ --> REVIEW[Manual Review Dashboard] REVIEW --> |Fixed| REPLAY[Replay to Main Topic] 
async def consume_with_dlq(msg, max_retries=3):
    retry_count = int(msg.headers().get('retry-count', 0))

    try:
        await process_message(msg)
    except RetryableError as e:
        if retry_count < max_retries:
            # Re-publish with incremented retry count
            producer.produce(
                topic=msg.topic(),
                key=msg.key(),
                value=msg.value(),
                headers={'retry-count': str(retry_count + 1)},
            )
        else:
            # Send to DLQ
            producer.produce(
                topic=f"{msg.topic()}.dlq",
                key=msg.key(),
                value=msg.value(),
                headers={
                    'original-topic': msg.topic(),
                    'failure-reason': str(e),
                    'failed-at': datetime.utcnow().isoformat(),
                },
            )
            alert_team(f"Message sent to DLQ: {e}")
    except NonRetryableError as e:
        # Immediately DLQ (e.g., schema validation failure)
        producer.produce(topic=f"{msg.topic()}.dlq", ...)

DLQ best practices:

  • Include metadata: original topic, failure reason, timestamp, retry count
  • Build a dashboard to view and replay DLQ messages
  • Alert on DLQ depth > 0 (every DLQ message deserves attention)
  • Distinguish retryable vs non-retryable errors — don't retry schema validation failures
Q: What is a schema registry and why do you need one?

A: A schema registry is a centralized store for message schemas (Avro, Protobuf, JSON Schema). It ensures producers and consumers agree on the message format.

Without schema registry:

  • Producer changes a field name → all consumers break at runtime
  • No way to discover what events look like without reading code
  • Breaking changes are only caught in production

With schema registry (e.g., Confluent Schema Registry):

  • Schemas are versioned and immutable
  • Compatibility checks happen at publish time (backward, forward, full)
  • Breaking changes are rejected before they reach production
  • Consumers auto-discover schema from message metadata

Compatibility modes:

ModeRuleAllows
BackwardNew schema can read old dataAdd optional fields, remove fields
ForwardOld schema can read new dataAdd fields, remove optional fields
FullBoth directionsAdd/remove optional fields only

Recommendation: Start with backward compatibility. This means consumers can be upgraded first (they can read both old and new messages), and producers upgraded after. This is the safest upgrade order.

Q: How do you monitor consumer lag and why does it matter?

A: Consumer lag is the difference between the latest offset in a partition and the consumer's committed offset. It tells you how far behind a consumer is.

Why it matters:

  • Growing lag = consumer can't keep up with producers = data freshness degradation
  • Sudden lag spike = consumer is stuck, crashed, or processing a poison message
  • Lag during peak hours = capacity planning needed

Monitoring tools:

  • Burrow (LinkedIn) — Dedicated consumer lag monitoring with automatic lag evaluation
  • Kafka Exporter + Prometheus — Export lag metrics to Prometheus/Grafana
  • Built-in: kafka-consumer-groups.sh --describe --group my-group

Alerting thresholds:

  • Warning: Lag > 10,000 messages for > 5 minutes
  • Critical: Lag increasing continuously for > 15 minutes
  • Emergency: Lag > 1 hour of messages (based on production rate)

Fixing lag: Add more consumers (up to partition count), optimize processing logic, increase partition count, or address the root cause (slow downstream dependency, poison messages).

Q: How do you handle message replay and reprocessing?

A: Kafka's log retention means you can reprocess historical messages. This is one of Kafka's killer features vs traditional message queues.

Replay scenarios:

  • Bug fix: Consumer had a bug, fix it, replay from before the bug was introduced
  • New consumer: A new service needs historical data, replay from the beginning
  • Rebuild read model: Your search index is corrupted, rebuild from events
# Reset consumer group to replay from a specific timestamp
# CLI:
# kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
#   --group order-processor \
#   --topic order-events \
#   --reset-offsets --to-datetime 2024-01-10T00:00:00.000 \
#   --execute

# Programmatic: seek to timestamp
from confluent_kafka import Consumer, TopicPartition

consumer = Consumer({...})
consumer.subscribe(['order-events'])

# Seek all partitions to a specific timestamp
partitions = consumer.assignment()
timestamps = {tp: 1705017600000 for tp in partitions}  # epoch ms
offsets = consumer.offsets_for_times(timestamps)
for tp_offset in offsets:
    consumer.seek(tp_offset)

Replay considerations:

  • Consumers MUST be idempotent for replay to work safely
  • Consider rate limiting replay to avoid overwhelming downstream systems
  • Use a separate consumer group for replay to not affect live processing
  • Set retention long enough for your replay needs (7 days minimum, often 30+)
Q: How do you choose the right message broker for your use case?

A: Decision framework:

  • Need event streaming with replay + high throughput? → Kafka
  • Need flexible routing (headers, patterns, priorities)? → RabbitMQ
  • Need simple decoupling in AWS with zero ops? → SQS/SNS
  • Need enterprise features in Azure (sessions, transactions)? → Azure Service Bus
  • Need managed Kafka without the ops burden? → Confluent Cloud, MSK, or Event Hubs (Kafka protocol)

Staff-level nuance: The broker choice is often less important than how you use it. A well-designed system on SQS will outperform a poorly designed system on Kafka. Focus on: idempotent consumers, proper error handling, DLQs, monitoring, and schema management. These patterns work regardless of the broker.

Multi-broker reality: Many organizations use multiple brokers. Kafka for event streaming, SQS for simple task queues, and RabbitMQ for legacy systems. This is fine — each has strengths. The platform team's job is to provide standardized client libraries and observability across all of them.

Q: What is the outbox pattern and why is it essential for event-driven systems?

A: The outbox pattern solves the dual-write problem: when you need to update a database AND publish an event, and you can't do both atomically.

The problem:

# DANGEROUS: dual write - not atomic
async def create_order(order):
    await db.insert(order)          # Step 1: DB write succeeds
    await kafka.publish(order_event) # Step 2: Kafka publish fails!
    # Now DB has the order but no event was published

The solution (outbox pattern):

# SAFE: write event to outbox table in same DB transaction
async def create_order(order):
    async with db.transaction():
        await db.insert("orders", order)
        await db.insert("outbox", {
            "id": uuid4(),
            "aggregate_type": "Order",
            "aggregate_id": order.id,
            "event_type": "OrderCreated",
            "payload": json.dumps(order_event),
        })

# Separate process polls outbox and publishes to Kafka
async def outbox_relay():
    while True:
        events = await db.fetch("SELECT * FROM outbox WHERE published = false LIMIT 100")
        for event in events:
            await kafka.publish(event.event_type, event.payload)
            await db.execute("UPDATE outbox SET published = true WHERE id = $1", event.id)
        await asyncio.sleep(1)

Alternative: Change Data Capture (CDC) with Debezium reads the database transaction log and publishes changes to Kafka. This avoids the outbox table entirely but requires more infrastructure.
