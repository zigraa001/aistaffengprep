Mock Interview 1: API Integration Design (45 min)

Interview Prompt & Interviewer Guide

Prompt: "We're building a unified API layer for our e-commerce platform. We have 8 backend services (users, products, orders, payments, shipping, inventory, reviews, search). Currently, our mobile and web apps make 15-20 API calls to render the home page. Design the API layer."

Interviewer follow-up questions:

  1. "How would you handle authentication across all services?"
  2. "What happens when the review service is slow? How do you prevent it from affecting the whole page?"
  3. "How would you handle API versioning when mobile apps can't be force-updated?"
  4. "What's your strategy for rate limiting across different client types?"
  5. "How would you test this before going to production?"

What the interviewer is evaluating:

  • Does the candidate ask clarifying questions before designing?
  • Do they consider the BFF pattern, GraphQL federation, or API gateway?
  • Do they address failure modes (circuit breaker, timeouts)?
  • Do they discuss operational concerns (monitoring, deployment)?
  • Do they consider the mobile vs web differences?
Sample Strong Answer (Key Points)

Approach: API Gateway + BFF pattern with optional GraphQL for complex queries.

  1. Clarifies: "What's the mobile vs web traffic split? Are there different data needs? What's the latency budget for the home page?"
  2. Proposes API Gateway for auth, rate limiting, routing. Separate Web BFF and Mobile BFF for different aggregation needs.
  3. Addresses failures: "Reviews being slow shouldn't block the product page. I'd set a 200ms timeout on the reviews call with a circuit breaker. If it trips, show 'Reviews temporarily unavailable' and serve the page with everything else."
  4. Versioning: "URL-based versioning (/v1, /v2) for the public API since mobile apps can't be force-updated. Internal services use header-based versioning. We'd support v(N) and v(N-1) with a 12-month deprecation policy."
  5. Testing: "Contract tests between BFF and each service, load test the aggregation layer at 3x peak, chaos test with Toxiproxy to simulate slow/failing services."
Sample Weak Answer (Anti-patterns)
  • "Let's just put everything behind an API gateway and route traffic." — No discussion of aggregation, BFF, or client-specific needs.
  • "We'll use GraphQL for everything." — No discussion of tradeoffs (complexity, caching difficulty, N+1 at resolver level).
  • No mention of failure handling, timeouts, or circuit breakers.
  • No consideration of mobile vs web differences.
  • Jumps to solution without asking clarifying questions.

Mock Interview 2: Testing Strategy Deep Dive (45 min)

Interview Prompt & Interviewer Guide

Prompt: "You're the staff engineer on a team building a new notification service. It consumes events from Kafka, applies user preference rules, and delivers notifications via email, SMS, push, and in-app. Design the testing strategy."

Follow-up questions:

  1. "How do you test the Kafka consumer? Do you use a real Kafka or mock it?"
  2. "How do you test that email/SMS actually gets delivered?"
  3. "What's your approach to testing user preference rules? There are 50+ combinations."
  4. "How do you prevent flaky tests in this system?"
  5. "What would you put in the CI pipeline vs run on a schedule?"
Sample Strong Answer (Key Points)

Testing portfolio:

  • Unit tests (35%): Preference rule engine (property-based testing with Hypothesis for 50+ rule combinations), notification template rendering, rate limiting logic
  • Integration tests (40%): Kafka consumer with Testcontainers, database queries with real PostgreSQL, email/SMS delivery via mock SMTP server and WireMock for Twilio
  • Contract tests (10%): Pact contracts with event producers (what events we expect) and delivery providers (Twilio, SendGrid API contracts)
  • E2E tests (10%): Publish test event to Kafka, verify notification appears in test mailbox/SMS simulator
  • Chaos (5%): Toxiproxy between consumer and Kafka to test consumer restart/rebalance, slow delivery provider simulation

Key insight: "For the 50+ preference combinations, I'd use property-based testing with Hypothesis. Instead of manually writing test cases for each combination, I'd define properties like 'a user who opted out of email should never receive email notifications' and let Hypothesis generate random preference sets."

CI pipeline: "Unit + integration tests on every commit (5 min). Contract tests on every commit (2 min). E2E tests pre-deploy to staging. Chaos tests weekly on a schedule."

Mock Interview 3: Code Review Exercise (30 min)

Interview Prompt

Prompt: "Here's a PR for a rate limiter middleware. Review it as you would in a real PR. Tell me what you'd approve, what you'd request changes on, and what you'd want to discuss."

from fastapi import Request, HTTPException
import redis
import time

r = redis.Redis(host="redis", port=6379)

async def rate_limit(request: Request):
    client_ip = request.client.host
    key = f"rate:{client_ip}"
    current = r.get(key)

    if current and int(current) > 100:
        raise HTTPException(429, "Rate limit exceeded")

    pipe = r.pipeline()
    pipe.incr(key)
    pipe.expire(key, 60)
    pipe.execute()

async def rate_limit_middleware(request: Request, call_next):
    await rate_limit(request)
    response = await call_next(request)
    return response

Expected review points (scored 1-5 per category):

Model Code Review Response

Issues (must fix):

  1. Race condition: Between get() and incr(), another request can slip through. Two requests at count=100 both pass the check, both increment. Use a Lua script for atomic check-and-increment.
  2. IP-based limiting is unreliable: Users behind NAT/proxies share IPs. Use API key or user ID instead (or in addition).
  3. No rate limit headers: Clients need X-RateLimit-Limit, X-RateLimit-Remaining, X-RateLimit-Reset headers.
  4. Redis connection is module-level: No connection pooling, no error handling if Redis is down. The entire API becomes unavailable if Redis fails.
  5. expire() may not set on first request: If the key doesn't exist, incr creates it with no TTL, then expire sets it. But if expire fails (Redis timeout between pipeline commands), the key lives forever.

Suggestions (non-blocking):

  • Make limit (100) and window (60s) configurable, not hardcoded
  • Add metrics: count of rate-limited requests for monitoring
  • Consider sliding window counter instead of fixed window (avoids burst at window boundaries)

Praise:

  • Good use of Redis pipeline to batch commands
  • Clean separation of rate_limit logic from middleware

Mock Interview 4: Production Incident Response (30 min)

Interview Prompt & Walkthrough

Prompt: "It's 3 AM. You're on call. PagerDuty fires: 'API error rate > 5% for 10 minutes.' Walk me through your response."

Expected structure:

  1. Acknowledge and assess (2 min): "First, I acknowledge the alert. Check the dashboard: which endpoints? Which services? Is it all traffic or a subset?"
  2. Initial triage (5 min): "Check: recent deployments? Infrastructure changes? External dependency issues? Error message patterns in logs?"
  3. Identify scope (5 min): "If it's a single service, check its health. If it's the database, check connection count and slow queries. If it's external, check the vendor status page."
  4. Mitigate (10 min): "If a deployment caused it: rollback. If a dependency is down: enable circuit breaker fallback. If it's traffic spike: scale up or enable load shedding."
  5. Communicate (ongoing): "Post in the incident channel. Update every 15 minutes. Escalate if not resolved in 30 minutes."
  6. Post-resolution: "Monitor for 30 minutes to confirm stable. Schedule postmortem within 48 hours."

What the interviewer evaluates:

  • Systematic approach (not random guessing)
  • Communication instincts (incident channel, stakeholder updates)
  • Mitigation before root cause (stop the bleeding first)
  • Knowledge of tools (dashboards, logs, traces)
  • Escalation judgment (when to wake up others)

Q: How do you build confidence before the interview?
  • Practice out loud: Explaining to a mirror or recording yourself is 10x more effective than reading silently.
  • Mock interviews: Do at least 2. One with a friend, one with a stranger (interviewing.io, Pramp).
  • Build something: Implement a rate limiter, a webhook system, or a circuit breaker. Real implementation deepens understanding.
  • Review your own work: Think about systems you've built. What were the tradeoffs? What would you do differently? These become your best interview stories.
  • Remember: The interviewer wants you to succeed. They're looking for reasons to hire you, not reasons to reject you. Show your thought process, be honest about gaps, and demonstrate that you can reason through novel problems.
Q: Final checklist — are you ready?

Before your interview, you should be able to:

  • Explain the difference between platform engineering, SRE, and DevOps in 2 minutes
  • Design an API with proper versioning, error handling, and pagination
  • Explain REST vs gRPC vs GraphQL tradeoffs with real examples
  • Design a webhook system with HMAC, retries, and DLQs
  • Compare Kafka vs RabbitMQ vs SQS for a given scenario
  • Implement retry with exponential backoff and explain why jitter matters
  • Draw a circuit breaker state machine
  • Design a testing strategy with the right mix of unit/integration/E2E
  • Review code and find N+1 queries, race conditions, and security issues
  • Define SLIs/SLOs and calculate error budgets
  • Diagnose a cascading failure from a monitoring alert
  • Present tradeoffs clearly: "I'd choose A over B because X, with the downside of Y"

If you can do all of these, you're ready. Good luck!
