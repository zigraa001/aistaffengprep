Review Dimensions

Staff-level prioritization: Focus review energy on correctness and security first. Style and naming are important but can be automated with linters. Don't spend 30 minutes arguing about variable names while a race condition goes unnoticed.

Common Bugs to Spot

Q: How do you spot N+1 query problems in code review?

A: The N+1 problem occurs when code fetches a list of items and then makes a separate query for each item's related data.

# BAD: N+1 queries (1 query for orders + N queries for users)
orders = db.query("SELECT * FROM orders WHERE status = 'pending'")
for order in orders:
    user = db.query("SELECT * FROM users WHERE id = %s", order.user_id)  # N queries!
    order.user_name = user.name

# GOOD: Single query with JOIN
orders = db.query("""
    SELECT o.*, u.name as user_name
    FROM orders o
    JOIN users u ON o.user_id = u.id
    WHERE o.status = 'pending'
""")

# GOOD: Batch fetch
orders = db.query("SELECT * FROM orders WHERE status = 'pending'")
user_ids = [o.user_id for o in orders]
users = db.query("SELECT * FROM users WHERE id = ANY(%s)", user_ids)
user_map = {u.id: u for u in users}

Red flags in review:

  • Database calls inside a for/while loop
  • ORM lazy loading in a loop (e.g., order.user.name in Python/Django)
  • Missing .prefetch_related() or .select_related() in Django
  • Missing @EntityGraph or JOIN FETCH in JPA/Hibernate
Q: How do you spot race conditions during code review?
# BUG: Race condition - check-then-act without atomicity
def transfer_funds(from_account, to_account, amount):
    balance = db.get_balance(from_account)
    if balance >= amount:          # Thread A checks: balance=100, amount=80
        # Thread B also checks: balance=100, amount=80 (BOTH pass!)
        db.deduct(from_account, amount)  # Thread A deducts 80 (balance=20)
        db.add(to_account, amount)       # Thread B deducts 80 (balance=-60!) BUG!

# FIX: Use database-level atomicity
def transfer_funds(from_account, to_account, amount):
    rows_updated = db.execute("""
        UPDATE accounts SET balance = balance - %s
        WHERE id = %s AND balance >= %s
    """, amount, from_account, amount)  # Atomic check-and-deduct

    if rows_updated == 0:
        raise InsufficientFundsError()
    db.execute("UPDATE accounts SET balance = balance + %s WHERE id = %s",
               amount, to_account)

Red flags in review:

  • Read-then-write patterns without locking or atomic operations
  • Shared mutable state without synchronization
  • if not exists then create patterns (TOCTOU)
  • Incrementing counters with count = get(); set(count + 1) instead of atomic increment

Spot the Bug Exercises

Spot the Bug #1: User registration endpoint
@app.post("/register")
async def register(request: RegisterRequest):
    # Check if email exists
    existing = await db.fetch_one(
        "SELECT id FROM users WHERE email = $1", request.email
    )
    if existing:
        raise HTTPException(409, "Email already exists")

    # Hash password
    hashed = hashlib.md5(request.password.encode()).hexdigest()

    # Create user
    user = await db.execute(
        "INSERT INTO users (email, password, role) VALUES ($1, $2, $3)",
        request.email, hashed, request.role
    )
    return {"id": user.id, "email": request.email}

Bugs found:

  1. MD5 for passwords — MD5 is cryptographically broken. Use bcrypt or argon2.
  2. Race condition — Two concurrent registrations with the same email can both pass the check. Add a UNIQUE constraint on the email column and handle the DB constraint violation.
  3. User-controlled role — request.role lets users set their own role (e.g., "admin"). Role should be set server-side.
  4. No input validation — No email format validation, no password strength check.
Spot the Bug #2: Pagination endpoint
@app.get("/orders")
async def list_orders(page: int = 1, per_page: int = 100):
    offset = (page - 1) * per_page
    orders = await db.fetch_all(
        f"SELECT * FROM orders ORDER BY created_at DESC OFFSET {offset} LIMIT {per_page}"
    )
    total = await db.fetch_val("SELECT COUNT(*) FROM orders")
    return {"orders": orders, "total": total, "page": page}

Bugs found:

  1. SQL injection — f-string with user input (offset, per_page). Use parameterized queries.
  2. No upper bound on per_page — User can request per_page=1000000 and OOM the server. Cap at a reasonable max (e.g., 100).
  3. No lower bound on page — page=0 or negative values produce invalid offsets.
  4. SELECT * — Returns all columns including potentially sensitive data.
  5. COUNT(*) on every request — Expensive on large tables. Consider caching or removing total count.
  6. Offset pagination — Slow on large datasets. Consider cursor-based pagination.
Spot the Bug #3: Cache invalidation
async def get_user_profile(user_id: str):
    # Check cache
    cached = await redis.get(f"user:{user_id}")
    if cached:
        return json.loads(cached)

    # Fetch from DB
    user = await db.fetch_one("SELECT * FROM users WHERE id = $1", user_id)
    if user:
        await redis.set(f"user:{user_id}", json.dumps(user))
    return user

async def update_user_profile(user_id: str, data: dict):
    await db.execute("UPDATE users SET name = $1 WHERE id = $2", data["name"], user_id)
    await redis.delete(f"user:{user_id}")

Bugs found:

  1. No TTL on cache — If update_user_profile fails to delete cache (Redis down), stale data persists forever. Always set a TTL.
  2. Race condition on update — Between DB update and cache delete, another request can read stale DB data and re-cache it. Use cache-aside with TTL, or write-through cache.
  3. No error handling on cache operations — If Redis is down, get_user_profile throws instead of falling back to DB.
  4. Cache stampede — If cache expires and 1000 requests hit simultaneously, all 1000 query the DB. Use a lock or probabilistic early expiration.
Spot the Bug #4: Concurrent counter update
public class ViewCounter {
    private Map<String, Integer> counts = new HashMap<>();

    public void incrementView(String pageId) {
        int current = counts.getOrDefault(pageId, 0);
        counts.put(pageId, current + 1);
    }

    public int getViews(String pageId) {
        return counts.getOrDefault(pageId, 0);
    }
}

Bugs found:

  1. Not thread-safe — HashMap is not concurrent. Multiple threads calling incrementView simultaneously can corrupt the map or lose increments. Use ConcurrentHashMap with compute() or AtomicInteger.
  2. Lost updates — Read-then-write is not atomic even with ConcurrentHashMap's get/put. Use counts.merge(pageId, 1, Integer::sum).
Spot the Bug #5: API retry logic
def call_payment_api(order_id, amount):
    for attempt in range(5):
        try:
            response = requests.post(
                "https://api.payment.com/charges",
                json={"order_id": order_id, "amount": amount},
                timeout=30
            )
            if response.status_code == 200:
                return response.json()
            elif response.status_code == 400:
                return response.json()  # Client error, retry won't help
        except requests.Timeout:
            time.sleep(1)
    raise PaymentError("Payment failed after 5 attempts")

Bugs found:

  1. Retrying 400 errors — The code continues looping on 400 (client error). These are not retryable. It should return/raise on 400.
  2. No idempotency key — Retrying a payment POST without an idempotency key can charge the customer multiple times.
  3. Fixed sleep (no backoff) — All retries are 1 second apart. Use exponential backoff with jitter.
  4. No circuit breaker — If the payment API is down, every order attempt waits 5 × 30s = 2.5 minutes.
  5. Swallowing non-timeout exceptions — Connection errors are not caught.

Review Culture

Q: How do you give constructive feedback in code reviews?

A:

Instead ofSay
"This is wrong""This could cause X when Y happens. What if we..."
"Why did you do it this way?""I'm curious about the thinking behind this approach. I was considering X because..."
"This is bad code""nit: this could be simplified to..." or "suggestion: consider X for readability"
Blocking for style"nit (non-blocking): prefer X style" and approve anyway

Comment prefixes:

  • nit: — Style/naming suggestion, non-blocking
  • suggestion: — Consider this alternative, non-blocking
  • question: — I don't understand this, help me understand
  • issue: — This is a bug or security issue, must fix before merge
  • praise: — This is well done! (Don't forget positive feedback)
Q: What should staff engineers prioritize when reviewing code?

A: Staff engineers should focus on what automated tools can't catch:

  • Architectural fit — Does this change align with the system's architecture? Does it introduce unnecessary coupling?
  • Failure modes — What happens when this code fails? Are error paths tested?
  • Scalability implications — Will this work at 10x current load?
  • API contract changes — Is this backward compatible? Will it break consumers?
  • Operational impact — Can this be deployed safely? Is it observable? Can it be rolled back?
  • Missing context — Is the PR description sufficient? Will the next person understand why this change was made?

What to delegate to automation: Style, formatting, import ordering, unused variables, simple type errors. Configure linters and formatters in CI so humans don't waste time on these.
