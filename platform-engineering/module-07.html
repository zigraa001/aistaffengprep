Testing Shapes

Q: What are the testing pyramid, trophy, and diamond, and which should you use?

A:

ShapeEmphasisBest For
Pyramid (Fowler)Many unit tests, fewer integration, fewest E2ELibraries, pure logic, microservices with clear boundaries
Trophy (Kent C. Dodds)Heavy integration tests, moderate unit, few E2EFrontend apps, services with many integrations
DiamondHeavy integration, moderate E2E, fewer unitPlatform services where integration correctness is paramount

Staff-level insight: The "right" shape depends on where your bugs live. If most production bugs are integration issues (API contract mismatches, database query bugs), invest more in integration tests. If bugs are logic errors in complex algorithms, invest in unit tests. Don't dogmatically follow any shape — measure where your defects come from and allocate testing effort there.

Platform engineering context: Platform services typically benefit from the diamond/trophy shape because their value is in correctly integrating with infrastructure (databases, message brokers, external APIs). A platform service with 100% unit test coverage but no integration tests is dangerously undertested.

Test Doubles

Q: What are the different types of test doubles and when should you use each?

A:

TypeDefinitionVerifiesExample
StubReturns predefined dataState (what was returned)Return fake user data from a user service
MockRecords interactions, has expectationsBehavior (was method called?)Verify email service was called with correct args
FakeWorking implementation (simplified)BothIn-memory database instead of PostgreSQL
SpyReal implementation that records callsBehavior + real outputReal HTTP client that logs all requests
DummyPassed but never usedNothingPlaceholder parameter to satisfy a signature
 
# Stub - returns canned data
class StubUserRepo:
    def get_user(self, user_id):
        return User(id=user_id, name="Test User", email="test@example.com")

# Fake - working in-memory implementation
class FakeUserRepo:
    def __init__(self):
        self.users = {}
    def save(self, user):
        self.users[user.id] = user
    def get_user(self, user_id):
        return self.users.get(user_id)

# Mock with pytest-mock
def test_sends_welcome_email(mocker):
    mock_email = mocker.patch("services.email.send_welcome")
    user_service.create_user({"name": "Alice", "email": "a@test.com"})
    mock_email.assert_called_once_with("a@test.com", "Alice")

Guideline: Prefer fakes over mocks for dependencies you own (repositories, caches). Use mocks for dependencies you don't own (external APIs, third-party services). Overuse of mocks leads to tests that pass but don't catch real integration bugs.

Advanced Testing Techniques

Q: What is property-based testing and when is it more effective than example-based testing?

A: Property-based testing generates random inputs and verifies that properties (invariants) always hold, rather than checking specific input/output pairs.

# Example-based: tests specific cases
def test_sort():
    assert sort([3, 1, 2]) == [1, 2, 3]
    assert sort([]) == []
    assert sort([1]) == [1]

# Property-based: tests invariants with random inputs
from hypothesis import given
import hypothesis.strategies as st

@given(st.lists(st.integers()))
def test_sort_properties(lst):
    result = sort(lst)
    assert len(result) == len(lst)           # Length preserved
    assert all(result[i] <= result[i+1]     # Actually sorted
               for i in range(len(result)-1))
    assert set(result) == set(lst)           # Same elements

Property-based testing excels at finding:

  • Edge cases you didn't think of (empty inputs, very large numbers, unicode)
  • Boundary conditions in parsers and serializers
  • Roundtrip properties (serialize then deserialize = original)
  • Invariants in state machines

Tools: Hypothesis (Python), jqwik (Java), fast-check (JavaScript), gopter (Go)

Q: What is mutation testing and what does it reveal about test quality?

A: Mutation testing introduces small changes (mutations) to your code and checks if your tests catch them. If a mutation survives (tests still pass), your tests have a blind spot.

Common mutations:

  • Change > to >= or <
  • Replace true with false
  • Remove method calls
  • Change arithmetic operators (+, -, *, /)
  • Replace return values with defaults (0, null, empty)

Mutation score = killed mutations / total mutations. A score of 85% means 15% of mutations survived — your tests wouldn't catch those bugs.

Why it's better than code coverage: 100% line coverage doesn't mean your tests actually verify anything. You can cover every line without asserting results. Mutation testing verifies that your assertions are meaningful.

Tools: PITest (Java), mutmut (Python), Stryker (JavaScript/.NET)

Staff-level take: Mutation testing is too slow for every CI run. Run it weekly or on critical modules. Target 80%+ mutation score on business logic. Don't bother with mutation testing on glue code or configuration.

Q: TDD vs BDD — what's the real difference and when do you use each?

A:

AspectTDDBDD
FocusDeveloper-facing, implementationBusiness-facing, behavior
LanguageCode (test frameworks)Natural language (Given/When/Then)
AudienceDevelopersDevelopers + product/QA
CycleRed → Green → RefactorDefine scenario → Implement → Verify
Toolspytest, JUnit, JestCucumber, Behave, SpecFlow

Staff-level view: TDD is a design tool, not a testing tool. Its value is in forcing you to think about interfaces before implementation. BDD is a communication tool — valuable when product managers and engineers need a shared specification language. Most teams benefit from TDD for core logic and BDD-style descriptions for acceptance criteria, without the overhead of a full BDD framework.

Practical Concerns

Q: How do you diagnose and fix flaky tests?

A: Flaky tests pass sometimes and fail sometimes without code changes. They erode confidence in the test suite.

Common causes and fixes:

CauseSymptomFix
Shared stateTest order mattersIsolate test data, reset state between tests
Timing/asyncPasses locally, fails in CIUse proper async waiting (not sleep), increase timeouts
External dependencyFails when network is slowMock external calls in unit tests
Date/timeFails at midnight or month boundariesInject a clock, use fixed timestamps
Resource contentionFails under parallel executionUse unique resources per test (random DB names, ports)
Non-deterministic dataIntermittent assertion failuresUse deterministic test data, seed random generators

Quarantine strategy: When you identify a flaky test, move it to a quarantine suite that runs separately. This unblocks the main pipeline while giving you time to fix the root cause. Track quarantine age — if a test is quarantined for >7 days, either fix or delete it.

Q: What are the limits of code coverage as a metric?

A: Code coverage measures which lines/branches are executed by tests, not whether the tests actually verify correct behavior.

# 100% coverage, 0% verification
def test_calculate_price():
    calculate_price(100, 0.1)  # Runs the function but doesn't assert anything!

# Better: actually verify
def test_calculate_price():
    assert calculate_price(100, 0.1) == 90.0  # 10% discount

What coverage IS good for:

  • Finding untested code paths (coverage < 50% is a red flag)
  • Preventing regressions in coverage over time (ratchet: never decrease)
  • Identifying dead code (code that's never executed even in production)

What coverage is NOT good for:

  • Measuring test quality (high coverage != good tests)
  • Setting as a gate (mandating 90% coverage leads to low-value tests)
  • Comparing across teams (a 70% coverage API might be better tested than a 95% coverage UI)

Staff-level recommendation: Track coverage trends, not absolute numbers. Set a floor (e.g., 60% for new services) and use a coverage ratchet (coverage can't decrease). Combine with mutation testing for a more accurate quality signal.

Q: What test naming conventions lead to maintainable test suites?

A: Test names should describe the scenario and expected outcome, acting as documentation:

# BAD: vague names
def test_create_user():
def test_error():
def test_1():

# GOOD: describes scenario and expectation
def test_create_user_with_valid_email_returns_201():
def test_create_user_with_duplicate_email_returns_409():
def test_create_user_without_name_returns_validation_error():

# BDD-style with describe/it (JavaScript)
describe('UserService.createUser', () => {
  it('should return 201 when email is valid', () => { ... });
  it('should return 409 when email already exists', () => { ... });
  it('should hash the password before storing', () => { ... });
});

Pattern: test_[unit]_[scenario]_[expected_result]

When tests fail in CI, the test name should tell you exactly what broke without reading the test code.

Q: When should you NOT write unit tests?

A: Unit tests have diminishing returns in some areas:

  • Thin wrappers/glue code — If a function just calls another function and returns the result, test the underlying function instead
  • Framework-generated code — ORM models, auto-generated API clients, serializers from schemas
  • UI layout — Visual regression testing (screenshots) is more effective than asserting CSS values
  • Code about to be deleted — If you're migrating to a new system, don't retroactively add tests to the old one
  • Integration-heavy code — If the function's value is in how it coordinates database queries and API calls, an integration test is more valuable

Staff-level principle: Every test has a maintenance cost. Write tests where the cost of a bug exceeds the cost of the test. For critical business logic, that threshold is very low (test everything). For configuration glue code, the threshold is higher.

Q: How should a staff engineer define a testing strategy for a new platform service?

A: A testing strategy document should cover:

  1. Risk analysis — What are the highest-risk areas? (e.g., payment processing, data integrity)
  2. Test distribution — What shape? How many of each type? Justified by risk analysis.
  3. Test boundaries — What's in-scope for unit vs integration vs E2E?
  4. External dependencies — Which are mocked? Which use testcontainers? Which use sandbox environments?
  5. CI pipeline — Which tests run on every commit? Which run nightly? Which run pre-deploy?
  6. Quality gates — Coverage floor, mutation score target, E2E pass rate requirements
  7. Ownership — Who owns each test suite? Who triages failures?

Example for a platform service (API Gateway):

  • Unit tests (30%): Routing logic, rate limiting algorithms, request transformation
  • Integration tests (50%): Upstream service connectivity, auth token validation, database queries
  • E2E tests (15%): Full request flow through gateway to upstream service and back
  • Performance tests (5%): Latency regression benchmarks at P99
