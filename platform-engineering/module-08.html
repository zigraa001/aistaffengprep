Integration Testing Fundamentals

Q: What is integration testing and how do you define its scope?

A: Integration tests verify that components work correctly together. The scope defines which boundaries you're testing across.

ScopeWhat's RealWhat's MockedSpeed
NarrowYour code + one dependencyEverything elseFast (seconds)
BroadYour code + all local deps (DB, cache, queue)External servicesMedium (10-60s)
FullYour code + all deps including externalNothingSlow (minutes)

Recommendation: Most value comes from broad integration tests using Testcontainers for databases, caches, and message brokers, with external HTTP services mocked via WireMock. This catches real integration bugs (SQL syntax, serialization, connection handling) without depending on external service availability.

Q: How does Testcontainers work and why is it a game-changer for integration testing?

A: Testcontainers spins up real Docker containers for your dependencies during tests. Instead of mocking PostgreSQL, you test against a real PostgreSQL instance.

# Python with testcontainers
from testcontainers.postgres import PostgresContainer
from testcontainers.redis import RedisContainer
import pytest

@pytest.fixture(scope="session")
def postgres():
    with PostgresContainer("postgres:15") as pg:
        yield pg

@pytest.fixture(scope="session")
def redis():
    with RedisContainer("redis:7") as r:
        yield r

@pytest.fixture
def db(postgres):
    engine = create_engine(postgres.get_connection_url())
    Base.metadata.create_all(engine)
    yield Session(engine)
    Base.metadata.drop_all(engine)

def test_create_user_persists_to_database(db):
    repo = UserRepository(db)
    user = repo.create(name="Alice", email="alice@test.com")
    found = repo.get_by_email("alice@test.com")
    assert found.name == "Alice"
    assert found.id == user.id
// Java with Testcontainers + JUnit 5
@Testcontainers
class OrderRepositoryTest {
    @Container
    static PostgreSQLContainer<?> postgres =
        new PostgreSQLContainer<>("postgres:15")
            .withDatabaseName("testdb");

    @Container
    static KafkaContainer kafka =
        new KafkaContainer(DockerImageName.parse("confluentinc/cp-kafka:7.4.0"));

    @Test
    void shouldPersistOrderAndPublishEvent() {
        var repo = new OrderRepository(postgres.getJdbcUrl());
        var publisher = new KafkaPublisher(kafka.getBootstrapServers());
        var service = new OrderService(repo, publisher);

        Order order = service.createOrder(new CreateOrderRequest("item-1", 2));

        assertThat(repo.findById(order.getId())).isPresent();
        // Verify Kafka message was published
        var records = KafkaTestUtils.getRecords(consumer);
        assertThat(records).hasSize(1);
    }
}

Why it's a game-changer: Before Testcontainers, teams chose between slow shared test databases (flaky, contention) or mocking (misses real bugs). Testcontainers gives you real database behavior with test isolation and parallelism.

Contract Testing

Q: What is consumer-driven contract testing and how does Pact work?

A: Consumer-driven contract testing verifies that a provider's API meets the expectations of its consumers, without running both services together.

sequenceDiagram participant Consumer as Consumer Team participant Broker as Pact Broker participant Provider as Provider Team Consumer->>Consumer: Write consumer test (generates pact) Consumer->>Broker: Publish pact contract Provider->>Broker: Fetch pact contracts Provider->>Provider: Verify pact against real API Provider->>Broker: Publish verification result Note over Broker: Both sides verified - safe to deploy 
# Consumer side (Python Pact)
from pact import Consumer, Provider

pact = Consumer('OrderService').has_pact_with(Provider('UserService'))

def test_get_user():
    expected = {"id": "user-123", "name": "Alice", "email": "alice@test.com"}

    (pact
     .given("user 123 exists")
     .upon_receiving("a request for user 123")
     .with_request("GET", "/users/user-123")
     .will_respond_with(200, body=expected))

    with pact:
        # This calls the Pact mock server
        result = user_client.get_user("user-123")
        assert result.name == "Alice"
    # Pact file is generated: orderservice-userservice.json
# Provider side: verify against the pact
def test_provider_honors_pact():
    verifier = Verifier(provider="UserService", provider_base_url="http://localhost:8080")
    output, _ = verifier.verify_pacts(
        "./pacts/orderservice-userservice.json",
        provider_states_setup_url="http://localhost:8080/_pact/setup"
    )
    assert output == 0
Q: Contract testing vs integration testing — when do you use each?
AspectContract TestingIntegration Testing
What it verifiesAPI shape and expectationsActual runtime behavior
Runs againstMock/pact fileReal (or containerized) services
SpeedFast (no real services)Slower (real dependencies)
CatchesBreaking API changesRuntime bugs, performance issues
MissesBugs in business logicNothing (if comprehensive)
Best forCross-team API compatibilitySingle service correctness

Staff-level answer: They're complementary, not competing. Contract tests are cheap guards against API breakage across team boundaries. Integration tests verify that your service actually works correctly. Use both: contract tests in every CI pipeline, integration tests for your own service's correctness.

Testing Specific Concerns

Q: How do you test async workflows (message brokers, event handlers)?

A: Async testing requires waiting for side effects and verifying eventual state.

# Testing Kafka consumer with Testcontainers
async def test_order_event_updates_inventory():
    # Arrange: publish event to Kafka
    await kafka_producer.send("order-events", value=json.dumps({
        "type": "OrderCreated",
        "data": {"product_id": "prod-1", "quantity": 5}
    }).encode())

    # Act: wait for consumer to process (with timeout)
    async def inventory_updated():
        inventory = await db.fetch_one(
            "SELECT quantity FROM inventory WHERE product_id = 'prod-1'"
        )
        return inventory and inventory["quantity"] == 95  # was 100

    result = await wait_for(inventory_updated, timeout=10, poll_interval=0.5)
    assert result, "Inventory was not updated within timeout"

async def wait_for(condition, timeout=10, poll_interval=0.5):
    """Poll until condition is true or timeout."""
    start = time.time()
    while time.time() - start < timeout:
        if await condition():
            return True
        await asyncio.sleep(poll_interval)
    return False

Key patterns:

  • Polling assertion: Check eventual state with timeout (as above)
  • Test consumer directly: Call the handler function with a crafted message, bypassing the broker
  • Capture published events: Use an in-memory event store in tests to verify what was published
Q: How do you test database interactions properly?

A:

  • Use Testcontainers for real database behavior (never mock your database layer)
  • Run migrations in tests to verify migration scripts work
  • Transaction rollback per test for fast cleanup: start a transaction before each test, roll it back after
  • Factory pattern for test data: don't use fixtures that create complex interdependent data
# Test data factory pattern
class UserFactory:
    @staticmethod
    def create(db, **overrides):
        defaults = {
            "name": f"User {uuid4().hex[:6]}",
            "email": f"{uuid4().hex[:6]}@test.com",
            "status": "active",
        }
        data = {**defaults, **overrides}
        return db.execute(
            "INSERT INTO users (name, email, status) VALUES (:name, :email, :status) RETURNING *",
            data
        )

def test_deactivate_user(db):
    user = UserFactory.create(db, status="active")
    service = UserService(db)
    service.deactivate(user.id)
    assert db.fetch_one("SELECT status FROM users WHERE id = :id", {"id": user.id})["status"] == "inactive"
Q: How do you use WireMock or MockServer for HTTP dependency testing?

A: WireMock/MockServer simulate external HTTP services with configurable responses, delays, and failures.

# Python with responses library (simple HTTP mocking)
import responses

@responses.activate
def test_payment_service_handles_timeout():
    responses.add(
        responses.POST,
        "https://payment-api.example.com/charges",
        body=requests.exceptions.Timeout()
    )
    result = payment_service.charge(amount=100, card_token="tok_123")
    assert result.status == "failed"
    assert result.error == "Payment gateway timeout"

@responses.activate
def test_payment_service_handles_partial_response():
    responses.add(
        responses.POST,
        "https://payment-api.example.com/charges",
        json={"id": "ch_123", "status": "pending"},
        status=202,
    )
    result = payment_service.charge(amount=100, card_token="tok_123")
    assert result.status == "pending"
// Java WireMock
@WireMockTest(httpPort = 8089)
class PaymentClientTest {
    @Test
    void shouldRetryOnServerError() {
        stubFor(post("/charges")
            .inScenario("retry")
            .whenScenarioStateIs(STARTED)
            .willReturn(serverError())
            .willSetStateTo("second-attempt"));

        stubFor(post("/charges")
            .inScenario("retry")
            .whenScenarioStateIs("second-attempt")
            .willReturn(okJson("{\"id\": \"ch_123\", \"status\": \"success\"}")));

        var result = paymentClient.charge(100, "tok_123");
        assertThat(result.getStatus()).isEqualTo("success");
        verify(2, postRequestedFor(urlEqualTo("/charges")));
    }
}

Test scenarios to cover: Success, 4xx errors, 5xx errors, timeouts, slow responses, malformed JSON, empty responses, network errors.

Q: How do you test webhooks in an integration test?

A: For testing webhook consumers: send HTTP requests to your endpoint with signed payloads. For testing webhook producers: spin up a mock server that captures deliveries.

# Testing your webhook consumer endpoint
import hmac, hashlib, json

def test_webhook_processes_payment_completed():
    payload = {"event": "payment.completed", "data": {"order_id": "ord-123", "amount": 99.99}}
    body = json.dumps(payload)
    timestamp = str(int(time.time()))
    signature = hmac.new(WEBHOOK_SECRET.encode(), f"{timestamp}.{body}".encode(), hashlib.sha256).hexdigest()

    response = client.post("/webhooks/stripe",
        data=body,
        headers={
            "Content-Type": "application/json",
            "X-Webhook-Signature": f"t={timestamp},v1={signature}",
        })
    assert response.status_code == 200
    # Verify side effect
    order = db.fetch_one("SELECT * FROM orders WHERE id = 'ord-123'")
    assert order["payment_status"] == "completed"

def test_webhook_rejects_invalid_signature():
    response = client.post("/webhooks/stripe",
        json={"event": "test"},
        headers={"X-Webhook-Signature": "t=123,v1=invalid"})
    assert response.status_code == 401
Q: What are the key CI/CD considerations for integration tests?

A:

ConcernStrategy
SpeedRun unit tests first (fast feedback), integration tests second. Parallelize integration tests.
Docker in CIUse Docker-in-Docker or mount the host Docker socket. Pre-pull common images to speed startup.
FlakinessRetry failed tests once (catches transient container startup issues). Track flake rate.
CostIntegration tests use more CI resources. Run full suite on main branch; run subset on PRs.
ParallelismEach test gets its own containers. No shared state between test classes.
 
# GitHub Actions example
jobs:
  unit-tests:
    runs-on: ubuntu-latest
    steps:
      - run: pytest tests/unit/ -x --timeout=30

  integration-tests:
    needs: unit-tests  # Only run if unit tests pass
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:15
        env:
          POSTGRES_PASSWORD: test
    steps:
      - run: pytest tests/integration/ --timeout=120 -n 4  # parallel
Q: How do you build a staff-level integration testing strategy?

A: A staff engineer's integration testing strategy considers the organizational context, not just the technical.

  1. Map service boundaries — Which services talk to each other? Where are the riskiest integrations?
  2. Standardize tooling — Pick one approach (Testcontainers + WireMock) and provide team-ready templates
  3. Contract tests at boundaries — Every cross-team API boundary should have Pact contracts
  4. Shared test infrastructure — Platform team provides Docker images, CI templates, and test utilities
  5. Test environment strategy — Define when teams use local containers vs shared staging vs production

The key insight: Integration tests are more valuable per test than unit tests for platform services, but they're also more expensive. Invest in making them fast and reliable (Testcontainers, good cleanup, parallelism) so teams actually write them.
