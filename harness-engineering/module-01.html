1. What Is a Harness?

The Core Definition ▼
Engineering Analogy: The Harness as an Electrical Panel

A raw LLM API is like bare electrical wire from the grid — immensely powerful, but you cannot plug anything useful into it directly. A harness is the electrical panel: it provides circuit breakers (permission layers), labeled outlets (tool registry), load balancing (model routing), and a consistent interface that hides the chaos behind the wall. The end user sees a reliable socket; the harness manages the dangerous complexity underneath.

A harness is the software layer that sits between a user (or an automated trigger) and an LLM, responsible for:

  1. Assembling context — gathering everything the model needs to know before each call
  2. Invoking the model — making the API call with the right parameters
  3. Parsing the response — extracting structured intent (tool calls, text, reasoning traces) from model output
  4. Executing actions — running tools, writing files, calling APIs based on model intent
  5. Managing the loop — deciding whether to loop back for another turn or return a final answer
  6. Enforcing safety — preventing the model from taking actions outside its authorized scope
Key Insight: The word "harness" is deliberate. A harness constrains and directs power. LLMs are probabilistic engines that will, if unconstrained, hallucinate tool names, write to paths they should not touch, or loop infinitely. The harness is the engineering discipline that makes the raw capability safe and useful.

Why "Harness" Not "Wrapper" — The Engineering Complexity ▼

Engineers unfamiliar with the domain often say "I'll just wrap the API." This misses the iceberg beneath the surface. Consider what a production harness actually handles:

Concern Naive Wrapper Production Harness
Context length Passes full conversation history until it 400s Summarizes, prunes, and ranks context to stay within window while preserving relevance
Tool call failures Crashes or returns error to user Retries with exponential backoff, reports structured error back to model so it can recover
Infinite loops Runs until API rate limit or OOM Tracks turn count, detects repetition, enforces budget limits, interrupts with HITL checkpoint
Concurrent users No isolation between sessions Per-session sandboxes, state isolation, resource quotas
Prompt injection Tool output goes directly into prompt Sanitizes tool output, uses separate trusted/untrusted context slots, validates model intent
Cost Every call uses maximum tokens Caches tool results, uses smaller models for simple steps, tracks per-session spend
Streaming Waits for complete response Streams tokens to UI, detects tool call boundaries mid-stream, starts tool execution before response ends
Observability Logs request/response pairs Traces full agent trajectory, records tool I/O, captures model reasoning, replays for debugging
Real Numbers: The Claude Code harness is approximately 30,000 lines of TypeScript. Cursor's agent runtime is a similar order of magnitude. Devin's cloud runtime is even larger because it manages full VM lifecycle. A "wrapper" is 50 lines. The gap is where all the product value lives.

2. The Agent Loop

The Canonical Loop — Six Steps ▼

Every agentic system, regardless of complexity, executes a variant of this loop. Understanding it deeply is the prerequisite for everything else in this course.

sequenceDiagram participant U as User participant H as Harness participant C as Context Manager participant M as LLM (Claude/GPT) participant P as Response Parser participant T as Tool Executor participant S as State Store U->>H: User message / trigger H->>S: Load session state H->>C: Assemble context (system prompt + history + tools) C-->>H: Context window (tokens) H->>M: API call (messages + tools schema) M-->>H: Response (text or tool_use block) H->>P: Parse response alt Tool call detected P-->>H: Tool name + args H->>T: Execute tool (filesystem, bash, API...) T-->>H: Tool result (stdout/stderr/return value) H->>S: Append tool call + result to history H->>C: Re-assemble context with tool result H->>M: Continue API call M-->>H: Next response else Final answer P-->>H: Text response H->>S: Save final state H-->>U: Return response end

The Six Steps in Detail

Step 1: User Input. This is not just a string. Production harnesses parse user input for intent signals (is this a clarification? a new task? a stop signal?), extract file references, strip prompt injection attempts, and attach metadata (session ID, timestamp, permissions context).

Step 2: Context Assembly. The most underrated step. The harness must construct the full messages array that will be sent to the model. This involves: selecting the right system prompt for the current task, pruning conversation history to fit the context window, injecting relevant tool results, formatting tool schemas, and inserting any retrieval results. A 1% improvement in context quality typically yields a 5-10% improvement in task success rate.

Step 3: Model Call. The actual API call. Production systems handle: streaming vs batch mode, timeout management, rate limit backoff, fallback to alternate models on failure, cost tracking, and request deduplication.

Step 4: Parse Response. The model returns either text or a structured tool_use block. The parser must: detect which type of response was returned, extract tool name and arguments (validating against the registry schema), handle partial or malformed JSON in tool arguments, and detect when the model is expressing uncertainty or asking for clarification.

Step 5: Tool Execution. Run the tool in a sandboxed environment, capture stdout/stderr/return values, handle timeouts, translate OS-level errors into model-readable error messages, and enforce permission policies (can this session write to this path?).

Step 6: Loop or Respond. The harness decides: did the model produce a final answer, or does it need another turn? This involves checking: has the model explicitly marked the task complete? Has a turn budget been exhausted? Has a human-in-the-loop checkpoint been triggered? Is the model in an obvious loop?

The Loop Termination Problem: Knowing when to stop is one of the hardest problems in harness engineering. A model that stops too early is unhelpful. A model that loops indefinitely wastes tokens and costs money. Production systems use a combination of: explicit "task_complete" tool calls, turn budgets, loop detection (same tool call + same args twice), and HITL checkpoints for risky actions.
Code: The Agent Loop in Python ▼

Minimal but Complete Agent Loop

"""
Minimal production-style agent loop in Python.
Uses the Anthropic SDK directly (no framework).
Demonstrates all six steps of the canonical loop.
"""
import json
import anthropic
from typing import Any

client = anthropic.Anthropic()  # reads ANTHROPIC_API_KEY from env

# ─────────────────────────────────────────
# Tool definitions (the registry)
# ─────────────────────────────────────────
TOOLS = [
    {
        "name": "read_file",
        "description": "Read the contents of a file at the given path.",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string", "description": "Absolute path to the file"}
            },
            "required": ["path"]
        }
    },
    {
        "name": "write_file",
        "description": "Write content to a file, creating it if it does not exist.",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["path", "content"]
        }
    },
    {
        "name": "bash",
        "description": "Run a bash command and return stdout + stderr.",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string"}
            },
            "required": ["command"]
        }
    }
]

# ─────────────────────────────────────────
# Tool executor (Step 5)
# ─────────────────────────────────────────
def execute_tool(name: str, inputs: dict[str, Any]) -> str:
    """Execute a tool and return its result as a string."""
    if name == "read_file":
        try:
            with open(inputs["path"]) as f:
                return f.read()
        except FileNotFoundError:
            return f"Error: file not found: {inputs['path']}"

    elif name == "write_file":
        with open(inputs["path"], "w") as f:
            f.write(inputs["content"])
        return f"Written {len(inputs['content'])} bytes to {inputs['path']}"

    elif name == "bash":
        import subprocess
        result = subprocess.run(
            inputs["command"],
            shell=True,
            capture_output=True,
            text=True,
            timeout=30
        )
        output = result.stdout
        if result.stderr:
            output += f"\nSTDERR:\n{result.stderr}"
        return output or "(no output)"

    else:
        return f"Error: unknown tool '{name}'"

# ─────────────────────────────────────────
# The agent loop
# ─────────────────────────────────────────
def run_agent(
    task: str,
    system_prompt: str = "You are a helpful coding assistant.",
    max_turns: int = 20,
) -> str:
    """
    Run the agent loop until the model produces a final text response
    or the turn budget is exhausted.

    Returns the final text response.
    """
    # Step 1: Initialize conversation history (state)
    messages = [{"role": "user", "content": task}]

    for turn in range(max_turns):
        # Step 2: Context assembly is implicit here —
        # in production you'd prune messages to fit context window

        # Step 3: Model call
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            system=system_prompt,
            tools=TOOLS,
            messages=messages,
        )

        # Step 4: Parse response
        # Append assistant's full response to history
        messages.append({"role": "assistant", "content": response.content})

        # Check stop reason
        if response.stop_reason == "end_turn":
            # Model produced a final answer — extract text
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
            return "(no text in final response)"

        elif response.stop_reason == "tool_use":
            # Step 5: Execute all tool calls in this response
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    print(f"  [turn {turn+1}] calling tool: {block.name}({json.dumps(block.input)[:80]})")
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })

            # Step 6: Loop — append tool results and continue
            messages.append({"role": "user", "content": tool_results})

        else:
            # Unexpected stop reason
            return f"Agent stopped unexpectedly: {response.stop_reason}"

    return f"Agent stopped: turn budget of {max_turns} exhausted"


# ─────────────────────────────────────────
# Example usage
# ─────────────────────────────────────────
if __name__ == "__main__":
    result = run_agent(
        task="List all Python files in the current directory, "
             "then read the first one and summarize what it does."
    )
    print("\n=== FINAL ANSWER ===")
    print(result)
What This Code Is Missing (Intentionally): This is the minimal loop. Production systems add: context window management (token counting and pruning), streaming response handling, structured error recovery, permission checks before tool execution, session persistence across restarts, cost tracking, and distributed tracing. Modules 2-12 cover each of these in depth.
Code: The Agent Loop in TypeScript ▼

Most production harnesses (Claude Code, Cursor) are written in TypeScript. Here is the same loop:

import Anthropic from "@anthropic-ai/sdk";
import { execSync } from "child_process";
import { readFileSync, writeFileSync } from "fs";

const client = new Anthropic(); // reads ANTHROPIC_API_KEY

// Tool registry
const TOOLS: Anthropic.Tool[] = [
  {
    name: "read_file",
    description: "Read the contents of a file at the given path.",
    input_schema: {
      type: "object" as const,
      properties: { path: { type: "string" } },
      required: ["path"],
    },
  },
  {
    name: "bash",
    description: "Run a bash command and return stdout + stderr.",
    input_schema: {
      type: "object" as const,
      properties: { command: { type: "string" } },
      required: ["command"],
    },
  },
];

// Tool executor
function executeTool(name: string, input: Record<string, string>): string {
  if (name === "read_file") {
    try {
      return readFileSync(input.path, "utf-8");
    } catch (e) {
      return `Error: ${(e as Error).message}`;
    }
  }
  if (name === "bash") {
    try {
      return execSync(input.command, { encoding: "utf-8", timeout: 30_000 });
    } catch (e) {
      return `Error: ${(e as Error).message}`;
    }
  }
  return `Error: unknown tool '${name}'`;
}

// Agent loop
async function runAgent(task: string, maxTurns = 20): Promise<string> {
  const messages: Anthropic.MessageParam[] = [
    { role: "user", content: task },
  ];

  for (let turn = 0; turn < maxTurns; turn++) {
    const response = await client.messages.create({
      model: "claude-opus-4-5",
      max_tokens: 4096,
      tools: TOOLS,
      messages,
    });

    messages.push({ role: "assistant", content: response.content });

    if (response.stop_reason === "end_turn") {
      const textBlock = response.content.find((b) => b.type === "text");
      return textBlock?.type === "text" ? textBlock.text : "";
    }

    if (response.stop_reason === "tool_use") {
      const toolResults: Anthropic.ToolResultBlockParam[] = [];
      for (const block of response.content) {
        if (block.type === "tool_use") {
          console.log(`[turn ${turn + 1}] ${block.name}(${JSON.stringify(block.input).slice(0, 80)})`);
          const result = executeTool(block.name, block.input as Record<string, string>);
          toolResults.push({ type: "tool_result", tool_use_id: block.id, content: result });
        }
      }
      messages.push({ role: "user", content: toolResults });
    }
  }

  return `Agent stopped: turn budget exhausted`;
}

// Run
runAgent("What is the Node.js version on this machine? Then list the top 3 largest files in /tmp.").then(console.log);
How Claude Code, Cursor, and Devin Implement the Loop ▼

Same abstract loop, dramatically different implementations driven by product constraints:

Claude Code

Claude Code runs as a CLI process on your local machine. The loop implementation is notable for:

  • Local tool execution: All tools run as child processes in your shell. No virtualization — the model has direct access to your filesystem (with permission checks).
  • REPL-style interaction: The loop persists across multiple user messages in a single session, maintaining full conversation history.
  • System prompt from CLAUDE.md: Project-specific context is injected into the system prompt from any CLAUDE.md file found in the directory tree.
  • Streaming-first: The UI renders tokens as they arrive. Tool calls are detected mid-stream and shown as "thinking" indicators.
  • Conservative permission model: First-time tool calls to sensitive operations (file writes, bash commands) prompt for confirmation unless pre-authorized.

Cursor

Cursor embeds the agent loop inside VS Code's process. Key differences:

  • Editor state as context: The harness automatically injects the currently open file, cursor position, selected text, and recent diffs into the system prompt.
  • LSP integration: Tools include language server calls (go-to-definition, find-references, get-diagnostics) that provide semantic code understanding unavailable to a filesystem-only harness.
  • Multi-file context: Uses a custom retrieval system to find relevant files across the project and inject them into context based on semantic similarity to the task.
  • Diff-based output: Rather than writing files directly, the agent produces diffs that are shown in a side-by-side review UI before being applied.

Devin (Cognition AI)

Devin is architecturally the most ambitious. The loop runs inside a cloud-hosted VM:

  • Full VM sandbox: Each session gets a fresh Ubuntu VM. Tools include not just bash/filesystem, but a full browser (via Playwright) and a graphical desktop.
  • Persistent sandboxes: The VM persists across sessions (not torn down after each task), allowing multi-day projects with environment state.
  • Planning layer: Before starting the action loop, Devin generates a structured plan (sub-tasks, dependencies, success criteria) that the loop then executes and tracks progress against.
  • Cloud tool latency: Every tool call crosses a network boundary. Devin's harness batches tool calls and uses asynchronous execution to hide latency.

OpenAI Assistants API

OpenAI's managed offering externalizes the loop to OpenAI's infrastructure:

  • Managed threads: OpenAI stores conversation history (messages, tool calls, tool results) in cloud "threads." Your harness just appends messages and polls for run status.
  • Code Interpreter: A built-in sandbox that executes Python in an isolated container. Your harness does not manage the execution environment.
  • File retrieval: Uploaded files are vectorized and retrieved automatically. Your harness does not manage embeddings.
  • Trade-off: You lose control over context assembly, turn logic, and loop termination. You gain simplicity. This is the right trade-off for prototypes and simple integrations, not for production products with custom requirements.

4. Core Components of a Harness

The Five Core Components ▼

Every production harness is built from the same five components, regardless of target architecture:

graph TD USER["User / Trigger"] CM["Context Manager\n(assembles model input)"] TR["Tool Registry\n(declares + routes tools)"] EE["Execution Engine\n(runs the loop)"] PL["Permission Layer\n(authorizes actions)"] SS["State Store\n(persists session data)"] USER --> EE EE --> CM EE --> TR EE --> PL EE --> SS CM --> SS PL --> TR style EE fill:#164e63,stroke:#06b6d4 style CM fill:#1e293b,stroke:#475569 style TR fill:#1e293b,stroke:#475569 style PL fill:#1e293b,stroke:#475569 style SS fill:#1e293b,stroke:#475569

1. Context Manager

Responsible for constructing the messages array sent to the model. Production context managers handle:

  • Token counting (using the model's tokenizer, not character counts)
  • History pruning strategies (sliding window, summarization, importance ranking)
  • System prompt composition (static + dynamic segments)
  • Tool schema injection (only include tools relevant to current task)
  • Retrieval result injection (RAG results, file contents)

2. Tool Registry

A catalog of every tool the agent can use. The registry is responsible for:

  • Storing tool schemas (names, descriptions, JSON Schema for parameters)
  • Routing tool calls to the correct executor function
  • Validating tool call arguments against the schema before execution
  • Generating the tools parameter for each API call
  • Tracking tool call metrics (latency, error rate, frequency)

3. Execution Engine

The main loop itself — the orchestrator. Responsibilities:

  • Driving the turn cycle (call model, parse, execute tools, continue)
  • Enforcing turn budgets and timeouts
  • Detecting and breaking loops
  • Triggering human-in-the-loop checkpoints
  • Handling streaming responses
  • Managing concurrency (multiple simultaneous agent sessions)

4. Permission Layer

Enforces what the agent is and is not allowed to do. Critical for production safety:

  • Path allowlists/denylists (can write to /tmp, cannot write to /etc)
  • Network access policies (can fetch from approved domains)
  • Command filters (cannot run rm -rf or sudo)
  • Human confirmation gates (require user approval for destructive operations)
  • Per-session and per-user permission scopes

5. State Store

Persists session data across turns and across process restarts:

  • Conversation history (messages array)
  • Tool call and result history
  • Session metadata (start time, user identity, task description)
  • Intermediate artifacts (files created, commands run)
  • Checkpoint state (for resumable long-running tasks)

"""
Minimal harness skeleton showing all five components.
This is the structural template — not production code.
Production implementations add error handling, streaming,
persistence, permission enforcement, and cost tracking.
"""
from dataclasses import dataclass, field
from typing import Callable, Any
import anthropic


# ── 1. State Store ──────────────────────────────────────────
@dataclass
class SessionState:
    session_id: str
    messages: list[dict] = field(default_factory=list)
    turn_count: int = 0
    total_tokens: int = 0

    def append(self, role: str, content: Any):
        self.messages.append({"role": role, "content": content})
        self.turn_count += 1


# ── 2. Tool Registry ────────────────────────────────────────
class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, dict] = {}          # name -> schema
        self._executors: dict[str, Callable] = {}   # name -> fn

    def register(self, schema: dict, executor: Callable):
        name = schema["name"]
        self._tools[name] = schema
        self._executors[name] = executor

    def schemas(self) -> list[dict]:
        return list(self._tools.values())

    def execute(self, name: str, inputs: dict) -> str:
        if name not in self._executors:
            return f"Error: unknown tool '{name}'"
        return self._executors[name](**inputs)


# ── 3. Permission Layer ─────────────────────────────────────
class PermissionLayer:
    def __init__(self, allowed_tools: set[str] | None = None):
        self.allowed_tools = allowed_tools  # None = all allowed

    def check(self, tool_name: str, inputs: dict) -> tuple[bool, str]:
        if self.allowed_tools and tool_name not in self.allowed_tools:
            return False, f"Tool '{tool_name}' is not authorized in this session"
        # Add more checks: path allowlists, command filters, etc.
        return True, "ok"


# ── 4. Context Manager ──────────────────────────────────────
class ContextManager:
    def __init__(self, system_prompt: str, max_tokens: int = 180_000):
        self.system_prompt = system_prompt
        self.max_tokens = max_tokens

    def build(self, state: SessionState) -> list[dict]:
        # In production: count tokens, prune history if needed
        # For now: return full history
        return state.messages


# ── 5. Execution Engine ─────────────────────────────────────
class ExecutionEngine:
    def __init__(
        self,
        registry: ToolRegistry,
        permissions: PermissionLayer,
        context_mgr: ContextManager,
        max_turns: int = 20,
    ):
        self.registry = registry
        self.permissions = permissions
        self.ctx = context_mgr
        self.max_turns = max_turns
        self.client = anthropic.Anthropic()

    def run(self, task: str, session: SessionState) -> str:
        session.append("user", task)

        for _ in range(self.max_turns):
            messages = self.ctx.build(session)
            response = self.client.messages.create(
                model="claude-opus-4-5",
                max_tokens=4096,
                system=self.ctx.system_prompt,
                tools=self.registry.schemas(),
                messages=messages,
            )
            session.total_tokens += response.usage.input_tokens + response.usage.output_tokens
            session.append("assistant", response.content)

            if response.stop_reason == "end_turn":
                for block in response.content:
                    if hasattr(block, "text"):
                        return block.text
                return ""

            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    allowed, reason = self.permissions.check(block.name, block.input)
                    if not allowed:
                        result = f"Permission denied: {reason}"
                    else:
                        result = self.registry.execute(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })

            session.append("user", tool_results)

        return "Turn budget exhausted"

    {

5. Design Decisions That Matter

Decision 1: Local vs Cloud Execution ▼

Where do tools run? This is the most consequential architectural decision for security and capability.

Dimension Local Execution (Claude Code) Cloud Execution (Devin)
Latency Zero tool latency — subprocess on local machine Network RTT per tool call (50-200ms per hop)
Access to local resources Full access to local files, services, credentials Must explicitly share files/secrets with cloud VM
Isolation No OS-level isolation; permission layer is a soft boundary Full VM isolation; catastrophic mistakes don't affect host
Persistence Session dies with terminal process VM persists; long-running tasks survive reconnects
Multi-user scaling Hard — each user needs local agent process Easy — provision VM per session in cloud
Cost Free (runs on user's machine) Cloud VM cost + network transfer
Data residency Data never leaves user's machine (except model API calls) Data lives in cloud VM; compliance implications
Best for Developer tools, personal productivity, accessing local dev environment Long-running tasks, untrusted code execution, multi-day projects
The Hybrid Approach: Cursor uses a hybrid: the agent itself runs locally in VS Code, but can optionally run tests in GitHub Actions (cloud). This gives low-latency file access with isolated CI execution.
Decision 2: Stateful vs Stateless Agent Loop ▼
Dimension Stateful Loop Stateless Loop
Definition State (history, file changes, env vars) persists across turns and sessions Each task starts from a clean environment; no carry-over
Examples Claude Code (persists .sh history), Devin (persistent VM), Cursor (persistent editor state) Most serverless function-calling wrappers, OpenAI Assistants (per-run)
Advantages Can accumulate installed packages, cached data, learned project context across tasks Reproducible; no state corruption; easy to scale horizontally
Disadvantages State can become inconsistent; session management is complex; hard to replay/debug Slow bootstrap per task (reinstall deps, re-clone repo); loses context
Production pattern Snapshot + restore: periodically snapshot VM state; can restore to known-good checkpoints Warm pool: pre-initialize containers with common dependencies; reduce cold start
Decision 3: Synchronous vs Streaming Responses ▼
Dimension Synchronous (Batch) Streaming
API call client.messages.create(...) — blocks until complete client.messages.stream(...) — yields token events
Time to first token Blocked until full response (2-30 seconds for long outputs) First token in ~300ms; user sees progressive output
Tool call detection Parse tool_use block after full response arrives Detect tool call boundary mid-stream; can start execution before response ends
UI experience Loading spinner; poor for interactive use Text appears in real time; feels responsive
Complexity Simple — single response object Complex — must buffer partial JSON for tool arguments, handle stream interruption
When to use sync Background jobs, batch processing, non-interactive agents Interactive CLI/IDE agents, chat interfaces, any human-facing product

import anthropic

    model="claude-opus-4-5",
    max_tokens=1024,
    messages=[{"role": "user", "content": "Explain how TCP handshake works."}],
) as stream:
    for text in stream.text_stream:
        print(text, end="", flush=True)  # Print each token immediately

# After stream completes, get the full message for history
final_message = stream.get_final_message()
print(f"\n\n[Used {final_message.usage.input_tokens} input tokens, "
      f"{final_message.usage.output_tokens} output tokens]")
Decision 4: Single-Model vs Multi-Model Routing ▼
Dimension Single Model Multi-Model Routing
Architecture Every request goes to one model (e.g., always claude-opus-4-5) Route requests to different models based on task complexity, cost, latency requirements
Cost High — opus-level pricing for all requests including simple ones Low — use haiku/sonnet for simple steps; opus only for hard reasoning
Latency Consistent but potentially high for simple tasks Can use fast small models for trivial steps (sub-1s)
Consistency Predictable behavior; same model handles everything Risk of inconsistency — different models have different knowledge cutoffs, capabilities
Routing signals N/A Task type, context length, estimated output length, error recovery mode, cost budget remaining
Production pattern Simple use cases; during initial development Production at scale — Claude Code uses haiku for ambient indexing, opus for complex edits
Cost Example — Model Routing Savings:
claude-opus-4-5: $15 input / $75 output per million tokens
claude-haiku-3-5: $0.80 input / $4.00 output per million tokens

If 60% of your tool calls are simple (read file, run ls): route those to haiku
40% complex (reason about code, write implementation): route to opus

Cost reduction: 60% * (0.80/15) + 40% * 1 = 0.032 + 0.40 = ~43% of original cost
i.e., roughly 57% cost reduction with smart routing

6. Interview Tips

Common Interview Questions on Harness Architecture ▼

Q: "Walk me through how you would design an AI coding assistant."

What interviewers want to hear: The five components (context manager, tool registry, execution engine, permission layer, state store), the agent loop sequence, and at least one non-obvious consideration like loop termination, context window management, or permission enforcement. Candidates who only describe "call the API and show the response" fail this question.

Ideal answer structure: Start with the user experience (what the user sees), then work backwards to the loop architecture, then go deep on the hardest component (usually context management or sandboxing).

Q: "How would you prevent an AI agent from doing something dangerous?"

What interviewers want to hear: Multi-layer defense: permission layer (allowlists/denylists), human-in-the-loop checkpoints for destructive operations, sandboxed execution environment (VM isolation), output validation before applying changes, and monitoring/alerting on anomalous tool call patterns.

Red flag answer: "Use a good system prompt to tell it not to." System prompts are not security boundaries — they are suggestions that a model can ignore under prompt injection or unexpected contexts.

Q: "How does Claude Code differ architecturally from the OpenAI Assistants API?"

What interviewers want to hear: Claude Code owns the full loop (context assembly, turn logic, tool execution, state persistence) locally. OpenAI Assistants externalizes the loop to OpenAI's infrastructure. Claude Code gives full control and lower latency; Assistants gives simplicity and managed state. The trade-off is control vs convenience.

Q: "How would you reduce the cost of an agent that makes 50+ tool calls per task?"

What interviewers want to hear: Model routing (use haiku for simple tool calls), tool result caching (memoize expensive read operations), context compression (summarize history rather than keeping full transcript), tool call batching (execute multiple independent tools in parallel), and early termination detection (stop as soon as task criteria are met).

What Interviewers Look For

  • Systems thinking: Can you reason about the full lifecycle of a request, not just the happy path?
  • Failure mode awareness: Do you proactively discuss loops, context overflow, permission violations, model hallucinations?
  • Production experience signals: Have you dealt with token budgets, streaming, error recovery, cost optimization?
  • Architecture vocabulary: Do you use precise terms (context window, tool registry, permission layer) rather than vague terms ("wrapper", "plugin")?

Red Flags to Avoid

  • Treating the LLM as a magic oracle rather than a probabilistic function with failure modes
  • Ignoring security (permission enforcement, sandboxing, prompt injection defense)
  • No mention of observability — how do you debug a 50-turn agent session?
  • Describing a framework (LangChain) without explaining the harness-level decisions it does not make for you
  • Not knowing the difference between synchronous and streaming API modes

Active Recall Questions

Q1: What is the difference between an SDK, a framework, and a harness? Give a real example of each in the LLM space.

Answer: An SDK is a typed client for a single provider API with no agent logic (e.g., the Anthropic Python SDK — handles auth, retries, streaming deserialization). A framework provides provider-agnostic abstractions for building agent systems: memory interfaces, tool protocols, multi-agent primitives (e.g., LangChain, LlamaIndex). A harness is the complete, opinionated runtime that drives a specific agent product — it makes all the product-level decisions about loop logic, sandboxing, permissions, state, and streaming (e.g., Claude Code, Cursor's agent runtime, Devin). You often use an SDK inside a harness, and may use framework primitives, but the harness is the full product-level system.

Q2: List the six steps of the canonical agent loop in order and explain what happens if Step 5 (tool execution) returns an error.

Answer: The six steps are: (1) User Input, (2) Context Assembly, (3) Model Call, (4) Parse Response, (5) Tool Execution, (6) Loop or Respond. If tool execution returns an error, the harness should: (a) format the error into a structured tool_result message (not crash), (b) append the error result to conversation history, (c) re-assemble context with the error included, (d) make another model call so the model can recover (try a different approach, report the error to the user, or ask for clarification). A robust harness gives the model the opportunity to self-correct from tool errors — a naive wrapper would just surface the exception to the user.

Q3: Why is context assembly described as "the most underrated step" in the agent loop?

Answer: Context assembly directly determines what information the model has available when making its decision. A poorly assembled context — one that includes irrelevant history, misses critical tool results, uses a generic system prompt instead of a task-specific one, or exceeds the token budget and gets truncated in the wrong place — can cause the model to fail tasks it would otherwise solve correctly. In production, a 1% improvement in context quality typically yields a 5-10% improvement in task success rate. Most developers focus on model selection and prompt wording, but context assembly (what you include, in what order, and how you summarize history) is often the highest-leverage optimization available.

Q4: Devin and Claude Code both implement the agent loop. Name two architectural differences and explain the trade-offs each choice implies.

Answer: (1) Execution location: Claude Code runs tools as subprocesses on the user's local machine (zero tool latency, direct access to local files and credentials, but no VM isolation); Devin runs tools inside a cloud VM (network latency per tool call, strong VM isolation, persists across sessions, but data leaves user's machine). (2) Planning layer: Claude Code is reactive (model decides next action after seeing each tool result); Devin generates a structured upfront plan before starting the action loop (better for long-horizon multi-day tasks, but plan can become stale if environment changes unexpectedly). The right choice depends on the product: local developer tools favor Claude Code's architecture; autonomous long-running task agents favor Devin's.

Q5: A user reports that your AI agent "ran forever and cost $50 on a simple task." What are the most likely root causes and how would you fix each?

Answer: Root causes: (1) No turn budget: The loop has no maximum turn count. Fix: enforce a hard max_turns limit (e.g., 50 turns) and return a graceful message when reached. (2) Loop detection missing: The model is calling the same tool with the same arguments repeatedly. Fix: track the last N tool calls; if the same (tool_name, args) pair appears twice, inject a message alerting the model it's in a loop. (3) No cost tracking: No per-session token budget. Fix: track cumulative token usage; interrupt and checkpoint when approaching budget. (4) Wrong model for task: Using claude-opus-4-5 for every small tool call. Fix: route simple operations (file reads, directory listings) to claude-haiku-3-5. (5) Context overflow causing re-work: History was trimmed poorly, causing the model to forget what it already did. Fix: improve context pruning to preserve task-critical information even when truncating.

import json
import subprocess
import anthropic

client = anthropic.Anthropic()

TOOLS = [
    {
        "name": "read_file",
        "description": "Read and return the contents of a file.",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}},
            "required": ["path"]
        }
    },
    {
        "name": "write_file",
        "description": "Write content to a file (creates or overwrites).",
        "input_schema": {
            "type": "object",
            "properties": {"path": {"type": "string"}, "content": {"type": "string"}},
            "required": ["path", "content"]
        }
    },
    {
        "name": "bash",
        "description": "Execute a bash command and return stdout + stderr.",
        "input_schema": {
            "type": "object",
            "properties": {"command": {"type": "string"}},
            "required": ["command"]
        }
    }
]


def execute_tool(name: str, inputs: dict) -> str:
    if name == "read_file":
        try:
            with open(inputs["path"]) as f:
                return f.read()
        except Exception as e:
            return f"Error reading file: {e}"

    elif name == "write_file":
        try:
            with open(inputs["path"], "w") as f:
                f.write(inputs["content"])
            return f"Written {len(inputs['content'])} bytes to {inputs['path']}"
        except Exception as e:
            return f"Error writing file: {e}"

    elif name == "bash":
        try:
            result = subprocess.run(
                inputs["command"], shell=True, capture_output=True, text=True, timeout=30
            )
            out = result.stdout
            if result.stderr:
                out += f"\nSTDERR: {result.stderr}"
            return out or "(no output)"
        except subprocess.TimeoutExpired:
            return "Error: command timed out after 30 seconds"
        except Exception as e:
            return f"Error running command: {e}"

    return f"Error: unknown tool '{name}'"


def run_agent(task: str, max_turns: int = 15) -> dict:
    messages = [{"role": "user", "content": task}]
    total_tokens = 0
    turn = 0

    for turn in range(max_turns):
        response = client.messages.create(
            model="claude-opus-4-5",
            max_tokens=4096,
            tools=TOOLS,
            messages=messages,
        )
        total_tokens += response.usage.input_tokens + response.usage.output_tokens
        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn":
            final_text = ""
            for block in response.content:
                if hasattr(block, "text"):
                    final_text += block.text
            return {"answer": final_text, "turns": turn + 1, "total_tokens": total_tokens}

        elif response.stop_reason == "tool_use":
            tool_results = []
            for block in response.content:
                if block.type == "tool_use":
                    args_preview = json.dumps(block.input)[:80]
                    print(f"  [turn {turn+1}] {block.name}({args_preview})")
                    result = execute_tool(block.name, block.input)
                    tool_results.append({
                        "type": "tool_result",
                        "tool_use_id": block.id,
                        "content": result,
                    })
            messages.append({"role": "user", "content": tool_results})

    return {
        "answer": f"Turn budget of {max_turns} exhausted",
        "turns": max_turns,
        "total_tokens": total_tokens
    }


if __name__ == "__main__":
    result = run_agent(
        "List all .py files in the current directory. "
        "For each one, count how many lines start with 'def '. "
        "Write the results to function_counts.txt in the format: "
        "'filename: N functions' (one per line)."
    )
    print(f"\nAnswer: {result['answer']}")
    print(f"Turns: {result['turns']}")
    print(f"Tokens: {result['total_tokens']}")
