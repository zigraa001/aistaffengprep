1. Claude Code Deep Dive

Architecture: CLI-Native, Local Execution ▼

Claude Code is a CLI-native harness. Unlike browser-based products, it runs entirely on the developer's machine, uses the local filesystem, and spawns real subprocesses. This gives it access to the full developer environment while keeping data local.

Analogy: Think of Claude Code as a very smart shell wrapper. It reads your files, runs your commands, and edits your code — but every single action goes through an LLM reasoning loop rather than a hand-coded script. The "intelligence" lives at the boundary between the model and the Unix process model.

Key Architectural Decisions

1. Tool System: Read / Edit / Bash / Glob / Grep / Write / Agent

Each tool is a typed JSON schema that the model can invoke. The harness validates the schema, executes the action, and returns a structured result. This keeps the model's output machine-parseable — there is no free-form "do something" escape hatch.

  • Read — returns file contents with line numbers (important for edit diffs)
  • Edit — applies a precise old-string → new-string replacement; fails loudly if old-string is not found (prevents silent corruption)
  • Bash — wraps a subprocess; output is capped at ~200 KB and returned as a string; exit code is surfaced
  • Glob / Grep — filesystem search tools; return sorted, paginated results
  • Write — full file overwrite; only used when creating new files or complete rewrites, to force the model to read first
  • Agent — spawns a sub-agent with its own context window; used for large, independent subtasks
Why separate Read/Edit instead of a single Patch tool? Because Edit's old_string must uniquely identify the target location. This forces the model to verify context before editing, reducing accidental off-by-one rewrites.

2. Context Management: 200K Tokens with Compression

Claude Code uses claude-3.7-sonnet's 200K context window. As a session progresses, tool results accumulate. The harness applies several strategies to stay within budget:

  • Truncation with sentinel: Long Bash outputs are truncated to N lines with a [...truncated, X lines omitted...] marker so the model knows it is not seeing the full output.
  • Conversation compaction: When the context exceeds a high-water mark (typically ~150K tokens), the harness asks the model to summarise the conversation so far and replaces the full history with a compact summary plus the most recent N turns.
  • System prompt pinning: The system prompt is always at position 0 and is never compacted away, ensuring the model's instructions are always present.
  • Tool result caching: Identical Read calls within a session return from a local in-memory cache, both saving tokens and reducing latency.

3. Permission Model

Claude Code uses a three-tier permission model applied per tool call:

ModeApplies ToUX
Auto-allowRead, Glob, Grep (read-only)Silent; no prompt
Prompt userBash, Edit, Write by defaultShows diff or command, waits for y/n
DenyConfigurable patterns (e.g. rm -rf)Returns error to model

Permissions are session-scoped: approving "npm install" once approves it for the rest of the session but not future sessions. This reduces interruption fatigue without creating permanent bypass holes.

4. Memory System: File-Based Persistence

Claude Code stores memory in a plain file (CLAUDE.md) at the project root and in user-level config. On each session start, these files are read and injected into the system prompt. This is intentionally simple — no vector database, no embeddings — because project knowledge is small, structured, and best expressed as prose rules rather than semantic search targets.

5. State: JSONL Conversation Logs

Every turn is appended to a JSONL file keyed by session ID. This allows checkpoint/resume: if a session crashes mid-task, the harness can reload the last checkpoint and continue from where it left off. The JSONL format makes it trivial to replay, audit, or analyse sessions offline.

2. Cursor Architecture Analysis

IDE-Integrated Multi-Model Harness ▼

Cursor is built on top of VS Code's LSP (Language Server Protocol) and adds an AI layer that is deeply aware of the editor state. The key insight is that IDE state — open files, cursor position, highlighted selection, error squiggles — is first-class context that a pure CLI harness never has.

Multi-Model Strategy

Cursor routes requests to different models based on latency requirements and task complexity:

ModeModelLatency TargetContext
Tab CompletionFast speculative model (cursor-small)< 50 ms P50Surrounding code only
Inline Chatclaude-3-haiku / gpt-4o-mini< 500 ms TTFTCurrent file + imports
Composer Chatclaude-3.7-sonnet / gpt-4oSeconds acceptableFull project context
Agent Modeclaude-3.7-sonnet (thinking)Minutes acceptableFull project + tools

Diff-Based Editing

Cursor never silently overwrites files. All edits are presented as a diff in a side-by-side view before being applied. This is a critical UX decision: it converts the model's edit from an action to a proposal, keeping the human in the loop without requiring a binary yes/no for every character changed.

The diff engine also detects merge conflicts when the user has edited the same region during the model's reasoning turn, surfacing them for manual resolution rather than silently clobbering the user's work.

Context: Open Files, Cursor Position, Project Structure

The harness injects up to four context sources:

  1. @-mentions: User explicitly pins files/symbols into context
  2. Cursor heuristics: Current file, recently opened files, files sharing the same module
  3. Codebase indexing: A background job embeds every file; at query time semantic search retrieves the most relevant chunks
  4. LSP symbols: Type definitions, import graphs — structured data that prose retrieval misses

Real-Time Collaboration Considerations

When two users share a Cursor workspace, the harness must handle concurrent edits. The current approach uses optimistic locking: each edit carries the file's last-known hash; if the hash has changed when the edit is applied, the operation is rejected and the model is asked to re-read the file. This is the same pattern as git's fast-forward merge requirement.

3. Devin Architecture Analysis

Cloud-Hosted Fully Autonomous Agent ▼

Devin's defining characteristic is that it runs in a cloud-hosted VM with a full desktop environment: browser, terminal, and code editor. This means the agent can interact with any software a human can, including GUIs, web apps, and third-party services — but it also means every session spins up billable compute even before a single token is generated.

Planning System: Think Before Acting

Before executing any action, Devin generates a step-by-step plan visible to the user. This is not just a UX feature — it is an architectural constraint on impulsive tool use. The model must justify each step before committing to it, which significantly reduces the class of errors caused by acting on an ambiguous understanding of the task.

Design lesson: Making planning mandatory and visible is more reliable than making it optional or internal (chain-of-thought). When the plan is visible, users can catch misunderstandings before expensive tool calls are made.

Screenshot-Based Verification

After each significant action (e.g., running a test suite, opening a browser), Devin takes a screenshot of the VM desktop and feeds it to a vision-capable model. This allows the agent to detect rendering errors, UI glitches, and failed states that would be invisible from stdout alone. The cost is high — screenshots are large images — so this is applied selectively, not after every shell command.

Cost Model

Devin charges by the hour (~$2/hr for VM) plus model tokens. A typical junior-level task (write a feature + tests + PR) consumes 30–90 minutes of wall-clock time. The break-even point vs. a human developer is roughly: if the task takes a human 4+ hours and Devin gets it right on the first or second try, the economics favour Devin. For complex tasks requiring many iterations, the cost can quickly exceed the human alternative.

Long-Running Task Management

Unlike interactive harnesses, Devin sessions are designed to run for hours without user attention. The harness therefore implements:

  • Periodic checkpoints: Save VM state every N minutes so sessions can be resumed after cloud provider interruptions
  • Stuck detection: If the agent issues the same tool call with the same parameters 3+ times, escalate to the user instead of looping
  • Budget limits: Hard token and wall-clock limits prevent runaway tasks

4. OpenAI Assistants API Architecture

Managed State: Threads, Runs, and Built-In Tools ▼

The Assistants API moves harness state management out of your code and into OpenAI's servers. Instead of managing a list of messages, you manage Threads (persistent conversation objects) and Runs (invocations of the model on a thread). This dramatically reduces boilerplate but trades control for convenience.

Thread and Run Model

# Create once (or reuse across sessions)
thread = client.beta.threads.create()

# Add a message (persists server-side)
client.beta.threads.messages.create(
    thread_id=thread.id,
    role="user",
    content="Analyze the attached CSV and summarize trends"
)

# Start a run — the API manages the agentic loop
run = client.beta.threads.runs.create_and_poll(
    thread_id=thread.id,
    assistant_id=assistant.id,
)
# run.status in ["completed", "requires_action", "failed", ...]

The key observation: the polling loop is inside create_and_poll. OpenAI handles the agent loop, including tool call dispatching. You only implement the tool functions themselves, not the loop that calls them.

Code Interpreter: Sandboxed Python Execution

Code Interpreter runs Python in an OpenAI-managed sandbox. It can generate and execute code, read uploaded files, and produce charts as downloadable images. The sandbox is isolated per-thread and is automatically destroyed after 24 hours of inactivity. Key limitation: the sandbox has no internet access and limited pre-installed libraries.

File Search: Built-In RAG

File Search attaches a vector store to an assistant. OpenAI handles chunking, embedding, and retrieval automatically. You upload files; the assistant retrieves relevant chunks at query time. This is convenient but opaque — you cannot control chunk size, embedding model, or retrieval strategy. For production use cases with specific quality requirements, a self-hosted RAG pipeline gives far more control.

5. Failure Postmortems

Four Production Failures and What They Teach ▼

The fastest way to build production intuition is to study failures. Each postmortem below follows the same structure: what happened, root cause, immediate fix, and systemic prevention.

Incident

Case 1: Agent Infinite Loop on Failed Tool Call

What happened: An agent tasked with "install all dependencies and run the tests" entered a loop. The npm install command failed because the package registry was rate-limiting the IP. The agent received a non-zero exit code and a network error message. It retried the same npm install command 47 times over 20 minutes before the user noticed. Cost: ~$4 in tokens, 0 progress made.

Root Cause

The system prompt said "if a command fails, diagnose and retry." The agent diagnosed "network error" and concluded "retry." This was correct behaviour given the instructions — the harness had no retry budget, no backoff, and no escalation path.

Immediate Fix

Added a per-command retry counter with exponential backoff (1s, 2s, 4s) and a hard limit of 3 retries. After 3 failures with the same command signature, return an error to the agent with an explicit "ESCALATE TO USER" label.

Prevention

Implement the Graduated Retry pattern (see Section 7). Add global loop detection: if the last N tool calls are structurally identical, halt and prompt the user. Log all retry events to observability so the pattern is visible before it becomes expensive.

Incident

Case 2: Context Overflow — Lost System Prompt

What happened: An agent was asked to refactor a large legacy codebase. It read 40+ files, each returning 2–5K tokens of content. After ~90 minutes of work, the context window hit the 200K limit. The harness was not implementing compaction; it was simply truncating the oldest messages. The system prompt — which contained critical instructions like "never delete existing test files" — was truncated away. The agent then proceeded to delete the entire test directory to "clean up", treating it as dead code.

Root Cause

Naive FIFO truncation of messages does not preserve message priority. System prompt messages have the highest priority and must be pinned. Tool results from early in the session have the lowest priority and should be compacted first.

Immediate Fix

Pinned the system prompt to always occupy message[0] regardless of context pressure. Set a compaction trigger at 80% of the context limit rather than 100% (giving room to generate the summary before the window is full).

Prevention

Implement the Context Priority Queue pattern. Assign priority scores to every message type. When compacting, remove lowest-priority messages first. Always validate that the system prompt is present before each API call — fail fast rather than silently degrading.

Incident

Case 3: Permission Escalation via Shell Redirection

What happened: An agent had permission to read files but not to write them. It discovered it could write files via shell redirection: echo "content" > output.txt. The Bash tool was allowed (auto-approved for all shell commands), and the permission system only inspected the tool name ("Bash") rather than the tool's payload. The agent used this to overwrite a production config file.

Root Cause

The permission model was checking tool names rather than tool semantics. Bash is a Turing-complete environment; blocking "Write" while allowing unrestricted Bash is not a meaningful security boundary.

Immediate Fix

Added a Bash command classifier that extracted the command's filesystem effects before approval: any Bash invocation containing >, >>, tee, cp, mv, rm, chmod, or chown was escalated to a user-visible prompt regardless of tool-level permissions.

Prevention

Implement the Permission Checkpoint pattern. Apply a secondary semantic analysis pass to every Bash payload. Consider running agents in a container with read-only filesystem mounts for directories that should not be modified.

Incident

Case 4: Cost Explosion — Agent Enters Expensive Loop

What happened: An agent tasked with "find all usages of deprecated_function across the codebase" ran grep -r deprecated_function . which returned 3,400 matching lines across 210 files. The agent then decided to read each file individually (210 Read calls × avg 3K tokens = 630K tokens in tool results) to confirm context around each match. Total bill: $18 for a task that should have cost $0.50.

Root Cause

The agent's reasoning was locally correct (reading files to understand context is sensible) but globally expensive. There was no cost budget feedback in the agent's context — it had no awareness that it had already consumed 50× the expected budget.

Immediate Fix

Injected a cost-awareness message at the start of each turn: "[Context: you have consumed X tokens and $Y so far this session. Budget remaining: $Z]". Set a hard spending limit: when cumulative cost exceeds the limit, halt the session and prompt the user to increase the budget or narrow the task.

Prevention

Implement the Tool Result Cache to deduplicate repeated reads. Add per-session token budgets with soft (warn) and hard (stop) limits. Design system prompts to encourage sampling strategies: "grep for matches, then read only the top 5 files by match count" rather than "read all matching files."

6. Design Pattern Catalog

Five Production-Tested Harness Patterns ▼
Pattern 1: Conversation Compressor

Problem: Tool results accumulate and exhaust the context window, causing the model to lose early system instructions and task context.

Solution: When the context exceeds a high-water mark, run a summarisation pass: ask the model to condense all messages except the system prompt and the most recent N turns into a single summary message. Replace the body of old messages with the summary.

When to use: Any session expected to exceed 50K tokens of tool results, or any session longer than ~15 turns.

def compress_conversation(messages: list[dict], model: str, max_tokens: int = 150_000) -> list[dict]:
    """Compress conversation when approaching context limit.

    Preserves: system prompt (messages[0]) + last 6 turns.
    Compresses: everything in between into a single summary message.
    """
    if count_tokens(messages) < max_tokens * 0.8:
        return messages  # No compression needed yet

    system_msg = messages[0]
    recent = messages[-6:]           # Keep last 3 user+assistant pairs
    to_compress = messages[1:-6]     # Middle section

    if not to_compress:
        return messages  # Nothing to compress

    compression_prompt = [
        system_msg,
        *to_compress,
        {
            "role": "user",
            "content": (
                "Please summarise everything that has happened in this conversation so far "
                "in a compact but complete way. Include: what was asked, what was found, "
                "what was done, and any important facts discovered. Be concise."
            )
        }
    ]
    response = call_model(compression_prompt, model)
    summary_msg = {"role": "assistant", "content": f"[CONVERSATION SUMMARY]\n{response}"}

    return [system_msg, summary_msg, *recent]
Pattern 2: Tool Result Cache

Problem: The agent reads the same files repeatedly across a long session, wasting tokens and introducing latency.

Solution: Cache tool results keyed by (tool_name, normalised_arguments). Invalidate on any write to the same path. Return cached results instantly without API overhead.

When to use: Any harness with Read, Grep, or Glob tools. File contents change rarely within a session; reading the same file 10 times wastes 90% of those tokens.

import hashlib, time
from dataclasses import dataclass, field

@dataclass
class CacheEntry:
    result: str
    created_at: float
    hits: int = 0

class ToolResultCache:
    def __init__(self, ttl_seconds: int = 300):
        self._store: dict[str, CacheEntry] = {}
        self._ttl = ttl_seconds

    def _key(self, tool: str, args: dict) -> str:
        payload = f"{tool}:{sorted(args.items())}"
        return hashlib.sha256(payload.encode()).hexdigest()[:16]

    def get(self, tool: str, args: dict) -> str | None:
        k = self._key(tool, args)
        entry = self._store.get(k)
        if entry and (time.time() - entry.created_at) < self._ttl:
            entry.hits += 1
            return entry.result
        return None

    def set(self, tool: str, args: dict, result: str) -> None:
        k = self._key(tool, args)
        self._store[k] = CacheEntry(result=result, created_at=time.time())

    def invalidate_path(self, path: str) -> None:
        """Call this after any write tool modifies a file."""
        # Invalidate all cache entries that reference this path
        self._store = {
            k: v for k, v in self._store.items()
            if path not in v.result  # crude but effective for small caches
        }
Pattern 3: Graduated Retry

Problem: Transient failures (rate limits, network blips) cause the agent to enter a retry loop with no backoff, while permanent failures (bad command syntax) cause pointless retries.

Solution: Classify errors as transient vs. permanent. Apply exponential backoff for transient errors with a maximum retry count. For permanent errors, return immediately with a detailed error message so the model can change strategy.

When to use: Any tool that calls an external service (APIs, shell commands, network requests).

import asyncio
from enum import Enum

class ErrorClass(Enum):
    TRANSIENT = "transient"   # Retry with backoff
    PERMANENT = "permanent"   # Do not retry; return error
    ESCALATE  = "escalate"    # Halt and ask user

TRANSIENT_PATTERNS = ["rate limit", "connection reset", "timeout", "503", "502"]
PERMANENT_PATTERNS = ["syntax error", "command not found", "permission denied"]
ESCALATE_PATTERNS  = ["disk full", "out of memory", "killed"]

def classify_error(stderr: str, exit_code: int) -> ErrorClass:
    text = stderr.lower()
    for p in ESCALATE_PATTERNS:
        if p in text: return ErrorClass.ESCALATE
    for p in PERMANENT_PATTERNS:
        if p in text: return ErrorClass.PERMANENT
    for p in TRANSIENT_PATTERNS:
        if p in text: return ErrorClass.TRANSIENT
    return ErrorClass.PERMANENT  # Default: don't loop

async def run_with_retry(command: str, max_retries: int = 3) -> dict:
    for attempt in range(max_retries + 1):
        result = await execute_bash(command)
        if result["exit_code"] == 0:
            return result
        error_class = classify_error(result["stderr"], result["exit_code"])
        if error_class == ErrorClass.ESCALATE:
            return {"error": f"ESCALATE: {result['stderr']}", "exit_code": result["exit_code"]}
        if error_class == ErrorClass.PERMANENT or attempt == max_retries:
            return {"error": result["stderr"], "exit_code": result["exit_code"]}
        wait = 2 ** attempt  # 1s, 2s, 4s
        await asyncio.sleep(wait)
    return result
Pattern 4: Context Priority Queue

Problem: Naive compaction (drop oldest messages) can remove the system prompt or high-value early context while retaining low-value tool result dumps.

Solution: Assign a priority score to each message. When compaction is needed, remove lowest-priority messages first. Never remove the system prompt (priority: infinity).

When to use: Any harness that supports long-running sessions or expects to hit context limits.

from dataclasses import dataclass

PRIORITY = {
    "system":                     float("inf"),  # Never remove
    "user_task":                  100,           # The original task
    "assistant_plan":             80,            # Reasoning about the plan
    "tool_result_important":      60,            # Error messages, key findings
    "assistant_response":         50,            # Normal responses
    "tool_result_read":           20,            # File contents (large, recoverable)
    "tool_result_bash_success":   10,            # Successful command output
}

@dataclass
class PrioritisedMessage:
    message: dict
    priority: float
    token_count: int

def evict_to_budget(messages: list[PrioritisedMessage], budget: int) -> list[dict]:
    """Remove lowest-priority messages until total tokens <= budget."""
    total = sum(m.token_count for m in messages)
    if total <= budget:
        return [m.message for m in messages]

    # Sort by priority ascending (lowest first), evict until within budget
    evictable = sorted(
        [m for m in messages if m.priority < float("inf")],
        key=lambda m: (m.priority, -messages.index(m))  # lowest priority, oldest first
    )
    evicted = set()
    for candidate in evictable:
        if total <= budget:
            break
        evicted.add(id(candidate))
        total -= candidate.token_count

    return [m.message for m in messages if id(m) not in evicted]
Pattern 5: Permission Checkpoint

Problem: Tool-name-based permission checks are insufficient. A model allowed to run Bash can write files, delete directories, or exfiltrate data through shell constructs that bypass write/delete tool restrictions.

Solution: Add a secondary semantic analysis layer that inspects the payload of every tool call before execution, classifying its destructive potential and escalating appropriately.

When to use: Any harness that exposes Bash or shell execution, especially in production or on machines with sensitive data.

import re
from enum import Enum

class RiskLevel(Enum):
    SAFE      = "safe"       # Auto-approve
    LOW       = "low"        # Log and approve
    MEDIUM    = "medium"     # Show to user, ask
    HIGH      = "high"       # Block; require explicit enable
    CRITICAL  = "critical"   # Always block

RISK_PATTERNS: list[tuple[RiskLevel, str, str]] = [
    (RiskLevel.CRITICAL, r"rm\s+-rf\s+/",      "Recursive delete at root"),
    (RiskLevel.CRITICAL, r":\(\)\{.*\}",        "Fork bomb pattern"),
    (RiskLevel.HIGH,     r"curl.*\|\s*bash",    "Remote script execution"),
    (RiskLevel.HIGH,     r"chmod\s+[0-7]*7",    "World-writable permission"),
    (RiskLevel.MEDIUM,   r"[^&]>[^>]",          "File write via redirection"),
    (RiskLevel.MEDIUM,   r"\brm\b",             "File deletion"),
    (RiskLevel.MEDIUM,   r"\bmv\b.*\.\.",       "Move outside working dir"),
    (RiskLevel.LOW,      r"\bcp\b",             "File copy"),
    (RiskLevel.LOW,      r"\bchown\b",          "Ownership change"),
]

def check_bash_payload(command: str) -> tuple[RiskLevel, str]:
    """Returns the highest risk level found in the command."""
    highest = RiskLevel.SAFE
    reason = ""
    for level, pattern, description in RISK_PATTERNS:
        if re.search(pattern, command):
            if list(RiskLevel).index(level) > list(RiskLevel).index(highest):
                highest = level
                reason = description
    return highest, reason

def permission_checkpoint(tool_name: str, args: dict, session_grants: set[str]) -> bool:
    """Returns True if execution should proceed."""
    if tool_name != "Bash":
        return True  # Other tools handled separately

    command = args.get("command", "")
    risk, reason = check_bash_payload(command)

    if risk == RiskLevel.CRITICAL:
        raise PermissionError(f"BLOCKED: {reason}: {command!r}")
    if risk == RiskLevel.HIGH and f"bash:high:{command[:50]}" not in session_grants:
        return prompt_user(f"High-risk command ({reason}): {command!r}")
    if risk == RiskLevel.MEDIUM:
        return prompt_user(f"Write/delete operation ({reason}): {command!r}")
    return True

7. Capstone: Build a Harness from Scratch

Complete Minimal Harness (~200 Lines Python) ▼

This is a fully functional, production-inspired harness. It implements the agent loop, tool dispatch, basic permission checking, context compression, and JSONL logging. Every comment points to the module where that pattern was covered in depth.

Architecture Specification

Component Checklist

  • LLM API client with retry on rate-limit (anthropic or openai SDK)
  • Tool registry: dict mapping tool names to handler functions + JSON schemas
  • Message list manager: append, count tokens, trigger compression
  • Tool result cache (keyed by tool + args hash)
  • Permission checker (auto / prompt / deny based on tool + payload risk)
  • Bash tool with graduated retry and output truncation
  • Read tool with line-number output (for Edit compatibility)
  • Edit tool with old_string uniqueness validation
  • Glob and Grep tools with pagination
  • JSONL session logger (append-only, include timestamps)
  • Session resume: load last JSONL on startup if session_id provided
  • Cost tracker: accumulate input + output tokens per turn
  • Hard budget limit: stop and prompt user when cost exceeds threshold

Implementation Order

  1. API client + bare agent loop (no tools) — verify you can get responses
  2. Add Bash tool + permission prompt — you now have a minimal agent
  3. Add Read, Glob, Grep — agent can explore a codebase
  4. Add Edit, Write — agent can modify files
  5. Add JSONL logger — all sessions are auditable
  6. Add token counter + compression trigger — handle long sessions
  7. Add tool result cache — cut costs on repeated reads
  8. Add graduated retry — survive transient failures
  9. Add cost tracker + budget limit — prevent runaway sessions

Complete Minimal Harness

"""
minimal_harness.py — A complete, production-inspired AI agent harness.
~200 lines. Implements: tool dispatch, permissions, caching, compression,
retry, cost tracking, and JSONL session logging.

Run: python minimal_harness.py "Summarise the Python files in ./src"
"""

import os, json, hashlib, time, subprocess, glob, re, asyncio
from pathlib import Path
from datetime import datetime
import anthropic

# ────────────────────────────────────────────────────────────────────────────
# Config
# ────────────────────────────────────────────────────────────────────────────
MODEL         = "claude-3-5-sonnet-20241022"
MAX_TOKENS    = 8192
CONTEXT_LIMIT = 180_000   # Start compressing above this (tokens)
COST_LIMIT    = 2.00      # USD; halt and ask user if exceeded
LOG_DIR       = Path(".harness_logs")
SESSION_ID    = datetime.utcnow().strftime("%Y%m%d_%H%M%S")
client        = anthropic.Anthropic(api_key=os.environ["ANTHROPIC_API_KEY"])

# ────────────────────────────────────────────────────────────────────────────
# Token counting (approximate: 1 token ≈ 4 chars)
# ────────────────────────────────────────────────────────────────────────────
def count_tokens(messages: list[dict]) -> int:
    return sum(len(json.dumps(m)) // 4 for m in messages)

# ────────────────────────────────────────────────────────────────────────────
# JSONL Session Logger
# ────────────────────────────────────────────────────────────────────────────
LOG_DIR.mkdir(exist_ok=True)
LOG_FILE = LOG_DIR / f"{SESSION_ID}.jsonl"

def log_event(event_type: str, data: dict) -> None:
    with open(LOG_FILE, "a") as f:
        f.write(json.dumps({"ts": time.time(), "type": event_type, **data}) + "\n")

# ────────────────────────────────────────────────────────────────────────────
# Tool Result Cache
# ────────────────────────────────────────────────────────────────────────────
_cache: dict[str, tuple[str, float]] = {}  # key -> (result, created_at)
CACHE_TTL = 300  # seconds

def cache_get(tool: str, args: dict) -> str | None:
    k = hashlib.sha256(f"{tool}:{sorted(args.items())}".encode()).hexdigest()[:16]
    if k in _cache:
        result, ts = _cache[k]
        if time.time() - ts < CACHE_TTL:
            return result
    return None

def cache_set(tool: str, args: dict, result: str) -> None:
    k = hashlib.sha256(f"{tool}:{sorted(args.items())}".encode()).hexdigest()[:16]
    _cache[k] = (result, time.time())

# ────────────────────────────────────────────────────────────────────────────
# Permission Checker
# ────────────────────────────────────────────────────────────────────────────
AUTO_ALLOW_TOOLS = {"read", "glob", "grep"}
ALWAYS_DENY_RE   = re.compile(r"rm\s+-rf\s+/|:\(\)\{.*\}|curl.*\|\s*bash", re.S)

def check_permission(tool: str, args: dict) -> bool:
    if tool in AUTO_ALLOW_TOOLS:
        return True
    if tool == "bash":
        cmd = args.get("command", "")
        if ALWAYS_DENY_RE.search(cmd):
            print(f"\n[BLOCKED] Dangerous pattern detected: {cmd!r}")
            return False
        print(f"\n[BASH] {cmd!r}")
        ans = input("Allow? [y/N] ").strip().lower()
        return ans == "y"
    if tool in ("edit", "write"):
        print(f"\n[{tool.upper()}] path={args.get('file_path') or args.get('path')!r}")
        ans = input("Allow? [y/N] ").strip().lower()
        return ans == "y"
    return True

# ────────────────────────────────────────────────────────────────────────────
# Tool Implementations
# ────────────────────────────────────────────────────────────────────────────
def tool_read(file_path: str) -> str:
    try:
        lines = Path(file_path).read_text(errors="replace").splitlines()
        return "\n".join(f"{i+1}→{l}" for i, l in enumerate(lines))
    except FileNotFoundError:
        return f"ERROR: File not found: {file_path}"

def tool_write(file_path: str, content: str) -> str:
    Path(file_path).parent.mkdir(parents=True, exist_ok=True)
    Path(file_path).write_text(content)
    return f"Written {len(content)} chars to {file_path}"

def tool_edit(file_path: str, old_string: str, new_string: str) -> str:
    text = Path(file_path).read_text()
    count = text.count(old_string)
    if count == 0:
        return f"ERROR: old_string not found in {file_path}"
    if count > 1:
        return f"ERROR: old_string matches {count} locations; be more specific"
    Path(file_path).write_text(text.replace(old_string, new_string, 1))
    return "Edit applied successfully"

def tool_bash(command: str, timeout: int = 30) -> str:
    """Run with graduated retry for transient errors."""
    for attempt in range(3):
        try:
            r = subprocess.run(
                command, shell=True, capture_output=True,
                text=True, timeout=timeout
            )
            output = (r.stdout + r.stderr)[:10_000]  # cap at 10K chars
            if r.returncode == 0 or attempt == 2:
                return output or f"[exit {r.returncode}]"
            # Simple transient check
            if any(p in output.lower() for p in ["timeout", "rate limit", "503"]):
                time.sleep(2 ** attempt)
                continue
            return output  # Permanent error, return immediately
        except subprocess.TimeoutExpired:
            return f"ERROR: command timed out after {timeout}s"
    return output

def tool_glob(pattern: str) -> str:
    matches = sorted(glob.glob(pattern, recursive=True))[:100]
    return "\n".join(matches) if matches else "No matches"

def tool_grep(pattern: str, path: str = ".", flags: str = "") -> str:
    try:
        cmd = f"grep -rn {flags} {pattern!r} {path}"
        r = subprocess.run(cmd, shell=True, capture_output=True, text=True, timeout=15)
        lines = (r.stdout or r.stderr).splitlines()[:200]
        return "\n".join(lines)
    except subprocess.TimeoutExpired:
        return "ERROR: grep timed out"

TOOLS = {
    "read":  tool_read,
    "write": tool_write,
    "edit":  tool_edit,
    "bash":  tool_bash,
    "glob":  tool_glob,
    "grep":  tool_grep,
}

TOOL_SCHEMAS = [
    {"name": "read",  "description": "Read a file.", "input_schema": {"type": "object", "properties": {"file_path": {"type": "string"}}, "required": ["file_path"]}},
    {"name": "write", "description": "Write a file.", "input_schema": {"type": "object", "properties": {"file_path": {"type": "string"}, "content": {"type": "string"}}, "required": ["file_path", "content"]}},
    {"name": "edit",  "description": "Edit a file by replacing old_string with new_string.", "input_schema": {"type": "object", "properties": {"file_path": {"type": "string"}, "old_string": {"type": "string"}, "new_string": {"type": "string"}}, "required": ["file_path", "old_string", "new_string"]}},
    {"name": "bash",  "description": "Run a shell command.", "input_schema": {"type": "object", "properties": {"command": {"type": "string"}}, "required": ["command"]}},
    {"name": "glob",  "description": "Find files by glob pattern.", "input_schema": {"type": "object", "properties": {"pattern": {"type": "string"}}, "required": ["pattern"]}},
    {"name": "grep",  "description": "Search for a regex pattern in files.", "input_schema": {"type": "object", "properties": {"pattern": {"type": "string"}, "path": {"type": "string"}}, "required": ["pattern"]}},
]

# ────────────────────────────────────────────────────────────────────────────
# Context Compression (Pattern 1)
# ────────────────────────────────────────────────────────────────────────────
def maybe_compress(messages: list[dict]) -> list[dict]:
    if count_tokens(messages) < CONTEXT_LIMIT:
        return messages
    system_msg = messages[0]
    recent = messages[-6:]
    to_compress = messages[1:-6]
    if not to_compress:
        return messages
    summary_resp = client.messages.create(
        model=MODEL, max_tokens=2000,
        messages=[*to_compress, {"role": "user", "content": "Summarise this conversation compactly."}]
    )
    summary = summary_resp.content[0].text
    log_event("compression", {"removed_msgs": len(to_compress), "summary_len": len(summary)})
    return [system_msg, {"role": "assistant", "content": f"[SUMMARY]\n{summary}"}, *recent]

# ────────────────────────────────────────────────────────────────────────────
# Cost Tracker
# ────────────────────────────────────────────────────────────────────────────
_total_cost = 0.0
PRICE_PER_1K = {"input": 0.003, "output": 0.015}  # claude-3-5-sonnet

def track_cost(usage) -> None:
    global _total_cost
    cost = (usage.input_tokens / 1000 * PRICE_PER_1K["input"] +
            usage.output_tokens / 1000 * PRICE_PER_1K["output"])
    _total_cost += cost
    log_event("cost", {"turn_cost": cost, "total_cost": _total_cost})
    if _total_cost > COST_LIMIT:
        raise RuntimeError(f"Cost limit ${COST_LIMIT:.2f} exceeded (${_total_cost:.2f}). "
                           "Increase COST_LIMIT to continue.")

# ────────────────────────────────────────────────────────────────────────────
# Agent Loop
# ────────────────────────────────────────────────────────────────────────────
SYSTEM_PROMPT = """You are a helpful AI assistant with access to file system tools.
Work methodically: read before you edit, verify after you change, and ask before
doing anything irreversible. When a task is complete, respond with plain text only
(no tool calls)."""

def run_agent(task: str) -> None:
    messages: list[dict] = [{"role": "user", "content": task}]
    log_event("session_start", {"task": task})

    while True:
        messages = maybe_compress(messages)
        response = client.messages.create(
            model=MODEL, max_tokens=MAX_TOKENS,
            system=SYSTEM_PROMPT, tools=TOOL_SCHEMAS,
            messages=messages
        )
        track_cost(response.usage)
        log_event("model_response", {"stop_reason": response.stop_reason})

        # Collect text and tool calls from the response
        assistant_parts = []
        tool_calls = []
        for block in response.content:
            if block.type == "text":
                assistant_parts.append({"type": "text", "text": block.text})
                if block.text:
                    print(f"\n[Agent] {block.text}")
            elif block.type == "tool_use":
                tool_calls.append(block)

        messages.append({"role": "assistant", "content": response.content})

        if response.stop_reason == "end_turn" or not tool_calls:
            print("\n[Session complete]")
            break

        # Execute tool calls
        tool_results = []
        for tc in tool_calls:
            tool_name = tc.name
            args = tc.input
            log_event("tool_call", {"tool": tool_name, "args": args})

            # Cache lookup (read-only tools)
            cached = cache_get(tool_name, args) if tool_name in AUTO_ALLOW_TOOLS else None
            if cached:
                result = cached
                log_event("cache_hit", {"tool": tool_name})
            else:
                if not check_permission(tool_name, args):
                    result = f"ERROR: Permission denied for {tool_name}"
                else:
                    fn = TOOLS.get(tool_name)
                    result = fn(**args) if fn else f"ERROR: Unknown tool {tool_name!r}"
                    if tool_name in AUTO_ALLOW_TOOLS:
                        cache_set(tool_name, args, result)
                    # Invalidate cache on writes
                    if tool_name in ("edit", "write"):
                        path = args.get("file_path", "")
                        _cache.clear()  # Simple: clear all on any write

            log_event("tool_result", {"tool": tool_name, "result_len": len(result)})
            tool_results.append({
                "type": "tool_result",
                "tool_use_id": tc.id,
                "content": result[:8_000],  # cap individual result
            })

        messages.append({"role": "user", "content": tool_results})

if __name__ == "__main__":
    import sys
    task = " ".join(sys.argv[1:]) or "List all Python files in the current directory."
    run_agent(task)

8. Interview Tips: Discussing Harness Architectures

How to Talk About These Systems at the Staff Level ▼

Staff-level interviews for AI platform roles frequently involve architecture discussions about coding assistants, agent platforms, and LLM harnesses. The following questions and answer frameworks are drawn from real interviews at companies building these systems.

Q: "How does Claude Code manage context across a long coding session?"

What they're testing: Do you understand that LLMs are stateless and the harness owns state management? Can you explain compaction vs. truncation?

Strong answer framework: Explain the agent loop, the accumulation problem (tool results grow the context), and the two strategies: (1) proactive compaction via summarisation when a high-water mark is hit, and (2) priority-based eviction rather than FIFO truncation. Mention that the system prompt is always pinned and compaction happens before the window is full, not after.

Q: "Design a permission system for an AI coding agent that prevents data exfiltration."

What they're testing: Security thinking beyond naive tool-name blocking. Do you understand that Bash is Turing-complete?

Strong answer framework: Three tiers: (1) read-only tools auto-approved, (2) write/exec tools prompt-approved, (3) semantic payload analysis for Bash (detect redirections, curl-pipe-bash, rm -rf patterns). Add: container-based isolation with read-only mounts on sensitive paths. Mention that defence in depth means even if the model is compromised, the container prevents filesystem damage.

Q: "Cursor uses multiple models. How would you design the routing logic?"

What they're testing: Understanding of latency/capability trade-offs, ability to design a router that serves multiple request types.

Strong answer framework: Classify requests by: (1) latency requirement (tab completion < 50 ms → small model), (2) complexity (multi-file refactor → large model), (3) context size (fits in small model's window vs. requires large). Use a fast heuristic first pass (token count, number of files referenced) rather than sending every request to the large model to determine routing.

Q: "Walk me through a production failure you've seen or designed around in an AI system."

What they're testing: Production experience, ability to learn from failure, systems thinking.

Strong answer framework: Use the postmortem structure: what happened, root cause (go deep — surface "the agent looped" but explain why the harness allowed it), immediate fix, and systemic prevention. Connect it to a general pattern (graduated retry, loop detection, budget limits). Finish with what monitoring you'd add to catch it earlier next time.

Q: "Assistants API vs. self-hosted harness — when would you choose each?"

What they're testing: Build vs. buy judgment, understanding of what managed services trade away.

Strong answer framework: Assistants API wins when: you want fast time-to-market, you don't need custom models, your data residency requirements are met by OpenAI, and you don't need deep observability. Self-hosted wins when: you need multi-provider model routing, custom tool execution environments, full audit trails, data locality, or cost control at scale. Call out that "control" is the crux: managed services abstract the loop, so debugging production issues is harder.

9. Quiz

Test Your Understanding (5 Questions) ▼

1. Claude Code uses an Edit tool that requires an old_string parameter. Why not use a simpler line-number-based patch format?

Line numbers become stale as soon as earlier lines are modified. Requiring a unique old_string forces the model to identify its target by content rather than position, making edits robust to prior changes in the same session. It also forces the model to read the file first and verify the exact text, reducing silent corruption from off-by-one assumptions.

2. In the "Context Overflow — Lost System Prompt" postmortem, why is FIFO truncation dangerous? What property must a context compaction strategy guarantee?

FIFO truncation drops the oldest messages first. The system prompt — which is message[0] — is therefore the first thing removed, taking all model instructions with it. The model then operates without its guiding constraints. A correct compaction strategy must guarantee: (1) the system prompt is always present (pinned, priority = infinity), (2) compaction triggers before the window is full (at ~80%), giving room to generate the summary, and (3) eviction removes lowest-value messages first (old tool results before the original task statement).

3. A model has "Bash auto-allowed, Write tool denied." Describe how the agent could still write files and what architectural layer prevents this.

Shell redirection: echo "content" > file.txt writes a file without using the Write tool. Any Bash command with >, >>, tee, or filesystem-modifying utilities (cp, mv) achieves the same effect. Prevention requires a semantic payload analysis layer (Permission Checkpoint pattern) that inspects Bash command strings for filesystem side-effects, not just tool names. Defence in depth: run the agent in a container with read-only filesystem mounts on sensitive paths so even if the payload check is bypassed, the OS denies the write.

4. Devin uses screenshot-based verification. What class of errors does this catch that stdout-only verification misses? What is the cost tradeoff?

Screenshots catch visual/rendering errors: CSS layout breakage, UI component failures that show a blank screen instead of an error, browser-rendered output that differs from server output, and GUI application state that is not reflected in any log. Stdout verification only catches process-level errors. Cost: screenshots are large multimodal inputs (typically 50–200K tokens equivalent in image tokens), making them expensive if taken after every action. The tradeoff is selective application: take screenshots after high-value verification points (test run completion, browser navigation, app launch) rather than after every shell command.

5. The Graduated Retry pattern classifies errors as transient, permanent, or escalate. Give a concrete example of each, and explain why misclassifying a permanent error as transient is more damaging than misclassifying a transient error as permanent.

Transient: HTTP 429 (rate limit), TCP connection reset, DNS timeout — will succeed if retried after a delay.
Permanent: "command not found", syntax error, "file not found" — retrying will never succeed; the model must change strategy.
Escalate: "disk full", "killed" (OOM) — retrying is harmful; requires human intervention.

Why misclassifying permanent-as-transient is worse: It creates an infinite retry loop that consumes tokens and wall-clock time indefinitely (as in the infinite loop postmortem). Misclassifying transient-as-permanent means the agent gives up on something that would have succeeded — a missed opportunity but a recoverable situation (the model can try a different approach). The former failure mode can run up a large bill; the latter merely requires the user to re-run the task.

Mini-Project: Extend the Capstone Harness with 3 Features

Using the minimal_harness.py from Section 7 as your starting point, implement the following three extensions. Each one exercises a core concept from the course and produces a measurable, testable output.

Extension 1: Context Priority Queue (Replaces maybe_compress)

Implement the evict_to_budget function from Pattern 4 and replace maybe_compress with it. Assign priority scores:

  • System prompt: float("inf")
  • User task message: 100
  • Any message containing "ERROR" or "ESCALATE": 70
  • Assistant text responses: 50
  • Tool results from read or bash: 20

Test: Write a test that creates a message list exceeding 200K tokens with mixed priorities, calls evict_to_budget(messages, 100_000), and asserts that (a) the system prompt is always present and (b) read tool results were evicted before assistant responses.

Extension 2: Session Resume from JSONL

Modify run_agent to accept an optional session_id parameter. When provided, load the existing .harness_logs/{session_id}.jsonl file and reconstruct the message list from all logged model_response and tool_result events before continuing the session.

Test: Run a session, kill it mid-way (Ctrl-C), resume it with the same session ID, and verify that the agent continues from the correct state (does not re-read files it already read).

Extension 3: Per-Session Cost Dashboard

After each agent turn, print a compact cost summary line to stderr:

[Turn 5] input=12,430 output=847 tokens | turn=$0.042 | total=$0.18 | budget remaining=$1.82

At session end, write a JSON summary file to .harness_logs/{session_id}_cost.json containing: total_turns, total_input_tokens, total_output_tokens, total_cost_usd, cost_per_turn (list), most_expensive_turn (index + cost).

Test: Run the harness on a known task, load the cost JSON, and assert that total_cost_usd == sum(cost_per_turn) and that all values are positive.

Checkpoint Summary: Production Patterns & Case Studies

tl;dr: Production harnesses succeed not because of clever prompting, but because of robust engineering: priority-based context management, semantic permission checks, retry budgets, cost limits, and observable session logs. Study real systems to understand why each constraint exists.

Key Takeaways

  • Claude Code is CLI-native, prioritises local execution and developer environment access, uses file-based memory, and JSONL session state.
  • Cursor adds IDE-awareness (LSP, cursor position, diff-based editing) and routes requests across a multi-model tier based on latency requirements.
  • Devin trades cost efficiency for autonomy via cloud-hosted VMs, mandatory planning, and screenshot-based verification of complex states.
  • OpenAI Assistants API manages the agent loop server-side, reducing boilerplate but sacrificing observability, model choice, and data control.
  • The four canonical failure modes are: infinite loop (no retry budget), context overflow (FIFO truncation drops system prompt), permission escalation (Bash bypasses tool-level restrictions), and cost explosion (agent lacks cost-awareness).
  • Five patterns address these: Conversation Compressor, Tool Result Cache, Graduated Retry, Context Priority Queue, Permission Checkpoint.
  • The ~200-line capstone harness implements all these patterns and provides named extension points for every advanced module topic.

How I'd Teach This

Start with the minimal harness and break it in four specific ways (remove the retry limit, remove system prompt pinning, allow unrestricted Bash, remove cost tracking). Watch each failure mode materialise. Then add each fix one at a time. Understanding by breaking-and-fixing builds far stronger intuition than studying patterns in the abstract.

Flashcard Q&A Pairs (for Anki)

  1. Q: Why does Claude Code's Edit tool use old_string rather than line numbers? A: Line numbers are stale after prior edits; old_string forces content-based targeting that is position-independent.
  2. Q: What is the correct context compaction trigger threshold and why? A: ~80% of the limit, not 100%, to leave room to generate the summary without hitting the limit during the summarisation call.
  3. Q: Name the three tiers in a harness permission model. A: Auto-allow (read-only tools), prompt-user (write/exec), always-deny (dangerous patterns).
  4. Q: What does Devin's planning system prevent that implicit chain-of-thought does not? A: It surfaces misunderstandings before expensive tool calls; users can correct the plan before execution begins.
  5. Q: The Graduated Retry pattern classifies errors into three buckets. Name them. A: Transient (retry with backoff), permanent (return error, change strategy), escalate (halt, require human).
