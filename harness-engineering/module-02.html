Context Engineering: Deep Dive

The context window is the most constrained and expensive resource in any agent harness. This module teaches you to treat it like a precious cache: what to put in, how to organize it, when to evict, and how to make the model cache work for you — not against you.

1. Why Context Engineering is THE Problem

The Context Window as a Scarce Resource ▼

Every API call to Claude, GPT-4, or Gemini sends a payload of tokens. The model reads them all — but its attention mechanism has a fixed capacity. Modern flagship models offer windows ranging from 128K (GPT-4o) to 200K (Claude 3.5 Sonnet) to 1M (Gemini 1.5 Pro). Despite that growth, production agents routinely hit the ceiling because each tool call injects fresh tokens while nothing leaves.

Analogy: RAM, not disk. Think of the context window the same way you think about L1/L2 cache in a CPU. You have a small, fast, expensive cache. Every cache line you load costs you; every eviction is information lost. The art of context engineering is identical to cache replacement policy design — you are writing the equivalent of LRU, LFU, or ARC algorithms, but for language model inputs.

The Real Dollar Cost

Context costs money on two dimensions: input tokens (reading) and output tokens (writing). With Claude 3.5 Sonnet pricing at $3/M input tokens and $15/M output tokens, let's compute the cost of a typical agent session:

Scenario: 50-turn coding session, Claude 3.5 Sonnet System prompt: 4,000 tokens (constant)
Tool schemas (15 tools): 3,500 tokens (constant)
Avg conversation/turn: +1,200 tokens (grows each turn)
Avg tool output/turn: +2,500 tokens (grows each turn)

Turn 1 context: ~9,000 tokens → $0.027 Turn 25 context: ~99,500 tokens → $0.299 Turn 50 context: ~189,500 tokens → $0.569 (this turn alone) Session total (naive): ~$7.20 input cost alone With smart budgeting: ~$1.80 (75% reduction via caching + truncation)

For a B2C product with 100,000 daily active sessions, that $5.40 per-session saving compounds to $540,000/day. This is why Anthropic, OpenAI, and every serious AI company dedicates entire engineering teams to context management.

Why "Dump Everything" Fails

Naive implementations append every message and tool result to a growing array and pass it all to the model. This approach fails in three ways:

Failure 1 Context Window Overflow After 20-40 turns on complex tasks, you hit the limit. The API returns a 400 error mid-task. The agent fails silently or crashes.
Failure 2 Attention Dilution ("Lost in the Middle") Stanford research (2023) shows LLMs perform worst on information placed in the middle of a long context. A 50-turn conversation buries the most recent user intent under thousands of old tokens.
Failure 3 Cost Explosion Each subsequent turn re-sends all previous tokens as input. Cost is O(n²) in the number of turns. A 200-turn debugging session can cost $40+ with no caching.

Real Systems: How Production Harnesses Handle It

Claude Code (~200K window)

  • System prompt ~8K tokens (model card + tool schemas)
  • File content injected as tool results, not conversation
  • Smart truncation: first 100 + last 100 lines of large files
  • Conversation history hard cap: ~40 turns retained
  • Bash outputs capped at 8K tokens per result
  • Prompt cache on system prompt + tool schemas

Cursor (~128K window)

  • Codebase indexed into vector embeddings first
  • Retrieves top-k relevant files per query (RAG-in-context)
  • Active file always gets highest priority allocation
  • Git diff injected as structured patch, not full file
  • Conversation history limited to last 10 turns by default
  • LSP symbols (function signatures) over full file bodies

2. Token Budgeting Algorithms

Fixed vs Dynamic Allocation ▼

There are two schools of thought on how to allocate the context window. The right choice depends on the predictability of your agent's tasks.

Fixed Allocation

Pre-assign hard token budgets to each category. Simple to reason about, no risk of one category starving another.

MAX = 200,000

SYSTEM_PROMPT  =  8,000  (4%)
TOOL_SCHEMAS   =  4,000  (2%)
RESPONSE_RSRV  = 16,000  (8%)
HISTORY        = 80,000  (40%)
TOOL_RESULTS   = 72,000  (36%)
FREE           = 20,000  (10%)

Best for: chatbots, customer service agents with predictable conversation shapes.

Dynamic Allocation

Measure actual usage of each category at runtime, allocate remaining budget to highest-priority needs.

available = MAX
           - count(system_prompt)
           - count(tool_schemas)
           - RESPONSE_RESERVE
# Divide rest by priority queue order
# Trim lower-priority items first

Best for: coding agents, research agents where tool output sizes are unpredictable.

The Priority Queue Approach

Dynamic allocation works via a priority queue. Each context element is assigned a priority. When budget runs low, the scheduler trims or evicts the lowest-priority elements first. Here is the canonical priority ordering for a coding agent:

P1 — Critical System Prompt + Tool Schemas ~8-15K tokens NEVER drop
P1 — Critical Response Reserve ~8-16K tokens NEVER drop
P2 — High Most Recent User Message ~500-3K tokens Keep verbatim
P2 — High Most Recent Tool Results (last 3) ~2-20K tokens Trim if large
P3 — Medium Recent Conversation History (last 10 turns) ~5-30K tokens Trim oldest first
P4 — Low Older Tool Results + File Contents ~0-80K tokens Summarize or drop
P5 — Ephemeral Intermediate Reasoning / Scratch Pad ~0-20K tokens Drop freely

Budget Calculation Formula

available_budget = total_window - count(system_prompt) - count(tool_schemas) - response_reserve
Example (Claude 3.5 Sonnet, 200K window):
available = 200,000 - 8,200 - 3,800 - 12,000 = 176,000 tokens

If history_tokens + tool_result_tokens > available_budget: overflow = (history + tools) - available
Trim P5 first → P4 → P3 → until overflow == 0

Python: TokenBudgetManager Class

from dataclasses import dataclass, field
from typing import Optional
import tiktoken

@dataclass
class ContextItem:
    """A single allocatable piece of context."""
    content: str
    priority: int          # 1=critical, 5=ephemeral
    category: str          # 'system', 'history', 'tool_result', etc.
    turn_index: int = 0    # When was this added?
    token_count: int = 0   # Populated by TokenBudgetManager

    def __post_init__(self):
        if self.token_count == 0:
            # Lazy count — will be filled by manager
            pass


class TokenBudgetManager:
    """
    Production token budget manager for agent harnesses.

    Implements a priority-queue-based allocation strategy:
    - P1 items (system, schemas, reserve) are NEVER dropped
    - Lower priority items are truncated/evicted when budget is exceeded
    - Supports both fixed and dynamic allocation modes
    """

    PRIORITIES = {
        'system_prompt': 1,
        'tool_schemas':  1,
        'response_reserve': 1,
        'last_user_msg': 2,
        'recent_tool_results': 2,
        'history_recent': 3,
        'tool_results_old': 4,
        'history_old': 4,
        'scratch_pad': 5,
    }

    def __init__(
        self,
        model: str = "claude-3-5-sonnet-20241022",
        total_tokens: int = 200_000,
        response_reserve: int = 12_000,
        encoding: str = "cl100k_base",   # approximation for Claude
    ):
        self.model = model
        self.total_tokens = total_tokens
        self.response_reserve = response_reserve
        self.enc = tiktoken.get_encoding(encoding)
        self._items: list[ContextItem] = []

    # ------------------------------------------------------------------ #
    # Public API                                                           #
    # ------------------------------------------------------------------ #

    def add(self, content: str, category: str, turn_index: int = 0) -> ContextItem:
        """Add a context item; category determines its priority."""
        priority = self.PRIORITIES.get(category, 3)
        token_count = self._count(content)
        item = ContextItem(
            content=content,
            priority=priority,
            category=category,
            turn_index=turn_index,
            token_count=token_count,
        )
        self._items.append(item)
        return item

    def build(self) -> list[ContextItem]:
        """
        Return the items that fit within budget, applying
        priority-based eviction for overflow.
        """
        fixed_cost = self._fixed_token_cost()
        available = self.total_tokens - fixed_cost - self.response_reserve

        if available <= 0:
            raise ValueError(
                f"Fixed context items ({fixed_cost} tokens) + response reserve "
                f"({self.response_reserve}) already exceeds total budget "
                f"({self.total_tokens})"
            )

        # Separate P1 items (must keep) from flexible items
        must_keep = [i for i in self._items if i.priority == 1]
        flexible  = sorted(
            [i for i in self._items if i.priority > 1],
            key=lambda x: (x.priority, -x.turn_index)  # P2 first, newest first
        )

        budget_used = 0
        selected: list[ContextItem] = list(must_keep)

        for item in flexible:
            if budget_used + item.token_count <= available:
                selected.append(item)
                budget_used += item.token_count
            # else: item is evicted (not included in output)

        return selected

    def budget_report(self) -> dict:
        """Return a dict summarising token usage per category."""
        report = {}
        for item in self._items:
            cat = item.category
            report[cat] = report.get(cat, 0) + item.token_count
        report['total'] = sum(report.values())
        report['available'] = self.total_tokens - self.response_reserve
        report['utilization_pct'] = round(
            100 * report['total'] / report['available'], 1
        )
        return report

    # ------------------------------------------------------------------ #
    # Private helpers                                                      #
    # ------------------------------------------------------------------ #

    def _count(self, text: str) -> int:
        """Count tokens (using tiktoken as a proxy for Claude)."""
        return len(self.enc.encode(text))

    def _fixed_token_cost(self) -> int:
        """Sum tokens for P1 items only."""
        return sum(
            i.token_count for i in self._items if i.priority == 1
        )


# Usage example
if __name__ == "__main__":
    mgr = TokenBudgetManager(total_tokens=200_000, response_reserve=12_000)

    mgr.add("You are an expert software engineer...", "system_prompt")
    mgr.add('{"name": "bash", "description": "Run shell commands..."}', "tool_schemas")

    for turn in range(30):
        mgr.add(f"User turn {turn}: Please fix the bug in auth.py", "history_recent", turn_index=turn)
        mgr.add(f"File content: {'x' * 2000}", "tool_results_old", turn_index=turn)

    selected = mgr.build()
    print(f"Selected {len(selected)} of {len(mgr._items)} items")
    print(mgr.budget_report())
Production insight: Claude Code uses a variation of this algorithm but with one extra wrinkle: tool results from the same "logical step" (e.g., bash call + its output) are treated as an atomic unit. They are never split — the entire unit is kept or dropped. This prevents the model from seeing a tool call without its result, which causes hallucination.

3. Context Assembly Pipeline

The 5-Stage Assembly Pipeline ▼

Raw context items — tool results, conversation history, file contents, user messages — cannot be passed directly to the model. They must flow through a structured pipeline that ensures budget compliance, correct ordering, and format consistency.

Stage 1 Collect Gather all raw items
Stage 2 Prioritize Score & rank items
Stage 3 Truncate Apply budgets
Stage 4 Format Build message array
Stage 5 Validate Final budget check
Stage 1: Collect all raw context items from memory, tools, and user input into a flat list.

TypeScript: ContextAssembler Class

import Anthropic from "@anthropic-ai/sdk";
import { encode } from "gpt-tokenizer"; // proxy tokenizer for Claude

type Priority = 1 | 2 | 3 | 4 | 5;

interface ContextEntry {
  id: string;
  priority: Priority;
  category: "system" | "history" | "tool_result" | "user" | "assistant";
  turnIndex: number;
  tokenCount: number;
  message: Anthropic.MessageParam | null; // null for system prompt
  systemText?: string;                    // only for system category
}

export class ContextAssembler {
  private readonly maxTokens: number;
  private readonly responseReserve: number;
  private entries: ContextEntry[] = [];
  private systemParts: string[] = [];

  constructor(
    maxTokens = 200_000,
    responseReserve = 12_000,
  ) {
    this.maxTokens = maxTokens;
    this.responseReserve = responseReserve;
  }

  /** Stage 1: Collect */
  addSystemPrompt(text: string): void {
    const tokens = this.countTokens(text);
    this.systemParts.push(text);
    this.entries.push({
      id: `system-${Date.now()}`,
      priority: 1,
      category: "system",
      turnIndex: -1,
      tokenCount: tokens,
      message: null,
      systemText: text,
    });
  }

  addTurn(
    role: "user" | "assistant",
    content: string | Anthropic.ContentBlock[],
    turnIndex: number,
    priority: Priority = 3,
  ): void {
    const text = typeof content === "string" ? content : JSON.stringify(content);
    const tokens = this.countTokens(text);
    this.entries.push({
      id: `${role}-${turnIndex}`,
      priority,
      category: role,
      turnIndex,
      tokenCount: tokens,
      message: { role, content },
    });
  }

  addToolResult(
    toolUseId: string,
    result: string,
    turnIndex: number,
  ): void {
    // Tool results are slightly higher priority than old history
    const priority: Priority = turnIndex === this.latestTurn() ? 2 : 4;
    const tokens = this.countTokens(result);
    this.entries.push({
      id: `tool-${toolUseId}`,
      priority,
      category: "tool_result",
      turnIndex,
      tokenCount: tokens,
      message: {
        role: "user",
        content: [{ type: "tool_result", tool_use_id: toolUseId, content: result }],
      },
    });
  }

  /** Stages 2-5: Prioritize → Truncate → Format → Validate */
  assemble(): { system: string; messages: Anthropic.MessageParam[] } {
    // Stage 2: Sort by priority asc, then by recency desc
    const sorted = [...this.entries].sort((a, b) => {
      if (a.priority !== b.priority) return a.priority - b.priority;
      return b.turnIndex - a.turnIndex;
    });

    // Stage 3: Truncate — greedy fill within budget
    const fixedCost = sorted
      .filter((e) => e.priority === 1)
      .reduce((sum, e) => sum + e.tokenCount, 0);
    const budget = this.maxTokens - fixedCost - this.responseReserve;
    let used = 0;
    const selected = new Set();

    for (const entry of sorted) {
      if (entry.priority === 1) {
        selected.add(entry.id);
        continue;
      }
      if (used + entry.tokenCount <= budget) {
        selected.add(entry.id);
        used += entry.tokenCount;
      }
    }

    // Stage 4: Format — rebuild ordered message array
    const systemText = this.entries
      .filter((e) => e.category === "system" && selected.has(e.id))
      .map((e) => e.systemText!)
      .join("\n\n");

    const messages: Anthropic.MessageParam[] = this.entries
      .filter((e) => e.category !== "system" && selected.has(e.id) && e.message)
      .sort((a, b) => a.turnIndex - b.turnIndex)
      .map((e) => e.message!);

    // Stage 5: Validate
    const totalTokens = this.countTokens(systemText) +
      messages.reduce((sum, m) => sum + this.countTokens(JSON.stringify(m)), 0);

    if (totalTokens + this.responseReserve > this.maxTokens) {
      throw new Error(
        `Context assembly validation failed: ${totalTokens} tokens exceeds budget`
      );
    }

    return { system: systemText, messages };
  }

  private latestTurn(): number {
    return Math.max(...this.entries.map((e) => e.turnIndex), 0);
  }

  private countTokens(text: string): number {
    return encode(text).length;
  }
}
Message ordering constraint: The Anthropic Messages API requires messages to alternate between user and assistant roles. If you drop a turn from the middle of a conversation (e.g., turn 7 out of 20), you may create consecutive same-role messages — which the API will reject with a 422 error. Always verify alternation invariant after truncation, and merge consecutive same-role messages if needed.

4. Sliding Window Strategies

Keeping the Most Relevant Context ▼

A sliding window is the core eviction strategy when conversation history grows beyond budget. The question is not just which messages to drop, but how to drop them without losing critical state. There are three main strategies:

Strategy 1: Naive Truncation (Drop Oldest)

Full history (30 turns, exceeds budget): SYS U1 A1 U2 A2 U3 A3 ... U20 A20 U21 A21 U22 A22

Result: Recent messages kept verbatim. Old context completely lost.

Strategy 2: Summarization-Based Compression

After summarization pass: SYS SUMMARY: Turns 1-19 U20 A20 U21 A21 U22 A22

Result: Old context compressed to a summary block. Retains gist, loses exact details.

Strategy 3: Hybrid Window (Recommended for Production)

Hybrid: keep system + recent + summarized middle: SYS U1 A1 U2 A2 SUMMARY: Turns 3-18 U19 A19 U20 A20

Result: Keeps original intent (first N turns) + recent context + compressed middle.
Why keep the first few turns? The first 1-2 turns typically contain the user's original goal or task specification. This is the highest-signal information in the entire conversation. Dropping it in favor of recent turns causes "goal drift" — the model forgets what it was originally trying to accomplish. Always anchor your window at both ends.

Python: Sliding Window with Summarization

import anthropic
from dataclasses import dataclass

@dataclass
class Message:
    role: str    # 'user' or 'assistant'
    content: str
    turn_index: int

class SlidingWindowContext:
    """
    Hybrid sliding window: keep first N turns + recent M turns,
    summarize the middle when budget is exceeded.
    """

    def __init__(
        self,
        client: anthropic.Anthropic,
        model: str = "claude-3-haiku-20240307",   # cheap summarizer
        max_history_tokens: int = 40_000,
        anchor_turns: int = 2,      # always keep first N turns verbatim
        recent_turns: int = 10,     # always keep last M turns verbatim
        summary_model: str = "claude-3-haiku-20240307",
    ):
        self.client = client
        self.model = model
        self.summary_model = summary_model
        self.max_history_tokens = max_history_tokens
        self.anchor_turns = anchor_turns
        self.recent_turns = recent_turns
        self.history: list[Message] = []
        self._summary: str | None = None
        self._summary_covers_up_to: int = -1

    def add_message(self, role: str, content: str) -> None:
        turn = len(self.history)
        self.history.append(Message(role=role, content=content, turn_index=turn))
        self._maybe_compress()

    def build_messages(self) -> list[dict]:
        """
        Build the messages array to send to the API.
        Returns: system_injection (optional) + messages
        """
        anchor = self.history[:self.anchor_turns]
        recent = self.history[-self.recent_turns:]

        messages = []
        for msg in anchor:
            messages.append({"role": msg.role, "content": msg.content})

        if self._summary:
            # Inject summary as a system-level note in a user message
            messages.append({
                "role": "user",
                "content": f"[Context from earlier in conversation: {self._summary}]",
            })
            # Need a placeholder assistant ACK to maintain alternation
            messages.append({
                "role": "assistant",
                "content": "Understood, I have that context.",
            })

        for msg in recent:
            if msg.turn_index >= self.anchor_turns:  # avoid duplicating anchors
                messages.append({"role": msg.role, "content": msg.content})

        return messages

    def _maybe_compress(self) -> None:
        """Check if history exceeds budget; compress if so."""
        total_chars = sum(len(m.content) for m in self.history)
        # rough char-to-token ratio: 4 chars ≈ 1 token
        estimated_tokens = total_chars // 4

        if estimated_tokens <= self.max_history_tokens:
            return

        # Identify turns eligible for summarization
        compressible_start = self.anchor_turns
        compressible_end   = max(self.anchor_turns, len(self.history) - self.recent_turns)

        if compressible_end <= compressible_start:
            return  # not enough turns to compress

        to_summarize = self.history[compressible_start:compressible_end]
        conversation_text = "\n".join(
            f"{m.role.upper()}: {m.content}" for m in to_summarize
        )

        response = self.client.messages.create(
            model=self.summary_model,
            max_tokens=1024,
            system="You are a conversation summarizer. Create a concise summary "
                   "of the key facts, decisions, and context from this conversation. "
                   "Focus on information that would be needed to continue the task.",
            messages=[{"role": "user", "content": conversation_text}],
        )
        self._summary = response.content[0].text
        self._summary_covers_up_to = compressible_end - 1

        # Remove compressed turns from history (keep anchors + recent)
        self.history = (
            self.history[:self.anchor_turns] +
            self.history[compressible_end:]
        )

5. Truncation vs Summarization

Choosing the Right Compression Strategy ▼

Not all context is worth the cost of summarization. The decision is economic: is the value of the compressed information greater than the cost of the summarization API call?

Truncate When...

  • Content is verbose scaffolding (stack traces, full file reads)
  • Tool output has a clear start/end with boilerplate in the middle
  • The exact text doesn't matter, only key values do
  • Content can be re-fetched on demand (idempotent tool calls)
  • The session is new — not enough history to summarize

Summarize When...

  • Conversation history encodes decisions + rationale
  • The user has given preferences that should persist
  • Earlier work produced results that later steps depend on
  • The session is long (30+ turns) with multiple topic shifts
  • You're running a multi-day agentic workflow

Cost Analysis: Summarization vs Lost Context

Summarization cost = input_tokens_compressed × $3/M + output_tokens_summary × $15/M
Example: compress 30K tokens of history into a 2K summary
Input cost: 30,000 × $3/1,000,000 = $0.090
Output cost: 2,000 × $15/1,000,000 = $0.030
Total summarization cost: $0.120

Value of keeping that 30K verbatim (if it fits): Cost per turn saved from re-work = estimated re-do cost × probability of needing it
If re-work probability = 20%, re-do cost = $2.00 per re-work
Expected value = 0.20 × $2.00 = $0.40

Decision: $0.40 > $0.12 → Summarize is worth it here.
But if re-work probability = 2%: expected value = $0.04 < $0.12 → Truncate instead.

Smart Truncation for File Contents

File contents are the biggest single consumer of context tokens in coding agents. A 1,000-line Python file is ~30K tokens. Reading 10 files would exhaust most of a 200K window. Smart truncation uses the first + last N lines strategy, which preserves imports, class/function signatures (top) and the most recent logic (bottom):

def smart_truncate_file(
    content: str,
    max_tokens: int = 4_000,
    head_lines: int = 80,
    tail_lines: int = 80,
    encoding: str = "cl100k_base",
) -> str:
    """
    Truncate a file to fit within max_tokens while preserving
    the most structurally important parts:
    - Head: imports, module docstring, class/function signatures
    - Tail: most recently written logic
    """
    import tiktoken
    enc = tiktoken.get_encoding(encoding)

    if len(enc.encode(content)) <= max_tokens:
        return content  # already fits

    lines = content.splitlines()
    total = len(lines)

    if total <= head_lines + tail_lines:
        # File is short; just encode as-is (may still exceed budget)
        return content

    head = lines[:head_lines]
    tail = lines[total - tail_lines:]
    omitted = total - head_lines - tail_lines

    truncated = (
        "\n".join(head) +
        f"\n\n# ... [{omitted} lines omitted for context budget] ...\n\n" +
        "\n".join(tail)
    )
    return truncated


class TruncationStrategy:
    """Strategy interface for different truncation approaches."""

    def truncate(self, content: str, max_tokens: int) -> str:
        raise NotImplementedError


class HeadTailTruncation(TruncationStrategy):
    """Keep first N% and last N% of content."""
    def __init__(self, head_fraction: float = 0.4, tail_fraction: float = 0.4):
        self.head_fraction = head_fraction
        self.tail_fraction = tail_fraction

    def truncate(self, content: str, max_tokens: int) -> str:
        lines = content.splitlines()
        head_n = int(len(lines) * self.head_fraction)
        tail_n = int(len(lines) * self.tail_fraction)
        return smart_truncate_file(content, max_tokens, head_n, tail_n)


class SentenceBoundaryTruncation(TruncationStrategy):
    """For prose/docs: truncate at sentence boundaries, not mid-sentence."""
    def truncate(self, content: str, max_tokens: int) -> str:
        import re, tiktoken
        enc = tiktoken.get_encoding("cl100k_base")
        sentences = re.split(r'(?<=[.!?])\s+', content)
        result, used = [], 0
        for s in sentences:
            t = len(enc.encode(s))
            if used + t > max_tokens:
                break
            result.append(s)
            used += t
        return " ".join(result) + " [truncated]"
Warning: Never truncate tool_use / tool_result pairs asymmetrically. The Anthropic API requires that every tool_use block has a corresponding tool_result block in a subsequent message. If you truncate one but not the other, the API returns a 400 error. Always treat them as an atomic pair — evict both or neither.

6. Prompt Caching

Anthropic Prompt Caching: Mechanics and Cost Math ▼

Anthropic's prompt caching feature (released 2024) allows you to mark portions of your context as cacheable. When the cache is hit, you pay 10% of the normal input token price for those tokens. This is the single highest-leverage optimization available for agent harnesses with stable system prompts.

How It Works

You add a cache_control object with type: "ephemeral" to content blocks you want cached. Anthropic stores a KV cache of the transformer's key-value state up to that point. On the next call, if the prefix matches, the cached state is reused — the model doesn't re-process those tokens, and you pay 10% of input price.

Cache write cost: 1.25× normal input price (one-time on first call) Cache read cost: 0.10× normal input price (every subsequent call) Cache TTL: 5 minutes (Anthropic resets if not used for 5 min)
Break-even: writing a 10K-token prefix costs 12,500 token-equivalents.
First cached read saves 9,000 token-equivalents (90% of 10K).
Break-even at: 2 API calls (write + 1 read already profitable)

Designing Cache-Friendly Context Layouts

The cache only works on prefixes. Anthropic computes a hash of the message array from the beginning up to the cache_control breakpoint. If you insert dynamic content before a cacheable block, the cache is busted. The golden rule is: static content must come before dynamic content.

BAD System prompt → User message → [cache_control] → Tool schemas Cache busts every turn because user message changes
GOOD System prompt [cache_control] → Tool schemas [cache_control] → User messages → Tool results Cache hits on every turn for the stable prefix

Code: Cache-Aware Context Builder

import Anthropic from "@anthropic-ai/sdk";

/**
 * Cache-aware context builder.
 *
 * Layout strategy (STATIC before DYNAMIC):
 *   1. System prompt          [cache_control: ephemeral]  ← stable, always cached
 *   2. Tool schemas           [cache_control: ephemeral]  ← stable, always cached
 *   3. Long static docs       [cache_control: ephemeral]  ← semi-stable, cached
 *   4. Conversation history                               ← dynamic, never cached
 *   5. Latest user message                                ← dynamic, never cached
 */
export class CacheAwareContextBuilder {
    private client: Anthropic;
    private systemPrompt: string;
    private toolSchemas: Anthropic.Tool[];
    private staticDocs: string[];

    constructor(
        client: Anthropic,
        systemPrompt: string,
        toolSchemas: Anthropic.Tool[],
        staticDocs: string[] = [],
    ) {
        this.client = client;
        this.systemPrompt = systemPrompt;
        this.toolSchemas = toolSchemas;
        this.staticDocs = staticDocs;
    }

    buildRequest(
        messages: Anthropic.MessageParam[],
        maxTokens = 8096,
    ): Anthropic.MessageCreateParams {
        // System: static prefix with cache breakpoint after tool schemas
        const systemBlocks: Anthropic.TextBlockParam[] = [
            {
                type: "text",
                text: this.systemPrompt,
                // @ts-ignore — cache_control is on beta, type definition varies
                cache_control: { type: "ephemeral" },
            },
        ];

        if (this.staticDocs.length > 0) {
            systemBlocks.push({
                type: "text",
                text: "## Reference Documentation\n\n" + this.staticDocs.join("\n\n---\n\n"),
                // @ts-ignore
                cache_control: { type: "ephemeral" },
            });
        }

        // Build tools array with cache marker on last schema
        const tools: Anthropic.Tool[] = this.toolSchemas.map((tool, idx) => {
            if (idx === this.toolSchemas.length - 1) {
                return {
                    ...tool,
                    // @ts-ignore
                    cache_control: { type: "ephemeral" },
                };
            }
            return tool;
        });

        return {
            model: "claude-3-5-sonnet-20241022",
            max_tokens: maxTokens,
            system: systemBlocks as unknown as string,  // SDK accepts array
            tools,
            messages,   // dynamic — never add cache_control here
        };
    }

    async sendWithCacheTracking(
        messages: Anthropic.MessageParam[],
        maxTokens = 8096,
    ) {
        const request = this.buildRequest(messages, maxTokens);
        const response = await this.client.messages.create(request);

        // Log cache performance
        const usage = response.usage as Anthropic.Usage & {
            cache_creation_input_tokens?: number;
            cache_read_input_tokens?: number;
        };

        const cacheReadTokens   = usage.cache_read_input_tokens ?? 0;
        const cacheWriteTokens  = usage.cache_creation_input_tokens ?? 0;
        const uncachedTokens    = usage.input_tokens;

        const normalCost  = (cacheReadTokens + cacheWriteTokens + uncachedTokens) * 3 / 1_000_000;
        const actualCost  = (cacheReadTokens * 0.3 + cacheWriteTokens * 3.75 + uncachedTokens * 3) / 1_000_000;
        const savings     = ((normalCost - actualCost) / normalCost * 100).toFixed(1);

        console.log(`Cache stats: read=${cacheReadTokens}, write=${cacheWriteTokens}, savings=${savings}%`);

        return response;
    }
}
Production tip: 5-minute TTL management. Anthropic's cache TTL is 5 minutes. In interactive coding sessions, the user may pause for longer. Design your harness to detect potential cache expiration and proactively re-warm the cache with a zero-content call (send system prompt + empty message) if a long idle period is detected, or accept the one-time cache re-write cost on the next real call.

Multi-Turn Cache Cascade

For long agentic sessions, you can create a cascade of cache breakpoints. Each breakpoint freezes the KV state up to that point. The key insight is that once a turn is more than 5 minutes old and unlikely to change, you can promote it into a new cache breakpoint:

8. Knowledge Check

Q1. You have a 200K token window. Your system prompt is 8K, tool schemas are 4K, and you want to reserve 12K for the response. A 60-turn session has accumulated 220K tokens of history and tool results. What is the maximum available budget for history + tool results, and by how much are you over-budget?

Available budget: 200,000 - 8,000 - 4,000 - 12,000 = 176,000 tokens
Over-budget by: 220,000 - 176,000 = 44,000 tokens

You need to truncate/evict 44,000 tokens from the P3-P5 history and old tool results. This is roughly 22 turns of average-sized messages (at ~2K tokens/turn). The strategy would be to summarize the oldest 30-40 turns into a 3-5K token summary, recovering approximately 55-75K tokens.

Q2. Why must a "tool_use" content block and its corresponding "tool_result" always be treated as an atomic pair when evicting context items?

The Anthropic Messages API validates that every tool_use block (in an assistant message) has a matching tool_result block (in the subsequent user message) with the same tool_use_id. If you evict one without the other, the API returns a 422/400 validation error mid-session, crashing the agent. Additionally, showing the model a tool call without its result (or a result without its call) confuses the model's attention about what happened and what step it's on in a multi-step task. Always evict the pair atomically.

Q3. You're building a customer support agent. The system prompt is 6K tokens and changes every quarter. You have 200 concurrent users each making 15 API calls per session. Should you use prompt caching? Calculate the hourly cost savings at $3/M input tokens with 85% cache hit rate.

Yes, absolutely use caching.

Without caching:
200 users × 15 calls × 6,000 tokens = 18,000,000 tokens/hour
Cost = 18M × $3/1M = $54/hour

With caching (85% hit rate, 10% price for hits, 125% for first write):
Write calls: 200 × 1 = 200 calls × 6,000 × $3.75/M ≈ $4.50 (first call per session)
Read calls: 200 × 14 × 0.85 = 2,380 calls × 6,000 × $0.30/M ≈ $4.28
Uncached hits: 200 × 14 × 0.15 = 420 calls × 6,000 × $3/M ≈ $7.56
Total cached cost ≈ $16.34/hour
Savings: ~70% ($37.66/hour)

Q4. Describe the "lost in the middle" failure mode and explain how it affects your context assembly strategy.

"Lost in the Middle" (Liu et al., Stanford 2023) demonstrated that LLMs perform significantly worse at retrieving information placed in the middle of a long context compared to the beginning or end. Attention scores for middle tokens are substantially lower.

Implications for context assembly:
  • Place the most critical instructions at the beginning (system prompt) and the current task/question at the end (latest user message).
  • Avoid burying critical tool results in the middle of long conversation history.
  • If you must include a large file, consider placing it immediately before the user's question rather than at the start of the conversation.
  • Use structured summaries rather than raw history dumps so key information is dense and near the top or bottom.

Q5. What is the maximum number of cache_control breakpoints you should add per API request, and what is the layout order that maximizes cache hits?

Anthropic supports up to 4 cache_control breakpoints per request (as of Claude 3.5 Sonnet). The optimal layout from most-static to most-dynamic:

  1. [cache #1] System prompt (most stable — only changes with major releases)
  2. [cache #2] Tool schemas (stable within a deployment version)
  3. [cache #3] Large static reference docs/codebase snapshot (semi-stable, update infrequently)
  4. [cache #4] Frozen conversation prefix (turns older than 5 min that won't change)
  5. Recent dynamic content (NO cache — changes every turn)
The cache works on prefixes: inserting dynamic content before a cache marker busts the cache for everything after it. Static content must always precede dynamic content.

Mini-Project: Build a Production Token Budget Manager

Objective: Build a complete, testable TokenBudgetManager in Python that handles all edge cases a production agent harness will encounter.

Requirements

  1. Priority Queue Eviction: Implement P1-P5 priority levels. P1 items are never dropped. Lower-priority items are evicted first. Within same priority, evict oldest-turn-index items first.
  2. Atomic Pair Enforcement: Tool-use and tool-result pairs must be evicted atomically. Implement add_tool_call_pair(call, result, turn_index) that adds them as a linked unit.
  3. Sliding Window: Implement apply_sliding_window(max_history_turns) that keeps the first 2 turns (anchor) + last N turns verbatim, with a summary slot for the middle.
  4. Budget Report: Implement get_budget_report() returning a dict with per-category token counts, utilization percentage, and items-evicted count.
  5. Alternation Validator: Implement validate_alternation(messages) that verifies the user/assistant alternation invariant and raises a descriptive error if violated.

def test_tool_pair_atomicity():
    mgr = TokenBudgetManager(total_tokens=5_000, response_reserve=500)
    mgr.add("system", "system_prompt")          # small
    mgr.add_tool_call_pair("bash('ls')", "file1.py\nfile2.py", turn_index=1)
    mgr.add_tool_call_pair("bash('cat file1.py')", "x" * 8_000, turn_index=2)
    result = mgr.build()
    # Both members of a pair must appear together or not at all
    pair_ids = {i.pair_id for i in result if i.pair_id}
    for pair_id in pair_ids:
        members = [i for i in result if i.pair_id == pair_id]
        assert len(members) == 2, f"Pair {pair_id} has {len(members)} members"

Checkpoint Summary

tl;dr for this module — what to take away and how to teach it back:

Core Mental Model

The context window is a cache. Context engineering is cache replacement policy design. The cardinal rule is: always reserve response tokens, never fill the window to 100%.

Key Formulas

available = total - system - schemas - reserve cache_savings ≈ 90% on stable prefix tokens summarize if: p(reuse) × reuse_cost > summarization_cost

5 Things to Remember

  1. P1 (system + schemas + reserve) = NEVER drop
  2. Static before dynamic = cache hits every turn
  3. Truncate tool outputs (head+tail), summarize history
  4. tool_use + tool_result = atomic pair, evict together
  5. Keep first N turns as anchor (user's original intent)

What Separates Good from Great

Good engineers implement truncation. Great engineers implement the full priority queue, validate alternation invariants, instrument per-turn cost/cache metrics, and design their system prompt layout for maximum cache hit rate from day one.
