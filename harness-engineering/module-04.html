Module 4: Sandbox Engineering

Building secure, performant execution environments for AI agents that run arbitrary code — without burning down the house.

What You'll Learn

  • Why sandbox isolation is a non-negotiable security requirement for code-executing agents
  • The trade-offs between Docker, gVisor, Firecracker microVMs, and nsjail
  • How to design layered filesystem and network isolation that prevents escapes
  • Resource limit configurations (CPU, memory, disk I/O, process count, wall-clock time)
  • Warm pool architecture to eliminate cold-start latency from container/VM boot
  • Security hardening: seccomp profiles, capability dropping, AppArmor, audit logging
  • How Devin, E2B, and Modal approach sandbox design in production

1. Why Sandboxes Are Non-Negotiable

The Security Threat Model ▾

When you give an AI agent a bash tool, you are handing it a general-purpose computer. Unlike calling a REST API with a well-defined schema, arbitrary shell execution can reach every resource the process has access to: the filesystem, the network, other processes, the kernel, and — if a container escape is possible — the host machine.

Real Incidents (anonymized from public post-mortems):
  • Recursive deletion: An agent tasked with "cleaning up temp files" ran rm -rf / after a path variable was not set, deleting the entire container filesystem — which was mounted from the host.
  • Credential exfiltration: An agent with network access read ~/.aws/credentials and POSTed the contents to an attacker-controlled webhook embedded in a prompt injection payload.
  • Fork bomb: An agent attempting to parallelize work spawned unlimited child processes, exhausting OS process table entries and taking down the host.
  • Crypto-mining via supply chain: A compromised npm package installed by an agent started a miner. With no CPU limits, it consumed 100% of shared compute for 6 hours before detection.

The fundamental issue is the capability gap: the minimal capability needed for a useful coding agent (write files, run code, install packages) is much smaller than the capability available to an unconfined Linux process. Your sandbox design must close this gap as tightly as possible without breaking legitimate use cases.

Defense in Depth Model

Layer 0: Prompt-level constraints (fragile — easily bypassed)
Layer 1: Process-level isolation (namespaces, seccomp, capabilities)
Layer 2: Filesystem isolation (overlay FS, read-only roots, volume policy)
Layer 3: Network isolation (default-deny, allowlist, egress proxy)
Layer 4: Resource quotas (CPU, memory, disk, PIDs, time)
Layer 5: VM-level isolation (separate kernel — Firecracker/gVisor)
Layer 6: Audit + anomaly detection (detect, alert, kill)

No single layer is sufficient. All must work together.
Analogy — Clean Rooms in Chip Fabrication:
A semiconductor clean room uses layered contamination control: positive air pressure keeps particles out, gowning procedures isolate humans from the environment, separate zones prevent cross-contamination between process steps, and real-time particle counters trigger alarms. A sandbox is the same idea applied to compute: each layer is independently insufficient, but together they reduce the blast radius of any single failure to near zero.

2. Sandbox Technologies Compared

Docker Containers — Process Isolation ▾

Docker uses Linux namespaces (PID, network, mount, UTS, IPC, user) to give a process group the illusion of an isolated system, and cgroups to enforce resource limits. The container shares the host kernel — there is no separate kernel instance. This is Docker's main weakness: a kernel exploit escapes to the host.

  • PID namespace: Container processes see a private PID tree starting at 1
  • Network namespace: Separate network stack, routing table, iptables rules
  • Mount namespace: Isolated filesystem view via overlay FS
  • User namespace: Root inside container maps to unprivileged UID on host
  • cgroups v2: CPU shares, memory limits, I/O throttling, PID count caps

Boot time for a pre-built Docker image is typically 50–200ms. With warm pool pre-loading, you can achieve effectively 0ms perceived startup for the agent.

gVisor — User-Space Kernel ▾

Google's gVisor interposes a user-space kernel (Sentry) between container processes and the host kernel. Syscalls from the container hit Sentry first; Sentry implements roughly 240 Linux syscalls in Go and only makes a small number of filtered syscalls to the real kernel. This dramatically shrinks the kernel attack surface.

  • Runsc runtime: Drop-in OCI replacement for runc
  • KVM mode: Sentry runs in a lightweight VM for stronger isolation
  • Ptrace mode: For environments without KVM; higher overhead
  • Overhead: Syscall-heavy workloads (many small file reads) see 2–5x slowdown vs native. Compute-heavy workloads see ~10–20% overhead.

gVisor is used in production by Google Cloud Run and GKE Sandbox. It is the sweet spot for multi-tenant code execution where you need stronger kernel isolation but can tolerate some I/O overhead.

Firecracker — MicroVMs ▾

AWS open-sourced Firecracker as the foundation for AWS Lambda and Fargate. Each sandbox gets its own lightweight VM with a separate kernel — true hardware-level isolation. The Firecracker VMM (Virtual Machine Monitor) exposes only a minimal device model: virtio-net, virtio-block, serial console. No PCI bus, no USB, no legacy devices.

Key Numbers (Firecracker paper, 2018 NSDI):
Boot time: <125ms to guest userspace
Memory overhead: ~5MB per microVM
Density: 150 microVMs per physical second on c5.metal instances
Security: full kernel isolation — a kernel exploit in the guest cannot reach the host

The trade-off is complexity: you need to manage VM images, kernel versions, and networking (TAP devices, bridges). Tools like Kata Containers and Flintlock layer Kubernetes-compatible interfaces on top of Firecracker. E2B (the open-source sandbox platform used by many coding agents) uses Firecracker under the hood.

nsjail — Lightweight Namespace Jail ▾

nsjail is a process-level sandboxing tool from Google that combines Linux namespaces, seccomp-bpf syscall filtering, resource limits, and chroot in a single lightweight binary. It has no container runtime overhead and is ideal for running short-lived compiler or interpreter invocations where startup time must be under 10ms.

  • Used by Google's Compiler Explorer (godbolt.org) to safely run user-submitted code
  • No daemon process required — just exec nsjail with the command
  • Seccomp profiles expressed in simple config files
  • Weaker isolation than Firecracker (shared kernel), stronger than plain Docker due to strict seccomp

3. Filesystem Isolation

Overlay Filesystems and Volume Strategy ▾

A well-designed sandbox filesystem uses OverlayFS to layer a read-only base image under a per-session writable layer. The agent sees a normal-looking filesystem but all writes go to an ephemeral upper layer that is discarded when the sandbox is destroyed. This enables fast sandbox recycling (just delete the upper layer) and prevents tampering with the base image.

OverlayFS Layer Model:

Upper layer (writable, per-session): /var/lib/sandbox/sessions/<id>/upper/
Lower layer (read-only, shared): /var/lib/sandbox/base/ubuntu-22.04/
Work directory: /var/lib/sandbox/sessions/<id>/work/
Merged view (seen by agent): /var/lib/sandbox/sessions/<id>/merged/

mount -t overlay overlay \
  -o lowerdir=/base,upperdir=/upper,workdir=/work \
  /merged

Volume Mounting Strategy — What to Expose:

  • Expose (read-write): /workspace — the agent's working directory for the task
  • Expose (read-only): Language toolchains, compilers, package manager caches
  • Never expose: Host /etc, /home, /root, /var/run/docker.sock, credential files
  • Tmpfs for temp: Mount /tmp as tmpfs with a size cap so disk-fill attacks are bounded
Filesystem Escape Vectors to Block:
  • Symlink attacks: Agent creates a symlink from inside the container to a host path. Mitigate with --no-new-privileges and user namespaces mapping root to unprivileged host UID.
  • Hardlink attacks: Creating hardlinks to SUID binaries to preserve them after patch. Mitigate with fs.protected_hardlinks=1 sysctl.
  • Docker socket mount: If /var/run/docker.sock is mounted, the agent effectively has root on the host. Never mount it.
  • Proc pseudo-filesystem leakage: Mount /proc with hidepid=2 so processes cannot inspect each other's environments.

How Devin handles it: Cognition's Devin uses a per-session VM image snapshot. The base VM has the development environment pre-installed; each session gets a CoW fork of the snapshot. Changes within the session are isolated and can be optionally committed back to create a persistent artifact (e.g., a Git commit). 

# Python: Docker sandbox with volume isolation
import docker
import tempfile
import os
from pathlib import Path

def create_isolated_workspace() -> dict:
    """Create an isolated workspace with proper volume mounts."""
    session_id = os.urandom(8).hex()
    workspace = Path(f"/var/lib/sandbox/workspaces/{session_id}")
    workspace.mkdir(parents=True, mode=0o700)
    return {"session_id": session_id, "workspace": str(workspace)}

def run_in_sandbox(
    code: str,
    session_id: str,
    workspace: str,
    image: str = "sandbox-python:3.11-slim",
    timeout: int = 30,
) -> dict:
    """Execute code in an isolated Docker container with filesystem restrictions."""
    client = docker.from_env()

    # Write code to workspace (host side)
    code_file = Path(workspace) / "run.py"
    code_file.write_text(code)

    try:
        container = client.containers.run(
            image=image,
            command=["python3", "/workspace/run.py"],
            volumes={
                workspace: {
                    "bind": "/workspace",
                    "mode": "rw",
                },
                # Read-only package cache to avoid re-downloading deps
                "/var/lib/sandbox/pip-cache": {
                    "bind": "/root/.cache/pip",
                    "mode": "ro",
                },
            },
            # Filesystem hardening
            read_only=False,      # workspace needs writes; use tmpfs for /tmp
            tmpfs={"/tmp": "size=64m,mode=1777"},
            # Security options
            security_opt=[
                "no-new-privileges:true",
                "seccomp=/etc/sandbox/seccomp-default.json",
            ],
            # Resource limits (covered in detail in Section 6)
            mem_limit="256m",
            nano_cpus=1_000_000_000,  # 1 CPU
            pids_limit=64,
            network_mode="none",      # full network isolation (see Section 5)
            # Remove all capabilities
            cap_drop=["ALL"],
            user="1000:1000",         # non-root
            remove=True,
            detach=False,
            stdout=True,
            stderr=True,
            timeout=timeout,
        )
        return {"status": "success", "output": container.decode("utf-8")}
    except docker.errors.ContainerError as e:
        return {"status": "error", "output": e.stderr.decode("utf-8")}
    except Exception as e:
        return {"status": "timeout", "output": str(e)}

// TypeScript: Workspace manager with overlay filesystem
import { exec } from "child_process";
import { promisify } from "util";
import * as fs from "fs/promises";
import * as path from "path";
import * as crypto from "crypto";

const execAsync = promisify(exec);

interface WorkspaceConfig {
  sessionId: string;
  upperDir: string;
  workDir: string;
  mergedDir: string;
  baseImage: string;
}

export class OverlayWorkspaceManager {
  private readonly baseDir = "/var/lib/sandbox/sessions";
  private readonly baseImage = "/var/lib/sandbox/base/ubuntu-22.04";

  async createWorkspace(): Promise<WorkspaceConfig> {
    const sessionId = crypto.randomBytes(8).toString("hex");
    const sessionDir = path.join(this.baseDir, sessionId);

    const upperDir = path.join(sessionDir, "upper");
    const workDir = path.join(sessionDir, "work");
    const mergedDir = path.join(sessionDir, "merged");

    await fs.mkdir(upperDir, { recursive: true, mode: 0o700 });
    await fs.mkdir(workDir, { recursive: true, mode: 0o700 });
    await fs.mkdir(mergedDir, { recursive: true, mode: 0o700 });

    // Mount the overlay filesystem
    await execAsync(
      `mount -t overlay overlay ` +
      `-o lowerdir=${this.baseImage},upperdir=${upperDir},workdir=${workDir} ` +
      mergedDir
    );

    return { sessionId, upperDir, workDir, mergedDir, baseImage: this.baseImage };
  }

  async destroyWorkspace(config: WorkspaceConfig): Promise<void> {
    // Unmount before deletion to avoid host filesystem contamination
    await execAsync(`umount ${config.mergedDir}`).catch(() => {});
    await fs.rm(path.dirname(config.upperDir), { recursive: true, force: true });
  }

  async exportWorkspaceArtifacts(
    config: WorkspaceConfig,
    targetDir: string
  ): Promise<void> {
    // Only export the upper layer diff — not the full base image
    await execAsync(`rsync -av ${config.upperDir}/ ${targetDir}/`);
  }
}

4. Network Isolation

Default-Deny Network Architecture ▾

The safest default for agent sandboxes is no network access. For coding tasks that need to install packages or call APIs, you need a carefully allowlisted egress policy. The architecture looks like this:

graph LR A[Agent Sandbox] -->|All traffic| B[Egress Proxy] B -->|DNS lookup| C[DNS Filter] C -->|Blocked domains| D[/dev/null] C -->|Allowed: pypi.org
npmjs.com
github.com| E[Internet] B -->|Audit log all requests| F[Log Store] B -->|Alert on suspicious| G[Security Monitor] H[Default-deny iptables] -.->|Blocks direct egress| A

Network isolation layers:

  1. Docker network mode none: Removes the container from all networks. Use this for purely computational tasks with no package installation.
  2. Custom bridge network with iptables: For tasks needing package managers, create a custom bridge, apply default-deny FORWARD rules, then add per-allowlist ACCEPT rules.
  3. Egress proxy with CONNECT logging: Route HTTPS through an intercepting proxy (mitmproxy, Squid) that logs every domain contacted for post-hoc audit.
  4. DNS sinkholing: Replace the container's DNS resolver with one that blocks known malware C2, data-exfiltration, and cryptocurrency domains.
# Python: nftables rules for agent sandbox network policy
SANDBOX_NFTABLES_RULES = """
#!/usr/sbin/nft -f

# Flush existing rules
flush ruleset

table inet sandbox_filter {
    # Allowlist of package registry CIDRs (pre-resolved, refreshed daily)
    set allowed_egress_v4 {
        type ipv4_addr
        flags interval
        # PyPI (pypi.org)
        elements = { 151.101.0.0/16, 151.101.64.0/18 }
        # npm (registry.npmjs.org)
        # GitHub (github.com/githubusercontent.com)
        # Add more as needed — resolved via allowlist resolver
    }

    chain forward {
        type filter hook forward priority 0; policy drop;

        # Allow established connections back in
        ct state established,related accept

        # Allow outbound to allowlist
        ip daddr @allowed_egress_v4 tcp dport { 80, 443 } accept

        # Log and drop everything else
        log prefix "SANDBOX_BLOCK: " flags all drop
    }

    chain output {
        type filter hook output priority 0; policy accept;
        # Host-level output: allow normally
    }
}
"""

import subprocess

def apply_sandbox_network_policy(namespace: str) -> None:
    """Apply nftables policy inside a network namespace."""
    rules_file = f"/tmp/sandbox-nft-{namespace}.rules"
    with open(rules_file, "w") as f:
        f.write(SANDBOX_NFTABLES_RULES)

    subprocess.run(
        ["ip", "netns", "exec", namespace, "nft", "-f", rules_file],
        check=True,
        capture_output=True,
    )
    print(f"[sandbox] Network policy applied to namespace: {namespace}")

// TypeScript: Docker network configuration with egress proxy
import Dockerode from "dockerode";

interface NetworkConfig {
  sandboxNetworkName: string;
  egressProxyIp: string;
  allowedDomains: string[];
}

export async function createSandboxNetwork(
  docker: Dockerode,
  config: NetworkConfig
): Promise<{ networkId: string; subnetCidr: string }> {
  const subnetCidr = "172.30.0.0/28"; // Small subnet — max 14 containers

  const network = await docker.createNetwork({
    Name: config.sandboxNetworkName,
    Driver: "bridge",
    Internal: false, // false = can reach host; iptables enforces the policy
    Options: {
      "com.docker.network.bridge.name": `sbx-${Date.now()}`,
    },
    IPAM: {
      Config: [{ Subnet: subnetCidr, Gateway: "172.30.0.1" }],
    },
    Labels: {
      "sandbox.managed": "true",
      "sandbox.egress-proxy": config.egressProxyIp,
    },
  });

  return { networkId: network.id, subnetCidr };
}

export function buildContainerNetworkOptions(
  networkId: string,
  egressProxyIp: string
): Record<string, string> {
  return {
    // Route HTTP_PROXY and HTTPS_PROXY through the logging egress proxy
    HTTP_PROXY: `http://${egressProxyIp}:3128`,
    HTTPS_PROXY: `http://${egressProxyIp}:3128`,
    // Prevent bypassing proxy via direct IP connections
    NO_PROXY: "localhost,127.0.0.1",
  };
}

5. Resource Limits

Complete Resource Quota Configuration ▾

Resource limits serve two purposes: security (preventing denial-of-service attacks via resource exhaustion) and cost control (ensuring a misbehaving or slow agent doesn't burn unlimited compute budget). Each limit targets a different attack surface.

Resource Limit Type Why It Matters Typical Value
CPU cgroups cpu.max Prevents crypto-mining, infinite loops consuming host CPU 1–2 vCPUs
Memory cgroups memory.max Prevents OOM cascades on shared hosts; must handle OOM gracefully 256MB–2GB depending on language
Disk writes cgroups blkio / rlimit FSIZE Prevents disk-fill DoS against shared storage 500MB–5GB upper layer cap
Process count cgroups pids.max + RLIMIT_NPROC Prevents fork bombs that exhaust the OS PID table 64–256 PIDs
Wall-clock time External watchdog + SIGKILL Prevents sessions that never terminate; enables timeout billing 30s (REPL) to 30min (long task)
CPU time (cumulative) RLIMIT_CPU Stops a process that runs slowly but continuously consumes CPU 60–300 CPU-seconds
Open files RLIMIT_NOFILE Prevents file descriptor exhaustion (e.g., opening thousands of sockets) 256–1024 
# Python: Resource limit configuration dataclass + Docker apply
from dataclasses import dataclass, field
from typing import Optional
import docker
import signal
import threading

@dataclass
class SandboxResourceLimits:
    # Memory
    memory_mb: int = 512
    memory_swap_mb: int = 0          # 0 = disable swap entirely
    # CPU
    cpu_count: float = 1.0           # fractional CPUs allowed
    cpu_period_us: int = 100_000     # cgroup period = 100ms
    # Derived: quota = period * cpu_count
    @property
    def cpu_quota_us(self) -> int:
        return int(self.cpu_period_us * self.cpu_count)
    # Processes
    pids_limit: int = 64
    # Disk (overlay upper layer quota via project quota or loop device)
    disk_write_mb: int = 1024
    # Time
    wall_clock_seconds: int = 120
    cpu_seconds: int = 60
    # Files
    nofile_soft: int = 512
    nofile_hard: int = 1024

    def to_docker_kwargs(self) -> dict:
        return {
            "mem_limit": f"{self.memory_mb}m",
            "memswap_limit": f"{self.memory_swap_mb}m" if self.memory_swap_mb else f"{self.memory_mb}m",
            "nano_cpus": int(self.cpu_count * 1_000_000_000),
            "cpu_period": self.cpu_period_us,
            "cpu_quota": self.cpu_quota_us,
            "pids_limit": self.pids_limit,
            "ulimits": [
                docker.types.Ulimit(name="nofile", soft=self.nofile_soft, hard=self.nofile_hard),
                docker.types.Ulimit(name="cpu", soft=self.cpu_seconds, hard=self.cpu_seconds + 5),
                docker.types.Ulimit(name="nproc", soft=self.pids_limit, hard=self.pids_limit),
            ],
        }

class WallClockWatchdog:
    """Kills a container after wall_clock_seconds regardless of CPU usage."""

    def __init__(self, container_id: str, timeout_seconds: int, client: docker.DockerClient):
        self.container_id = container_id
        self.timeout = timeout_seconds
        self.client = client
        self._timer: Optional[threading.Timer] = None

    def start(self) -> None:
        self._timer = threading.Timer(self.timeout, self._kill)
        self._timer.daemon = True
        self._timer.start()

    def cancel(self) -> None:
        if self._timer:
            self._timer.cancel()

    def _kill(self) -> None:
        try:
            container = self.client.containers.get(self.container_id)
            container.kill(signal.SIGKILL)
            print(f"[watchdog] Killed container {self.container_id[:12]} (wall-clock timeout)")
        except docker.errors.NotFound:
            pass  # Already exited

export class OomMonitor extends EventEmitter {
  constructor(private docker: Dockerode, private containerId: string) {
    super();
  }

  async watch(): Promise<void> {
    const stream = await this.docker.getEvents({
      filters: { container: [this.containerId], event: ["oom"] },
    });
    stream.on("data", (chunk: Buffer) => {
      const event = JSON.parse(chunk.toString());
      if (event.Action === "oom") {
        this.emit("oom", {
          containerId: this.containerId,
          timestamp: new Date(event.time * 1000),
        });
      }
    });
  }
}

6. Warm Pool Management

Eliminating Cold-Start Latency ▾

Even with 50ms Docker container startup, a user executing code in a conversational agent perceives a noticeable pause on the first tool call. For Firecracker microVMs the boot time is 125ms. Across thousands of concurrent users, this also serializes demand onto container-start infrastructure. The solution is a warm pool of pre-started, idle sandboxes ready to accept work immediately.

Lambda's approach: AWS Lambda maintains warm execution environments (called "execution contexts") across invocations of the same function. The first invocation (cold start) pays the full init cost; subsequent invocations within the keep-alive window (~5–15 minutes) reuse the warm environment. Lambda achieves <1ms assignment latency for warm invocations vs 100–500ms for cold starts.
stateDiagram-v2 [*] --> Provisioning : Pool needs refill Provisioning --> Warm : Container started,\nfilesystem mounted,\nruntime initialized Warm --> Assigned : Agent requests sandbox Assigned --> Running : Code execution begins Running --> Draining : Execution complete Draining --> Warm : Recycle (clean upper layer) Draining --> Destroyed : Recycle limit reached\nor security policy Warm --> Destroyed : Pool over-capacity\nor TTL expired Destroyed --> [*]

Pool sizing algorithm: Use an exponential moving average (EMA) of demand over a rolling window plus a safety buffer. Proactively scale up before traffic peaks using time-of-day patterns (if known).

Pool Target Formula:

ema_demand(t) = α × requests_per_second(t) + (1-α) × ema_demand(t-1)
avg_session_duration = 45 seconds (empirical)
concurrency_estimate = ema_demand × avg_session_duration
pool_target = concurrency_estimate × (1 + buffer_factor)

where buffer_factor = 0.3 (30% overhead for burst absorption)
and α = 0.1 (slow EMA for stability, 0.3 for responsive scaling) 
# Python: WarmPoolManager with demand-driven auto-scaling
import asyncio
import time
import uuid
from collections import deque
from dataclasses import dataclass, field
from typing import Optional
import docker

@dataclass
class SandboxHandle:
    sandbox_id: str
    container_id: str
    created_at: float = field(default_factory=time.monotonic)
    recycle_count: int = 0
    max_recycles: int = 5      # destroy after 5 reuses (security hygiene)

    @property
    def age_seconds(self) -> float:
        return time.monotonic() - self.created_at

class WarmPoolManager:
    """Maintains a pool of pre-warmed Docker sandboxes for low-latency assignment."""

    MAX_POOL_SIZE = 50
    MIN_POOL_SIZE = 5
    SANDBOX_TTL_SECONDS = 600          # Force-recycle idle sandboxes after 10 min
    REFILL_CHECK_INTERVAL = 2.0        # seconds between pool size checks
    EMA_ALPHA = 0.15

    def __init__(self, image: str, resource_limits: dict):
        self.image = image
        self.resource_limits = resource_limits
        self.client = docker.from_env()
        self._pool: asyncio.Queue[SandboxHandle] = asyncio.Queue()
        self._ema_demand: float = self.MIN_POOL_SIZE
        self._request_times: deque[float] = deque(maxlen=1000)
        self._running = False

    async def start(self) -> None:
        """Pre-fill pool and start background maintenance loop."""
        self._running = True
        # Pre-fill to minimum
        await asyncio.gather(*[self._provision_sandbox() for _ in range(self.MIN_POOL_SIZE)])
        asyncio.create_task(self._maintenance_loop())

    async def acquire(self) -> SandboxHandle:
        """Acquire a warm sandbox. Blocks if pool is empty."""
        self._request_times.append(time.monotonic())
        self._update_ema()
        try:
            # Try non-blocking first
            return self._pool.get_nowait()
        except asyncio.QueueEmpty:
            # Pool exhausted — provision on demand (cold path)
            return await self._provision_sandbox()

    async def release(self, handle: SandboxHandle, recycle: bool = True) -> None:
        """Return sandbox to pool or destroy it."""
        handle.recycle_count += 1
        if (
            recycle
            and handle.recycle_count < handle.max_recycles
            and handle.age_seconds < self.SANDBOX_TTL_SECONDS
        ):
            await self._clean_sandbox(handle)
            await self._pool.put(handle)
        else:
            await self._destroy_sandbox(handle)
            # Replace with fresh sandbox to maintain pool size
            asyncio.create_task(self._provision_sandbox())

    async def _provision_sandbox(self) -> SandboxHandle:
        container = self.client.containers.run(
            self.image,
            command=["sleep", "infinity"],   # idle, waiting for work
            detach=True,
            **self.resource_limits,
        )
        return SandboxHandle(
            sandbox_id=str(uuid.uuid4()),
            container_id=container.id,
        )

    async def _clean_sandbox(self, handle: SandboxHandle) -> None:
        """Remove workspace artifacts so sandbox can be reused cleanly."""
        container = self.client.containers.get(handle.container_id)
        container.exec_run("rm -rf /workspace/*", user="root")

    async def _destroy_sandbox(self, handle: SandboxHandle) -> None:
        try:
            container = self.client.containers.get(handle.container_id)
            container.remove(force=True)
        except docker.errors.NotFound:
            pass

    def _update_ema(self) -> None:
        now = time.monotonic()
        recent = sum(1 for t in self._request_times if now - t <= 10.0)
        rps = recent / 10.0
        self._ema_demand = self.EMA_ALPHA * rps * 45 + (1 - self.EMA_ALPHA) * self._ema_demand

    async def _maintenance_loop(self) -> None:
        while self._running:
            await asyncio.sleep(self.REFILL_CHECK_INTERVAL)
            target = min(
                max(int(self._ema_demand * 1.3), self.MIN_POOL_SIZE),
                self.MAX_POOL_SIZE,
            )
            current_size = self._pool.qsize()
            deficit = target - current_size
            if deficit > 0:
                await asyncio.gather(*[self._provision_sandbox() for _ in range(deficit)])

7. Security Hardening

Seccomp, Capabilities, AppArmor, and Audit Logging ▾

Namespace isolation and resource limits reduce the blast radius of an attack but do not prevent a compromised container from exploiting kernel vulnerabilities via syscalls. Seccomp-BPF adds a syscall-level allowlist: any syscall not in the profile immediately kills the process with SIGSYS. Combined with Linux capabilities dropping and mandatory access control (AppArmor or SELinux), you create a container that can do almost nothing except run the intended workload.

Docker default vs hardened seccomp:
Docker's default seccomp profile blocks ~44 of ~350+ syscalls. A hardened profile for a Python sandbox can block ~280+ syscalls, allowing only read, write, open, close, stat, fstat, mmap, mprotect, brk, exit_group, futex, epoll_wait, clock_gettime, getpid and ~30 others. This is a significant reduction in kernel attack surface.

Audit Logging

Every sandbox operation should emit structured audit events to an append-only log store. This enables post-incident forensics (what did the agent do?), billing (how much compute was used?), and anomaly detection (is an agent behaving unusually?). 

# Python: Structured sandbox audit logger
import json
import time
from dataclasses import dataclass, asdict
from enum import Enum
from typing import Any, Optional

class SandboxEvent(str, Enum):
    CREATED      = "sandbox.created"
    ASSIGNED     = "sandbox.assigned"
    EXEC_START   = "sandbox.exec.start"
    EXEC_END     = "sandbox.exec.end"
    OOM_KILL     = "sandbox.oom_kill"
    TIMEOUT_KILL = "sandbox.timeout_kill"
    SECCOMP_DENY = "sandbox.seccomp_deny"
    NET_BLOCK    = "sandbox.network.blocked"
    DESTROYED    = "sandbox.destroyed"

@dataclass
class AuditRecord:
    event: str
    sandbox_id: str
    session_id: str
    agent_id: str
    timestamp: float
    duration_ms: Optional[float] = None
    exit_code: Optional[int] = None
    syscall_name: Optional[str] = None     # for seccomp denials
    blocked_domain: Optional[str] = None  # for network blocks
    metadata: Optional[dict] = None

class SandboxAuditLogger:
    def __init__(self, log_path: str = "/var/log/sandbox/audit.jsonl"):
        self.log_path = log_path

    def log(self, record: AuditRecord) -> None:
        entry = json.dumps(asdict(record), default=str)
        with open(self.log_path, "a") as f:
            f.write(entry + "\n")

    def log_exec(
        self,
        sandbox_id: str,
        session_id: str,
        agent_id: str,
        command: str,
        exit_code: int,
        duration_ms: float,
    ) -> None:
        self.log(AuditRecord(
            event=SandboxEvent.EXEC_END,
            sandbox_id=sandbox_id,
            session_id=session_id,
            agent_id=agent_id,
            timestamp=time.time(),
            exit_code=exit_code,
            duration_ms=duration_ms,
            metadata={"command_hash": hash(command) & 0xFFFFFFFF},
        ))

What Interviewers Are Looking For ▾

"Walk me through how you'd design a secure execution environment for an AI coding agent that needs to run arbitrary user-provided Python code."

What strong candidates do:

  • Start with the threat model before jumping to implementation — "the agent can run arbitrary code, so we have these attack surfaces: filesystem, network, process, kernel"
  • Explicitly discuss defense in depth — no single mechanism is sufficient
  • Make concrete technology choices with trade-off justification (Docker vs Firecracker vs gVisor)
  • Address operational concerns: warm pool for latency, recycle vs destroy, pool sizing
  • Mention audit logging and what you'd detect in post-incident forensics
  • Know the difference between namespace isolation and kernel isolation

Common follow-up questions:

  1. "How does Firecracker achieve <125ms boot time while providing VM-level isolation?"
    Answer: minimal device model (no PCI, USB, etc.), pre-built kernel + initrd image, virtio fast path for I/O, KVM hardware acceleration
  2. "A container escape vulnerability is disclosed. How do you respond?"
    Answer: kill all running sandboxes immediately, rotate to gVisor or Firecracker, patch and redeploy base images, audit logs for exploitation evidence
  3. "How do you handle agents that need to install npm packages on each run without re-downloading everything?"
    Answer: read-only bind mount of a shared npm cache, plus overlay FS so each sandbox gets its own writable layer over the cache
  4. "What happens when a sandbox runs out of memory mid-execution?"
    Answer: OOM killer terminates the container process, we catch the Docker OOM event, return an error to the agent, log the incident, recycle the sandbox
Red Flags to Avoid:
  • Saying "we just run it in Docker and it's fine" without discussing network isolation, filesystem policy, capability dropping, or seccomp
  • Conflating namespace isolation with hypervisor-level isolation
  • Not discussing the warm pool problem — "we start a new container per request" ignores latency
  • Treating resource limits as optional nice-to-haves rather than security requirements
  • Not mentioning the Docker socket exposure risk (/var/run/docker.sock)

9. Knowledge Check

Q1. Why is sharing a host kernel between containers (as in standard Docker) a security risk compared to Firecracker microVMs?

A kernel vulnerability (e.g., a privilege escalation exploit) can be triggered by code running inside a container, and since the container shares the host kernel, a successful exploit grants attacker access to the host machine and all other containers on it. Firecracker microVMs each run a separate guest kernel. A kernel exploit inside a microVM only compromises that VM — the host kernel is protected by the KVM hypervisor boundary. The attack surface to escape to the host is the much smaller VMM (Virtual Machine Monitor), not the full kernel.

Q2. A coding agent sandbox uses network_mode=none but the agent needs to install Python packages. How would you enable this securely?

Option A (preferred for high-security): Pre-install all required packages into the base image at build time. The agent can use packages but cannot install new ones at runtime. Fast and fully air-gapped.

Option B (for dynamic installs): Remove network_mode=none and instead create a custom Docker network with default-deny iptables/nftables rules. Add ACCEPT rules only for the IP ranges of pypi.org and files.pythonhosted.org. Route traffic through an egress proxy that logs all requests. Apply DNS filtering to block non-allowlisted domains. This allows pip install while preventing arbitrary internet access.

Q3. What is a fork bomb and how does the pids.max cgroup limit prevent it?

A fork bomb is a program that recursively calls fork() to create exponentially more child processes: :(){ :|:& };: in bash. Each process consumes a slot in the OS PID table (typically 32,768–4,194,304 entries on Linux). When the table fills, no new processes can be created on the entire host, causing a denial-of-service for all users and system processes including init.

The cgroups pids.max limit places a hard cap on the total number of processes (threads) within a cgroup hierarchy. When a fork() call would exceed pids.max, the kernel returns EAGAIN (resource temporarily unavailable). The fork bomb stays contained within its cgroup and cannot exhaust the global PID table. Setting pids.max=64 is typically sufficient for most coding tasks and provides strong fork-bomb protection.

Q4. Explain the OverlayFS layer model: what are the lower, upper, and merged layers, and why is this useful for sandbox recycling?

OverlayFS presents a unified "merged" directory that stacks two layers:

Lower layer (read-only): The base filesystem, typically a shared Docker image or OS image. All sandbox instances share a single copy — no duplication.
Upper layer (read-write, per-session): A private directory where all writes go. Reads check the upper layer first; if the file doesn't exist there, they fall through to the lower layer (copy-on-first-write semantics).
Merged layer: The view the container process sees — it looks like a single normal filesystem.

For sandbox recycling, cleanup is O(1): simply delete the upper layer directory and create a new empty one. The base image never needs to be re-mounted or re-copied. This makes recycling a sandbox nearly as fast as creating one from scratch but avoids the overhead of pulling/extracting a full image.

Q5. What is the warm pool sizing formula and what trade-off does the buffer_factor represent?

The formula is:
pool_target = (ema_rps × avg_session_seconds) × (1 + buffer_factor)

ema_rps × avg_session_seconds estimates the current concurrency (Little's Law): if requests arrive at 2 RPS and each session lasts 45 seconds, expect ~90 concurrent sessions.

buffer_factor (typically 0.2–0.5) represents the trade-off between:
- Cost: Higher buffer = more idle pre-warmed containers consuming CPU/memory with no revenue
- Latency: Lower buffer = more pool exhaustion events, falling back to cold starts during traffic spikes

At 0.3 buffer, you accept 30% waste in exchange for absorbing burst traffic of up to 30% above the moving average without any cold starts. For latency-sensitive production agents, a buffer_factor of 0.3–0.5 is typical.
