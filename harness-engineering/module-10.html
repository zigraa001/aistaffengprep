Module 10: State Management & Persistence

Master how agent harnesses store, checkpoint, and recover state across tool calls, crashes, and sessions — the infrastructure that makes long-running agents reliable.

What You'll Learn

  • Why state management is the hardest unsolved problem in production agent systems
  • Model agent conversations as formal state machines with typed transitions
  • Design and implement checkpoint/resume systems for crash recovery
  • Distinguish short-term context state from long-term persistent memory
  • Choose between file-based and database-backed state storage with clear trade-off reasoning
  • Implement crash recovery with write-ahead logging and idempotent tool execution
  • Build an idempotency middleware layer that protects every tool call

1. The State Problem in Agent Systems

Why State Is Fundamentally Hard for Agents

A stateless web request completes in milliseconds and forgets everything. An agent task runs for minutes, hours, or even days — invoking dozens of tools, modifying files, calling external APIs, and awaiting human approval. Every one of those steps accumulates state that must be tracked, persisted, and recoverable.

Analogy: Think of an agent like a surgeon mid-operation. A surgeon cannot "restart from scratch" if they drop a tool. They need to know exactly what has been cut, what is sutured, and what still needs doing. Agent state is the surgical notes that make continuity possible.

Where Agent State Lives

State is not in one place — it is distributed across four layers:

Layer What Lives There Lifetime Lost On Crash?
Context Window Current conversation messages, tool results, reasoning Single API call Yes — always
Process Memory In-memory variables, caches, pending tool queues Process lifetime Yes
Filesystem Modified files, generated code, downloaded artifacts Persistent (if durable volume) Only if ephemeral
External Systems DB rows written, APIs called, PRs opened Permanent side effects No — but untracked
The Fundamental Tension: The model only sees the context window. The real world only sees external systems. Your harness must bridge these two by serializing and replaying state across the gap — and doing so safely when crashes occur between any two tool calls.

How Claude Code Handles State

Claude Code uses a pragmatic approach: the filesystem is the state. Conversation turns are written as JSONL to ~/.claude/projects/<project-hash>/conversations/<session-id>.jsonl. Each line is an atomic message record appended in real time. This means:

  • A crash mid-conversation leaves a partial JSONL file — recovery simply reads up to the last complete line
  • The git repository records file-level state — git diff shows exactly what the agent changed
  • CLAUDE.md files act as persistent cross-session memory injected at the start of every turn
  • No database required — the project directory itself is the source of truth
Key Insight: Claude Code's file-first state model means any developer can inspect, diff, and understand agent state using standard Unix tools. Opacity is the enemy of trust in agent systems.

2. Conversation State Machines

Modeling Agent Conversations as Formal State Machines

Every agent conversation follows a deterministic cycle of states. Formalizing this as a state machine gives you three powerful properties: (1) you can serialize and restore the machine at any state, (2) you can validate that every transition is legal, and (3) you can emit structured events for observability.

The Six Core Agent States

IDLE THINKING EXECUTING_TOOL AWAITING_APPROVAL RESPONDING ERROR

  • IDLE — Waiting for a user message. No compute running.
  • THINKING — Model API call in flight. Streaming tokens from LLM.
  • EXECUTING_TOOL — A tool_use block was returned; tool is running in sandbox.
  • AWAITING_APPROVAL — Tool requires human confirmation before execution continues.
  • RESPONDING — Final assistant text message being streamed to user.
  • ERROR — Unrecoverable failure. Agent paused pending operator intervention.

Implementing a ConversationStateMachine

from enum import Enum, auto
from dataclasses import dataclass, field
from typing import Optional, Callable, Dict, List
import time
import json


class AgentState(str, Enum):
    IDLE              = "IDLE"
    THINKING          = "THINKING"
    EXECUTING_TOOL    = "EXECUTING_TOOL"
    AWAITING_APPROVAL = "AWAITING_APPROVAL"
    RESPONDING        = "RESPONDING"
    ERROR             = "ERROR"


@dataclass
class StateTransition:
    from_state: AgentState
    to_state: AgentState
    trigger: str
    timestamp: float = field(default_factory=time.time)
    metadata: dict = field(default_factory=dict)


# Legal transitions: (from_state, trigger) -> to_state
TRANSITION_TABLE: Dict[tuple, AgentState] = {
    (AgentState.IDLE,              "user_message_received"):   AgentState.THINKING,
    (AgentState.THINKING,          "tool_use_block_emitted"):  AgentState.EXECUTING_TOOL,
    (AgentState.THINKING,          "text_only_response"):      AgentState.RESPONDING,
    (AgentState.THINKING,          "api_error"):               AgentState.ERROR,
    (AgentState.THINKING,          "timeout"):                 AgentState.ERROR,
    (AgentState.EXECUTING_TOOL,    "tool_requires_approval"):  AgentState.AWAITING_APPROVAL,
    (AgentState.EXECUTING_TOOL,    "tool_result_ready"):       AgentState.THINKING,
    (AgentState.EXECUTING_TOOL,    "tool_exception"):          AgentState.ERROR,
    (AgentState.EXECUTING_TOOL,    "sandbox_crash"):           AgentState.ERROR,
    (AgentState.AWAITING_APPROVAL, "user_approved"):           AgentState.EXECUTING_TOOL,
    (AgentState.AWAITING_APPROVAL, "user_denied"):             AgentState.IDLE,
    (AgentState.RESPONDING,        "response_complete"):       AgentState.IDLE,
    (AgentState.ERROR,             "operator_reset"):          AgentState.IDLE,
    (AgentState.ERROR,             "operator_retry"):          AgentState.THINKING,
}


class ConversationStateMachine:
    """
    Manages agent state transitions with full audit trail.
    Serializable to/from dict for checkpoint/resume.
    """

    def __init__(self, session_id: str):
        self.session_id = session_id
        self.current_state = AgentState.IDLE
        self.history: List[StateTransition] = []
        self._listeners: Dict[AgentState, List[Callable]] = {}

    def transition(self, trigger: str, metadata: dict = None) -> AgentState:
        """
        Apply a trigger and move to the next state.
        Raises ValueError on illegal transition.
        """
        key = (self.current_state, trigger)
        next_state = TRANSITION_TABLE.get(key)
        if next_state is None:
            raise ValueError(
                f"Illegal transition: {self.current_state} --[{trigger}]--> ?"
                f"\nLegal triggers from {self.current_state}: "
                f"{[t for (s, t) in TRANSITION_TABLE if s == self.current_state]}"
            )

        record = StateTransition(
            from_state=self.current_state,
            to_state=next_state,
            trigger=trigger,
            metadata=metadata or {},
        )
        self.history.append(record)
        self.current_state = next_state

        # Notify listeners
        for cb in self._listeners.get(next_state, []):
            cb(record)

        return next_state

    def on_enter(self, state: AgentState, callback: Callable):
        """Register a callback invoked whenever state is entered."""
        self._listeners.setdefault(state, []).append(callback)

    def to_dict(self) -> dict:
        """Serialize for checkpoint storage."""
        return {
            "session_id": self.session_id,
            "current_state": self.current_state.value,
            "history": [
                {
                    "from": t.from_state.value,
                    "to": t.to_state.value,
                    "trigger": t.trigger,
                    "ts": t.timestamp,
                    "meta": t.metadata,
                }
                for t in self.history
            ],
        }

    @classmethod
    def from_dict(cls, data: dict) -> "ConversationStateMachine":
        """Reconstruct from checkpoint data."""
        sm = cls(data["session_id"])
        sm.current_state = AgentState(data["current_state"])
        for t in data["history"]:
            sm.history.append(StateTransition(
                from_state=AgentState(t["from"]),
                to_state=AgentState(t["to"]),
                trigger=t["trigger"],
                timestamp=t["ts"],
                metadata=t.get("meta", {}),
            ))
        return sm


# Usage example
sm = ConversationStateMachine(session_id="sess_abc123")
sm.on_enter(AgentState.ERROR, lambda t: print(f"ALERT: Agent entered ERROR via {t.trigger}"))

sm.transition("user_message_received")          # IDLE -> THINKING
sm.transition("tool_use_block_emitted",         # THINKING -> EXECUTING_TOOL
              metadata={"tool": "bash", "id": "tool_xyz"})
sm.transition("tool_result_ready")              # EXECUTING_TOOL -> THINKING
sm.transition("text_only_response")             # THINKING -> RESPONDING
sm.transition("response_complete")              # RESPONDING -> IDLE
print(sm.current_state)                         # AgentState.IDLE
Why a formal state machine? Ad-hoc boolean flags like is_running and waiting_for_tool become inconsistent under concurrent access and crashes. A state machine has a single authoritative field (current_state) that is trivially serializable and auditable.

3. Checkpoint and Resume

Designing a Robust Checkpoint System

A checkpoint is a point-in-time snapshot of all recoverable state. Resume means reading that snapshot and reconstructing the agent's context so it can continue as if the crash never happened — without re-executing side effects that already succeeded.

What to Checkpoint

checkpoint = {
  session_id,
  state_machine_snapshot, # current state + history
  messages[], # full conversation so far
  tool_call_log[], # each call: id, name, args, result, status
  file_snapshot_refs[], # git commit hash or content hashes
  cursor_position, # which tool call we're at in the plan
  pending_approval_queue[], # tools awaiting human sign-off
  metadata { created_at, version, harness_version }
}

Checkpoint Storage Options

  • Local JSONL file (Claude Code model) — each message appended atomically. Crash-safe because partial writes are detected by JSON parse failure on the last line.
  • SQLite — single-file embedded DB, great for structured queries over tool call history. Useful for analytics.
  • Object storage (S3, GCS) — required for distributed/cloud agents where the process may resume on a different machine.

CheckpointManager Implementation

import json
import os
import hashlib
from pathlib import Path
from dataclasses import dataclass, asdict, field
from typing import List, Optional, Any
import time


@dataclass
class ToolCallRecord:
    call_id: str           # idempotency key
    tool_name: str
    arguments: dict
    result: Optional[Any]  # None if not yet completed
    status: str            # "pending" | "success" | "error"
    started_at: float
    completed_at: Optional[float] = None
    error_msg: Optional[str] = None


@dataclass
class Checkpoint:
    session_id: str
    version: int
    created_at: float
    state_machine: dict
    messages: list
    tool_call_log: List[ToolCallRecord]
    file_hashes: dict          # path -> sha256 of file content at checkpoint time
    cursor: int                # index into tool_call_log of next pending call
    pending_approvals: list
    harness_version: str = "1.0"

    def to_json(self) -> str:
        def _serial(obj):
            if isinstance(obj, ToolCallRecord):
                return asdict(obj)
            raise TypeError(f"Unserializable: {type(obj)}")
        return json.dumps(asdict(self), default=_serial, indent=2)

    @classmethod
    def from_json(cls, raw: str) -> "Checkpoint":
        data = json.loads(raw)
        data["tool_call_log"] = [
            ToolCallRecord(**t) for t in data["tool_call_log"]
        ]
        return cls(**data)


class CheckpointManager:
    """
    Atomically writes checkpoints to disk.
    Uses write-then-rename for crash-safe updates (POSIX rename is atomic).
    """

    def __init__(self, checkpoint_dir: str):
        self.dir = Path(checkpoint_dir)
        self.dir.mkdir(parents=True, exist_ok=True)

    def _path(self, session_id: str) -> Path:
        return self.dir / f"{session_id}.checkpoint.json"

    def _tmp_path(self, session_id: str) -> Path:
        return self.dir / f"{session_id}.checkpoint.json.tmp"

    def save(self, checkpoint: Checkpoint) -> None:
        """
        Atomic write: write to .tmp, then rename over the real file.
        This guarantees the checkpoint file is always either the old
        complete version or the new complete version — never corrupted.
        """
        tmp = self._tmp_path(checkpoint.session_id)
        final = self._path(checkpoint.session_id)

        tmp.write_text(checkpoint.to_json(), encoding="utf-8")
        tmp.replace(final)  # atomic on POSIX, near-atomic on Windows

    def load(self, session_id: str) -> Optional[Checkpoint]:
        """Load the most recent valid checkpoint for a session."""
        path = self._path(session_id)
        if not path.exists():
            return None
        try:
            return Checkpoint.from_json(path.read_text(encoding="utf-8"))
        except (json.JSONDecodeError, KeyError, TypeError) as e:
            # Checkpoint corrupted — try .tmp backup
            tmp = self._tmp_path(session_id)
            if tmp.exists():
                return Checkpoint.from_json(tmp.read_text(encoding="utf-8"))
            raise RuntimeError(f"Checkpoint unreadable for {session_id}: {e}") from e

    def delete(self, session_id: str) -> None:
        """Remove checkpoint after successful completion."""
        for p in [self._path(session_id), self._tmp_path(session_id)]:
            p.unlink(missing_ok=True)

    def list_sessions(self) -> List[str]:
        return [p.stem.replace(".checkpoint", "")
                for p in self.dir.glob("*.checkpoint.json")]

    @staticmethod
    def hash_file(path: str) -> str:
        """SHA-256 of file content for change detection."""
        h = hashlib.sha256()
        with open(path, "rb") as f:
            for chunk in iter(lambda: f.read(65536), b""):
                h.update(chunk)
        return h.hexdigest()


# Resume protocol
def resume_from_checkpoint(
    checkpoint: Checkpoint,
    sm: "ConversationStateMachine",
) -> dict:
    """
    Reconstruct agent context from a checkpoint.
    Returns a dict with everything the agent loop needs.
    """
    # 1. Restore state machine
    sm_restored = ConversationStateMachine.from_dict(checkpoint.state_machine)

    # 2. Find the last SUCCESSFUL tool call (replay point)
    completed = [t for t in checkpoint.tool_call_log if t.status == "success"]
    pending   = [t for t in checkpoint.tool_call_log if t.status == "pending"]

    # 3. Build messages context (full history up to checkpoint)
    messages = checkpoint.messages

    return {
        "state_machine":    sm_restored,
        "messages":         messages,
        "completed_calls":  completed,
        "pending_calls":    pending,
        "replay_from":      checkpoint.cursor,
    }
Atomic rename pattern: Writing to a .tmp file then renaming is the standard Unix crash-safety idiom. It guarantees readers always see either the old checkpoint or the new one, never a partial write. SQLite uses a similar approach with its WAL journal.

4. Short-Term vs Long-Term Memory

Memory Architecture for Agent Systems

Agents need two fundamentally different kinds of memory, just like humans do. Confusing them leads to either bloated context windows (short-term overload) or forgotten knowledge (long-term neglect).

Dimension Short-Term Memory Long-Term Memory
Location Context window (messages array) CLAUDE.md files, vector DB, key-value store
Scope Current session / task Across sessions, all tasks
Retrieval Always present (attention over all tokens) Explicit load: read file, semantic search, key lookup
Cost Token cost per API call (linear) Storage cost + retrieval latency
Update Append new messages Write/upsert to persistent store
Example "You asked me to use Python 3.11 earlier" "This project always uses Black formatter"

Memory Consolidation: When to Promote

Not every short-term fact deserves long-term storage. Promotion criteria:

  • Repeated access — If a fact was referenced 3+ times in the session, it should be in long-term memory.
  • Cross-session relevance — Preferences about coding style, project structure, API keys format.
  • High-value corrections — When the user corrects the agent, that correction is worth persisting.
  • Reference material — File paths, API endpoints, build commands that won't change.

How Claude Code's Memory System Works

Claude Code uses a layered CLAUDE.md hierarchy as its long-term memory:

  1. ~/.claude/CLAUDE.md — Global user preferences, always loaded
  2. /project-root/CLAUDE.md — Project-level conventions, loaded when in project
  3. /project-root/src/CLAUDE.md — Directory-level notes, loaded when working in src/

This means memory retrieval is purely filesystem-based: the harness reads the relevant CLAUDE.md files at session start and injects them into the system prompt. No vector search, no embeddings — just structured text files that developers can read, edit, and version control.

MemoryManager Implementation

import os
import json
import time
from pathlib import Path
from dataclasses import dataclass, field, asdict
from typing import List, Optional, Dict
from enum import Enum


class MemoryTier(str, Enum):
    SHORT_TERM = "short_term"   # current session only
    LONG_TERM  = "long_term"    # persists across sessions


@dataclass
class MemoryEntry:
    key: str
    content: str
    tier: MemoryTier
    created_at: float = field(default_factory=time.time)
    accessed_at: float = field(default_factory=time.time)
    access_count: int = 0
    tags: List[str] = field(default_factory=list)
    source: str = ""           # "user_correction" | "inferred" | "explicit"


class MemoryManager:
    """
    Dual-tier memory manager.
    Short-term: in-memory dict (lost on crash, that's fine).
    Long-term: persisted to .claude/memory.json in project root.
    """

    PROMOTE_AFTER_ACCESSES = 3    # auto-promote if accessed this many times
    MAX_SHORT_TERM_ENTRIES = 100  # evict LRU after this limit

    def __init__(self, project_root: str):
        self.project_root = Path(project_root)
        self.memory_file = self.project_root / ".claude" / "memory.json"
        self.memory_file.parent.mkdir(parents=True, exist_ok=True)

        self._short_term: Dict[str, MemoryEntry] = {}
        self._long_term: Dict[str, MemoryEntry] = {}
        self._load_long_term()

    # ---- Write -------------------------------------------------------

    def write(self, key: str, content: str,
              tier: MemoryTier = MemoryTier.SHORT_TERM,
              tags: List[str] = None,
              source: str = "inferred") -> MemoryEntry:
        entry = MemoryEntry(
            key=key, content=content, tier=tier,
            tags=tags or [], source=source,
        )
        if tier == MemoryTier.SHORT_TERM:
            self._evict_if_needed()
            self._short_term[key] = entry
        else:
            self._long_term[key] = entry
            self._persist_long_term()
        return entry

    # ---- Read --------------------------------------------------------

    def read(self, key: str) -> Optional[str]:
        """Read from short-term first, then long-term."""
        entry = self._short_term.get(key) or self._long_term.get(key)
        if entry:
            entry.accessed_at = time.time()
            entry.access_count += 1
            self._maybe_promote(entry)
            return entry.content
        return None

    def search(self, query_tags: List[str]) -> List[MemoryEntry]:
        """Retrieve all entries matching ANY of the given tags."""
        tag_set = set(query_tags)
        results = []
        for store in (self._short_term, self._long_term):
            for entry in store.values():
                if tag_set & set(entry.tags):
                    results.append(entry)
        return sorted(results, key=lambda e: e.accessed_at, reverse=True)

    # ---- Consolidation -----------------------------------------------

    def consolidate_session(self) -> int:
        """
        At end of session: promote frequently-accessed short-term entries.
        Returns number of entries promoted.
        """
        promoted = 0
        for key, entry in list(self._short_term.items()):
            if entry.access_count >= self.PROMOTE_AFTER_ACCESSES:
                self._long_term[key] = entry
                self._long_term[key].tier = MemoryTier.LONG_TERM
                del self._short_term[key]
                promoted += 1
        if promoted:
            self._persist_long_term()
        return promoted

    # ---- CLAUDE.md Integration ---------------------------------------

    def load_from_claude_md(self, path: str) -> None:
        """
        Parse a CLAUDE.md file and load its sections as long-term memories.
        Each H2 heading becomes a memory key.
        """
        content = Path(path).read_text(encoding="utf-8")
        sections = content.split("\n## ")
        for section in sections[1:]:  # skip text before first H2
            lines = section.strip().split("\n")
            key = lines[0].strip().lower().replace(" ", "_")
            body = "\n".join(lines[1:]).strip()
            self.write(key, body, tier=MemoryTier.LONG_TERM, source="claude_md")

    def render_for_system_prompt(self, max_chars: int = 4000) -> str:
        """Format long-term memory as a system prompt injection."""
        lines = ["## Persistent Memory\n"]
        total = 0
        for entry in sorted(self._long_term.values(),
                             key=lambda e: e.accessed_at, reverse=True):
            snippet = f"- [{entry.key}]: {entry.content}\n"
            if total + len(snippet) > max_chars:
                break
            lines.append(snippet)
            total += len(snippet)
        return "".join(lines)

    # ---- Private helpers ---------------------------------------------

    def _maybe_promote(self, entry: MemoryEntry) -> None:
        if (entry.tier == MemoryTier.SHORT_TERM
                and entry.access_count >= self.PROMOTE_AFTER_ACCESSES):
            entry.tier = MemoryTier.LONG_TERM
            self._long_term[entry.key] = entry
            self._short_term.pop(entry.key, None)
            self._persist_long_term()

    def _evict_if_needed(self) -> None:
        if len(self._short_term) >= self.MAX_SHORT_TERM_ENTRIES:
            lru_key = min(self._short_term,
                          key=lambda k: self._short_term[k].accessed_at)
            del self._short_term[lru_key]

    def _persist_long_term(self) -> None:
        data = {k: asdict(v) for k, v in self._long_term.items()}
        tmp = self.memory_file.with_suffix(".json.tmp")
        tmp.write_text(json.dumps(data, indent=2), encoding="utf-8")
        tmp.replace(self.memory_file)

    def _load_long_term(self) -> None:
        if not self.memory_file.exists():
            return
        try:
            data = json.loads(self.memory_file.read_text(encoding="utf-8"))
            self._long_term = {
                k: MemoryEntry(**v) for k, v in data.items()
            }
        except (json.JSONDecodeError, TypeError):
            pass  # corrupt file — start fresh

5. File vs Database State

Choosing Your State Backend

There is no universally correct answer. The choice depends on whether you need queryability, concurrent access, or simplicity. Let's examine both ends of the spectrum and the hybrid middle.

Dimension File-Based (Claude Code) Database-Backed (OpenAI Assistants) Hybrid
Simplicity Excellent — ls, cat, grep More complex — schema, migrations Moderate
Query capability Limited — grep/jq only Full SQL / NoSQL SQL for metadata
Concurrent access Dangerous — file locks needed Native with transactions DB handles concurrency
Git integration Native — diff, history, blame Not applicable File parts only
Portability Copy directory = copy state Dump/restore required Mixed
Scale Breaks >10K sessions on one filesystem Scales to millions of sessions Files per session, DB for index
Multi-machine resume Requires shared filesystem or S3 Any machine with DB access DB coordinates, files replicated

File-Based State (Claude Code Approach)

import json
import fcntl  # POSIX file locking
from pathlib import Path
from typing import List
from datetime import datetime


class FileStateStore:
    """
    JSONL-based state store.
    Each session is a directory containing:
      - conversation.jsonl   (append-only message log)
      - tool_calls.jsonl     (append-only tool call log)
      - metadata.json        (session-level metadata)
    """

    def __init__(self, base_dir: str):
        self.base = Path(base_dir)

    def _session_dir(self, session_id: str) -> Path:
        d = self.base / session_id
        d.mkdir(parents=True, exist_ok=True)
        return d

    def append_message(self, session_id: str, message: dict) -> None:
        """Append a single message atomically to the conversation log."""
        log_path = self._session_dir(session_id) / "conversation.jsonl"
        line = json.dumps(message) + "\n"
        with open(log_path, "a", encoding="utf-8") as f:
            fcntl.flock(f, fcntl.LOCK_EX)  # exclusive write lock
            f.write(line)
            f.flush()
            os.fsync(f.fileno())            # ensure durability on crash
            fcntl.flock(f, fcntl.LOCK_UN)

    def read_messages(self, session_id: str) -> List[dict]:
        """Read all valid messages; skip any partial/corrupt final line."""
        log_path = self._session_dir(session_id) / "conversation.jsonl"
        if not log_path.exists():
            return []
        messages = []
        with open(log_path, "r", encoding="utf-8") as f:
            for line in f:
                line = line.strip()
                if not line:
                    continue
                try:
                    messages.append(json.loads(line))
                except json.JSONDecodeError:
                    pass  # skip corrupt tail line from crash
        return messages

    def list_sessions(self, limit: int = 100) -> List[dict]:
        """List recent sessions sorted by modification time."""
        sessions = []
        for d in sorted(self.base.iterdir(),
                         key=lambda p: p.stat().st_mtime, reverse=True):
            if d.is_dir():
                meta_path = d / "metadata.json"
                meta = json.loads(meta_path.read_text()) if meta_path.exists() else {}
                sessions.append({"session_id": d.name, **meta})
        return sessions[:limit]

class SQLiteStateStore:
    """
    SQLite-backed state store for structured queries and concurrent access.
    Uses WAL mode for better read concurrency.
    Suitable for a single-machine deployment with multiple agent workers.
    """

    SCHEMA = """
    CREATE TABLE IF NOT EXISTS sessions (
        session_id   TEXT PRIMARY KEY,
        created_at   REAL NOT NULL,
        updated_at   REAL NOT NULL,
        state        TEXT NOT NULL DEFAULT 'IDLE',
        metadata     TEXT NOT NULL DEFAULT '{}'
    );

    CREATE TABLE IF NOT EXISTS messages (
        id           INTEGER PRIMARY KEY AUTOINCREMENT,
        session_id   TEXT NOT NULL,
        role         TEXT NOT NULL,
        content      TEXT NOT NULL,
        created_at   REAL NOT NULL,
        token_count  INTEGER DEFAULT 0,
        FOREIGN KEY (session_id) REFERENCES sessions(session_id)
    );

    CREATE TABLE IF NOT EXISTS tool_calls (
        call_id      TEXT PRIMARY KEY,
        session_id   TEXT NOT NULL,
        tool_name    TEXT NOT NULL,
        arguments    TEXT NOT NULL,
        result       TEXT,
        status       TEXT NOT NULL DEFAULT 'pending',
        started_at   REAL NOT NULL,
        completed_at REAL,
        FOREIGN KEY (session_id) REFERENCES sessions(session_id)
    );

    CREATE INDEX IF NOT EXISTS idx_messages_session
        ON messages(session_id, created_at);
    CREATE INDEX IF NOT EXISTS idx_tool_calls_session
        ON tool_calls(session_id, started_at);
    """

    def __init__(self, db_path: str):
        self.db_path = db_path
        self._init_db()

    def _init_db(self):
        with self._conn() as conn:
            conn.executescript(self.SCHEMA)
            conn.execute("PRAGMA journal_mode=WAL")
            conn.execute("PRAGMA synchronous=NORMAL")

    @contextmanager
    def _conn(self):
        conn = sqlite3.connect(self.db_path, check_same_thread=False)
        conn.row_factory = sqlite3.Row
        try:
            yield conn
            conn.commit()
        except Exception:
            conn.rollback()
            raise
        finally:
            conn.close()

    def upsert_tool_call(self, call: dict) -> None:
        with self._conn() as conn:
            conn.execute("""
                INSERT INTO tool_calls
                    (call_id, session_id, tool_name, arguments,
                     result, status, started_at, completed_at)
                VALUES (:call_id, :session_id, :tool_name, :arguments,
                        :result, :status, :started_at, :completed_at)
                ON CONFLICT(call_id) DO UPDATE SET
                    result=excluded.result,
                    status=excluded.status,
                    completed_at=excluded.completed_at
            """, call)

    def get_failed_sessions(self, since_ts: float) -> List[dict]:
        """Example of a query that's trivial in SQL, painful with files."""
        with self._conn() as conn:
            rows = conn.execute("""
                SELECT s.session_id, s.state, s.updated_at,
                       COUNT(tc.call_id) as failed_tools
                FROM sessions s
                JOIN tool_calls tc ON tc.session_id = s.session_id
                WHERE s.state = 'ERROR'
                  AND s.updated_at > ?
                  AND tc.status = 'error'
                GROUP BY s.session_id
                ORDER BY s.updated_at DESC
            """, (since_ts,)).fetchall()
            return [dict(r) for r in rows]

6. Crash Recovery

Building a Crash-Resilient Agent Loop

Agents fail in ways that web servers don't. The failure surface includes: process OOM kill, LLM API timeout mid-stream, sandbox container crash, network partition, and host VM rescheduling in Kubernetes. Each failure mode requires a different recovery strategy.

Failure Taxonomy

Failure Mode State Lost Recovery Strategy
Process OOM / SIGKILL All in-memory state Load checkpoint; replay from cursor
LLM API timeout mid-stream Partial assistant message Discard partial; re-issue API call with same messages
Sandbox crash during tool execution Tool result not written Re-execute tool (must be idempotent)
Network partition to LLM API None (call not sent) Retry with exponential backoff
Harness bug / exception Varies Catch at outer loop; persist error state; alert ops

Harness->>Sandbox: re_execute(tool_call[5]) with same call_id Sandbox-->>Harness: tool_result (idempotent — same as before) Harness->>Checkpoint: save(cursor=6, tool_log updated) Harness->>LLM: Continue with tool_result in messages LLM-->>Harness: Next response Note over Harness: Normal loop resumes

CrashRecoveryManager Implementation

import signal
import atexit
import logging
from typing import Optional, Callable
from dataclasses import dataclass
from pathlib import Path

logger = logging.getLogger("crash_recovery")


@dataclass
class RecoveryContext:
    session_id: str
    checkpoint: "Checkpoint"
    resume_from_cursor: int
    messages: list
    is_recovery: bool = True


class CrashRecoveryManager:
    """
    Manages crash detection and recovery for an agent session.

    Usage:
        mgr = CrashRecoveryManager(session_id, checkpoint_manager)
        ctx = mgr.start_or_resume()
        # if ctx.is_recovery: replay from ctx.resume_from_cursor
        try:
            run_agent_loop(ctx)
            mgr.complete()
        except Exception as e:
            mgr.record_error(e)
            raise
    """

    def __init__(self, session_id: str, ckpt_mgr: "CheckpointManager"):
        self.session_id = session_id
        self.ckpt_mgr = ckpt_mgr
        self._checkpoint: Optional["Checkpoint"] = None
        self._setup_signal_handlers()

    def _setup_signal_handlers(self):
        """Flush checkpoint on SIGTERM (graceful pod shutdown)."""
        def handler(signum, frame):
            logger.warning("SIGTERM received — flushing checkpoint before exit")
            if self._checkpoint:
                self.ckpt_mgr.save(self._checkpoint)
            raise SystemExit(0)

        signal.signal(signal.SIGTERM, handler)
        atexit.register(self._on_exit)

    def _on_exit(self):
        if self._checkpoint and self._checkpoint.state_machine.get("current_state") not in ("IDLE",):
            logger.warning("Abnormal exit — checkpoint preserved for recovery")
            self.ckpt_mgr.save(self._checkpoint)

    def start_or_resume(self) -> RecoveryContext:
        """
        Check for an existing checkpoint.
        If found, this is a recovery; otherwise start fresh.
        """
        existing = self.ckpt_mgr.load(self.session_id)
        if existing:
            logger.info(
                "Recovery: resuming session %s from cursor=%d",
                self.session_id, existing.cursor
            )
            return RecoveryContext(
                session_id=self.session_id,
                checkpoint=existing,
                resume_from_cursor=existing.cursor,
                messages=existing.messages,
                is_recovery=True,
            )
        else:
            import time
            from dataclasses import field
            fresh = Checkpoint(
                session_id=self.session_id,
                version=1,
                created_at=time.time(),
                state_machine={"current_state": "IDLE", "history": []},
                messages=[],
                tool_call_log=[],
                file_hashes={},
                cursor=0,
                pending_approvals=[],
            )
            self._checkpoint = fresh
            return RecoveryContext(
                session_id=self.session_id,
                checkpoint=fresh,
                resume_from_cursor=0,
                messages=[],
                is_recovery=False,
            )

    def update_checkpoint(self, checkpoint: "Checkpoint") -> None:
        """Call after each successful tool execution to advance the cursor."""
        self._checkpoint = checkpoint
        self.ckpt_mgr.save(checkpoint)

    def record_error(self, error: Exception) -> None:
        if self._checkpoint:
            self._checkpoint.state_machine["current_state"] = "ERROR"
            self._checkpoint.state_machine.setdefault("errors", []).append(str(error))
            self.ckpt_mgr.save(self._checkpoint)

    def complete(self) -> None:
        """Called on successful session completion — remove checkpoint."""
        self.ckpt_mgr.delete(self.session_id)
        self._checkpoint = None
        logger.info("Session %s completed; checkpoint deleted", self.session_id)
Write-Ahead Logging for Tool Operations: Before executing any irreversible tool (file write, API call, DB mutation), write an intent record to a WAL file. On recovery, check the WAL to determine which operations were started but not confirmed complete. This prevents silent data loss where a tool executed but its result was never written to the checkpoint. 
class WriteAheadLog:
    """
    Simple append-only WAL for tool operation intent.
    Pattern: WRITE intent -> EXECUTE tool -> WRITE completion -> EXECUTE next
    """

    def __init__(self, wal_path: str):
        self.path = Path(wal_path)

    def write_intent(self, call_id: str, tool_name: str, args: dict) -> None:
        entry = {"type": "intent", "call_id": call_id,
                 "tool": tool_name, "args": args, "ts": time.time()}
        self._append(entry)

    def write_completion(self, call_id: str, result: any, success: bool) -> None:
        entry = {"type": "completion", "call_id": call_id,
                 "success": success, "result": str(result)[:500], "ts": time.time()}
        self._append(entry)

    def get_incomplete_calls(self) -> list:
        """Return call_ids that have an intent but no completion record."""
        intents = {}
        completions = set()
        for entry in self._read_all():
            if entry["type"] == "intent":
                intents[entry["call_id"]] = entry
            elif entry["type"] == "completion":
                completions.add(entry["call_id"])
        return [v for k, v in intents.items() if k not in completions]

    def _append(self, entry: dict) -> None:
        with open(self.path, "a") as f:
            fcntl.flock(f, fcntl.LOCK_EX)
            f.write(json.dumps(entry) + "\n")
            f.flush()
            fcntl.flock(f, fcntl.LOCK_UN)

    def _read_all(self) -> list:
        if not self.path.exists():
            return []
        records = []
        for line in self.path.read_text().splitlines():
            try:
                records.append(json.loads(line))
            except json.JSONDecodeError:
                pass
        return records

7. Idempotent Tool Execution

Making Every Tool Call Safe to Retry

Idempotency means executing a tool call multiple times with the same inputs produces the same outcome as executing it once. This is the critical enabler of crash recovery: if you cannot safely re-execute a tool, you cannot safely recover from a crash that happened during its execution.

Analogy: "Press the elevator button N times" is idempotent — the elevator arrives once. "Add $10 to your account N times" is not idempotent — you get $10N. Agent tool calls must behave like the elevator button.

Idempotency Key Strategy

Each tool call gets a deterministic idempotency key derived from:

call_id = sha256(session_id + ":" + turn_number + ":" + tool_name + ":" + canonical_json(args))

This means the same logical operation always generates the same key. On crash recovery, we check whether the key already exists in the completed-calls log before re-executing.

IdempotencyLayer Middleware

import hashlib
import json
import time
import functools
from typing import Any, Callable, Dict, Optional
from dataclasses import dataclass


@dataclass
class CachedResult:
    call_id: str
    result: Any
    executed_at: float
    hit_count: int = 0


class IdempotencyLayer:
    """
    Middleware that wraps tool executors with idempotency guarantees.

    Tools are registered with a type hint:
      - "readonly"     -> never cache, always re-execute
      - "write_once"   -> cache result after first success, return cached on retry
      - "idempotent"   -> re-execute is safe but cache anyway for speed
    """

    def __init__(self, store: "CheckpointManager", wal: "WriteAheadLog"):
        self.store = store
        self.wal = wal
        self._cache: Dict[str, CachedResult] = {}
        self._tool_types: Dict[str, str] = {}

    def register_tool(self, name: str, tool_type: str = "write_once"):
        """Register a tool with its idempotency type."""
        assert tool_type in ("readonly", "write_once", "idempotent")
        self._tool_types[name] = tool_type

    def make_call_id(self, session_id: str, turn: int,
                     tool_name: str, args: dict) -> str:
        canonical = json.dumps(args, sort_keys=True)
        raw = f"{session_id}:{turn}:{tool_name}:{canonical}"
        return hashlib.sha256(raw.encode()).hexdigest()[:32]

    def execute(self,
                executor: Callable,
                call_id: str,
                tool_name: str,
                args: dict) -> Any:
        """
        Execute a tool with idempotency protection.
        1. Check cache for existing result
        2. Write WAL intent
        3. Execute
        4. Write WAL completion
        5. Cache result
        """
        tool_type = self._tool_types.get(tool_name, "write_once")

        # Readonly tools always re-execute (no side effects)
        if tool_type == "readonly":
            return executor(**args)

        # Check if already executed
        cached = self._cache.get(call_id)
        if cached:
            cached.hit_count += 1
            return cached.result

        # Write intent to WAL before execution
        self.wal.write_intent(call_id, tool_name, args)

        try:
            result = executor(**args)
            success = True
        except Exception as e:
            self.wal.write_completion(call_id, str(e), success=False)
            raise

        # Record completion
        self.wal.write_completion(call_id, result, success=True)

        # Cache result for future retries
        self._cache[call_id] = CachedResult(
            call_id=call_id,
            result=result,
            executed_at=time.time(),
        )
        return result

    def load_from_checkpoint(self, tool_log: list) -> None:
        """
        Rebuild in-memory cache from a checkpoint's tool log.
        Called during crash recovery so already-completed calls are not re-run.
        """
        for record in tool_log:
            if record.get("status") == "success":
                self._cache[record["call_id"]] = CachedResult(
                    call_id=record["call_id"],
                    result=record.get("result"),
                    executed_at=record.get("completed_at", 0),
                )


# Making file operations idempotent
class IdempotentFileOps:
    """
    Idempotent wrappers for common file operations.
    These are safe to call multiple times with the same arguments.
    """

    @staticmethod
    def write_file(path: str, content: str) -> dict:
        """Write file atomically. Re-running produces same result."""
        p = Path(path)
        p.parent.mkdir(parents=True, exist_ok=True)
        tmp = p.with_suffix(p.suffix + ".tmp")
        tmp.write_text(content, encoding="utf-8")
        tmp.replace(p)
        return {"path": str(p), "bytes": len(content.encode()), "ok": True}

    @staticmethod
    def create_directory(path: str) -> dict:
        """mkdir -p — idempotent by definition."""
        Path(path).mkdir(parents=True, exist_ok=True)
        return {"path": path, "ok": True}

    @staticmethod
    def delete_file_if_exists(path: str) -> dict:
        """Idempotent delete — no error if already gone."""
        p = Path(path)
        existed = p.exists()
        p.unlink(missing_ok=True)
        return {"path": path, "existed": existed, "ok": True}


# Making API calls idempotent
class IdempotentAPIClient:
    """
    Wraps HTTP API calls with idempotency headers.
    Many APIs (Stripe, GitHub) honor Idempotency-Key headers.
    """

    def __init__(self, session: "requests.Session"):
        self.session = session

    def post(self, url: str, call_id: str, payload: dict) -> dict:
        """POST with Idempotency-Key header — safe to retry."""
        resp = self.session.post(
            url,
            json=payload,
            headers={
                "Idempotency-Key": call_id,
                "Content-Type": "application/json",
            },
            timeout=30,
        )
        resp.raise_for_status()
        return resp.json()
Non-Idempotent Operations: Some operations are inherently non-idempotent: sending an email, posting a tweet, charging a credit card. For these, the correct pattern is: (1) generate an idempotency key before the call, (2) record "intent to call" in the WAL, (3) make the call with the idempotency key as a header, (4) record completion. The API server deduplicates on its end.

8. Interview Tips

What Senior Interviewers Look For

State Serialization Questions

  • "How would you design crash recovery for an agent that runs for 30 minutes?" — Talk about checkpointing after each tool call, atomic writes (rename pattern), WAL for intent logging, and idempotency keys. Show you understand the difference between "checkpoint written" and "tool effect committed."
  • "What's in a checkpoint?" — Enumerate all four layers: context window state (messages), state machine position, tool call log with results, and filesystem state (file hashes or git commit). Interviewers who ask this want to see you think systematically about all the places state lives.
  • "How do you avoid double-executing a tool after crash recovery?" — Idempotency keys + result cache loaded from checkpoint. Distinguish readonly (always safe to re-run) vs. write_once (must deduplicate).

Memory Architecture Questions

  • "How does an agent remember things across sessions?" — Distinguish short-term (context window) from long-term (persistent store). Describe promotion criteria: access frequency, cross-session relevance, user corrections. Mention Claude Code's CLAUDE.md hierarchy as a concrete example.
  • "When would you use a database instead of files for agent state?" — Key signals: concurrent multi-agent access (DB wins), need for ad-hoc analytics queries (DB wins), developer transparency/git integration (files win), single-developer local tool (files win).

Red Flags to Avoid

  • Saying "just restart from scratch" — shows you haven't thought about long-running tasks
  • No mention of atomicity in file writes — partial writes corrupt checkpoints
  • Treating all tools as idempotent without distinguishing read vs. write operations
  • Forgetting that external side effects (API calls, emails) cannot be "undone" — must be idempotent or tracked via WAL
  • Over-engineering memory to require vector search when file-based CLAUDE.md is often sufficient

Architectural Trade-off Questions

  • "State machine vs. ad-hoc booleans?" — State machine gives you: serializable state, illegal-transition detection, audit trail, listener hooks. Booleans give you: initial simplicity, nothing else.
  • "How granular should checkpoints be?" — After each tool call is the right granularity. Before API calls is too early (no result yet). After full turns is too coarse (a 10-tool turn may take minutes — losing it is expensive).

Knowledge Check

Q1. An agent process is killed by OOM mid-way through executing the 7th tool call in a session. The checkpoint was last saved after the 6th tool call completed. What is the correct recovery sequence?

Correct sequence:
  1. Load checkpoint (cursor = 6, messages include results of calls 1-6)
  2. Reconstruct state machine — state should be EXECUTING_TOOL (or advance to it from the saved state)
  3. Check WAL for tool call 7: was an "intent" record written? Was a "completion" record written?
  4. If intent-only (no completion): the tool may or may not have fully executed. Re-execute using the same call_id (idempotency key). The tool must be idempotent or the API must honor the idempotency key header.
  5. Write the tool result back to the checkpoint and advance cursor to 7
  6. Continue the agent loop normally from turn 7 onward

Key insight: the WAL tells you what was started; the checkpoint tells you what was completed. The gap between them is the recovery window.

Q2. What is the "atomic rename" pattern and why does checkpoint writing use it?

Pattern: Write new content to a temporary file (checkpoint.json.tmp), then call os.rename(tmp, final). On POSIX systems, rename is atomic at the filesystem level — readers see either the old complete file or the new complete file, never a partial write.

Why it matters for checkpoints: If a process crashes while writing a checkpoint file directly, the file may be half-written and unparseable. A half-written checkpoint is worse than no checkpoint (it blocks recovery). The rename pattern eliminates this risk: the tmp file may be partial, but the final checkpoint file is always a valid complete snapshot.

Q3. Describe the difference between short-term and long-term agent memory, and give a concrete trigger for promoting a fact from short-term to long-term.

Short-term memory lives in the context window — it is the messages array passed to the LLM on each API call. It is lost when the session ends (or when context compression evicts old messages). It has no retrieval cost but has a hard token limit.

Long-term memory is persisted to disk (CLAUDE.md files, a key-value store, a vector DB). It survives process restarts and session boundaries. It requires explicit retrieval: loading a file, semantic search, or key lookup.

Promotion triggers:
  • The fact was accessed 3+ times in the current session (high utility signal)
  • The user explicitly corrected the agent (high-value ground truth)
  • The fact is project-wide (coding style, build commands) — not task-specific
  • The session is ending and the fact is marked as "sticky" by the agent's reasoning

Q4. You are building a harness for an agent that sends Slack messages as a tool. How do you make this tool idempotent for crash recovery purposes?

Approach 1 — API-level idempotency: Generate a deterministic call_id = sha256(session+turn+tool+args) before calling. Pass it as a custom header or query parameter. Build (or use an existing) deduplication layer server-side that ignores duplicate calls with the same key within a time window (e.g., 24 hours).

Approach 2 — WAL + result cache: Write intent to WAL before calling. After successful send, write the Slack message_ts to the checkpoint's tool log. On crash recovery, check the log: if call_id has a result, return the cached result without re-calling Slack.

Approach 3 — Check-before-send: Before sending, query Slack's history for a message with your unique call_id in the text or metadata. If found, skip. This works but adds latency.

Best practice is Approach 2 (WAL + result cache) for most cases, with Approach 1 as a belt-and-suspenders addition for critical messages.

Q5. When would you choose a SQLite-backed state store over a file-based JSONL store for an agent harness? Give three concrete scenarios.

Choose SQLite (database) when:
  1. Multiple concurrent agent workers share the same session. SQLite's WAL mode allows multiple readers and one writer concurrently without file-lock conflicts. JSONL requires external coordination (advisory locks) for concurrent writes.
  2. You need operational analytics over agent runs. Example: "Show me all sessions that entered ERROR state in the last hour with a failed bash tool call." This is trivial SQL but painful with grep over thousands of JSONL files.
  3. You need to enforce referential integrity between state entities. Example: every tool_call must reference a valid session, and every message must reference a valid turn. FK constraints in SQLite catch bugs at write time; JSONL files have no such enforcement.
Stick with JSONL when: Single-user local tool, git-based audit trail is important, developer transparency over raw state is a priority, or simplicity of deployment matters (no DB dependency).

State management is the invisible backbone of reliable agent systems. You now understand:

  • State distribution — Agent state lives in four layers (context, process memory, filesystem, external systems). Your harness must bridge them all.
  • State machines — Formalizing agent conversations as 6-state FSMs gives you serializable, auditable, and recoverable control flow.
  • Checkpointing — Atomic write-then-rename pattern, checkpoint after each tool call, and the WAL for intent tracking are the three pillars of crash-safe persistence.
  • Memory tiers — Short-term lives in the context window; long-term lives in files or a store. Promote based on access frequency, correction signals, and cross-session utility.
  • File vs. DB — Files win on simplicity, transparency, and git integration. Databases win on concurrency, queryability, and multi-machine scenarios. Hybrids use files per session with a DB index.
  • Idempotency — Deterministic call_ids + result cache + WAL make crash recovery safe. Non-idempotent external calls require API-level idempotency keys.
Reliability formula for agent state:

P(successful recovery) = P(checkpoint valid) × P(tool idempotent | retry) × P(WAL complete)

Maximize each factor independently:
- Atomic writes for checkpoint validity
- Idempotency keys + result cache for tool safety
- Sync-after-write (fsync) for WAL completeness
