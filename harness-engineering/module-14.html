Module 14: Multi-Agent Orchestration

Move beyond the single-agent loop. Learn how to design, coordinate, budget, and debug systems where multiple AI agents collaborate — and why you usually should not.

What You'll Learn

  • When multi-agent architectures add genuine value versus unnecessary complexity
  • Supervisor/worker patterns: delegation, aggregation, and error handling
  • Shared context protocols: filesystem, message passing, shared memory, and context inheritance
  • Fan-out / fan-in for parallel task decomposition and result aggregation
  • Conflict resolution when multiple agents touch the same resource
  • Agent lifecycle management: creation, resource limits, timeouts, and cleanup
  • Cost budgeting across multi-agent systems — the most underestimated problem
  • The four canonical multi-agent architecture patterns and when to use each

1. Why Multi-Agent?

Before designing any multi-agent system, you must answer honestly: does this task actually require multiple agents, or am I reaching for complexity? The majority of tasks that engineers reach for multi-agent to solve can be handled by a single agent with better tool design and context management.

Single-Agent Limitations

Context Window Ceiling

Even with 200K-token windows, a codebase of 50,000 files cannot fit. When the full working set exceeds what can be held in one context, you need agents that each own a slice.

Specialization

A single agent prompted to do everything (security review, test writing, documentation) is less effective than specialized agents each with a focused system prompt and relevant tools.

Parallelism

LLM inference is serial within a single agent loop. If you have 10 independent subtasks, 10 parallel workers reduce wall-clock time by ~10x at the cost of coordination overhead.

Fault Isolation

A monolithic agent that fails mid-task loses everything. Worker agents can fail and be retried independently without restarting the entire job.

When Multi-Agent Adds Value vs. Complexity

Scenario Single Agent Multi-Agent Verdict
Fix a single bug in a known file Trivial, fast, cheap Massive overhead for no benefit Single agent
Migrate 200 files to a new API Context window overflow, slow serial Fan-out to 20 workers, each handles 10 files Multi-agent
Write feature + tests + docs Fine if tasks are sequential Parallel if tasks are independent; overhead otherwise Depends on coupling
Security audit of a codebase Struggles with breadth Specialist agents per layer (auth, crypto, injection, etc.) Multi-agent
Answer a complex research question Single deep reasoner works well Coordination latency exceeds thinking benefit Single agent
Real Example: Claude Code's Agent Tool

Claude Code exposes an Agent tool that allows the top-level agent to spawn sub-agents with their own system prompts, tool sets, and context windows. A typical use: the top agent identifies 15 independent files that need refactoring, spawns 15 sub-agents in parallel (each with the relevant file in context), and aggregates the edits. The sub-agents never see each other's context — they communicate only through the filesystem.

The Coordination Tax

Every multi-agent system pays a coordination cost that single-agent systems do not: supervisor LLM calls to plan and delegate, parsing/validation of worker outputs, handling partial failures and retries, additional token spend for context duplication, and debugging complexity that grows super-linearly with agent count. Before adding an agent, ask: does the expected gain exceed this tax?

2. Supervisor / Worker Patterns

The most common multi-agent topology is the supervisor/worker pattern. The supervisor is a planning agent that decomposes a high-level goal into concrete subtasks, delegates each subtask to a specialized worker, and then aggregates the results back into a coherent output.

Responsibilities

  • Supervisor: Holds the task decomposition plan, worker registry, result buffer, and final synthesis logic. It should be promted with a "planner" persona and given tools to spawn, query, and cancel workers.
  • Worker: Receives a single, focused task definition. Has a specialized system prompt, a restricted tool set, a dedicated context window, and a bounded token budget. Workers should be stateless where possible.
  • Communication protocol: Workers return structured results (JSON with result, status, error fields) — never free-form prose that the supervisor must parse.

import asyncio
import json
import uuid
from dataclasses import dataclass, field
from enum import Enum
from typing import Any

import anthropic


class WorkerStatus(str, Enum):
    OK = "ok"
    ERROR = "error"
    TIMEOUT = "timeout"


@dataclass
class WorkerTask:
    task_id: str
    description: str
    context: dict[str, Any]
    max_tokens: int = 4096
    timeout_seconds: float = 60.0


@dataclass
class WorkerResult:
    task_id: str
    status: WorkerStatus
    output: Any = None
    error: str | None = None
    tokens_used: int = 0


@dataclass
class WorkerAgent:
    """A focused, stateless agent for a single subtask."""
    worker_id: str
    system_prompt: str
    tools: list[dict]
    client: anthropic.Anthropic = field(default_factory=anthropic.Anthropic)

    async def execute(self, task: WorkerTask) -> WorkerResult:
        messages = [
            {
                "role": "user",
                "content": f"{task.description}\n\nContext:\n{json.dumps(task.context, indent=2)}"
            }
        ]
        total_tokens = 0
        try:
            async with asyncio.timeout(task.timeout_seconds):
                # Simplified: single non-streaming call for clarity
                response = self.client.messages.create(
                    model="claude-opus-4-5",
                    max_tokens=task.max_tokens,
                    system=self.system_prompt,
                    tools=self.tools,
                    messages=messages,
                )
                total_tokens = response.usage.input_tokens + response.usage.output_tokens
                # Extract structured output from the last text block
                output_text = next(
                    (b.text for b in response.content if b.type == "text"), ""
                )
                # Workers must return JSON
                output = json.loads(output_text)
                return WorkerResult(
                    task_id=task.task_id,
                    status=WorkerStatus.OK,
                    output=output,
                    tokens_used=total_tokens,
                )
        except asyncio.TimeoutError:
            return WorkerResult(
                task_id=task.task_id,
                status=WorkerStatus.TIMEOUT,
                error=f"Worker timed out after {task.timeout_seconds}s",
                tokens_used=total_tokens,
            )
        except Exception as exc:
            return WorkerResult(
                task_id=task.task_id,
                status=WorkerStatus.ERROR,
                error=str(exc),
                tokens_used=total_tokens,
            )


class SupervisorAgent:
    """Plans, delegates, retries, and synthesizes across workers."""

    def __init__(
        self,
        worker_registry: dict[str, WorkerAgent],
        max_retries: int = 2,
    ):
        self.workers = worker_registry
        self.max_retries = max_retries
        self.client = anthropic.Anthropic()

    async def _delegate(
        self, worker_name: str, task: WorkerTask, attempt: int = 0
    ) -> WorkerResult:
        worker = self.workers[worker_name]
        result = await worker.execute(task)

        if result.status != WorkerStatus.OK and attempt < self.max_retries:
            print(f"[Supervisor] Worker {worker_name} failed ({result.error}), retry {attempt+1}")
            # Fresh task_id on retry so we can distinguish in logs
            task.task_id = str(uuid.uuid4())
            return await self._delegate(worker_name, task, attempt + 1)

        return result

    async def run(self, goal: str, decomposition: list[dict]) -> dict:
        """
        decomposition: list of {worker: str, task: str, context: dict}
        Returns aggregated results keyed by task_id.
        """
        tasks = [
            (
                step["worker"],
                WorkerTask(
                    task_id=str(uuid.uuid4()),
                    description=step["task"],
                    context=step.get("context", {}),
                ),
            )
            for step in decomposition
        ]

        # Fan-out: run all tasks concurrently
        coros = [self._delegate(worker_name, task) for worker_name, task in tasks]
        results: list[WorkerResult] = await asyncio.gather(*coros)

        # Fan-in: aggregate
        aggregated = {r.task_id: r for r in results}
        failed = [r for r in results if r.status != WorkerStatus.OK]

        return {
            "goal": goal,
            "results": {tid: {"status": r.status, "output": r.output} for tid, r in aggregated.items()},
            "failed_count": len(failed),
            "total_tokens": sum(r.tokens_used for r in results),
        }
Key Engineering Decision

Workers must return structured JSON, not prose. If a worker returns free text that the supervisor must parse, you have introduced an unreliable deserialization step at every aggregation boundary. Force structured output by using tool use or a strict JSON system prompt, and validate the schema before the supervisor acts on it.

3. Shared Context Protocols

Agents that collaborate need to exchange information. There is no single right answer — the choice depends on the size of data being shared, the need for real-time visibility, and the acceptable coupling between agents.

Four Protocols

Protocol Mechanism Best For Drawbacks
Shared Filesystem Workers write results to files; others read via path convention Large artifacts (code diffs, reports); natural for coding agents No ordering guarantees; polling needed; conflict on concurrent writes
Message Passing Structured messages routed through supervisor or a queue Event-driven pipelines; clear provenance; easy to log Supervisor becomes a bottleneck; latency for each hop
Shared Memory Store Key-value store (Redis, in-memory dict) accessible by all agents Small, frequently-read state (counters, flags, interim results) Race conditions; requires locking; added infrastructure
Context Inheritance Parent serializes its conversation history and passes it as context to child Hierarchical planning where children need full parent context Token duplication; child context grows with parent; expensive

import asyncio
import time
from dataclasses import dataclass, field
from typing import Any


@dataclass
class ContextEntry:
    value: Any
    written_by: str
    written_at: float
    version: int = 1


class SharedContextStore:
    """
    Thread-safe in-process shared context store.
    For multi-process deployments, swap the backing store for Redis.
    """

    def __init__(self) -> None:
        self._store: dict[str, ContextEntry] = {}
        self._lock = asyncio.Lock()
        self._watchers: dict[str, list[asyncio.Event]] = {}

    async def write(self, key: str, value: Any, agent_id: str) -> int:
        """Write a value. Returns the new version number."""
        async with self._lock:
            existing = self._store.get(key)
            version = (existing.version + 1) if existing else 1
            self._store[key] = ContextEntry(
                value=value,
                written_by=agent_id,
                written_at=time.monotonic(),
                version=version,
            )
            # Notify waiters
            for event in self._watchers.pop(key, []):
                event.set()
            return version

    async def read(self, key: str) -> ContextEntry | None:
        async with self._lock:
            return self._store.get(key)

    async def read_or_wait(self, key: str, timeout: float = 30.0) -> ContextEntry | None:
        """Block until a key is available or timeout expires."""
        async with self._lock:
            if key in self._store:
                return self._store[key]
            event = asyncio.Event()
            self._watchers.setdefault(key, []).append(event)

        try:
            await asyncio.wait_for(event.wait(), timeout=timeout)
        except asyncio.TimeoutError:
            return None

        async with self._lock:
            return self._store.get(key)

    async def compare_and_swap(
        self, key: str, expected_version: int, new_value: Any, agent_id: str
    ) -> bool:
        """Optimistic concurrency: only write if version matches."""
        async with self._lock:
            entry = self._store.get(key)
            current_version = entry.version if entry else 0
            if current_version != expected_version:
                return False
            self._store[key] = ContextEntry(
                value=new_value,
                written_by=agent_id,
                written_at=time.monotonic(),
                version=current_version + 1,
            )
            return True

    def snapshot(self) -> dict[str, Any]:
        """Return a read-only snapshot for debugging/logging."""
        return {k: {"value": v.value, "by": v.written_by, "v": v.version}
                for k, v in self._store.items()}

4. Fan-Out / Fan-In

Fan-out/fan-in is the core pattern for parallel work. You split a task into N independent subtasks, execute them in parallel across N agents, then aggregate the results into a single coherent output.

Work Decomposition Strategies

  • Data partitioning: Split a dataset (files, chunks, records) evenly across workers. Works when work per item is uniform.
  • Functional specialization: Each worker applies a different function to the same input (security check, style check, logic check).
  • Hierarchical decomposition: Supervisor decomposes the goal recursively; workers may themselves spawn sub-workers.
  • Dynamic load balancing: A work queue is consumed by workers as they become available, rather than pre-assigning partitions.

import asyncio
from dataclasses import dataclass, field
from typing import Callable, TypeVar

T = TypeVar("T")
R = TypeVar("R")


@dataclass
class FanOutResult:
    index: int
    item: any
    result: any = None
    error: str | None = None
    attempts: int = 0


class FanOutFanIn:
    """
    Generic fan-out/fan-in executor.

    worker_fn:   async function that processes one item
    aggregator:  function that merges all successful results
    max_workers: max concurrency
    max_retries: retries per failed item
    """

    def __init__(
        self,
        worker_fn: Callable,
        aggregator: Callable,
        max_workers: int = 10,
        max_retries: int = 2,
    ):
        self.worker_fn = worker_fn
        self.aggregator = aggregator
        self.semaphore = asyncio.Semaphore(max_workers)
        self.max_retries = max_retries

    async def _process_item(self, index: int, item) -> FanOutResult:
        fo_result = FanOutResult(index=index, item=item)
        for attempt in range(self.max_retries + 1):
            fo_result.attempts = attempt + 1
            async with self.semaphore:
                try:
                    fo_result.result = await self.worker_fn(item)
                    fo_result.error = None
                    return fo_result
                except Exception as exc:
                    fo_result.error = str(exc)
                    if attempt < self.max_retries:
                        wait = 2 ** attempt  # exponential backoff
                        await asyncio.sleep(wait)
        return fo_result  # failed after all retries

    async def run(self, items: list) -> dict:
        coros = [self._process_item(i, item) for i, item in enumerate(items)]
        raw_results: list[FanOutResult] = await asyncio.gather(*coros)

        successes = [r for r in raw_results if r.error is None]
        failures  = [r for r in raw_results if r.error is not None]

        # Sort by original index to preserve ordering for aggregator
        successes.sort(key=lambda r: r.index)
        aggregated = self.aggregator([r.result for r in successes])

        return {
            "aggregated": aggregated,
            "success_count": len(successes),
            "failure_count": len(failures),
            "failures": [{"index": r.index, "error": r.error} for r in failures],
        }

5. Conflict Resolution

When multiple agents can modify the same resource (a file, a database row, a shared state key), you must have a conflict resolution strategy. This is not theoretical — Claude Code's multi-agent mode surfaces this constantly when two workers both edit the same source file.

Strategies

Strategy How It Works When to Use Risk
Pessimistic Lock Agent acquires exclusive lock before reading; others block High write contention; correctness critical Deadlock; reduced parallelism
Optimistic + Retry Read version, write if version unchanged, else retry Low contention; high-throughput scenarios Livelock under very high contention
Partition by Resource Assign each resource to exactly one worker at planning time File-level work in coding agents; clear ownership Load imbalance; requires upfront partition plan
Three-Way Merge Supervisor merges concurrent edits using a common ancestor Document / code edits where diffs are meaningful Merge conflicts still possible; needs semantic merge logic
Last-Write-Wins Most recent timestamp overwrites prior value Idempotent data with no ordering semantics Silent data loss; should be last resort

import asyncio
import difflib
import time
from dataclasses import dataclass
from enum import Enum


class MergeStrategy(str, Enum):
    PARTITION   = "partition"
    OPTIMISTIC  = "optimistic"
    THREE_WAY   = "three_way"
    LWW         = "last_write_wins"


@dataclass
class FileVersion:
    path: str
    content: str
    version: int
    modified_by: str
    modified_at: float


class ConflictResolver:
    def __init__(self, strategy: MergeStrategy = MergeStrategy.OPTIMISTIC):
        self.strategy = strategy
        self._files: dict[str, FileVersion] = {}
        self._lock = asyncio.Lock()
        self._ancestors: dict[str, str] = {}  # path -> original content

    async def checkout(self, path: str, agent_id: str) -> FileVersion:
        """Agent reads the current version before editing."""
        async with self._lock:
            if path not in self._files:
                self._files[path] = FileVersion(
                    path=path, content="", version=0,
                    modified_by="init", modified_at=time.monotonic()
                )
            fv = self._files[path]
            # Store ancestor for three-way merge
            self._ancestors.setdefault(path, fv.content)
            return fv

    async def commit(
        self,
        path: str,
        new_content: str,
        expected_version: int,
        agent_id: str,
    ) -> tuple[bool, str]:
        """
        Returns (success, final_content).
        On conflict, applies the configured strategy.
        """
        async with self._lock:
            current = self._files[path]

            if current.version == expected_version:
                # No conflict — clean commit
                self._files[path] = FileVersion(
                    path=path, content=new_content,
                    version=current.version + 1,
                    modified_by=agent_id, modified_at=time.monotonic()
                )
                return True, new_content

            # Conflict detected
            if self.strategy == MergeStrategy.LWW:
                self._files[path] = FileVersion(
                    path=path, content=new_content,
                    version=current.version + 1,
                    modified_by=agent_id, modified_at=time.monotonic()
                )
                return True, new_content

            elif self.strategy == MergeStrategy.THREE_WAY:
                ancestor = self._ancestors.get(path, "")
                merged = self._three_way_merge(ancestor, current.content, new_content)
                if merged is None:
                    return False, current.content  # unresolvable conflict
                self._files[path] = FileVersion(
                    path=path, content=merged,
                    version=current.version + 1,
                    modified_by=f"{agent_id}+merge", modified_at=time.monotonic()
                )
                return True, merged

            # OPTIMISTIC: signal failure; caller should re-read and retry
            return False, current.content

    @staticmethod
    def _three_way_merge(ancestor: str, ours: str, theirs: str) -> str | None:
        """
        Simplified line-level three-way merge.
        Returns None if there are conflicting hunks.
        """
        a_lines = ancestor.splitlines(keepends=True)
        o_lines = ours.splitlines(keepends=True)
        t_lines = theirs.splitlines(keepends=True)

        matcher_o = difflib.SequenceMatcher(None, a_lines, o_lines)
        matcher_t = difflib.SequenceMatcher(None, a_lines, t_lines)

        # Build change sets
        o_changes = {tag: (i1, i2, j1, j2) for tag, i1, i2, j1, j2 in matcher_o.get_opcodes() if tag != "equal"}
        t_changes = {tag: (i1, i2, j1, j2) for tag, i1, i2, j1, j2 in matcher_t.get_opcodes() if tag != "equal"}

        # If no changes touch the same ancestor lines, merge is trivial
        # For production use: adopt a proper merge library (e.g., diff3)
        # This stub returns theirs if no explicit conflict detected
        return theirs
Production Advice

Partition by resource at planning time whenever possible. The cleanest conflict avoidance strategy is to never have two agents touch the same file. The supervisor's decomposition step should assign non-overlapping file sets to workers. Only fall back to merge strategies for cases where shared ownership is unavoidable (e.g., a shared configuration file or a central registry).

6. Agent Lifecycle Management

In a multi-agent system, agents are resources. They consume memory, LLM API quota, and compute. Production harnesses need explicit lifecycle management — just as you would manage threads or processes in a concurrent system.

Lifecycle States

stateDiagram-v2 [*] --> PENDING: spawn() called by supervisor PENDING --> INITIALIZING: worker slot available INITIALIZING --> RUNNING: context loaded, tools bound RUNNING --> WAITING: awaiting tool result WAITING --> RUNNING: tool result received RUNNING --> COMPLETED: task finished RUNNING --> FAILED: exception / model error RUNNING --> TIMED_OUT: deadline exceeded FAILED --> PENDING: retry scheduled TIMED_OUT --> PENDING: retry scheduled COMPLETED --> [*]: result delivered to supervisor FAILED --> [*]: max retries exhausted TIMED_OUT --> [*]: max retries exhausted

Resource Limits Per Agent

  • Token budget: Each worker receives a hard cap. Exceeding it triggers graceful truncation of its context, not an exception.
  • Wall-clock timeout: Prevents a stuck agent from blocking the entire fan-in. Set to 2-3x the P99 latency of that agent type.
  • Tool call limit: Cap the maximum number of tool calls per agent turn to prevent runaway loops (e.g., max 50 tool calls per worker execution).
  • Concurrency cap: Use a semaphore to limit simultaneous agents to avoid overwhelming the model provider's rate limits and your sandbox infrastructure.
import asyncio
from dataclasses import dataclass, field
from enum import Enum
import time
import logging

logger = logging.getLogger(__name__)


class AgentState(str, Enum):
    PENDING      = "PENDING"
    INITIALIZING = "INITIALIZING"
    RUNNING      = "RUNNING"
    WAITING      = "WAITING"
    COMPLETED    = "COMPLETED"
    FAILED       = "FAILED"
    TIMED_OUT    = "TIMED_OUT"


@dataclass
class AgentHandle:
    agent_id: str
    task_id: str
    state: AgentState = AgentState.PENDING
    created_at: float = field(default_factory=time.monotonic)
    started_at: float | None = None
    completed_at: float | None = None
    token_budget: int = 8192
    tokens_used: int = 0
    tool_calls: int = 0
    max_tool_calls: int = 50

    def duration(self) -> float | None:
        if self.started_at and self.completed_at:
            return self.completed_at - self.started_at
        return None

    def budget_remaining(self) -> int:
        return max(0, self.token_budget - self.tokens_used)


class AgentLifecycleManager:
    """Tracks and enforces lifecycle of all active agents."""

    def __init__(self, max_concurrent: int = 20):
        self._agents: dict[str, AgentHandle] = {}
        self._semaphore = asyncio.Semaphore(max_concurrent)
        self._lock = asyncio.Lock()

    async def spawn(self, task_id: str, token_budget: int = 8192) -> AgentHandle:
        import uuid
        agent_id = str(uuid.uuid4())[:8]
        handle = AgentHandle(agent_id=agent_id, task_id=task_id, token_budget=token_budget)
        async with self._lock:
            self._agents[agent_id] = handle
        logger.info(f"[Lifecycle] Spawned agent {agent_id} for task {task_id}")
        return handle

    async def transition(self, agent_id: str, new_state: AgentState) -> None:
        async with self._lock:
            handle = self._agents[agent_id]
            old_state = handle.state
            handle.state = new_state
            if new_state == AgentState.RUNNING:
                handle.started_at = time.monotonic()
            elif new_state in (AgentState.COMPLETED, AgentState.FAILED, AgentState.TIMED_OUT):
                handle.completed_at = time.monotonic()
            logger.debug(f"[Lifecycle] Agent {agent_id}: {old_state} -> {new_state}")

    async def record_tool_call(self, agent_id: str, tokens_delta: int) -> bool:
        """Returns False if agent has exceeded its limits."""
        async with self._lock:
            handle = self._agents[agent_id]
            handle.tokens_used += tokens_delta
            handle.tool_calls += 1
            if handle.tokens_used > handle.token_budget:
                logger.warning(f"[Lifecycle] Agent {agent_id} exceeded token budget")
                return False
            if handle.tool_calls > handle.max_tool_calls:
                logger.warning(f"[Lifecycle] Agent {agent_id} exceeded tool call limit")
                return False
            return True

    async def cleanup(self, agent_id: str) -> AgentHandle:
        async with self._lock:
            return self._agents.pop(agent_id, None)

    def active_agents(self) -> list[AgentHandle]:
        return [h for h in self._agents.values()
                if h.state in (AgentState.RUNNING, AgentState.WAITING)]

    def metrics(self) -> dict:
        handles = list(self._agents.values())
        return {
            "total": len(handles),
            "by_state": {s.value: sum(1 for h in handles if h.state == s) for s in AgentState},
            "avg_tokens": sum(h.tokens_used for h in handles) / max(len(handles), 1),
        }

7. Cost Budgeting Across Agents

Cost is the most commonly underestimated dimension of multi-agent systems. A task that costs $0.05 with a single agent can easily cost $2.00 when orchestrated across a supervisor and 10 workers — because the supervisor itself consumes tokens, every worker gets a duplicated copy of shared context, and retries multiply costs. You must model costs before shipping.

Total Cost Model:

total_cost = supervisor_cost + Σ(worker_i_cost) + retry_overhead

supervisor_cost = (input_tokens × price_in) + (output_tokens × price_out)

worker_cost = (shared_context_tokens + task_tokens) × price_in + output_tokens × price_out

retry_overhead = failed_fraction × avg_worker_cost × avg_retries

Key insight: shared_context_tokens is duplicated for EVERY worker. If you pass a 50K-token codebase context to 10 workers, you pay for 500K input tokens of shared context alone.

Worked Example: 5-Agent Coding Task

Agent Input Tokens Output Tokens Cost @ $3/M in, $15/M out
Supervisor (plan + aggregate) 8,000 2,000 $0.024 + $0.030 = $0.054
Worker 1: Security Review 12,000 1,500 $0.036 + $0.023 = $0.059
Worker 2: Unit Tests 10,000 3,000 $0.030 + $0.045 = $0.075
Worker 3: Documentation 9,000 2,500 $0.027 + $0.038 = $0.065
Worker 4: Refactor (retry x1) 11,000 × 2 = 22,000 2,000 × 2 = 4,000 $0.066 + $0.060 = $0.126
Total 61,000 13,000 $0.379

Code: MultiAgentBudgetManager

import asyncio
import time
from dataclasses import dataclass, field


@dataclass
class AgentBudget:
    agent_id: str
    allocated_usd: float
    spent_usd: float = 0.0
    input_tokens: int = 0
    output_tokens: int = 0


class MultiAgentBudgetManager:
    """
    Tracks and enforces per-agent and total cost budgets.

    Pricing is configurable per model. Default: Claude Opus 4.5 rates.
    """

    def __init__(
        self,
        total_budget_usd: float,
        price_per_m_input: float = 3.0,
        price_per_m_output: float = 15.0,
    ):
        self.total_budget_usd = total_budget_usd
        self.price_in = price_per_m_input / 1_000_000
        self.price_out = price_per_m_output / 1_000_000
        self._budgets: dict[str, AgentBudget] = {}
        self._lock = asyncio.Lock()

    async def allocate(self, agent_id: str, fraction: float) -> float:
        """Allocate a fraction of the remaining total budget to an agent."""
        async with self._lock:
            remaining = self.total_budget_usd - self._total_spent()
            allocated = remaining * fraction
            self._budgets[agent_id] = AgentBudget(
                agent_id=agent_id, allocated_usd=allocated
            )
            return allocated

    async def record_usage(
        self, agent_id: str, input_tokens: int, output_tokens: int
    ) -> bool:
        """
        Record token usage. Returns False if the agent has exceeded its budget
        (caller should stop the agent gracefully).
        """
        cost = input_tokens * self.price_in + output_tokens * self.price_out
        async with self._lock:
            if agent_id not in self._budgets:
                # Auto-register with remaining budget
                self._budgets[agent_id] = AgentBudget(
                    agent_id=agent_id,
                    allocated_usd=self.total_budget_usd - self._total_spent()
                )
            budget = self._budgets[agent_id]
            budget.spent_usd += cost
            budget.input_tokens += input_tokens
            budget.output_tokens += output_tokens
            if budget.spent_usd > budget.allocated_usd:
                return False  # over-budget
            if self._total_spent() > self.total_budget_usd:
                return False  # global budget exhausted
            return True

    async def redistribute(self, source_id: str, target_id: str, amount_usd: float) -> bool:
        """Transfer unused budget from a completed agent to another agent."""
        async with self._lock:
            source = self._budgets.get(source_id)
            target = self._budgets.get(target_id)
            if not source or not target:
                return False
            transferable = source.allocated_usd - source.spent_usd
            if transferable < amount_usd:
                return False
            source.allocated_usd -= amount_usd
            target.allocated_usd += amount_usd
            return True

    def _total_spent(self) -> float:
        return sum(b.spent_usd for b in self._budgets.values())

    async def summary(self) -> dict:
        async with self._lock:
            return {
                "total_budget_usd": self.total_budget_usd,
                "total_spent_usd": round(self._total_spent(), 6),
                "remaining_usd": round(self.total_budget_usd - self._total_spent(), 6),
                "agents": {
                    aid: {
                        "allocated": round(b.allocated_usd, 6),
                        "spent": round(b.spent_usd, 6),
                        "utilization_pct": round(b.spent_usd / max(b.allocated_usd, 1e-9) * 100, 1),
                    }
                    for aid, b in self._budgets.items()
                },
            }
Dynamic Budget Redistribution

Allocate budgets conservatively and redistribute dynamically. Assign each worker only 60-70% of what you expect it to need. As workers complete early, redistribute their unused budget to agents performing complex retries. This prevents a single expensive worker from exhausting the global budget and starving others.

8. Architecture Patterns

Multi-agent systems fall into four canonical topology patterns. Real systems often combine them, but understanding the pure forms helps you reason about trade-offs.

Pattern 1: Pipeline (Sequential)

Each agent processes the output of the previous. Useful when each stage transforms the artifact and the next stage depends on the full transformed output. Low parallelism; easy to reason about; each agent's context is clean.

Parse + Structure] A1 --> A2[Agent B
Enrich + Validate] A2 --> A3[Agent C
Format + Output] A3 --> O([Final Output]) style A1 fill:#0e7490,stroke:#06b6d4,color:#fff style A2 fill:#0e7490,stroke:#06b6d4,color:#fff style A3 fill:#0e7490,stroke:#06b6d4,color:#fff

Pattern 2: Map-Reduce

Split input into N shards, process each shard independently (map), then combine results (reduce). The canonical pattern for breadth-first tasks like auditing a large codebase or summarizing many documents.

graph TD I([Large Input]) --> Split[Splitter] Split --> M1[Map Agent 1] Split --> M2[Map Agent 2] Split --> M3[Map Agent 3] Split --> M4[Map Agent 4] M1 --> R[Reduce Agent] M2 --> R M3 --> R M4 --> R R --> O([Aggregated Output]) style Split fill:#0e7490,stroke:#06b6d4,color:#fff style R fill:#064e3b,stroke:#10b981,color:#fff

Pattern 3: Swarm

Agents operate autonomously toward a shared objective. No explicit supervisor — agents communicate through a shared blackboard (context store) and self-coordinate. High parallelism; hard to debug; suitable for exploration tasks where the search space is too large for a supervisor to plan upfront.

graph TD Obj([Shared Objective]) --> SB[(Shared
Blackboard)] A1[Agent 1] <-->|read/write| SB A2[Agent 2] <-->|read/write| SB A3[Agent 3] <-->|read/write| SB A4[Agent 4] <-->|read/write| SB SB --> Done([Objective Achieved?]) style SB fill:#1e3a5f,stroke:#3b82f6,color:#fff style Done fill:#064e3b,stroke:#10b981,color:#fff

Pattern 4: Hierarchical (Multi-Level Supervisor)

A top-level supervisor delegates to mid-level supervisors, each of which manages its own worker pool. Suitable for very large tasks (thousands of files) where a single supervisor would become a bottleneck or exceed its own context window planning all the work.

9. Interview Tips

What Interviewers Look For

Architecture Discussion

  • Can you articulate why you need multiple agents rather than reaching for it by default? Interviewers respect engineers who argue against multi-agent when it is not warranted.
  • Can you draw the information flow: who delegates to whom, where shared state lives, how the fan-in works?
  • Do you proactively bring up failure modes — worker failures, partial results, budget overruns — without being prompted?

Cost Awareness

  • Multi-agent systems multiply costs. Interviewers at AI-native companies will ask you to estimate the cost of your design. Have the math ready.
  • Mention context duplication: if every worker gets the same shared context, input tokens scale linearly with worker count.

Conflict and Concurrency

  • Show that you treat shared resources as a concurrency problem, not just an AI problem. Mention versioning, optimistic locking, and partition-by-resource.
  • Know the difference between pessimistic and optimistic concurrency and when each applies.

Red Flags to Avoid

  • Proposing multi-agent for every problem without articulating the cost/benefit.
  • Ignoring the coordination tax — assuming N agents = N× speedup.
  • Not having a conflict resolution strategy for shared resources.
  • No budget management — treating LLM costs as free.
  • Workers returning unstructured prose that the supervisor must re-parse with another LLM call.

Strong Signal Answers

  • "I would first check whether better tool design or context management could solve this without multiple agents."
  • "The supervisor should partition the work so that no two workers touch the same file — that's the cleanest conflict avoidance."
  • "Each worker returns structured JSON with a status field so the supervisor can make decisions without another LLM call."
  • "I'd allocate 60% of the total token budget to workers, hold 20% in reserve for retries, and use 20% for the supervisor."

10. Quiz

Q1. A single-agent coding assistant exceeds its context window trying to refactor a 10,000-file monorepo. You propose a multi-agent system. What is the single most important thing the supervisor must do during decomposition?

Partition the file set so no two workers are assigned overlapping files. This eliminates merge conflicts at the source. If two workers can touch the same file, you need a conflict resolution protocol that adds significant complexity. Partition-by-resource is the correct default; only introduce merge logic for the rare cases (shared config files, global type definitions) where it is unavoidable.

Q2. You have a fan-out of 20 workers, each receiving a 10,000-token shared context plus a 2,000-token task-specific context. Using Claude Opus 4.5 ($3/M input tokens), what is the input token cost of the shared context alone?

20 workers × 10,000 tokens = 200,000 input tokens. At $3 / 1,000,000 tokens, that is $0.60 in shared-context input costs alone — before any task-specific tokens or output tokens. This illustrates why minimizing shared context duplication (via a shared context store that workers query on-demand rather than receiving upfront) can significantly reduce costs at scale.

Q3. What is the difference between optimistic concurrency and pessimistic locking in the context of multi-agent file modifications? When does each fail?

Pessimistic locking requires an agent to acquire an exclusive lock before reading. Others block until the lock is released. Fails under: high contention causing long queues, and risk of deadlock if two agents each hold a lock the other needs.

Optimistic concurrency lets agents read freely and attach the version they read. The write succeeds only if the version is still current; otherwise the agent re-reads and retries. Fails under: very high contention where retries loop without making progress (livelock).

For agent systems, optimistic concurrency is usually preferred because contention is low when work is properly partitioned.

Q4. You have a supervisor + 4 workers. Workers 1, 2, and 3 succeed. Worker 4 returns a timeout error. What are the correct next steps in order?

  1. Do not discard results from workers 1-3. Store them in the shared context or result buffer.
  2. Check the retry count for worker 4. If below max_retries, re-spawn worker 4 with the same task. Use exponential backoff to avoid hammering a rate-limited API.
  3. If max retries exhausted, decide whether the task can proceed with partial results (degraded output) or must fail entirely. This is a product decision, not just an engineering one.
  4. Never retry all workers because one failed — that would discard correct results and multiply costs.

Q5. What makes the Swarm pattern fundamentally harder to debug than the Supervisor/Worker pattern?

Emergent coordination and non-deterministic ordering. In a supervisor/worker system, every delegation decision is made by the supervisor and is logged with clear attribution. In a swarm, agents self-coordinate through a shared blackboard — there is no central authority recording who decided what and why. When the swarm produces an incorrect result, determining which agent made the bad decision that cascaded into the final output requires replaying the blackboard state at every moment, which is expensive. Additionally, swarms are sensitive to the order in which agents read and write the blackboard, making bugs difficult to reproduce reliably.

Checkpoint Summary

  • Justify first: Multi-agent adds genuine value only for tasks exceeding single-agent context limits, requiring true parallelism, or benefiting from specialization. The coordination tax is real — quantify it before proceeding.
  • Supervisor/Worker: Supervisor plans and aggregates; workers are stateless, focused, and return structured JSON. Error handling lives in the supervisor: detect failures, retry with backoff, handle partial results gracefully.
  • Shared context protocols: Choose based on data size and coupling — filesystem for large artifacts, message passing for clear provenance, shared memory for small frequently-read state, context inheritance only when children truly need full parent history.
  • Fan-out/Fan-in: Partition work carefully; use a semaphore to cap concurrency; aggregate after all workers complete (or after retries exhaust); never discard partial results from successful workers when one worker fails.
  • Conflict resolution: Prefer partitioning at planning time. For shared resources, use optimistic concurrency (version check + retry) as the default; pessimistic locks only when correctness demand is absolute.
  • Agent lifecycle: Every agent is a resource. Enforce token budgets, wall-clock timeouts, and tool call limits. Track state transitions and clean up agents on completion — no leaks.
  • Cost budgeting: Model costs before building. Shared context is duplicated N times — keep it minimal. Allocate budgets conservatively with a reserve pool for retries. Monitor and redistribute dynamically.
  • Architecture patterns: Pipeline for sequential transforms, Map-Reduce for independent shards, Swarm for open-ended exploration, Hierarchical for massive scale with domain separation.
