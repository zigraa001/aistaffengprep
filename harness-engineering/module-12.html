Why Testing Agent Systems is Hard

The Unique Challenges of Testing Agent Harnesses ▼

Testing traditional software is hard. Testing agent harnesses is harder in several dimensions simultaneously. Understanding the root causes is the prerequisite for designing a test strategy that actually works.

Challenge 1 — Non-Determinism

The same prompt sent to a model twice will produce different outputs. Temperature, top-p sampling, and server-side load all introduce variance. Standard assertion-based testing breaks: assert output == expected fails on the second run.

Challenge 2 — Multi-Step Interactions

An agent loop might execute 10–30 tool calls before completing a task. Errors compound: a wrong file read in step 3 poisons the context window for steps 4–30. Testing individual steps is necessary but not sufficient.

Challenge 3 — External Dependencies

Real agents depend on filesystem state, network APIs, database contents, and model APIs simultaneously. Each creates a new axis of flakiness. A test that worked yesterday fails today because a remote API changed its response schema.

Challenge 4 — Cost of Real API Calls

A single E2E agent task might cost $0.05–$2.00 in model API fees. A CI suite with 500 tests running on every commit could burn thousands of dollars per week. Cost is a first-class testing constraint, not an afterthought.

The Testing Paradox

The most rigorous way to test an agent harness is to run it against the real model and check whether it completes real tasks correctly. But you need a model to judge whether the output is "correct" — you need the model to test the model. This circularity forces a layered strategy: mock at low levels, use model-as-judge sparingly and on budget at high levels.

How Industry Teams Handle This

  • Anthropic / Claude Code: Separates harness logic tests (deterministic, run on every commit) from capability evals (model-in-the-loop, run nightly with budget caps).
  • Cursor: Records real IDE interaction sessions and replays them with mocked model responses for regression; uses a small golden set of real-model tasks run weekly.
  • Devin / SWE-bench: Maintains a curated set of 300 real GitHub issues as the eval set; runs against real models on a weekly cadence with cost dashboards.

Mock Model Interfaces

Mocking LLM Responses for Deterministic Testing ▼

The single most impactful technique for harness testing is decoupling your harness logic from the live model API. This is not optional — it is the foundation that makes all other tests feasible.

Analogy

Mocking a model client is the same as mocking a database in application code. You don't run real SQL in unit tests; you inject a mock repository. The harness should never know — or care — whether it is talking to a real model or a fixture.

Three Mocking Strategies

  • 1
    Static Fixture Mocks — Hardcode expected responses per test. Use when you need exact control over what the "model" says. Fast, zero cost, but brittle if the harness code changes the prompt shape.
  • 2
    Record-Replay (VCR / Cassettes) — Run once against the real model and record the exact HTTP response. Subsequent test runs replay the cassette. Captures real model behaviour at a point in time without ongoing cost.
  • 3
    Scripted Mock Sequences — Provide a queue of responses consumed in order. Useful for simulating multi-turn dialogues: response[0] triggers a tool call, response[1] processes the tool result, response[2] produces the final answer.
Python — MockModelClient with fixture queues
from dataclasses import dataclass, field
from typing import Iterator
import json


@dataclass
class MockToolUse:
    tool_name: str
    tool_input: dict
    tool_use_id: str = "tu_mock_001"


@dataclass
class MockMessage:
    """Simulates an Anthropic-style model response."""
    text: str | None = None
    tool_uses: list[MockToolUse] = field(default_factory=list)
    stop_reason: str = "end_turn"
    input_tokens: int = 100
    output_tokens: int = 50

    def is_tool_call(self) -> bool:
        return len(self.tool_uses) > 0


class MockModelClient:
    """
    Drop-in replacement for a real model client.
    Consumes responses from a pre-loaded queue; raises if exhausted.
    """

    def __init__(self, responses: list[MockMessage]):
        self._queue = list(responses)
        self.call_log: list[dict] = []  # For assertions in tests

    def create_message(
        self,
        messages: list[dict],
        tools: list[dict] | None = None,
        system: str | None = None,
        **kwargs,
    ) -> MockMessage:
        if not self._queue:
            raise RuntimeError(
                f"MockModelClient exhausted. Received unexpected call with "
                f"{len(messages)} messages."
            )
        self.call_log.append({
            "messages": messages,
            "tools": tools,
            "system": system,
        })
        return self._queue.pop(0)

    def stream_message(
        self,
        messages: list[dict],
        **kwargs,
    ) -> Iterator[str]:
        """Simulate streaming by yielding the text word by word."""
        msg = self.create_message(messages, **kwargs)
        if msg.text:
            for word in msg.text.split():
                yield word + " "

    def assert_called_with_tool(self, tool_name: str) -> None:
        """Convenience assertion: did any call include the given tool schema?"""
        for call in self.call_log:
            tools = call.get("tools") or []
            if any(t.get("name") == tool_name for t in tools):
                return
        raise AssertionError(f"Model was never offered tool '{tool_name}'")

    def assert_call_count(self, expected: int) -> None:
        actual = len(self.call_log)
        assert actual == expected, (
            f"Expected {expected} model calls, got {actual}"
        )

TypeScript — MockModelClient with async generator streaming
interface ToolUse {
  type: "tool_use";
  id: string;
  name: string;
  input: Record<string, unknown>;
}

interface MockResponse {
  stopReason: "end_turn" | "tool_use";
  text?: string;
  toolUses?: ToolUse[];
  usage?: { inputTokens: number; outputTokens: number };
}

export class MockModelClient {
  private queue: MockResponse[];
  public callLog: { messages: unknown[]; tools?: unknown[] }[] = [];

  constructor(responses: MockResponse[]) {
    this.queue = [...responses];
  }

  async createMessage(params: {
    messages: unknown[];
    tools?: unknown[];
    system?: string;
  }): Promise<MockResponse> {
    if (this.queue.length === 0) {
      throw new Error(
        `MockModelClient exhausted — unexpected call with ` +
        `${params.messages.length} messages`
      );
    }
    this.callLog.push({ messages: params.messages, tools: params.tools });
    return this.queue.shift()!;
  }

  async *streamMessage(params: {
    messages: unknown[];
  }): AsyncGenerator<string> {
    const response = await this.createMessage(params);
    const text = response.text ?? "";
    for (const word of text.split(" ")) {
      yield word + " ";
      // Simulate realistic chunk latency without sleeping real time
    }
  }

  assertCallCount(expected: number): void {
    if (this.callLog.length !== expected) {
      throw new Error(
        `Expected ${expected} model calls, got ${this.callLog.length}`
      );
    }
  }
}

import pytest
from pathlib import Path
from my_harness.tools import ReadFileTool, ToolPermissionError


@pytest.fixture
def sandbox(tmp_path: Path) -> Path:

    result = tool.execute(path="src/main.py")

    assert result.lines == 1

    assert result.error is not None
    assert "not found" in result.error.lower()

Layer 2 — Integration Tests

Integration tests exercise the interaction between the harness orchestrator and its tools, with the model client providing scripted multi-turn responses.

Python — Integration test: model requests tool, harness executes it
from my_harness import AgentHarness
from my_harness.tools import ReadFileTool, WriteFileTool
from tests.mocks import MockModelClient, MockMessage, MockToolUse


def test_harness_executes_tool_and_returns_result_to_model(tmp_path):
    """
    Verify that:
    1. Harness passes tool result back to model as a tool_result message.
    2. Harness stops after model returns end_turn.
    3. Tool was actually executed (file exists on disk).
    """
    (tmp_path / "notes.txt").write_text("Deploy at 17:00 UTC")

    mock = MockModelClient([
        MockMessage(
            tool_uses=[MockToolUse(
                tool_name="read_file",
                tool_input={"path": "notes.txt"},
                tool_use_id="tu_001",
            )],
            stop_reason="tool_use",
        ),
        MockMessage(
            text="The deployment is scheduled for 17:00 UTC.",
            stop_reason="end_turn",
        ),
    ])

    harness = AgentHarness(
        model_client=mock,
        tools=[ReadFileTool(sandbox_root=tmp_path)],
    )
    result = harness.run("When is the deployment?")

    # Model was called twice (plan + summarise after tool result)
    mock.assert_call_count(2)

    # Second call must include tool_result message
    second_call_messages = mock.call_log[1]["messages"]
    tool_result_msgs = [
        m for m in second_call_messages if m.get("role") == "user"
        and isinstance(m.get("content"), list)
        and any(c.get("type") == "tool_result" for c in m["content"])
    ]
    assert len(tool_result_msgs) == 1

    assert "17:00 UTC" in result.final_answer

Layer 3 — E2E Tests (Recorded Sessions)

E2E tests replay complete recorded sessions — the full sequence of model calls and tool executions — to detect regressions in end-to-end behaviour without live API costs.

Python — E2E test using a recorded session JSON
import json
import pytest
from pathlib import Path
from my_harness import AgentHarness, load_tools_from_config
from tests.mocks import MockModelClient, MockMessage, MockToolUse


def load_session(name: str) -> list[MockMessage]:
    """Load a recorded session from tests/sessions/{name}.json"""
    path = Path(__file__).parent / "sessions" / f"{name}.json"
    data = json.loads(path.read_text())
    messages = []
    for step in data["steps"]:
        if step["type"] == "tool_use":
            messages.append(MockMessage(
                tool_uses=[MockToolUse(**tu) for tu in step["tool_uses"]],
                stop_reason="tool_use",
                input_tokens=step.get("input_tokens", 0),
                output_tokens=step.get("output_tokens", 0),
            ))
        else:
            messages.append(MockMessage(
                text=step["text"],
                stop_reason=step.get("stop_reason", "end_turn"),
            ))
    return messages


@pytest.mark.parametrize("session_name,user_input,expected_substring", [
    ("add_logging", "Add logging to src/server.py", "logging"),
    ("refactor_auth", "Refactor the auth module to use JWT", "JWT"),
    ("write_tests",  "Write pytest tests for utils.py",   "def test_"),
])
def test_e2e_session_replay(tmp_path, session_name, user_input, expected_substring):
    mock = MockModelClient(responses=load_session(session_name))
    harness = AgentHarness(
        model_client=mock,
        tools=load_tools_from_config("tests/fixtures/tools.yaml", sandbox=tmp_path),
    )
    result = harness.run(user_input)
    assert expected_substring in result.final_answer, (
        f"Expected '{expected_substring}' in answer for session '{session_name}'"
    )

Snapshot Testing for Tool Schemas

Tool Schemas Are Contracts — Guard Them with Snapshots ▼
Key Insight

A tool schema is a contract between your harness and the model. If you rename a parameter from file_path to path, the model's ability to use that tool correctly can silently break — it learned from training that "read_file" has a parameter named file_path. Snapshot tests catch schema drift before it reaches production.

What Can Break When Schemas Change

  • Parameter renames: Model passes old name, tool receives nothing or raises an error.
  • Type changes: Model passes a string, schema now expects an array.
  • Required → optional: Model stops including a parameter it believes is optional; tool crashes on None.
  • Description changes: Model's understanding of when to use the tool shifts. Silent quality regression.
  • Tool name changes: Model won't call a tool it doesn't know by name.
Python — Snapshot test for tool JSON Schema
"""
Snapshot tests for tool schemas.

Usage:
  Update snapshots: pytest --snapshot-update
  Assert no drift:  pytest (normal CI run)
"""
import json
import pytest
from pathlib import Path
from my_harness.tools import (
    ReadFileTool, WriteFileTool, RunCommandTool,
    SearchFilesTool, ListDirectoryTool,
)

SNAPSHOT_DIR = Path(__file__).parent / "snapshots" / "tool_schemas"

ALL_TOOLS = [
    ReadFileTool,
    WriteFileTool,
    RunCommandTool,
    SearchFilesTool,
    ListDirectoryTool,
]


@pytest.mark.parametrize("ToolClass", ALL_TOOLS, ids=lambda t: t.__name__)
def test_tool_schema_snapshot(ToolClass, snapshot_update: bool):
    tool = ToolClass.__new__(ToolClass)  # no sandbox needed for schema export
    schema = tool.to_api_schema()  # returns Anthropic tool-use dict
    schema_json = json.dumps(schema, indent=2, sort_keys=True)

    snapshot_path = SNAPSHOT_DIR / f"{ToolClass.__name__}.json"

    if snapshot_update:
        SNAPSHOT_DIR.mkdir(parents=True, exist_ok=True)
        snapshot_path.write_text(schema_json)
        pytest.skip(f"Snapshot updated for {ToolClass.__name__}")
    else:
        assert snapshot_path.exists(), (
            f"No snapshot for {ToolClass.__name__}. "
            f"Run with --snapshot-update to create it."
        )
        expected = snapshot_path.read_text()
        assert schema_json == expected, (
            f"Tool schema for {ToolClass.__name__} has drifted!\n\n"
            f"If this change is intentional, run: pytest --snapshot-update\n"
            f"Then review the diff and update the model system prompt if needed."
        )


def pytest_addoption(parser):
    parser.addoption(
        "--snapshot-update",
        action="store_true",
        default=False,
        help="Overwrite snapshot files with current output",
    )


@pytest.fixture
def snapshot_update(request):
    return request.config.getoption("--snapshot-update")

Backward Compatibility Strategy

When you must change a schema, use a compatibility shim to support both the old and new parameter names while the model re-learns the new schema through prompt updates:

Python — Backward-compatible parameter aliasing
class ReadFileTool:
    """
    v2 schema uses 'path'. v1 used 'file_path'.
    We accept both during the transition window.
    """

    SCHEMA_VERSION = "2.0"

    def to_api_schema(self) -> dict:
        return {
            "name": "read_file",
            "description": "Read the contents of a file at the given path.",
            "input_schema": {
                "type": "object",
                "properties": {
                    "path": {
                        "type": "string",
                        "description": "Absolute or sandbox-relative file path to read.",
                    },
                    # Deprecated alias — remove in v3
                    "file_path": {
                        "type": "string",
                        "description": "[DEPRECATED] Use 'path' instead.",
                    },
                },
                "required": ["path"],
            },
        }

    def execute(self, path: str | None = None, file_path: str | None = None) -> dict:
        """Accept both old and new parameter names."""
        resolved_path = path or file_path
        if resolved_path is None:
            raise ValueError("Either 'path' or 'file_path' must be provided")
        # ... rest of implementation


class SchemaVersionGuard:
    """
    Fail fast if a deployed schema version is older than minimum supported.
    Prevents silent model-schema mismatches in production.
    """

    MINIMUM_SCHEMA_VERSION = "2.0"

    def validate(self, tools: list) -> None:
        from packaging.version import Version
        min_v = Version(self.MINIMUM_SCHEMA_VERSION)
        for tool in tools:
            v = Version(getattr(tool, "SCHEMA_VERSION", "1.0"))
            if v < min_v:
                raise RuntimeError(
                    f"{tool.__class__.__name__} has schema version {v}, "
                    f"minimum required is {min_v}"
                )

Cost-Aware CI/CD

Running Tests Without Bankrupting Your API Budget ▼
Cost Estimate
500 tests × $0.10 avg cost (claude-opus, long context) = $50 per CI run
10 engineers × 5 pushes/day × $50 = $2,500/day = $50,000/month

With mocking: 500 tests × $0 (mock) = $0 per CI run (unit + integration)
Nightly real-model eval: 50 golden tasks × $0.50 = $25/night = $750/month

The Five-Strategy Framework

Strategy When Applied Cost Savings Trade-off
Mock in CI, real model nightly Unit + integration tests ~98% 24-hour lag on real-model regressions
Response caching across runs Cassette replay (same prompts) ~80% Cache invalidation complexity
Budget cap enforcement Real-model eval suites Hard stop Tests may be skipped when budget exceeded
Selective test execution Changed-file gating ~60% Misses indirect regressions
Cheaper model for CI smoke tests Real-model sanity checks ~90% vs flagship Haiku ≠ Sonnet behaviour

class EvalBudgetTracker:
    budget_usd: float
    spent_usd: float = 0.0
    skipped: int = 0
    completed: int = 0
    log: list[dict] = field(default_factory=list)

    def can_afford(self, estimated_cost: float) -> bool:
        return (self.spent_usd + estimated_cost) <= self.budget_usd

    def record(self, task_id: str, cost: float, passed: bool) -> None:
        self.spent_usd += cost
        self.completed += 1
        self.log.append({"task": task_id, "cost": cost, "passed": passed})

    def skip(self, task_id: str, reason: str) -> None:
        self.skipped += 1
        self.log.append({"task": task_id, "skipped": True, "reason": reason})

    @property
    def pass_rate(self) -> float:
        passed = sum(1 for e in self.log if e.get("passed"))
        total = self.completed
        return passed / total if total > 0 else 0.0


def run_eval_suite(suite_path: str, budget_usd: float, model: str) -> EvalBudgetTracker:
    tasks = json.loads(Path(suite_path).read_text())
    tracker = EvalBudgetTracker(budget_usd=budget_usd)
    harness = AgentHarness(model=model)

    for task in tasks:
        est = estimate_cost_usd(
            prompt_tokens=task.get("estimated_prompt_tokens", 2000),
            max_output_tokens=task.get("max_output_tokens", 1000),
            model=model,
        )
        if not tracker.can_afford(est):
            tracker.skip(task["id"], reason=f"Budget exhausted (spent ${tracker.spent_usd:.2f})")
            print(f"  SKIP {task['id']} — budget at ${tracker.spent_usd:.2f}/{budget_usd}")
            continue

        try:
            result = harness.run(task["input"])
            passed = task["expected_substring"].lower() in result.final_answer.lower()
            actual_cost = estimate_cost_usd(
                result.total_input_tokens, result.total_output_tokens, model
            )
            tracker.record(task["id"], actual_cost, passed)
            status = "PASS" if passed else "FAIL"
            print(f"  {status} {task['id']} (${actual_cost:.4f})")
        except Exception as e:
            tracker.record(task["id"], 0, False)
            print(f"  ERROR {task['id']}: {e}")

    return tracker

Eval Frameworks for Harness Quality

Measuring What Actually Matters in Production ▼

Unit and integration tests tell you "the code is correct." Evals tell you "the harness is useful." These are different questions. A harness can pass every unit test and still fail 40% of real user tasks.

Task Completion Rate (TCR)

Did the agent successfully complete the user's request? TCR is the primary business metric. Target: >85% for production launch. Measure with a golden eval set of 50–200 representative tasks.

Tool Selection Accuracy (TSA)

Did the model choose the right tool for each step? TSA catches poor tool descriptions or schemas that cause the model to use suboptimal tools. Measure per-step with human-labelled ground truth.

Context Utilisation Efficiency (CUE)

What fraction of context tokens actually contributed to the final answer? High CUE (close to 1.0) means you're reading only what the model uses. Low CUE (<0.3) means your context builder is retrieving irrelevant content.

Cost per Successful Task (CpST)

Total model API cost divided by successful completions. A harness with 90% TCR at $2.00/task may be worse than 80% TCR at $0.05/task depending on the business context. Always track both dimensions.

Python — HarnessEvaluator with all four metrics
from dataclasses import dataclass, field
from typing import Protocol
import json
import time


@dataclass
class EvalTask:
    id: str
    user_input: str
    expected_substring: str          # Simple ground truth check
    expected_tools: list[str] = field(default_factory=list)  # For TSA
    expected_context_keys: list[str] = field(default_factory=list)  # For CUE


@dataclass
class EvalResult:
    task_id: str
    final_answer: str
    tools_used: list[str]
    context_keys_used: list[str]
    input_tokens: int
    output_tokens: int
    latency_ms: float
    error: str | None = None

    def task_completed(self, task: EvalTask) -> bool:
        if self.error:
            return False
        return task.expected_substring.lower() in self.final_answer.lower()

    def tool_selection_accuracy(self, task: EvalTask) -> float:
        if not task.expected_tools:
            return 1.0
        correct = set(self.tools_used) & set(task.expected_tools)
        return len(correct) / len(task.expected_tools)

    def context_utilisation_efficiency(self, task: EvalTask) -> float:
        if not task.expected_context_keys:
            return 1.0
        used = set(self.context_keys_used) & set(task.expected_context_keys)
        total_retrieved = max(len(self.context_keys_used), 1)
        return len(used) / total_retrieved

    def cost_usd(self, model: str) -> float:
        from my_harness.cost import estimate_cost_usd
        return estimate_cost_usd(self.input_tokens, self.output_tokens, model)


class HarnessEvaluator:
    def __init__(self, harness, model: str):
        self.harness = harness
        self.model = model

    def run_task(self, task: EvalTask) -> EvalResult:
        start = time.perf_counter()
        try:
            result = self.harness.run(task.user_input)
            return EvalResult(
                task_id=task.id,
                final_answer=result.final_answer,
                tools_used=[tc.tool_name for tc in result.tool_calls],
                context_keys_used=result.context_keys_used,
                input_tokens=result.total_input_tokens,
                output_tokens=result.total_output_tokens,
                latency_ms=(time.perf_counter() - start) * 1000,
            )
        except Exception as e:
            return EvalResult(
                task_id=task.id,
                final_answer="",
                tools_used=[],
                context_keys_used=[],
                input_tokens=0,
                output_tokens=0,
                latency_ms=(time.perf_counter() - start) * 1000,
                error=str(e),
            )

    def evaluate(self, tasks: list[EvalTask]) -> dict:
        results = [self.run_task(t) for t in tasks]

        tcr = sum(r.task_completed(t) for r, t in zip(results, tasks)) / len(tasks)
        tsa = sum(r.tool_selection_accuracy(t) for r, t in zip(results, tasks)) / len(tasks)
        cue = sum(r.context_utilisation_efficiency(t) for r, t in zip(results, tasks)) / len(tasks)
        total_cost = sum(r.cost_usd(self.model) for r in results)
        successful = sum(r.task_completed(t) for r, t in zip(results, tasks))
        cpst = total_cost / max(successful, 1)
        avg_latency = sum(r.latency_ms for r in results) / len(results)

        return {
            "task_completion_rate": round(tcr, 4),
            "tool_selection_accuracy": round(tsa, 4),
            "context_utilisation_efficiency": round(cue, 4),
            "cost_per_successful_task_usd": round(cpst, 6),
            "total_cost_usd": round(total_cost, 4),
            "avg_latency_ms": round(avg_latency, 1),
            "n_tasks": len(tasks),
            "n_successful": successful,
            "per_task": [
                {
                    "id": r.task_id,
                    "passed": r.task_completed(t),
                    "tsa": r.tool_selection_accuracy(t),
                    "cost": r.cost_usd(self.model),
                    "error": r.error,
                }
                for r, t in zip(results, tasks)
            ],
        }

Building a Golden Eval Dataset

  • 1
    Seed from real user sessions. Export 200 real sessions from production logs. Strip PII. Select a stratified sample across task types (read, write, refactor, debug).
  • 2
    Add human-labelled ground truth. An engineer annotates each session with: (a) did it complete the task? (b) which tools should have been used? (c) was the answer correct?
  • 3
    Include adversarial cases. Add tasks designed to probe known failure modes: path traversal attempts, infinite tool loops, tasks with ambiguous intent.
  • 4
    Version-control the dataset. The eval dataset is code. Changes to it require the same review process as changes to production code.
  • 5
    Refresh quarterly. User task distributions shift. Retire tasks the agent solves 100% and add new tasks from recent sessions to maintain signal.

Regression Testing for Agent Behavior

Detecting Behavioral Regressions Before Users Do ▼

Behavioral regression in agents is subtle. A change to a system prompt, a tool description, or the context assembly order can silently shift which tools get called, how often, and with what arguments — without changing the final answer for easy tasks. Regression suites catch these shifts.

Analogy

Traditional regression testing uses exact string matching. Agent regression testing needs fuzzy matching because the model may phrase the same correct answer in different words each time. The challenge is: how fuzzy is too fuzzy? A match too strict creates false positives; a match too loose misses real regressions.

Three Matching Strategies

Strategy How It Works Precision Use For
Substring containment Check if expected keyword/phrase appears in answer Low Fast smoke tests, golden task pass/fail
Semantic similarity (embedding cosine) Compare answer embedding to reference embedding Medium Detecting paraphrase vs meaning drift
Model-as-judge Ask a separate model "Does answer A match answer B in intent?" High High-value tasks, release gates
Python — RegressionTestSuite with multi-strategy matching
from dataclasses import dataclass
from enum import Enum
import json
from pathlib import Path


class MatchStrategy(str, Enum):
    SUBSTRING = "substring"
    SEMANTIC = "semantic"
    MODEL_JUDGE = "model_judge"


@dataclass
class GoldenExample:
    id: str
    user_input: str
    reference_answer: str
    match_strategy: MatchStrategy = MatchStrategy.SUBSTRING
    similarity_threshold: float = 0.85
    expected_tools: list[str] = None


@dataclass
class RegressionResult:
    example_id: str
    passed: bool
    actual_answer: str
    similarity_score: float | None
    judge_verdict: str | None


class RegressionTestSuite:
    def __init__(
        self,
        harness,
        golden_path: str,
        embedding_client=None,
        judge_client=None,
    ):
        self.harness = harness
        self.embedding_client = embedding_client
        self.judge_client = judge_client
        self.golden: list[GoldenExample] = self._load(golden_path)

    def _load(self, path: str) -> list[GoldenExample]:
        data = json.loads(Path(path).read_text())
        return [GoldenExample(**ex) for ex in data]

    def _match_substring(self, actual: str, reference: str) -> tuple[bool, float | None]:
        # Check all key phrases from reference appear in actual
        phrases = [p.strip() for p in reference.split(",") if len(p.strip()) > 4]
        hits = sum(1 for p in phrases if p.lower() in actual.lower())
        score = hits / max(len(phrases), 1)
        return score >= 0.5, score

    def _match_semantic(
        self, actual: str, reference: str, threshold: float
    ) -> tuple[bool, float]:
        import numpy as np

        def embed(text: str) -> list[float]:
            return self.embedding_client.embed(text)

        vec_a = np.array(embed(actual))
        vec_r = np.array(embed(reference))
        cosine = float(
            np.dot(vec_a, vec_r) / (np.linalg.norm(vec_a) * np.linalg.norm(vec_r))
        )
        return cosine >= threshold, cosine

    def _match_model_judge(
        self, actual: str, reference: str
    ) -> tuple[bool, str]:
        prompt = (
            f"Reference answer: {reference}\n\n"
            f"Candidate answer: {actual}\n\n"
            "Does the candidate answer convey the same essential information as the "
            "reference? Reply with PASS or FAIL followed by one sentence of reasoning."
        )
        verdict = self.judge_client.complete(prompt, max_tokens=100)
        passed = verdict.strip().upper().startswith("PASS")
        return passed, verdict

    def run(self, example: GoldenExample) -> RegressionResult:
        result = self.harness.run(example.user_input)
        actual = result.final_answer

        if example.match_strategy == MatchStrategy.SUBSTRING:
            passed, score = self._match_substring(actual, example.reference_answer)
            return RegressionResult(example.id, passed, actual, score, None)

        elif example.match_strategy == MatchStrategy.SEMANTIC:
            passed, score = self._match_semantic(
                actual, example.reference_answer, example.similarity_threshold
            )
            return RegressionResult(example.id, passed, actual, score, None)

        elif example.match_strategy == MatchStrategy.MODEL_JUDGE:
            passed, verdict = self._match_model_judge(actual, example.reference_answer)
            return RegressionResult(example.id, passed, actual, None, verdict)

        raise ValueError(f"Unknown strategy: {example.match_strategy}")

    def run_all(self) -> dict:
        results = [self.run(ex) for ex in self.golden]
        n_pass = sum(r.passed for r in results)
        return {
            "pass_rate": n_pass / len(results),
            "n_pass": n_pass,
            "n_fail": len(results) - n_pass,
            "failures": [
                {"id": r.example_id, "actual": r.actual_answer[:200]}
                for r in results if not r.passed
            ],
        }

A/B Testing Agent Versions in Production

When deploying a new harness version, A/B testing lets you compare the new behaviour against baseline using real traffic before a full rollout:

Python — Shadow mode A/B harness wrapper
import threading
from dataclasses import dataclass


@dataclass
class ABResult:
    control_answer: str
    treatment_answer: str
    control_cost: float
    treatment_cost: float
    control_latency_ms: float
    treatment_latency_ms: float


class ShadowHarness:
    """
    Runs control harness normally and treatment harness in shadow mode.
    Returns control result to the user; records both for comparison.
    Useful for A/B testing without exposing users to treatment failures.
    """

    def __init__(self, control, treatment, results_store):
        self.control = control
        self.treatment = treatment
        self.store = results_store

    def run(self, user_input: str):
        import time

        # Control runs synchronously (user sees this result)
        t0 = time.perf_counter()
        control_result = self.control.run(user_input)
        control_latency = (time.perf_counter() - t0) * 1000

        # Treatment runs in a background thread (shadow)
        def run_treatment():
            t1 = time.perf_counter()
            try:
                treatment_result = self.treatment.run(user_input)
                self.store.record(ABResult(
                    control_answer=control_result.final_answer,
                    treatment_answer=treatment_result.final_answer,
                    control_cost=control_result.cost_usd,
                    treatment_cost=treatment_result.cost_usd,
                    control_latency_ms=control_latency,
                    treatment_latency_ms=(time.perf_counter() - t1) * 1000,
                ))
            except Exception as e:
                self.store.record_error(user_input, str(e))

        thread = threading.Thread(target=run_treatment, daemon=True)
        thread.start()

        return control_result  # User never waits for treatment

Interview Tips

What Senior Interviewers Look For

Strong Signals

  • Immediately separates "testing harness logic" from "testing model capability" — treats them as different problems
  • Mentions cost as a first-class constraint without being prompted
  • Understands why tool schemas are contracts and that schema drift silently breaks the model
  • Proposes a layered strategy: mock unit → mock integration → replayed E2E → nightly real model
  • Can articulate the testing paradox (model to test model) and a practical workaround
  • Distinguishes TCR (did task complete?) from TSA (were right tools used?) from CpST (at what cost?)

Red Flags

  • "Just assert the output equals the expected string" — ignores non-determinism
  • No mention of cost implications of running real models in CI
  • Conflating unit tests (deterministic tool logic) with evals (model quality)
  • No strategy for schema versioning or backward compatibility
  • Proposing only E2E tests with real models as the testing strategy
  • Treating model quality as binary (pass/fail) without acknowledging degrees of correctness

Common Follow-Up Questions

  • "How do you decide when to update a cassette vs investigate a failure?" — Answer: if the model's API changes (new stop_reason field) you update cassettes; if a tool schema test fails, you investigate schema drift before updating.
  • "How do you test multi-agent harnesses?" — Answer: mock each sub-agent separately; integration-test agent-to-agent message passing with fixture responses; E2E with full recorded orchestration.
  • "How would you set an eval budget for a startup vs a large company?" — Answer: startup might spend $100/month on nightly evals with a 50-task golden set; large company might spend $10k/month with 1,000-task set and per-feature regression baselines.
  • "What happens when you change the system prompt?" — Answer: treat it as a schema change; run the full golden eval set before and after; gate deployment on TCR ≥ baseline − 2%.

Knowledge Check

Q1. You have 500 harness tests. Running them all against the real model API would cost ~$50 per CI run. What is the correct architecture?

Mock the model client at unit and integration test level (zero cost, runs on every commit). Use recorded cassettes for E2E replay (zero marginal cost). Run a curated golden set against the real model only on a nightly or weekly schedule with a hard budget cap. The $50/run problem is caused by using the real API for all tests — not by having 500 tests.

Q2. A developer renames the ReadFileTool parameter from file_path to path and all unit tests still pass. What test would catch this regression before it reaches production?

A snapshot test for the tool's JSON Schema. The snapshot records the exact schema at a point in time. When the parameter name changes, the snapshot diff immediately flags it — even if all functional tests pass, because functional tests use the mock client which doesn't care about parameter names. The snapshot forces a conscious review of any schema change before it can be deployed.

Q3. What is the "testing paradox" in agent harness testing, and what is the practical resolution?

The paradox: the most rigorous way to evaluate an agent is to check whether its outputs are "correct" — but evaluating correctness for open-ended tasks requires intelligence similar to the model being tested. You need the model to evaluate the model. The practical resolution is a layered approach: (1) For structured outputs, use deterministic assertions (substring, schema validation). (2) For semantic correctness, use embedding cosine similarity as a middle ground. (3) For high-value complex tasks, use a separate model-as-judge (typically a larger, cheaper-to-call model). (4) For ground truth, use human annotation on a small sample quarterly.

Q4. Distinguish Task Completion Rate (TCR), Tool Selection Accuracy (TSA), and Cost per Successful Task (CpST). Why do you need all three?

TCR measures the end-user outcome: did the agent complete what the user asked? TSA measures internal quality: did the agent pick the right tool at each step? CpST measures economic sustainability: is the harness viable at scale? You need all three because they can move independently. A harness might have high TCR (lucky final answers) but low TSA (using 5 tools when 1 would do) and high CpST (burning $5/task when the business can afford $0.10). Conversely, a harness with perfect TSA can still have low TCR if the tools return insufficient context.

Q5. What is shadow mode A/B testing and when should you use it for a harness upgrade?

Shadow mode runs the new (treatment) harness in parallel with the existing (control) harness on real production traffic, but only shows users the control result. Treatment results are recorded for offline comparison. Use it when: you have changed a system prompt or tool description that should improve quality; you want to measure the real-traffic improvement in TCR/CpST before committing to a full rollout; or when the regression test suite cannot fully capture the diversity of real user inputs. It is safer than a live A/B test (users never see treatment failures) at the cost of doubled inference spend during the shadow period.

Checkpoint Summary — Module 12: Testing & Evaluation

Key Concepts
  • Agent testing has five unique challenges: non-determinism, multi-step compounding, external dependencies, API cost, and the testing paradox.
  • Mock model clients are the single most important enabler of cost-free, deterministic testing — inject them via dependency injection, not monkey-patching.
  • The test pyramid maps to: unit (tools/permissions/context), integration (tool chains with mocked model), E2E (recorded session replay), real-model evals (nightly, budgeted).
  • Tool schemas are contracts. Snapshot tests guard against silent schema drift that breaks the model's ability to use your tools.
Patterns & Strategies
  • VCR cassette recording: capture real model responses once, replay forever at zero cost.
  • Cost-aware CI: mock in CI, real model nightly, hard budget cap on eval suites.
  • Four eval metrics: TCR (outcomes), TSA (tool quality), CUE (context efficiency), CpST (economics).
  • Regression matching: substring for smoke tests, semantic similarity for quality, model-as-judge for release gates.
  • Shadow mode A/B: test harness upgrades on real traffic without exposing users to treatment failures.
One-Sentence Summary: Test agent harnesses by mocking aggressively at unit and integration level to achieve zero-cost CI coverage, then use cost-budgeted real-model evals nightly to catch the capability regressions that mocks cannot.
