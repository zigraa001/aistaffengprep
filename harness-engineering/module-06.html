Human-in-the-Loop Patterns

Design approval workflows, trust escalation, interrupt/resume flows, work-in-progress visibility, and undo systems so AI agents remain safe and controllable even as they grow more capable.

What You Will Learn

  • Why HITL is a product requirement, not just a safety afterthought
  • The trust continuum from fully manual to fully autonomous
  • Pre-execution, post-execution, and batch approval workflows
  • Progressive trust escalation with action classification
  • Graceful interrupt and resume without losing agent state
  • Streaming work-in-progress visibility to the user
  • Undo and rollback for every class of agent action
  • Production design decisions behind Claude Code and Cursor

Why Human-in-the-Loop Matters

AI Agents Aren't Perfect — Humans Need Control

An AI agent operating without any human oversight is like giving a junior contractor unescorted access to your production database. Even if the model is highly capable, it can:

  • Misinterpret intent — "clean up the logs folder" interpreted as "delete everything in /var/log"
  • Cascade mistakes — one bad file edit triggers a chain of build failures across a monorepo
  • Exceed scope — drift from "fix the bug" to "refactor the whole module" while still being technically correct
  • Be exploited via prompt injection — malicious content in files or web pages hijacking tool calls

Human-in-the-Loop (HITL) is the set of engineering patterns that keep a human in the decision loop for actions that cross a risk threshold. Done well, it makes agents safer and more trusted, not just slower.

Analogy — Surgical Checklist

The WHO Surgical Safety Checklist doesn't slow surgeries down because it makes surgeons incompetent. It catches the predictable class of mistakes that skilled people make under pressure. HITL approval workflows serve the same function for AI agents.

The Trust Continuum

Agent products exist along a spectrum. The engineering challenge is choosing the right point on this spectrum for the current context — and making that point adjustable over time.

Fully Manual
AI suggests, human executes every step
HITL Hybrid
AI acts autonomously on safe ops, asks for risky ones
Fully Autonomous
AI acts; human reviews aggregate output only

Most production agent products today live in the middle band. Claude Code, for example, prompts the user before running shell commands that delete files or make network requests, but autonomously reads files and runs non-destructive commands.

UX Impact: Interruptions vs Safety

Too many approval prompts destroy the value of automation. Too few create silent disasters. The practical design target is:

Design Target

Minimise approvals-per-task while maximising coverage of irreversible or high-blast-radius actions. Every prompt that blocks the user should carry real information content — if the user always clicks "Yes", the prompt shouldn't exist.

Approval Workflows

Pre-Execution, Post-Execution, and Batch Approval

There are three fundamental shapes for approval workflows. Each involves different tradeoffs between latency, safety, and cognitive load on the user.

1. Pre-Execution Approval (Ask Before Doing)

The agent presents the proposed action and waits for explicit permission before executing. This is the safest pattern and is used for destructive or irreversible operations. The cost is latency — the agent is blocked waiting for a human response that could take seconds to hours.

2. Post-Execution Review (Do, Then Confirm)

The agent executes and presents the result for retrospective approval. Useful for low-risk mutations (e.g., writing a new file) where showing the output is more informative than showing the plan. A rejected result triggers rollback. This keeps the agent moving without blocking on every step.

3. Batch Approval (Approve Categories of Actions)

Rather than prompting per-action, the user approves a policy: "You may read any file, write to src/, but must ask before touching config/ or running shell commands." The agent then proceeds autonomously within that approved policy envelope. This is how Claude Code's "auto-approve" settings work — users grant blanket trust to specific tool categories.

Key Insight

Batch approval is the key to breaking out of the interruption trap. It shifts the conversation from "should I do this action?" to "should I grant this class of capability?" — a much higher-leverage human decision that scales across hundreds of actions.

Design Patterns: ApprovalGate and ApprovalPolicy

📄 approval_workflow.py — ApprovalGate & ApprovalPolicy
from dataclasses import dataclass, field
from enum import Enum
from typing import Callable, Awaitable
import asyncio


class RiskLevel(Enum):
    SAFE      = "safe"
    RISKY     = "risky"
    DANGEROUS = "dangerous"


@dataclass
class Action:
    tool_name: str
    parameters: dict
    rationale: str   # Why the agent wants to take this action


@dataclass
class ApprovalDecision:
    approved: bool
    modified_action: Action | None = None  # User may edit the action
    denial_reason: str | None = None


# ── ApprovalPolicy ────────────────────────────────────────────────────────────
# A policy is a pure function: Action -> RiskLevel.
# Policies are composable: they can be stacked in a pipeline.

class ApprovalPolicy:
    """Classifies an action's risk level based on configurable rules."""

    def __init__(self):
        # Each rule is (predicate, risk_level). First match wins.
        self._rules: list[tuple[Callable[[Action], bool], RiskLevel]] = []
        self._default = RiskLevel.RISKY

    def add_rule(self, predicate: Callable[[Action], bool], risk: RiskLevel):
        self._rules.append((predicate, risk))
        return self  # fluent API

    def classify(self, action: Action) -> RiskLevel:
        for predicate, risk in self._rules:
            if predicate(action):
                return risk
        return self._default


def build_default_policy() -> ApprovalPolicy:
    policy = ApprovalPolicy()

    # Safe read-only operations
    policy.add_rule(lambda a: a.tool_name in ("read_file", "list_dir", "search_files",
                                               "run_tests", "grep"), RiskLevel.SAFE)

    # Dangerous — always require explicit confirmation
    policy.add_rule(lambda a: a.tool_name == "bash" and any(
        tok in a.parameters.get("command", "")
        for tok in ("rm -rf", "DROP TABLE", "> /dev/", "sudo")), RiskLevel.DANGEROUS)

    policy.add_rule(lambda a: a.tool_name in ("network_request", "send_email",
                                               "deploy", "publish_package"), RiskLevel.DANGEROUS)

    # Risky by default for anything that mutates state
    policy.add_rule(lambda a: a.tool_name in ("write_file", "patch_file",
                                               "bash", "git_commit"), RiskLevel.RISKY)
    return policy


# ── ApprovalGate ──────────────────────────────────────────────────────────────
# Sits between the agent loop and the executor.
# It either auto-approves, auto-denies, or delegates to a UI callback.

PromptFn = Callable[[Action, RiskLevel], Awaitable[ApprovalDecision]]


class ApprovalGate:
    def __init__(self,
                 policy: ApprovalPolicy,
                 prompt_fn: PromptFn,
                 batch_grants: set[str] | None = None):
        self.policy = policy
        self.prompt_fn = prompt_fn
        # batch_grants: tool names the user has pre-approved for this session
        self.batch_grants: set[str] = batch_grants or set()

    async def request(self, action: Action) -> ApprovalDecision:
        # Batch grants override policy for pre-approved tool names
        if action.tool_name in self.batch_grants:
            return ApprovalDecision(approved=True)

        risk = self.policy.classify(action)

        if risk == RiskLevel.SAFE:
            return ApprovalDecision(approved=True)

        # RISKY and DANGEROUS both go to the user, but the UI should
        # communicate the severity difference clearly.
        return await self.prompt_fn(action, risk)

    def grant_batch(self, tool_name: str):
        """User grants blanket approval for a tool category this session."""
        self.batch_grants.add(tool_name)


# ── Example usage in an agent loop ───────────────────────────────────────────

async def mock_prompt(action: Action, risk: RiskLevel) -> ApprovalDecision:
    """Simulates a UI prompt. In production, this streams to the terminal/UI."""
    print(f"\n[APPROVAL REQUIRED — {risk.value.upper()}]")
    print(f"  Tool: {action.tool_name}")
    print(f"  Args: {action.parameters}")
    print(f"  Why:  {action.rationale}")
    answer = input("  Approve? [y/n/e(dit)]: ").strip().lower()
    if answer == "y":
        return ApprovalDecision(approved=True)
    return ApprovalDecision(approved=False, denial_reason="User denied")


async def agent_step(gate: ApprovalGate, action: Action):
    decision = await gate.request(action)
    if decision.approved:
        effective_action = decision.modified_action or action
        print(f"Executing: {effective_action.tool_name}({effective_action.parameters})")
        # ... call executor ...
    else:
        print(f"Action blocked: {decision.denial_reason}. Replanning...")

Progressive Trust Escalation

Start Restrictive, Earn Autonomy Over Time

A newly onboarded agent should behave conservatively. As it demonstrates reliable judgment — executing safe actions correctly, asking appropriate questions, not overstepping — the harness can relax constraints and require fewer interruptions. This mirrors how you'd delegate to a new employee.

Analogy — New Employee Probation

A new employee on day one can't expense anything over $50 without approval. After six months of good judgment, that limit rises to $500. After a year, they have a corporate card with a monthly budget. HITL trust escalation follows the same logic: demonstrated reliability unlocks capability.

Action Classification

Class Examples Blast Radius Default Policy
SAFE Read file, list directory, search codebase, run read-only test Zero — no state mutation Auto-approve always
RISKY Write / patch file, git commit, run build, install package Local repo; reversible via git Prompt on first use; auto-approve after N successful runs
DANGEROUS Network request, delete files, deploy, run arbitrary shell, send email External systems; potentially irreversible Always prompt; require typed confirmation for highest tier

Escalation Triggers and Thresholds

Trust can escalate (grant more autonomy) or de-escalate (require more approval) based on signals from the session:

  • Escalate: N consecutive approvals with no denials, no rollbacks, and no user corrections
  • Escalate: User explicitly grants batch approval for a tool category
  • De-escalate: User denies an action that the agent auto-approved (indicates over-trust)
  • De-escalate: Agent triggered a rollback or undo in the last K actions
  • Hard floor: DANGEROUS actions always require human approval regardless of trust level

Session-Based vs Persistent Trust

Design Decision

Session trust resets on every conversation start. It is safer by default and simpler to reason about — trust earned in one task does not transfer to another. Persistent trust is stored in a user profile and carries over. It reduces friction for power users but requires richer audit trails and must be scoped carefully (e.g., "trust write_file in repo X" not "trust write_file everywhere").

📄 trust_escalation.py — TrustEscalationStateMachine
from dataclasses import dataclass, field
from enum import Enum
from typing import Literal


class TrustTier(Enum):
    STRICT    = 0   # All risky actions require approval
    MODERATE  = 1   # Risky auto-approved after threshold; dangerous still prompts
    RELAXED   = 2   # Only DANGEROUS class requires approval
    # Note: DANGEROUS never auto-approves — that's a hard-coded floor


@dataclass
class TrustRecord:
    """Per-tool-category trust record within a session."""
    tool_category: str
    consecutive_approvals: int = 0
    consecutive_denials: int = 0
    rollback_count: int = 0
    tier: TrustTier = TrustTier.STRICT

    # Thresholds (tune per product requirements)
    ESCALATE_AFTER   : int = field(default=5, repr=False)   # consecutive approvals
    DEESCALATE_AFTER : int = field(default=1, repr=False)   # consecutive denials


class TrustEscalationFSM:
    """
    Manages trust state across tool categories for a single session.

    State transitions:
        STRICT -> MODERATE : consecutive_approvals >= ESCALATE_AFTER
        MODERATE -> RELAXED : consecutive_approvals >= ESCALATE_AFTER * 2
        Any tier -> STRICT  : denial OR rollback
    """

    def __init__(self, initial_tier: TrustTier = TrustTier.STRICT):
        self._default_tier = initial_tier
        self._records: dict[str, TrustRecord] = {}

    def _get(self, category: str) -> TrustRecord:
        if category not in self._records:
            self._records[category] = TrustRecord(
                tool_category=category,
                tier=self._default_tier
            )
        return self._records[category]

    def record_approval(self, category: str):
        rec = self._get(category)
        rec.consecutive_approvals += 1
        rec.consecutive_denials = 0
        self._maybe_escalate(rec)

    def record_denial(self, category: str):
        rec = self._get(category)
        rec.consecutive_denials += 1
        rec.consecutive_approvals = 0
        rec.tier = TrustTier.STRICT   # Hard reset on denial

    def record_rollback(self, category: str):
        rec = self._get(category)
        rec.rollback_count += 1
        rec.tier = TrustTier.STRICT   # Rollback = signal of bad auto-approve

    def grant_batch(self, category: str):
        """User explicitly grants batch approval — jump to RELAXED immediately."""
        rec = self._get(category)
        rec.tier = TrustTier.RELAXED
        rec.consecutive_approvals = rec.ESCALATE_AFTER * 2

    def requires_approval(self, category: str, risk: RiskLevel) -> bool:
        if risk == RiskLevel.DANGEROUS:
            return True   # Hard floor — always ask
        if risk == RiskLevel.SAFE:
            return False  # Never ask
        # RISKY: depends on tier
        tier = self._get(category).tier
        return tier == TrustTier.STRICT

    def _maybe_escalate(self, rec: TrustRecord):
        if rec.tier == TrustTier.STRICT and rec.consecutive_approvals >= rec.ESCALATE_AFTER:
            rec.tier = TrustTier.MODERATE
        elif rec.tier == TrustTier.MODERATE and rec.consecutive_approvals >= rec.ESCALATE_AFTER * 2:
            rec.tier = TrustTier.RELAXED

    def summary(self) -> dict:
        return {cat: {"tier": r.tier.name, "approvals": r.consecutive_approvals}
                for cat, r in self._records.items()}

Interrupt and Resume

Graceful Mid-Task Interruption Without Losing State

A user watching an agent work will sometimes want to redirect it mid-task — "wait, use the v2 API instead of v1", or "stop, that file is read-only". An agent that can only be stopped by killing the process and restarting from scratch is not production-grade.

Interrupt handling in an agent harness requires three things:

  1. An interrupt channel — a way to signal the running agent loop from outside the main async flow
  2. Cooperative yield points — the agent checks for interrupt signals at natural boundaries (between tool calls) rather than forcing preemption mid-execution
  3. State serialization — enough context is captured at each checkpoint to resume intelligently

Priority Message Injection

One elegant mechanism is a priority message queue. The user's redirect is injected as a high-priority message into the context at the next yield point. The LLM sees it as if the user had spoken during the task and updates its plan accordingly — no special "interrupt handling" code path in the model call itself.

Warning — Don't Interrupt Mid-Tool-Call

Never interrupt an agent while an external tool call is in flight (e.g., a file write, a shell command). Check for interrupt signals only between tool calls. Otherwise you risk leaving external systems in a half-written state that is harder to recover from than letting the current action finish.

Interrupt / Resume State Diagram

stateDiagram-v2 [*] --> Idle Idle --> Running : user submits task Running --> Executing : agent selects tool call Executing --> Running : tool call completes Running --> WaitingApproval : risky action encountered WaitingApproval --> Running : user approves WaitingApproval --> Replanning : user denies / edits Running --> Interrupted : user sends interrupt signal Executing --> Interrupted : tool finishes, interrupt pending Interrupted --> Replanning : inject priority message into context Replanning --> Running : new plan generated Running --> Completed : agent signals task_done Running --> Failed : unrecoverable error Completed --> Idle Failed --> Idle
📄 interruptible_agent.py — InterruptibleAgent pattern
import asyncio
from dataclasses import dataclass, field
from typing import AsyncIterator
import json


@dataclass
class AgentCheckpoint:
    """Serializable snapshot of agent state at a yield point."""
    task_description: str
    completed_steps: list[dict]
    pending_tool_call: dict | None
    context_messages: list[dict]
    metadata: dict = field(default_factory=dict)

    def to_json(self) -> str:
        return json.dumps({
            "task": self.task_description,
            "steps": self.completed_steps,
            "context": self.context_messages[-20:],  # Keep last 20 for resumption
            "metadata": self.metadata,
        })


class InterruptibleAgent:
    """
    Agent loop with cooperative interruption at tool-call boundaries.

    Interrupt flow:
      1. External code calls agent.interrupt(priority_message)
      2. _interrupt_event is set; message queued in _priority_queue
      3. Agent loop checks for interrupt after each tool call completes
      4. On interrupt: checkpoint saved, priority message injected, replanning triggered
    """

    def __init__(self, llm_client, tool_executor, approval_gate):
        self.llm = llm_client
        self.executor = tool_executor
        self.gate = approval_gate

        self._interrupt_event = asyncio.Event()
        self._priority_queue: asyncio.Queue[str] = asyncio.Queue()
        self._checkpoint: AgentCheckpoint | None = None
        self._running = False

    async def run(self, task: str) -> AsyncIterator[dict]:
        """
        Execute a task, yielding events (tool_call, result, interrupt, done).
        Caller can iterate and push events to the UI.
        """
        self._running = True
        messages = [{"role": "user", "content": task}]
        completed_steps: list[dict] = []

        while self._running:
            # ── Check for interrupt before calling the model ──
            if self._interrupt_event.is_set():
                priority_msg = await self._priority_queue.get()
                self._interrupt_event.clear()
                # Inject priority message as a user turn
                messages.append({"role": "user", "content": f"[PRIORITY] {priority_msg}"})
                yield {"type": "interrupt", "message": priority_msg}
                # Save checkpoint for potential resume
                self._checkpoint = AgentCheckpoint(
                    task_description=task,
                    completed_steps=completed_steps,
                    pending_tool_call=None,
                    context_messages=messages,
                )

            # ── Call the model ──
            response = await self.llm.chat(messages)

            if response.stop_reason == "end_turn":
                yield {"type": "done", "output": response.text}
                self._running = False
                break

            if response.stop_reason != "tool_use":
                yield {"type": "error", "reason": f"Unexpected stop: {response.stop_reason}"}
                self._running = False
                break

            # ── Process tool call ──
            tool_call = response.tool_use
            yield {"type": "tool_call", "tool": tool_call.name, "params": tool_call.input}

            # Request approval (may block on user input)
            action = Action(
                tool_name=tool_call.name,
                parameters=tool_call.input,
                rationale=response.text or "Agent step",
            )
            decision = await self.gate.request(action)

            if not decision.approved:
                # Inject denial into conversation so model can replan
                messages.append({"role": "assistant", "content": response.text,
                                  "tool_uses": [tool_call]})
                messages.append({"role": "user", "content": None,
                                  "tool_results": [{"tool_use_id": tool_call.id,
                                                    "content": f"Denied: {decision.denial_reason}"}]})
                yield {"type": "denied", "tool": tool_call.name}
                continue

            # Execute approved action (do NOT check interrupt here — mid-call)
            result = await self.executor.run(tool_call.name, tool_call.input)
            completed_steps.append({"tool": tool_call.name, "result": result})
            yield {"type": "tool_result", "tool": tool_call.name, "result": result}

            # Append to messages for next model call
            messages.append({"role": "assistant", "content": response.text,
                              "tool_uses": [tool_call]})
            messages.append({"role": "user", "content": None,
                              "tool_results": [{"tool_use_id": tool_call.id,
                                                "content": str(result)}]})

    async def interrupt(self, priority_message: str):
        """Thread-safe: can be called from UI event handler or another coroutine."""
        await self._priority_queue.put(priority_message)
        self._interrupt_event.set()

    async def resume_from_checkpoint(self, checkpoint: AgentCheckpoint,
                                      new_direction: str) -> AsyncIterator[dict]:
        """Resume a previously interrupted task with a new direction."""
        modified_messages = checkpoint.context_messages.copy()
        modified_messages.append({"role": "user",
                                   "content": f"Resume from checkpoint. New direction: {new_direction}"})
        async for event in self.run(checkpoint.task_description):
            yield event

Showing Work-in-Progress

Streaming Visibility into Agent Actions

A user staring at a spinner for 90 seconds has no idea whether the agent is making progress or stuck in a loop. Visibility into work-in-progress is both a UX necessity and a safety mechanism — users can spot mistakes early and interrupt before they propagate.

What Needs to Be Visible

  • Current tool call: Which tool is running right now and with what arguments
  • Partial results: Streaming output from long-running commands (e.g., test runner output line-by-line)
  • Step count / progress: "Step 4 of estimated 7" — even rough estimates reduce anxiety
  • Thinking/planning phase: Extended thinking tokens, when visible, tell users the model is reasoning rather than stuck
  • Audit trail: A persistent log of all actions taken, available after the task completes

How Claude Code Does It

Claude Code streams every tool call in real time to the terminal before it executes. The user sees exactly:

  1. The tool name and argument (e.g., Bash(command="npm test"))
  2. Whether it ran automatically or required permission
  3. The full stdout/stderr output as it streams from the subprocess

This makes the agent's reasoning transparent and its actions auditable. If a command runs for 30 seconds, the user sees output accumulating and knows the process is alive.

📄 progress_reporter.py — ProgressReporter pattern
import asyncio
from datetime import datetime
from typing import AsyncIterator, Callable
from dataclasses import dataclass, field
from enum import Enum


class EventType(Enum):
    TOOL_START   = "tool_start"
    TOOL_STREAM  = "tool_stream"    # Partial output from long-running tools
    TOOL_DONE    = "tool_done"
    APPROVAL_REQ = "approval_required"
    APPROVAL_RES = "approval_resolved"
    INTERRUPT    = "interrupt"
    AGENT_THINK  = "agent_thinking"
    DONE         = "done"
    ERROR        = "error"


@dataclass
class ProgressEvent:
    event_type: EventType
    tool_name: str | None = None
    data: dict = field(default_factory=dict)
    timestamp: str = field(default_factory=lambda: datetime.utcnow().isoformat())


class ProgressReporter:
    """
    Collects raw agent events and transforms them into UI-friendly progress events.
    Supports multiple subscriber callbacks (terminal renderer, web socket, audit log).

    Design principle: reporters are passive observers — they do NOT affect agent execution.
    """

    def __init__(self):
        self._subscribers: list[Callable[[ProgressEvent], None]] = []
        self._audit_log: list[ProgressEvent] = []
        self._active_tools: dict[str, datetime] = {}

    def subscribe(self, callback: Callable[[ProgressEvent], None]):
        self._subscribers.append(callback)

    def _emit(self, event: ProgressEvent):
        self._audit_log.append(event)
        for cb in self._subscribers:
            cb(event)

    # ── Public emit methods (called by agent loop) ──

    def tool_started(self, tool_name: str, params: dict):
        self._active_tools[tool_name] = datetime.utcnow()
        self._emit(ProgressEvent(
            event_type=EventType.TOOL_START,
            tool_name=tool_name,
            data={"params": params}
        ))

    def tool_stream(self, tool_name: str, chunk: str):
        """Called for each partial output line from a long-running tool."""
        self._emit(ProgressEvent(
            event_type=EventType.TOOL_STREAM,
            tool_name=tool_name,
            data={"chunk": chunk}
        ))

    def tool_done(self, tool_name: str, result: dict, approved: bool = True):
        elapsed = None
        if tool_name in self._active_tools:
            elapsed = (datetime.utcnow() - self._active_tools.pop(tool_name)).total_seconds()
        self._emit(ProgressEvent(
            event_type=EventType.TOOL_DONE,
            tool_name=tool_name,
            data={"result": result, "elapsed_s": elapsed, "approved": approved}
        ))

    def approval_required(self, tool_name: str, risk: str, params: dict):
        self._emit(ProgressEvent(
            event_type=EventType.APPROVAL_REQ,
            tool_name=tool_name,
            data={"risk": risk, "params": params}
        ))

    def done(self, output: str):
        self._emit(ProgressEvent(event_type=EventType.DONE, data={"output": output}))

    # ── Audit trail export ──

    def export_audit_log(self) -> list[dict]:
        return [{"ts": e.timestamp, "type": e.event_type.value,
                 "tool": e.tool_name, "data": e.data}
                for e in self._audit_log]


# ── Example: terminal renderer subscriber ─────────────────────────────────────

def terminal_renderer(event: ProgressEvent):
    ts = event.timestamp[11:19]  # HH:MM:SS
    tool = event.tool_name or "agent"
    match event.event_type:
        case EventType.TOOL_START:
            print(f"\033[36m{ts}\033[0m  \033[1m{tool}\033[0m  Starting with {event.data.get('params')}")
        case EventType.TOOL_STREAM:
            print(f"       {event.data.get('chunk')}", end="", flush=True)
        case EventType.TOOL_DONE:
            elapsed = event.data.get('elapsed_s', 0)
            print(f"\033[36m{ts}\033[0m  \033[1m{tool}\033[0m  Done ({elapsed:.1f}s)")
        case EventType.APPROVAL_REQ:
            print(f"\033[33m{ts}  APPROVAL REQUIRED [{event.data.get('risk')}]  {tool}\033[0m")
        case EventType.DONE:
            print(f"\033[32m{ts}  Task complete.\033[0m")

Undo and Rollback

Every Agent Action Should Be Reversible

The ability to undo agent actions transforms the risk calculus of HITL design. If every action is reversible with low cost, you can afford to auto-approve more of them — because a mistake is not a crisis, it is just one Ctrl+Z away. Undo is a force multiplier for trust escalation.

Three Classes of Undo

ClassMechanismExampleLimitation
File-level undo Git snapshot or file-system copy before each write Restore src/api.py to its pre-agent state Only covers file mutations, not external side effects
Command-level undo Inverse operation: if the action was mkdir foo, the undo is rmdir foo Revert a git commit, delete a created file Not all operations have clean inverses (e.g., npm install)
Transaction log rollback Replay the undo log from the last good checkpoint Revert a 12-step refactor back to the state before the agent started Rollback cost grows with task length; may conflict with unrelated edits

import shutil
import hashlib
import json
from pathlib import Path
from dataclasses import dataclass, field
from typing import Callable, Awaitable
from datetime import datetime


@dataclass
class UndoEntry:
    """One reversible agent action."""
    action_id: str
    tool_name: str
    parameters: dict
    timestamp: str = field(default_factory=lambda: datetime.utcnow().isoformat())
    # One of: snapshot path, inverse_op dict, or None (not undoable)
    snapshot_path: str | None = None
    inverse_op: dict | None = None
    description: str = ""

    @property
    def is_undoable(self) -> bool:
        return self.snapshot_path is not None or self.inverse_op is not None


class SnapshotManager:
    """
    Creates and restores file snapshots before destructive operations.
    Uses content-addressed storage (SHA256) to deduplicate identical snapshots.
    """

    def __init__(self, snapshot_dir: str = ".agent_snapshots"):
        self.snapshot_dir = Path(snapshot_dir)
        self.snapshot_dir.mkdir(parents=True, exist_ok=True)

    def capture(self, file_path: str) -> str | None:
        """Snapshot a file before modification. Returns snapshot path or None."""
        src = Path(file_path)
        if not src.exists():
            return None
        content = src.read_bytes()
        digest = hashlib.sha256(content).hexdigest()[:12]
        dest = self.snapshot_dir / f"{digest}_{src.name}"
        if not dest.exists():
            shutil.copy2(src, dest)
        # Store metadata: original path + snapshot path
        meta = {"original": str(src), "snapshot": str(dest), "mtime": src.stat().st_mtime}
        meta_path = dest.with_suffix(".meta.json")
        meta_path.write_text(json.dumps(meta))
        return str(dest)

    def restore(self, snapshot_path: str) -> bool:
        """Restore a file from its snapshot. Returns success."""
        snap = Path(snapshot_path)
        meta_path = snap.with_suffix(".meta.json")
        if not snap.exists() or not meta_path.exists():
            return False
        meta = json.loads(meta_path.read_text())
        shutil.copy2(snap, meta["original"])
        return True

    def cleanup(self, older_than_hours: int = 24):
        """Prune old snapshots to reclaim disk space."""
        import time
        cutoff = time.time() - older_than_hours * 3600
        for f in self.snapshot_dir.iterdir():
            if f.stat().st_mtime < cutoff:
                f.unlink()


class UndoManager:
    """
    Wraps the executor to log every action with its undo information.
    Provides undo(n) to revert the last n actions.
    """

    def __init__(self, executor, snapshots: SnapshotManager):
        self.executor = executor
        self.snapshots = snapshots
        self._stack: list[UndoEntry] = []

    async def execute(self, tool_name: str, params: dict, action_id: str) -> dict:
        entry = UndoEntry(
            action_id=action_id,
            tool_name=tool_name,
            parameters=params,
        )

        # Capture snapshot for file-mutating tools
        if tool_name in ("write_file", "patch_file") and "path" in params:
            entry.snapshot_path = self.snapshots.capture(params["path"])
            entry.description = f"Modified {params['path']}"

        # Record inverse operations for other reversible tools
        elif tool_name == "create_file" and "path" in params:
            entry.inverse_op = {"tool": "delete_file", "params": {"path": params["path"]}}
            entry.description = f"Created {params['path']}"

        elif tool_name == "git_commit" and "sha" in params:
            entry.inverse_op = {"tool": "bash",
                                  "params": {"command": f"git revert {params['sha']} --no-edit"}}
            entry.description = f"Git commit {params.get('sha', '')[:8]}"

        result = await self.executor.run(tool_name, params)
        self._stack.append(entry)
        return result

    async def undo(self, n: int = 1) -> list[str]:
        """Undo the last n actions. Returns list of descriptions of what was undone."""
        undone = []
        for _ in range(n):
            if not self._stack:
                break
            entry = self._stack.pop()
            if not entry.is_undoable:
                undone.append(f"[SKIP — not undoable] {entry.tool_name}")
                continue
            if entry.snapshot_path:
                ok = self.snapshots.restore(entry.snapshot_path)
                undone.append(f"Restored: {entry.description}" if ok
                               else f"[FAIL] Could not restore: {entry.description}")
            elif entry.inverse_op:
                await self.executor.run(entry.inverse_op["tool"],
                                         entry.inverse_op["params"])
                undone.append(f"Reverted: {entry.description}")
        return undone

    def undo_history(self) -> list[dict]:
        return [{"id": e.action_id, "tool": e.tool_name, "desc": e.description,
                 "ts": e.timestamp, "undoable": e.is_undoable}
                for e in reversed(self._stack)]

Production Insight — Git as the Universal Undo

For code agents, committing a git snapshot before each agent run is the simplest and most powerful undo mechanism available. Users already understand git; "undo agent changes" maps naturally to "git reset --hard HEAD~1". Claude Code leverages this by encouraging users to work inside a git repo and by showing which files would be modified before executing.

Interview Tips

What Interviewers Look For

System Design Signals

  • You understand that HITL is a spectrum, not a binary — and that the right point on the spectrum depends on the action's blast radius, reversibility, and cost of interruption.
  • You separate mechanism from policy: The approval gate (mechanism) is agnostic to which actions require approval; that's defined by the policy object, making it configurable without code changes.
  • You mention undo as a trust enabler: Most candidates focus on HITL as a brake. Strong candidates see undo as an accelerator — it lets you reduce approval friction because mistakes are cheap to fix.
  • You consider the async approval case: In long-running agents (Devin-style), the user cannot be expected to respond within milliseconds. Async approval — queue the action, notify the user, unblock when they respond — is production-realistic.

Common Follow-Up Questions

  • "How would you handle approval for an agent that runs unattended overnight?" — async approval queue with configurable fallback policies (fail-safe: skip and log; fail-open: execute and notify)
  • "What happens if the user approves an action but the executor crashes halfway through?" — idempotency, partial-state detection on resume, and the undo log
  • "How do you prevent approval fatigue?" — batch grants, progressive trust escalation, and risk-weighted sampling (don't prompt for every low-risk action even if it's technically "risky")
  • "How would you A/B test different approval UX patterns?" — feature-flag the approval policy, track denial rate, undo rate, and task completion rate as proxy metrics for the right friction level

Red Flags to Avoid

  • Treating HITL as "just show a confirm dialog" — misses the engineering depth around async flows, trust escalation, and undo
  • Not separating policy from mechanism — hard-coding "always ask for shell commands" instead of making it configurable
  • Ignoring undo — describing an approval-only system that has no recovery path if an approved action turns out to be wrong
  • Forgetting the interrupt-mid-tool-call race condition — describing an interrupt handler that fires during an in-flight external call

Module Quiz

Q1. An agent is about to run git push origin main. The harness classifies this as DANGEROUS. The user has granted a batch approval for the "git" tool category. Should the gate prompt the user?

It depends on the policy implementation. As designed in the module, batch grants override the risk classification, so the gate would not prompt. However, this is a valid design tension: a blanket "git" grant should probably exclude DANGEROUS sub-operations like force-push or push to a protected branch. The better design is to scope batch grants by tool and risk tier — e.g., "auto-approve git RISKY, still ask for git DANGEROUS". The key interview point is recognising that tool-category grants should be granular enough to exclude highest-risk operations.

Q2. Why is it a mistake to check for interrupt signals while a tool call is executing rather than after it completes?

Interrupting mid-tool-call risks leaving external systems in a partial or inconsistent state. For example, if a file write is 60% complete when the interrupt fires and you kill the call, the file on disk is corrupted. The cooperative interruption model defers the interrupt check to the boundary after the tool call finishes — the current action runs to completion, then the interrupt is processed. The trade-off is that the user may have to wait for the current action to finish before the redirect takes effect, but this is almost always preferable to half-applied mutations.

Q3. What is the difference between session-based and persistent trust, and when would you choose each?

Session trust resets on every conversation start. It is the safe default: trust earned in one task cannot be abused in a future session. It is simpler to implement and reason about. Choose it for products with high security requirements or where tasks are short and varied.

Persistent trust is stored in a user profile and carries over across sessions. It reduces friction for power users who repeatedly perform the same class of actions (e.g., a developer who always approves "write_file in src/"). It requires a richer audit trail, explicit revocation UI, and careful scoping (trust should be bound to a repo or project, not globally). Choose it for expert-user products where automation value is high and the blast radius is bounded.

Q4. Describe how you would make an undo system work for a multi-step agent task that included a git commit.

For git commit, the undo entry records an inverse operation: git revert <sha> --no-edit (or git reset --hard HEAD~1 if you prefer a hard undo). For file patches applied before the commit, the undo entries have snapshot paths captured before each patch was applied. On undo, you pop entries LIFO — first revert the commit, then restore each file from its snapshot in reverse order. The key challenge is that git revert creates a new commit (preserving history) while git reset rewrites it. For shared repos, revert is safer. For local solo work, reset is simpler. The UndoManager should expose both options.

Q5. How would you measure whether your HITL approval workflow has the right amount of friction?

Key metrics to track:

Too much friction: High "always approve" rate (users approve without reading — approval is theater), low task completion rate, high session abandonment rate after approval prompts, user feedback mentioning interruptions.

Too little friction: High undo/rollback rate (users frequently reverting agent actions), high post-task "agent made a mistake" reports, denial rate close to zero (the gate never prompts for things users would actually deny).

The target is a denial rate of 5–15% (prompts are meaningful) and an undo rate well below the denial rate (the approved actions mostly worked out). You can A/B test different approval policies and use task completion rate + undo rate as a joint objective function.

Checkpoint Summary

  • HITL is a product requirement driven by safety, trust, and UX — not just an engineering checkbox.
  • The trust continuum runs from fully manual to fully autonomous; most production agents live in the HITL hybrid band.
  • Approval workflows come in three shapes: pre-execution (ask before doing), post-execution (do then confirm), and batch (approve categories). Batch grants are the key to breaking out of the interruption trap.
  • Separate mechanism (ApprovalGate) from policy (ApprovalPolicy) so risk rules are configurable without code changes.
  • Progressive trust escalation lets agents earn autonomy through demonstrated reliable behavior — STRICT to MODERATE to RELAXED, with hard resets on denials and rollbacks.
  • Cooperative interruption checks for signals only at tool-call boundaries, never mid-execution, to avoid partial-state corruption.
  • Work-in-progress visibility via streaming tool call events is both a UX requirement and a safety mechanism — users can spot and interrupt mistakes early.
  • Undo is a trust accelerator: every reversible action allows you to approve more aggressively, because a mistake is recoverable. Three undo classes — snapshot, inverse-op, and transaction-log rollback — cover all action types.
