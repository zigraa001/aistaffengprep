1. Tool Schema Design

JSON Schema Anatomy ▼

Every tool exposed to an LLM is serialised as a JSON object that follows a fixed structure. All major APIs (Anthropic, OpenAI, Google) share the same high-level shape, though they differ in edge cases:

{
  "name": "read_file",
  "description": "Read the contents of a file at the given path. Use this when you need to examine source code, configuration files, or any text file. Returns the file content as a string with line numbers prepended.",
  "input_schema": {
    "type": "object",
    "properties": {
      "path": {
        "type": "string",
        "description": "Absolute path to the file. Must start with /."
      },
      "limit": {
        "type": "integer",
        "description": "Maximum number of lines to return. Defaults to 200 if omitted.",
        "minimum": 1,
        "maximum": 2000
      },
      "offset": {
        "type": "integer",
        "description": "Line number to start reading from (1-indexed). Use with limit for pagination.",
        "minimum": 1
      }
    },
    "required": ["path"]
  }
}

The four load-bearing parts are:

Field Purpose Common Mistakes
name Unique identifier the model emits verbatim when calling the tool Spaces or hyphens (some APIs reject them); names longer than ~30 chars hurt recall
description The model's primary guide on when and how to use the tool Vague one-liners; no mention of failure modes; too long (wastes tokens every turn)
input_schema Validates the model's argument structure and provides per-parameter guidance All params optional (model guesses); missing minimum/maximum constraints
Parameter description Micro-prompt telling the model what value to supply Repeating the param name; not specifying format (e.g., absolute vs relative path)
Key Insight: The model never reads your code. It only reads the schema. Every behavioural expectation you have must be expressed as text in description fields. The schema is a contract written in natural language, not a type system.
How Claude, GPT-4, and Gemini Differ ▼

All three providers accept a broadly similar schema format, but the models were trained differently and exhibit distinct behaviours:

BehaviourClaude 3.x / Sonnet 4GPT-4oGemini 1.5 Pro
Parallel tool calls in one response Yes — emits multiple tool_use blocks Yes — tool_calls array Yes — functionCall parts
Strict schema adherence High; rarely emits extra fields High with strict: true; lax otherwise Medium; may omit optional params
Handling unknown enum values Defaults to first listed value or asks May hallucinate a plausible value Often omits the param entirely
Max tools in a single schema Tested up to 200+; quality degrades ~50+ Quality degrades noticeably ~64+ Documented limit 64 tools
Tool choice forcing tool_choice: {type: "tool", name: "..."} tool_choice: {type: "function", function: {name: "..."}} tool_config: {mode: "ANY", allowed_function_names: [...]}
Production tip: If you target multiple providers, write an adapter layer that translates your internal tool definitions to each provider's wire format. The schema shapes are similar but not identical — Anthropic uses input_schema, OpenAI uses parameters, Gemini uses parameters under function_declarations.
Real Examples: Claude Code's Core Tools ▼

Claude Code ships a small, carefully designed tool set. Studying them reveals deliberate schema engineering choices:

  "input_schema": {
    "type": "object",
    "properties": {
      "file_path": { "type": "string", "description": "The absolute path to the file to read" },
      "limit":     { "type": "number", "description": "The number of lines to read." },
      "offset":    { "type": "number", "description": "The line number to start reading from." }
    },
    "required": ["file_path"]
  }
}

// Bash - power tool with explicit safety rails in the description
{
  "name": "Bash",
  "description": "Executes a bash command in a persistent shell session. Use for running tests, build commands, git operations, or any shell task. Avoid commands that produce unbounded output — pipe through head/tail. Do NOT use for file reads when the Read tool suffices.",
  "input_schema": {
    "type": "object",
    "properties": {
      "command":     { "type": "string", "description": "The bash command to run." },
      "description": { "type": "string", "description": "Clear description of what this command does and why." },
      "timeout":     { "type": "number", "description": "Timeout in milliseconds. Default 120000." }
    },
    "required": ["command", "description"]
  }
}

// Glob - scoped file discovery
{
  "name": "Glob",
  "description": "Fast file pattern matching tool. Supports glob patterns like '**/*.js'. Returns matching file paths sorted by modification time. Use this to find files by name pattern, not to read their contents.",
  "input_schema": {
    "type": "object",
    "properties": {
      "pattern": { "type": "string", "description": "The glob pattern to match files against" },
      "path":    { "type": "string", "description": "Directory to search in. Omit to use current working directory." }
    },
    "required": ["pattern"]
  }
}

Notice the patterns:

  • One required param, rest optional — reduces friction; the model can call the tool with minimal arguments
  • Negative instructions in descriptions — "Do NOT use for file reads when the Read tool suffices" prevents tool overuse
  • Usage bullets in description — concise, scannable rules the model re-reads every turn
  • No enum for path — dynamic values are never constrained with enums; only finite static sets should use enum

2. Description Optimization

The Description Is a Prompt, Not Documentation ▼
Mental Model: Tool descriptions are exactly like system prompt instructions except they live inside the tool schema. The model reads all descriptions on every inference pass. A 500-token description for a rarely-used tool is always in context, even when it's irrelevant. Treat each token as expensive.

The description must answer three questions a model asks (implicitly) before selecting a tool:

  1. What does this tool do? — action + side effects
  2. When should I use it? — triggering conditions
  3. When should I NOT use it? — disambiguation from similar tools

The brevity vs clarity tradeoff resolves differently for different tool categories:

Tool CategoryIdeal Description LengthReasoning
Utility (read_file, list_dir)2-4 sentencesAction is obvious; spend tokens on constraints and format
Domain-specific API (send_email, create_issue)4-8 sentencesSide effects must be explicit; rate limits and auth notes help
Dangerous / irreversible (delete, deploy)6-10 sentencesModel needs enough context to avoid accidents; negative instructions critical
Disambiguation pair (search_web vs search_codebase)Add 1-2 contrast sentencesModel confuses similar tools without explicit "use X for Y, not Z"
A/B Testing Tool Descriptions & the ToolDescriptionBuilder Pattern ▼

Small wording changes cause measurable shifts in tool call accuracy. Treat description copy as a tunable hyperparameter.

Before — vague, no rails

The ToolDescriptionBuilder pattern lets you version, test, and iterate descriptions systematically:

from dataclasses import dataclass, field
from typing import Optional
import hashlib, json

@dataclass
class ToolDescriptionBuilder:
    """
    Builds and versions tool descriptions for A/B testing.
    Tracks which description variant is active so you can correlate
    with task success metrics in your observability layer.
    """
    name: str
    action: str                        # "Reads a file from the filesystem."
    when_to_use: list[str] = field(default_factory=list)
    do_not_use_for: list[str] = field(default_factory=list)
    format_notes: list[str] = field(default_factory=list)
    examples: list[str] = field(default_factory=list)
    max_tokens: int = 200              # soft budget

    def build(self) -> str:
        parts = [self.action]

        if self.when_to_use:
            parts.append("Use for: " + "; ".join(self.when_to_use) + ".")
        if self.do_not_use_for:
            parts.append("Do NOT use for: " + "; ".join(self.do_not_use_for) + ".")
        if self.format_notes:
            parts.append("\n".join(self.format_notes))
        if self.examples:
            parts.append("Examples:\n" + "\n".join(f"  - {e}" for e in self.examples))

        description = "\n\n".join(parts)
        # warn if over budget (approximate token count)
        approx_tokens = len(description.split())
        if approx_tokens > self.max_tokens:
            import warnings
            warnings.warn(
                f"Tool '{self.name}' description ~{approx_tokens} words, "
                f"exceeds budget of {self.max_tokens}. Consider trimming."
            )
        return description

    def fingerprint(self) -> str:
        """Stable hash for logging which description variant was active."""
        return hashlib.sha1(self.build().encode()).hexdigest()[:8]


# Usage
read_desc = ToolDescriptionBuilder(
    name="Read",
    action="Reads a file from the local filesystem and returns its content with line numbers.",
    when_to_use=["examining source code", "reading config files", "inspecting any text file"],
    do_not_use_for=["finding files by name (use Glob)", "searching content by pattern (use Grep)"],
    format_notes=["file_path must be absolute.", "Use limit + offset for large files."],
)
print(read_desc.build())
print("Variant:", read_desc.fingerprint())  # e.g. "a3f8c12b"
Measurement tip: Log the description fingerprint alongside each tool call in your traces. When you change a description, you can query "did tool call accuracy improve on task type X after switching from fingerprint a3f8c12b to 7bd29e44?"

3. Execution Engines

Four Archetypes of Tool Execution ▼

Once the model emits a tool call your harness must decide how to run it. The choice of execution engine determines isolation, latency, and resource control:

Engine TypeHow It WorksIsolationLatencyUse For
In-process Direct function call in the harness process None <1 ms File reads, in-memory lookups, pure computations
Subprocess subprocess.run() / exec() OS process boundary 5–50 ms startup Shell commands, CLI tools, build systems
Network call HTTP/gRPC to external service Network + auth boundary 10–500 ms External APIs, databases, search engines
Sandboxed Docker container, gVisor, Firecracker VM Full kernel isolation 100 ms–2 s cold start Arbitrary code execution, untrusted inputs

ToolExecutor Class with Timeout & Resource Management ▼ 
import asyncio
import subprocess
import resource
import time
from dataclasses import dataclass
from typing import Any, Callable, Awaitable, Optional

@dataclass
class ExecutionConfig:
    timeout_seconds: float = 30.0
    max_output_bytes: int = 1_000_000   # 1 MB
    max_memory_mb:   int = 512
    working_dir:     Optional[str] = None

@dataclass
class ExecutionResult:
    stdout:       str
    stderr:       str
    exit_code:    int
    duration_ms:  float
    truncated:    bool = False

class ToolExecutor:
    """
    Unified executor that wraps subprocess execution with
    timeout enforcement, output capping, and resource limits.
    """

    def __init__(self, config: ExecutionConfig):
        self.config = config

    async def run_bash(self, command: str) -> ExecutionResult:
        start = time.monotonic()

        try:
            proc = await asyncio.create_subprocess_shell(
                command,
                stdout=asyncio.subprocess.PIPE,
                stderr=asyncio.subprocess.PIPE,
                cwd=self.config.working_dir,
                # Apply memory limit via preexec_fn on Linux
                preexec_fn=self._apply_resource_limits,
            )

            try:
                stdout_b, stderr_b = await asyncio.wait_for(
                    proc.communicate(),
                    timeout=self.config.timeout_seconds,
                )
            except asyncio.TimeoutError:
                proc.kill()
                await proc.communicate()
                return ExecutionResult(
                    stdout="",
                    stderr=f"Command timed out after {self.config.timeout_seconds}s",
                    exit_code=124,
                    duration_ms=(time.monotonic() - start) * 1000,
                )

        except Exception as e:
            return ExecutionResult(
                stdout="",
                stderr=f"Execution error: {e}",
                exit_code=1,
                duration_ms=(time.monotonic() - start) * 1000,
            )

        truncated = False
        stdout = stdout_b.decode("utf-8", errors="replace")
        stderr = stderr_b.decode("utf-8", errors="replace")

        if len(stdout_b) > self.config.max_output_bytes:
            stdout = stdout[: self.config.max_output_bytes].rsplit("\n", 1)[0]
            stdout += f"\n[... output truncated at {self.config.max_output_bytes} bytes ...]"
            truncated = True

        return ExecutionResult(
            stdout=stdout,
            stderr=stderr[:4096],       # cap stderr separately
            exit_code=proc.returncode,
            duration_ms=(time.monotonic() - start) * 1000,
            truncated=truncated,
        )

    def _apply_resource_limits(self):
        """Called in child process before exec (Linux only)."""
        try:
            mem_bytes = self.config.max_memory_mb * 1024 * 1024
            resource.setrlimit(resource.RLIMIT_AS, (mem_bytes, mem_bytes))
            # Prevent core dumps
            resource.setrlimit(resource.RLIMIT_CORE, (0, 0))
        except Exception:
            pass  # Windows / macOS: silently skip

    async def run_in_process(
        self,
        fn: Callable[..., Awaitable[Any]],
        *args,
        **kwargs,
    ) -> ExecutionResult:
        """Run a coroutine as a tool, wrapping exceptions."""
        start = time.monotonic()
        try:
            result = await asyncio.wait_for(
                fn(*args, **kwargs),
                timeout=self.config.timeout_seconds,
            )
            return ExecutionResult(
                stdout=str(result),
                stderr="",
                exit_code=0,
                duration_ms=(time.monotonic() - start) * 1000,
            )
        except asyncio.TimeoutError:
            return ExecutionResult(
                stdout="",
                stderr=f"Function timed out after {self.config.timeout_seconds}s",
                exit_code=124,
                duration_ms=(time.monotonic() - start) * 1000,
            )
        except Exception as e:
            return ExecutionResult(
                stdout="",
                stderr=f"{type(e).__name__}: {e}",
                exit_code=1,
                duration_ms=(time.monotonic() - start) * 1000,
            )

4. Result Formatting & Truncation

Why Truncation Is an Engineering Problem, Not a Detail ▼
Context budget math: Suppose you have a 200 k-token window. Your system prompt costs 2 k tokens. Each tool result from a file read averages 500 tokens when untruncated. After just 25 sequential tool calls, you've spent 12.5 k tokens on results — and you haven't even included the model's reasoning or the conversation history. At 200 tool calls, you've blown the window even if every file was tiny.

Smart truncation must be content-aware. The right strategy differs by tool type:

Tool TypeTruncation StrategyRationale
File read First N lines + last M lines with a gap marker; always include line numbers Imports/headers at top, main logic near bottom; middle is often boilerplate
Grep / search results First K matches; show ±2 lines context per match; total lines capped First match is usually the most relevant; the model can re-search with refined pattern
Bash output First N lines; append last M lines if exit_code != 0 Errors appear at the end; always surface them even if body is truncated
JSON API response Trim arrays > K items; omit null/empty fields; add truncation annotation LLMs parse structure, not raw JSON; compact is easier to read
Directory listing Sort by modification time; cap at 50 entries; group by extension Most-recently-modified files are usually most relevant
ResultFormatter with Truncation Policies ▼ 
from dataclasses import dataclass
from typing import Optional
import json

@dataclass
class TruncationPolicy:
    max_lines:       int = 200
    head_lines:      int = 100   # lines from top
    tail_lines:      int = 50    # lines from bottom (errors, return stmts)
    max_chars:       int = 20_000
    include_numbers: bool = True

class ResultFormatter:
    """
    Converts raw tool execution output into model-ready strings
    that respect context budgets while preserving information density.
    """

    def __init__(self, policy: TruncationPolicy = TruncationPolicy()):
        self.policy = policy

    # ── File content ──────────────────────────────────────────────────
    def format_file(
        self,
        content: str,
        path: str,
        offset: int = 1,
    ) -> str:
        lines = content.splitlines()
        total = len(lines)

        if total <= self.policy.max_lines:
            numbered = self._number_lines(lines, start=offset)
            return f"<file path='{path}' lines='{offset}-{offset+total-1}'>\n{numbered}\n</file>"

        head = lines[: self.policy.head_lines]
        tail = lines[-self.policy.tail_lines :]
        omitted = total - self.policy.head_lines - self.policy.tail_lines

        head_str = self._number_lines(head, start=offset)
        tail_start = offset + total - self.policy.tail_lines
        tail_str  = self._number_lines(tail, start=tail_start)

        return (
            f"<file path='{path}' total_lines='{total}' truncated='true'>\n"
            f"{head_str}\n"
            f"... [{omitted} lines omitted — use offset={offset + self.policy.head_lines} to read more] ...\n"
            f"{tail_str}\n"
            f"</file>"
        )

    # ── Grep results ──────────────────────────────────────────────────
    def format_grep(
        self,
        matches: list[dict],   # [{file, line_no, content, context_before, context_after}]
        pattern: str,
        max_matches: int = 20,
    ) -> str:
        if not matches:
            return f"No matches found for pattern: {pattern!r}"

        truncated = len(matches) > max_matches
        shown = matches[:max_matches]
        parts = [f"Found {len(matches)} match(es) for {pattern!r}" +
                 (f" (showing first {max_matches})" if truncated else "") + ":"]

        for m in shown:
            header = f"\n{m['file']}:{m['line_no']}"
            ctx = "\n".join(
                f"  {m['line_no'] - len(m.get('context_before', [])) + i} | {l}"
                for i, l in enumerate(m.get("context_before", []))
            )
            match_line = f"> {m['line_no']} | {m['content']}"
            ctx_after = "\n".join(
                f"  {m['line_no'] + i + 1} | {l}"
                for i, l in enumerate(m.get("context_after", []))
            )
            parts.append(header + "\n" + "\n".join(filter(None, [ctx, match_line, ctx_after])))

        return "\n".join(parts)

    # ── JSON API responses ────────────────────────────────────────────
    def format_json(
        self,
        data: Any,
        max_array_items: int = 10,
        max_depth: int = 5,
    ) -> str:
        pruned = self._prune_json(data, max_array_items, max_depth, 0)
        return json.dumps(pruned, indent=2, default=str)

    def _prune_json(self, obj, max_items, max_depth, depth):
        if depth >= max_depth:
            return "<max depth reached>"
        if isinstance(obj, dict):
            return {
                k: self._prune_json(v, max_items, max_depth, depth + 1)
                for k, v in obj.items()
                if v not in (None, "", [], {})
            }
        if isinstance(obj, list):
            truncated = len(obj) > max_items
            pruned = [self._prune_json(i, max_items, max_depth, depth + 1)
                      for i in obj[:max_items]]
            if truncated:
                pruned.append(f"... [{len(obj) - max_items} more items]")
            return pruned
        return obj

    # ── Helpers ───────────────────────────────────────────────────────
    def _number_lines(self, lines: list[str], start: int = 1) -> str:
        width = len(str(start + len(lines)))
        return "\n".join(
            f"{str(start + i).rjust(width)}\t{line}"
            for i, line in enumerate(lines)
        )
Why XML-style wrappers? Wrapping results in <file> / </file> tags trains the model to treat tool results as distinct data regions, not as part of the conversation. It reduces hallucination where the model confuses prior tool output with its own prior responses. Claude is particularly responsive to this pattern.

5. Error Recovery

Error Categorization: Not All Failures Are Equal ▼

The harness, not the model, should classify errors. Give the model a rich enough error message to decide its next action, but don't expose raw stack traces that waste tokens.

Error ClassExamplesHarness ActionMessage to Model
Retryable / Transient Network timeout, rate limit (429), disk I/O spike Retry with exponential backoff (max 3 attempts) If all retries fail, surface with "temporarily unavailable, try again"
User Fixable File not found, permission denied, invalid path Return immediately; no retry Specific: "File /foo/bar.py does not exist. Use Glob to find the correct path."
Schema / Usage Error Missing required param, wrong type, enum mismatch Validate before execution; return early "Invalid arguments: 'path' is required but was not provided."
Fatal / System Error Sandbox crash, OOM, unrecoverable DB failure Log + alert; return graceful degradation message "Tool unavailable due to internal error. Proceed without this tool or ask user for help."
Anti-pattern: Returning an empty string or null on error. The model will often re-call the same tool with the same arguments, entering an infinite loop. Always return a non-empty error message that explains what happened and, where possible, suggests a corrective action.
Error Handling Middleware & Retry Decorator ▼ 
import asyncio
import functools
import logging
from typing import Callable, TypeVar, Awaitable

logger = logging.getLogger(__name__)
T = TypeVar("T")

# ── Retry decorator ───────────────────────────────────────────────────
def retryable(
    max_attempts: int = 3,
    base_delay: float = 0.5,
    retryable_exceptions: tuple = (TimeoutError, ConnectionError, OSError),
):
    """
    Decorator that retries async tool functions on transient errors.
    Uses exponential backoff with full jitter.
    """
    def decorator(fn: Callable[..., Awaitable[T]]) -> Callable[..., Awaitable[T]]:
        @functools.wraps(fn)
        async def wrapper(*args, **kwargs) -> T:
            import random
            for attempt in range(1, max_attempts + 1):
                try:
                    return await fn(*args, **kwargs)
                except retryable_exceptions as exc:
                    if attempt == max_attempts:
                        raise
                    delay = base_delay * (2 ** (attempt - 1)) * random.uniform(0.5, 1.5)
                    logger.warning(
                        "Tool %s attempt %d/%d failed: %s. Retrying in %.2fs",
                        fn.__name__, attempt, max_attempts, exc, delay,
                    )
                    await asyncio.sleep(delay)
        return wrapper
    return decorator


# ── Error middleware ──────────────────────────────────────────────────
class ToolErrorMiddleware:
    """
    Wraps tool execution to produce model-friendly error messages
    regardless of what exception was thrown.
    """

    RETRYABLE = (TimeoutError, ConnectionError, OSError)
    USAGE     = (ValueError, TypeError, KeyError)

    def __init__(self, tool_name: str):
        self.tool_name = tool_name

    async def execute(self, coro: Awaitable[str]) -> str:
        try:
            return await coro

        except self.RETRYABLE as exc:
            logger.error("Transient error in tool '%s': %s", self.tool_name, exc, exc_info=True)
            return (
                f"[Tool error — transient] {self.tool_name} failed due to a temporary issue: "
                f"{type(exc).__name__}: {exc}. "
                f"You may retry this tool call. If it persists, try an alternative approach."
            )

        except PermissionError as exc:
            return (
                f"[Tool error — permission] Cannot access resource: {exc}. "
                f"Check that the path exists and you have read permission."
            )

        except FileNotFoundError as exc:
            return (
                f"[Tool error — not found] {exc}. "
                f"Use Glob to verify the file path before reading."
            )

        except self.USAGE as exc:
            return (
                f"[Tool error — bad arguments] {type(exc).__name__}: {exc}. "
                f"Review the tool schema and correct the arguments."
            )

        except Exception as exc:
            logger.critical(
                "Unhandled exception in tool '%s'", self.tool_name, exc_info=True
            )
            return (
                f"[Tool error — internal] {self.tool_name} encountered an unexpected error. "
                f"Proceed without this tool result or ask the user for help."
            )


# ── Usage example ────────────────────────────────────────────────────
async def handle_tool_call(tool_name: str, tool_fn, args: dict) -> str:
    middleware = ToolErrorMiddleware(tool_name)
    return await middleware.execute(tool_fn(**args))

6. Parallel vs Sequential Execution

When Parallel Execution Matters ▼
Engineering analogy: Tool calls are like database queries. You wouldn't execute SELECT * FROM users and SELECT * FROM orders sequentially when they're independent. The same applies to tool calls — parallel dispatch cuts total wall-clock latency by the number of independent calls in a batch.

Claude Code (and many modern agents) emit multiple tool calls in a single response when the model determines they're independent. The harness must:

  1. Detect that a response contains multiple tool_use blocks
  2. Identify dependencies between them (can call B before A finishes?)
  3. Execute independent groups in parallel; dependent calls in sequence
  4. Collect all results and return them as a batch in the next message
flowchart LR subgraph "Sequential (naive)" S1["Read file A"] --> S2["Read file B"] --> S3["Grep pattern"] --> S4["Model turn 2"] T1["⏱ 3 × 50ms = 150ms"] end subgraph "Parallel (smart)" P1["Read file A"] P2["Read file B"] P3["Grep pattern"] PG["Gather results"] --> P4["Model turn 2"] P1 --> PG P2 --> PG P3 --> PG T2["⏱ max(50ms) = 50ms"] end style T1 fill:#dc2626,color:#fff style T2 fill:#16a34a,color:#fff

The 3x latency reduction compounds over a long task. A 50-step coding task with average 3 parallel tool calls per step goes from ~7.5 s (sequential) to ~2.5 s (parallel) for tool execution alone — before model inference time.

Race conditions to watch for: Two parallel Bash calls that both write to the same file, or two Edit calls targeting overlapping regions. The dependency detector must flag these as sequential. A simple heuristic: if any two calls touch the same file path, serialize them.
ParallelToolExecutor with Dependency Graph ▼ 
import asyncio
from dataclasses import dataclass, field
from typing import Any

@dataclass
class ToolCall:
    id:        str
    name:      str
    arguments: dict[str, Any]

@dataclass
class ToolResult:
    tool_use_id: str
    content:     str

class DependencyDetector:
    """
    Heuristic dependency detection between tool calls.
    Conservative: when in doubt, serialize.
    """

    WRITE_TOOLS = {"Edit", "Write", "Bash"}
    READ_TOOLS  = {"Read", "Glob", "Grep"}

    def extract_paths(self, call: ToolCall) -> set[str]:
        """Best-effort path extraction from tool arguments."""
        paths = set()
        for key in ("path", "file_path", "command"):
            val = call.arguments.get(key, "")
            if isinstance(val, str) and val.startswith("/"):
                # crude: first token that looks like a path
                for token in val.split():
                    if token.startswith("/"):
                        paths.add(token)
        return paths

    def depends_on(self, a: ToolCall, b: ToolCall) -> bool:
        """Return True if b must run after a."""
        # Write-after-write or write-after-read on same path
        paths_a = self.extract_paths(a)
        paths_b = self.extract_paths(b)
        shared  = paths_a & paths_b

        if not shared:
            return False

        # If either is a write tool, enforce ordering
        return a.name in self.WRITE_TOOLS or b.name in self.WRITE_TOOLS


class ParallelToolExecutor:
    """
    Executes a batch of tool calls with dependency-aware parallelism.

    Algorithm:
    1. Build dependency graph (DAG)
    2. Topological sort into levels
    3. Execute each level in parallel, next level waits for previous
    """

    def __init__(self, tool_registry: dict[str, Any], detector: DependencyDetector = None):
        self.registry = tool_registry
        self.detector = detector or DependencyDetector()

    async def execute_batch(self, calls: list[ToolCall]) -> list[ToolResult]:
        if not calls:
            return []

        # Build adjacency list: call_id -> set of call_ids that must come BEFORE it
        n = len(calls)
        in_edges: dict[str, set[str]] = {c.id: set() for c in calls}

        for i in range(n):
            for j in range(n):
                if i != j and self.detector.depends_on(calls[i], calls[j]):
                    # calls[j] depends on calls[i]
                    in_edges[calls[j].id].add(calls[i].id)

        # Kahn's topological sort into levels
        levels: list[list[ToolCall]] = []
        remaining = {c.id: c for c in calls}
        completed: set[str] = set()

        while remaining:
            # Find calls with all dependencies satisfied
            ready = [
                c for c in remaining.values()
                if in_edges[c.id].issubset(completed)
            ]
            if not ready:
                # Cycle detected — fall back to sequential
                ready = list(remaining.values())[:1]
            levels.append(ready)
            for c in ready:
                del remaining[c.id]
                completed.add(c.id)

        # Execute level by level
        all_results: list[ToolResult] = []
        for level in levels:
            level_results = await asyncio.gather(
                *[self._execute_one(call) for call in level],
                return_exceptions=False,
            )
            all_results.extend(level_results)

        return all_results

    async def _execute_one(self, call: ToolCall) -> ToolResult:
        fn = self.registry.get(call.name)
        if fn is None:
            return ToolResult(
                tool_use_id=call.id,
                content=f"[Error] Unknown tool: {call.name!r}",
            )
        try:
            result = await fn(**call.arguments)
            return ToolResult(tool_use_id=call.id, content=str(result))
        except Exception as exc:
            return ToolResult(
                tool_use_id=call.id,
                content=f"[Tool error] {type(exc).__name__}: {exc}",
            )

7. Tool Routing & Deferred Loading

The Cost of Large Tool Schemas ▼

Every tool definition is serialised into the API request, consuming input tokens before the model generates a single output token. This has three cascading effects:

Token cost of a naive "dump all tools" approach:
If each tool schema averages 150 tokens and you register 100 tools:
150 × 100 = 15,000 tokens consumed per turn regardless of which tools are used.
At $15 / M input tokens (Sonnet 4), that's $0.225 per turn just in tool overhead.

Beyond cost, there's a quality degradation effect. With >~50 tools in context:

  • The model's tool selection accuracy drops — it starts picking semantically similar but wrong tools
  • "Lost in the middle" attention effects mean tools defined in the middle of the list are selected less frequently
  • The model occasionally hallucinates tool names not in the schema

Three mitigation strategies:

StrategyHow It WorksTradeoff
Context-sensitive filtering Inspect conversation context; inject only tools relevant to the current task phase Requires a router heuristic; wrong filter = tool unavailable when needed
Deferred / lazy loading Register a stub "fetch_tool_schema" tool; model calls it to discover and load specific tools on demand Adds one extra turn per tool discovery; no quality degradation
Tool namespacing Group tools into namespaces (file_*, git_*, web_*); expose only one namespace at a time Model must know which namespace to request; works well with a task-planning step

This is exactly what Claude Code's ToolSearch tool implements: at startup only a minimal set of tools is loaded; a ToolSearch tool is always available that lets the model discover and fetch the full schema for deferred tools. This keeps the default tool payload small while preserving access to the full toolset.

Tool Registry with Context-Sensitive Filtering ▼ 
from dataclasses import dataclass, field
from enum import Enum, auto
from typing import Optional, Callable, Awaitable, Any
import re

class ToolPhase(Enum):
    """Coarse task phase — used for context-sensitive tool filtering."""
    EXPLORATION = auto()   # Reading, listing, searching
    EDITING     = auto()   # Writing, editing files
    EXECUTION   = auto()   # Running commands, tests
    VALIDATION  = auto()   # Reviewing, diffing, linting
    ANY         = auto()   # Always available

@dataclass
class RegisteredTool:
    schema:      dict                    # Full JSON schema
    handler:     Callable               # Async callable
    phases:      list[ToolPhase] = field(default_factory=lambda: [ToolPhase.ANY])
    tags:        list[str]       = field(default_factory=list)
    deferred:    bool            = False  # Only load on explicit request

class ToolRegistry:
    """
    Central registry for all tools.
    Supports phase-based filtering and deferred (lazy) tool loading.
    """

    def __init__(self):
        self._tools: dict[str, RegisteredTool] = {}

    def register(
        self,
        schema: dict,
        handler: Callable,
        phases: list[ToolPhase] = None,
        tags: list[str] = None,
        deferred: bool = False,
    ):
        name = schema["name"]
        self._tools[name] = RegisteredTool(
            schema=schema,
            handler=handler,
            phases=phases or [ToolPhase.ANY],
            tags=tags or [],
            deferred=deferred,
        )

    def get_schemas_for_phase(
        self,
        phase: ToolPhase,
        include_deferred: bool = False,
    ) -> list[dict]:
        """Return tool schemas relevant for the given phase."""
        result = []
        for tool in self._tools.values():
            if tool.deferred and not include_deferred:
                continue
            if ToolPhase.ANY in tool.phases or phase in tool.phases:
                result.append(tool.schema)
        return result

    def get_handler(self, name: str) -> Optional[Callable]:
        t = self._tools.get(name)
        return t.handler if t else None

    def get_deferred_stub_schema(self) -> dict:
        """
        Schema for the ToolSearch tool — lets the model discover
        and fetch deferred tool definitions on demand.
        """
        deferred_names = [
            name for name, t in self._tools.items() if t.deferred
        ]
        return {
            "name": "ToolSearch",
            "description": (
                "Fetches full schema definitions for deferred tools so they can be called. "
                "Deferred tools appear by name only until fetched — there is no parameter schema. "
                "Use this when you need a tool that is not yet in your schema. "
                f"Available deferred tools: {', '.join(deferred_names)}"
            ),
            "input_schema": {
                "type": "object",
                "properties": {
                    "query": {
                        "type": "string",
                        "description": (
                            "Tool name or keyword. "
                            "Use 'select:ToolName' to fetch a specific tool by name."
                        ),
                    },
                    "max_results": {
                        "type": "integer",
                        "description": "Maximum number of tool schemas to return.",
                        "default": 3,
                    },
                },
                "required": ["query"],
            },
        }

    async def handle_tool_search(self, query: str, max_results: int = 3) -> str:
        """Handler for the ToolSearch meta-tool."""
        import json

        if query.startswith("select:"):
            names = [n.strip() for n in query[7:].split(",")]
        else:
            # Keyword search against name and description
            words = query.lower().split()
            names = [
                name for name, t in self._tools.items()
                if any(w in name.lower() or w in t.schema.get("description", "").lower()
                       for w in words)
            ][:max_results]

        schemas = []
        for name in names[:max_results]:
            if name in self._tools:
                schemas.append(self._tools[name].schema)

        if not schemas:
            return f"No tools found matching: {query!r}"

        return (
            "Fetched tool schemas (now callable):\n"
            + "\n".join(f"<function>{json.dumps(s)}</function>" for s in schemas)
        )


# ── Example setup ─────────────────────────────────────────────────────
async def example_setup():
    registry = ToolRegistry()

    registry.register(
        schema={"name": "Read", "description": "...", "input_schema": {}},
        handler=lambda file_path, **kw: open(file_path).read(),
        phases=[ToolPhase.EXPLORATION, ToolPhase.VALIDATION],
    )
    registry.register(
        schema={"name": "Bash", "description": "...", "input_schema": {}},
        handler=lambda command, **kw: __import__("subprocess").getoutput(command),
        phases=[ToolPhase.EXECUTION],
    )
    # Deferred — not sent until model explicitly calls ToolSearch
    registry.register(
        schema={"name": "WebFetch", "description": "Fetches a URL...", "input_schema": {}},
        handler=lambda url, **kw: None,
        phases=[ToolPhase.EXPLORATION],
        deferred=True,
    )

    # What you'd send for an exploration-phase turn:
    schemas = registry.get_schemas_for_phase(ToolPhase.EXPLORATION)
    schemas.append(registry.get_deferred_stub_schema())
    return schemas

8. Interview Tips

What Interviewers Are Testing ▼

Tool design questions at staff-level interviews probe three competencies simultaneously:

  1. Systems thinking: Can you reason about the full data flow — from model output, through execution, through result formatting, back into context — as a system with latency, cost, and reliability properties?
  2. ML product intuition: Do you understand that the model is a consumer of your schemas and that schema quality directly affects product quality? Interviewers look for candidates who treat the description as a first-class engineering artefact.
  3. Production engineering depth: Do you know about timeouts, sandboxing, context budgets, retry strategies, and graceful degradation — not just the happy path?
Strong signal phrases to use: "The description is a prompt the model reads every turn, so I treat it as a hyperparameter and A/B test it." "I classify errors before surfacing them to the model — retryable vs fatal — because an empty error response causes the model to loop." "Deferred tool loading keeps the base schema small, which matters both for cost and for tool selection quality with 50+ tools."

Common Follow-up Questions

  • "How would you prevent a model from using a dangerous tool like 'delete_file' accidentally?" — Discuss: description negative instructions, permission system, confirmation gates (Module 6)
  • "If tool call latency is 200 ms and you have 100 turns, what's the impact?" — Calculate: 100 × 200 ms = 20 s. Discuss parallel execution, caching, eager tool pre-execution
  • "How do you handle a tool that returns 50,000 tokens of output?" — Truncation policies, head+tail strategy, streaming results, pagination via follow-up calls
  • "What happens if the model hallucinates a tool name that doesn't exist?" — Discuss: registry validation, returning explicit "Unknown tool" message (not raising an exception), logging for schema debugging

Red Flags to Avoid

  • Treating tool schemas as "just JSON config" with no engineering investment
  • No mention of timeouts, resource limits, or output size caps
  • Proposing to return raw exceptions / stack traces to the model
  • Not considering context budget when designing tool result formats
  • Assuming all tools execute sequentially without considering parallelism

9. Knowledge Check

Q1. A model is calling search_codebase when it should be calling search_web. Both tools have similar names. What is the most targeted fix?

Add a mutual disambiguation line to both descriptions. In search_codebase: "Do NOT use to search the internet or external documentation — use search_web for that." In search_web: "Do NOT use to search local files or source code — use search_codebase for that." Explicit negative instructions outperform relying on the model to infer the distinction from the tool name alone.

Q2. Your agent executes 4 tool calls per turn on average, each taking 80 ms. The task takes 40 turns. What is the minimum possible time spent on tool execution, assuming optimal parallelism?

With optimal parallelism, all calls within a turn execute concurrently. The wall-clock cost per turn is max(call latencies) = 80 ms (since all calls take the same time). Over 40 turns: 40 × 80 ms = 3,200 ms (3.2 seconds). Naively sequential: 40 × 4 × 80 ms = 12,800 ms. The parallel gain is 4x here. In practice, dependency constraints reduce this gain somewhat.

Q3. A file read tool returns the full 10,000-line output of a minified JavaScript file. What truncation strategy preserves the most signal for a "find the bug" task?

For a minified file, head+tail truncation gives little value since the file has no semantic structure at line boundaries. Better strategies: (1) ask the model to search with Grep for specific patterns rather than reading the whole file, (2) return only the first N characters with a hint to use offset+limit for pagination, or (3) pre-process the file through a formatter/prettifier and then truncate. The key insight is that truncation policy should be content-aware, not uniform.

Q4. Your tool executor receives a 429 Too Many Requests response from an external API. How should the harness handle this, and what should the model receive?

A 429 is a retryable / transient error. The harness should: (1) read the Retry-After header if present and wait that duration, otherwise use exponential backoff; (2) retry up to 3 times; (3) only surface the error to the model if all retries exhaust. The message to the model should read: "Tool temporarily unavailable: rate limit exceeded. Retry after a short wait, or try an alternative approach." The model should never receive a raw HTTP 429 response — it adds no actionable information.

Q5. You have 120 tools registered. Tool selection accuracy is dropping. What are two orthogonal mitigations?

Mitigation 1 — Deferred loading: Move rarely-used tools to a deferred list and expose a ToolSearch meta-tool. The model sees only the 20-30 most relevant tools by default and fetches others on demand. Keeps schema small, preserves accuracy.

Mitigation 2 — Phase-based filtering: Detect the current task phase (exploration, editing, execution) and inject only the tool subset appropriate for that phase. A model in "editing" mode doesn't need web search tools, and a model in "exploration" mode doesn't need deployment tools. Reduces both token cost and selection confusion.

Module 3 Checkpoint

tl;dr: Tools are the interface between the model and the world. The description is a prompt, not documentation — every token is re-read every turn. Execution engines come in four flavours (in-process, subprocess, network, sandboxed); always enforce timeouts and output caps. Smart truncation preserves information density within context budgets. Errors must be classified and formatted for the model, never surfaced as raw exceptions. Parallel execution is a 3-5x latency win for multi-tool turns. Deferred loading solves the quality degradation problem when you have more than ~50 tools.

Key Takeaways

  • JSON Schema fields are your only communication channel with the model — invest in them
  • Negative instructions ("Do NOT use for...") resolve ambiguity between similar tools
  • A/B test descriptions using a fingerprint hash logged alongside tool call outcomes
  • Classify errors before surfacing to the model: retryable, user-fixable, usage error, fatal
  • Parallel tool execution: use a dependency graph; serialize only calls that share a write path
  • Head+tail truncation for files; first-K truncation for search results; always preserve error lines
  • Deferred tool loading keeps schemas small; expose a ToolSearch meta-tool for discovery
