Scaling Agent Systems

Traditional web services scale by adding stateless servers behind a load balancer. Agent systems break every assumption that makes that easy: requests last minutes, not milliseconds; each turn consumes thousands of tokens; sessions carry stateful context that can't move mid-flight; and every unit of usage has a direct dollar cost billed by a third-party API. This module teaches you how to reason about, architect, and operate agent systems at scale.

What You'll Learn

  • Why agent scaling is fundamentally different from stateless web scaling
  • Horizontal worker pool design and session affinity strategies
  • Queue-based architectures with priority tiers and backpressure
  • Token-bucket rate limiting and graceful degradation under load
  • Prompt caching internals and cache-hit-rate optimization
  • HTTP/2 connection pool sizing for model APIs
  • Back-of-envelope cost modeling at 1K / 10K / 100K DAU
  • Top-ranked cost optimization strategies by ROI

1. Scaling Challenges Unique to Agent Systems

Before designing solutions, you need to deeply understand why existing playbooks fail. Here are the five fundamental differences between scaling a REST API and scaling an agent harness.

Why Agent Workloads Break Standard Scaling Assumptions
1

Long-running requests (minutes, not milliseconds). A typical REST API handler completes in <100 ms. An agent coding session can run for 10–20 minutes across dozens of tool calls. This wrecks assumptions baked into HTTP proxies (nginx's 60 s default timeout), load balancers, and connection draining logic. You cannot drain a worker in 30 seconds when it has 15-minute sessions in progress.

2

High and unpredictable token throughput. A 5-turn agent session with tool results can consume 50 K–200 K tokens. Anthropic's Claude API caps at 1 M tokens per minute (TPM) across all calls from your account. A single burst of 20 concurrent sessions each hitting claude-opus can saturate that limit in seconds, causing 429s that cascade into retries, compounding the overload.

3

Stateful sessions that can't be load balanced mid-flight. Each agent turn appends to a growing conversation history. The next turn must see the complete prior context. If you route turn N to worker A and turn N+1 to worker B, worker B has no context. You either store context in shared persistent storage (adds latency per turn) or implement session affinity so all turns for a session hit the same worker (limits horizontal scalability).

4

Unpredictable per-request resource consumption. A "list files" tool call takes 2 ms. A "run all tests" tool call takes 45 s and pegs a CPU core. A simple question uses 800 tokens. An architectural analysis request uses 80 K tokens. You cannot pre-allocate resources with the determinism you rely on for traditional services. P99 latency for agent requests can be 1,000× P50.

5

Cost scales with usage in non-linear ways. A database query costs the same whether your service handles 100 or 100 K QPS (mostly). LLM API cost scales linearly with tokens consumed, which scales with both user count AND how deeply each user engages. A power user who runs 10 long coding sessions per day can cost 100× more than a light user. Cost spikes are hard to predict and easy to accidentally create (e.g., a bug that causes the agent to loop and regenerate the same large context repeatedly).

Analogy: Think of standard web servers as a fleet of taxis: any available cab picks up any passenger, rides are short, and cars are interchangeable. Agent workers are more like dedicated chauffeurs assigned to a client for the day: the passenger expects continuity, the ride is long, and you can't swap drivers mid-journey without briefing the new one on the entire trip so far.

2. Horizontal Worker Scaling

The goal is to serve N concurrent agent sessions across a pool of workers while maintaining session continuity and enabling elastic scale-out when demand grows.

Worker Pool Architecture and Session Affinity

Stateless vs. Stateful Workers

Stateless Workers

No in-memory session state. Each turn fetches context from a shared store (Redis, DynamoDB) before calling the model, then writes updated context back after.

+ Any worker can serve any turn. True horizontal scale.

- 2 extra network round-trips per turn (read + write). ~5–20 ms added latency.

Stateful Workers (Session Affinity)

Session state held in worker memory. The load balancer routes all turns for a given session to the same worker using a session ID in the header or cookie.

+ No external state store needed per turn. Lower latency.

- Worker death = session loss. Uneven load if long sessions cluster. Complicates scale-in.

Production pattern: Use stateful workers for interactive latency-sensitive sessions (streaming, real-time UI) but checkpoint context to Redis every N turns. On worker failure or scale-in, sessions migrate to a new worker which bootstraps from the checkpoint. This is "mostly stateful" — you pay the Redis write cost only at checkpoints, not every turn.

Auto-Scaling Based on Queue Depth

The right scaling signal for agent systems is pending work in the queue, not CPU or memory. An agent worker can sit at 5% CPU while a tool call blocks on a slow external API. CPU-based autoscaling would not trigger even though users are waiting.

Scaling Trigger Formula target_worker_count = ceil(queue_depth / target_sessions_per_worker)

Example: queue_depth = 120, target_sessions_per_worker = 10
→ target_worker_count = ceil(120 / 10) = 12 workers

Add scale-up buffer: provision 20% extra
→ target_worker_count = 12 * 1.2 = 15 workers

Min workers (always warm): 3 | Max workers (cost cap): 50

Worker Pool Manager — Code

"""
WorkerPoolManager: manages a pool of agent worker processes/containers,
routes sessions with affinity, and triggers autoscaling based on queue depth.
"""
import asyncio
import time
import math
import random
from dataclasses import dataclass, field
from typing import Dict, Optional
from collections import defaultdict


@dataclass
class Worker:
    worker_id: str
    max_sessions: int = 10
    active_sessions: int = 0
    started_at: float = field(default_factory=time.time)
    is_draining: bool = False   # no new sessions assigned during drain

    @property
    def available_capacity(self) -> int:
        if self.is_draining:
            return 0
        return self.max_sessions - self.active_sessions

    @property
    def utilization(self) -> float:
        return self.active_sessions / self.max_sessions


class WorkerPoolManager:
    def __init__(
        self,
        min_workers: int = 3,
        max_workers: int = 50,
        target_sessions_per_worker: int = 10,
        scale_up_buffer: float = 1.2,
        scale_up_threshold_seconds: int = 30,
        scale_down_cooldown_seconds: int = 300,
    ):
        self.min_workers = min_workers
        self.max_workers = max_workers
        self.target_sessions_per_worker = target_sessions_per_worker
        self.scale_up_buffer = scale_up_buffer
        self.scale_up_threshold_seconds = scale_up_threshold_seconds
        self.scale_down_cooldown_seconds = scale_down_cooldown_seconds

        self.workers: Dict[str, Worker] = {}
        # session_id -> worker_id for affinity routing
        self.session_affinity: Dict[str, str] = {}
        self._last_scale_down = 0.0
        self._worker_counter = 0

    # ------------------------------------------------------------------ #
    # Session routing
    # ------------------------------------------------------------------ #

    def assign_worker(self, session_id: str) -> Optional[str]:
        """Return worker_id for a session, creating affinity on first call."""
        # Honour existing affinity
        if session_id in self.session_affinity:
            wid = self.session_affinity[session_id]
            if wid in self.workers and not self.workers[wid].is_draining:
                return wid
            # Worker gone or draining; re-assign
            del self.session_affinity[session_id]

        # Pick least-loaded worker with capacity
        candidates = [
            w for w in self.workers.values()
            if w.available_capacity > 0
        ]
        if not candidates:
            return None  # caller must queue the request

        # Least-connections selection
        chosen = min(candidates, key=lambda w: w.active_sessions)
        chosen.active_sessions += 1
        self.session_affinity[session_id] = chosen.worker_id
        return chosen.worker_id

    def release_session(self, session_id: str) -> None:
        """Call when a session completes or errors out."""
        wid = self.session_affinity.pop(session_id, None)
        if wid and wid in self.workers:
            self.workers[wid].active_sessions = max(
                0, self.workers[wid].active_sessions - 1
            )

    # ------------------------------------------------------------------ #
    # Autoscaling
    # ------------------------------------------------------------------ #

    def compute_desired_workers(self, queue_depth: int) -> int:
        active_sessions = sum(w.active_sessions for w in self.workers.values())
        total_load = active_sessions + queue_depth
        raw = math.ceil(total_load / self.target_sessions_per_worker)
        with_buffer = math.ceil(raw * self.scale_up_buffer)
        return max(self.min_workers, min(self.max_workers, with_buffer))

    async def reconcile(self, queue_depth: int) -> Dict:
        """
        Called by a control loop every ~10 s.
        Returns a dict describing the scaling action taken.
        """
        desired = self.compute_desired_workers(queue_depth)
        current = len(self.workers)
        actions = {"desired": desired, "current": current, "action": "none"}

        if desired > current:
            # Scale up immediately
            to_add = desired - current
            for _ in range(to_add):
                await self._spawn_worker()
            actions["action"] = f"scale_up +{to_add}"

        elif desired < current:
            # Scale down with cooldown to avoid flapping
            now = time.time()
            if now - self._last_scale_down > self.scale_down_cooldown_seconds:
                to_remove = current - desired
                # Drain emptiest workers first (least disruptive)
                candidates = sorted(
                    self.workers.values(),
                    key=lambda w: w.active_sessions
                )
                for worker in candidates[:to_remove]:
                    await self._drain_worker(worker.worker_id)
                self._last_scale_down = now
                actions["action"] = f"scale_down -{to_remove} (draining)"

        return actions

    async def _spawn_worker(self) -> Worker:
        self._worker_counter += 1
        wid = f"worker-{self._worker_counter:04d}"
        # In production this calls your container orchestrator
        # (ECS RunTask, k8s Deployment scale, etc.)
        print(f"[pool] Spawning {wid}")
        worker = Worker(worker_id=wid)
        self.workers[wid] = worker
        return worker

    async def _drain_worker(self, worker_id: str) -> None:
        """Mark worker for draining; it will be removed when sessions = 0."""
        if worker_id in self.workers:
            self.workers[worker_id].is_draining = True
            print(f"[pool] Draining {worker_id} "
                  f"({self.workers[worker_id].active_sessions} active sessions)")
            # A background task watches for active_sessions == 0 then deregisters


# Usage example
async def demo():
    pool = WorkerPoolManager(min_workers=2, max_workers=20)
    # Initialise minimum workers
    for _ in range(2):
        await pool._spawn_worker()

    # Simulate 15 incoming sessions
    for i in range(15):
        wid = pool.assign_worker(f"session-{i}")
        print(f"session-{i} -> {wid}")

    # Reconcile with queue depth 30
    result = await pool.reconcile(queue_depth=30)
    print(f"Scaling result: {result}")

asyncio.run(demo())

3. Queue-Based Architectures

For non-interactive (batch) agent tasks — nightly code reviews, automated test generation, document summarisation — a queue decouples submission from execution, provides natural backpressure, and allows priority-based scheduling.

Priority Queues, Dead Letter Queues, and Backpressure

Queue Tiers

Tier Use case Max wait target Technology
P0 – Interactive Streaming chat turns, real-time inline completions < 200 ms Redis LPUSH/BRPOP (in-memory)
P1 – Near-real-time Agent task dispatch (user-initiated, UI showing spinner) < 5 s Redis or SQS FIFO
P2 – Background Automated code review, indexing jobs < 2 min SQS Standard or Kafka topic
P3 – Batch Nightly bulk analysis, training data generation < 1 hr Kafka or managed batch (AWS Batch)
Anti-pattern: Putting all workloads in a single queue. A burst of P3 batch jobs will starve P1 interactive sessions because workers drain the queue in FIFO order. Always implement at least two tiers: interactive and batch.

Dead Letter Queues (DLQ)

Agent tasks fail in non-obvious ways: a tool call hits a timeout, the model returns malformed JSON, a sandbox OOMs. Without a DLQ, failed messages are either silently dropped or loop forever consuming retries.

1

Set maxReceiveCount = 3 on SQS (or delivery.attempt.limit = 3 on Kafka). After 3 failures the message moves to the DLQ.

2

Alert on DLQ depth > 0. Every message in the DLQ is a user task that silently failed.

3

DLQ messages retain the full task payload plus error metadata. A human or automated remediation worker can inspect, fix, and replay them.

QueuedAgentDispatcher — Code

"""
QueuedAgentDispatcher: wraps SQS (or a local asyncio.Queue for dev)
with priority tiers, retry logic, and DLQ forwarding.
"""
import asyncio
import json
import time
import uuid
from dataclasses import dataclass, field
from enum import IntEnum
from typing import Any, Callable, Awaitable, Dict, Optional
import logging

logger = logging.getLogger(__name__)


class Priority(IntEnum):
    INTERACTIVE = 0   # P0/P1 – user waiting
    BACKGROUND  = 1   # P2 – user not watching
    BATCH       = 2   # P3 – scheduled


@dataclass
class AgentTask:
    task_id: str = field(default_factory=lambda: str(uuid.uuid4()))
    session_id: str = ""
    payload: Dict[str, Any] = field(default_factory=dict)
    priority: Priority = Priority.BACKGROUND
    attempt: int = 0
    max_attempts: int = 3
    enqueued_at: float = field(default_factory=time.time)

    @property
    def should_dlq(self) -> bool:
        return self.attempt >= self.max_attempts


class QueuedAgentDispatcher:
    def __init__(self, max_workers_per_tier: int = 5):
        # Three in-memory priority queues (asyncio.PriorityQueue sorts ascending)
        self._queues: Dict[Priority, asyncio.Queue] = {
            Priority.INTERACTIVE: asyncio.Queue(maxsize=500),
            Priority.BACKGROUND:  asyncio.Queue(maxsize=2000),
            Priority.BATCH:       asyncio.Queue(maxsize=10000),
        }
        self._dlq: asyncio.Queue = asyncio.Queue()
        self._max_workers = max_workers_per_tier
        self._stats = {"processed": 0, "failed": 0, "dlq": 0}

    async def enqueue(self, task: AgentTask) -> None:
        """Non-blocking enqueue. Raises QueueFull if tier is at capacity."""
        q = self._queues[task.priority]
        try:
            q.put_nowait(task)
            logger.info(
                f"Enqueued task={task.task_id} priority={task.priority.name} "
                f"queue_depth={q.qsize()}"
            )
        except asyncio.QueueFull:
            # Apply backpressure: reject or spill to lower-priority queue
            logger.warning(
                f"Queue full for {task.priority.name}, "
                f"returning 429 to caller"
            )
            raise

    async def _process_task(
        self,
        task: AgentTask,
        handler: Callable[[AgentTask], Awaitable[None]],
    ) -> None:
        try:
            await asyncio.wait_for(handler(task), timeout=1200)  # 20 min max
            self._stats["processed"] += 1
        except (asyncio.TimeoutError, Exception) as exc:
            task.attempt += 1
            logger.warning(
                f"Task {task.task_id} failed attempt {task.attempt}: {exc}"
            )
            if task.should_dlq:
                logger.error(f"Moving {task.task_id} to DLQ after {task.attempt} attempts")
                await self._dlq.put(task)
                self._stats["dlq"] += 1
            else:
                # Exponential back-off before re-queue
                backoff = 2 ** task.attempt
                await asyncio.sleep(backoff)
                await self._queues[task.priority].put(task)
            self._stats["failed"] += 1

    async def _worker_loop(
        self,
        priority: Priority,
        handler: Callable[[AgentTask], Awaitable[None]],
    ) -> None:
        q = self._queues[priority]
        while True:
            task = await q.get()
            await self._process_task(task, handler)
            q.task_done()

    async def start(
        self,
        handler: Callable[[AgentTask], Awaitable[None]],
    ) -> None:
        """Launch worker coroutines for all tiers."""
        workers = []
        for priority in Priority:
            for _ in range(self._max_workers):
                workers.append(
                    asyncio.create_task(self._worker_loop(priority, handler))
                )
        await asyncio.gather(*workers)

    def queue_depths(self) -> Dict[str, int]:
        return {
            p.name: self._queues[p].qsize()
            for p in Priority
        } | {"DLQ": self._dlq.qsize()}

4. Rate Limiting and Backpressure

The Anthropic API enforces per-account rate limits measured in tokens per minute (TPM) and requests per minute (RPM). Exceeding them triggers 429 responses. Without client-side rate limiting, your retry storm from a 429 makes the problem worse.

Token Bucket Rate Limiter and Graceful Degradation

Anthropic API Limits (as of 2025)

ModelRPMTPMContext window
claude-3-5-haiku4 0002 000 000200 K tokens
claude-3-5-sonnet2 0001 600 000200 K tokens
claude-opus-41 000800 000200 K tokens
Real failure mode: At 50 concurrent long agent sessions on claude-opus, each averaging 10 K tokens/turn and executing a turn every 30 s, you consume 50 × 10 K / 30 s = ~16 667 TPM. That's comfortably under 800 K TPM. But if those 50 sessions all start simultaneously after a queue flush, turn 1 for all 50 fires within seconds: 50 × 10 K = 500 K tokens in one burst, potentially over the per-minute bucket. Always model burst patterns, not just steady-state.

Token Bucket Algorithm

A token bucket holds up to capacity tokens. Tokens refill at a constant rate. Each API call draws tokens proportional to the estimated token count. If the bucket is empty, the caller waits (or gets rejected, depending on policy).

"""
TokenBucketRateLimiter: enforces both TPM and RPM limits against the
Anthropic (or any) model API. Thread-safe via asyncio.Lock.
"""
import asyncio
import time
import logging
from dataclasses import dataclass
from typing import Optional

logger = logging.getLogger(__name__)


@dataclass
class RateLimitConfig:
    rpm: int           # requests per minute
    tpm: int           # tokens per minute
    burst_factor: float = 1.5  # allow short bursts above steady state


class TokenBucketRateLimiter:
    def __init__(self, config: RateLimitConfig):
        # RPM bucket
        self._rpm_capacity = config.rpm * config.burst_factor
        self._rpm_tokens = float(self._rpm_capacity)
        self._rpm_refill_rate = config.rpm / 60.0  # tokens per second

        # TPM bucket
        self._tpm_capacity = config.tpm * config.burst_factor
        self._tpm_tokens = float(self._tpm_capacity)
        self._tpm_refill_rate = config.tpm / 60.0  # tokens per second

        self._last_refill = time.monotonic()
        self._lock = asyncio.Lock()

    def _refill(self) -> None:
        now = time.monotonic()
        elapsed = now - self._last_refill
        self._last_refill = now

        self._rpm_tokens = min(
            self._rpm_capacity,
            self._rpm_tokens + elapsed * self._rpm_refill_rate
        )
        self._tpm_tokens = min(
            self._tpm_capacity,
            self._tpm_tokens + elapsed * self._tpm_refill_rate
        )

    async def acquire(
        self,
        estimated_tokens: int,
        timeout: float = 60.0,
    ) -> float:
        """
        Wait until the bucket has capacity, then consume.
        Returns the wait time incurred (seconds).
        Raises asyncio.TimeoutError if wait exceeds timeout.
        """
        start = time.monotonic()
        deadline = start + timeout

        while True:
            async with self._lock:
                self._refill()

                rpm_wait = max(0.0, (1 - self._rpm_tokens) / self._rpm_refill_rate)
                tpm_wait = max(0.0, (estimated_tokens - self._tpm_tokens) / self._tpm_refill_rate)
                wait_needed = max(rpm_wait, tpm_wait)

                if wait_needed == 0:
                    self._rpm_tokens -= 1
                    self._tpm_tokens -= estimated_tokens
                    waited = time.monotonic() - start
                    if waited > 0.1:
                        logger.warning(
                            f"Rate limit: waited {waited:.2f}s for {estimated_tokens} tokens"
                        )
                    return waited

            # Release lock while sleeping so other coroutines can check
            remaining = deadline - time.monotonic()
            if wait_needed > remaining:
                raise asyncio.TimeoutError(
                    f"Rate limit wait {wait_needed:.1f}s exceeds timeout {timeout}s"
                )
            await asyncio.sleep(min(wait_needed, 0.5))  # wake up early to re-check


class BackpressureHandler:
    """
    Translates rate limit pressure into user-visible feedback
    and graceful degradation strategies.
    """
    def __init__(self, limiter: TokenBucketRateLimiter):
        self.limiter = limiter
        self._rejected_count = 0

    async def call_with_backpressure(
        self,
        estimated_tokens: int,
        api_call,
        user_callback=None,
    ):
        """
        Attempt the API call with rate-limit-aware backpressure.
        Degradation tiers:
          - wait < 2 s   : transparent to user
          - wait 2-10 s  : show "thinking..." spinner update
          - wait > 10 s  : queue the request, notify user of estimated wait
          - timeout       : reject with 503, suggest retry-after header
        """
        try:
            waited = await self.limiter.acquire(
                estimated_tokens, timeout=30.0
            )
            if waited > 2.0 and user_callback:
                await user_callback(
                    f"High demand – your request waited {waited:.0f}s in queue"
                )
            return await api_call()
        except asyncio.TimeoutError:
            self._rejected_count += 1
            raise ServiceOverloadError(
                "Service is temporarily overloaded. Please retry in 60 seconds.",
                retry_after=60,
            )


class ServiceOverloadError(Exception):
    def __init__(self, message: str, retry_after: int = 60):
        super().__init__(message)
        self.retry_after = retry_after


# Example configuration for claude-opus-4
opus_limiter = TokenBucketRateLimiter(RateLimitConfig(rpm=1000, tpm=800_000))

5. Prompt Caching at Scale

Anthropic's prompt caching feature allows a portion of the prompt to be cached server-side. On subsequent requests that share the same cache prefix, you pay 10% of the normal input price for cached tokens. At scale, this is often the single highest-ROI optimization available.

How Prompt Caching Works and Cache-Hit Rate Optimization

Technical Mechanism

When you include {"type": "text", "text": "...", "cache_control": {"type": "ephemeral"}} in a message, Anthropic's servers compute a hash of all tokens up to that breakpoint and store the KV-cache state. On the next request that shares the same prefix hash, the prefill phase is skipped for those tokens — cutting TTFT dramatically and reducing your billed input tokens by 90%.

Prompt Cache Economics Normal input price (claude-sonnet): $3.00 / M tokens
Cached input price: $0.30 / M tokens (90% savings)
Cache write surcharge: $3.75 / M tokens (one-time on first request)

Break-even: cache write pays off after 2+ requests use the cache
At 10 requests sharing a 50K-token system prompt:
Without cache: 10 × 50K × $3.00/M = $1.50
With cache: 1 write × 50K × $3.75/M
+ 9 reads × 50K × $0.30/M
= $0.1875 + $0.135 = $0.3225
Savings: 78.5%

Cache Architecture Patterns

Pattern A: Static System Prompt Cache

Place cache_control at the end of your system prompt. Every user shares this cache entry. Works when the system prompt is identical across users.

Cache hit rate: ~99% (once warm)

Pattern B: Per-User Context Cache

Cache at the end of user-specific context (e.g., their codebase summary). Each user has their own cache slot. Cache stays warm for the duration of their session.

Cache hit rate: ~80–95% (degrades if context changes)

Key constraint: The cache is keyed on the exact byte sequence of all tokens before the cache breakpoint. Adding even one token before the breakpoint (e.g., a timestamp, a per-request ID) busts the cache. Design prompts so that volatile content always comes after the last cache breakpoint.

Cache-Optimized Prompt Builder — Code

"""
CacheOptimizedPromptBuilder: constructs Anthropic API message payloads
with cache_control breakpoints placed for maximum cache reuse.

Design principle:
  [STATIC system prompt] ← cache breakpoint 1 (shared by ALL users)
  [User-specific context] ← cache breakpoint 2 (shared within a user session)
  [Current conversation history]
  [New user message]
"""
from dataclasses import dataclass, field
from typing import List, Dict, Any, Optional


@dataclass
class CacheBreakpoint:
    content: str
    is_shared: bool = True  # True = same across multiple users, False = user-specific


class CacheOptimizedPromptBuilder:
    def __init__(self, system_prompt: str):
        """
        system_prompt: the static part shared by all users.
        This is placed with a cache_control breakpoint so all users benefit.
        """
        self._system_prompt = system_prompt
        self._user_context: Optional[str] = None
        self._conversation_history: List[Dict[str, Any]] = []

    def set_user_context(self, context: str) -> "CacheOptimizedPromptBuilder":
        """
        Per-user, per-session context (e.g., repo summary, user preferences).
        Placed with a second cache breakpoint — cached across turns within a session.
        MUST be stable within a session (no timestamps, no per-request IDs).
        """
        self._user_context = context
        return self

    def add_turn(self, role: str, content: str) -> "CacheOptimizedPromptBuilder":
        """Add a conversation turn. These are NOT cached (they change each turn)."""
        self._conversation_history.append({"role": role, "content": content})
        return self

    def build_system(self) -> List[Dict[str, Any]]:
        """
        Returns the 'system' parameter for the Anthropic messages API.
        The static system prompt gets a cache breakpoint.
        If user context exists, it gets a second breakpoint (appended after static prompt).
        """
        blocks = [
            {
                "type": "text",
                "text": self._system_prompt,
                "cache_control": {"type": "ephemeral"},
                # ^ This breakpoint is shared by all users
            }
        ]

        if self._user_context:
            blocks.append(
                {
                    "type": "text",
                    "text": self._user_context,
                    "cache_control": {"type": "ephemeral"},
                    # ^ This breakpoint is specific to this user's session
                }
            )

        return blocks

    def build_messages(self, new_user_message: str) -> List[Dict[str, Any]]:
        """
        Returns the 'messages' parameter for the API call.
        History is passed verbatim (not cached — it changes every turn).
        """
        messages = list(self._conversation_history)
        messages.append({"role": "user", "content": new_user_message})
        return messages

    def estimate_cache_savings(
        self,
        turns_in_session: int,
        system_tokens: int,
        user_context_tokens: int,
        price_per_million_input: float = 3.00,
        price_per_million_cached: float = 0.30,
        price_per_million_cache_write: float = 3.75,
    ) -> Dict[str, float]:
        """Estimate cost savings from caching for a session."""
        cacheable_tokens = system_tokens + user_context_tokens

        # Without cache
        cost_no_cache = (turns_in_session * cacheable_tokens / 1_000_000) * price_per_million_input

        # With cache: 1 write + (N-1) reads for user_context; system is shared
        # (assume system prompt cache is already warm from another user)
        cost_with_cache = (
            (user_context_tokens / 1_000_000) * price_per_million_cache_write  # write once
            + ((turns_in_session - 1) * user_context_tokens / 1_000_000) * price_per_million_cached  # reads
            + (turns_in_session * system_tokens / 1_000_000) * price_per_million_cached  # system always cached
        )

        savings = cost_no_cache - cost_with_cache
        return {
            "cost_no_cache_usd": round(cost_no_cache, 4),
            "cost_with_cache_usd": round(cost_with_cache, 4),
            "savings_usd": round(savings, 4),
            "savings_pct": round(100 * savings / cost_no_cache, 1) if cost_no_cache else 0,
        }

6. Connection Pooling

Every API call to Anthropic over HTTPS/HTTP2 involves a TLS handshake if a new connection is established. At 50+ concurrent agent sessions, each making multiple turns per minute, uncontrolled connection creation causes latency spikes and risks hitting Anthropic's per-connection limits.

HTTP/2 Connection Pool Design and Sizing

Why HTTP/2 Matters for Agent Systems

HTTP/2 supports multiplexing: multiple requests can be in-flight over a single TCP connection simultaneously. For a model API:

  • HTTP/1.1: 1 request per connection. 50 concurrent requests = 50 TCP connections = 50 TLS handshakes (each ~50–100 ms).
  • HTTP/2: N requests per connection. 50 concurrent requests = 5–10 connections. New requests attach to existing connections instantly.

Python's httpx supports HTTP/2 with http2=True. The default requests library does not.

Pool Sizing Heuristics

Connection Pool Sizing max_connections = ceil(max_concurrent_requests / streams_per_connection)

streams_per_connection for Anthropic API: ~25–50 (empirical, check your account)
For 200 concurrent agent turns: ceil(200 / 25) = 8 connections

keepalive_expiry: 60–120 s (balance between reuse and stale connections)
connect_timeout: 5 s
read_timeout: 1200 s (20 min — agent sessions are long)

ModelAPIConnectionPool — Code

"""
ModelAPIConnectionPool: manages an httpx AsyncClient (HTTP/2) with
connection pooling, health checks, and automatic retry on connection errors.
"""
import asyncio
import time
import httpx
import logging
from contextlib import asynccontextmanager
from typing import AsyncIterator, Optional, Dict, Any

logger = logging.getLogger(__name__)


class ModelAPIConnectionPool:
    """
    Singleton-pattern connection pool for Anthropic API calls.
    Use as an async context manager or call connect()/close() manually.

    Usage:
        pool = ModelAPIConnectionPool(api_key="sk-ant-...", max_connections=10)
        async with pool:
            response = await pool.post("/v1/messages", json=payload)
    """

    BASE_URL = "https://api.anthropic.com"

    def __init__(
        self,
        api_key: str,
        max_connections: int = 10,
        max_keepalive_connections: int = 8,
        keepalive_expiry: float = 90.0,
        connect_timeout: float = 5.0,
        read_timeout: float = 1200.0,
        max_retries: int = 3,
    ):
        self._api_key = api_key
        self._client: Optional[httpx.AsyncClient] = None
        self._limits = httpx.Limits(
            max_connections=max_connections,
            max_keepalive_connections=max_keepalive_connections,
            keepalive_expiry=keepalive_expiry,
        )
        self._timeout = httpx.Timeout(
            connect=connect_timeout,
            read=read_timeout,
            write=30.0,
            pool=10.0,
        )
        self._max_retries = max_retries
        self._request_count = 0
        self._error_count = 0
        self._last_health_check = 0.0

    async def connect(self) -> None:
        self._client = httpx.AsyncClient(
            base_url=self.BASE_URL,
            http2=True,
            limits=self._limits,
            timeout=self._timeout,
            headers={
                "x-api-key": self._api_key,
                "anthropic-version": "2023-06-01",
                "content-type": "application/json",
            },
        )
        logger.info(
            f"ModelAPIConnectionPool connected "
            f"(max_connections={self._limits.max_connections}, http2=True)"
        )

    async def close(self) -> None:
        if self._client:
            await self._client.aclose()
            self._client = None
            logger.info("ModelAPIConnectionPool closed")

    async def __aenter__(self):
        await self.connect()
        return self

    async def __aexit__(self, *args):
        await self.close()

    async def post(
        self,
        path: str,
        json: Dict[str, Any],
        attempt: int = 0,
    ) -> httpx.Response:
        if not self._client:
            raise RuntimeError("Pool not connected. Use 'async with pool:' or call connect().")

        self._request_count += 1
        try:
            response = await self._client.post(path, json=json)

            if response.status_code == 429:
                # Rate limited — respect Retry-After header
                retry_after = float(response.headers.get("retry-after", 60))
                logger.warning(f"429 rate limited, retry in {retry_after}s")
                if attempt < self._max_retries:
                    await asyncio.sleep(retry_after)
                    return await self.post(path, json, attempt + 1)

            if response.status_code == 529:
                # Anthropic overloaded — exponential back-off
                if attempt < self._max_retries:
                    backoff = 2 ** attempt * 5
                    logger.warning(f"529 overloaded, retry in {backoff}s")
                    await asyncio.sleep(backoff)
                    return await self.post(path, json, attempt + 1)

            response.raise_for_status()
            return response

        except (httpx.ConnectError, httpx.RemoteProtocolError) as exc:
            self._error_count += 1
            if attempt < self._max_retries:
                await asyncio.sleep(2 ** attempt)
                return await self.post(path, json, attempt + 1)
            raise

    def stats(self) -> Dict[str, Any]:
        return {
            "requests": self._request_count,
            "errors": self._error_count,
            "error_rate": (
                self._error_count / self._request_count
                if self._request_count else 0
            ),
            "active_connections": (
                len(self._client._transport._pool._connections)  # type: ignore
                if self._client else 0
            ),
        }

10. Knowledge Check

Q1. Why is queue depth a better autoscaling signal than CPU utilization for agent workers?

Agent workers frequently block on I/O — waiting for tool execution results, model API responses, or sandbox operations. During these waits, CPU utilization is near zero even though the worker is fully occupied and users are waiting. CPU-based scaling would not trigger, leaving the queue growing. Queue depth directly measures pending user work, regardless of what the workers are blocked on.

Q2. You have 50K tokens of static system prompt used by all users. Where do you place the cache_control breakpoint, and why must nothing volatile appear before it?

Place cache_control: {type: "ephemeral"} at the end of the static system prompt block. Anthropic's cache is keyed on the exact byte sequence of all tokens up to the breakpoint. If even a single token changes (e.g., a request ID, timestamp, or per-user insertion before the breakpoint), the cache key changes and the entry is a miss — you pay full input price AND the cache write surcharge with zero benefit. Keep all volatile/user-specific content strictly after the last cache breakpoint.

Q3. A session is in progress on Worker A when a scale-in event drains it. How do you preserve the session without dropping it?

Two-part strategy: (1) When drain is triggered, mark Worker A with is_draining=True — no new sessions are routed to it, but existing sessions continue. (2) Checkpoint: at regular intervals (or at each turn), serialize the session's conversation history and metadata to Redis with a TTL. If the worker is forced down before draining completes, the Session Router detects the missing affinity target and bootstraps a new worker from the most recent checkpoint. The session resumes from the last known state.

Q4. Your system handles 10K DAU, each running 2 sessions of 15 turns at 12K input / 1.5K output tokens per turn with 70% cache hit rate on input. Estimate the monthly API cost using claude-3-5-sonnet pricing.

Total turns/day = 10 000 × 2 × 15 = 300 000 turns/day

Input tokens/day = 300 000 × 12 000 = 3.6 B tokens
  Cached (70%): 2.52 B tokens @ $0.30/M = $756/day
  Uncached (30%): 1.08 B tokens @ $3.00/M = $3 240/day

Output tokens/day = 300 000 × 1 500 = 450 M tokens @ $15.00/M = $6 750/day

Total/day = $756 + $3 240 + $6 750 = $10 746/day
Monthly = $10 746 × 30 = ~$322 000/month

Q5. What is the break-even point for enabling prompt caching on a 50K-token block at $3.75/M cache write and $0.30/M cached read vs $3.00/M uncached?

Cost of N requests WITHOUT cache: N × 50 000 × $3.00/M = N × $0.15
Cost of N requests WITH cache: (1 × 50 000 × $3.75/M) + ((N-1) × 50 000 × $0.30/M)
= $0.1875 + (N-1) × $0.015

Break-even: N × $0.15 = $0.1875 + (N-1) × $0.015
0.15N = 0.1875 + 0.015N - 0.015
0.135N = 0.1725
N = 1.28 → break-even at 2 requests (caching pays off from the second request onward).
