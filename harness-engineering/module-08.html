Module 8: Observability & Debugging

Master the art of making non-deterministic agent systems inspectable, traceable, and debuggable in production — from OpenTelemetry instrumentation to replay-based debugging and cost circuit breakers.

What You'll Learn

  • Why agent observability fundamentally differs from traditional service observability
  • Design a full OpenTelemetry trace hierarchy for agent loops: session → turn → model call → tool execution
  • Build structured logging schemas with correlation IDs that survive async fan-out
  • Implement accurate per-request cost tracking with real model pricing and budget circuit breakers
  • Measure and alert on TTFT, tool latency, and end-to-end waterfall breakdowns
  • Record and replay agent sessions for deterministic bug reproduction
  • Design production dashboards and on-call playbooks specific to agent workloads

1. Why Agent Observability is Different

The Five Fundamental Challenges ▲

Traditional observability was designed for deterministic systems: a request comes in, a predictable code path executes, a response goes out. Distributed tracing, structured logs, and RED metrics (Rate, Errors, Duration) handle this well. Agent systems break all of those assumptions simultaneously.

1. Non-Determinism

Send the same user message twice and the agent may take entirely different paths. It might call different tools, in a different order, producing a different answer — not because of a bug, but because sampling temperature introduces variance. Classical regression testing that compares exact outputs will produce false failures constantly.

Analogy: Observing a function call is like watching a vending machine — same input (B3), same output (Snickers), every time. Observing an agent conversation is like watching a human customer service rep — same customer complaint, potentially different resolution path every call. You can't assert on the exact transcript; you have to assert on outcomes and quality dimensions.

2. Multi-Step Execution Depth

A single "user message" in a harness like Claude Code can trigger a waterfall of operations:

  • Context assembly (reading files, injecting tools)
  • A model call that returns a tool_use block
  • Tool execution (bash, file read, file write)
  • Injecting tool_result back into context
  • Another model call that may itself return more tool_use blocks
  • Repeat 5–20 more times
  • A final text response

In production, Claude Code sessions regularly involve 20–40 model calls per complex user request. Each call has its own latency, token count, and failure mode. A single span for the entire user request tells you almost nothing useful.

3. Token Economics as a First-Class Signal

Every model call has a hard dollar cost. Unlike CPU cycles (essentially free to use more), tokens cost real money at marginal rates. A naive agent that loops unnecessarily or builds bloated contexts can cost 10x what an efficient agent would for the same task. This means cost is not a billing concern — it is a performance signal that must be tracked at the same granularity as latency.

Cost formula per model call:
cost = (input_tokens × input_price_per_token) + (output_tokens × output_price_per_token)

Example — claude-sonnet-4-5 pricing (approximate):
input_price = $3.00 / 1M tokens = $0.000003 / token
output_price = $15.00 / 1M tokens = $0.000015 / token

A call with 10,000 input + 500 output tokens costs:
(10,000 × 0.000003) + (500 × 0.000015) = $0.03 + $0.0075 = $0.0375

Multiply by 20 calls/session × 100 sessions/hour = $75/hour

4. The Black Box Problem

You cannot inspect the model's reasoning directly. You see inputs and outputs, not the "why." When an agent makes a wrong tool choice or produces a hallucinated answer, the output is the only evidence. This means your observability layer must capture the full input context and the full output — including any chain-of-thought if your prompt elicits it — so post-hoc analysis is possible.

5. Async and Concurrent Execution

Production agent systems often run multiple tool calls in parallel (e.g., reading three files simultaneously), spawn subagents, or operate across multiple processes. A correlation ID that only lives on the initial HTTP request will get lost in async fan-out. You need baggage propagation that flows through every thread, task, and subprocess boundary.

The observability anti-pattern: Many teams bolt on logging as an afterthought — adding print(f"model called with {len(messages)} messages") everywhere. This creates unstructured, uncorrelated noise that is impossible to query in production. Observability for agents must be designed from the ground up, before the first model call is ever made.

2. OpenTelemetry for Agent Loops

Trace Hierarchy Design ▲

OpenTelemetry (OTel) is the industry standard for distributed tracing and is the right foundation for agent observability. The key insight is designing the right span hierarchy — one that maps to the semantic structure of agent execution rather than to HTTP request boundaries.

Baggage Propagation for Session Context

OTel baggage allows you to attach key-value pairs to the trace context that propagate across process boundaries. For agent harnesses that spawn subprocesses (e.g., bash commands), or make HTTP calls to external tools, baggage ensures the session_id and conv_id travel with every operation — even ones that don't use the Python OTel SDK directly.

@dataclass
class AgentSpanContext:
    session_id: str
    conv_id: str
    turn_id: Optional[str] = None

class AgentTracer:
    """Wraps OpenTelemetry spans with agent-semantic naming and attributes."""

    def __init__(self, session_id: str, user_id: str, harness_version: str):
        self.session_id = session_id
        self.user_id = user_id
        self.harness_version = harness_version
        self._session_span = None
        self._session_token = None

    def start_session(self):
        """Create the root session span. Call once at harness startup."""
        ctx = baggage.set_baggage("session_id", self.session_id)
        ctx = baggage.set_baggage("user_id", self.user_id, context=ctx)
        self._session_token = context.attach(ctx)

        self._session_span = tracer.start_span(
            "agent.session",
            attributes={
                "session.id": self.session_id,
                "session.user_id": self.user_id,
                "harness.version": self.harness_version,
                "session.start_time": int(time.time() * 1000),
            }
        )
        return self._session_span

    def end_session(self, total_cost_usd: float, total_turns: int):
        if self._session_span:
            self._session_span.set_attribute("session.total_cost_usd", total_cost_usd)
            self._session_span.set_attribute("session.total_turns", total_turns)
            self._session_span.end()
        if self._session_token:
            context.detach(self._session_token)

    def conversation_span(self, conv_id: str):
        """Returns a context manager for a conversation."""
        return tracer.start_as_current_span(
            "agent.conversation",
            attributes={
                "conv.id": conv_id,
                "session.id": self.session_id,
            }
        )

    def turn_span(self, turn_id: str, user_message: str):
        """Returns a context manager for a single user turn."""
        return tracer.start_as_current_span(
            "agent.turn",
            attributes={
                "turn.id": turn_id,
                "turn.user_message_length": len(user_message),
                # Hash the message — don't store PII in traces
                "turn.user_message_hash": str(hash(user_message) % (10**9)),
            }
        )

    def model_call_span(self, model_id: str, input_tokens: int,
                        output_tokens: int, cost_usd: float,
                        stop_reason: str, ttft_ms: float, latency_ms: float):
        """Record a completed model call. Must be called within a turn span."""
        span = trace.get_current_span()
        # Emit a child event rather than a full span for completed model calls
        # (reduces span cardinality while preserving all data)
        with tracer.start_as_current_span(
            "agent.model_call",
            attributes={
                "model.id": model_id,
                "model.input_tokens": input_tokens,
                "model.output_tokens": output_tokens,
                "model.cost_usd": cost_usd,
                "model.stop_reason": stop_reason,
                "model.ttft_ms": ttft_ms,
                "model.latency_ms": latency_ms,
            }
        ) as model_span:
            yield model_span

    def tool_span(self, tool_name: str, args_hash: str):
        """Context manager for a single tool execution."""
        return tracer.start_as_current_span(
            "agent.tool_execution",
            attributes={
                "tool.name": tool_name,
                "tool.args_hash": args_hash,
            }
        )


# ---- Usage example in the main agent loop ----
async def run_agent_turn(
    agent_tracer: AgentTracer,
    conv_id: str,
    turn_id: str,
    user_message: str,
    model_client,
    tools: dict
):
    with agent_tracer.conversation_span(conv_id):
        with agent_tracer.turn_span(turn_id, user_message) as turn_span:
            model_calls_count = 0
            turn_cost = 0.0
            messages = [{"role": "user", "content": user_message}]

            while True:
                t_start = time.perf_counter()
                response = await model_client.call(messages)
                latency_ms = (time.perf_counter() - t_start) * 1000

                cost = compute_cost(
                    response.model, response.usage.input_tokens,
                    response.usage.output_tokens
                )
                turn_cost += cost
                model_calls_count += 1

                with tracer.start_as_current_span(
                    "agent.model_call",
                    attributes={
                        "model.id": response.model,
                        "model.input_tokens": response.usage.input_tokens,
                        "model.output_tokens": response.usage.output_tokens,
                        "model.cost_usd": cost,
                        "model.stop_reason": response.stop_reason,
                        "model.latency_ms": latency_ms,
                        "model.call_index": model_calls_count,
                    }
                ):
                    if response.stop_reason == "end_turn":
                        break

                    for tool_use in response.tool_uses:
                        tool_fn = tools[tool_use.name]
                        t_tool_start = time.perf_counter()
                        try:
                            result = await tool_fn(tool_use.input)
                            success = True
                            error_type = None
                        except Exception as e:
                            result = str(e)
                            success = False
                            error_type = type(e).__name__

                        tool_latency_ms = (time.perf_counter() - t_tool_start) * 1000

                        with tracer.start_as_current_span(
                            "agent.tool_execution",
                            attributes={
                                "tool.name": tool_use.name,
                                "tool.duration_ms": tool_latency_ms,
                                "tool.success": success,
                                "tool.output_bytes": len(str(result)),
                                "tool.error_type": error_type or "",
                            }
                        ):
                            pass  # span auto-ends on exit

                        messages.append({
                            "role": "assistant",
                            "content": [{"type": "tool_use", "id": tool_use.id, "name": tool_use.name, "input": tool_use.input}]
                        })
                        messages.append({
                            "role": "user",
                            "content": [{"type": "tool_result", "tool_use_id": tool_use.id, "content": str(result)}]
                        })

            turn_span.set_attribute("turn.total_cost_usd", turn_cost)
            turn_span.set_attribute("turn.model_calls_count", model_calls_count)

3. Structured Logging

Log Schema, Event Types, and Correlation ▲

Traces capture the structure of execution; logs capture the content — what the model actually said, what tools returned, what errors occurred. For agent systems, logs must be structured (JSON), correlated (tied to trace IDs), and schema-versioned (so you can query historical data after schema changes).

The Agent Log Schema

Every log event in an agent system should carry a common envelope:

{
  "timestamp": "2025-04-27T14:32:01.234Z",
  "level": "INFO",
  "event_type": "model_call",
  "schema_version": "2025-04-01",

  // Correlation IDs — must be present on ALL events
  "session_id": "sess_abc123",
  "conv_id": "conv_xyz789",
  "turn_id": "turn_001",
  "trace_id": "4bf92f3577b34da6a3ce929d0e0e4736",
  "span_id": "00f067aa0ba902b7",

  // Event-specific payload
  "payload": {
    "model": "claude-sonnet-4-5",
    "input_tokens": 8432,
    "output_tokens": 312,
    "cost_usd": 0.0298,
    "stop_reason": "tool_use",
    "tool_uses": [{"name": "bash", "id": "toolu_01"}],
    "latency_ms": 1842,
    "ttft_ms": 312
  }
}

Always hash content for correlation, log metadata only, and store full content in a separate encrypted audit store with strict access controls.

Structured Logger — TypeScript Implementation

Many agent harnesses are built in TypeScript. Here is a production-grade structured logger with OTel trace correlation:

import { trace, context } from '@opentelemetry/api';

interface LogEnvelope {
  timestamp: string;
  level: 'DEBUG' | 'INFO' | 'WARN' | 'ERROR';
  event_type: string;
  schema_version: string;
  session_id: string;
  conv_id?: string;
  turn_id?: string;
  trace_id?: string;
  span_id?: string;
  payload: Record<string, unknown>;
}

type LogSink = (entry: LogEnvelope) => void;

const stdoutSink: LogSink = (entry) => {
  process.stdout.write(JSON.stringify(entry) + '\n');
};

class AgentLogger {
  private sessionId: string;
  private convId?: string;
  private turnId?: string;
  private schemaVersion = '2025-04-01';
  private sinks: LogSink[];

  constructor(sessionId: string, sinks: LogSink[] = [stdoutSink]) {
    this.sessionId = sessionId;
    this.sinks = sinks;
  }

  setConversationContext(convId: string, turnId?: string): void {
    this.convId = convId;
    this.turnId = turnId;
  }

  private getTraceContext(): { trace_id?: string; span_id?: string } {
    const span = trace.getActiveSpan();
    if (!span) return {};
    const ctx = span.spanContext();
    return {
      trace_id: ctx.traceId,
      span_id: ctx.spanId,
    };
  }

  private emit(
    level: LogEnvelope['level'],
    event_type: string,
    payload: Record<string, unknown>
  ): void {
    const entry: LogEnvelope = {
      timestamp: new Date().toISOString(),
      level,
      event_type,
      schema_version: this.schemaVersion,
      session_id: this.sessionId,
      conv_id: this.convId,
      turn_id: this.turnId,
      ...this.getTraceContext(),
      payload,
    };
    for (const sink of this.sinks) {
      try { sink(entry); } catch { /* sink errors must not crash the agent */ }
    }
  }

  modelCall(params: {
    model: string;
    inputTokens: number;
    outputTokens: number;
    costUsd: number;
    stopReason: string;
    latencyMs: number;
    ttftMs: number;
    callIndex: number;
  }): void {
    this.emit('INFO', 'model_call', {
      model: params.model,
      input_tokens: params.inputTokens,
      output_tokens: params.outputTokens,
      cost_usd: params.costUsd,
      stop_reason: params.stopReason,
      latency_ms: params.latencyMs,
      ttft_ms: params.ttftMs,
      call_index: params.callIndex,
    });
  }

  toolInvoke(toolName: string, toolUseId: string, argsHash: string): void {
    this.emit('INFO', 'tool_invoke', {
      tool_name: toolName,
      tool_use_id: toolUseId,
      args_hash: argsHash,
    });
  }

  toolResult(params: {
    toolName: string;
    toolUseId: string;
    durationMs: number;
    outputBytes: number;
    success: boolean;
    exitCode?: number;
    errorType?: string;
  }): void {
    const level = params.success ? 'INFO' : 'WARN';
    this.emit(level, 'tool_result', {
      tool_name: params.toolName,
      tool_use_id: params.toolUseId,
      duration_ms: params.durationMs,
      output_bytes: params.outputBytes,
      success: params.success,
      exit_code: params.exitCode,
      error_type: params.errorType,
    });
  }

  error(errorType: string, message: string, recoverable: boolean): void {
    this.emit('ERROR', 'error', {
      error_type: errorType,
      error_message: message, // OK to log non-PII error messages
      recoverable,
    });
  }

  budgetAlert(thresholdUsd: number, actualUsd: number, actionTaken: string): void {
    this.emit('WARN', 'budget_alert', {
      threshold_usd: thresholdUsd,
      actual_usd: actualUsd,
      action_taken: actionTaken,
    });
  }
}

Log Levels for Agent Systems

Standard log level guidance applies, but with agent-specific nuances:

  • DEBUG: Full context window contents (only in development — never in production due to PII and volume). Token-level details.
  • INFO: All model calls, all tool invocations and results, session start/end, turn boundaries. This is your primary operational data.
  • WARN: Failed tool executions, budget alerts, retried model calls, unexpectedly long tool execution times.
  • ERROR: Model API errors, unrecoverable tool failures, context overflow, authentication failures.

4. Cost Tracking

Per-Request Economics and Budget Circuit Breakers ▲

Cost tracking for agent systems is not a finance problem — it is an engineering problem. An agent that silently enters a loop and burns $50 in 5 minutes is indistinguishable from a healthy agent until the billing statement arrives. You need real-time cost visibility with automated circuit breakers.

def compute_cost(model_id: str, input_tokens: int, output_tokens: int) -> float:
    """Calculate USD cost for a single model call."""
    pricing = MODEL_PRICING.get(model_id)
    if pricing is None:
        # Unknown model — use a conservative estimate and log a warning
        import logging
        logging.warning(f"Unknown model '{model_id}' — using default pricing $3/$15 per 1M")
        pricing = {"input": 3.00, "output": 15.00}

    input_cost = (input_tokens / 1_000_000) * pricing["input"]
    output_cost = (output_tokens / 1_000_000) * pricing["output"]
    return round(input_cost + output_cost, 8)


@dataclass
class CostBreakdown:
    model_id: str
    call_index: int
    input_tokens: int
    output_tokens: int
    cost_usd: float
    timestamp: float = field(default_factory=time.time)


class BudgetExceededError(Exception):
    """Raised when a cost budget threshold is exceeded."""
    def __init__(self, budget_usd: float, actual_usd: float):
        self.budget_usd = budget_usd
        self.actual_usd = actual_usd
        super().__init__(
            f"Cost budget exceeded: ${actual_usd:.4f} > ${budget_usd:.4f}"
        )


class CostTracker:
    """
    Tracks cost at session, conversation, and turn granularity.
    Supports hard budget limits (raise exception) and soft limits (callback).
    """

    def __init__(
        self,
        session_id: str,
        hard_budget_usd: Optional[float] = None,
        soft_budget_usd: Optional[float] = None,
        soft_budget_callback: Optional[Callable[[float, float], None]] = None,
    ):
        self.session_id = session_id
        self.hard_budget_usd = hard_budget_usd
        self.soft_budget_usd = soft_budget_usd
        self.soft_budget_callback = soft_budget_callback

        self._lock = Lock()
        self._session_cost: float = 0.0
        self._conv_costs: Dict[str, float] = {}
        self._turn_costs: Dict[str, float] = {}
        self._call_log: list[CostBreakdown] = []
        self._call_index: int = 0
        self._soft_alert_fired: bool = False

    def record_call(
        self,
        model_id: str,
        input_tokens: int,
        output_tokens: int,
        conv_id: Optional[str] = None,
        turn_id: Optional[str] = None,
    ) -> float:
        """
        Record a completed model call and return its cost.
        Raises BudgetExceededError if hard_budget_usd is set and exceeded.
        """
        cost = compute_cost(model_id, input_tokens, output_tokens)

        with self._lock:
            self._call_index += 1
            breakdown = CostBreakdown(
                model_id=model_id,
                call_index=self._call_index,
                input_tokens=input_tokens,
                output_tokens=output_tokens,
                cost_usd=cost,
            )
            self._call_log.append(breakdown)
            self._session_cost += cost

            if conv_id:
                self._conv_costs[conv_id] = self._conv_costs.get(conv_id, 0.0) + cost
            if turn_id:
                self._turn_costs[turn_id] = self._turn_costs.get(turn_id, 0.0) + cost

            # Soft limit: fire callback once per session
            if (
                self.soft_budget_usd
                and not self._soft_alert_fired
                and self._session_cost >= self.soft_budget_usd
            ):
                self._soft_alert_fired = True
                if self.soft_budget_callback:
                    self.soft_budget_callback(self.soft_budget_usd, self._session_cost)

            # Hard limit: raise immediately (agent loop must catch this)
            if self.hard_budget_usd and self._session_cost > self.hard_budget_usd:
                raise BudgetExceededError(self.hard_budget_usd, self._session_cost)

        return cost

    @property
    def session_total(self) -> float:
        return self._session_cost

    def conv_total(self, conv_id: str) -> float:
        return self._conv_costs.get(conv_id, 0.0)

    def turn_total(self, turn_id: str) -> float:
        return self._turn_costs.get(turn_id, 0.0)

    def top_costly_calls(self, n: int = 5) -> list[CostBreakdown]:
        """Returns the n most expensive model calls — useful for debugging cost spikes."""
        return sorted(self._call_log, key=lambda c: c.cost_usd, reverse=True)[:n]

    def cost_by_model(self) -> Dict[str, float]:
        """Aggregate cost grouped by model ID."""
        totals: Dict[str, float] = {}
        for call in self._call_log:
            totals[call.model_id] = totals.get(call.model_id, 0.0) + call.cost_usd
        return totals

    def summary(self) -> dict:
        return {
            "session_id": self.session_id,
            "total_cost_usd": self._session_cost,
            "total_calls": self._call_index,
            "cost_by_model": self.cost_by_model(),
            "hard_budget_usd": self.hard_budget_usd,
            "budget_utilization_pct": (
                round(self._session_cost / self.hard_budget_usd * 100, 1)
                if self.hard_budget_usd else None
            ),
        }


# ---- Usage in agent loop ----
def build_cost_tracker(session_id: str, logger) -> CostTracker:
    def soft_alert(threshold: float, actual: float):
        logger.budgetAlert(threshold, actual, action_taken="warning_logged")

    return CostTracker(
        session_id=session_id,
        hard_budget_usd=5.00,   # Hard stop at $5/session

5. Latency Monitoring

TTFT, Waterfall Breakdown, and Percentile Monitoring ▲

Agent systems have a fundamentally different latency profile from traditional APIs. A single user request involves multiple sequential model calls (each 1–8 seconds), tool executions (microseconds to minutes), and context assembly. Understanding where time is spent requires waterfall analysis, not just aggregate request duration.

Key Latency Metrics

TTFT (time to first token) Time from API call to first token received 200 – 800ms > 2000ms
Model call latency (total) TTFT + streaming completion time 1000 – 6000ms > 15000ms
Tool execution latency From tool_invoke to tool_result 1ms (file read) to 60s+ (build) Tool-specific SLOs
Context assembly latency Time to build messages array for model 1 – 50ms > 200ms
Turn total latency User message to final response 5 – 60s > 120s

Optimizing tool execution latency gives minimal benefit; reducing the number of model calls (through better prompting or routing to faster models for intermediate steps) is the highest-leverage optimization.

Latency Tracking Middleware — Python

import time
import asyncio
from typing import AsyncIterator, Optional
from dataclasses import dataclass
from collections import deque
import statistics


@dataclass
class ModelCallLatency:
    call_id: str
    model_id: str
    ttft_ms: float
    total_ms: float
    input_tokens: int
    output_tokens: int
    timestamp: float = 0.0

    def __post_init__(self):
        if self.timestamp == 0.0:
            self.timestamp = time.time()


class LatencyTracker:
    """
    Tracks model call latencies with a rolling window for percentile computation.
    Designed to be thread-safe for concurrent agent sessions.
    """

    def __init__(self, window_size: int = 1000):
        self._window: deque[ModelCallLatency] = deque(maxlen=window_size)
        self._lock = asyncio.Lock()

    async def record(self, latency: ModelCallLatency) -> None:
        async with self._lock:
            self._window.append(latency)

    def percentiles(self, metric: str = "total_ms") -> dict:
        """Compute p50/p95/p99 for the rolling window."""
        if not self._window:
            return {}
        values = sorted(getattr(r, metric) for r in self._window)
        n = len(values)

        def pct(p: float) -> float:
            idx = max(0, int(n * p / 100) - 1)
            return round(values[idx], 1)

        return {
            "p50": pct(50),
            "p95": pct(95),
            "p99": pct(99),
            "mean": round(statistics.mean(values), 1),
            "count": n,
        }

    def ttft_percentiles(self) -> dict:
        return self.percentiles("ttft_ms")

    def total_latency_percentiles(self) -> dict:
        return self.percentiles("total_ms")


class TimedModelClient:
    """
    Wraps an Anthropic-compatible model client to measure TTFT and total latency
    for streaming responses.
    """

    def __init__(self, client, tracker: LatencyTracker, logger):
        self._client = client
        self._tracker = tracker
        self._logger = logger

    async def call_streaming(
        self,
        messages: list,
        model: str,
        max_tokens: int,
        call_id: str,
        **kwargs
    ) -> AsyncIterator:
        t_start = time.perf_counter()
        ttft_ms: Optional[float] = None
        first_token_seen = False
        output_tokens = 0

        async with self._client.messages.stream(
            model=model,
            messages=messages,
            max_tokens=max_tokens,
            **kwargs
        ) as stream:
            async for event in stream:
                if not first_token_seen and hasattr(event, 'type'):
                    if event.type in ('content_block_delta', 'text'):
                        ttft_ms = (time.perf_counter() - t_start) * 1000
                        first_token_seen = True

                if hasattr(event, 'type') and event.type == 'message_stop':
                    total_ms = (time.perf_counter() - t_start) * 1000
                    usage = getattr(stream, 'usage', None)

                    latency = ModelCallLatency(
                        call_id=call_id,
                        model_id=model,
                        ttft_ms=ttft_ms or total_ms,
                        total_ms=total_ms,
                        input_tokens=getattr(usage, 'input_tokens', 0) if usage else 0,
                        output_tokens=getattr(usage, 'output_tokens', 0) if usage else 0,
                    )
                    await self._tracker.record(latency)

                yield event

    async def get_latency_report(self) -> dict:
        return {
            "ttft": self._tracker.ttft_percentiles(),
            "total": self._tracker.total_latency_percentiles(),
        }

6. Replay and Time-Travel Debugging

Recording Sessions and Deterministic Replay ▲

The most powerful debugging technique for agent systems is session recording with deterministic replay. By recording every model response alongside the input that produced it, you can replay a production bug in development — the model will "say" exactly what it said in production because you feed it the recorded response instead of calling the real API.

Analogy: This is identical to how browser automation testing tools like Playwright record network responses and replay them — your tests pass deterministically even without network access. For agents, you're recording model API calls instead of HTTP calls, but the principle is identical.

What to Record

For a complete replay, you need to record:

  1. The input context — the full messages array sent to the model (post-assembly)
  2. The model response — complete API response including stop_reason, tool_uses, text content
  3. Tool results — what each tool returned (for tools that are non-deterministic like bash)
  4. Timestamps — for debugging timing-related issues
  5. Random seeds — if any randomness is introduced outside the model

SessionRecorder and ReplayEngine — Python

import json
import gzip
import hashlib
import time
from pathlib import Path
from dataclasses import dataclass, field, asdict
from typing import Any, Optional


@dataclass
class RecordedModelCall:
    """One entry in the recording: input context + model response."""
    call_index: int
    timestamp: float
    model_id: str
    input_messages: list        # Full messages array sent to model
    input_tokens: int
    response: dict              # Full API response as dict
    output_tokens: int
    stop_reason: str
    latency_ms: float
    # Hash of input_messages for integrity checking during replay
    input_hash: str = field(init=False)

    def __post_init__(self):
        content = json.dumps(self.input_messages, sort_keys=True)
        self.input_hash = hashlib.sha256(content.encode()).hexdigest()[:16]


@dataclass
class RecordedToolCall:
    """One tool execution: args + result."""
    call_index: int
    tool_use_id: str
    tool_name: str
    args: dict
    result: Any
    duration_ms: float
    success: bool
    timestamp: float = field(default_factory=time.time)


@dataclass
class SessionRecording:
    session_id: str
    recorded_at: float
    harness_version: str
    model_calls: list[RecordedModelCall] = field(default_factory=list)
    tool_calls: list[RecordedToolCall] = field(default_factory=list)

    def save(self, path: Path) -> None:
        """Serialize to gzipped JSON (recordings can be large)."""
        data = {
            "session_id": self.session_id,
            "recorded_at": self.recorded_at,
            "harness_version": self.harness_version,
            "model_calls": [asdict(c) for c in self.model_calls],
            "tool_calls": [asdict(c) for c in self.tool_calls],
        }
        with gzip.open(path, 'wt', encoding='utf-8') as f:
            json.dump(data, f, indent=2)

    @classmethod
    def load(cls, path: Path) -> 'SessionRecording':
        with gzip.open(path, 'rt', encoding='utf-8') as f:
            data = json.load(f)
        recording = cls(
            session_id=data["session_id"],
            recorded_at=data["recorded_at"],
            harness_version=data["harness_version"],
        )
        # Reconstruct dataclasses (simplified — production would use dacite or pydantic)
        recording.model_calls = [RecordedModelCall(**c) for c in data["model_calls"]]
        recording.tool_calls = [RecordedToolCall(**c) for c in data["tool_calls"]]
        return recording


class SessionRecorder:
    """Wraps a real model client and records all interactions."""

    def __init__(self, real_client, session_id: str, harness_version: str):
        self._client = real_client
        self._recording = SessionRecording(
            session_id=session_id,
            recorded_at=time.time(),
            harness_version=harness_version,
        )
        self._call_index = 0

    async def call(self, messages: list, model: str, max_tokens: int, **kwargs) -> dict:
        """Call the real model and record the interaction."""
        self._call_index += 1
        t_start = time.perf_counter()
        response = await self._client.messages.create(
            model=model, messages=messages, max_tokens=max_tokens, **kwargs
        )
        latency_ms = (time.perf_counter() - t_start) * 1000

        recorded = RecordedModelCall(
            call_index=self._call_index,
            timestamp=time.time(),
            model_id=model,
            input_messages=messages,
            input_tokens=response.usage.input_tokens,
            response=response.model_dump(),   # pydantic model to dict
            output_tokens=response.usage.output_tokens,
            stop_reason=response.stop_reason,
            latency_ms=latency_ms,
        )
        self._recording.model_calls.append(recorded)
        return response

    def record_tool_result(
        self, call_index: int, tool_use_id: str, tool_name: str,
        args: dict, result: Any, duration_ms: float, success: bool
    ) -> None:
        self._recording.tool_calls.append(RecordedToolCall(
            call_index=call_index,
            tool_use_id=tool_use_id,
            tool_name=tool_name,
            args=args,
            result=result,
            duration_ms=duration_ms,
            success=success,
        ))

    def save(self, path: Path) -> None:
        self._recording.save(path)


class ReplayEngine:
    """
    Replays a recorded session deterministically.
    Model calls return recorded responses; tool calls return recorded results.
    Raises if the input context diverges from the recording (detects replay bugs).
    """

    def __init__(self, recording: SessionRecording, strict: bool = True):
        self._recording = recording
        self._strict = strict
        self._model_call_idx = 0
        self._tool_call_idx = 0

    def next_model_response(self, messages: list) -> dict:
        """Return the next recorded model response, validating input hash."""
        if self._model_call_idx >= len(self._recording.model_calls):
            raise IndexError(
                f"ReplayEngine exhausted: tried to get call {self._model_call_idx + 1} "
                f"but recording only has {len(self._recording.model_calls)} model calls"
            )

        recorded = self._recording.model_calls[self._model_call_idx]
        self._model_call_idx += 1

        # Validate that the input context matches what was recorded
        actual_hash = hashlib.sha256(
            json.dumps(messages, sort_keys=True).encode()
        ).hexdigest()[:16]

        if self._strict and actual_hash != recorded.input_hash:
            raise ValueError(
                f"Replay divergence at model call {self._model_call_idx}: "
                f"input context hash mismatch. "
                f"Expected {recorded.input_hash}, got {actual_hash}. "
                f"The agent loop produced different context than during recording."
            )

        return recorded.response

    def next_tool_result(self, tool_use_id: str) -> Any:
        """Return the next recorded tool result."""
        # Find the matching tool call by tool_use_id
        for tc in self._recording.tool_calls[self._tool_call_idx:]:
            if tc.tool_use_id == tool_use_id:
                self._tool_call_idx += 1
                return tc.result

        raise KeyError(f"No recorded result for tool_use_id={tool_use_id}")

    def step_to(self, call_index: int) -> dict:
        """
        Time-travel: reset to a specific model call index.
        Useful for debugging a specific decision point.
        """
        if call_index < 1 or call_index > len(self._recording.model_calls):
            raise IndexError(f"call_index {call_index} out of range")
        self._model_call_idx = call_index - 1
        return self._recording.model_calls[call_index - 1].response

    @property
    def current_call_index(self) -> int:
        return self._model_call_idx

    @property
    def total_calls(self) -> int:
        return len(self._recording.model_calls)
Production workflow: When a user reports a bug ("the agent deleted the wrong file in my session sess_abc123"), you: (1) pull the session recording from object storage, (2) load it into ReplayEngine with strict=False for initial investigation, (3) step through call by call until you find the decision point where the agent went wrong, (4) fix the prompt or tool behavior, (5) run the full recording to confirm the fix produces the correct result.

When alerts fire, the on-call engineer needs a clear runbook. Agent systems have unique failure modes:

Infinite Loop Detection: The most dangerous failure mode is an agent that loops — calling the same tool repeatedly with minor variations. 8. Interview Tips

What Interviewers Look for in Observability Questions

  • Systems thinking: Can you design a tracing hierarchy that reflects the semantic structure of agent execution, not just HTTP boundaries? Interviewers want to see you reason about what the data model should look like before reaching for tools.
  • Cost awareness: Can you demonstrate that you treat token cost as a first-class operational metric, not a billing afterthought? Know real pricing numbers for at least two models.
  • Specificity over generality: Avoid "we would add logging." Instead, say: "We would emit structured JSON events with a session_id and trace_id on every model call, capturing input/output token counts, cost in USD, and stop_reason, shipping to a central log aggregation system queryable by Grafana."
  • Debugging workflow: Be able to walk through a concrete debugging scenario end to end — from a user complaint to a production trace to a recording replay to a fix.
  • Failure mode reasoning: Know the agent-specific failure modes: infinite loops, context overflow, hallucinated tool calls, cost spikes. Interviewers respect candidates who have clearly thought about what can go wrong.

Common Follow-up Questions

  • "How do you test your observability layer itself?" — Answer: Inject synthetic events, use recorded sessions, verify that alerts fire correctly against replayed anomaly traces.
  • "What do you do if logging the full context window is too expensive?" — Answer: Sample at a configurable rate (e.g., 10% of sessions), always log metadata, store full context in cold storage (S3/GCS) indexed by trace_id, only retrieve on demand.
  • "How do you handle PII in agent logs?" — Answer: Hash user messages at ingestion, store full content in an encrypted, access-controlled audit store separate from operational logs, apply retention policies.
  • "How would you detect prompt injection in your observability data?" — Answer: Monitor for anomalous patterns — unusually long context windows, unexpected tool_use sequences (e.g., network calls from a coding agent), cost spikes on single turns. Alert on tool invocation patterns that deviate from baseline.
The "tell me about a time you debugged a production AI system" question: If you have built an agent, prepare a concrete story that covers: (1) what the symptom was, (2) which observability data you consulted first and why, (3) what the root cause turned out to be, and (4) what you changed in the system as a result. Bonus points for mentioning a fix to the observability layer itself.

9. Knowledge Check

Q1. You are designing the OpenTelemetry span hierarchy for an agent harness. A user sends one message that triggers 12 model calls over 2 minutes. How many root spans should there be, and what is the correct hierarchy?

One root span for the session, with one conversation span, one turn span (for the single user message), and 12 model_call child spans under the turn. The hierarchy is: session → conversation → turn → model_call → tool_execution. Creating a root span per model call would break trace correlation — you would lose the ability to query "show me all model calls for session X" without expensive scatter-gather across trace IDs.

Q2. Your cost dashboard shows a sudden 10x spike in cost/hour. The session count has not changed. Walk through your investigation steps.

Step 1: Check if the spike is broad (all sessions) or concentrated (one or a few sessions) using the top_costly_calls() or a per-session cost aggregation. Step 2: If concentrated — pull that session's recording and check the model_calls for the pattern. Look for: (a) the same tool being called repeatedly with similar args (loop), (b) unusually large input_tokens per call (context growing unboundedly), (c) suddenly using a more expensive model. Step 3: If broad — check if there was a recent deploy that changed the system prompt (making it much longer), or a routing change that shifted traffic to a more expensive model. Step 4: Engage the budget circuit breaker to stop further spend while investigating.

Q3. A developer asks: "Why do I need structured logging AND distributed tracing? Aren't they redundant?" How do you respond?

They are complementary, not redundant. Traces capture the timing and causal structure of execution — they tell you what happened in what order and how long each step took. They are optimized for waterfall views and latency analysis. Logs capture content and events at specific points — they tell you what the values were, what the error message was, what the model said. Logs are optimized for text search and content inspection. In practice: a trace shows you "turn 3 had an unusually long model_call at 12:34:01"; the correlated log entry shows you "input_tokens=45,000, cost=$0.18" which reveals the root cause is context bloat. You need both.

Q4. You want to reproduce a production bug where the agent deleted the wrong file. The session recording exists. Describe the exact steps using the ReplayEngine to isolate the decision.

1. Load the recording: recording = SessionRecording.load(path). 2. Create engine = ReplayEngine(recording, strict=False) — use non-strict mode first to identify the divergence without crashing. 3. Use engine.step_to(1) through step_to(N), inspecting each recorded.response for tool_use blocks. Find the model call that produced a file_write or bash tool_use targeting the wrong file. 4. Examine recording.model_calls[k].input_messages to see the full context the model had at that decision point. 5. Identify the context issue — was the wrong filename mentioned prominently? Was there ambiguity in the system prompt? 6. Fix the prompt or tool design, switch to strict=True, and rerun the full recording to confirm the fix results in the correct tool target.

Q5. Design a TTFT (time to first token) alerting strategy. What threshold should trigger an alert, and what should the alert drive the on-call engineer to do?

TTFT baseline for claude-sonnet-4-5 is roughly 200–500ms p50, 800ms p95. Alert thresholds: Warning at p95 > 2000ms (4x baseline), Critical at p95 > 5000ms. The alert should include the model_id label so the engineer can immediately check if it is model-specific or all models. First action: check the model provider's status page (status.anthropic.com, status.openai.com). If a provider incident is confirmed: activate fallback model routing (switch that model to an equivalent alternative). If no provider incident: check for context size regression — unusually large input_tokens increase time-to-first-token as the model processes more prefix. Alert resolution: TTFT p95 back below 1000ms for 10 minutes. Track TTFT trends weekly — gradual increase can indicate context window growth over time as prompts get longer.

Module 8 Checkpoint Summary

tl;dr: Agent observability is not traditional APM with logging bolted on. It requires a purpose-built span hierarchy aligned to agent semantics, cost as a first-class signal, and session recording for deterministic bug reproduction.

Key Concepts to Retain

  • The 5-level OTel trace hierarchy: session → conversation → turn → model_call → tool_execution — each level exists because it maps to a distinct operational boundary with its own failure modes.
  • Structured logs must have a common envelope with session_id, conv_id, turn_id, trace_id, and span_id on every event — without these, log events from async operations are uncorrelated and useless.
  • Cost = (input_tokens / 1M × input_price) + (output_tokens / 1M × output_price). Know approximate pricing for at least claude-sonnet and claude-haiku.
  • TTFT (time to first token) is the key user-experience latency metric; total model call latency determines agent throughput. They require different alerting thresholds.
  • Session recording with deterministic replay is the production debugging superpower for agent systems — it converts a non-reproducible user report into a reproducible test case.
  • The four agent-specific alert categories: cost spike (loop detection), model API error rate, TTFT degradation (provider issue), and session stuck (hung tool or loop).

5 Suggested Flashcard Q&A Pairs

  1. Q: What is TTFT and why does it matter for agents? A: Time to First Token — the latency from model API call submission to receiving the first streamed token. It determines perceived responsiveness; for streaming agents it is often more important to UX than total model call duration. Typical range: 200–800ms for claude-sonnet.
  2. Q: Why do you hash user messages in logs instead of storing them? A: User messages contain PII, source code, credentials, and proprietary content. Hashes preserve correlation ability (you can still match a log event to a trace) without storing sensitive content. Full content goes in a separate encrypted audit store with strict access controls.
  3. Q: What is baggage propagation in OpenTelemetry and why do agent systems need it? A: OTel baggage is a key-value map carried in the trace context that propagates across process and thread boundaries. Agents need it because they spawn subprocesses (bash tool) and make HTTP calls (external APIs). Without baggage, the session_id and conv_id are lost at process boundaries, making cross-process correlation impossible.
  4. Q: What is the purpose of the input_hash field in a RecordedModelCall? A: It is a hash of the input messages array used for replay validation. During replay, the ReplayEngine computes the hash of the context being replayed and compares it to the recorded hash. A mismatch means the replay diverged — the agent loop produced different context than during recording — which indicates a bug in the replay logic itself.
  5. Q: Name three agent-specific failure modes that standard APM tools will not alert on. A: (1) Infinite loop — agent calling same tool repeatedly, visible as cost spike + high model_calls_per_session. (2) Context overflow — input_tokens approaching context limit, causing truncation and degraded responses. (3) Budget exceeded — session cost crossing threshold; without cost tracking this is invisible until the billing statement.
