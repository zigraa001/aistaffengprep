Module 7: Model Routing & Switching

Design cost-aware, fallback-safe, capability-matched routing logic that intelligently dispatches requests to the right model — and gracefully degrades when things go wrong.

What You'll Learn

  • Why no single model is best — and how to exploit that fact
  • Cost-aware routing: estimating spend before dispatching
  • Latency-aware routing: TTFT vs throughput trade-offs per task class
  • Fallback chains with circuit breakers to handle API outages
  • Capability-based routing: classifying tasks and matching to model strengths
  • A/B testing frameworks for rigorous model comparison
  • Model-specific prompt adaptation — the adapter pattern at the API level
  • Real-world reference: how Cursor, Copilot, and Claude Code use multi-model setups

1. Why Multi-Model Architectures?

The Fundamental Problem: No Model Wins Everywhere ▶

Every LLM is a point on a multi-dimensional Pareto frontier of cost, latency, context length, reasoning depth, code ability, multilingual support, and tool-calling reliability. Optimizing for one dimension typically degrades others.

Analogy: A hospital doesn't send every patient to a neurosurgeon. A GP handles routine cases, a specialist handles complex ones, and an ER triage system decides who goes where in real time. Your AI harness needs the same triage layer.

Here's what the landscape looks like across four key dimensions:

Key insight: Cursor (the AI code editor) uses at least three models concurrently: a fast local or Haiku-class model for inline autocomplete (must respond in <50 ms), Sonnet-class for chat and single-file edits, and Opus or GPT-4 for multi-file agent runs. This split reduces their per-completion cost by an estimated 70% versus routing everything through the flagship model.
Three Routing Motivations: Cost, Latency, Capability ▶

The three motivations are not mutually exclusive — a good router optimizes all three simultaneously:

Cost Optimization

  • Use cheap models for bulk/simple work
  • Estimate tokens before dispatch
  • Budget per conversation or per user tier
  • Route overflow to open-source self-hosted

Latency Optimization

  • Interactive UX needs <500 ms TTFT
  • Background tasks can tolerate 5-10 s
  • Streaming vs batch dispatch differ
  • Geographic routing to nearest inference region

Capability Matching

  • Code models outperform general models on benchmarks
  • Reasoning models for multi-step planning
  • Vision models when input contains images
  • Function-calling reliability varies significantly

2. Cost/Latency-Aware Routing

Estimating Cost Before Dispatching ▶

You cannot control what you don't measure. Before a request is dispatched you should estimate its cost, compare it against the session/user budget, and then pick the cheapest model that meets the quality threshold.

Cost Estimation Formula
estimated_cost = (prompt_tokens * input_price_per_token)
               + (expected_output_tokens * output_price_per_token)

prompt_tokens ≈ len(text) / 4         [rough heuristic]
             or use tiktoken/tokenize() for precision

expected_output_tokens = historical_p50 for task_type
                            or a classifier's predicted length bucket

The key engineering challenge is that you don't know output length at routing time. Strategies to handle this:

  • Historical p50/p95 by task type: Code generation averages 300 tokens; summarization averages 150 tokens. Store these in a lookup table updated by your observability pipeline.
  • Output length prediction model: A tiny classifier (even logistic regression on task features) can predict "short/medium/long" with 75%+ accuracy.
  • Conservative cap with streaming abort: Assume max_tokens and route accordingly; abort early via streaming if budget is reached.
CostAwareRouter — Implementation ▶

Below is a production-grade cost-aware router. It ranks candidate models by cost, checks budget constraints, and selects the cheapest model that satisfies minimum quality requirements.

from typing import Optional

@dataclass
class RoutingRequest:
    messages: list[dict]
    task_type: str                         # "code_gen" | "chat" | "summarize" | "plan" | "tool_use"
    max_budget_usd: Optional[float] = None # Per-request budget cap
    max_latency_ms: Optional[int] = None   # Deadline for TTFT
    min_quality: float = 0.0               # Minimum acceptable quality score
    needs_tools: bool = False
    needs_vision: bool = False
    expected_output_tokens: Optional[int] = None  # Override for length estimation


# Per-task expected output length (p50 from production data)
OUTPUT_TOKEN_P50 = {
    "code_gen": 400,
    "chat": 200,
    "summarize": 150,
    "plan": 350,
    "tool_use": 250,
}


class CostAwareRouter:
    def __init__(
        self,
        registry: dict[str, ModelConfig] = MODEL_REGISTRY,
        tokenizer_model: str = "cl100k_base",
    ):
        self.registry = registry
        self.enc = tiktoken.get_encoding(tokenizer_model)

    def _count_prompt_tokens(self, messages: list[dict]) -> int:
        total = 0
        for m in messages:
            content = m.get("content", "")
            if isinstance(content, str):
                total += len(self.enc.encode(content))
            elif isinstance(content, list):
                # Vision / multi-part messages
                for part in content:
                    if part.get("type") == "text":
                        total += len(self.enc.encode(part.get("text", "")))
                    elif part.get("type") == "image":
                        total += 765  # Approximate fixed cost for an image token
            # Role + formatting overhead
            total += 4
        return total + 2  # reply primer

    def _estimate_cost(
        self, model: ModelConfig, prompt_tokens: int, expected_output: int
    ) -> float:
        input_cost = (prompt_tokens / 1000) * model.input_price_per_1k
        output_cost = (expected_output / 1000) * model.output_price_per_1k
        return input_cost + output_cost

    def route(self, request: RoutingRequest) -> tuple[ModelConfig, dict]:
        """
        Returns (chosen_model, routing_metadata).
        Raises ValueError if no model satisfies all constraints.
        """
        prompt_tokens = self._count_prompt_tokens(request.messages)
        expected_output = (
            request.expected_output_tokens
            or OUTPUT_TOKEN_P50.get(request.task_type, 200)
        )

        candidates = []
        rejection_reasons = {}

        for model_id, model in self.registry.items():
            reasons = []

            # Hard constraints
            if request.needs_tools and not model.supports_tools:
                reasons.append("no tool support")
            if request.needs_vision and not model.supports_vision:
                reasons.append("no vision support")
            if prompt_tokens + expected_output > model.max_context:
                reasons.append(f"context overflow ({prompt_tokens + expected_output} > {model.max_context})")
            if model.quality_score < request.min_quality:
                reasons.append(f"quality {model.quality_score:.2f} < required {request.min_quality:.2f}")
            if request.max_latency_ms and model.median_ttft_ms > request.max_latency_ms:
                reasons.append(f"TTFT {model.median_ttft_ms}ms > budget {request.max_latency_ms}ms")

            estimated_cost = self._estimate_cost(model, prompt_tokens, expected_output)
            if request.max_budget_usd and estimated_cost > request.max_budget_usd:
                reasons.append(f"cost ${estimated_cost:.4f} > budget ${request.max_budget_usd:.4f}")

            if reasons:
                rejection_reasons[model_id] = reasons
                continue

            candidates.append((estimated_cost, model.median_ttft_ms, model))

        if not candidates:
            raise ValueError(
                f"No model satisfies constraints. Rejections: {rejection_reasons}"
            )

        # Primary sort: cost. Tiebreak: latency.
        candidates.sort(key=lambda x: (x[0], x[1]))
        chosen_cost, chosen_ttft, chosen_model = candidates[0]

        metadata = {
            "prompt_tokens": prompt_tokens,
            "expected_output_tokens": expected_output,
            "estimated_cost_usd": round(chosen_cost, 6),
            "estimated_ttft_ms": chosen_ttft,
            "candidates_considered": len(candidates),
            "rejected": rejection_reasons,
        }
        return chosen_model, metadata
Production tip: Don't use static p50 values for long. Wire your observability pipeline to update OUTPUT_TOKEN_P50 from a rolling 7-day window of actual completions, segmented by task type. This alone can reduce cost estimation error from 40% to under 10%.

3. Fallback Chains with Circuit Breakers

Why Single-Provider APIs Are a Liability ▶

Every major model provider has experienced outages: Anthropic, OpenAI, Google — none are immune. In 2024, OpenAI had 14 significant incidents. Production AI products that rely on a single model endpoint are one outage away from complete user-facing failure.

The solution is a fallback chain: an ordered list of models, where each model is tried in sequence until one succeeds. Combined with a circuit breaker, this avoids hammering a broken provider and wastes latency on doomed requests.

Warning: Naive retry loops without circuit breakers can amplify outage impact. If your primary provider is down and you send 1000 RPM * 3 retries * 10 second timeout = 30 000 seconds of wasted thread time. A circuit breaker stops this cascade immediately.

ModelFallbackChain with CircuitBreaker — Implementation ▶ 
import asyncio
import time
import logging
from collections import deque
from enum import Enum, auto
from dataclasses import dataclass, field
from typing import Any, Callable, Awaitable

logger = logging.getLogger(__name__)


class BreakerState(Enum):
    CLOSED = auto()    # Normal: passing requests through
    OPEN = auto()      # Tripped: fast-fail all requests
    HALF_OPEN = auto() # Probing: let one request through


@dataclass
class CircuitBreakerConfig:
    failure_threshold: int = 5       # Failures in window before OPEN
    window_seconds: int = 60         # Rolling failure window
    recovery_timeout: int = 30       # Seconds to wait before HALF_OPEN
    probe_timeout_multiplier: float = 2.0  # Probe gets 2x normal timeout


class CircuitBreaker:
    def __init__(self, name: str, config: CircuitBreakerConfig = CircuitBreakerConfig()):
        self.name = name
        self.config = config
        self._state = BreakerState.CLOSED
        self._failures: deque[float] = deque()  # timestamps
        self._opened_at: float = 0.0
        self._lock = asyncio.Lock()

    @property
    def state(self) -> BreakerState:
        return self._state

    def _prune_old_failures(self) -> None:
        cutoff = time.monotonic() - self.config.window_seconds
        while self._failures and self._failures[0] < cutoff:
            self._failures.popleft()

    async def _try_recover(self) -> bool:
        """Returns True if we should enter HALF_OPEN."""
        if self._state == BreakerState.OPEN:
            elapsed = time.monotonic() - self._opened_at
            if elapsed >= self.config.recovery_timeout:
                async with self._lock:
                    if self._state == BreakerState.OPEN:
                        self._state = BreakerState.HALF_OPEN
                        logger.info(f"CircuitBreaker[{self.name}]: OPEN -> HALF_OPEN")
                        return True
        return False

    async def call(
        self,
        fn: Callable[..., Awaitable[Any]],
        *args,
        timeout: float = 30.0,
        **kwargs,
    ) -> Any:
        # Attempt recovery if we're OPEN
        if self._state == BreakerState.OPEN:
            recovered = await self._try_recover()
            if not recovered:
                raise RuntimeError(
                    f"CircuitBreaker[{self.name}] is OPEN — fast fail"
                )

        effective_timeout = (
            timeout * self.config.probe_timeout_multiplier
            if self._state == BreakerState.HALF_OPEN
            else timeout
        )

        try:
            result = await asyncio.wait_for(fn(*args, **kwargs), timeout=effective_timeout)
            # Success — close or stay closed
            async with self._lock:
                if self._state == BreakerState.HALF_OPEN:
                    self._state = BreakerState.CLOSED
                    self._failures.clear()
                    logger.info(f"CircuitBreaker[{self.name}]: HALF_OPEN -> CLOSED (recovered)")
            return result

        except Exception as exc:
            async with self._lock:
                now = time.monotonic()
                self._failures.append(now)
                self._prune_old_failures()

                if self._state == BreakerState.HALF_OPEN:
                    self._state = BreakerState.OPEN
                    self._opened_at = now
                    logger.warning(f"CircuitBreaker[{self.name}]: HALF_OPEN -> OPEN (probe failed: {exc})")
                elif len(self._failures) >= self.config.failure_threshold:
                    self._state = BreakerState.OPEN
                    self._opened_at = now
                    logger.error(
                        f"CircuitBreaker[{self.name}]: CLOSED -> OPEN "
                        f"({len(self._failures)} failures in {self.config.window_seconds}s)"
                    )
            raise


@dataclass
class FallbackModel:
    model_id: str
    call_fn: Callable[..., Awaitable[Any]]   # async fn(messages, **kwargs) -> response
    timeout_s: float = 30.0
    priority: int = 0                         # Lower = try first


class ModelFallbackChain:
    """
    Tries models in priority order. Each model has its own CircuitBreaker.
    Moves to the next fallback if the current one raises any exception.
    """

    def __init__(self, models: list[FallbackModel], breaker_config: CircuitBreakerConfig = CircuitBreakerConfig()):
        self._models = sorted(models, key=lambda m: m.priority)
        self._breakers = {
            m.model_id: CircuitBreaker(m.model_id, breaker_config)
            for m in self._models
        }

    async def complete(self, messages: list[dict], **kwargs) -> tuple[Any, str]:
        """
        Returns (response, model_id_used).
        Raises RuntimeError if all models fail.
        """
        last_exc: Exception = RuntimeError("No models configured")

        for model in self._models:
            breaker = self._breakers[model.model_id]
            try:
                response = await breaker.call(
                    model.call_fn,
                    messages,
                    timeout=model.timeout_s,
                    **kwargs,
                )
                logger.info(f"ModelFallbackChain: succeeded with {model.model_id}")
                return response, model.model_id

            except RuntimeError as e:
                if "fast fail" in str(e):
                    logger.debug(f"ModelFallbackChain: {model.model_id} circuit open, skipping")
                    last_exc = e
                    continue
                last_exc = e
                logger.warning(f"ModelFallbackChain: {model.model_id} failed: {e}")

            except Exception as e:
                last_exc = e
                logger.warning(f"ModelFallbackChain: {model.model_id} failed: {e}")

        raise RuntimeError(f"All models in fallback chain failed. Last error: {last_exc}")


# --- Usage example ---
# chain = ModelFallbackChain([
#     FallbackModel("claude-3-5-sonnet", call_anthropic, priority=0),
#     FallbackModel("gpt-4o",            call_openai,    priority=1),
#     FallbackModel("llama-3.1-70b",     call_local,     priority=2),
# ])

Design note: The CircuitBreaker here uses a sliding-window failure count, not a fixed-interval counter. This is more accurate under bursty failure patterns — e.g., a 5-second timeout spike won't persist as "failures" a full minute later.

CapabilityRouter with Task Classifier — Implementation ▶ 
import re
from dataclasses import dataclass
from typing import Optional


@dataclass
class TaskSignal:
    task_type: str                   # Classified task category
    has_code_block: bool
    has_image: bool
    estimated_steps: int             # Multi-step planning indicator
    context_token_count: int
    tool_names_required: list[str]


class TaskClassifier:
    """
    Lightweight heuristic classifier for routing decisions.
    In production, replace or augment with a fine-tuned embedding classifier.
    """

    CODE_PATTERNS = [
        r"```[\w]*\n",                   # Fenced code block
        r"\bdef \w+\(",                  # Python function def
        r"\bclass \w+[\(:]",             # Python class
        r"\bfunction \w+\s*\(",          # JS function
        r"\bpublic (static )?(void|int|String)", # Java
        r"\bfn \w+\s*\(",                # Rust
    ]

    PLANNING_KEYWORDS = [
        "plan", "roadmap", "strategy", "architecture",
        "design", "steps to", "how would you", "what is the best approach",
        "tradeoffs", "compare", "pros and cons",
    ]

    TOOL_HINT_PATTERNS = {
        "web_search":  [r"\bsearch\b", r"\blatest\b", r"\bcurrent\b", r"\bnews\b"],
        "code_runner": [r"\brun\b", r"\bexecute\b", r"\btest\b", r"\boutput of\b"],
        "file_read":   [r"\bread file\b", r"\bopen file\b", r"\bcontents of\b"],
    }

    def classify(self, messages: list[dict], context_tokens: int = 0) -> TaskSignal:
        # Flatten all message content to a single text blob
        full_text = ""
        has_image = False
        for m in messages:
            content = m.get("content", "")
            if isinstance(content, str):
                full_text += content + "\n"
            elif isinstance(content, list):
                for part in content:
                    if part.get("type") == "text":
                        full_text += part["text"] + "\n"
                    elif part.get("type") == "image":
                        has_image = True

        full_lower = full_text.lower()

        # Detect code
        has_code = any(re.search(p, full_text) for p in self.CODE_PATTERNS)

        # Detect planning
        planning_hits = sum(kw in full_lower for kw in self.PLANNING_KEYWORDS)
        estimated_steps = min(planning_hits * 2, 10)

        # Classify task type
        if has_image:
            task_type = "vision"
        elif has_code or re.search(r"\b(implement|write|fix|refactor|debug)\b", full_lower):
            task_type = "code_gen"
        elif planning_hits >= 2:
            task_type = "plan"
        elif re.search(r"\b(summarize|tldr|brief|overview)\b", full_lower):
            task_type = "summarize"
        elif re.search(r"\b(call|use|invoke|run)\b.*\btool\b", full_lower):
            task_type = "tool_use"
        else:
            task_type = "chat"

        # Detect required tools
        required_tools = []
        for tool_name, patterns in self.TOOL_HINT_PATTERNS.items():
            if any(re.search(p, full_lower) for p in patterns):
                required_tools.append(tool_name)

        return TaskSignal(
            task_type=task_type,
            has_code_block=has_code,
            has_image=has_image,
            estimated_steps=estimated_steps,
            context_token_count=context_tokens,
            tool_names_required=required_tools,
        )


class CapabilityRouter:
    """
    Routes based on task signal and model capability scores.
    Integrates with CostAwareRouter for final selection.
    """

    # Minimum quality scores required per task type
    QUALITY_FLOOR = {
        "code_gen":  0.85,
        "plan":      0.90,
        "tool_use":  0.80,
        "chat":      0.65,
        "summarize": 0.65,
        "vision":    0.80,
    }

    # Maximum acceptable TTFT per task context
    LATENCY_BUDGET = {
        "autocomplete": 200,
        "interactive":  800,
        "background":  None,  # No limit
    }

    def __init__(self, cost_router: "CostAwareRouter"):
        self.classifier = TaskClassifier()
        self.cost_router = cost_router

    def route(
        self,
        messages: list[dict],
        ux_context: str = "interactive",  # "autocomplete" | "interactive" | "background"
        budget_usd: Optional[float] = None,
        context_tokens: int = 0,
    ) -> tuple["ModelConfig", dict]:
        signal = self.classifier.classify(messages, context_tokens)

        routing_request = RoutingRequest(
            messages=messages,
            task_type=signal.task_type,
            max_budget_usd=budget_usd,
            max_latency_ms=self.LATENCY_BUDGET.get(ux_context),
            min_quality=self.QUALITY_FLOOR.get(signal.task_type, 0.65),
            needs_tools=bool(signal.tool_names_required),
            needs_vision=signal.has_image,
        )

        model, metadata = self.cost_router.route(routing_request)
        metadata["task_signal"] = signal.__dict__
        metadata["ux_context"] = ux_context
        return model, metadata

5. A/B Testing Models

Why Model A/B Testing Is Different From Regular A/B Testing ▶

Standard web A/B testing compares two deterministic UIs: the control and variant are stable. Model A/B testing is harder because:

  • Outputs are stochastic — the same input may yield different outputs across temperature settings
  • Quality is subjective — "better" requires human or LLM judge evaluation, not just a click metric
  • Cost and latency co-vary — a model that's better in quality but 5x more expensive may not be worth it at scale
  • Sample size requirements are large — LLM quality differences are often small; you need hundreds of paired samples for statistical significance
Sample Size for Model Comparison (Two-proportion z-test)
n = 2 * (z_α/2 + z_β)² * p̄(1 - p̄) / (p1 - p2)²

where:
  p1 = win rate model A, p2 = win rate model B
  p̄ = (p1 + p2) / 2
  z_α/2 = 1.96 (95% confidence)
  z_β = 0.84 (80% power)

Example: detecting a 5% win rate difference (60% vs 55%)
requires ~780 paired evaluations per arm.
ABTestRouter — Implementation ▶ 
import hashlib
import random
import time
from collections import defaultdict
from dataclasses import dataclass, field
from typing import Optional, Callable, Any


@dataclass
class ExperimentArm:
    name: str                   # e.g., "control", "treatment_sonnet"
    model_id: str
    traffic_fraction: float     # 0.0 - 1.0; must sum to 1.0 across all arms
    call_fn: Callable           # async fn(messages, **kwargs) -> response


@dataclass
class EvaluationResult:
    arm_name: str
    model_id: str
    response: Any
    latency_ms: float
    estimated_cost_usd: float
    quality_score: Optional[float] = None  # From judge, if available
    user_feedback: Optional[float] = None  # 1-5 rating, if available


@dataclass
class ExperimentStats:
    arm_name: str
    model_id: str
    n_requests: int = 0
    total_latency_ms: float = 0.0
    total_cost_usd: float = 0.0
    quality_scores: list[float] = field(default_factory=list)

    @property
    def mean_latency_ms(self) -> float:
        return self.total_latency_ms / self.n_requests if self.n_requests else 0.0

    @property
    def mean_cost_usd(self) -> float:
        return self.total_cost_usd / self.n_requests if self.n_requests else 0.0

    @property
    def mean_quality(self) -> Optional[float]:
        return sum(self.quality_scores) / len(self.quality_scores) if self.quality_scores else None


class ABTestRouter:
    """
    Routes traffic across experiment arms with deterministic user-level bucketing.
    Collects metrics for offline analysis.
    """

    def __init__(self, experiment_name: str, arms: list[ExperimentArm]):
        assert abs(sum(a.traffic_fraction for a in arms) - 1.0) < 1e-6, \
            "Arm traffic fractions must sum to 1.0"
        self.experiment_name = experiment_name
        self.arms = arms
        self.stats: dict[str, ExperimentStats] = {
            arm.name: ExperimentStats(arm.name, arm.model_id)
            for arm in arms
        }
        self._event_log: list[dict] = []  # Replace with Kafka/database in production

    def _assign_arm(self, user_id: str) -> ExperimentArm:
        """
        Deterministic bucketing: same user always gets same arm.
        Uses MD5 hash for uniform distribution.
        """
        hash_val = int(hashlib.md5(f"{self.experiment_name}:{user_id}".encode()).hexdigest(), 16)
        bucket = (hash_val % 10_000) / 10_000.0   # [0.0, 1.0)

        cumulative = 0.0
        for arm in self.arms:
            cumulative += arm.traffic_fraction
            if bucket < cumulative:
                return arm
        return self.arms[-1]  # Edge case: return last arm

    async def complete(
        self,
        messages: list[dict],
        user_id: str,
        cost_usd: float = 0.0,
        judge_fn: Optional[Callable] = None,
        **kwargs,
    ) -> EvaluationResult:
        arm = self._assign_arm(user_id)
        stats = self.stats[arm.name]

        start = time.monotonic()
        try:
            response = await arm.call_fn(messages, **kwargs)
        except Exception as exc:
            # Log the failure but re-raise so caller can handle/fallback
            self._event_log.append({
                "experiment": self.experiment_name,
                "arm": arm.name,
                "model_id": arm.model_id,
                "user_id": user_id,
                "error": str(exc),
                "ts": time.time(),
            })
            raise

        latency_ms = (time.monotonic() - start) * 1000

        # Optionally evaluate quality via an LLM judge
        quality_score = None
        if judge_fn:
            try:
                quality_score = await judge_fn(messages, response)
                stats.quality_scores.append(quality_score)
            except Exception:
                pass  # Judge failures are non-fatal

        # Update stats
        stats.n_requests += 1
        stats.total_latency_ms += latency_ms
        stats.total_cost_usd += cost_usd

        result = EvaluationResult(
            arm_name=arm.name,
            model_id=arm.model_id,
            response=response,
            latency_ms=latency_ms,
            estimated_cost_usd=cost_usd,
            quality_score=quality_score,
        )

        self._event_log.append({
            "experiment": self.experiment_name,
            "arm": arm.name,
            "model_id": arm.model_id,
            "user_id": user_id,
            "latency_ms": latency_ms,
            "cost_usd": cost_usd,
            "quality_score": quality_score,
            "ts": time.time(),
        })
        return result

    def summary(self) -> dict:
        return {
            arm_name: {
                "n_requests": s.n_requests,
                "mean_latency_ms": round(s.mean_latency_ms, 1),
                "mean_cost_usd": round(s.mean_cost_usd, 6),
                "mean_quality": round(s.mean_quality, 3) if s.mean_quality else None,
            }
            for arm_name, s in self.stats.items()
        }
Deterministic bucketing is critical. If users are randomly re-bucketed on each request, a user's experience will be inconsistent across a session. Always hash on (experiment_name + user_id) so the same user always hits the same arm for the lifetime of the experiment.

6. Model-Specific Prompt Adaptation

Why Models Need Different Prompts ▶

The same logical instruction can produce dramatically different quality outputs depending on how it is phrased for each model. This is because models differ in:

  • Training data and RLHF style: GPT-4 responds well to "You are a..." personas. Claude responds better to direct instructions about character and values.
  • Tool call format: Anthropic uses tools + tool_use content blocks. OpenAI uses functions/tools with JSON schema in a different structure.
  • System prompt placement: Some open-source models (Llama) inject system prompts inside the first user message via a special token format.
  • Response format preferences: GPT-4 tends to use markdown headers eagerly; Haiku tends to be terse without prompting for structure.
Common mistake: Writing one "universal" system prompt and using it across all models. In practice, prompts tuned for Claude degrade GPT-4 quality by up to 15% on structured output tasks, and vice versa. Use the adapter pattern.
ModelAdapter Pattern — Implementation ▶

The adapter pattern provides a uniform interface regardless of which model receives the request, translating between a canonical internal format and model-specific API formats.

from abc import ABC, abstractmethod
from typing import Any
import json


# ---- Canonical internal types ----

@dataclass
class CanonicalTool:
    name: str
    description: str
    parameters: dict   # JSON Schema


@dataclass
class CanonicalRequest:
    system: str
    messages: list[dict]   # [{role, content}]
    tools: list[CanonicalTool] = field(default_factory=list)
    max_tokens: int = 2048
    temperature: float = 0.0


# ---- Base adapter ----

class ModelAdapter(ABC):
    """
    Translates CanonicalRequest to provider-specific API call format
    and normalises the response back to a CanonicalResponse.
    """

    @abstractmethod
    def build_request_payload(self, request: CanonicalRequest) -> dict:
        """Build provider-specific API payload."""
        ...

    @abstractmethod
    def parse_response(self, raw_response: Any) -> dict:
        """
        Returns normalised dict:
          {
            "content": str,
            "tool_calls": [{"name": str, "arguments": dict}],
            "stop_reason": str,
            "usage": {"input_tokens": int, "output_tokens": int}
          }
        """
        ...

    @abstractmethod
    async def call(self, request: CanonicalRequest, client: Any) -> dict:
        """Make the API call and return normalised response."""
        ...


# ---- Anthropic adapter ----

class AnthropicAdapter(ModelAdapter):
    def __init__(self, model_id: str):
        self.model_id = model_id

    def build_request_payload(self, request: CanonicalRequest) -> dict:
        payload = {
            "model": self.model_id,
            "system": request.system,
            "messages": request.messages,
            "max_tokens": request.max_tokens,
            "temperature": request.temperature,
        }
        if request.tools:
            payload["tools"] = [
                {
                    "name": t.name,
                    "description": t.description,
                    "input_schema": t.parameters,
                }
                for t in request.tools
            ]
        return payload

    def parse_response(self, raw: Any) -> dict:
        content_text = ""
        tool_calls = []
        for block in raw.content:
            if block.type == "text":
                content_text += block.text
            elif block.type == "tool_use":
                tool_calls.append({"name": block.name, "arguments": block.input})
        return {
            "content": content_text,
            "tool_calls": tool_calls,
            "stop_reason": raw.stop_reason,
            "usage": {
                "input_tokens": raw.usage.input_tokens,
                "output_tokens": raw.usage.output_tokens,
            },
        }

    async def call(self, request: CanonicalRequest, client: Any) -> dict:
        payload = self.build_request_payload(request)
        raw = await client.messages.create(**payload)
        return self.parse_response(raw)


# ---- OpenAI adapter ----

class OpenAIAdapter(ModelAdapter):
    def __init__(self, model_id: str):
        self.model_id = model_id

    def build_request_payload(self, request: CanonicalRequest) -> dict:
        # OpenAI encodes system as a message, not a top-level field
        messages = [{"role": "system", "content": request.system}] + request.messages
        payload = {
            "model": self.model_id,
            "messages": messages,
            "max_tokens": request.max_tokens,
            "temperature": request.temperature,
        }
        if request.tools:
            payload["tools"] = [
                {
                    "type": "function",
                    "function": {
                        "name": t.name,
                        "description": t.description,
                        "parameters": t.parameters,
                    },
                }
                for t in request.tools
            ]
            payload["tool_choice"] = "auto"
        return payload

    def parse_response(self, raw: Any) -> dict:
        choice = raw.choices[0]
        msg = choice.message
        tool_calls = []
        if msg.tool_calls:
            for tc in msg.tool_calls:
                tool_calls.append({
                    "name": tc.function.name,
                    "arguments": json.loads(tc.function.arguments),
                })
        return {
            "content": msg.content or "",
            "tool_calls": tool_calls,
            "stop_reason": choice.finish_reason,
            "usage": {
                "input_tokens": raw.usage.prompt_tokens,
                "output_tokens": raw.usage.completion_tokens,
            },
        }

    async def call(self, request: CanonicalRequest, client: Any) -> dict:
        payload = self.build_request_payload(request)
        raw = await client.chat.completions.create(**payload)
        return self.parse_response(raw)


# ---- Llama (self-hosted / Ollama) adapter ----

class LlamaAdapter(ModelAdapter):
    """
    Llama 3 chat template expects:
    <|begin_of_text|><|start_header_id|>system<|end_header_id|>
    {system}<|eot_id|>
    <|start_header_id|>user<|end_header_id|>{message}<|eot_id|>
    <|start_header_id|>assistant<|end_header_id|>
    """
    def __init__(self, model_id: str, base_url: str = "http://localhost:11434"):
        self.model_id = model_id
        self.base_url = base_url

    def build_request_payload(self, request: CanonicalRequest) -> dict:
        # Inject system into the messages list as Ollama expects
        messages = [{"role": "system", "content": request.system}] + request.messages
        return {
            "model": self.model_id,
            "messages": messages,
            "stream": False,
            "options": {
                "temperature": request.temperature,
                "num_predict": request.max_tokens,
            },
        }

    def parse_response(self, raw: Any) -> dict:
        return {
            "content": raw["message"]["content"],
            "tool_calls": [],     # Ollama tool support varies by version
            "stop_reason": "stop",
            "usage": {
                "input_tokens": raw.get("prompt_eval_count", 0),
                "output_tokens": raw.get("eval_count", 0),
            },
        }

    async def call(self, request: CanonicalRequest, client: Any) -> dict:
        import aiohttp
        payload = self.build_request_payload(request)
        async with aiohttp.ClientSession() as session:
            async with session.post(
                f"{self.base_url}/api/chat",
                json=payload,
            ) as resp:
                raw = await resp.json()
        return self.parse_response(raw)

def get_adapter(model_id: str) -> ModelAdapter:
    if model_id not in ADAPTER_REGISTRY:
        raise ValueError(f"No adapter registered for model '{model_id}'")
    return ADAPTER_REGISTRY[model_id]

8. Module Quiz

Q1. You have a code assistant with 10 000 daily active users, each averaging 50 completions/day. At $0.003/1k input tokens and $0.015/1k output tokens with average prompt=400 tokens and output=350 tokens, what is your daily LLM bill for Sonnet, and how much could you save by routing 70% of completions to Haiku ($0.00025/$0.00125)?

Sonnet baseline:
Requests/day = 10 000 * 50 = 500 000
Input cost = 500 000 * 400/1000 * 0.003 = $600/day
Output cost = 500 000 * 350/1000 * 0.015 = $2 625/day
Total = $3 225/day

Mixed routing (70% Haiku, 30% Sonnet):
Haiku: 350 000 req * (400 * 0.00025 + 350 * 0.00125) / 1000 = 350 000 * (0.10 + 0.4375)/1000 = $189/day
Sonnet: 150 000 req * (400 * 0.003 + 350 * 0.015) / 1000 = 150 000 * (1.2 + 5.25)/1000 = $967.50/day
Total = ~$1 157/day — saving ~$2 068/day (~64%)

Q2. A circuit breaker is in the OPEN state. A request arrives. What happens? When might it transition to HALF_OPEN, and what decides if it returns to CLOSED?

In the OPEN state, the circuit breaker fast-fails immediately without sending the request to the provider — preventing cascaded timeout load. After a configured recovery_timeout elapses (e.g., 30 s), it transitions to HALF_OPEN and allows one probe request through. If the probe succeeds, the breaker returns to CLOSED (normal operation); if it fails, it returns to OPEN and resets the recovery timer.

Q3. What is the purpose of the ModelAdapter pattern in a multi-provider harness? Give two concrete differences between the Anthropic and OpenAI adapters.

The adapter pattern provides a uniform canonical interface for the rest of the harness, isolating provider-specific API differences. Two concrete differences:

1. System prompt placement: Anthropic takes system as a top-level field; OpenAI inserts it as a {"role": "system"} message at index 0 of the messages list.
2. Tool definition format: Anthropic uses input_schema inside a tools array; OpenAI wraps the function in a {"type": "function", "function": {...}} object and uses parameters (not input_schema).

Q4. You want to run an A/B test between Claude Sonnet and GPT-4o on chat quality. Explain why deterministic user bucketing is preferable to random per-request bucketing.

With random per-request bucketing, the same user might get Sonnet on one turn and GPT-4o on the next within the same conversation. This creates:
1. Inconsistent UX — the response style, knowledge cutoff, and capability level changes mid-conversation, confusing the user.
2. Contaminated measurements — you cannot cleanly attribute quality signals (thumbs-up/down, follow-up messages) to a specific model.
Deterministic bucketing by hashing (experiment_name + user_id) ensures each user always sees the same arm, enabling clean per-user attribution and consistent experience.

Q5. Describe "cascade routing". When is it more appropriate than purely cost-based routing?

Cascade routing sends every request to a cheap model first. If the cheap model produces a low-confidence or low-quality response (detected by a confidence classifier or self-evaluation prompt), the request is automatically escalated to a more capable (and expensive) model.

It is more appropriate than static cost-based routing when: (1) task difficulty is highly variable and unpredictable — e.g., customer support questions range from simple FAQs to complex legal disputes; (2) you want automatic quality safeguards without manual task classification; (3) the cheap model's accuracy on easy tasks is sufficient, and escalation overhead is acceptable. The downside is added latency from the cascade step and the need for a reliable confidence signal.

  • No single model wins: Routing 70% of requests to cheaper models can cut costs by 60%+ with minimal quality impact on simple tasks.
  • Cost estimation precedes dispatch: Count prompt tokens, predict output length from historical p50 by task type, estimate USD cost, compare against budget.
