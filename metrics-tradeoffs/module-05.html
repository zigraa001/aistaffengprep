Consistency & Availability Tradeoffs

Q: What does CAP theorem actually mean with real numbers?

A: CAP theorem states that during a network partition, a distributed system must choose between Consistency (every read gets the most recent write) and Availability (every request gets a non-error response). The key insight that most discussions miss: partitions are rare but they do happen, and what you choose during normal operation matters more.

Quantifying partition frequency:

EnvironmentPartition FrequencyTypical DurationSource
Within single datacenter~1-2 per yearSeconds to minutesGoogle/Microsoft studies
Cross-AZ (same region)~4-8 per yearMinutes to hoursAWS post-mortems
Cross-region~12-24 per yearMinutes to hoursBailis & Kingsbury 2014
Cross-cloud provider~20-50 per yearSeconds to hoursIndustry estimates

Real system choices and their measured impact:

SystemCAP ChoiceWhat Happens During PartitionMeasured Impact
Google SpannerCP (chooses consistency)Writes block until partition healsWrite latency spikes to seconds-minutes; reads from stale replicas still work
Amazon DynamoDBAP (chooses availability)Both sides accept writes, conflict resolution later0 downtime but conflict rate spikes to 0.01-0.1% of writes
Cassandra (QUORUM)Tunable (CP at QUORUM)Operations fail if quorum unreachableAvailability drops from 99.99% to ~66% during AZ loss
MongoDB (primary reads)CPElection takes 10-30 seconds, writes fail10-30s write unavailability on primary failure

Staff-level insight: In practice, most systems operate in a non-partitioned state 99.9%+ of the time. The more important question is: what consistency and latency tradeoffs do you make during normal operation? This is where PACELC comes in.

Q: What is PACELC and why is it more useful than CAP?

A: PACELC extends CAP by addressing what happens when there is no partition (the common case). It states: if there is a Partition, choose Availability or Consistency; Else (normal operation), choose Latency or Consistency.

PACELC decision matrix:

SystemDuring Partition (PA/PC)Normal Operation (EL/EC)ClassificationWrite Latency (normal)
DynamoDBPA (available)EL (low latency)PA/EL~5ms single-region
Cassandra (ONE)PAELPA/EL~2ms local
SpannerPC (consistent)EC (consistent)PC/EC~7ms single-region, ~100ms+ cross-region
MongoDBPCECPC/EC~2-5ms local
CockroachDBPCECPC/EC~10-20ms cross-AZ
Cassandra (QUORUM)PCECPC/EC~5-10ms (waits for majority)

Why PACELC is more useful: CAP only matters during partitions (rare). PACELC tells you the cost you pay every single request. Spanner's choice of EC means every write pays a cross-replica synchronization cost (~7ms minimum for TrueTime). DynamoDB's EL choice means writes are fast but you may read stale data. This latency-consistency tradeoff affects your system's performance profile 100% of the time, not just during rare partition events.

Decision guide for staff engineers:

  • Financial transactions, inventory: PC/EC (Spanner, CockroachDB) -- correctness is worth the latency cost
  • Social feeds, product catalogs: PA/EL (DynamoDB, Cassandra ONE) -- stale data is acceptable, speed matters
  • User sessions, shopping carts: PA/EL with conflict resolution -- availability matters, last-write-wins is fine
  • Leaderboards, analytics: PA/EL -- eventual consistency is the natural model

Latency Deep Dive

Q: How do you quantify replication lag? (sync vs async with real latency numbers)

A: Replication lag is the time between a write being committed on the primary and being visible on a replica. This is the fundamental measure of data freshness in distributed systems.

Replication ModeWithin AZCross-AZ (same region)Cross-RegionWrite Latency Impact
Synchronous<1ms lag, 0 data loss1-3ms lag, 0 data loss50-200ms lag, 0 data lossWrite blocked until replica ACKs
Semi-synchronous<1ms lag1-3ms lag50-200ms lagWrite blocked until 1 replica ACKs
Asynchronous0-10ms lag typical1-50ms lag typical100ms-2s lag typicalNo write impact

Measuring replication lag in PostgreSQL:

-- On primary: check replication lag for each replica
SELECT
    client_addr,
    state,
    sent_lsn,
    write_lsn,
    flush_lsn,
    replay_lsn,
    pg_wal_lsn_diff(sent_lsn, replay_lsn) AS replay_lag_bytes,
    write_lag,
    flush_lag,
    replay_lag
FROM pg_stat_replication;

-- Example output:
-- client_addr | replay_lag_bytes | replay_lag
-- 10.0.1.5   | 1048576          | 00:00:00.023   (23ms, same region)
-- 10.0.2.5   | 8388608          | 00:00:00.847   (847ms, cross-region)

The staleness budget: Different features tolerate different staleness. Define a staleness budget per feature:

  • Account balance after deposit: 0ms (must read-your-writes) -- route to primary
  • Product inventory count: <5s (approximate is fine) -- replica is fine
  • User profile display: <30s (eventual is acceptable) -- any replica
  • Analytics dashboard: <5min (batch lag acceptable) -- read from analytics replica

Staff-level pattern: Implement read-your-writes consistency by routing the writing user's subsequent reads to the primary (or a synchronous replica) for a brief window (e.g., 5 seconds), while routing all other users' reads to async replicas. This gives the consistency guarantee where it matters without paying the latency cost globally.

Q: What is tail latency amplification and how do you calculate it?

A: Tail latency amplification occurs when a single user request fans out to multiple backend services, and the overall latency is determined by the slowest response. This is one of the most important phenomena in microservice architectures.

The math:

If a single service has p99 latency of 10ms,
and a user request fans out to N services in parallel:

P(all N services respond within 10ms) = 0.99^N

N=1:  99.0% within 10ms   (p99 = 10ms)
N=5:  95.1% within 10ms   (effective p95 = 10ms, p99 ~ 15ms)
N=10: 90.4% within 10ms   (effective p90 = 10ms, p99 ~ 25ms)
N=20: 81.8% within 10ms   (effective p82 = 10ms, p99 ~ 50ms)
N=50: 60.5% within 10ms   (effective p60 = 10ms, p99 ~ 100ms+)
N=100: 36.6% within 10ms  (p99 becomes unusable)

Google's real-world data (from Jeff Dean's "Tail at Scale" paper): A Google web search fans out to thousands of leaf servers. Even if each server's p99 is 10ms, the overall p99 of the search would be terrible without mitigation. Google measured that a request touching 100 servers has a 63% chance of experiencing at least one server's p99 latency.

Mitigation strategies with measured impact:

StrategyHow It WorksMeasured ImprovementCost
Hedged requestsSend same request to 2 replicas, use first responsep99 reduced by 30-50%~5% extra load (cancel second after first returns)
Tied requestsSend to 2 replicas, first to start processing cancels the otherp99 reduced by 40-60%~2% extra load
Backup requests with delaySend backup only if primary does not respond within p95 timep99 reduced by 20-40%~2% extra load
Reducing fan-outRestructure to call fewer servicesProportional to fan-out reductionArchitectural change

Python implementation of hedged requests:

import asyncio
from typing import Any

async def hedged_request(
    func,
    *args,
    replicas: list[str] = None,
    hedge_delay_ms: float = 10.0,  # p95 of the service
    **kwargs
) -> Any:
    """Send request to primary; if no response within hedge_delay,
    send to a second replica and return whichever responds first."""

    async def call_replica(replica: str):
        return await func(replica, *args, **kwargs)

    primary = replicas[0]
    backup = replicas[1]

    # Start primary request
    primary_task = asyncio.create_task(call_replica(primary))

    # Wait for hedge delay
    done, pending = await asyncio.wait(
        {primary_task},
        timeout=hedge_delay_ms / 1000.0
    )

    if done:
        return primary_task.result()

    # Primary is slow; send hedged request
    backup_task = asyncio.create_task(call_replica(backup))

    done, pending = await asyncio.wait(
        {primary_task, backup_task},
        return_when=asyncio.FIRST_COMPLETED
    )

    # Cancel the slower one
    for task in pending:
        task.cancel()

    return done.pop().result()
Q: How do you measure and reduce tail latency?

A: Measuring tail latency requires high-resolution histograms, not averages. Averages hide the worst cases that affect real users.

What to measure:

Essential latency percentiles:
  p50  (median): The "typical" experience
  p90:  10% of users see this or worse
  p95:  Your SLO target for most services
  p99:  The "tail" -- affects 1 in 100 requests
  p99.9: The "far tail" -- matters at high traffic (10K rps = 10 req/sec at p99.9)

Latency budget example:
  User-facing API SLO: p99 < 200ms

  Breakdown:
    Load balancer:    ~1ms
    Auth middleware:   ~5ms
    Business logic:   ~10ms
    Database query:   ~20ms p50, ~100ms p99  <-- main contributor
    External API:     ~50ms p50, ~500ms p99  <-- tail latency source
    Serialization:    ~2ms

  Total p50: ~88ms (fine)
  Total p99: ~618ms (SLO violation!)

Common tail latency causes and fixes:

CauseSymptomFixExpected Improvement
GC pauses (Java/Go)Periodic p99 spikes of 50-200msTune GC, reduce allocation rate, use off-heap50-80% p99 reduction
Lock contentionp99 grows with concurrencyShard locks, use lock-free structures40-70% p99 reduction
Noisy neighbors (shared infra)Random spikes not correlated with loadDedicated instances, CPU pinning30-60% p99 reduction
Cold cache on deploysp99 spikes after each deploymentCache warming, canary deploysEliminates deploy-related spikes
Connection pool exhaustionp99 spikes under moderate loadRight-size pools, add connection timeout60-90% p99 reduction

Key principle: Optimize the p99, not the p50. Reducing p50 from 10ms to 5ms has minimal user impact. Reducing p99 from 500ms to 100ms dramatically improves user experience for the users who are most likely to churn.

Q: What are the real costs of strong consistency vs eventual consistency?

A: The costs are measurable across four dimensions: latency, throughput, availability, and operational complexity.

DimensionStrong ConsistencyEventual ConsistencyQuantified Difference
Write latencyMust wait for quorum ACKACK after local write2-10x higher for strong (5ms vs 1ms local, 150ms vs 5ms cross-region)
Write throughputLimited by slowest quorum memberLimited only by local node3-10x lower for strong
Read latencyMay need to check leader or read quorumAny replica, local1-5x higher for strong reads
Availability during AZ lossUnavailable if quorum lostReduced capacity but available99.95% vs 99.99% typical
Conflict resolutionNot needed (no conflicts by design)Required (LWW, vector clocks, CRDTs)0 engineering cost vs significant
Bug surface areaLower (serial execution model)Higher (concurrent updates, stale reads)~3x more consistency-related bugs with eventual

The hidden cost of eventual consistency: Engineering time spent debugging "impossible" bugs caused by stale reads, race conditions, and conflict resolution. At Amazon, the DynamoDB team estimates that 30% of customer support tickets for DynamoDB are related to eventual consistency confusion. The operational cost of training engineers and debugging consistency-related issues is substantial.

Hybrid approach (most common in practice): Use strong consistency for the critical path (payments, inventory decrement, user auth) and eventual consistency for everything else (product reviews, recommendations, activity feeds). This gives you the correctness guarantees where they matter while keeping costs low for the majority of operations.

Q: How do you quantify the "cost of coordination"? (Raft/Paxos latency overhead)

A: Coordination (consensus) requires network round trips between nodes to agree on the order of operations. This adds a fixed latency floor to every write operation.

Consensus protocol latency breakdown:

Raft/Paxos single write:
  1. Client sends write to leader          ~0.5ms (network)
  2. Leader appends to local log            ~0.1ms (disk fsync)
  3. Leader sends AppendEntries to followers ~0.5ms (network)
  4. Followers append + ACK                 ~0.6ms (disk + network)
  5. Leader commits after majority ACK      ~0.1ms
  6. Leader responds to client              ~0.5ms

  Minimum: ~2.3ms within same AZ (dominated by fsync + RTT)
  Cross-AZ: ~5-10ms (RTT is 1-3ms per hop)
  Cross-region: ~100-300ms (RTT is 50-150ms per hop)

Throughput impact of coordination:

ConfigurationMax Write ThroughputLatency (p50)Why
Single node (no coordination)100K+ writes/sec<1msOnly local disk latency
3-node Raft (same AZ)20-50K writes/sec~2-3ms2 network RTTs + fsync
5-node Raft (cross-AZ)5-15K writes/sec~5-10msCross-AZ RTT dominates
5-node Raft (cross-region)500-2K writes/sec~100-300msCross-region RTT dominates

Batching to amortize coordination cost: Instead of one consensus round per write, batch multiple writes into a single consensus round. etcd uses this approach -- batching 100 writes into one Raft round reduces per-write latency overhead from 2.3ms to ~0.023ms. The tradeoff: individual write latency increases slightly (waiting for the batch to fill) but throughput increases dramatically.

Throughput, Capacity & Scaling

Q: What is the throughput-latency tradeoff? (batching example with numbers)

A: Batching increases throughput by amortizing per-operation overhead, but it increases individual request latency because each request must wait for the batch to fill or a timeout to fire.

Concrete Kafka producer example:

Kafka producer configuration impact:

batch.size=1 (no batching):
  Throughput: ~10,000 messages/sec
  Latency per message: ~2ms
  Network utilization: High (1 TCP packet per message)

batch.size=16384, linger.ms=5:
  Throughput: ~200,000 messages/sec (20x improvement)
  Latency per message: ~7ms (2ms network + 5ms batch wait)
  Network utilization: Low (hundreds of messages per TCP packet)

batch.size=65536, linger.ms=50:
  Throughput: ~500,000 messages/sec (50x improvement)
  Latency per message: ~52ms (2ms network + 50ms batch wait)
  Network utilization: Very low

Database batching (INSERT):

PostgreSQL INSERT performance:

Single row INSERTs (1 at a time):
  Throughput: ~5,000 rows/sec
  Each INSERT: parse + plan + execute + fsync ack = ~0.2ms

Batched INSERT (100 rows per statement):
  Throughput: ~200,000 rows/sec (40x improvement)
  Each batch: parse + plan + execute + fsync ack = ~0.5ms
  Per-row amortized: ~0.005ms

  Throughput: ~500,000 rows/sec
  Bypasses query planner entirely

The sweet spot: For most systems, batching with a 5-20ms linger/timeout provides 10-20x throughput improvement with minimal latency impact. Beyond 50ms, user-perceivable latency becomes a concern for interactive workloads. For batch processing, linger in the hundreds of milliseconds is acceptable.

Q: How do you model capacity planning? (formulas for QPS to compute to cost)

A: Capacity planning translates business growth projections into infrastructure requirements and cost. It is one of the most impactful exercises a staff engineer performs.

The capacity planning pipeline:

Step 1: Business metrics --> QPS
  Users (MAU): 10M
  DAU/MAU ratio: 30% --> 3M DAU
  Actions per user per day: 20
  Peak-to-average ratio: 3x

  Avg QPS = (3M * 20) / 86,400 = 694 req/sec
  Peak QPS = 694 * 3 = 2,083 req/sec
  Design capacity (1.5x headroom) = 3,125 req/sec

Step 2: QPS --> Compute
  Load test result: 1 c6i.xlarge handles 500 req/sec at p99 < 100ms
  Instances needed: ceil(3,125 / 500) = 7 instances
  With N+2 redundancy: 9 instances (survives 2 failures)

Step 3: Compute --> Cost
  9 * c6i.xlarge on-demand: 9 * $0.17/hr * 730 = $1,117/mo
  With 1yr Savings Plan: 9 * $0.107/hr * 730 = $703/mo
  Plus ALB, RDS, ElastiCache, etc.: ~$2,500/mo total

Step 4: Growth projection
  Expected growth: 50% YoY
  Year 2: 15M MAU --> ~$3,750/mo
  Year 3: 22.5M MAU --> ~$5,625/mo (if linear scaling)
  Year 3 with optimization: ~$4,200/mo (sub-linear due to caching improvements)

Key formulas:

QPS = (DAU * actions_per_user) / seconds_per_day
Peak QPS = avg_QPS * peak_multiplier  (typically 2-5x)
Instances = ceil(peak_QPS / throughput_per_instance) + redundancy
Monthly cost = instances * hourly_rate * 730

Storage growth:
  Monthly data = new_records_per_day * avg_record_size * 30
  Cumulative after N months = sum(monthly_data_i for i in 1..N) * (1 - compression_ratio)

Common mistakes:

  • Using p50 throughput from load tests instead of p99. Size for the throughput at which p99 latency meets your SLO, not the maximum throughput the instance can handle.
  • Forgetting connection overhead. A 4-vCPU instance handling 2000 WebSocket connections may have only 50% CPU headroom for actual request processing.
  • Not accounting for data growth. Storage costs are cumulative; compute costs are concurrent. A service that grows 50% YoY in users may grow 100% YoY in storage (more data per user over time).
Q: What is Amdahl's Law and how does it limit distributed system scaling?

A: Amdahl's Law states that the maximum speedup from parallelization is limited by the sequential portion of the workload. In distributed systems, this translates to: no matter how many nodes you add, the serial bottleneck limits overall throughput.

The formula:

Speedup(N) = 1 / (S + (1 - S) / N)

Where:
  S = fraction of work that is sequential (cannot be parallelized)
  N = number of parallel workers/nodes

Examples:
  S = 5% sequential (e.g., lock contention, consensus):
    N=10:  Speedup = 6.9x   (69% efficiency)
    N=100: Speedup = 16.8x  (17% efficiency)
    N=1000: Speedup = 19.6x (2% efficiency)  <-- diminishing returns
    Maximum possible: 20x (1/0.05)

  S = 1% sequential:
    N=10:  Speedup = 9.2x   (92% efficiency)
    N=100: Speedup = 50.3x  (50% efficiency)
    N=1000: Speedup = 90.9x (9% efficiency)
    Maximum possible: 100x

Real distributed system bottlenecks that create "S":

BottleneckEffective S (%)Max Theoretical SpeedupExample
Global lock / single leader10-50%2-10xSingle-primary database writes
Consensus per write2-10%10-50xRaft-based state machines
Shared counter / sequence5-20%5-20xAuto-increment IDs
Cross-shard transactions1-5% of transactions20-100xDistributed 2PC
Aggregation step (MapReduce)1-10%10-100xReduce phase in large jobs

Staff-level takeaway: Before adding more nodes, identify and measure your sequential fraction. If 10% of your processing is sequential, adding more than 20 nodes gives nearly zero additional throughput. Invest in reducing S (sharding the bottleneck, removing coordination, using CRDTs) before investing in adding N (more nodes).

Sharding, Storage & Architecture Tradeoffs

Q: What are the tradeoffs of different sharding strategies?

A: Sharding strategy determines how data is distributed across nodes. The choice affects query patterns, hot spots, rebalancing cost, and operational complexity.

StrategyDistributionRange QueriesHot SpotsRebalancingBest For

Range-basedNon-uniform (ranges of keys)Efficient within shardCommon (e.g., recent dates on one shard)Split/merge individual rangesTime-series, alphabetical lookups

Quantified impact of shard key choice:

E-commerce platform, 100M orders/year:

Shard by order_id (hash):
  Distribution: Even (~20M orders per shard with 5 shards)
  "Get order by ID": 1 shard hit, ~2ms
  "Get all orders for user X": scatter to all 5 shards, ~15ms

Shard by user_id (hash):
  Distribution: Skewed (power users have 10,000x more orders)
  "Get order by ID": need user_id lookup first, ~4ms
  "Get all orders for user X": 1 shard hit, ~2ms
  Hot shard risk: top 1% of users on same shard

Shard by user_id with virtual sharding:
  Use consistent hashing with 1000 virtual nodes
  "Get all orders for user X": 1 virtual node lookup, ~3ms
  Rebalancing: add node, move ~1/N of data

The cross-shard query tax: Any query that spans multiple shards incurs a scatter-gather cost. If your most common query pattern does not align with your shard key, you will pay this tax on every request. Measure the percentage of queries that are cross-shard before choosing your strategy. Target <10% cross-shard queries for production workloads.

Q: What is the read-write tradeoff? (read replicas, write amplification, CQRS metrics)

A: Most systems are read-heavy (90-99% reads). Optimizing reads often means making writes more expensive (write amplification) and vice versa. CQRS (Command Query Responsibility Segregation) formalizes this tradeoff.

Read-write ratio benchmarks:

SystemRead:Write RatioOptimization Strategy
Twitter feed1000:1Fan-out on write (precompute feeds)
E-commerce catalog100:1Read replicas + heavy caching
IoT telemetry1:100Write-optimized (LSM trees, time-series DBs)
Chat messages10:1Balanced (write-behind cache)
Financial ledger5:1Write-optimized with materialized read views

Write amplification in practice:

Twitter's fan-out-on-write approach:
  When a user with 10M followers tweets:
  1 logical write --> 10M timeline inserts (physical writes)
  Write amplification factor: 10,000,000x

  This is why Twitter uses a hybrid:
  - Users with <500K followers: fan-out on write (fast reads)
  - Users with 500K+ followers: fan-out on read (merge at read time)
  - This reduces write amplification by 99% for celebrity tweets

CQRS metrics to track:

  • Write path latency: Time from command to committed state change
  • Projection lag: Time from write commit to read model update (typically 10-500ms)
  • Read path latency: Time from query to response (should be <10ms with denormalized views)
  • Write amplification ratio: Physical writes / logical writes (target <10x for most systems)
  • Storage amplification: Total storage / unique data (CQRS typically 2-5x due to multiple views)

Q: How do you quantify the blast radius of a failure?

A: Blast radius measures the scope of impact when a component fails. Quantifying it helps prioritize reliability investments and design isolation boundaries.

Blast radius formula:

Blast Radius = Affected Users * Impact Severity * Duration

Where:
  Affected Users = % of total users who experience degradation
  Impact Severity = 0 (no impact) to 1 (complete outage)
  Duration = minutes of impact

Blast Radius Score = Affected_Users% * Severity * Duration_minutes

Example failures:
  Single pod crash (Kubernetes):
    Affected: ~0.1% (1 of 1000 pods), Severity: 1.0, Duration: 0.5 min
    Score: 0.001 * 1.0 * 0.5 = 0.0005 (negligible)

  AZ outage (one of three):
    Affected: ~33%, Severity: 0.5 (degraded, not down), Duration: 30 min
    Score: 0.33 * 0.5 * 30 = 4.95 (moderate)

  DNS failure (global):
    Affected: 100%, Severity: 1.0, Duration: 60 min
    Score: 1.0 * 1.0 * 60 = 60 (critical)

  Database primary failure:
    Affected: 100%, Severity: 0.8 (reads may work), Duration: 5 min (failover)
    Score: 1.0 * 0.8 * 5 = 4.0 (moderate)

Reducing blast radius -- architectural patterns:

PatternBlast Radius ReductionCost IncreaseExample
Cell-based architectureLimit to 1 cell (~5-10% of users)20-50% (duplicate infra per cell)Slack, AWS (per-region cells)
Bulkhead isolationLimit to 1 service10-20% (dedicated resources)Netflix: separate thread pools per dependency
Circuit breakersPrevent cascading failuresMinimal (~5%)Hystrix, Resilience4j
Multi-region active-activeLimit to 1 region80-100% (full duplication)CockroachDB, Spanner

Staff-level practice: Create a blast radius map of your architecture. For each component, document: what percentage of users are affected if it fails, what is the expected severity, and what is the expected recovery time. Use this to prioritize reliability investments on the highest blast-radius components first.

Q: How do you measure consensus protocol performance? (Raft vs Paxos vs Zab)

A: Consensus protocols have different performance characteristics that matter for system design choices.

MetricRaft (etcd)Multi-Paxos (Chubby)Zab (ZooKeeper)EPaxos
Normal write latency2-5ms2-5ms2-5ms1-3ms (no leader)
Leader election time1-3 seconds5-15 seconds1-5 secondsN/A (leaderless)
Throughput (3 nodes)~15K writes/sec~15K writes/sec~10K writes/sec~20K writes/sec
Message complexityO(N) per writeO(N) per writeO(N) per writeO(N) per write
UnderstandabilityHigh (designed for it)Low (notoriously complex)MediumLow
Used byetcd, CockroachDB, TiKVChubby, SpannerZooKeeper, Kafka (KRaft)Research, some new DBs

Key performance factor: In practice, the protocol itself rarely determines performance. Network RTT, disk fsync latency, and batching strategy dominate. A well-tuned Raft implementation and a well-tuned Multi-Paxos implementation will have nearly identical performance. The real difference is operational: Raft is easier to debug, reason about, and implement correctly.

Kafka's evolution (ZAB to KRaft): Kafka moved from ZooKeeper (ZAB) to KRaft (Raft-based) for metadata management. The measured improvements: metadata propagation latency dropped from 10-100ms to 1-10ms, partition count scalability increased from ~200K to millions, and operational complexity decreased significantly (no separate ZooKeeper cluster to manage).

Q: How do you measure the overhead of service mesh / sidecar proxies?

A: Service meshes (Istio, Linkerd, Consul Connect) add a sidecar proxy to every pod, intercepting all network traffic. This adds measurable latency, CPU, and memory overhead.

Measured overhead (Istio with Envoy sidecar):

MetricWithout MeshWith IstioOverhead
p50 latency (per hop)1ms2.5ms+1.5ms (+150%)
p99 latency (per hop)5ms12ms+7ms (+140%)
CPU per sidecar050-100m (millicores)+50-100m per pod
Memory per sidecar050-100 MB+50-100 MB per pod
Throughput impact100%85-95%-5 to -15%

At scale, these numbers compound:

Fleet: 1,000 pods across 50 services
  CPU overhead: 1,000 * 75m = 75 vCPU = ~19 c6i.xlarge instances
  Memory overhead: 1,000 * 75 MB = 75 GB
  Cost: 19 * $0.17/hr * 730 = $2,359/mo just for sidecar overhead

Latency: Request path touching 5 services
  Without mesh: 5 hops * 1ms = 5ms
  With mesh: 5 hops * 2.5ms = 12.5ms (2.5x increase)

  At p99: 5 hops * 12ms = 60ms (vs 25ms without mesh)

When the overhead is worth it: The service mesh provides mTLS, observability, traffic management, and circuit breaking. Without a mesh, you would implement these in application code, which has its own (higher) development cost. The mesh overhead is justified when you have 20+ services and need consistent security and observability. For smaller architectures (<10 services), the overhead is rarely worth it.

Lighter alternatives: Linkerd adds only 0.5-1ms latency per hop (vs Istio's 1.5ms) and uses ~20MB per sidecar (vs Istio's 50-100MB). Cilium uses eBPF at the kernel level, adding <0.5ms with near-zero memory overhead. Choose based on your feature needs and latency budget.

Q: Your service needs 99.99% availability across 3 regions. Design the metrics.

A: 99.99% (four nines) means at most 52.6 minutes of downtime per year, or ~4.38 minutes per month. Across 3 regions, this requires active-active architecture with robust failover.

Metric design:

SLI (Service Level Indicator):
  Success rate: % of requests returning 2xx within 500ms
  Measured at: Load balancer (closest to user)
  Window: 5-minute rolling

SLO (Service Level Objective):
  99.99% of requests succeed within 500ms over a 30-day window
  Error budget: 0.01% = ~4,320 failed requests per 43.2M total
  (assuming 1000 rps: 1000 * 86400 * 30 = 2.592B requests/month
   Error budget: 259,200 failed requests)

Regional SLOs:
  Each region independently: 99.95% (allows for brief regional degradation)
  Combined (with failover): 99.99%
  Math: P(all 3 regions down) = (1 - 0.9995)^3 = 1.25e-10 (essentially zero)
  P(at least 1 region up) = 1 - (0.0005)^3 = 99.9999999875%

Metrics to instrument:

MetricGranularityAlert ThresholdWhy
Request success ratePer region, per minute<99.95% for 5 minDetect regional degradation before users notice
Cross-region replication lagPer second>5 secondsStale failover data = data loss risk
Failover timePer event>30 secondsMust failover within error budget
DNS propagation timePer change>60 secondsDNS-based failover has inherent delay
Data consistency checkHourlyAny divergenceDetect split-brain or replication bugs
Error budget burn rateDaily>3x normal burnEarly warning before SLO breach

Architecture requirements for 99.99%:

  • Active-active in all 3 regions (not active-passive; passive failover is too slow)
  • Health-check-based global load balancing (Route 53 health checks, or better: Anycast DNS)
  • Data replication with RPO < 5 seconds (Recovery Point Objective: maximum data loss on failover)
  • RTO < 30 seconds (Recovery Time Objective: time to complete failover)
  • Chaos engineering: Monthly region failover drills to verify the failover path works
Q: Two architectures -- event-driven vs synchronous. How do you compare them quantitatively?

A: This is a common staff-level decision. You need a quantitative framework, not just qualitative "event-driven is more decoupled."

Comparison framework:

DimensionSynchronous (REST/gRPC)Event-Driven (Kafka/SQS)Winner
End-to-end latencySum of all hops: 5 services * 10ms = 50msAsync: 5ms (just enqueue) + eventual processingSync for user-facing; Async for background
ThroughputLimited by slowest service in chainEach service scales independentlyEvent-driven (decoupled scaling)
Failure couplingAny service down = entire chain failsMessages buffer; consumers process when readyEvent-driven (resilient)
Debugging complexityStack traces, request tracing is straightforwardDistributed tracing across async boundaries is hardSynchronous (simpler debugging)
Data consistencyTransactions possible within requestEventual consistency, saga patterns neededSynchronous (simpler consistency)
Infrastructure costCompute onlyCompute + message broker ($500-$5K/mo for Kafka)Synchronous (lower base cost)
Operational overheadStandard monitoringQueue depth monitoring, DLQ management, offset trackingSynchronous (less ops burden)

Quantified scenario -- order processing:

Synchronous: Order API --> Payment --> Inventory --> Shipping --> Email
  Latency: 50 + 200 + 30 + 20 + 50 = 350ms per order
  Failure rate: 1 - (0.999 * 0.998 * 0.999 * 0.999 * 0.997) = 0.8% failure rate
  Cost: 5 services compute only = $3,000/mo

  Problem: 0.8% of orders fail and need retry logic in every service

Event-driven: Order API --> Kafka --> [Payment, Inventory, Shipping, Email consume independently]
  Order API latency: 5ms (just write to Kafka)
  End-to-end processing: ~500ms (async, user does not wait)
  Failure rate: Each consumer retries independently, overall <0.01%
  Cost: $3,000 compute + $1,500 Kafka = $4,500/mo

  Problem: More complex to debug, eventual consistency

Verdict for this case: Event-driven wins on reliability and user experience
(fast order confirmation), costs $1,500/mo more but eliminates 0.8% order failures
which at 100K orders/month with $50 AOV = $40K/mo in recovered revenue.

Decision framework: Use synchronous for operations where the user needs an immediate, consistent response (e.g., checking inventory before confirming order). Use event-driven for operations where the user does not need to wait (e.g., sending confirmation email, updating analytics). Most production systems use a hybrid: synchronous for the critical path, event-driven for side effects.

Q: How do you measure data freshness vs consistency?

A: Data freshness measures how recent the data is, while consistency measures whether all readers see the same version. These are related but distinct concepts.

Freshness metrics:

Staleness = current_time - timestamp_of_data_read
Freshness SLO = % of reads where staleness < threshold

Example: Product price in an e-commerce search index
  Source of truth: PostgreSQL (updated via admin panel)
  Search index: Elasticsearch (updated via CDC pipeline)

  Measured pipeline:
    Postgres --> Debezium CDC --> Kafka --> Consumer --> Elasticsearch
    Typical latency: 500ms - 2s end-to-end
    p99 latency: 10s (during Kafka rebalances or ES backpressure)

  Freshness SLO: 99% of reads have staleness < 5s
  Current performance: 99.7% (meeting SLO)

  Monitoring query (in ES):
    stale_reads = count(reads where doc._last_updated < now() - 5s)
    freshness_percent = 1 - (stale_reads / total_reads)

When freshness matters more than consistency: For a product catalog, showing a price that is 2 seconds stale to all users is fine. Showing different prices to different users simultaneously (inconsistency) causes customer complaints. Freshness and consistency are often conflated but require different solutions: freshness requires fast replication pipelines; consistency requires coordination protocols.

Q: What autoscaling metrics matter beyond CPU?

A: CPU-based autoscaling is the default but often the worst choice. The best scaling metric depends on your workload's bottleneck.

MetricBest ForWhy CPU Fails HereTarget Value
Queue depthAsync workers, batch jobsCPU may be low while queue grows<1000 messages (job-dependent)
Request latency (p99)User-facing APIsCPU at 50% but latency spiking due to contention< SLO target (e.g., <100ms)
Concurrent connectionsWebSocket servers, long-poll10K idle connections = low CPU but high memory<80% of max connections
Memory utilizationIn-memory caches, JVM appsMemory pressure causes GC, not CPU spike<75% to avoid OOM
Custom business metricDomain-specific workloadsCPU does not correlate with business loadVaries (e.g., orders/min)

Multi-signal scaling: The best production setups use composite metrics. For example, scale up when ANY of: CPU > 70%, p99 latency > 100ms, or queue depth > 5000. Scale down only when ALL metrics are below their thresholds. This asymmetric approach (aggressive scale-up, conservative scale-down) prevents oscillation.

Kubernetes HPA with custom metrics:

apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: api-server
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: api-server
  minReplicas: 3
  maxReplicas: 50
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Pods
    pods:
      metric:
        name: http_request_duration_seconds_p99
      target:
        type: AverageValue
        averageValue: "100m"  # 100ms
  - type: External
    external:
      metric:
        name: sqs_queue_depth
        selector:
          matchLabels:
            queue: api-tasks
      target:
        type: Value
        value: "5000"
