Classical ML Metrics

Q: What are the fundamental ML model metrics and their formulas?

A: These are the building blocks of ML evaluation. Every staff engineer working with ML systems must understand these deeply, not just their definitions but when each one is the right metric to optimize.

The confusion matrix foundation:

                    Predicted Positive    Predicted Negative
Actual Positive     True Positive (TP)    False Negative (FN)
Actual Negative     False Positive (FP)   True Negative (TN)

Accuracy  = (TP + TN) / (TP + TN + FP + FN)
Precision = TP / (TP + FP)       -- "Of predicted positives, how many are correct?"
Recall    = TP / (TP + FN)       -- "Of actual positives, how many did we catch?"
F1 Score  = 2 * (Precision * Recall) / (Precision + Recall)  -- harmonic mean

Specificity = TN / (TN + FP)    -- "Of actual negatives, how many did we correctly identify?"
FPR = FP / (FP + TN) = 1 - Specificity  -- False Positive Rate

AUC-ROC explained: The ROC curve plots True Positive Rate (recall) vs False Positive Rate at various classification thresholds. AUC (Area Under Curve) summarizes this as a single number from 0 to 1. AUC = 0.5 means random guessing; AUC = 1.0 means perfect separation.

from sklearn.metrics import (
    accuracy_score, precision_score, recall_score,
    f1_score, roc_auc_score, classification_report,
    confusion_matrix
)
import numpy as np

def evaluate_model(y_true, y_pred, y_prob=None):
    """Comprehensive model evaluation with business context."""
    cm = confusion_matrix(y_true, y_pred)
    tn, fp, fn, tp = cm.ravel()

    metrics = {
        'accuracy': accuracy_score(y_true, y_pred),
        'precision': precision_score(y_true, y_pred),
        'recall': recall_score(y_true, y_pred),
        'f1': f1_score(y_true, y_pred),
        'true_positives': int(tp),
        'false_positives': int(fp),
        'false_negatives': int(fn),
        'true_negatives': int(tn),
    }

    if y_prob is not None:
        metrics['auc_roc'] = roc_auc_score(y_true, y_prob)

    # Business impact calculation
    cost_fp = 10   # Cost of a false positive (e.g., manual review)
    cost_fn = 1000 # Cost of a false negative (e.g., missed fraud)

    metrics['total_cost'] = fp * cost_fp + fn * cost_fn
    metrics['cost_per_prediction'] = metrics['total_cost'] / len(y_true)

    return metrics

Why accuracy is often the wrong metric: For imbalanced datasets (99% negative, 1% positive), a model that always predicts "negative" achieves 99% accuracy but 0% recall. In fraud detection with 0.1% fraud rate, accuracy is meaningless. Use precision-recall AUC or F1 instead.

Q: When do you optimize for precision vs recall? (real examples)

A: The choice depends on the relative cost of false positives vs false negatives. This is a business decision, not a technical one.

ScenarioOptimize ForWhyAcceptable Tradeoff
Spam filter (email)Precision (avoid FP)False positive = legitimate email in spam = missed business opportunityLet some spam through (low recall) rather than block good email
Cancer screeningRecall (catch all cases)False negative = missed cancer = potentially fatalMore false positives (unnecessary biopsies) are acceptable
Fraud detectionRecall (with precision floor)False negative = undetected fraud = direct financial lossBlock some legitimate transactions (FP) but maintain precision >50% to limit customer friction
Content moderationRecall (remove harmful content)False negative = harmful content shown to users = brand/legal riskSome over-moderation (FP) is acceptable; appeal process handles it
Search rankingPrecision@KUsers only look at top 10 results; irrelevant results destroy trustMissing some relevant results (low recall) is fine if top results are good
Autonomous drivingRecall for pedestrians (near 100%)False negative = hitting a pedestrian = catastrophicMassive FP rate (phantom braking) is better than missing a pedestrian

Q: What is the cost of a false positive vs false negative? (business impact calculation)

A: Assigning dollar values to prediction errors transforms ML evaluation from a technical exercise into a business decision framework. This is one of the most powerful tools a staff engineer can bring to an ML team.

Cost matrix framework:

Expected Cost = (FP_count * Cost_FP) + (FN_count * Cost_FN) + (TP_count * Benefit_TP)

Threshold optimization:
  For each possible classification threshold (0.0 to 1.0):
    Calculate expected cost
    Choose threshold that minimizes total cost

  This is NOT the same as choosing threshold = 0.5!
  If Cost_FN >> Cost_FP, optimal threshold is lower (catch more positives)

Python implementation of cost-sensitive threshold optimization:

import numpy as np
from sklearn.metrics import confusion_matrix

def find_optimal_threshold(y_true, y_prob, cost_fp, cost_fn, benefit_tp=0):
    """Find classification threshold that minimizes total business cost."""
    thresholds = np.arange(0.01, 1.0, 0.01)
    best_threshold = 0.5
    min_cost = float('inf')

    results = []
    for threshold in thresholds:
        y_pred = (y_prob >= threshold).astype(int)
        tn, fp, fn, tp = confusion_matrix(y_true, y_pred).ravel()

        total_cost = (fp * cost_fp) + (fn * cost_fn) - (tp * benefit_tp)
        results.append({
            'threshold': round(threshold, 2),
            'cost': total_cost,
            'precision': tp / (tp + fp) if (tp + fp) > 0 else 0,
            'recall': tp / (tp + fn) if (tp + fn) > 0 else 0,
        })

        if total_cost < min_cost:
            min_cost = total_cost
            best_threshold = threshold

    return best_threshold, min_cost, results

Q: What is model drift and how do you detect it?

A: Model drift occurs when the statistical properties of the data or the relationship between features and targets change over time, causing model performance to degrade. There are three types of drift, each with different detection methods.

Detection methods with implementation:

from scipy import stats
import numpy as np

def detect_data_drift(reference_data, current_data, feature_name, threshold=0.05):
    """Detect data drift using Kolmogorov-Smirnov test."""
    statistic, p_value = stats.ks_2samp(reference_data, current_data)

    return {
        'feature': feature_name,
        'ks_statistic': round(statistic, 4),
        'p_value': round(p_value, 6),
        'drift_detected': p_value < threshold,
        'severity': 'high' if statistic > 0.2 else 'medium' if statistic > 0.1 else 'low'
    }

def population_stability_index(expected, actual, bins=10):
    """Calculate PSI to measure distribution shift.
    PSI < 0.1: No significant shift
    PSI 0.1-0.25: Moderate shift, investigate
    PSI > 0.25: Significant shift, retrain
    """
    expected_pct = np.histogram(expected, bins=bins)[0] / len(expected)
    actual_pct = np.histogram(actual, bins=bins)[0] / len(actual)

    # Avoid division by zero
    expected_pct = np.clip(expected_pct, 0.001, None)
    actual_pct = np.clip(actual_pct, 0.001, None)

    psi = np.sum((actual_pct - expected_pct) * np.log(actual_pct / expected_pct))
    return round(psi, 4)

Real-world drift examples:

  • COVID-19 (2020): Every e-commerce, travel, and financial model drifted massively. Purchase patterns, fraud patterns, and credit risk all changed overnight. Companies that had drift detection caught this in days; others discovered it in quarterly reviews.
  • Seasonal drift: A retail demand forecasting model trained on summer data will drift every winter. Solution: include seasonality features or retrain quarterly.
  • Feature pipeline failures: A feature that stops updating (always returns the same value) causes silent drift. The model still produces outputs, just increasingly wrong ones. Monitor feature freshness alongside model performance.

Monitoring cadence: Run drift detection daily for high-stakes models (fraud, pricing), weekly for medium-stakes (recommendations, search ranking), and monthly for low-stakes (internal analytics). Alert on PSI > 0.1 and auto-trigger retraining pipeline at PSI > 0.25.

Q: How do you measure model fairness and bias?

A: Model fairness is both an ethical imperative and a regulatory requirement (EU AI Act, US algorithmic accountability proposals). Staff engineers must build fairness metrics into every ML pipeline that affects people.

Key fairness metrics:

MetricDefinitionFormulaWhen to Use
Demographic ParityPrediction rate is equal across groupsP(Y_hat=1|A=a) = P(Y_hat=1|A=b)When selection rate should be independent of protected attribute
Equalized OddsTPR and FPR are equal across groupsTPR_a = TPR_b AND FPR_a = FPR_bWhen errors should be equally distributed
Predictive ParityPrecision is equal across groupsPrecision_a = Precision_bWhen positive predictions should be equally reliable
CalibrationPredicted probabilities match actual rates per groupP(Y=1|score=s, A=a) = P(Y=1|score=s, A=b)When risk scores are used for decision-making

Important impossibility result: Choquet-Deny theorem shows that except in trivial cases, you cannot simultaneously achieve demographic parity, equalized odds, and predictive parity. You must choose which fairness criterion matters most for your use case.

def fairness_metrics(y_true, y_pred, sensitive_attr):
    """Calculate fairness metrics across a sensitive attribute."""
    groups = np.unique(sensitive_attr)
    results = {}

    for group in groups:
        mask = sensitive_attr == group
        y_t = y_true[mask]
        y_p = y_pred[mask]

        tp = np.sum((y_t == 1) & (y_p == 1))
        fp = np.sum((y_t == 0) & (y_p == 1))
        fn = np.sum((y_t == 1) & (y_p == 0))
        tn = np.sum((y_t == 0) & (y_p == 0))

        results[group] = {
            'selection_rate': np.mean(y_p),  # demographic parity check
            'tpr': tp / (tp + fn) if (tp + fn) > 0 else 0,  # equalized odds
            'fpr': fp / (fp + tn) if (fp + tn) > 0 else 0,  # equalized odds
            'precision': tp / (tp + fp) if (tp + fp) > 0 else 0,  # predictive parity
        }

    # Disparate impact ratio (4/5 rule: ratio should be > 0.8)
    rates = [r['selection_rate'] for r in results.values()]
    results['disparate_impact_ratio'] = min(rates) / max(rates) if max(rates) > 0 else 0

    return results

The 4/5 (80%) rule: A widely used legal standard (US EEOC guidelines). If the selection rate for a protected group is less than 80% of the rate for the highest group, there is evidence of adverse impact. For example, if a hiring model selects 50% of Group A but only 35% of Group B, the ratio is 35/50 = 70%, which is below 80% and flags potential bias.

LLM-Specific Metrics

Q: What are LLM-specific metrics? (TTFT, tokens/sec, total latency, cost per query)

A: LLMs have a unique performance profile because they generate output tokens sequentially. This creates metrics that do not exist in traditional ML systems.

MetricDefinitionGood TargetWhy It Matters
TTFT (Time to First Token)Time from request to first output token<500msUser perception of responsiveness; dominated by prompt processing time
TPS (Tokens per Second)Output token generation rate>30 tok/s for streamingDetermines perceived speed of completion
Total LatencyTTFT + (output_tokens / TPS)Varies by use caseEnd-to-end response time
Input Token CountTokens in the promptMinimize while maintaining qualityDirectly impacts cost and TTFT
Output Token CountTokens generatedRight-sized for taskDirectly impacts cost and total latency
Cost per QueryInput tokens * price + Output tokens * price<$0.01 for most appsDetermines unit economics viability

Latency breakdown for a typical LLM query:

User sends a 2000-token prompt, receives 500-token response:

TTFT breakdown:
  Network latency (client to API):          ~50ms
  Prompt tokenization:                      ~5ms
  Prompt processing (prefill):              ~200ms (2000 tokens at ~10K tok/s)
  First token generation:                   ~30ms
  Network latency (API to client):          ~50ms
  Total TTFT:                               ~335ms

Streaming phase:
  500 tokens at 40 tok/s:                   ~12.5 seconds

Total wall-clock time:                      ~12.8 seconds
Total latency (non-streaming):              ~12.8 seconds
Perceived latency (streaming):              ~335ms to first token, then continuous flow

Why streaming matters: Without streaming, the user waits 12.8 seconds for any response. With streaming, they see the first token at 335ms and read as the response generates. This dramatically changes perceived performance without changing actual compute time.

Cost calculation with code:

def calculate_llm_cost(
    input_tokens: int,
    output_tokens: int,
    model: str = "gpt-4o"
) -> dict:
    """Calculate cost of an LLM API call."""
    pricing = {
        "gpt-4o":          {"input": 2.50,  "output": 10.00},
        "gpt-4o-mini":     {"input": 0.15,  "output": 0.60},
        "claude-sonnet":   {"input": 3.00,  "output": 15.00},
        "claude-haiku":    {"input": 0.25,  "output": 1.25},
    }

    rates = pricing[model]
    input_cost = (input_tokens / 1_000_000) * rates["input"]
    output_cost = (output_tokens / 1_000_000) * rates["output"]
    total = input_cost + output_cost

    return {
        "model": model,
        "input_tokens": input_tokens,
        "output_tokens": output_tokens,
        "input_cost": f"${input_cost:.6f}",
        "output_cost": f"${output_cost:.6f}",
        "total_cost": f"${total:.6f}",
        "cost_per_1k_queries": f"${total * 1000:.2f}",
    }

Q: What are hallucination metrics and how do you measure them?

A: Hallucinations are outputs that are fluent and confident but factually incorrect or unsupported by the provided context. Measuring them is one of the hardest problems in LLM evaluation.

Types of hallucination:

TypeDefinitionExampleDetection Method
Intrinsic hallucinationContradicts the source materialDocument says "revenue grew 10%", LLM says "revenue grew 25%"NLI (Natural Language Inference) against source
Extrinsic hallucinationAdds information not in sourceDocument mentions a company, LLM adds a founding date not in contextFaithfulness scoring against provided context
FabricationInvents facts, citations, URLs"According to Smith et al. (2023)..." when no such paper existsEntity and citation verification

Key metrics:

Faithfulness = (claims in response supported by context) / (total claims in response)
  Target: >0.95 for factual applications

Groundedness = (sentences grounded in provided documents) / (total sentences)
  Target: >0.90 for RAG systems

Hallucination Rate = 1 - Faithfulness
  Industry benchmarks:
    GPT-4 (vanilla): ~5-15% hallucination rate
    GPT-4 + RAG:     ~2-5% hallucination rate
    GPT-4 + RAG + guardrails: ~0.5-2% hallucination rate

Automated hallucination detection pipeline:

from openai import OpenAI

def check_faithfulness(context: str, response: str) -> dict:
    """Use model-as-judge to evaluate faithfulness of a response."""
    client = OpenAI()

    judge_prompt = f"""You are a factual accuracy evaluator.

Given the context below and a response, identify every factual claim
in the response and check if it is supported by the context.

Context:
{context}

Response to evaluate:
{response}

For each claim, output:
- claim: the factual statement
- supported: true/false
- evidence: the supporting text from context, or "not found"

Then provide:
- faithfulness_score: (supported claims / total claims)
- hallucinated_claims: list of unsupported claims

Output as JSON."""

    result = client.chat.completions.create(
        model="gpt-4o",
        messages=[{"role": "user", "content": judge_prompt}],
        response_format={"type": "json_object"},
    )

    return result.choices[0].message.content

Cost of hallucinations in production: A legal tech startup found that a 3% hallucination rate in their contract review tool led to 12% of users losing trust and canceling within 3 months. In healthcare applications, even 0.1% hallucination rate can have life-threatening consequences. The acceptable hallucination rate depends entirely on the domain and the downstream impact of incorrect information.

Diagnosing RAG failures:

Failure Mode 1: Low faithfulness, good retrieval
  Symptom: Relevant chunks retrieved but answer hallucinates
  Root cause: LLM ignoring context or adding information
  Fix: Stronger system prompt, lower temperature, use extracted quotes

Failure Mode 2: Low relevancy, poor retrieval
  Symptom: Retrieved chunks are off-topic
  Root cause: Embedding model mismatch, poor chunking strategy
  Fix: Re-embed with domain-tuned model, improve chunk boundaries

Failure Mode 3: Good retrieval, incomplete answer
  Symptom: Relevant info is in chunks but not reflected in answer
  Root cause: Too many chunks (context overload) or key info in middle
  Fix: Reduce K, rerank to put best chunks first (lost-in-the-middle effect)
Q: How do you build an eval framework for LLMs?

A: An LLM eval framework is the most critical infrastructure for any LLM-powered product. Without it, you cannot measure quality, detect regressions, or confidently ship changes.

Three layers of evaluation:

LayerMethodSpeedAccuracyCostWhen to Use
Unit evalsAutomated assertions, regex, semantic similaritySecondsLow-Medium~$0Every commit, CI/CD
Model-as-judgeGPT-4/Claude evaluates outputs against rubricMinutesMedium-High~$0.01/evalDaily, pre-deploy
Human evalDomain experts rate outputsHours-DaysHighest$1-10/evalWeekly, before major releases

Eval dataset structure:

eval_dataset = [
    {
        "id": "support-001",
        "input": "How do I reset my password?",
        "context": "Users can reset passwords at settings > security > reset password...",
        "expected_output": None,  # For open-ended generation
        "rubric": {
            "mentions_settings_page": {"weight": 0.3, "type": "contains"},
            "provides_step_by_step": {"weight": 0.3, "type": "judge"},
            "tone_is_helpful": {"weight": 0.2, "type": "judge"},
            "no_hallucination": {"weight": 0.2, "type": "faithfulness"},
        },
        "tags": ["password", "easy", "common"],
    },

]

def run_eval_suite(model_fn, dataset, judge_model="gpt-4o"):
    """Run evaluation suite and return aggregate metrics."""
    results = []
    for example in dataset:
        output = model_fn(example["input"], example.get("context"))

        scores = {}
        for criterion, config in example["rubric"].items():
            if config["type"] == "contains":
                scores[criterion] = 1.0 if criterion_text_in(output) else 0.0
            elif config["type"] == "judge":
                scores[criterion] = model_judge(output, criterion, judge_model)
            elif config["type"] == "faithfulness":
                scores[criterion] = faithfulness_check(output, example["context"])

        weighted_score = sum(
            scores[k] * example["rubric"][k]["weight"]
            for k in scores
        )
        results.append({"id": example["id"], "score": weighted_score, **scores})

    return {
        "mean_score": np.mean([r["score"] for r in results]),
        "pass_rate": np.mean([r["score"] >= 0.8 for r in results]),
        "results": results,
    }

Staff-level insight: The eval dataset is your product specification. Invest heavily in curating it. A 500-example eval set with well-defined rubrics is more valuable than the model itself, because it lets you confidently evaluate every model change, prompt update, and RAG improvement. Budget 2-3 engineering weeks to build the initial eval set and 2-4 hours/week to maintain it.

Q: How do you measure agent task completion rate?

A: AI agents (LLMs that use tools and take multi-step actions) require metrics beyond traditional LLM evaluation because they operate in a loop with real-world side effects.

Agent metric taxonomy:

MetricDefinitionTargetMeasurement
Task Completion Rate% of tasks fully completed correctly>85%Human validation of final state
Partial Success Rate% of tasks partially completedTrack as diagnosticRubric scoring (0-1 per sub-task)
Tool Use Accuracy% of tool calls with correct parameters>95%Validate tool inputs against schema
Steps to CompletionAverage number of actions takenMinimize (close to optimal path)Log action count per task
Error Recovery Rate% of errors the agent recovers from>70%Track retry success after tool failure
Cost per TaskTotal token cost for all agent stepsBelow business value of taskSum token costs across all LLM calls
Wall-Clock TimeTotal time from task start to completionFaster than human alternativeTimestamp delta

Failure taxonomy for agents:

Agent Failure Types (in order of frequency):

1. Planning Failure (30% of failures)
   - Agent takes wrong approach to solve the task
   - Example: Searches wrong database, uses wrong API
   - Detection: Trajectory analysis against optimal path

2. Tool Use Failure (25%)
   - Correct tool, wrong parameters
   - Example: SQL query with wrong column name
   - Detection: Tool execution error rate

3. Hallucination in Reasoning (20%)
   - Agent reasons incorrectly about intermediate results
   - Example: Misinterprets API response, wrong arithmetic
   - Detection: Intermediate state validation

4. Infinite Loop / Stuck (15%)
   - Agent repeats the same action or cannot progress
   - Detection: Action repetition count, timeout

5. Context Window Overflow (10%)
   - Multi-step tasks exceed context, agent loses early information
   - Detection: Token count monitoring, summarization degradation

Benchmarking framework: Use task suites like SWE-bench (software engineering tasks), WebArena (web browsing tasks), or custom domain-specific task sets. Run weekly to detect regression. A 2% drop in task completion rate after a model or prompt change should block deployment.

Semantic caching implementation:

import hashlib
import numpy as np
from redis import Redis

class SemanticCache:
    """Cache LLM responses for semantically similar queries."""

    def __init__(self, embedding_fn, redis_client: Redis, similarity_threshold=0.95):
        self.embed = embedding_fn
        self.redis = redis_client
        self.threshold = similarity_threshold

    def get(self, query: str):
        """Check if a semantically similar query has been cached."""
        query_embedding = self.embed(query)

        # Search for similar embeddings in cache
        # In production, use a vector DB (Pinecone, pgvector) for this
        cached_keys = self.redis.keys("cache:*")
        for key in cached_keys:
            cached = self.redis.hgetall(key)
            cached_embedding = np.frombuffer(cached[b'embedding'], dtype=np.float32)
            similarity = np.dot(query_embedding, cached_embedding) / (
                np.linalg.norm(query_embedding) * np.linalg.norm(cached_embedding)
            )
            if similarity >= self.threshold:
                return cached[b'response'].decode()
        return None

    def set(self, query: str, response: str, ttl_seconds=3600):
        """Cache a query-response pair."""
        embedding = self.embed(query)
        key = f"cache:{hashlib.md5(query.encode()).hexdigest()}"
        self.redis.hset(key, mapping={
            'query': query,
            'response': response,
            'embedding': embedding.tobytes(),
        })
        self.redis.expire(key, ttl_seconds)

Real-world caching results: A customer support chatbot with 100K daily queries found that 45% of queries were semantically similar to previous queries (e.g., "how do I reset my password" vs "reset password help"). Implementing semantic caching with a 0.95 similarity threshold reduced LLM API calls by 42% with zero quality impact.

Q: How do you detect model degradation in production?

A: Model degradation in production is insidious because there is often no error -- the model still returns plausible outputs, just worse ones. Detection requires proactive monitoring, not reactive alerting.

Implicit quality signals (no user action required):

Positive signals:
  - User copies the response (clipboard event)
  - Session ends after response (task completed)
  - No follow-up clarification question
  - User returns to the product next day (retention)

Negative signals:
  - User immediately asks a follow-up (response insufficient)
  - User rephrases the same question (response was off-topic)
  - User abandons session within 10 seconds of response
  - User clicks "contact human support" after AI response

Composite Quality Score:
  quality = w1 * (1 - followup_rate) + w2 * copy_rate + w3 * (1 - abandon_rate)
  Typical weights: w1=0.4, w2=0.3, w3=0.3

Companies with automated eval suites detected this within hours and switched to Claude or reverted to a pinned model version. Companies without monitoring discovered it weeks later through customer complaints. The eval suite investment paid for itself in a single incident.

Q: What are A/B testing challenges specific to AI features?

A: A/B testing AI features has unique challenges that make standard testing frameworks insufficient.

Challenge 1: High variance in outputs

LLMs produce different outputs for the same input (non-deterministic). A user in group A might get a great response, the same user 5 minutes later might get a mediocre one. This increases the sample size needed for statistical significance.

Standard A/B test (deterministic feature):
  Minimum detectable effect: 2%
  Required sample: ~10,000 per arm

LLM A/B test (high output variance):
  Minimum detectable effect: 2%
  Required sample: ~50,000-100,000 per arm (5-10x more)
  Reason: Within-group variance is much higher

Challenge 2: Delayed and indirect metrics

AI features often affect metrics indirectly. An AI-powered search improvement might increase conversion, but the causal chain is: better search results -> user finds product -> user purchases. Each step adds noise and delay.

Challenge 3: User adaptation

Users learn to use AI features differently over time. A chatbot that performs well in week 1 (users ask simple questions) may appear to degrade in week 4 (users ask harder questions because they trust it). This is actually a sign of success, not failure.

Challenge 4: Cost asymmetry

The "better" variant might cost 10x more to run. You need to evaluate cost-adjusted quality, not just quality alone.

Recommended approach:

  • Use interleaving (show results from both models side-by-side) for search/ranking features to reduce variance
  • Use longer experiment windows (3-4 weeks minimum) to capture user adaptation
  • Track composite metrics: quality * (1 / cost) to evaluate cost-efficiency, not just quality
  • Use Bayesian analysis instead of frequentist p-values for faster and more interpretable results with high-variance outcomes

Staff+ Scenarios

Q: Your RAG system has 85% answer accuracy. Is that good enough? How do you decide?

A: "Good enough" depends entirely on the domain, the cost of wrong answers, and the alternative the user has. This is a business decision backed by data, not a technical threshold.

Framework for evaluation:

Step 1: What is the baseline?
  - No AI (human-only): 95% accuracy, 10 min avg resolution time
  - Previous system (keyword search): 60% accuracy, self-service
  - No system (user gives up): 0% accuracy

  85% AI accuracy is a massive improvement over 60% keyword search
  and only marginally worse than 95% human accuracy.

Step 2: What is the cost of the 15% wrong answers?
  Low cost domain (e.g., internal IT help desk):
    Wrong answer = user tries something that does not work, asks again
    Impact: ~5 minutes wasted per wrong answer
    Cost: 15% * 1000 queries/day * 5 min * $0.50/min = $375/day

  High cost domain (e.g., medical advice):
    Wrong answer = potentially dangerous action
    Impact: Liability, patient harm
    Cost: Potentially catastrophic; 85% is NOT acceptable

Step 3: What is the value of the 85% right answers?
  IT help desk: 85% * 1000 queries * $15 (avg ticket cost saved) = $12,750/day
  Net value: $12,750 - $375 = $12,375/day

Step 4: What would it cost to get to 95%?
  Improving from 85% to 95% typically requires:
  - Better retrieval (reranking, hybrid search): +5% accuracy, 2 weeks eng
  - More context (increase K, better chunking): +3% accuracy, 1 week eng
  - Human-in-the-loop for low-confidence: +5% accuracy, ongoing cost
  - Fine-tuning: +2% accuracy, 4 weeks eng

  Estimated investment: 4-8 weeks + ongoing HITL cost
  Expected accuracy: ~93-95%

Decision: For the IT help desk, 85% is already generating $12,375/day in value. Invest in improving to 93%+ over the next quarter, but ship at 85% now. For medical advice, do not ship until accuracy exceeds 98% with human verification on every response.

Staff-level framing: Present this as an ROI calculation, not a quality metric. "Our RAG system saves $12,000/day at 85% accuracy. Each 5% improvement is worth an additional $1,500/day. I recommend shipping now and investing 4 weeks to reach 93%." This is far more compelling than "we need to get to 95% before launching."

Q: How do you measure the ROI of an AI feature?

A: AI feature ROI must account for both direct value (revenue, cost savings) and indirect value (user experience, competitive positioning), minus the full cost including infrastructure, engineering, and ongoing maintenance.

ROI formula:

ROI = (Value Generated - Total Cost) / Total Cost * 100%

Value Generated:
  Direct savings = (manual_cost_per_task - AI_cost_per_task) * tasks_automated
  Revenue uplift = conversion_increase * traffic * average_order_value
  Productivity gain = time_saved_per_user * users * hourly_cost

Total Cost:
  Development = engineers * months * loaded_cost
  Infrastructure = compute + API + storage (monthly * 12)
  Maintenance = ongoing_eng_cost + monitoring + retraining
  Risk = probability_of_failure * cost_of_failure

Example: AI-powered customer support chatbot

Current state:
  Support volume: 50,000 tickets/month
  Cost per human-handled ticket: $15 (20 min * $45/hr loaded cost)
  Monthly support cost: $750,000

AI chatbot (handles 60% of tickets automatically):
  Automated: 30,000 tickets/month at $0.10/ticket (LLM API cost)
  Human-handled: 20,000 tickets at $15/ticket

  Monthly support cost: (30,000 * $0.10) + (20,000 * $15) = $303,000
  Monthly savings: $447,000

AI chatbot costs:
  Development: 3 engineers * 3 months * $25K/mo = $225,000 (one-time)
  Infrastructure: $3,000/month (LLM API + vector DB + hosting)
  Maintenance: 0.5 engineer = $12,500/month

  Year 1 total cost: $225,000 + (12 * $15,500) = $411,000

Year 1 ROI:
  Value: $447,000/mo * 12 = $5,364,000
  Cost: $411,000
  ROI: ($5,364,000 - $411,000) / $411,000 = 1,204%

Payback period: $225,000 / $447,000 = 0.5 months (after launch)

Common ROI traps:

  • Overestimating automation rate: Most AI features automate 40-60% of cases, not 90%+. The remaining cases are the hard ones that still need humans.
  • Ignoring quality costs: If AI accuracy is 85%, the 15% of wrong answers generate rework, customer dissatisfaction, and support escalation. Factor this in.
  • Forgetting maintenance: AI features require ongoing prompt tuning, eval maintenance, model updates, and data pipeline monitoring. Budget 0.25-0.5 FTE per major AI feature.
  • Not measuring displacement: If the AI chatbot handles easy tickets, the remaining human tickets are harder and take longer. The cost per human ticket may increase from $15 to $25.
Q: How do you measure data quality for ML pipelines?

A: Data quality is the single biggest determinant of ML model quality. "Garbage in, garbage out" applies with a vengeance. Staff engineers must build data quality monitoring into every ML pipeline.

Data quality checks implementation:

from dataclasses import dataclass
from typing import Any, Callable
import pandas as pd

@dataclass
class DataQualityCheck:
    name: str
    check_fn: Callable[[pd.DataFrame], bool]
    severity: str  # "critical", "warning", "info"

def build_quality_suite(df: pd.DataFrame) -> list[dict]:
    """Run data quality checks on a DataFrame."""
    checks = [
        DataQualityCheck(
            "null_rate_user_id",
            lambda d: d['user_id'].isnull().mean() < 0.001,
            "critical"
        ),
        DataQualityCheck(
            "null_rate_features",
            lambda d: all(d[col].isnull().mean() < 0.05 for col in d.columns),
            "warning"
        ),
        DataQualityCheck(
            "no_future_timestamps",
            lambda d: (d['event_time'] <= pd.Timestamp.now()).all(),
            "critical"
        ),
        DataQualityCheck(
            "value_range_age",
            lambda d: d['age'].between(0, 150).all() if 'age' in d.columns else True,
            "warning"
        ),
        DataQualityCheck(
            "row_count_reasonable",
            lambda d: len(d) > 1000,  # expect at least 1K rows per batch
            "critical"
        ),
        DataQualityCheck(
            "no_duplicate_ids",
            lambda d: d['user_id'].duplicated().mean() < 0.01,
            "warning"
        ),
    ]

    results = []
    for check in checks:
        try:
            passed = check.check_fn(df)
        except Exception as e:
            passed = False

        results.append({
            "check": check.name,
            "passed": passed,
            "severity": check.severity,
        })

    return results

The "silent failure" problem: Unlike software bugs that crash, data quality issues silently degrade model performance. A feature pipeline that starts returning stale data (e.g., yesterday's prices instead of today's) will not throw an error -- the model will just make worse predictions. Only proactive monitoring catches these issues.

Real-world example: A recommendation system at a major e-commerce company experienced a 15% drop in click-through rate over 2 weeks. Root cause: a upstream data pipeline change started mapping product category IDs incorrectly, causing 30% of products to be in the wrong category. The model was serving recommendations based on wrong categories. A simple data quality check (distribution of category IDs should be stable week-over-week) would have caught this in hours, not weeks.

Q: What is the "last mile" problem in ML deployment and how do you measure it?

A: The "last mile" problem refers to the gap between model performance in offline evaluation (test set, notebooks) and actual production performance. Studies consistently show a 5-20% drop in key metrics when moving from offline to online.

Sources of the last-mile gap:

SourceTypical ImpactDetectionMitigation
Train-serve skew5-10% accuracy dropCompare feature distributions in training vs servingUse a feature store that serves identical features for training and inference
Data freshness gap2-5% accuracy dropMonitor feature staleness in real-timeOnline feature computation, streaming pipelines
Latency constraintsAffects model choice (simpler = faster)Compare offline model vs deployed modelModel distillation, feature precomputation
Edge cases in production3-8% accuracy drop on long tailStratified performance monitoringEnsemble models, rule-based fallbacks for edge cases
Feedback loopsGradual degradation over timeMonitor prediction distribution over timeExploration/exploitation, randomized audits

Measuring the gap:

Offline metrics (test set):
  AUC-ROC: 0.92
  Precision@10%: 0.85
  Recall@10%: 0.78

Online metrics (first week in production):
  AUC-ROC: 0.87 (-5.4%)     <-- Acceptable (<10% drop)
  Precision@10%: 0.72 (-15%) <-- Concerning (investigate)
  Recall@10%: 0.71 (-9%)     <-- Borderline

Investigation reveals:
  - 60% of the precision drop is from train-serve skew on 2 features
  - Feature "user_session_duration" computed differently in serving vs training
  - Fix: Align feature computation; precision recovers to 0.81

Staff-level practice: Always plan for a 2-week "bake-in" period after deploying a new model. Monitor online metrics against offline baselines with pre-defined thresholds. If online performance drops more than 10% below offline metrics, trigger an investigation. The feature store is the most important infrastructure investment for reducing train-serve skew.
