Module 8: Code Quality & Refactoring

What You'll Learn

  • Identify and eliminate Java-specific code smells that signal deeper design issues
  • Apply a catalog of proven refactoring techniques with before/after Java 17+ code
  • Configure and interpret static analysis tools (SpotBugs, SonarQube) on real codebases
  • Diagnose and prevent common memory leak patterns in Java applications
  • Leverage immutability (records, unmodifiable collections) as a quality and concurrency lever
  • Articulate refactoring decisions at the staff-engineer level during interviews

Why Code Quality Matters at Staff Level

The Staff Engineer's Responsibility ▼

At the senior level, you write clean code. At the staff level, you set the standard for what clean code means across your organization. The difference is not just personal output—it is multiplicative impact. A staff engineer who introduces a consistent refactoring culture can reduce defect rates across dozens of engineers.

Why this matters in interviews: Staff-level LLD rounds increasingly test whether you can identify structural problems in existing code and propose systematic improvements, not just produce a correct solution from scratch. Interviewers look for candidates who can reason about code evolution, maintainability costs, and organizational impact of technical debt.

flowchart TD A[Code Smell Detected] --> B{Severity?} B -->|Low| C[Add TODO + Track] B -->|Medium| D[Refactor in Current Sprint] B -->|High / Blocking| E[Refactor Immediately] C --> F[Review in Quarterly Cleanup] D --> G[Pair Review + Tests] E --> G G --> H[Measure Improvement] H --> I[Update Team Standards]

Real-world example: At Google, the "Readability" process mandates that every change in a language must be reviewed by someone with readability approval in that language. This is not bureaucracy—it is a code-quality multiplier. Staff engineers are often the ones who define and evolve readability guidelines for their teams. At Microsoft, code quality gates in the pull-request pipeline block merges when static analysis thresholds regress, and staff engineers own the configuration and evolution of those gates.

Interview Tip: When asked about refactoring in an interview, always frame it in terms of business impact: "I refactored the payment pipeline because the cyclomatic complexity was causing a 30% defect escape rate on that module, which translated to ~2 customer-impacting incidents per quarter."

Java-Specific Code Smells

Primitive Obsession ▼

What it is: Using primitive types (String, int, double) to represent domain concepts. This scatters validation logic, makes method signatures ambiguous, and allows invalid states to exist.

Why it matters: Primitive obsession is the root cause of a surprising number of production bugs. When an email address is just a String, nothing prevents you from passing a phone number where an email is expected. The compiler cannot help you, and neither can your IDE.

Before (Smell)

public class UserService {
    // Which string is which? Easy to swap arguments by accident.
    public User createUser(String email, String phone, String zipCode,
                           int age, double accountBalance) {
        // Validation scattered everywhere this method is called
        if (!email.contains("@")) throw new IllegalArgumentException("Bad email");
        if (age < 0 || age > 150) throw new IllegalArgumentException("Bad age");
        if (accountBalance < 0) throw new IllegalArgumentException("Negative balance");
        // ...
        return new User(email, phone, zipCode, age, accountBalance);
    }
}

After (Refactored with Java 17+ Records)

// Value objects enforce validity at construction time
public record EmailAddress(String value) {
    public EmailAddress {
        Objects.requireNonNull(value, "Email must not be null");
        if (!value.matches("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+$")) {
            throw new IllegalArgumentException("Invalid email: " + value);
        }
        value = value.toLowerCase().strip();
    }
}

public record PhoneNumber(String value) {
    public PhoneNumber {
        Objects.requireNonNull(value);
        value = value.replaceAll("[^0-9+]", "");
        if (value.length() < 10 || value.length() > 15) {
            throw new IllegalArgumentException("Invalid phone: " + value);
        }
    }
}

public record Age(int value) {
    public Age {
        if (value < 0 || value > 150) {
            throw new IllegalArgumentException("Age out of range: " + value);
        }
    }
}

public record Money(BigDecimal amount, Currency currency) {
    public Money {
        Objects.requireNonNull(amount);
        Objects.requireNonNull(currency);
        if (amount.signum() < 0) {
            throw new IllegalArgumentException("Negative amount: " + amount);
        }
    }

    public static Money usd(String val) {
        return new Money(new BigDecimal(val), Currency.getInstance("USD"));
    }
}

// Now the service is type-safe and self-documenting
public class UserService {
    public User createUser(EmailAddress email, PhoneNumber phone,
                           ZipCode zipCode, Age age, Money balance) {
        // No validation needed here - value objects guarantee correctness
        return new User(email, phone, zipCode, age, balance);
    }
}
When NOT to refactor: If a primitive is used in exactly one place, wrapping it adds ceremony without benefit. Value objects pay off when the same concept appears across multiple classes or layers. A good rule of thumb: if you see the same validation logic for a primitive in two or more places, introduce a value object.
Feature Envy ▼

What it is: A method that uses more data from another class than from its own. This signals that the logic belongs in the other class.

Before (Smell)

public class InvoicePrinter {
    // This method "envies" the Invoice class - it reaches into Invoice's
    // internals repeatedly. The calculation logic belongs on Invoice itself.
    public String formatTotal(Invoice invoice) {
        double subtotal = 0;
        for (LineItem item : invoice.getItems()) {
            subtotal += item.getPrice() * item.getQuantity();
        }
        double tax = subtotal * invoice.getTaxRate();
        double discount = subtotal * invoice.getDiscountPercent();
        double total = subtotal + tax - discount;
        return String.format("$%.2f (tax: $%.2f, discount: $%.2f)", total, tax, discount);
    }
}

After (Refactored)

public class Invoice {
    private final List<LineItem> items;
    private final double taxRate;
    private final double discountPercent;

    public Money subtotal() {
        return items.stream()
            .map(LineItem::totalPrice)
            .reduce(Money.ZERO, Money::add);
    }

    public Money tax() {
        return subtotal().multiply(taxRate);
    }

    public Money discount() {
        return subtotal().multiply(discountPercent);
    }

    public Money total() {
        return subtotal().add(tax()).subtract(discount());
    }

    // Formatting can now use the domain methods
    public String formattedTotal() {
        return String.format("$%s (tax: $%s, discount: $%s)",
            total(), tax(), discount());
    }
}

// InvoicePrinter is now thin - it delegates to Invoice's own logic
public class InvoicePrinter {
    public String formatTotal(Invoice invoice) {
        return invoice.formattedTotal();
    }
}
Interview Tip: Feature envy is one of the most common smells interviewers plant in LLD code snippets. When you spot a method reaching into another object's data repeatedly, name the smell explicitly and move the logic.
God Class ▼

What it is: A class that knows too much or does too much. It violates the Single Responsibility Principle by accumulating unrelated responsibilities over time. In Java, any class over ~300 lines or with more than 7-8 public methods should trigger scrutiny.

Before (Smell)

// A "god class" that handles orders, payments, inventory, and notifications
public class OrderManager {
    // Order management
    public Order createOrder(Customer c, List<Item> items) { /* ... */ }
    public void cancelOrder(long orderId) { /* ... */ }
    public void updateOrderStatus(long orderId, Status s) { /* ... */ }

    // Payment processing - different responsibility!
    public PaymentResult chargeCard(CreditCard card, Money amount) { /* ... */ }
    public void processRefund(long orderId) { /* ... */ }

    // Inventory - yet another responsibility!
    public boolean checkStock(long itemId) { /* ... */ }
    public void reserveInventory(long orderId) { /* ... */ }
    public void releaseInventory(long orderId) { /* ... */ }

    // Notifications - and another!
    public void sendOrderConfirmation(long orderId) { /* ... */ }
    public void sendShippingUpdate(long orderId) { /* ... */ }

    // 500+ more lines...
}

After (Refactored - Single Responsibility)

// Each class owns one responsibility
public class OrderService {
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    private final NotificationService notificationService;

    public Order createOrder(Customer customer, List<Item> items) {
        inventoryService.reserve(items);
        Order order = Order.create(customer, items);
        paymentService.authorize(order);
        notificationService.sendConfirmation(order);
        return order;
    }

    public void cancelOrder(long orderId) {
        Order order = findOrder(orderId);
        paymentService.refund(order);
        inventoryService.release(order.getItems());
        order.cancel();
        notificationService.sendCancellation(order);
    }
}

public class PaymentService {
    public PaymentResult authorize(Order order) { /* ... */ }
    public void refund(Order order) { /* ... */ }
}

public class InventoryService {
    public void reserve(List<Item> items) { /* ... */ }
    public void release(List<Item> items) { /* ... */ }
    public boolean checkAvailability(long itemId) { /* ... */ }
}

public class NotificationService {
    public void sendConfirmation(Order order) { /* ... */ }
    public void sendShippingUpdate(Order order) { /* ... */ }
    public void sendCancellation(Order order) { /* ... */ }
}
When NOT to split: Avoid splitting a class just to reduce line count if the methods are genuinely cohesive. A MathUtils class with 20 static math functions is cohesive, not a god class. The test is: "If I change one method, do I likely need to change unrelated methods too?" If yes, split.
Long Parameter List ▼

What it is: Methods with more than 3-4 parameters become hard to read, easy to misuse (swapping arguments), and painful to extend. This often signals a missing abstraction.

Before (Smell)

public class ReportGenerator {
    public Report generate(String title, String author, LocalDate startDate,
                           LocalDate endDate, ReportFormat format,
                           boolean includeCharts, boolean includeSummary,
                           int maxPages, String watermarkText,
                           List<String> recipientEmails) {
        // Which boolean is which? Easy to confuse.
        // Adding a new option requires changing every call site.
    }
}

After (Refactored with Parameter Object + Builder)

public record ReportConfig(
    String title,
    String author,
    LocalDate startDate,
    LocalDate endDate,
    ReportFormat format,
    ReportOptions options,
    List<String> recipientEmails
) {
    // Compact constructor for validation
    public ReportConfig {
        Objects.requireNonNull(title, "Title required");
        Objects.requireNonNull(startDate, "Start date required");
        Objects.requireNonNull(endDate, "End date required");
        if (endDate.isBefore(startDate)) {
            throw new IllegalArgumentException("End date before start date");
        }
        recipientEmails = List.copyOf(recipientEmails); // defensive copy
    }

    // Builder for convenience
    public static Builder builder(String title) {
        return new Builder(title);
    }

    public static class Builder {
        private String title;
        private String author = "System";
        private LocalDate startDate = LocalDate.now().minusMonths(1);
        private LocalDate endDate = LocalDate.now();
        private ReportFormat format = ReportFormat.PDF;
        private ReportOptions options = ReportOptions.defaults();
        private List<String> recipientEmails = List.of();

        Builder(String title) { this.title = title; }

        public Builder author(String a) { this.author = a; return this; }
        public Builder dateRange(LocalDate s, LocalDate e) {
            this.startDate = s; this.endDate = e; return this;
        }
        public Builder format(ReportFormat f) { this.format = f; return this; }
        public Builder options(ReportOptions o) { this.options = o; return this; }
        public Builder recipients(List<String> r) { this.recipientEmails = r; return this; }

        public ReportConfig build() {
            return new ReportConfig(title, author, startDate, endDate,
                                    format, options, recipientEmails);
        }
    }
}

public record ReportOptions(
    boolean includeCharts,
    boolean includeSummary,
    int maxPages,
    String watermarkText
) {
    public static ReportOptions defaults() {
        return new ReportOptions(true, true, 50, null);
    }
}

// Clean usage
public class ReportGenerator {
    public Report generate(ReportConfig config) {
        // Single parameter, self-documenting, easy to extend
    }
}

// Call site is readable
Report report = generator.generate(
    ReportConfig.builder("Q4 Revenue")
        .author("Finance Team")
        .dateRange(LocalDate.of(2025, 10, 1), LocalDate.of(2025, 12, 31))
        .format(ReportFormat.PDF)
        .recipients(List.of("cfo@company.com"))
        .build()
);

Refactoring Catalog

Extract Method ▼

Why: Long methods are hard to test, hard to name, and hard to reuse. Extracting a coherent fragment into its own method improves readability and enables independent testing. Martin Fowler calls this the single most important refactoring.

Before

public class OrderProcessor {
    public OrderResult processOrder(Order order) {
        // Validate order - 15 lines of validation
        if (order.getItems().isEmpty()) {
            throw new IllegalArgumentException("Empty order");
        }
        for (var item : order.getItems()) {
            if (item.getQuantity() <= 0) {
                throw new IllegalArgumentException("Invalid qty: " + item);
            }
            if (!inventoryService.isAvailable(item.getProductId(), item.getQuantity())) {
                throw new InsufficientStockException(item.getProductId());
            }
        }

        // Calculate totals - 10 lines of arithmetic
        BigDecimal subtotal = BigDecimal.ZERO;
        for (var item : order.getItems()) {
            subtotal = subtotal.add(
                item.getUnitPrice().multiply(BigDecimal.valueOf(item.getQuantity()))
            );
        }
        BigDecimal tax = subtotal.multiply(TAX_RATE);
        BigDecimal shipping = calculateShipping(order);
        BigDecimal total = subtotal.add(tax).add(shipping);

        // Process payment - 8 lines
        PaymentResult paymentResult = paymentGateway.charge(
            order.getPaymentMethod(), total
        );
        if (!paymentResult.isSuccess()) {
            throw new PaymentFailedException(paymentResult.getError());
        }

        // ... more steps
        return new OrderResult(order, total, paymentResult);
    }
}

After

public class OrderProcessor {
    public OrderResult processOrder(Order order) {
        validateOrder(order);
        OrderTotals totals = calculateTotals(order);
        PaymentResult payment = processPayment(order, totals.total());
        reserveInventory(order);
        return new OrderResult(order, totals, payment);
    }

    private void validateOrder(Order order) {
        if (order.getItems().isEmpty()) {
            throw new IllegalArgumentException("Empty order");
        }
        order.getItems().forEach(this::validateLineItem);
    }

    private void validateLineItem(LineItem item) {
        if (item.getQuantity() <= 0) {
            throw new IllegalArgumentException("Invalid qty: " + item);
        }
        if (!inventoryService.isAvailable(item.getProductId(), item.getQuantity())) {
            throw new InsufficientStockException(item.getProductId());
        }
    }

    private OrderTotals calculateTotals(Order order) {
        BigDecimal subtotal = order.getItems().stream()
            .map(item -> item.getUnitPrice()
                .multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);

        BigDecimal tax = subtotal.multiply(TAX_RATE);
        BigDecimal shipping = calculateShipping(order);
        return new OrderTotals(subtotal, tax, shipping);
    }

    private PaymentResult processPayment(Order order, BigDecimal total) {
        PaymentResult result = paymentGateway.charge(order.getPaymentMethod(), total);
        if (!result.isSuccess()) {
            throw new PaymentFailedException(result.getError());
        }
        return result;
    }
}
When NOT to use: Do not extract a method that is called exactly once and is only 2-3 lines. The indirection can harm readability more than the inlining. Also avoid extracting when it would require passing 5+ parameters to the new method—that signals you may need a different refactoring (Introduce Parameter Object or Extract Class) first.
Replace Conditional with Polymorphism ▼

Why: Switch statements or long if-else chains on type codes are a magnet for bugs: every new type requires modifying every switch. Polymorphism distributes the logic to the types themselves, making the system open for extension and closed for modification (OCP).

Before

public class NotificationSender {
    public void send(Notification notification) {
        switch (notification.getType()) {
            case "EMAIL":
                String body = renderEmailTemplate(notification);
                emailClient.send(notification.getRecipient(), body);
                break;
            case "SMS":
                String text = truncate(notification.getMessage(), 160);
                smsGateway.send(notification.getPhone(), text);
                break;
            case "PUSH":
                pushService.send(notification.getDeviceToken(),
                    notification.getTitle(), notification.getMessage());
                break;
            case "SLACK":
                slackClient.postMessage(notification.getChannel(),
                    formatSlackMessage(notification));
                break;
            // Every new channel = modify this class
        }
    }
}

After

// Sealed interface (Java 17+) constrains implementations
public sealed interface NotificationChannel
    permits EmailChannel, SmsChannel, PushChannel, SlackChannel {

    void send(Notification notification);
    boolean supports(Notification notification);
}

public final class EmailChannel implements NotificationChannel {
    private final EmailClient emailClient;
    private final TemplateEngine templateEngine;

    @Override
    public void send(Notification notification) {
        String body = templateEngine.render(notification);
        emailClient.send(notification.getRecipient(), body);
    }

    @Override
    public boolean supports(Notification notification) {
        return notification.getRecipient() != null
            && notification.getRecipient().contains("@");
    }
}

public final class SmsChannel implements NotificationChannel {
    private final SmsGateway smsGateway;

    @Override
    public void send(Notification notification) {
        String text = notification.getMessage().length() > 160
            ? notification.getMessage().substring(0, 157) + "..."
            : notification.getMessage();
        smsGateway.send(notification.getPhone(), text);
    }

    @Override
    public boolean supports(Notification notification) {
        return notification.getPhone() != null;
    }
}

// Dispatcher is now open/closed - no switch needed
public class NotificationSender {
    private final List<NotificationChannel> channels;

    public NotificationSender(List<NotificationChannel> channels) {
        this.channels = channels;
    }

    public void send(Notification notification) {
        channels.stream()
            .filter(ch -> ch.supports(notification))
            .findFirst()
            .orElseThrow(() -> new UnsupportedNotificationException(notification))
            .send(notification);
    }
}
When NOT to use: If you have a simple 2-3 case switch that is unlikely to grow, polymorphism adds unnecessary class proliferation. Also, Java 17+ pattern matching with switch expressions and sealed types can be a cleaner alternative for type-based dispatch that the compiler can exhaustiveness-check.
Introduce Parameter Object ▼

Why: Groups of parameters that travel together indicate a missing concept. A parameter object names the concept, reduces parameter count, and provides a natural home for validation and derived behavior.

Before

public class MetricsService {
    // These four parameters always travel together
    public List<DataPoint> query(LocalDateTime start, LocalDateTime end,
                                  String metricName, Duration granularity) { ... }

    public double average(LocalDateTime start, LocalDateTime end,
                          String metricName, Duration granularity) { ... }

    public Map<String, Double> percentiles(LocalDateTime start, LocalDateTime end,
                                            String metricName, Duration granularity,
                                            List<Double> percentiles) { ... }
}

After

public record TimeRange(LocalDateTime start, LocalDateTime end) {
    public TimeRange {
        Objects.requireNonNull(start);
        Objects.requireNonNull(end);
        if (end.isBefore(start)) {
            throw new IllegalArgumentException("end before start");
        }
    }

    public Duration duration() { return Duration.between(start, end); }

    public boolean contains(LocalDateTime point) {
        return !point.isBefore(start) && !point.isAfter(end);
    }

    public static TimeRange lastHours(int hours) {
        LocalDateTime now = LocalDateTime.now();
        return new TimeRange(now.minusHours(hours), now);
    }
}

public record MetricQuery(TimeRange range, String metricName, Duration granularity) {
    public MetricQuery {
        Objects.requireNonNull(range);
        Objects.requireNonNull(metricName);
        if (granularity != null && granularity.compareTo(range.duration()) > 0) {
            throw new IllegalArgumentException("Granularity exceeds range");
        }
    }
}

public class MetricsService {
    public List<DataPoint> query(MetricQuery query) { ... }
    public double average(MetricQuery query) { ... }
    public Map<String, Double> percentiles(MetricQuery query,
                                            List<Double> pcts) { ... }
}
Replace Magic Numbers with Constants ▼

Why: Numeric literals scattered through code hide intent and create maintenance landmines. When the same value appears in 12 places and needs to change, you are guaranteed to miss at least one.

Before

public class RateLimiter {
    public boolean allowRequest(String clientId) {
        long count = redis.increment(clientId);
        if (count == 1) {
            redis.expire(clientId, 60);  // 60 what? seconds? minutes?
        }
        return count <= 100;  // why 100?
    }

    public void cleanup() {
        // Remove entries older than 3600 seconds
        redis.removeOlderThan(3600);  // what is 3600?
    }
}

After

public class RateLimiter {
    private static final int MAX_REQUESTS_PER_WINDOW = 100;
    private static final Duration RATE_LIMIT_WINDOW = Duration.ofSeconds(60);
    private static final Duration CLEANUP_THRESHOLD = Duration.ofHours(1);

    public boolean allowRequest(String clientId) {
        long count = redis.increment(clientId);
        if (count == 1) {
            redis.expire(clientId, RATE_LIMIT_WINDOW.toSeconds());
        }
        return count <= MAX_REQUESTS_PER_WINDOW;
    }

    public void cleanup() {
        redis.removeOlderThan(CLEANUP_THRESHOLD.toSeconds());
    }
}
Pro tip: Consider externalizing these constants into configuration (environment variables or a config file) so that operators can tune them without a code change. In production systems at scale, rate-limit thresholds frequently change.
Extract Interface ▼

Why: Extracting an interface decouples consumers from concrete implementations, enabling testability (mocking), multiple implementations, and adherence to the Dependency Inversion Principle. This is the single most impactful refactoring for testability.

Before

// Directly coupled to a concrete HTTP client - untestable without network
public class WeatherService {
    private final OkHttpClient httpClient = new OkHttpClient();

    public WeatherData getCurrentWeather(String city) {
        Request request = new Request.Builder()
            .url("https://api.weather.com/v1/current?city=" + city)
            .build();
        Response response = httpClient.newCall(request).execute();
        return parseResponse(response.body().string());
    }
}

After

// Interface defines the contract
public interface WeatherProvider {
    WeatherData getCurrentWeather(String city);
    Optional<Forecast> getForecast(String city, int days);
}

// Production implementation
public class OpenWeatherProvider implements WeatherProvider {
    private final HttpClient httpClient;
    private final String apiKey;

    public OpenWeatherProvider(HttpClient httpClient, String apiKey) {
        this.httpClient = httpClient;
        this.apiKey = apiKey;
    }

    @Override
    public WeatherData getCurrentWeather(String city) {
        // Real HTTP call
        HttpResponse<String> response = httpClient.send(
            HttpRequest.newBuilder()
                .uri(URI.create("https://api.openweathermap.org/data/2.5/weather?q="
                    + city + "&appid=" + apiKey))
                .build(),
            HttpResponse.BodyHandlers.ofString()
        );
        return parseResponse(response.body());
    }

    @Override
    public Optional<Forecast> getForecast(String city, int days) { /* ... */ }
}

// Test implementation - no network needed
public class FakeWeatherProvider implements WeatherProvider {
    private final Map<String, WeatherData> data = new HashMap<>();

    public void setWeather(String city, WeatherData weather) {
        data.put(city, weather);
    }

    @Override
    public WeatherData getCurrentWeather(String city) {
        return data.getOrDefault(city,
            new WeatherData(city, 20.0, "Clear"));
    }

    @Override
    public Optional<Forecast> getForecast(String city, int days) {
        return Optional.empty();
    }
}

// Service depends on abstraction, not concrete class
public class WeatherService {
    private final WeatherProvider provider;  // injected

    public WeatherService(WeatherProvider provider) {
        this.provider = provider;
    }
}
When NOT to use: Do not extract an interface when there is only one implementation and no realistic prospect of a second. "Speculative generality" (creating abstractions you do not need yet) adds navigational complexity. Extract when you have a concrete testing or extensibility need.

Static Analysis

SpotBugs, SonarQube & Common Rules ▼

Why static analysis matters: Humans are inconsistent reviewers. Static analysis tools catch entire categories of bugs—null dereferences, resource leaks, concurrency errors—that even experienced engineers miss in code review. At staff level, you are expected to champion and configure these tools for your team, not just use them passively.

flowchart LR A[Developer Commits] --> B[CI Pipeline] B --> C[SpotBugs] B --> D[SonarQube] B --> E[Checkstyle] C --> F{Quality Gate} D --> F E --> F F -->|Pass| G[Merge Allowed] F -->|Fail| H[Block + Notify]

SpotBugs: What It Catches

SpotBugs (successor to FindBugs) performs bytecode analysis to find real bugs, not just style issues. Key categories:

// BUG: NP_NULL_ON_SOME_PATH - possible null dereference
public String getUserCity(long userId) {
    User user = userRepository.findById(userId);  // may return null
    return user.getAddress().getCity();  // SpotBugs flags this
}

// FIX:
public Optional<String> getUserCity(long userId) {
    return Optional.ofNullable(userRepository.findById(userId))
        .map(User::getAddress)
        .map(Address::getCity);
}

// BUG: RCN_REDUNDANT_NULLCHECK - redundant null check after guaranteed non-null
public void process(String input) {
    Objects.requireNonNull(input);
    if (input != null) {  // SpotBugs flags: always true after requireNonNull
        // ...
    }
}

// BUG: EI_EXPOSE_REP - may expose internal mutable state
public class Config {
    private final Date createdAt;

    public Date getCreatedAt() {
        return createdAt;  // SpotBugs flags: returns mutable Date reference
    }

    // FIX: return defensive copy (or use Instant instead)
    public Date getCreatedAtSafe() {
        return new Date(createdAt.getTime());
    }
}

// BUG: OBL_UNSATISFIED_OBLIGATION - resource not closed
public String readFile(Path path) throws IOException {
    BufferedReader reader = Files.newBufferedReader(path);
    return reader.readLine();  // reader never closed!
}

// FIX: use try-with-resources
public String readFile(Path path) throws IOException {
    try (BufferedReader reader = Files.newBufferedReader(path)) {
        return reader.readLine();
    }
}

Memory Leaks in Java

Common Memory Leak Patterns ▼

Why this matters: "Java has garbage collection, so you cannot have memory leaks" is a myth. You absolutely can—and production incidents from memory leaks are common even at companies like Netflix and LinkedIn. The GC only collects objects that are unreachable. If you accidentally hold a reference, the GC cannot help you.

1. Listeners Not Removed

// LEAK: Registering listener without ever unregistering
public class StockWidget {
    public StockWidget(StockFeed feed) {
        // feed holds a reference to this widget forever
        feed.addListener(this::onPriceUpdate);
    }

    private void onPriceUpdate(StockPrice price) {
        updateDisplay(price);
    }

    // Widget is "closed" but feed still references it!
    public void close() {
        // Oops - forgot to remove listener
    }
}

// FIX: Always pair registration with deregistration
public class StockWidget implements AutoCloseable {
    private final StockFeed feed;
    private final StockFeed.Listener listener;

    public StockWidget(StockFeed feed) {
        this.feed = feed;
        this.listener = this::onPriceUpdate;
        feed.addListener(listener);
    }

    @Override
    public void close() {
        feed.removeListener(listener);  // break the reference
    }
}

2. Non-Static Inner Classes Holding Outer References

// LEAK: Non-static inner class holds implicit reference to outer instance
public class DataProcessor {
    private final byte[] largeBuffer = new byte[10_000_000]; // 10 MB

    // Inner class holds hidden reference to DataProcessor
    // (and therefore to largeBuffer)
    public class ResultIterator implements Iterator<Result> {
        private int index = 0;
        @Override
        public boolean hasNext() { return index < results.size(); }
        @Override
        public Result next() { return results.get(index++); }
    }

    public Iterator<Result> getResults() {
        return new ResultIterator(); // keeps DataProcessor alive!
    }
}

// FIX: Use static inner class (no implicit outer reference)
public class DataProcessor {
    private final byte[] largeBuffer = new byte[10_000_000];

    private static class ResultIterator implements Iterator<Result> {
        private final List<Result> results;
        private int index = 0;

        ResultIterator(List<Result> results) {
            this.results = results; // only holds what it needs
        }

        @Override
        public boolean hasNext() { return index < results.size(); }
        @Override
        public Result next() { return results.get(index++); }
    }

    public Iterator<Result> getResults() {
        return new ResultIterator(List.copyOf(this.results));
    }
}

3. ThreadLocal Not Cleaned Up

// LEAK: ThreadLocal in a thread-pool environment
// In a servlet container or thread pool, threads are reused.
// If you set a ThreadLocal but never remove it, data accumulates.
public class RequestContext {
    private static final ThreadLocal<UserSession> currentSession =
        new ThreadLocal<>();

    public static void setSession(UserSession session) {
        currentSession.set(session);
    }

    public static UserSession getSession() {
        return currentSession.get();
    }

    // CRITICAL: Must call this when the request ends!
    public static void clear() {
        currentSession.remove();
    }
}

// FIX: Use try-finally or a servlet filter to guarantee cleanup
public class RequestFilter implements Filter {
    @Override
    public void doFilter(ServletRequest req, ServletResponse res,
                         FilterChain chain) throws IOException, ServletException {
        try {
            UserSession session = authenticateRequest(req);
            RequestContext.setSession(session);
            chain.doFilter(req, res);
        } finally {
            RequestContext.clear();  // always clean up
        }
    }
}

4. Static Collections Growing Without Bound

// LEAK: Static cache that never evicts entries
public class CacheManager {
    // This map lives as long as the classloader - forever in most apps
    private static final Map<String, Object> cache = new HashMap<>();

    public static void put(String key, Object value) {
        cache.put(key, value);  // entries accumulate indefinitely
    }
}

// FIX 1: Use WeakHashMap for keys that should be GC'd
private static final Map<String, Object> cache =
    Collections.synchronizedMap(new WeakHashMap<>());

// FIX 2: Use a bounded cache with eviction (preferred)
private static final Map<String, Object> cache =
    new LinkedHashMap<>(16, 0.75f, true) {
        @Override
        protected boolean removeEldestEntry(Map.Entry<String, Object> eldest) {
            return size() > MAX_CACHE_SIZE;
        }
    };

// FIX 3: Use Caffeine (production-grade)
private static final Cache<String, Object> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(Duration.ofMinutes(30))
    .recordStats()
    .build();
Diagnosis tools: When you suspect a memory leak in production, use:
  • jmap -histo:live <pid> — shows live object counts by class
  • VisualVM / JFR (Java Flight Recorder) — real-time heap monitoring
  • Eclipse MAT — analyzes heap dumps to find "dominator trees" (who holds the most memory)
  • -XX:+HeapDumpOnOutOfMemoryError — auto-dumps on OOM for post-mortem

Immutability as a Quality Lever

Records, Unmodifiable Collections & Defensive Copies ▼

Why immutability matters: Immutable objects are inherently thread-safe, easier to reason about, safe to cache, and free of aliasing bugs. In Java, mutability is the default, so you must actively design for immutability. At staff level, promoting immutability-by-default is one of the highest-leverage quality improvements you can make.

Java Records (Java 16+)

// Records are immutable by design: final fields, no setters, built-in
// equals/hashCode/toString
public record Money(BigDecimal amount, Currency currency) {
    // Compact constructor for validation + normalization
    public Money {
        Objects.requireNonNull(amount, "amount must not be null");
        Objects.requireNonNull(currency, "currency must not be null");
        amount = amount.setScale(2, RoundingMode.HALF_UP);
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Currency mismatch");
        }
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money multiply(int factor) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(factor)), currency);
    }

    public static final Money ZERO_USD =
        new Money(BigDecimal.ZERO, Currency.getInstance("USD"));
}

Unmodifiable Collections

public class Team {
    private final List<Member> members;

    public Team(List<Member> members) {
        // Defensive copy at construction - caller cannot mutate our list
        this.members = List.copyOf(members);
    }

    // Returns unmodifiable view - caller cannot add/remove
    public List<Member> getMembers() {
        return members;  // already unmodifiable from List.copyOf
    }

    // To "modify", return a new Team (functional style)
    public Team addMember(Member member) {
        var updated = new ArrayList<>(this.members);
        updated.add(member);
        return new Team(updated);
    }

    public Team removeMember(Member member) {
        var updated = this.members.stream()
            .filter(m -> !m.equals(member))
            .toList();
        return new Team(updated);
    }
}

// Java collection factory methods create unmodifiable collections:
var ids = List.of(1, 2, 3);            // unmodifiable
var names = Set.of("Alice", "Bob");     // unmodifiable
var config = Map.of("key", "value");    // unmodifiable

// Collectors.toUnmodifiableList() for streams:
var filtered = items.stream()
    .filter(Item::isActive)
    .collect(Collectors.toUnmodifiableList());

Defensive Copies

// PROBLEM: Mutable field exposed
public class Event {
    private final Date startTime;  // Date is mutable!

    public Event(Date startTime) {
        this.startTime = startTime;  // caller still holds reference
    }

    public Date getStartTime() {
        return startTime;  // caller can mutate!
    }
}

// An attacker or careless caller can do:
Date date = new Date();
Event event = new Event(date);
date.setTime(0);  // mutates the event's internal state!

// FIX 1: Defensive copies
public class Event {
    private final Date startTime;

    public Event(Date startTime) {
        this.startTime = new Date(startTime.getTime()); // copy in
    }

    public Date getStartTime() {
        return new Date(startTime.getTime()); // copy out
    }
}

// FIX 2 (preferred): Use immutable types from java.time
public record Event(Instant startTime, String title) {
    // Instant is immutable - no defensive copies needed!
}

Benefits for Concurrency

// Immutable objects are thread-safe by construction
// No synchronization needed!
public record PriceSnapshot(
    String symbol,
    BigDecimal price,
    Instant timestamp,
    Map<String, BigDecimal> metadata
) {
    public PriceSnapshot {
        Objects.requireNonNull(symbol);
        Objects.requireNonNull(price);
        timestamp = (timestamp != null) ? timestamp : Instant.now();
        metadata = (metadata != null) ? Map.copyOf(metadata) : Map.of();
    }
}

// Can safely share across threads without locks
public class PriceCache {
    // AtomicReference for thread-safe swaps of the entire snapshot
    private final AtomicReference<PriceSnapshot> latest =
        new AtomicReference<>();

    public void update(PriceSnapshot snapshot) {
        latest.set(snapshot);  // atomic, no locks
    }

    public PriceSnapshot getLatest() {
        return latest.get();  // safe - snapshot is immutable
    }
}
When NOT to use immutability:
  • High-frequency updates on large objects: Creating a new object every time can pressure the GC. For hot paths (e.g., game loop state, high-throughput stream buffers), mutable state with explicit synchronization may be necessary.
  • Builder patterns during construction: Builders are inherently mutable—that is fine. The key is that the built object should be immutable.
  • Interop with frameworks: Some frameworks (JPA, Jackson) require mutable beans. Use DTOs at the boundary and convert to immutable domain objects internally.

Question Answer
What is Primitive Obsession and how do Java records fix it? Using raw primitives (String, int) for domain concepts, scattering validation. Records create immutable value objects that enforce invariants at construction, making method signatures type-safe and self-documenting.
Name three common Java memory leak patterns. 1) Listeners registered but never removed, 2) Non-static inner classes holding implicit outer references, 3) ThreadLocal values not cleared in thread-pool environments, 4) Static collections growing without eviction.
When should you NOT extract a method? When the fragment is 2-3 lines called once (indirection hurts readability) or when extraction requires 5+ parameters (signals need for Extract Class or Parameter Object instead).
What does SonarQube rule S3776 measure? Cognitive complexity. Unlike cyclomatic complexity which counts paths, cognitive complexity weights nested control structures more heavily, better reflecting how hard code is for humans to understand. Default threshold is 15.
Why are immutable objects thread-safe? Their state cannot change after construction, so there are no race conditions, no need for synchronization, and they can be safely shared across threads. Use records, List.copyOf(), and java.time types.

Practice Exercise: Refactor a Messy Class

Exercise: Step-by-Step Refactoring ▼

Scenario: You have inherited the following class from a departing engineer. Identify all code smells and refactor step by step.

The Messy Class

public class UserManager {
    private static Map<String, Object> cache = new HashMap<>();
    private Connection dbConn;

    public Map<String, Object> processUser(String email, String fname,
            String lname, int age, String street, String city,
            String zip, String country, double balance, String type) {

        // Validate
        if (email == null || !email.contains("@")) return null;
        if (age < 0 || age > 200) return null;
        if (balance < 0) return null;

        // Calculate discount
        double discount = 0;
        if (type.equals("GOLD")) {
            discount = balance * 0.1;
        } else if (type.equals("PLATINUM")) {
            discount = balance * 0.2;
        } else if (type.equals("DIAMOND")) {
            discount = balance * 0.3;
        }

        // Store in DB (resource leak potential)
        PreparedStatement stmt = dbConn.prepareStatement(
            "INSERT INTO users VALUES (?, ?, ?, ?, ?, ?, ?)");
        stmt.setString(1, email);
        stmt.setString(2, fname + " " + lname);
        stmt.setInt(3, age);
        stmt.setString(4, street + ", " + city + " " + zip + ", " + country);
        stmt.setDouble(5, balance);
        stmt.setDouble(6, discount);
        stmt.setString(7, type);
        stmt.executeUpdate();

        // Cache the user
        cache.put(email, fname + "|" + lname + "|" + age);

        // Build response
        Map<String, Object> result = new HashMap<>();
        result.put("email", email);
        result.put("name", fname + " " + lname);
        result.put("discount", discount);
        result.put("status", "created");
        return result;
    }
}

Step 1: Identify All Smells

  • Long Parameter List — 10 parameters, many of which form logical groups
  • Primitive Obsession — email, address, balance, type all raw primitives
  • God Class — validation, discount logic, DB access, caching, response building
  • Replace Conditional with Polymorphism — discount type switch
  • Resource Leak — PreparedStatement never closed
  • Static mutable collection — unbounded cache
  • Returning null on error — hides failures
  • Magic numbers — 0.1, 0.2, 0.3 discount rates
  • Returning raw Map instead of a typed object

Step 2: Introduce Value Objects

public record EmailAddress(String value) {
    public EmailAddress {
        Objects.requireNonNull(value);
        if (!value.contains("@")) throw new IllegalArgumentException("Invalid email");
        value = value.toLowerCase().strip();
    }
}

public record Address(String street, String city, String zip, String country) {
    public Address {
        Objects.requireNonNull(street);
        Objects.requireNonNull(city);
    }

    public String formatted() {
        return String.join(", ", street, city + " " + zip, country);
    }
}

public record Age(int value) {
    public Age { if (value < 0 || value > 150) throw new IllegalArgumentException(); }
}

public record Money(BigDecimal amount) {
    public Money { if (amount.signum() < 0) throw new IllegalArgumentException(); }
    public Money multiply(BigDecimal factor) { return new Money(amount.multiply(factor)); }
}

Step 3: Replace Conditional with Enum Strategy

public enum MembershipTier {
    STANDARD(BigDecimal.ZERO),
    GOLD(new BigDecimal("0.10")),
    PLATINUM(new BigDecimal("0.20")),
    DIAMOND(new BigDecimal("0.30"));

    private final BigDecimal discountRate;

    MembershipTier(BigDecimal discountRate) {
        this.discountRate = discountRate;
    }

    public Money calculateDiscount(Money balance) {
        return balance.multiply(this.discountRate);
    }
}

Step 4: Extract Responsibilities

// Parameter object
public record CreateUserRequest(
    EmailAddress email, String firstName, String lastName,
    Age age, Address address, Money balance, MembershipTier tier
) {}

// Result type instead of raw Map
public record CreateUserResult(
    EmailAddress email, String fullName,
    Money discount, String status
) {}

// Separated service
public class UserService {
    private final UserRepository repository;
    private final UserCache cache;

    public UserService(UserRepository repository, UserCache cache) {
        this.repository = repository;
        this.cache = cache;
    }

    public CreateUserResult createUser(CreateUserRequest request) {
        Money discount = request.tier().calculateDiscount(request.balance());

        User user = new User(
            request.email(), request.firstName(), request.lastName(),
            request.age(), request.address(), request.balance(),
            discount, request.tier()
        );

        repository.save(user);
        cache.put(request.email(), user);

        return new CreateUserResult(
            request.email(),
            request.firstName() + " " + request.lastName(),
            discount,
            "created"
        );
    }
}
