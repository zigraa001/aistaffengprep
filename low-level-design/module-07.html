Module 7: Concurrency Patterns

Learning Objectives

  • Understand the Java Memory Model and happens-before guarantees
  • Choose the right locking mechanism: synchronized, ReentrantLock, or StampedLock
  • Implement producer-consumer, reader-writer, and async composition patterns
  • Detect, prevent, and reason about deadlocks in interview settings
  • Size thread pools correctly and explain work-stealing to interviewers
  • Use concurrent collections and synchronization utilities at scale

Why Concurrency Matters

Real-World Motivation for Concurrency ▼

Every large-scale system is concurrent. Netflix processes ~2 million API requests per second. Google handles ~99,000 search queries per second. Uber dispatches rides across millions of concurrent users. None of this works with a single thread.

Why you cannot avoid concurrency at staff level:

  • Moore's Law shifted: Clock speeds plateaued around 2005. Performance gains now come from multi-core parallelism, not faster single cores. A modern server has 64-128 cores.
  • Latency hiding: While one thread waits on a 10ms network call, hundreds of other requests could be served. Blocking wastes 99% of available CPU cycles on I/O-heavy workloads.
  • Correctness under load: Race conditions, visibility bugs, and deadlocks are the top three causes of production incidents that only appear at scale. They never show up in unit tests.
  • Interview signal: Concurrency questions separate senior from staff candidates. Interviewers probe whether you can reason about ordering, visibility, and liveness -- not just write code that compiles.

1. synchronized vs ReentrantLock vs StampedLock

Choosing the Right Lock ▼

Java offers three primary locking mechanisms, each designed for different use cases. Choosing the wrong one is a common source of bugs and performance problems.

synchronized -- The Built-In Monitor

Why it exists: Every Java object has an intrinsic lock (monitor). synchronized is the simplest way to achieve mutual exclusion. The JVM handles lock acquisition and release automatically, even in the presence of exceptions.

public class BankAccount {
    private long balance;

    // Method-level synchronization: locks on 'this'
    public synchronized void deposit(long amount) {
        // Only one thread can execute this at a time per instance
        this.balance += amount;
    }

    public synchronized void withdraw(long amount) {
        if (this.balance < amount) {
            throw new IllegalStateException("Insufficient funds");
        }
        this.balance -= amount;
    }

    // Block-level: finer granularity, locks on specific object
    public void transfer(BankAccount target, long amount) {
        // DANGER: naive approach can deadlock (see Deadlock section)
        synchronized (this) {
            synchronized (target) {
                this.withdraw(amount);
                target.deposit(amount);
            }
        }
    }

    public synchronized long getBalance() {
        return this.balance;
    }
}

When to use: Simple mutual exclusion, short critical sections, code where simplicity matters more than flexibility. Modern JVMs optimize synchronized aggressively (biased locking, lock coarsening, lock elision).

ReentrantLock -- Explicit Control

Why it exists: synchronized cannot be interrupted, cannot time out, cannot try without blocking, and does not support fairness policies. ReentrantLock solves all of these.

import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class InventoryService {
    private final ReentrantLock lock = new ReentrantLock(/* fair = */ true);
    private final Map<String, Integer> stock = new HashMap<>();

    /**
     * Reserves stock with a timeout -- essential for high-throughput systems.
     * At Netflix scale, you cannot afford threads blocking indefinitely
     * on a lock held by a slow downstream call.
     */
    public boolean reserveStock(String sku, int quantity, long timeoutMs) {
        boolean acquired = false;
        try {
            // tryLock: non-blocking attempt with timeout
            acquired = lock.tryLock(timeoutMs, TimeUnit.MILLISECONDS);
            if (!acquired) {
                // Log, increment metric, return graceful failure
                return false;
            }

            int available = stock.getOrDefault(sku, 0);
            if (available < quantity) {
                return false;
            }
            stock.put(sku, available - quantity);
            return true;

        } catch (InterruptedException e) {
            Thread.currentThread().interrupt(); // Restore interrupt flag
            return false;
        } finally {
            if (acquired) {
                lock.unlock(); // MUST unlock in finally -- no auto-release!
            }
        }
    }

    /**
     * lockInterruptibly: allows cancellation of waiting threads.
     * Critical for systems with request deadlines.
     */
    public void restockInterruptible(String sku, int quantity)
            throws InterruptedException {
        lock.lockInterruptibly();
        try {
            stock.merge(sku, quantity, Integer::sum);
        } finally {
            lock.unlock();
        }
    }
}

StampedLock -- Optimistic Reads for Read-Heavy Workloads

Why it exists: In read-heavy systems (e.g., a configuration cache read 10,000x/sec, written 1x/min), even ReadWriteLock adds unnecessary overhead because readers still acquire a shared lock. StampedLock offers optimistic reads that are essentially free when there is no contention with writers.

import java.util.concurrent.locks.StampedLock;

public class GeoPoint {
    private double latitude;
    private double longitude;
    private final StampedLock lock = new StampedLock();

    public void move(double newLat, double newLon) {
        long stamp = lock.writeLock();
        try {
            this.latitude = newLat;
            this.longitude = newLon;
        } finally {
            lock.unlockWrite(stamp);
        }
    }

    /**
     * Optimistic read: no lock acquired! Just a stamp validation.
     * If a writer intervened, we fall back to a full read lock.
     * This pattern gives near-zero overhead for reads when writes are rare.
     */
    public double distanceFromOrigin() {
        long stamp = lock.tryOptimisticRead(); // non-blocking, returns stamp
        double currentLat = this.latitude;
        double currentLon = this.longitude;

        if (!lock.validate(stamp)) {
            // A writer intervened -- fall back to pessimistic read
            stamp = lock.readLock();
            try {
                currentLat = this.latitude;
                currentLon = this.longitude;
            } finally {
                lock.unlockRead(stamp);
            }
        }

        return Math.sqrt(currentLat * currentLat + currentLon * currentLon);
    }
}

When NOT to Use:
  • StampedLock: Never use if you need reentrancy or condition variables. Never use in recursive code paths.
  • ReentrantLock: Overkill for simple cases -- the try/finally boilerplate is error-prone. Use synchronized unless you need timeouts, interruptibility, or fairness.
  • synchronized: Avoid when you need lock polling or timed waits. Also avoid for long-running critical sections that could block other threads indefinitely.
Interview Tip: Interviewers love asking "Why not just use synchronized everywhere?" The staff-level answer: "synchronized is fine for most cases thanks to JVM optimizations, but when I need tryLock with timeout for latency SLAs, multiple condition queues for producer-consumer, or optimistic reads for read-heavy caches, I reach for the java.util.concurrent alternatives."

2. volatile, happens-before, and the Java Memory Model

Memory Visibility Guarantees ▼

Why the JMM exists: Modern CPUs have per-core caches (L1, L2) and store buffers. Without explicit memory ordering, Thread A's write to a variable may never become visible to Thread B -- it can sit in Thread A's store buffer or cache indefinitely. The Java Memory Model (JMM, JSR-133) defines the rules for when writes become visible across threads.

Store buffers (red) are where visibility bugs hide. Writes may stay here and never reach main memory without a memory barrier.

The happens-before Relationship

The JMM defines happens-before as a partial ordering. If action A happens-before action B, then A's effects are guaranteed visible to B. Key rules:

  • Monitor rule: An unlock on a monitor happens-before every subsequent lock on that same monitor.
  • Volatile rule: A write to a volatile field happens-before every subsequent read of that same field.
  • Thread start rule: thread.start() happens-before any action in the started thread.
  • Thread join rule: All actions in a thread happen-before join() returns in another thread.
  • Transitivity: If A happens-before B, and B happens-before C, then A happens-before C.

volatile -- Visibility Without Mutual Exclusion

public class FeatureFlagService {
    // volatile ensures all threads see the latest value immediately
    // after a write. Without volatile, Thread B might read a stale
    // cached value of 'enabled' indefinitely.
    private volatile boolean enabled = false;

    // Writer thread (e.g., config reload)
    public void enable() {
        this.enabled = true; // volatile write: flushes store buffer
    }

    // Reader threads (e.g., request handlers) -- thousands concurrent
    public boolean isEnabled() {
        return this.enabled; // volatile read: invalidates cache line
    }
}

// CLASSIC BUG: double-checked locking without volatile
public class Singleton {
    // WITHOUT volatile, another thread could see a partially
    // constructed object (allocated but constructor not finished)
    private static volatile Singleton instance;

    public static Singleton getInstance() {
        if (instance == null) {                  // 1st check (no lock)
            synchronized (Singleton.class) {
                if (instance == null) {          // 2nd check (with lock)
                    instance = new Singleton();  // volatile write
                }
            }
        }
        return instance;                         // volatile read
    }
}

What volatile does NOT do: It does not make compound operations atomic. volatileCounter++ is still a race (read-modify-write). For atomic increments, use AtomicLong or LongAdder.

Interview Tip: If asked about the JMM, start by saying: "The JMM solves two problems -- visibility and ordering. Visibility: when does Thread B see Thread A's write? Ordering: can the compiler or CPU reorder my instructions? The happens-before relationship is the formal guarantee that links them together." This framing immediately signals staff-level understanding.

3. Concurrent Collections

Thread-Safe Data Structures ▼

Why not just Collections.synchronizedMap()? Synchronized wrappers lock the entire collection for every operation, creating a bottleneck. At Google scale (millions of QPS), this serialization destroys throughput. The java.util.concurrent collections use fine-grained locking, lock striping, or lock-free algorithms to allow much higher concurrency.

ConcurrentHashMap -- The Workhorse

import java.util.concurrent.ConcurrentHashMap;
import java.util.concurrent.atomic.LongAdder;

public class HitCounter {
    // ConcurrentHashMap uses lock striping internally:
    // the table is divided into segments, each with its own lock.
    // In Java 8+, it uses CAS + synchronized on individual bins.
    private final ConcurrentHashMap<String, LongAdder> counters =
        new ConcurrentHashMap<>();

    /**
     * computeIfAbsent is atomic -- no check-then-act race.
     * LongAdder is faster than AtomicLong under high contention
     * because it uses cell striping (each thread increments its
     * own cell, cells are summed on read).
     */
    public void recordHit(String endpoint) {
        counters.computeIfAbsent(endpoint, k -> new LongAdder())
                .increment();
    }

    public long getHitCount(String endpoint) {
        LongAdder adder = counters.get(endpoint);
        return adder == null ? 0 : adder.sum();
    }

    // Bulk operations (Java 8+): parallel with a parallelism threshold
    public Map<String, Long> getAllCounts() {
        Map<String, Long> snapshot = new HashMap<>();
        // forEach with parallelism threshold of 1000:
        // if map has >1000 entries, operations run in ForkJoinPool
        counters.forEach(1000, (key, adder) ->
            snapshot.put(key, adder.sum())
        );
        return snapshot;
    }
}

CopyOnWriteArrayList -- For Read-Dominated Lists

import java.util.concurrent.CopyOnWriteArrayList;

/**
 * Event listener registry: written rarely (on startup or config change),
 * read on every event (thousands/sec). CopyOnWriteArrayList creates a
 * new array on every write, so reads never block and never see
 * ConcurrentModificationException.
 */
public class EventBus {
    private final CopyOnWriteArrayList<EventListener> listeners =
        new CopyOnWriteArrayList<>();

    public void register(EventListener listener) {
        listeners.add(listener); // O(n) copy -- fine if writes are rare
    }

    public void unregister(EventListener listener) {
        listeners.remove(listener);
    }

    public void publish(Event event) {
        // Safe iteration: snapshot semantics, no lock needed
        for (EventListener listener : listeners) {
            listener.onEvent(event);
        }
    }
}

BlockingQueue Implementations

// ArrayBlockingQueue: bounded, backed by array, single lock
// LinkedBlockingQueue: optionally bounded, separate put/take locks (higher throughput)
// PriorityBlockingQueue: unbounded, priority-ordered
// SynchronousQueue: zero-capacity, direct handoff (used in CachedThreadPool)

// Choosing the right one:
// - Bounded + backpressure needed?    -> ArrayBlockingQueue
// - High put/take throughput needed?  -> LinkedBlockingQueue
// - Priority ordering needed?         -> PriorityBlockingQueue
// - Direct producer-consumer handoff? -> SynchronousQueue
When NOT to Use:
  • CopyOnWriteArrayList: Never use for write-heavy workloads. Each write copies the entire array -- O(n) per write.
  • ConcurrentHashMap: Does not support null keys or values (unlike HashMap). If you need nulls, rethink your design or use a sentinel value.
  • size() / isEmpty(): On concurrent collections, these are approximations, not exact. Never make business decisions based on concurrentMap.size().

4. Producer-Consumer Pattern

Decoupling Production from Consumption ▼

Why this pattern matters: Almost every backend system is a producer-consumer pipeline. Web servers produce requests into a queue; worker threads consume them. Kafka producers write events; consumers process them. The pattern decouples the rate of production from the rate of consumption, absorbing bursts and enabling backpressure.

Bounded buffer absorbs burst traffic; backpressure blocks producers when full.

Implementation with BlockingQueue

import java.util.concurrent.*;

public class LogIngestionPipeline {
    // Bounded queue provides backpressure: producers block when full
    private final BlockingQueue<LogEntry> queue;
    private final ExecutorService producers;
    private final ExecutorService consumers;
    private volatile boolean running = true;

    public LogIngestionPipeline(int queueCapacity, int numProducers, int numConsumers) {
        this.queue = new ArrayBlockingQueue<>(queueCapacity);
        this.producers = Executors.newFixedThreadPool(numProducers);
        this.consumers = Executors.newFixedThreadPool(numConsumers);
    }

    public void start() {
        // Start consumer threads
        for (int i = 0; i < 4; i++) {
            consumers.submit(this::consumeLoop);
        }
    }

    /**
     * Called by HTTP handler threads (producers).
     * offer() with timeout avoids indefinite blocking -- critical
     * for maintaining response time SLAs.
     */
    public boolean submit(LogEntry entry) {
        try {
            boolean accepted = queue.offer(entry, 100, TimeUnit.MILLISECONDS);
            if (!accepted) {
                // Queue full -- shed load gracefully
                Metrics.increment("log.ingestion.dropped");
            }
            return accepted;
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        }
    }

    private void consumeLoop() {
        while (running) {
            try {
                // poll with timeout so we can check the 'running' flag
                LogEntry entry = queue.poll(500, TimeUnit.MILLISECONDS);
                if (entry != null) {
                    processEntry(entry);
                }
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }
    }

    private void processEntry(LogEntry entry) {
        // Parse, enrich, write to Elasticsearch / S3
        // Real systems batch entries for throughput
    }

    public void shutdown() {
        running = false;
        producers.shutdown();
        consumers.shutdown();
        try {
            consumers.awaitTermination(30, TimeUnit.SECONDS);
        } catch (InterruptedException e) {
            consumers.shutdownNow();
        }
    }
}
Interview Tip: When designing any LLD system with async processing, mention the producer-consumer pattern by name and discuss: (1) bounded vs unbounded queue, (2) backpressure strategy (block, drop, or buffer to disk), (3) graceful shutdown (drain the queue before stopping consumers).

5. Reader-Writer Locks

Maximizing Read Throughput ▼

Why: In many systems, reads vastly outnumber writes. A cache might serve 100,000 reads/sec but only update every 30 seconds. Using a single mutex for both reads and writes forces all reads to serialize -- a massive throughput loss. Reader-writer locks allow concurrent reads while ensuring exclusive write access.

ReentrantReadWriteLock

import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ConfigurationCache {
    private final ReentrantReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final ReentrantReadWriteLock.ReadLock readLock = rwLock.readLock();
    private final ReentrantReadWriteLock.WriteLock writeLock = rwLock.writeLock();
    private Map<String, String> config = new HashMap<>();

    /**
     * Multiple threads can read simultaneously.
     * Only blocks if a writer is active.
     */
    public String getProperty(String key) {
        readLock.lock();
        try {
            return config.get(key);
        } finally {
            readLock.unlock();
        }
    }

    /**
     * Write lock is exclusive: blocks all readers AND writers.
     * Reload the entire config atomically.
     */
    public void reloadConfig(Map<String, String> newConfig) {
        writeLock.lock();
        try {
            this.config = new HashMap<>(newConfig);
        } finally {
            writeLock.unlock();
        }
    }

    /**
     * Lock downgrade: write lock -> read lock (supported).
     * Useful when you want to update and then read without
     * releasing the lock (which would allow another writer in).
     */
    public String computeIfAbsent(String key, Supplier<String> supplier) {
        readLock.lock();
        try {
            String value = config.get(key);
            if (value != null) {
                return value;
            }
        } finally {
            readLock.unlock();
        }

        // Must acquire write lock to update
        writeLock.lock();
        try {
            // Double-check after acquiring write lock
            String value = config.get(key);
            if (value == null) {
                value = supplier.get();
                config.put(key, value);
            }
            // Downgrade: acquire read lock before releasing write lock
            readLock.lock();
            return value;
        } finally {
            writeLock.unlock();
            // Still holding read lock -- other readers can proceed
            readLock.unlock();
        }
    }
}

StampedLock Optimistic Read Pattern

import java.util.concurrent.locks.StampedLock;

/**
 * Rate limiter that tracks request counts per window.
 * Reads (checking if rate limit exceeded) happen on every request.
 * Writes (incrementing counter) happen on every request too,
 * but we can optimistically read the current count first.
 */
public class SlidingWindowRateLimiter {
    private final StampedLock lock = new StampedLock();
    private long windowStart;
    private int requestCount;
    private final int maxRequests;
    private final long windowMs;

    public SlidingWindowRateLimiter(int maxRequests, long windowMs) {
        this.maxRequests = maxRequests;
        this.windowMs = windowMs;
        this.windowStart = System.currentTimeMillis();
        this.requestCount = 0;
    }

    public boolean tryAcquire() {
        // Optimistic read first: if we're clearly under limit, no lock needed
        long stamp = lock.tryOptimisticRead();
        long ws = this.windowStart;
        int rc = this.requestCount;
        long now = System.currentTimeMillis();

        if (lock.validate(stamp)) {
            // No writer intervened
            if (now - ws > windowMs) {
                // Window expired -- need write lock to reset
            } else if (rc < maxRequests) {
                // Under limit but still need to increment (write)
            } else {
                return false; // Over limit, no write needed
            }
        }

        // Fall back to write lock for mutation
        stamp = lock.writeLock();
        try {
            if (System.currentTimeMillis() - windowStart > windowMs) {
                windowStart = System.currentTimeMillis();
                requestCount = 1;
                return true;
            }
            if (requestCount < maxRequests) {
                requestCount++;
                return true;
            }
            return false;
        } finally {
            lock.unlockWrite(stamp);
        }
    }
}
When NOT to Use:
  • ReentrantReadWriteLock: Avoid if writes are frequent (>20% of operations). The overhead of managing read/write separation is not worth it. Use a simple ReentrantLock instead.
  • StampedLock: Avoid if your read logic has side effects or is long-running. The optimistic read may need to retry, so your read code must be idempotent and fast.
  • Note: ReentrantReadWriteLock supports lock upgrade from read to write only through a downgrade path (write then read). Attempting read-to-write upgrade causes deadlock.

6. CompletableFuture & Async Composition

Non-Blocking Async Pipelines ▼

Why: In microservice architectures, a single API call often fans out to 3-5 downstream services. Calling them sequentially wastes time. CompletableFuture lets you compose async operations -- running them in parallel, chaining transformations, and handling errors -- without callback hell.

Chaining, Combining, and Error Handling

import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class ProductPageService {
    private final ProductService productService;
    private final ReviewService reviewService;
    private final RecommendationService recService;
    private final PricingService pricingService;
    private final ExecutorService executor = Executors.newFixedThreadPool(10);

    /**
     * Build a product page by fetching data from 4 services in parallel.
     * Total latency = max(individual latencies), not sum.
     * At Amazon scale, this reduces P99 from ~800ms to ~250ms.
     */
    public CompletableFuture<ProductPage> getProductPage(String productId) {

        // Fan-out: all four calls start immediately
        CompletableFuture<Product> productFuture =
            CompletableFuture.supplyAsync(
                () -> productService.getProduct(productId), executor);

        CompletableFuture<List<Review>> reviewsFuture =
            CompletableFuture.supplyAsync(
                () -> reviewService.getReviews(productId), executor)
            .exceptionally(ex -> {
                // Graceful degradation: reviews are non-critical
                log.warn("Failed to fetch reviews: {}", ex.getMessage());
                return List.of();
            });

        CompletableFuture<List<Product>> recsFuture =
            CompletableFuture.supplyAsync(
                () -> recService.getRecommendations(productId), executor)
            .completeOnTimeout(List.of(), 200, TimeUnit.MILLISECONDS);
            // Timeout: if recs take >200ms, show empty

        CompletableFuture<Price> priceFuture =
            CompletableFuture.supplyAsync(
                () -> pricingService.getPrice(productId), executor);

        // Combine all results
        return productFuture.thenCombine(priceFuture, (product, price) -> {
                product.setPrice(price);
                return product;
            })
            .thenCombine(reviewsFuture, (product, reviews) -> {
                return new ProductPage(product, reviews);
            })
            .thenCombine(recsFuture, (page, recs) -> {
                page.setRecommendations(recs);
                return page;
            })
            .orTimeout(500, TimeUnit.MILLISECONDS) // Overall deadline
            .exceptionally(ex -> {
                // Final fallback: return cached/default page
                log.error("Product page assembly failed", ex);
                return ProductPage.fallback(productId);
            });
    }
}

// Chaining transformations
CompletableFuture<String> pipeline =
    CompletableFuture.supplyAsync(() -> fetchRawData())    // Step 1
        .thenApplyAsync(data -> parse(data), executor)      // Step 2
        .thenApplyAsync(parsed -> validate(parsed), executor)// Step 3
        .thenApplyAsync(valid -> enrich(valid), executor)    // Step 4
        .handle((result, ex) -> {
            if (ex != null) {
                log.error("Pipeline failed at some stage", ex);
                return "default";
            }
            return result;
        });

Common Composition Patterns

// allOf: Wait for ALL futures (fan-out, fan-in)
CompletableFuture<Void> all = CompletableFuture.allOf(f1, f2, f3);

// anyOf: Wait for FIRST to complete (racing / hedged requests)
CompletableFuture<Object> first = CompletableFuture.anyOf(f1, f2, f3);

// Hedged request: send same request to two replicas, take fastest
public CompletableFuture<Response> hedgedRequest(Request req) {
    CompletableFuture<Response> primary =
        CompletableFuture.supplyAsync(() -> replica1.call(req));
    CompletableFuture<Response> secondary =
        CompletableFuture.supplyAsync(() -> replica2.call(req));
    return CompletableFuture.anyOf(primary, secondary)
        .thenApply(obj -> (Response) obj);
}
When NOT to Use:
  • Do not use CompletableFuture for simple sequential logic -- it adds complexity for no benefit.
  • Be careful with the default ForkJoinPool.commonPool(). Blocking I/O in the common pool starves CPU-bound tasks. Always pass a dedicated executor for I/O work.
  • In Java 21+, consider Virtual Threads (Project Loom) as a simpler alternative to CompletableFuture for I/O-bound workloads.
Interview Tip: When designing an API gateway or aggregation service in an LLD interview, use CompletableFuture to fan out requests and combine results. Mention: (1) per-call timeouts via completeOnTimeout, (2) graceful degradation via exceptionally, (3) overall deadline via orTimeout. This is exactly how Amazon, Netflix, and Uber build their aggregation layers.

7. Deadlock Detection & Prevention

Breaking the Deadlock Cycle ▼

Four conditions for deadlock (all must hold simultaneously):

  1. Mutual exclusion: A resource can only be held by one thread.
  2. Hold and wait: A thread holds one resource while waiting for another.
  3. No preemption: Resources cannot be forcibly taken from a thread.
  4. Circular wait: Thread A waits for Thread B, which waits for Thread A.

Classic deadlock: circular dependency between two threads and two locks.

Prevention Strategy 1: Lock Ordering

/**
 * The classic "transfer between bank accounts" deadlock.
 *
 * BAD: transfer(A, B) locks A then B
 *      transfer(B, A) locks B then A  -- DEADLOCK!
 *
 * FIX: Always acquire locks in a consistent global order.
 * Use a unique, immutable ID (like account number) to define order.
 */
public class SafeBankTransfer {

    public void transfer(BankAccount from, BankAccount to, long amount) {
        // Determine lock order by account ID
        BankAccount first = from.getId() < to.getId() ? from : to;
        BankAccount second = from.getId() < to.getId() ? to : from;

        // Tie-breaker for equal IDs (same account -- no-op, but be safe)
        if (from.getId() == to.getId()) {
            throw new IllegalArgumentException("Cannot transfer to same account");
        }

        synchronized (first) {
            synchronized (second) {
                if (from.getBalance() < amount) {
                    throw new InsufficientFundsException();
                }
                from.debit(amount);
                to.credit(amount);
            }
        }
    }
}

Prevention Strategy 2: tryLock with Timeout

/**
 * When you cannot impose a global lock order (e.g., dynamic resource
 * graphs), use tryLock with timeout to break the "no preemption" condition.
 */
public boolean transferWithRetry(BankAccount from, BankAccount to,
                                  long amount, int maxRetries) {
    for (int attempt = 0; attempt < maxRetries; attempt++) {
        boolean fromLocked = false;
        boolean toLocked = false;
        try {
            fromLocked = from.getLock().tryLock(50, TimeUnit.MILLISECONDS);
            toLocked = to.getLock().tryLock(50, TimeUnit.MILLISECONDS);

            if (fromLocked && toLocked) {
                // Both locks acquired -- proceed
                from.debit(amount);
                to.credit(amount);
                return true;
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        } finally {
            if (toLocked) to.getLock().unlock();
            if (fromLocked) from.getLock().unlock();
        }

        // Back off with jitter before retrying
        try {
            Thread.sleep(ThreadLocalRandom.current().nextInt(10, 50));
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
            return false;
        }
    }
    return false; // All retries exhausted
}

Detection: Thread Dumps and JMX

// Programmatic deadlock detection via ThreadMXBean
import java.lang.management.ManagementFactory;
import java.lang.management.ThreadMXBean;

public class DeadlockDetector {
    private final ThreadMXBean threadBean =
        ManagementFactory.getThreadMXBean();

    /**
     * Run periodically (e.g., every 30s) in a monitoring thread.
     * In production at Netflix, deadlock detection triggers alerts
     * and automatic thread dumps for diagnosis.
     */
    public void checkForDeadlocks() {
        long[] deadlockedThreads = threadBean.findDeadlockedThreads();
        if (deadlockedThreads != null) {
            ThreadInfo[] infos = threadBean.getThreadInfo(
                deadlockedThreads, true, true);
            StringBuilder sb = new StringBuilder("DEADLOCK DETECTED:\n");
            for (ThreadInfo info : infos) {
                sb.append(info.toString());
            }
            log.error(sb.toString());
            alertService.sendCritical("Deadlock detected", sb.toString());
        }
    }
}
Interview Tip: If an interviewer asks "How would you prevent deadlocks?", give three strategies in order of preference: (1) Lock ordering -- eliminates circular wait, simplest and most reliable. (2) tryLock with timeout -- handles cases where ordering is impractical. (3) Lock-free algorithms (CAS, atomic variables) -- eliminates locks entirely for simple operations. Then mention that in production, you always add deadlock detection via ThreadMXBean as a safety net.

8. Synchronization Utilities

CountDownLatch, CyclicBarrier, Phaser ▼

These utilities coordinate groups of threads. They are not about mutual exclusion (that is locks) -- they are about timing: making threads wait until a condition is met.

CountDownLatch -- One-Shot Gate

import java.util.concurrent.CountDownLatch;

/**
 * Real scenario: Service startup -- don't accept requests until
 * all dependencies (DB, cache, config) are initialized.
 * CountDownLatch counts down from N to 0. Once it hits 0, all
 * waiting threads proceed. It cannot be reset -- it's a one-shot gate.
 */
public class ServiceBootstrap {
    private final CountDownLatch readyLatch;
    private final List<Subsystem> subsystems;

    public ServiceBootstrap(List<Subsystem> subsystems) {
        this.subsystems = subsystems;
        this.readyLatch = new CountDownLatch(subsystems.size());
    }

    public void start() throws InterruptedException {
        for (Subsystem sub : subsystems) {
            new Thread(() -> {
                try {
                    sub.initialize();
                    log.info("{} ready", sub.getName());
                } finally {
                    readyLatch.countDown(); // Decrement regardless of success
                }
            }).start();
        }

        // Main thread blocks until all subsystems are initialized
        boolean allReady = readyLatch.await(30, TimeUnit.SECONDS);
        if (!allReady) {
            throw new RuntimeException("Subsystems failed to start in 30s");
        }

        log.info("All subsystems ready -- accepting traffic");
        httpServer.start();
    }
}

CyclicBarrier -- Reusable Sync Point

import java.util.concurrent.CyclicBarrier;

/**
 * Real scenario: Parallel data pipeline where each phase must
 * complete before the next begins. E.g., MapReduce-style processing.
 * Unlike CountDownLatch, CyclicBarrier can be reused across rounds.
 */
public class ParallelMatrixComputation {
    private final int numWorkers;
    private final CyclicBarrier barrier;
    private final double[][] matrix;

    public ParallelMatrixComputation(double[][] matrix, int numWorkers) {
        this.matrix = matrix;
        this.numWorkers = numWorkers;
        // Barrier action runs after all threads arrive
        this.barrier = new CyclicBarrier(numWorkers, () -> {
            log.info("All workers finished phase -- merging results");
            mergePartialResults();
        });
    }

    public void compute() {
        int rowsPerWorker = matrix.length / numWorkers;

        for (int i = 0; i < numWorkers; i++) {
            int startRow = i * rowsPerWorker;
            int endRow = (i == numWorkers - 1) ? matrix.length : startRow + rowsPerWorker;

            new Thread(() -> {
                for (int phase = 0; phase < 10; phase++) {
                    processRows(startRow, endRow, phase);
                    try {
                        barrier.await(); // Wait for all workers
                    } catch (Exception e) {
                        Thread.currentThread().interrupt();
                        return;
                    }
                }
            }).start();
        }
    }
}

Phaser -- Dynamic, Flexible Coordination

import java.util.concurrent.Phaser;

/**
 * Phaser is like a CyclicBarrier but supports:
 * - Dynamic registration/deregistration of parties
 * - Per-phase actions
 * - Termination conditions
 *
 * Real scenario: Web crawler where threads join/leave dynamically
 * as new URLs are discovered or crawl limits are reached.
 */
public class WebCrawler {
    private final Phaser phaser = new Phaser(1); // Register self
    private final Set<String> visited = ConcurrentHashMap.newKeySet();
    private final int maxDepth;

    public void crawl(String seedUrl) {
        submitCrawlTask(seedUrl, 0);

        // Wait for all phases to complete
        // arriveAndDeregister: "I'm done, and remove me from the party count"
        phaser.arriveAndDeregister();
    }

    private void submitCrawlTask(String url, int depth) {
        if (depth > maxDepth || !visited.add(url)) return;

        phaser.register(); // Add a new party dynamically
        CompletableFuture.runAsync(() -> {
            try {
                List<String> links = fetchAndParse(url);
                for (String link : links) {
                    submitCrawlTask(link, depth + 1);
                }
            } finally {
                phaser.arriveAndDeregister(); // Done with this task
            }
        });
    }
}

Quick Comparison

Utility Reusable? Dynamic Parties? Use Case
CountDownLatch No No Wait for N events (startup, test synchronization)
CyclicBarrier Yes No Multi-phase parallel computation
Phaser Yes Yes Dynamic task graphs, crawlers, fork-join-like coordination
When NOT to Use:
  • CountDownLatch: Don't use if you need to reset the gate. Use CyclicBarrier instead.
  • CyclicBarrier: Don't use if threads join/leave dynamically. One broken thread breaks the barrier for all.
  • Phaser: Overkill for simple "wait for N things" scenarios. Use CountDownLatch for simplicity.

9. Thread Pools & Executor Framework

Sizing, Work-Stealing, and Virtual Threads ▼

Why thread pools: Creating a thread costs ~1MB of stack memory and ~100us of OS time. At 10,000 requests/sec, you would create and destroy 10,000 threads per second -- the OS would spend more time managing threads than running your code. Thread pools amortize creation cost and bound resource usage.

Sizing Guidelines

/**
 * Thread pool sizing formulas (Brian Goetz, "Java Concurrency in Practice"):
 *
 * CPU-bound tasks: threads = N_cpu + 1
 *   Why +1? If a thread occasionally blocks (page fault, GC pause),
 *   the extra thread keeps the CPU fully utilized.
 *
 * I/O-bound tasks: threads = N_cpu * (1 + W/C)
 *   W = average wait time (I/O, network)
 *   C = average compute time
 *   Example: 8 cores, 100ms wait, 10ms compute
 *            threads = 8 * (1 + 100/10) = 88 threads
 *
 * In practice, benchmark and adjust. These are starting points.
 */
public class ThreadPoolFactory {

    // CPU-bound: image processing, encryption, compression
    public static ExecutorService cpuBoundPool() {
        int cpus = Runtime.getRuntime().availableProcessors();
        return new ThreadPoolExecutor(
            cpus + 1,                          // core pool size
            cpus + 1,                          // max pool size (same for CPU-bound)
            60L, TimeUnit.SECONDS,             // idle thread keepalive
            new LinkedBlockingQueue<>(1000),   // bounded queue for backpressure
            new ThreadPoolExecutor.CallerRunsPolicy() // backpressure: caller thread runs task
        );
    }

    // I/O-bound: HTTP calls, DB queries, file I/O
    public static ExecutorService ioBoundPool() {
        int cpus = Runtime.getRuntime().availableProcessors();
        int ioThreads = cpus * 10; // Assumes W/C ratio ~9
        return new ThreadPoolExecutor(
            cpus,                              // core: keep some always alive
            ioThreads,                         // max: scale up under load
            120L, TimeUnit.SECONDS,
            new SynchronousQueue<>(),          // direct handoff -- forces thread creation
            new ThreadPoolExecutor.AbortPolicy() // reject if at max
        );
    }
}

ForkJoinPool -- Work-Stealing for Recursive Tasks

import java.util.concurrent.ForkJoinPool;
import java.util.concurrent.RecursiveTask;

/**
 * ForkJoinPool uses work-stealing: each thread has its own deque.
 * When a thread finishes its tasks, it steals from the tail of
 * another thread's deque. This provides excellent load balancing
 * for recursive divide-and-conquer problems.
 *
 * Used internally by parallel streams, CompletableFuture (common pool).
 */
public class ParallelMergeSort extends RecursiveTask<int[]> {
    private static final int THRESHOLD = 1024;
    private final int[] array;
    private final int start;
    private final int end;

    public ParallelMergeSort(int[] array, int start, int end) {
        this.array = array;
        this.start = start;
        this.end = end;
    }

    @Override
    protected int[] compute() {
        int length = end - start;

        if (length <= THRESHOLD) {
            // Base case: sort sequentially
            int[] subArray = Arrays.copyOfRange(array, start, end);
            Arrays.sort(subArray);
            return subArray;
        }

        int mid = start + length / 2;

        // Fork: submit left half to another thread
        ParallelMergeSort left = new ParallelMergeSort(array, start, mid);
        left.fork();

        // Compute right half in current thread
        ParallelMergeSort right = new ParallelMergeSort(array, mid, end);
        int[] rightResult = right.compute();

        // Join: wait for left half
        int[] leftResult = left.join();

        return merge(leftResult, rightResult);
    }

    // Usage
    public static int[] sort(int[] array) {
        ForkJoinPool pool = new ForkJoinPool(
            Runtime.getRuntime().availableProcessors()
        );
        return pool.invoke(new ParallelMergeSort(array, 0, array.length));
    }
}

Java 21+: Virtual Threads (Project Loom)

// Virtual threads: lightweight (few KB vs 1MB), scheduled by JVM not OS.
// Ideal for I/O-bound workloads -- you can have millions of them.
// No need to size thread pools for I/O anymore.

// Java 21+
try (var executor = Executors.newVirtualThreadPerTaskExecutor()) {
    List<Future<String>> futures = new ArrayList<>();
    for (String url : urls) {
        futures.add(executor.submit(() -> fetchUrl(url)));
    }
    for (Future<String> f : futures) {
        String result = f.get(); // Blocks virtual thread, not OS thread
    }
}

// CAUTION: Virtual threads are NOT faster for CPU-bound work.
// They excel at I/O-bound work where traditional threads waste
// time blocked on network/disk calls.
Interview Tip: When asked about thread pool sizing, always start with: "It depends on whether the workload is CPU-bound or I/O-bound." Give the formulas, then say: "But these are starting points -- in production, I instrument the pool (queue depth, active threads, rejection count) and tune based on load testing. I also prefer bounded queues with a rejection policy over unbounded queues, because unbounded queues hide backpressure and cause OOM."
When NOT to Use:
  • ForkJoinPool: Don't use for I/O-bound tasks. Blocked threads in ForkJoinPool cause thread starvation since work-stealing only helps when threads are computing, not waiting.
  • CachedThreadPool: Dangerous in production -- it creates unlimited threads. A burst of traffic can spawn thousands of threads and OOM the JVM.
  • SingleThreadExecutor: Fine for sequential task queues, but never use for latency-sensitive work (everything waits in line).

Self-Check Quiz

Flashcard Suggestions

5 Q/A Pairs for Spaced Repetition ▼
  1. Q: What three problems does the Java Memory Model address?
    A: Atomicity (compound actions aren't indivisible), visibility (threads may see stale values due to CPU caches), and ordering (compilers/CPUs can reorder instructions). The happens-before relationship is the formal guarantee that addresses all three.
  2. Q: When should you use StampedLock over ReentrantReadWriteLock?
    A: When reads vastly outnumber writes and you can tolerate retrying reads. StampedLock's optimistic read acquires no lock and is essentially free, but requires validation. Use ReentrantReadWriteLock when you need reentrancy or cannot retry.
  3. Q: What is the thread pool sizing formula for I/O-bound tasks?
    A: threads = N_cpu * (1 + W/C), where W is average wait time and C is average compute time. For example, 8 cores with 100ms wait and 10ms compute = 8 * 11 = 88 threads.
  4. Q: Name the four conditions required for deadlock.
    A: Mutual exclusion (resource held exclusively), hold and wait (thread holds one lock while waiting for another), no preemption (locks can't be forcibly taken), and circular wait (A waits for B, B waits for A). Breaking any one condition prevents deadlock.
  5. Q: What is the difference between CountDownLatch and CyclicBarrier?
    A: CountDownLatch is a one-shot gate that counts down from N to 0 and cannot be reset. CyclicBarrier is reusable across phases -- once all parties arrive, it resets automatically. Use CountDownLatch for "wait for N events", CyclicBarrier for "synchronize N threads repeatedly".

Practice Exercise

Implement a Thread-Safe Bounded Buffer ▼

Challenge: Implement a generic BoundedBuffer<T> with the following requirements:

  • put(T item) -- blocks if the buffer is full
  • T take() -- blocks if the buffer is empty
  • int size() -- returns the current number of items
  • Use ReentrantLock with two Condition variables (notFull, notEmpty)
  • Support graceful shutdown via an interrupt()-friendly design

Stretch goals:

  • Add offer(T item, long timeout, TimeUnit unit) with timeout
  • Add drain(Collection<T> dest, int maxElements) for batch consumption
  • Write a stress test with 10 producers and 5 consumers that verifies no data is lost

Sample Solution

import java.util.concurrent.locks.Condition;
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;
import java.util.Collection;

public class BoundedBuffer<T> {
    private final Object[] items;
    private int head; // index of next take
    private int tail; // index of next put
    private int count;

    private final ReentrantLock lock = new ReentrantLock();
    private final Condition notFull = lock.newCondition();
    private final Condition notEmpty = lock.newCondition();

    public BoundedBuffer(int capacity) {
        if (capacity <= 0) throw new IllegalArgumentException();
        this.items = new Object[capacity];
    }

    public void put(T item) throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == items.length) {
                notFull.await(); // Release lock and wait
            }
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal(); // Wake one waiting consumer
        } finally {
            lock.unlock();
        }
    }

    @SuppressWarnings("unchecked")
    public T take() throws InterruptedException {
        lock.lockInterruptibly();
        try {
            while (count == 0) {
                notEmpty.await(); // Release lock and wait
            }
            T item = (T) items[head];
            items[head] = null; // Help GC
            head = (head + 1) % items.length;
            count--;
            notFull.signal(); // Wake one waiting producer
            return item;
        } finally {
            lock.unlock();
        }
    }

    public boolean offer(T item, long timeout, TimeUnit unit)
            throws InterruptedException {
        long nanos = unit.toNanos(timeout);
        lock.lockInterruptibly();
        try {
            while (count == items.length) {
                if (nanos <= 0) return false;
                nanos = notFull.awaitNanos(nanos);
            }
            items[tail] = item;
            tail = (tail + 1) % items.length;
            count++;
            notEmpty.signal();
            return true;
        } finally {
            lock.unlock();
        }
    }

    public int drain(Collection<T> dest, int maxElements) {
        lock.lock();
        try {
            int n = Math.min(count, maxElements);
            for (int i = 0; i < n; i++) {
                @SuppressWarnings("unchecked")
                T item = (T) items[head];
                dest.add(item);
                items[head] = null;
                head = (head + 1) % items.length;
                count--;
            }
            notFull.signalAll(); // Wake all waiting producers
            return n;
        } finally {
            lock.unlock();
        }
    }

    public int size() {
        lock.lock();
        try {
            return count;
        } finally {
            lock.unlock();
        }
    }
}

Key design decisions to discuss in an interview:

  • Two conditions vs one: Using separate notFull and notEmpty conditions avoids waking producers when only consumers should wake (and vice versa). With a single condition, signalAll() would wake all waiters unnecessarily.
  • Circular array: O(1) put and take with no array shifting. The modular arithmetic (% items.length) wraps around.
  • lockInterruptibly: Allows the thread to be interrupted while waiting for the lock, enabling graceful shutdown.
  • signal() vs signalAll(): We use signal() for single put/take (only one waiter needs to wake) and signalAll() for drain() (multiple producers may need to wake).
