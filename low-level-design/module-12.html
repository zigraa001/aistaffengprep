Module 12: Mock Interviews

What You'll Learn

  • How to execute a complete 45-minute LLD interview from start to finish
  • Three full mock interview walkthroughs with staff-level model answers
  • Timing management strategies for each interview phase
  • Common mistakes and how to recover from them mid-interview
  • What distinguishes a staff-level answer from a senior-level answer
  • Techniques for handling "I don't know" moments and interviewer disagreement

Why Mock Practice is Essential

The Case for Deliberate Practice

Knowing design patterns and principles is necessary but insufficient. Staff-level interviews test your ability to apply knowledge under pressure, communicate trade-offs fluently, and navigate ambiguity with confidence. The gap between "understanding the theory" and "delivering under interview conditions" is where most candidates fail.

Why Reading Alone Doesn't Work

  • Time pressure changes behavior: Under a 45-minute clock, candidates over-engineer, forget concurrency, or panic and go silent. Mock practice builds the muscle memory to stay composed.
  • Communication is half the score: You may design a perfect system in your head, but if you can't articulate your reasoning, the interviewer sees nothing. Practice forces verbalization.
  • Feedback loops are critical: Without a mock partner or self-review, you can't identify blind spots -- perhaps you always forget to clarify requirements, or you never address thread safety first.

How to Use These Mock Interviews

  1. First pass: Set a 45-minute timer and attempt the problem yourself before reading the walkthrough.
  2. Compare: Read the model walkthrough and note what you missed.
  3. Repeat: Do the same problem again a week later. Your second attempt should be noticeably sharper.
  4. Practice aloud: Speak your thought process out loud, even alone. This is the single highest-leverage practice technique.

// --- Shopping Cart with thread-safe operations ---
public class ShoppingCart {
    private final String cartId;
    private final User user;
    private final ConcurrentHashMap<String, CartItem> items = new ConcurrentHashMap<>();

    public ShoppingCart(User user) {
        this.cartId = UUID.randomUUID().toString();
        this.user = user;
    }

    public void addItem(Book book, int quantity) {
        items.merge(book.getIsbn(),
            new CartItem(book, quantity),
            (existing, added) -> {
                existing.setQuantity(existing.getQuantity() + added.getQuantity());
                return existing;
            });
    }

    public void removeItem(String isbn) {
        items.remove(isbn);
    }

    public BigDecimal getTotal() {
        return items.values().stream()
            .map(item -> item.getBook().getPrice()
                .multiply(BigDecimal.valueOf(item.getQuantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    public Map<String, CartItem> getItems() {
        return Collections.unmodifiableMap(items);
    }

    public void clear() {
        items.clear();
    }
}

// --- Inventory Manager: thread-safe stock reservation ---
public class InventoryManager {
    private final ConcurrentHashMap<String, AtomicInteger> stock =
        new ConcurrentHashMap<>();

    public boolean reserve(String isbn, int quantity) {
        AtomicInteger current = stock.get(isbn);
        if (current == null) return false;

        // CAS loop for atomic reservation
        while (true) {
            int existing = current.get();
            if (existing < quantity) return false;
            if (current.compareAndSet(existing, existing - quantity)) {
                return true;
            }
            // Another thread modified stock; retry
        }
    }

    public void release(String isbn, int quantity) {
        stock.computeIfPresent(isbn, (key, val) -> {
            val.addAndGet(quantity);
            return val;
        });
    }
}

// --- Order Service: orchestrates checkout ---
public class OrderService {
    private final InventoryManager inventory;
    private final PaymentProcessor paymentProcessor;

    public OrderService(InventoryManager inventory,
                        PaymentProcessor paymentProcessor) {
        this.inventory = inventory;
        this.paymentProcessor = paymentProcessor;
    }

    public Order checkout(ShoppingCart cart, PaymentInfo paymentInfo) {
        // Step 1: Reserve inventory for all items
        List<Map.Entry<String, CartItem>> reserved = new ArrayList<>();
        try {
            for (var entry : cart.getItems().entrySet()) {
                boolean success = inventory.reserve(
                    entry.getKey(), entry.getValue().getQuantity());
                if (!success) {
                    // Rollback already-reserved items
                    rollbackReservations(reserved);
                    throw new OutOfStockException(
                        "Book " + entry.getKey() + " is out of stock");
                }
                reserved.add(entry);
            }

            // Step 2: Process payment
            PaymentResult result = paymentProcessor.processPayment(
                cart.getTotal(), paymentInfo);

            if (!result.isSuccess()) {
                rollbackReservations(reserved);
                throw new PaymentFailedException("Payment declined");
            }

            // Step 3: Create order
            List<OrderItem> orderItems = cart.getItems().values().stream()
                .map(ci -> new OrderItem(
                    ci.getBook(), ci.getQuantity(),
                    ci.getBook().getPrice()))
                .toList();

            Order order = new Order(
                UUID.randomUUID().toString(),
                cart.getUser(),
                orderItems,
                cart.getTotal(),
                result.getTransactionId());

            cart.clear();
            return order;

        } catch (Exception e) {
            rollbackReservations(reserved);
            throw e;
        }
    }

    private void rollbackReservations(
            List<Map.Entry<String, CartItem>> reserved) {
        for (var entry : reserved) {
            inventory.release(entry.getKey(),
                entry.getValue().getQuantity());
        }
    }
}

Staff-Level Follow-Up Questions & Model Answers

Q: "How would you handle two users buying the last copy simultaneously?"

Model Answer: "The InventoryManager.reserve() uses a CAS (compare-and-set) loop on an AtomicInteger. Only one thread can successfully decrement from 1 to 0. The losing thread gets false from reserve() and we surface an OutOfStockException. In a distributed setting, I'd replace this with optimistic locking on the database row -- UPDATE books SET stock = stock - 1 WHERE isbn = ? AND stock >= 1 -- and check the affected row count."

Q: "What if payment succeeds but order creation fails?"

Model Answer: "This is the classic distributed transaction problem. I'd use the Saga pattern: if order creation fails after payment, we issue a compensating transaction (refund). In the code, I'd wrap the post-payment logic in a try-catch and call paymentProcessor.refund(transactionId) on failure. For idempotency, the order ID should be generated before the payment call and passed as an idempotency key."

Q: "How would you add a recommendation engine without changing existing classes?"

Model Answer: "I'd use the Observer pattern. When an order is placed, the OrderService publishes an OrderPlacedEvent. A RecommendationService subscribes to these events and updates user preference profiles. This follows the Open/Closed Principle -- we add behavior without modifying OrderService. The event could carry the user ID and list of ISBNs purchased."

Phase 3: Core Implementation -- Seat Locking & Concurrency

The hardest part of this problem is concurrent seat selection. Two users viewing the same show must not book the same seat. We solve this with a temporary lock mechanism.

// --- Seat lock with expiration ---
public record SeatLock(
    String userId,
    LocalDateTime lockedAt,
    LocalDateTime expiresAt
) {}

// --- Thread-safe seat lock manager ---
public class SeatLockManager {
    private final ConcurrentHashMap<String, SeatLock> activeLocks =
        new ConcurrentHashMap<>();
    private final Duration lockTimeout;

    public SeatLockManager(Duration lockTimeout) {
        this.lockTimeout = lockTimeout;
    }

    /**
     * Attempts to lock a set of seats atomically.
     * Either all seats are locked, or none are.
     */
    public boolean lockSeats(String showId, List<String> seatIds,
                             String userId) {
        List<String> lockKeys = seatIds.stream()
            .map(seatId -> buildKey(showId, seatId))
            .toList();

        // Acquire locks in sorted order to prevent deadlocks
        List<String> sortedKeys = new ArrayList<>(lockKeys);
        Collections.sort(sortedKeys);

        List<String> acquiredKeys = new ArrayList<>();

        for (String key : sortedKeys) {
            LocalDateTime now = LocalDateTime.now();
            SeatLock newLock = new SeatLock(
                userId, now, now.plus(lockTimeout));

            SeatLock existing = activeLocks.putIfAbsent(key, newLock);

            if (existing != null) {
                // Check if existing lock has expired
                if (existing.expiresAt().isBefore(now)) {
                    // Try to replace the expired lock
                    if (activeLocks.replace(key, existing, newLock)) {
                        acquiredKeys.add(key);
                        continue;
                    }
                }
                // Lock is held by another user -- rollback
                releaseKeys(acquiredKeys);
                return false;
            }
            acquiredKeys.add(key);
        }
        return true;
    }

    public void releaseLock(String showId, List<String> seatIds) {
        List<String> keys = seatIds.stream()
            .map(seatId -> buildKey(showId, seatId))
            .toList();
        releaseKeys(keys);
    }

    private void releaseKeys(List<String> keys) {
        keys.forEach(activeLocks::remove);
    }

    private String buildKey(String showId, String seatId) {
        return showId + ":" + seatId;
    }
}

// --- Booking Service ---
public class BookingService {
    private final SeatLockManager lockManager;
    private final PaymentProcessor paymentProcessor;
    private final BookingRepository bookingRepo;
    private final ShowRepository showRepo;

    public BookingService(SeatLockManager lockManager,
                          PaymentProcessor paymentProcessor,
                          BookingRepository bookingRepo,
                          ShowRepository showRepo) {
        this.lockManager = lockManager;
        this.paymentProcessor = paymentProcessor;
        this.bookingRepo = bookingRepo;
        this.showRepo = showRepo;
    }

    /**
     * Step 1: Lock seats and create a pending booking.
     * User has N minutes to complete payment.
     */
    public Booking initiateBooking(User user, String showId,
                                    List<String> seatIds) {
        Show show = showRepo.findById(showId);

        // Validate seats are bookable
        for (String seatId : seatIds) {
            if (show.getSeatStatus(seatId) != SeatStatus.AVAILABLE) {
                throw new SeatUnavailableException(
                    "Seat " + seatId + " is not available");
            }
        }

        // Attempt to lock
        boolean locked = lockManager.lockSeats(
            showId, seatIds, user.getUserId());
        if (!locked) {
            throw new SeatUnavailableException(
                "Could not lock requested seats -- " +
                "another user may be booking them");
        }

        // Create pending booking
        BigDecimal total = calculatePrice(show, seatIds);
        Booking booking = new Booking(
            UUID.randomUUID().toString(),
            user, show, seatIds,
            total, BookingStatus.PENDING);

        bookingRepo.save(booking);
        return booking;
    }

    /**
     * Step 2: Process payment and confirm.
     */
    public Booking confirmBooking(String bookingId,
                                   PaymentInfo paymentInfo) {
        Booking booking = bookingRepo.findById(bookingId);

        if (booking.getStatus() != BookingStatus.PENDING) {
            throw new IllegalStateException(
                "Booking is not in PENDING state");
        }

        PaymentResult result = paymentProcessor.processPayment(
            booking.getTotalAmount(), paymentInfo);

        if (!result.isSuccess()) {
            // Release locks on payment failure
            lockManager.releaseLock(
                booking.getShow().getShowId(),
                booking.getSeatIds());
            booking.setStatus(BookingStatus.FAILED);
            bookingRepo.save(booking);
            throw new PaymentFailedException("Payment declined");
        }

        // Mark seats as booked in the show
        booking.getShow().confirmBooking(booking.getSeatIds());
        booking.setStatus(BookingStatus.CONFIRMED);
        booking.setPaymentTransactionId(result.getTransactionId());
        bookingRepo.save(booking);

        // Release temporary locks (seats are now permanently booked)
        lockManager.releaseLock(
            booking.getShow().getShowId(),
            booking.getSeatIds());

        return booking;
    }

    private BigDecimal calculatePrice(Show show,
                                       List<String> seatIds) {
        return seatIds.stream()
            .map(id -> show.getScreen().getSeat(id))
            .map(seat -> switch (seat.getType()) {
                case PREMIUM -> new BigDecimal("15.00");
                case REGULAR -> new BigDecimal("10.00");
                case ECONOMY -> new BigDecimal("7.00");
            })
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }
}

// --- Show scheduling ---
public class ShowScheduler {
    private final ShowRepository showRepo;
    private final ScreenRepository screenRepo;

    public Show scheduleShow(Movie movie, String screenId,
                              LocalDateTime startTime) {
        Screen screen = screenRepo.findById(screenId);

        // Check for conflicts: no overlapping shows on same screen
        LocalDateTime endTime = startTime.plusMinutes(
            movie.getDurationMinutes() + 30); // 30 min buffer for cleanup

        List<Show> conflicts = showRepo.findOverlapping(
            screenId, startTime, endTime);

        if (!conflicts.isEmpty()) {
            throw new ScheduleConflictException(
                "Screen " + screenId + " is occupied from " +
                conflicts.get(0).getStartTime());
        }

        // Initialize all seats as available
        Map<String, SeatStatus> seatAvailability = new HashMap<>();
        for (Seat seat : screen.getSeats()) {
            seatAvailability.put(seat.getSeatId(), SeatStatus.AVAILABLE);
        }

        Show show = new Show(
            UUID.randomUUID().toString(),
            movie, screen, startTime, seatAvailability);

        showRepo.save(show);
        return show;
    }
}

Staff-Level Follow-Up Questions & Model Answers

Q: "What happens if a user locks seats but never completes payment?"

Model Answer: "The SeatLock has an expiresAt field. A background scheduled task runs every 30 seconds to clean up expired locks. Additionally, any new lock attempt checks if the existing lock is expired and atomically replaces it using ConcurrentHashMap.replace(key, oldVal, newVal). The timeout is configurable -- typically 7-10 minutes for movie booking."

Q: "How would you prevent a bot from locking all seats repeatedly?"

Model Answer: "Three layers: (1) Rate limiting per user ID -- max 3 active lock sessions per user. (2) CAPTCHA on the seat selection page. (3) A per-user cooldown if they fail to complete payment more than 3 times in a row. At scale, I'd also add anomaly detection on locking patterns."

Q: "How would this design change for a distributed, multi-server deployment?"

Model Answer: "The in-memory ConcurrentHashMap for locks would be replaced by Redis with SET key value NX EX timeout for distributed locking. The NX flag ensures atomicity across servers. For seat status, I'd use a database row-level lock: SELECT ... FOR UPDATE within a transaction, or optimistic locking with a version column."

// --- Post creation and management ---
public class Post {
    private final String postId;
    private final String authorId;
    private final String content;
    private final String imageUrl; // nullable
    private final LocalDateTime createdAt;
    private final AtomicInteger likeCount = new AtomicInteger(0);
    private final CopyOnWriteArrayList<Comment> comments =
        new CopyOnWriteArrayList<>();

    public Post(String authorId, String content, String imageUrl) {
        this.postId = UUID.randomUUID().toString();
        this.authorId = authorId;
        this.content = Objects.requireNonNull(content);
        this.imageUrl = imageUrl;
        this.createdAt = LocalDateTime.now();
    }

    public void addLike() { likeCount.incrementAndGet(); }
    public void removeLike() { likeCount.decrementAndGet(); }

    public void addComment(Comment comment) {
        comments.add(comment);
    }

    // Getters omitted for brevity
}

// --- Follow/Unfollow with thread safety ---
public class UserRelationshipManager {
    // userId -> set of userIds they follow
    private final ConcurrentHashMap<String, Set<String>> followGraph =
        new ConcurrentHashMap<>();

    public void follow(String followerId, String followeeId) {
        if (followerId.equals(followeeId)) {
            throw new IllegalArgumentException("Cannot follow yourself");
        }
        followGraph
            .computeIfAbsent(followerId,
                k -> ConcurrentHashMap.newKeySet())
            .add(followeeId);
    }

    public void unfollow(String followerId, String followeeId) {
        Set<String> following = followGraph.get(followerId);
        if (following != null) {
            following.remove(followeeId);
        }
    }

    public Set<String> getFollowing(String userId) {
        return followGraph.getOrDefault(userId, Set.of());
    }
}

// --- Feed Generation with Strategy Pattern ---
public interface FeedStrategy {
    List<Post> generateFeed(String userId, Set<String> followingIds,
                             int page, int pageSize);
}

public class ChronologicalFeedStrategy implements FeedStrategy {
    private final PostRepository postRepo;

    public ChronologicalFeedStrategy(PostRepository postRepo) {
        this.postRepo = postRepo;
    }

    @Override
    public List<Post> generateFeed(String userId,
                                     Set<String> followingIds,
                                     int page, int pageSize) {
        return postRepo.findByAuthorsOrderByCreatedAtDesc(
            followingIds, page, pageSize);
    }
}

public class RankedFeedStrategy implements FeedStrategy {
    private final PostRepository postRepo;

    public RankedFeedStrategy(PostRepository postRepo) {
        this.postRepo = postRepo;
    }

    @Override
    public List<Post> generateFeed(String userId,
                                     Set<String> followingIds,
                                     int page, int pageSize) {
        // Fetch recent posts (e.g., last 48 hours)
        List<Post> candidates = postRepo.findRecentByAuthors(
            followingIds, Duration.ofHours(48));

        // Score and rank
        return candidates.stream()
            .sorted(Comparator.comparingDouble(
                (Post p) -> calculateScore(p, userId)).reversed())
            .skip((long) page * pageSize)
            .limit(pageSize)
            .toList();
    }

    private double calculateScore(Post post, String userId) {
        double recencyScore = computeRecency(post.getCreatedAt());
        double engagementScore = post.getLikeCount() * 1.0
            + post.getComments().size() * 2.0;
        double affinityScore = computeAffinity(
            userId, post.getAuthorId());

        return (recencyScore * 0.4)
             + (engagementScore * 0.35)
             + (affinityScore * 0.25);
    }

    private double computeRecency(LocalDateTime createdAt) {
        long hoursAgo = Duration.between(
            createdAt, LocalDateTime.now()).toHours();
        return Math.max(0, 100.0 - hoursAgo * 2);
    }

    private double computeAffinity(String userId, String authorId) {
        // In production: based on interaction history
        // (likes, comments, profile views)
        return 50.0; // placeholder
    }
}

// --- Feed Service ---
public class FeedService {
    private final UserRelationshipManager relationshipManager;
    private volatile FeedStrategy strategy;

    public FeedService(UserRelationshipManager relationshipManager,
                       FeedStrategy strategy) {
        this.relationshipManager = relationshipManager;
        this.strategy = strategy;
    }

    public void setStrategy(FeedStrategy strategy) {
        this.strategy = strategy;
    }

    public List<Post> getFeed(String userId, int page, int pageSize) {
        Set<String> following =
            relationshipManager.getFollowing(userId);

        if (following.isEmpty()) {
            return List.of(); // Could return trending posts instead
        }

        return strategy.generateFeed(
            userId, following, page, pageSize);
    }
}

// --- Notification System (Observer Pattern) ---
public interface NotificationObserver {
    void onNewPost(String targetUserId, Post post);
}

public class PushNotificationObserver implements NotificationObserver {
    @Override
    public void onNewPost(String targetUserId, Post post) {
        // Send push notification via FCM/APNs
        System.out.println("Push to " + targetUserId +
            ": New post from " + post.getAuthorId());
    }
}

public class InAppNotificationObserver implements NotificationObserver {
    private final NotificationRepository notifRepo;

    public InAppNotificationObserver(NotificationRepository notifRepo) {
        this.notifRepo = notifRepo;
    }

    @Override
    public void onNewPost(String targetUserId, Post post) {
        notifRepo.save(new Notification(
            targetUserId,
            "New post from " + post.getAuthorId(),
            NotificationType.NEW_POST,
            post.getPostId()));
    }
}

public class PostService {
    private final PostRepository postRepo;
    private final UserRelationshipManager relationships;
    private final List<NotificationObserver> observers;
    private final ExecutorService notificationExecutor;

    public PostService(PostRepository postRepo,
                       UserRelationshipManager relationships,
                       List<NotificationObserver> observers) {
        this.postRepo = postRepo;
        this.relationships = relationships;
        this.observers = observers;
        this.notificationExecutor =
            Executors.newFixedThreadPool(4);
    }

    public Post createPost(String authorId, String content,
                            String imageUrl) {
        Post post = new Post(authorId, content, imageUrl);
        postRepo.save(post);

        // Notify followers asynchronously
        notificationExecutor.submit(() -> {
            Set<String> followers =
                relationships.getFollowers(authorId);
            for (String followerId : followers) {
                for (NotificationObserver observer : observers) {
                    observer.onNewPost(followerId, post);
                }
            }
        });

        return post;
    }
}

Staff-Level Follow-Up Questions & Model Answers

Q: "How would you handle a celebrity with 10M followers posting?"

Model Answer: "This is the fan-out problem. For users with massive follower counts, I'd switch from push-based (write to each follower's feed on post) to pull-based (followers query the celebrity's posts at read time). A hybrid approach: users with fewer than 10K followers use push; above that, use pull. The FeedStrategy interface already supports this -- we create a HybridFeedStrategy that merges pre-computed feed entries with live pulls from high-follower accounts."

Q: "What if the ranking algorithm needs to change frequently?"

Model Answer: "The Strategy pattern already isolates the algorithm. To make it runtime-configurable, I'd introduce a FeedStrategyFactory that reads weights from a configuration service. For A/B testing, the factory can assign different strategies based on user segment: if (user.isInExperimentGroup('ranked-v2')) return new RankedV2Strategy(). The key insight is that FeedService never changes -- only the injected strategy does."

"Graceful Fail" Techniques

How to Handle "I Don't Know" Moments

Even the best candidates encounter moments where they don't know the answer. What matters is how you handle it. Interviewers evaluate your problem-solving process, not just your knowledge.

Technique 1: Acknowledge and Reason From First Principles

Instead of: "I don't know how to handle that."

Say: "I haven't implemented that specific pattern before, but let me reason through it. The core problem is X, so I'd need a mechanism that ensures Y. One approach would be..."

Technique 2: Partial Answer + Bridge to Strength

Instead of: "I'm not sure about distributed locking."

Say: "For the in-process case, I'd use ReentrantLock with tryLock and a timeout. For the distributed case, I know Redis provides SET NX EX for atomic lock acquisition, though I'd want to research the fencing token approach for correctness. Let me show you the single-process version, which demonstrates the same locking semantics..."

Technique 3: State What You Do Know

Instead of: "I'm blanking on the pattern name."

Say: "The pattern I'm thinking of separates the request from its handler and lets you chain handlers together. Each handler decides whether to process or pass along. Let me sketch the interface..." (This is Chain of Responsibility -- showing you understand the concept matters more than naming it.)

Technique 4: Ask a Clarifying Question to Buy Time

Instead of: Staring at the whiteboard in silence.

Say: "That's an interesting edge case. Before I address it -- are we assuming a single-server deployment or multi-server? That changes whether I'd use an in-memory lock or a distributed lock..."

This is not stalling -- it genuinely narrows the problem space and gives you a moment to think.

Disagreeing with the Interviewer

How to Push Back Professionally

Staff engineers are expected to have opinions and defend them. Interviewers sometimes intentionally suggest suboptimal approaches to see if you'll push back or blindly comply. But how you disagree matters enormously.

The ACE Framework for Disagreement

  • Acknowledge their point: "That's a valid approach..."
  • Contrast with your reasoning: "My concern with that is..."
  • Evaluate together: "What if we compared both approaches against our requirements?"

Example Scenarios

Scenario: Interviewer suggests Singleton for everything

Bad response: "No, Singleton is an anti-pattern."

Good response: "Singleton works well for the ParkingLot since there's genuinely one instance. For PaymentProcessor, though, I'd prefer dependency injection -- it makes the class testable (we can mock the processor) and lets us support multiple payment providers. Would you like me to show both approaches?"

Scenario: Interviewer questions your pattern choice

Bad response: "Trust me, Strategy is the right pattern here."

Good response: "Good question -- let me justify. Without Strategy, we'd need if-else chains in the pricing logic that grow with every new pricing rule. With Strategy, adding holiday pricing is one new class that implements the interface. The trade-off is an extra level of indirection, but I think it's worth it given that pricing rules are the most likely change vector. What are your thoughts?"

Scenario: You realize the interviewer is right

Bad response: Silently changing your approach without acknowledging.

Good response: "Actually, you're right -- I was over-complicating this. A simple enum with a method is sufficient here since we only have three types and they're unlikely to change. Let me simplify."

Changing your mind when presented with a good argument is a strength, not a weakness. It signals intellectual honesty.

Staff vs Senior: The Key Difference

A senior engineer solves the problem correctly. A staff engineer solves it correctly while also:

  • Driving the conversation -- You lead, not follow. You set scope, propose trade-offs, and proactively identify risks.
  • Showing production awareness -- Concurrency, failure modes, monitoring hooks, and operational concerns come up naturally.
  • Demonstrating design judgment -- You choose the simplest solution that works, not the most impressive one. You say "We don't need X here because..." which is harder than adding X.
  • Communicating like a tech lead -- You explain decisions as if onboarding a team member. "I chose this approach because..." is the most powerful phrase in an LLD interview.

Self-Check Quiz

Test Your Understanding (6 Questions)
  1. You're designing a movie ticket booking system and realize at the 20-minute mark that you haven't addressed concurrent seat selection. What do you do?
    Answer: Verbalize the issue immediately: "I notice I haven't addressed concurrency yet, and that's the hardest part of this problem. Let me add a SeatLockManager with temporary locks and CAS operations." The worst thing you can do is ignore it. Acknowledging and correcting shows self-awareness.
  2. The interviewer asks you to add a "waitlist" feature for sold-out shows. How does this test staff-level thinking?
    Answer: This tests whether your design is extensible. A staff answer uses Observer pattern: when a booking is cancelled, the Show emits a SeatsAvailableEvent. A WaitlistService subscribes, checks the waitlist queue, and offers seats to the next user with a time-limited lock. The key insight is that your core BookingService should not change.
  3. In the Online Bookstore problem, why is rollback logic in OrderService critical, and what pattern does it follow?
    Answer: It follows the Saga pattern (compensating transactions). If payment fails after inventory is reserved, we must release the reservation. Without rollback, "phantom reservations" permanently reduce available stock. The try-finally block ensures rollback happens even on unexpected exceptions.
  4. What's wrong with using synchronized on the entire bookSeat() method in a movie booking system?
    Answer: It creates a global bottleneck -- only one user can book any seat at a time, even for different shows. Fine-grained locking (per-seat or per-show) allows concurrent bookings for different shows. Use ConcurrentHashMap with putIfAbsent or per-seat ReentrantLock.
  5. How would you explain the difference between your Social Media Feed's ChronologicalFeedStrategy and RankedFeedStrategy to a non-technical stakeholder?
    Answer: "Chronological shows posts in the order they were created -- newest first, like a timeline. Ranked uses a scoring formula that considers how recent a post is, how many people engaged with it, and how close you are to the author. Think of it as showing you the 'best' posts instead of the 'newest' posts." Staff engineers must translate technical decisions into business language.
  6. An interviewer suggests using inheritance (BookWithImages extends Book, BookWithAudio extends Book) for different media types. You think composition is better. How do you push back?
    Answer: Use the ACE framework. "Inheritance could work if the types are fixed (Acknowledge). My concern is that a book might have both images and audio, and Java doesn't support multiple inheritance -- we'd get a combinatorial explosion of subclasses (Contrast). What if we used composition with a List<MediaAttachment> on Book? Each attachment could be ImageAttachment or AudioAttachment. This follows 'favor composition over inheritance' and handles any combination (Evaluate)."
