Module 10: Advanced OOP Concepts

Learning Objectives

  • Master Java generics including PECS, wildcards, and type erasure trade-offs
  • Use sealed classes and records (Java 17+) to model constrained domain hierarchies
  • Apply pattern matching with instanceof, switch expressions, and guarded patterns
  • Compose behavior using functional interfaces and method references
  • Create and process custom annotations for cross-cutting concerns

  • Articulate staff-level depth: WHY these features exist, not just HOW to use them

Why Advanced OOP for Staff Level

Senior engineers know the syntax. Staff engineers understand the type theory behind the syntax, explain trade-offs to their team, and choose the right abstraction for the next five years of code evolution. In an interview, demonstrating command of advanced OOP signals that you can:

Design Expressive APIs

Generics and sealed types let you encode invariants into the type system so the compiler catches bugs before code review does. Google's Guava and the JDK's Stream API are masterclasses in this.

Reduce Boilerplate Safely

Records, pattern matching, and functional interfaces eliminate defensive code without sacrificing type safety. Netflix reduced their DTO layer by 40% migrating to records.

Mentor Effectively

1. Generics Deep Dive

Generics - Type Safety Without Sacrifice ▼

Why Generics Exist: Before Java 5, collections stored Object. Every retrieval required a cast, and every cast was a potential ClassCastException at runtime. Generics move this check to compile time. But the implementation via type erasure introduces subtleties that every staff engineer must understand.

The PECS Principle (Producer Extends, Consumer Super)

PECS is the single most important rule for writing generic APIs. It answers: "When do I use ? extends T vs ? super T?"

  • Producer (reads data out): Use ? extends T — the collection produces items of type T (or subtypes). You can read from it, but not write to it.
  • Consumer (writes data in): Use ? super T — the collection consumes items of type T (or supertypes). You can write T into it, but reads give you only Object.
// PECS in action: a utility that copies elements from a producer to a consumer
public class Collections {

    // 'src' is a PRODUCER of T (we read from it) -> ? extends T
    // 'dest' is a CONSUMER of T (we write to it) -> ? super T
    public static <T> void copy(
            List<? super T> dest,
            List<? extends T> src) {
        for (int i = 0; i < src.size(); i++) {
            dest.set(i, src.get(i));
        }
    }
}

// Why this matters: maximum flexibility for callers
List<Number> numbers = new ArrayList<>(List.of(0, 0, 0));
List<Integer> ints = List.of(1, 2, 3);

// Works! Integer extends Number, Number supers Integer
Collections.copy(numbers, ints);

// Without PECS, the signature would be:
//   copy(List<T> dest, List<T> src)
// And this call would FAIL because List<Number> != List<Integer>

Wildcards in Practice

// Unbounded wildcard: "I don't care about the type"
// Useful for methods that only use Object methods
public static void printAll(List<?> items) {
    for (Object item : items) {
        System.out.println(item);  // toString() is on Object
    }
}

// Upper-bounded: "any Number or subtype"
public static double sum(List<? extends Number> numbers) {
    double total = 0;
    for (Number n : numbers) {
        total += n.doubleValue();
    }
    return total;
    // Note: you CANNOT do numbers.add(42) here!
    // The compiler doesn't know if it's List<Double> or List<Integer>
}

// Lower-bounded: "any supertype of Integer"
public static void addIntegers(List<? super Integer> list) {
    list.add(1);
    list.add(2);
    list.add(3);
    // Reading gives Object: Object item = list.get(0);
}

// Real-world: Comparator uses PECS
// Comparator<? super T> because it CONSUMES T values to compare
public static <T> T max(Collection<? extends T> coll,
                          Comparator<? super T> comp) {
    T best = null;
    for (T item : coll) {
        if (best == null || comp.compare(item, best) > 0) {
            best = item;
        }
    }
    return best;
}

Type Erasure: What the JVM Actually Sees

// What you write:
public class Box<T> {
    private T value;
    public T getValue() { return value; }
    public void setValue(T value) { this.value = value; }
}

// What the JVM sees after erasure:
public class Box {
    private Object value;
    public Object getValue() { return value; }
    public void setValue(Object value) { this.value = value; }
}

// Consequences of type erasure:
// 1. Cannot use instanceof with generics
//    if (obj instanceof List<String>) { }  // COMPILE ERROR

// 2. Cannot create generic arrays
//    T[] arr = new T[10];  // COMPILE ERROR
//    Workaround:
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[10];

// 3. Cannot overload on generic type alone
//    void process(List<String> s) { }
//    void process(List<Integer> i) { }  // COMPILE ERROR: same erasure

// 4. Bridge methods: compiler generates synthetic methods
public class StringBox extends Box<String> {
    @Override
    public String getValue() { return super.getValue(); }
    // Compiler also generates:
    // public Object getValue() { return getValue(); }  // bridge method
}

Reified Generics Discussion

Why Java chose erasure: Backward compatibility. Java 5 generics had to work with pre-generics bytecode. C# took a different path with reified generics (type info preserved at runtime) because .NET 2.0 could break backward compatibility.

Modern workaround — type tokens:

// Type token pattern (used by Jackson, Gson, Spring)
public abstract class TypeReference<T> {
    private final Type type;

    protected TypeReference() {
        // Captures generic type via anonymous subclass
        Type superclass = getClass().getGenericSuperclass();
        this.type = ((ParameterizedType) superclass).getActualTypeArguments()[0];
    }

    public Type getType() { return type; }
}

// Usage: Jackson deserialization
List<User> users = objectMapper.readValue(json,
    new TypeReference<List<User>>() {});
// The anonymous class preserves List<User> in its superclass metadata
When NOT to Use Complex Generics:
  • If the method only works with one concrete type, don't make it generic for "future flexibility"
  • If the wildcard nesting goes deeper than two levels (Map<String, List<? extends Comparable<? super T>>>), simplify with a named type alias or intermediate type
  • If your team struggles to read the signature, the abstraction is too clever
Interview Tip: When asked about generics, start with the problem they solve (type safety without casts), then explain PECS with a concrete example like Collections.copy. Mentioning type erasure and its consequences unprompted signals depth. If you can compare Java's erasure approach with C#'s reified generics, you demonstrate breadth.

2. Sealed Classes and Records (Java 17+)

Sealed Classes & Records - Constrained Domain Modeling ▼

Why Sealed Classes: Before sealed, Java had two extremes: final (no subclasses) or open (anyone can subclass). Sealed classes let you declare a closed set of permitted subtypes, enabling exhaustive pattern matching and making illegal states unrepresentable.

Sealed Class Fundamentals

// A payment can ONLY be one of these three types
public sealed interface Payment
    permits CreditCardPayment, BankTransferPayment, CryptoPayment {

    Money amount();
    String transactionId();
}

// Each permitted subtype must be final, sealed, or non-sealed
public record CreditCardPayment(
    Money amount,
    String transactionId,
    String cardLast4,
    String network   // VISA, MASTERCARD
) implements Payment {}

public record BankTransferPayment(
    Money amount,
    String transactionId,
    String bankCode,
    String accountNumber
) implements Payment {}

// non-sealed: allows further subclassing (escape hatch)
public non-sealed class CryptoPayment implements Payment {
    private final Money amount;
    private final String transactionId;
    private final String walletAddress;
    private final CryptoNetwork network;

    // constructor, getters...

    @Override
    public Money amount() { return amount; }

    @Override
    public String transactionId() { return transactionId; }
}

Records as Value Objects / DTOs

// Record = immutable data carrier with auto-generated
// equals(), hashCode(), toString(), accessors, constructor

public record UserDTO(
    String id,
    String name,
    String email,
    Instant createdAt
) {
    // Compact constructor for validation
    public UserDTO {
        Objects.requireNonNull(id, "id must not be null");
        Objects.requireNonNull(email, "email must not be null");
        if (!email.contains("@")) {
            throw new IllegalArgumentException("Invalid email: " + email);
        }
        // Fields are auto-assigned after compact constructor
    }

    // You can add derived methods
    public String displayName() {
        return name != null ? name : email.split("@")[0];
    }
}

// Usage
var user = new UserDTO("u123", "Alice", "alice@example.com", Instant.now());
// user.name() returns "Alice" (accessor, not getName())
// Immutable: no setters, fields are final

Records vs Builders

// Problem: records require ALL fields in constructor
// For 10+ fields, this gets unwieldy
var config = new ServerConfig("localhost", 8080, true, 100,
    Duration.ofSeconds(30), "TLSv1.3", null, true, 4, "INFO");
// Unreadable! Which boolean is which?

// Solution 1: Record + Builder (best of both worlds)
public record ServerConfig(
    String host,
    int port,
    boolean ssl,
    int maxConnections,
    Duration timeout,
    String tlsVersion,
    String certPath,
    boolean http2,
    int workerThreads,
    String logLevel
) {
    // Nested builder inside the record
    public static Builder builder() {
        return new Builder();
    }

    public static class Builder {
        private String host = "localhost";
        private int port = 8080;
        private boolean ssl = false;
        private int maxConnections = 100;
        private Duration timeout = Duration.ofSeconds(30);
        private String tlsVersion = "TLSv1.3";
        private String certPath;
        private boolean http2 = false;
        private int workerThreads = Runtime.getRuntime().availableProcessors();
        private String logLevel = "INFO";

        public Builder host(String host) { this.host = host; return this; }
        public Builder port(int port) { this.port = port; return this; }
        public Builder ssl(boolean ssl) { this.ssl = ssl; return this; }
        // ... more setters

        public ServerConfig build() {
            return new ServerConfig(host, port, ssl, maxConnections,
                timeout, tlsVersion, certPath, http2, workerThreads, logLevel);
        }
    }
}

// Clean usage:
var config = ServerConfig.builder()
    .host("api.example.com")
    .port(443)
    .ssl(true)
    .http2(true)
    .build();
When NOT to Use Records:
  • When you need mutable state (records are immutable by design)
  • When you need inheritance (records are implicitly final and extend java.lang.Record)
  • When you need to control the field storage layout (e.g., lazy initialization)
  • For JPA entities — Hibernate needs a no-arg constructor and mutable fields
Interview Tip: When modeling a domain in an LLD interview, use sealed interfaces for variant types (payment methods, notification channels, game piece types). This shows the interviewer you understand algebraic data types and makes your design self-documenting. Mention that sealed + records give you something close to Kotlin's sealed classes or Rust's enums.

3. Pattern Matching

Pattern Matching - Expressive Control Flow ▼

Why Pattern Matching: Traditional Java code is littered with instanceof checks followed by casts. Pattern matching eliminates the redundancy, reduces errors, and combined with sealed types creates exhaustive, compiler-verified branching. This is the Java equivalent of Scala's match or Rust's match.

instanceof Pattern Matching (Java 16+)

// Before: verbose and error-prone
public String format(Object obj) {
    if (obj instanceof String) {
        String s = (String) obj;  // redundant cast
        return "String[" + s.length() + "]: " + s;
    } else if (obj instanceof Integer) {
        Integer i = (Integer) obj;
        return "Integer: " + i;
    } else if (obj instanceof List) {
        List<?> list = (List<?>) obj;
        return "List[" + list.size() + "]";
    }
    return obj.toString();
}

// After: pattern matching binds the variable in one step
public String format(Object obj) {
    if (obj instanceof String s) {
        return "String[" + s.length() + "]: " + s;
    } else if (obj instanceof Integer i) {
        return "Integer: " + i;
    } else if (obj instanceof List<?> list) {
        return "List[" + list.size() + "]";
    }
    return obj.toString();
}

// The binding variable's scope is flow-sensitive
if (!(obj instanceof String s)) {
    // s is NOT in scope here
    throw new IllegalArgumentException("Expected String");
}
// s IS in scope here because we only reach this point if instanceof was true
return s.toUpperCase();

Switch Expressions with Patterns (Java 21+)

// Combining sealed types + switch + pattern matching
public sealed interface Shape
    permits Circle, Rectangle, Triangle {}

public record Circle(double radius) implements Shape {}
public record Rectangle(double width, double height) implements Shape {}
public record Triangle(double base, double height) implements Shape {}

// Exhaustive switch - compiler verifies all cases are covered
public double area(Shape shape) {
    return switch (shape) {
        case Circle c     -> Math.PI * c.radius() * c.radius();
        case Rectangle r  -> r.width() * r.height();
        case Triangle t   -> 0.5 * t.base() * t.height();
        // No default needed! Sealed type guarantees exhaustiveness
    };
}

// If you add a new Shape subtype, the compiler forces you to update
// every switch that uses it. This is enormously valuable at scale.

Guarded Patterns

// Add conditions to pattern cases with 'when' (Java 21+)
public String classifyShape(Shape shape) {
    return switch (shape) {
        case Circle c when c.radius() > 100
            -> "Large circle (r=" + c.radius() + ")";
        case Circle c when c.radius() > 10
            -> "Medium circle (r=" + c.radius() + ")";
        case Circle c
            -> "Small circle (r=" + c.radius() + ")";
        case Rectangle r when r.width() == r.height()
            -> "Square (" + r.width() + "x" + r.height() + ")";
        case Rectangle r
            -> "Rectangle (" + r.width() + "x" + r.height() + ")";
        case Triangle t
            -> "Triangle (base=" + t.base() + ")";
    };
}

// Real-world: processing domain events
public void handleEvent(DomainEvent event) {
    switch (event) {
        case OrderCreated e when e.totalAmount().isGreaterThan(Money.of(10000))
            -> flagForFraudReview(e);
        case OrderCreated e
            -> processNormally(e);
        case PaymentReceived e when e.method() instanceof CryptoPayment
            -> processWithExtraVerification(e);
        case PaymentReceived e
            -> recordPayment(e);
        case OrderCancelled e
            -> initiateRefund(e);
    }
}

with sealed types] D --> G G --> H[No runtime ClassCastException possible]
When NOT to Use Pattern Matching:
  • When polymorphism (virtual dispatch) is a better fit — if every type has its own area() method, let them implement it instead of a central switch
  • When the set of types is open-ended — pattern matching shines with sealed types; without them, you always need a default case
  • When the guard conditions become complex — extract them into named methods for readability
Interview Tip: If you're solving an LLD problem with a finite set of types (piece types in chess, vehicle types in a parking lot), model them as a sealed hierarchy and use switch expressions. The interviewer sees that you know modern Java and that your design prevents invalid states. Always mention the tradeoff: adding a new variant requires updating all switches (good for closed domains, bad for open ones).

4. Functional Interfaces

Functional Interfaces - Composable Behavior ▼

Why Functional Interfaces: Java 8 introduced lambdas, but they need a type to target. A functional interface is any interface with exactly one abstract method, and the java.util.function package provides the standard ones. Mastering composition (andThen, compose) lets you build processing pipelines without class explosion.

The Core Four

import java.util.function.*;

// 1. Predicate<T>: T -> boolean (testing/filtering)
Predicate<String> isNotBlank = s -> s != null && !s.isBlank();
Predicate<String> isShort = s -> s.length() < 10;

// Composition: and, or, negate
Predicate<String> isValidTag = isNotBlank.and(isShort);
List<String> tags = List.of("java", "", "microservices", "a-very-long-tag-name");
List<String> valid = tags.stream().filter(isValidTag).toList();
// ["java", "microservices"]

// 2. Function<T, R>: T -> R (transformation)
Function<String, Integer> length = String::length;
Function<Integer, String> stars = n -> "*".repeat(n);

// Composition: andThen (left to right), compose (right to left)
Function<String, String> lengthAsStars = length.andThen(stars);
// "hello" -> 5 -> "*****"

Function<String, String> sameResult = stars.compose(length);
// Same pipeline, different reading direction

// 3. Consumer<T>: T -> void (side effects)
Consumer<User> sendWelcomeEmail = user ->
    emailService.send(user.email(), "Welcome!");
Consumer<User> logRegistration = user ->
    logger.info("New user: {}", user.id());

// Composition: andThen (chain side effects)
Consumer<User> onUserRegistered = sendWelcomeEmail.andThen(logRegistration);
onUserRegistered.accept(newUser);  // sends email THEN logs

// 4. Supplier<T>: () -> T (lazy creation)
Supplier<Connection> connectionFactory = () ->
    DriverManager.getConnection(url, user, pass);

// Useful for deferred/lazy evaluation
Supplier<ExpensiveReport> lazyReport = () -> generateReport();
// Report is NOT generated until lazyReport.get() is called

Beyond the Core Four

// BiFunction<T, U, R>: (T, U) -> R
BiFunction<String, Integer, String> repeat = String::repeat;
// "ha".repeat(3) -> "hahaha"

// UnaryOperator<T>: T -> T (special case of Function<T, T>)
UnaryOperator<String> trim = String::trim;
UnaryOperator<String> lower = String::toLowerCase;
UnaryOperator<String> normalize = trim.andThen(lower)::apply;
// or: Function.identity() returns a no-op UnaryOperator

// BinaryOperator<T>: (T, T) -> T
BinaryOperator<Integer> max = Integer::max;
int result = max.apply(3, 7);  // 7

// Real-world: building a validation pipeline
public class Validator<T> {
    private final List<Predicate<T>> rules = new ArrayList<>();
    private final List<String> ruleNames = new ArrayList<>();

    public Validator<T> addRule(String name, Predicate<T> rule) {
        rules.add(rule);
        ruleNames.add(name);
        return this;
    }

    public ValidationResult validate(T value) {
        List<String> failures = new ArrayList<>();
        for (int i = 0; i < rules.size(); i++) {
            if (!rules.get(i).test(value)) {
                failures.add(ruleNames.get(i));
            }
        }
        return new ValidationResult(failures.isEmpty(), failures);
    }
}

// Usage
var userValidator = new Validator<UserDTO>()
    .addRule("name required", u -> u.name() != null && !u.name().isBlank())
    .addRule("valid email", u -> u.email().contains("@"))
    .addRule("age >= 18", u -> u.age() >= 18);

ValidationResult result = userValidator.validate(newUser);
When NOT to Use Functional Interfaces:
  • When the lambda body exceeds 3 lines — extract to a named method or class
  • When you need state or multiple methods — use a proper interface/class
  • When readability suffers from deeply nested composition — a simple for loop may be clearer
  • When debugging — stack traces through lambdas are harder to read than named methods
Interview Tip: In an LLD interview, use functional interfaces for strategy-like behaviors where a full Strategy class would be overkill. For example, a sorting comparator or a filter predicate. Show andThen composition to demonstrate you understand pipelines. But don't force lambdas everywhere — show judgment about when a named class is better.

5. Method References

Method References - Concise Lambda Syntax ▼

Why Method References: When a lambda simply delegates to an existing method, a method reference is more readable and signals intent directly. There are exactly four forms, and knowing all four is a signal of Java fluency.

The Four Forms

import java.util.*;
import java.util.stream.*;

// 1. Reference to a STATIC method: ClassName::staticMethod
//    Lambda equivalent: (args) -> ClassName.staticMethod(args)
Function<String, Integer> parse = Integer::parseInt;
// same as: s -> Integer.parseInt(s)

List<String> numStrs = List.of("1", "2", "3");
List<Integer> nums = numStrs.stream()
    .map(Integer::parseInt)  // static method ref
    .toList();

// 2. Reference to an instance method of a PARTICULAR object: instance::method
//    Lambda equivalent: (args) -> instance.method(args)
String prefix = "Hello, ";
Function<String, String> greeter = prefix::concat;
// same as: name -> prefix.concat(name)

var logger = LoggerFactory.getLogger(MyClass.class);
Consumer<String> logInfo = logger::info;
// same as: msg -> logger.info(msg)

// 3. Reference to an instance method of an ARBITRARY object of a type:
//    ClassName::instanceMethod
//    Lambda equivalent: (obj, args) -> obj.instanceMethod(args)
Function<String, String> upper = String::toUpperCase;
// same as: s -> s.toUpperCase()

// Used in stream operations where the stream element IS the receiver
List<String> names = List.of("alice", "bob", "charlie");
List<String> upperNames = names.stream()
    .map(String::toUpperCase)     // arbitrary object ref
    .toList();

// Sorting with arbitrary instance method ref
List<String> sorted = names.stream()
    .sorted(String::compareToIgnoreCase)  // BiFunction form
    .toList();

// 4. Reference to a CONSTRUCTOR: ClassName::new
//    Lambda equivalent: (args) -> new ClassName(args)
Supplier<ArrayList<String>> listFactory = ArrayList::new;
// same as: () -> new ArrayList<>()

Function<String, File> fileCreator = File::new;
// same as: path -> new File(path)

// Array constructor reference
IntFunction<String[]> arrayFactory = String[]::new;
String[] arr = arrayFactory.apply(5);  // new String[5]

// Real-world: converting DTOs
record UserEntity(Long id, String name, String email) {}
record UserDTO(String name, String email) {
    static UserDTO fromEntity(UserEntity entity) {
        return new UserDTO(entity.name(), entity.email());
    }
}

List<UserEntity> entities = repository.findAll();
List<UserDTO> dtos = entities.stream()
    .map(UserDTO::fromEntity)   // static method ref
    .toList();

Lambda Best Practices

// DO: Prefer method references when they're equally clear
list.forEach(System.out::println);        // clear
list.stream().map(String::toLowerCase);   // clear

// DON'T: Use method ref when lambda is clearer
// Method ref: ambiguous which overload
list.stream().map(String::valueOf);
// Lambda: explicit about parameter
list.stream().map(s -> String.valueOf(s));

// DO: Keep lambdas short (1-3 lines)
users.stream()
    .filter(u -> u.isActive() && u.age() >= 18)
    .map(User::email)
    .toList();

// DON'T: Multi-line lambdas in streams
users.stream().map(u -> {
    var address = addressService.lookup(u.id());
    var preferences = prefService.get(u.id());
    return new UserProfile(u, address, preferences);
}).toList();

// DO: Extract to a method instead
users.stream()
    .map(this::buildUserProfile)  // instance method ref
    .toList();

private UserProfile buildUserProfile(User user) {
    var address = addressService.lookup(user.id());
    var preferences = prefService.get(user.id());
    return new UserProfile(user, address, preferences);
}
Interview Tip: Use method references in your LLD code to show fluency. If the interviewer asks you to explain, describe the four forms with a brief example of each. The "arbitrary instance" form (Form 3) trips up most candidates — being able to explain it clearly is a differentiator.

6. Custom Annotations

Custom Annotations - Declarative Metadata ▼

Why Custom Annotations: Annotations let you attach metadata to code elements and process it at compile time or runtime. Frameworks like Spring, JPA, and Jackson are built on annotations. Understanding how to create them (not just consume them) is a staff-level skill because it unlocks building internal frameworks and reducing boilerplate for your team.

Retention Policies

flowchart LR A[Annotation Defined] --> B{Retention Policy} B -->|SOURCE| C["Discarded by compiler
(e.g., @Override, @SuppressWarnings)"] B -->|CLASS| D["In .class file but NOT
available at runtime
(default if unspecified)"] B -->|RUNTIME| E["Available via reflection

Creating Custom Annotations

import java.lang.annotation.*;

// Validation annotation
@Target(ElementType.FIELD)          // Can only be applied to fields
@Retention(RetentionPolicy.RUNTIME) // Available at runtime via reflection
@Documented                         // Included in Javadoc
public @interface NotBlank {
    String message() default "must not be blank";
}

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Range {
    int min() default 0;
    int max() default Integer.MAX_VALUE;
    String message() default "out of range";
}

@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface Email {
    String message() default "invalid email format";
}

// Applying annotations
public class CreateUserRequest {
    @NotBlank(message = "Username is required")
    private String username;

    @Email
    @NotBlank
    private String email;

    @Range(min = 18, max = 150, message = "Age must be between 18 and 150")
    private int age;

    // constructors, getters...
}

Annotation Processing at Runtime

// A simple validation framework using reflection
public class AnnotationValidator {

    public static List<String> validate(Object obj) {
        List<String> errors = new ArrayList<>();
        Class<?> clazz = obj.getClass();

        for (Field field : clazz.getDeclaredFields()) {
            field.setAccessible(true);

            try {
                Object value = field.get(obj);

                // Process @NotBlank
                if (field.isAnnotationPresent(NotBlank.class)) {
                    NotBlank ann = field.getAnnotation(NotBlank.class);
                    if (value == null || (value instanceof String s && s.isBlank())) {
                        errors.add(field.getName() + ": " + ann.message());
                    }
                }

                // Process @Range
                if (field.isAnnotationPresent(Range.class)) {
                    Range ann = field.getAnnotation(Range.class);
                    if (value instanceof Number num) {
                        int val = num.intValue();
                        if (val < ann.min() || val > ann.max()) {
                            errors.add(field.getName() + ": " + ann.message());
                        }
                    }
                }

                // Process @Email
                if (field.isAnnotationPresent(Email.class)) {
                    Email ann = field.getAnnotation(Email.class);
                    if (value instanceof String s && !s.matches("^[\\w.-]+@[\\w.-]+\\.[a-zA-Z]{2,}$")) {
                        errors.add(field.getName() + ": " + ann.message());
                    }
                }

            } catch (IllegalAccessException e) {
                throw new RuntimeException("Validation failed for field: " + field.getName(), e);
            }
        }
        return errors;
    }
}

// Usage
var request = new CreateUserRequest("", "not-an-email", 15);

// ["username: Username is required", "email: invalid email format",
//  "age: Age must be between 18 and 150"]

7. Covariance vs Contravariance

Variance - The Theory Behind PECS ▼

Why Variance Matters: Variance answers the question: "If Dog is a subtype of Animal, is List<Dog> a subtype of List<Animal>?" The answer depends on whether the generic type is covariant, contravariant, or invariant. Getting this wrong causes ClassCastException at runtime or overly restrictive APIs.

flowchart TD subgraph "Type Hierarchy" N[Number] --> I[Integer] N --> D[Double] end subgraph "Covariant (? extends Number)" direction TB LN1["List<Number>"] -.-> LI1["List<Integer> is subtype"] LN1 -.-> LD1["List<Double> is subtype"] LI1 -.- R1["Read OK, Write BLOCKED"] end subgraph "Contravariant (? super Integer)" direction TB LI2["List<Integer>"] -.-> LN2["List<Number> is subtype"] LI2 -.-> LO2["List<Object> is subtype"] LN2 -.- R2["Write OK, Read gives Object"] end

Arrays vs Generics: A Cautionary Tale

// Java arrays are COVARIANT (and this was a design mistake)
Number[] numbers = new Integer[5];  // Compiles! Integer[] is a subtype of Number[]
numbers[0] = 3.14;  // Compiles! But throws ArrayStoreException at RUNTIME

// Java generics are INVARIANT by default (fixed the mistake)
// List<Number> numbers = new ArrayList<Integer>();  // COMPILE ERROR!
// This prevents the equivalent of the ArrayStoreException

// To get covariance/contravariance with generics, you use wildcards explicitly
List<? extends Number> covariant = new ArrayList<Integer>();  // OK
List<? super Integer> contravariant = new ArrayList<Number>(); // OK

Covariance (? extends T) - For Reading

// Covariant: "I produce values of type T or its subtypes"
// Safe for reading, unsafe for writing

public static double sumAll(List<? extends Number> numbers) {
    double sum = 0;
    for (Number n : numbers) {     // READ is safe
        sum += n.doubleValue();
    }
    // numbers.add(42);            // COMPILE ERROR: cannot write
    // numbers.add(3.14);          // COMPILE ERROR: cannot write
    // Why? If it's a List<Double>, adding Integer would corrupt it
    return sum;
}

// Works with any List of Numbers or subtypes
sumAll(List.of(1, 2, 3));              // List<Integer>
sumAll(List.of(1.0, 2.0, 3.0));       // List<Double>
sumAll(List.of(1L, 2L, 3L));          // List<Long>

Contravariance (? super T) - For Writing

// Contravariant: "I consume values of type T or its subtypes"
// Safe for writing T, unsafe for specific reading

public static void fillWithIntegers(List<? super Integer> list, int count) {
    for (int i = 0; i < count; i++) {
        list.add(i);                  // WRITE is safe
    }
    // Integer n = list.get(0);      // COMPILE ERROR: can only read as Object
    Object obj = list.get(0);        // OK but loses type info
}

// Works with any List that can hold Integers
List<Integer> ints = new ArrayList<>();
List<Number> nums = new ArrayList<>();
List<Object> objs = new ArrayList<>();

fillWithIntegers(ints, 3);   // OK
fillWithIntegers(nums, 3);   // OK
fillWithIntegers(objs, 3);   // OK

Putting It All Together: A Real API Design

// Designing a type-safe event system using variance
public class EventBus {

    // Map from event type to list of handlers
    // Handler<? super T> means: a handler for T or any supertype of T
    private final Map<Class<?>, List<Consumer<?>>> handlers = new ConcurrentHashMap<>();

    // Register: handler consumes events of type T (contravariant)
    public <T> void subscribe(Class<T> eventType, Consumer<? super T> handler) {
        handlers.computeIfAbsent(eventType, k -> new CopyOnWriteArrayList<>())
                .add(handler);
    }

    // Publish: event is produced (covariant concept)
    @SuppressWarnings("unchecked")
    public <T> void publish(T event) {
        Class<?> eventType = event.getClass();

        // Walk up the class hierarchy to find handlers
        while (eventType != null) {
            List<Consumer<?>> eventHandlers = handlers.get(eventType);
            if (eventHandlers != null) {
                for (Consumer<?> handler : eventHandlers) {
                    ((Consumer<T>) handler).accept(event);
                }
            }
            eventType = eventType.getSuperclass();
        }
    }
}

When NOT to Worry About Variance:
  • Internal implementation code that doesn't expose a public API — just use concrete types
  • When the type parameter is both read and written in the same method — it must be invariant
  • When the team finds the wildcards confusing — sometimes a less flexible but more readable signature is the right call
Interview Tip: If you get asked about variance, draw the type hierarchy on the whiteboard and walk through a concrete example: "If I have List<? extends Number>, it might be a List<Integer> or a List<Double>. If I could add an Integer, I might corrupt a List of Doubles. That's why the compiler blocks writes." This intuitive explanation is far more impressive than reciting the rule.

# Question Answer
1 What does PECS stand for, and what problem does it solve? Producer Extends, Consumer Super. It determines when to use ? extends T (reading/producing) vs ? super T (writing/consuming) for maximum API flexibility.
2 Why can't you add an Integer to a List<? extends Number>? Because the actual list might be a List<Double>. The compiler prevents writes to covariant collections to avoid type corruption at runtime.

4 Name the four forms of Java method references. 1) Static: ClassName::staticMethod, 2) Instance of particular object: instance::method, 3) Instance of arbitrary object: ClassName::instanceMethod, 4) Constructor: ClassName::new.

public class TypeSafeEventBus {
    private final Map<Class<?>, List<Consumer<?>>> handlers = new ConcurrentHashMap<>();

    /**
     * Subscribe a handler for a specific event type.
     * Consumer<? super T> allows handlers for supertypes (e.g.,
     * a Consumer<AppEvent> can handle OrderPlaced events).
     */
    public <T extends AppEvent> void subscribe(Class<T> eventType, Consumer<? super T> handler) {
        handlers.computeIfAbsent(eventType, k -> new CopyOnWriteArrayList<>())
                .add(handler);
    }

    /**
     * Publish an event. Walks the class hierarchy to invoke
     * handlers registered for the event's type AND its supertypes.
     */
    @SuppressWarnings("unchecked")
    public <T extends AppEvent> void publish(T event) {
        Objects.requireNonNull(event, "event must not be null");

        Class<?> type = event.getClass();

        // Walk up the type hierarchy: OrderPlaced -> AppEvent -> Object
        while (type != null) {
            List<Consumer<?>> typeHandlers = handlers.get(type);
            if (typeHandlers != null) {
                for (Consumer<?> handler : typeHandlers) {
                    ((Consumer<T>) handler).accept(event);
                }
            }

            // Also check interfaces (for sealed interface hierarchies)
            for (Class<?> iface : type.getInterfaces()) {
                List<Consumer<?>> ifaceHandlers = handlers.get(iface);
                if (ifaceHandlers != null) {
                    for (Consumer<?> handler : ifaceHandlers) {
                        ((Consumer<T>) handler).accept(event);
                    }
                }
            }

            type = type.getSuperclass();
        }
    }

    /** Returns the count of handlers for a given event type (useful for testing). */
    public int handlerCount(Class<? extends AppEvent> eventType) {
        return handlers.getOrDefault(eventType, List.of()).size();
    }
}
