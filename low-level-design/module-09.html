Module 9: Testing & TDD

Learning Objectives

  • Master JUnit 5 features used in real production codebases: parameterized tests, nested classes, extensions
  • Use Mockito effectively, knowing when to mock and when not to
  • Understand the test pyramid and what belongs at each level
  • Practice the TDD Red-Green-Refactor cycle through a complete worked example
  • Test concurrent code safely using CountDownLatch and Awaitility
  • Design code for testability from the start using dependency injection and interface extraction
  • Articulate testing strategy at a staff-engineer level in interviews

Why Testing Matters for Staff Engineers

Beyond "Write More Tests" — Strategic Testing ▼

At the staff level, testing is not about code coverage metrics. It is about confidence in change. A well-tested system lets your organization ship faster because engineers can refactor, extend, and deploy without fear. A poorly-tested system calcifies — every change carries risk, so teams slow down.

Why interviewers ask about testing:

  • Signal for production-readiness: Candidates who mention testing organically during LLD problems show they think about real systems, not just happy paths
  • Design quality indicator: Code that is hard to test is almost always poorly designed. If you can't test it, it has too many responsibilities or hidden dependencies
  • Staff-level influence: Staff engineers set testing standards for their teams. Interviewers want to see you can articulate what to test, how much, and why

Real-world impact: Google's engineering practices research found that teams with high test coverage experienced 40% fewer production incidents and were able to deploy 2x more frequently. At Amazon, the "two-pizza team" model works precisely because automated tests serve as contracts between services, replacing the need for manual coordination.

When NOT to test obsessively:
  • Prototypes and throwaway spikes — testing slows exploration
  • Pure data classes (records/DTOs) with no logic
  • Trivial getters/setters generated by Lombok or similar
  • Third-party library internals — trust their tests, verify your integration

The goal is meaningful coverage, not 100% line coverage. Test behavior, not implementation details.

1. JUnit 5 Deep Dive

JUnit 5 — Modern Testing Essentials ▼

Why JUnit 5 over JUnit 4: JUnit 5 (Jupiter) was a ground-up rewrite that introduced composable annotations, parameterized tests without runners, and an extension model that replaced the rigid @Rule/@RunWith system. Understanding these features lets you write expressive, maintainable tests.

@ParameterizedTest — Eliminate Repetition

Parameterized tests let you run the same test logic with different inputs. This is invaluable for boundary testing and equivalence partitioning.

import org.junit.jupiter.params.ParameterizedTest;
import org.junit.jupiter.params.provider.*;

class PasswordValidatorTest {

    private final PasswordValidator validator = new PasswordValidator();

    @ParameterizedTest(name = "Password \"{0}\" should be {1}")
    @CsvSource({
        "'Str0ng!Pass', true",
        "'short',       false",
        "'nouppercas3!', false",
        "'NOLOWERCASE3!', false",
        "'NoSpecialChar3', false",
        "'NoDigits!!aa', false",
        "'',            false"
    })
    void shouldValidatePasswordStrength(String password, boolean expected) {
        assertEquals(expected, validator.isStrong(password));
    }

    @ParameterizedTest
    @NullAndEmptySource
    @ValueSource(strings = {"   ", "\t", "\n"})
    void shouldRejectBlankPasswords(String password) {
        assertFalse(validator.isStrong(password));
    }

    // Custom argument provider for complex objects
    @ParameterizedTest
    @MethodSource("provideEdgeCasePasswords")
    void shouldHandleEdgeCases(String password, boolean expected, String reason) {
        assertEquals(expected, validator.isStrong(password),
            () -> "Failed for: " + reason);
    }

    static Stream<Arguments> provideEdgeCasePasswords() {
        return Stream.of(
            Arguments.of("A1!aaaaa", true, "minimum length exactly 8"),
            Arguments.of("A1!aaaa", false, "one char below minimum"),
            Arguments.of("A1!" + "a".repeat(128), true, "very long password"),
            Arguments.of("P@ss\u00E9w0rd", true, "unicode characters")
        );
    }
}

@Nested — Organize Tests by Context

Nested test classes create a BDD-style hierarchy that makes test intent clear. Each nested class represents a scenario or state.

@DisplayName("BankAccount")
class BankAccountTest {

    private BankAccount account;

    @BeforeEach
    void setUp() {
        account = new BankAccount("ACC-001", "Alice");
    }

    @Nested
    @DisplayName("when newly created")
    class WhenNew {

        @Test
        @DisplayName("has zero balance")
        void hasZeroBalance() {
            assertEquals(BigDecimal.ZERO, account.getBalance());
        }

        @Test
        @DisplayName("rejects withdrawal")
        void rejectsWithdrawal() {
            assertThrows(InsufficientFundsException.class,
                () -> account.withdraw(BigDecimal.ONE));
        }
    }

    @Nested
    @DisplayName("after depositing $100")
    class AfterDeposit {

        @BeforeEach
        void deposit() {
            account.deposit(new BigDecimal("100.00"));
        }

        @Test
        @DisplayName("has correct balance")
        void hasCorrectBalance() {
            assertEquals(new BigDecimal("100.00"), account.getBalance());
        }

        @Test
        @DisplayName("allows partial withdrawal")
        void allowsPartialWithdrawal() {
            account.withdraw(new BigDecimal("40.00"));
            assertEquals(new BigDecimal("60.00"), account.getBalance());
        }

        @Nested
        @DisplayName("and then withdrawing $60")
        class AfterWithdrawal {

            @BeforeEach
            void withdraw() {
                account.withdraw(new BigDecimal("60.00"));
            }

            @Test
            @DisplayName("has $40 remaining")
            void hasCorrectBalance() {
                assertEquals(new BigDecimal("40.00"), account.getBalance());
            }
        }
    }
}

assertAll — Grouped Assertions

assertAll runs every assertion even if one fails, giving you a complete picture of what broke instead of stopping at the first failure.

@Test
void shouldCreateUserWithAllFields() {
    User user = userService.createUser("alice", "alice@example.com", "Engineering");

    assertAll("user properties",
        () -> assertNotNull(user.getId(), "id should be generated"),
        () -> assertEquals("alice", user.getUsername()),
        () -> assertEquals("alice@example.com", user.getEmail()),
        () -> assertEquals("Engineering", user.getDepartment()),
        () -> assertNotNull(user.getCreatedAt(), "createdAt should be set"),
        () -> assertTrue(user.isActive(), "new users should be active")
    );
}

Custom Extensions

Extensions replace JUnit 4's @Rule and @RunWith. They let you inject behavior at any lifecycle point.

// Timing extension - logs slow tests
public class TimingExtension implements BeforeTestExecutionCallback,
                                        AfterTestExecutionCallback {

    private static final Logger log = LoggerFactory.getLogger(TimingExtension.class);

    @Override
    public void beforeTestExecution(ExtensionContext context) {
        getStore(context).put("start", System.currentTimeMillis());
    }

    @Override
    public void afterTestExecution(ExtensionContext context) {
        long start = getStore(context).remove("start", long.class);
        long duration = System.currentTimeMillis() - start;
        if (duration > 500) {
            log.warn("SLOW TEST: {} took {}ms",
                context.getDisplayName(), duration);
        }
    }

    private ExtensionContext.Store getStore(ExtensionContext context) {
        return context.getStore(
            ExtensionContext.Namespace.create(getClass(), context.getRequiredTestMethod()));
    }
}

// Usage
@ExtendWith(TimingExtension.class)
class PerformanceSensitiveTest {
    // All tests in this class will be timed
}
When NOT to use these features:
  • @ParameterizedTest — Don't use when each case needs different assertion logic; write separate tests instead
  • @Nested — Don't nest more than 3 levels deep; it becomes hard to trace which @BeforeEach applies
  • Custom extensions — Don't write one for something Mockito or Spring already handles
Interview Tip: When solving an LLD problem, mention "@ParameterizedTest for boundary validation" or "@Nested for state-based testing." It shows practical experience, not textbook knowledge.

2. Mockito Mastery

Mockito — Isolating Units Under Test ▼

Why mock: Mocks let you test a class in isolation by replacing its collaborators with controlled substitutes. This makes tests fast, deterministic, and focused on one behavior at a time.

Argument Captors — Verifying Complex Arguments

@ExtendWith(MockitoExtension.class)
class OrderServiceTest {

    @Mock private EmailService emailService;
    @Mock private OrderRepository orderRepository;
    @InjectMocks private OrderService orderService;

    @Captor private ArgumentCaptor<Email> emailCaptor;

    @Test
    void shouldSendConfirmationEmailOnOrderPlacement() {
        // Arrange
        Order order = new Order("ORD-123", "alice@example.com",
            List.of(new LineItem("Widget", 2, new BigDecimal("29.99"))));

        when(orderRepository.save(any(Order.class)))
            .thenReturn(order);

        // Act
        orderService.placeOrder(order);

        // Assert - capture the email that was sent
        verify(emailService).send(emailCaptor.capture());

        Email sentEmail = emailCaptor.getValue();
        assertAll(
            () -> assertEquals("alice@example.com", sentEmail.getTo()),
            () -> assertTrue(sentEmail.getSubject().contains("ORD-123")),
            () -> assertTrue(sentEmail.getBody().contains("Widget")),
            () -> assertTrue(sentEmail.getBody().contains("$59.98"))
        );
    }
}

Verify Interactions — Behavioral Assertions

@Test
void shouldRetryCacheOnFailure() {
    // Arrange
    when(cacheService.get("user:42"))
        .thenThrow(new CacheException("timeout"))  // first call fails
        .thenReturn(Optional.of(cachedUser));        // retry succeeds

    // Act
    User user = userService.getUser(42);

    // Assert
    verify(cacheService, times(2)).get("user:42");   // called exactly twice
    verify(databaseService, never()).findById(42);    // DB was NOT hit
    assertEquals("Alice", user.getName());
}

@Test
void shouldFallThroughToDatabaseWhenCacheMisses() {
    when(cacheService.get("user:42")).thenReturn(Optional.empty());
    when(databaseService.findById(42)).thenReturn(dbUser);

    userService.getUser(42);

    // Verify order of operations
    InOrder inOrder = inOrder(cacheService, databaseService, cacheService);
    inOrder.verify(cacheService).get("user:42");
    inOrder.verify(databaseService).findById(42);
    inOrder.verify(cacheService).put("user:42", dbUser);  // cache populated
}

Spy vs Mock

A mock is a complete substitute — all methods return defaults unless stubbed. A spy wraps a real object — unstubbed methods call the real implementation.

@Test
void spyExample() {
    // Spy wraps real ArrayList - add/size work normally
    List<String> spyList = spy(new ArrayList<>());
    spyList.add("one");
    spyList.add("two");
    assertEquals(2, spyList.size());  // real implementation

    // Override specific behavior
    doReturn(100).when(spyList).size();
    assertEquals(100, spyList.size()); // stubbed
    assertEquals("one", spyList.get(0)); // still real
}

// When to use spy: testing legacy code you can't refactor yet
@Test
void testLegacyServiceWithSpy() {
    LegacyPaymentGateway realGateway = new LegacyPaymentGateway();
    LegacyPaymentGateway spyGateway = spy(realGateway);

    // Override only the HTTP call, keep validation logic real
    doReturn(new PaymentResponse(true, "TXN-001"))
        .when(spyGateway).callRemoteApi(any());

    PaymentResult result = spyGateway.processPayment(validPayment);
    assertTrue(result.isSuccess());
}

@InjectMocks — Automatic Dependency Wiring

@ExtendWith(MockitoExtension.class)
class NotificationServiceTest {

    @Mock private EmailService emailService;
    @Mock private SmsService smsService;
    @Mock private PushNotificationService pushService;
    @Mock private UserPreferenceRepository prefRepo;

    // Mockito injects all @Mock fields that match constructor/field types
    @InjectMocks private NotificationService notificationService;

    @Test
    void shouldSendViaPreferredChannel() {
        when(prefRepo.getPreference("user-1"))
            .thenReturn(new UserPreference(Channel.SMS));

        notificationService.notify("user-1", "Your order shipped!");

        verify(smsService).send(eq("user-1"), contains("order shipped"));
        verify(emailService, never()).send(any(), any());
        verify(pushService, never()).send(any(), any());
    }
}
When NOT to Mock:
  • Value objects and data classes: Just use the real thing. Mocking BigDecimal or a record is absurd
  • Simple pure functions: If it has no side effects, call it directly
  • The class under test: Never mock what you are testing (use spy only as a last resort for legacy code)
  • Everything in integration tests: The whole point is to test components working together

Rule of thumb: Mock at architectural boundaries (network, database, external API). Do not mock internal domain logic.

Interview Tip: Interviewers love hearing "I would mock the repository here but use real domain objects." It shows you understand test isolation and its limits.

The Boundary Question: Unit vs Integration

// UNIT TEST - mock the repository, test the service logic
@Test
void shouldApplyDiscountForLoyalCustomers() {
    when(customerRepo.findById("C-1"))
        .thenReturn(Optional.of(new Customer("C-1", Tier.GOLD, 3)));

    OrderSummary summary = pricingService.calculatePrice(order, "C-1");

    assertEquals(new BigDecimal("85.00"), summary.getTotal()); // 15% discount

// INTEGRATION TEST - real database, test the query
@SpringBootTest
@Testcontainers
class CustomerRepositoryIntegrationTest {

    @Container
    static PostgreSQLContainer<?> postgres = new PostgreSQLContainer<>("postgres:15");

    @Autowired private CustomerRepository customerRepo;

    @Test
    void shouldFindCustomersByTier() {
        customerRepo.save(new Customer("C-1", Tier.GOLD, 3));
        customerRepo.save(new Customer("C-2", Tier.SILVER, 1));
        customerRepo.save(new Customer("C-3", Tier.GOLD, 5));

        List<Customer> goldCustomers = customerRepo.findByTier(Tier.GOLD);

        assertEquals(2, goldCustomers.size());
        assertTrue(goldCustomers.stream().allMatch(c -> c.getTier() == Tier.GOLD));
    }
}
When NOT to follow the pyramid blindly:
  • Data-pipeline systems: Integration tests matter more than unit tests because the logic lives in how components connect
  • CRUD apps with little business logic: A thin unit layer is fine; integration tests provide more value
  • UI-heavy apps: Consider the "testing trophy" (Kent C. Dodds) which emphasizes integration tests over unit tests
Interview Tip: If asked "How would you test this system?" walk through the pyramid. "Unit tests for the pricing engine, integration tests for the repository layer, and one E2E test for the critical checkout path."

public class ShoppingCart {
    public BigDecimal getTotal() {

Cycle 2: RED — Adding one item

@Test
void totalReflectsSingleItem() {
    ShoppingCart cart = new ShoppingCart();
    cart.addItem("Widget", new BigDecimal("9.99"), 1);
    assertEquals(new BigDecimal("9.99"), cart.getTotal());
}

Cycle 2: GREEN

public class ShoppingCart {
    private final List<LineItem> items = new ArrayList<>();

    public void addItem(String name, BigDecimal price, int quantity) {
        items.add(new LineItem(name, price, quantity));
    }

    public BigDecimal getTotal() {
        return items.stream()
            .map(item -> item.price().multiply(BigDecimal.valueOf(item.quantity())))
            .reduce(BigDecimal.ZERO, BigDecimal::add);
    }

    record LineItem(String name, BigDecimal price, int quantity) {}
}

Cycle 3: RED — Multiple items with quantities

@Test
void totalSumsMultipleItems() {
    ShoppingCart cart = new ShoppingCart();
    cart.addItem("Widget", new BigDecimal("9.99"), 2);  // 19.98
    cart.addItem("Gadget", new BigDecimal("24.50"), 1); // 24.50
    assertEquals(new BigDecimal("44.48"), cart.getTotal());
}
// GREEN: already passes with current implementation!

Cycle 4: RED — Removing items

@Test

    ShoppingCart cart = new ShoppingCart();

    cart.addItem("Gadget", new BigDecimal("24.50"), 1);

    cart.removeItem("Widget");

    assertEquals(new BigDecimal("24.50"), cart.getTotal());
    assertEquals(1, cart.getItemCount());
}

Cycle 4: GREEN

public void removeItem(String name) {
    items.removeIf(item -> item.name().equals(name));

public int getItemCount() {

public void addItem(String name, BigDecimal price, int quantity) {
    if (quantity <= 0) {
        throw new IllegalArgumentException("Quantity must be positive, got: " + quantity);
    }
    if (price.compareTo(BigDecimal.ZERO) < 0) {
        throw new IllegalArgumentException("Price must be non-negative, got: " + price);

    items.add(new LineItem(name, price, quantity));
}
When NOT to use strict TDD:
  • Exploratory prototypes: When you don't know the interface yet, spiking first is faster
  • UI/presentation code: Visual output is better validated visually
  • Framework glue code: Configuration classes, annotations — test through integration tests instead

Many experienced engineers use "test-first" for business logic and "test-after" for infrastructure code. That is fine.

Interview Tip: If asked "Do you practice TDD?", avoid absolutist answers. Say: "I use TDD when designing new business logic because it clarifies the API. For integration and infrastructure code, I often write tests after the spike to lock in behavior."

5. Testing Concurrent Code

Concurrency Testing — Making Nondeterminism Deterministic ▼

Why this is hard: Concurrency bugs (race conditions, deadlocks, visibility issues) are nondeterministic. A test that passes 999 times and fails once is worse than no test at all because it creates false confidence. The key is to use synchronization primitives to force the interleaving you want to test.

CountDownLatch — Coordinating Concurrent Test Threads

@Test
void shouldHandleConcurrentDeposits() throws InterruptedException {
    BankAccount account = new BankAccount("ACC-1", BigDecimal.ZERO);
    int threadCount = 100;
    BigDecimal depositAmount = new BigDecimal("10.00");

    CountDownLatch startGate = new CountDownLatch(1);  // all threads wait
    CountDownLatch endGate = new CountDownLatch(threadCount);  // main waits

    for (int i = 0; i < threadCount; i++) {
        new Thread(() -> {
            try {
                startGate.await();  // wait for signal
                account.deposit(depositAmount);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
            } finally {
                endGate.countDown();
            }
        }).start();
    }

    startGate.countDown();    // release all threads simultaneously
    endGate.await(5, TimeUnit.SECONDS);  // wait for all to finish

    // If deposit is not thread-safe, this will likely fail
    assertEquals(new BigDecimal("1000.00"), account.getBalance());
}

Awaitility — Elegant Async Assertions

Awaitility eliminates fragile Thread.sleep() calls and replaces them with polling-based assertions that wait only as long as needed.

import static org.awaitility.Awaitility.*;

@Test
void shouldProcessEventAsynchronously() {
    EventBus eventBus = new EventBus();
    TestEventHandler handler = new TestEventHandler();
    eventBus.subscribe("order.created", handler);

    eventBus.publish(new OrderCreatedEvent("ORD-123"));

    // Polls every 100ms, fails after 5s
    await().atMost(5, TimeUnit.SECONDS)
           .pollInterval(100, TimeUnit.MILLISECONDS)
           .untilAsserted(() -> {
               assertTrue(handler.wasInvoked());
               assertEquals("ORD-123", handler.getLastOrderId());
           });
}

@Test
void shouldDrainQueueWithinTimeout() {
    WorkQueue queue = new WorkQueue(4);  // 4 worker threads
    queue.submitAll(generateTasks(50));

    await().atMost(10, TimeUnit.SECONDS)
           .until(() -> queue.getPendingCount() == 0);

    assertEquals(50, queue.getCompletedCount());
    assertEquals(0, queue.getFailedCount());
}

Testing Thread Safety of a Cache

@RepeatedTest(10)  // Run multiple times to increase chance of catching races
void cacheShouldBeThreadSafe() throws InterruptedException {
    ThreadSafeCache<String, Integer> cache = new ThreadSafeCache<>(100);
    int threads = 50;
    CountDownLatch latch = new CountDownLatch(1);
    CountDownLatch done = new CountDownLatch(threads);
    AtomicInteger errors = new AtomicInteger(0);

    for (int i = 0; i < threads; i++) {
        final int id = i;
        new Thread(() -> {
            try {
                latch.await();
                // Mix reads and writes
                cache.put("key-" + id, id);
                cache.get("key-" + (id % 10));
                cache.put("shared-key", id);  // contention point
                Integer val = cache.get("shared-key");
                if (val == null) errors.incrementAndGet();
            } catch (Exception e) {
                errors.incrementAndGet();
            } finally {
                done.countDown();
            }
        }).start();
    }

    latch.countDown();
    assertTrue(done.await(10, TimeUnit.SECONDS));
    assertEquals(0, errors.get(), "Concurrent operations should not cause errors");
}
When NOT to write concurrency tests:
  • If the class is inherently single-threaded (called only from one thread, e.g., a request-scoped bean)
  • If thread safety is guaranteed by the container (e.g., stateless Spring singletons with no mutable fields)
  • If the concurrency is handled by a well-tested library (ConcurrentHashMap, Executors) — test your logic, not the JDK
Interview Tip: If your LLD design has shared mutable state (e.g., parking lot's available spots, elevator's current floor), mention that you would write a concurrency test using CountDownLatch. This signals production awareness.

6. Property-Based Testing

Property-Based Testing with jqwik ▼

Why property-based testing: Instead of testing with specific examples ("add 3 and 5, expect 8"), you specify properties that must hold for all inputs ("for any two positive integers, their sum is greater than either"). The framework generates hundreds of random inputs and tries to find a counterexample. This catches edge cases humans overlook.

jqwik Example: Testing a Sorting Function

import net.jqwik.api.*;

class SortingPropertyTest {

    @Property
    void sortedOutputHasSameSizeAsInput(@ForAll List<Integer> input) {
        List<Integer> sorted = Sorter.sort(input);
        assertEquals(input.size(), sorted.size());
    }

    @Property
    void sortedOutputIsOrdered(@ForAll List<Integer> input) {
        List<Integer> sorted = Sorter.sort(input);
        for (int i = 1; i < sorted.size(); i++) {
            assertTrue(sorted.get(i - 1) <= sorted.get(i),
                "Element at " + (i - 1) + " should be <= element at " + i);
        }
    }

    @Property
    void sortedOutputContainsSameElements(@ForAll List<Integer> input) {
        List<Integer> sorted = Sorter.sort(input);
        // Same elements, same frequencies
        assertEquals(
            input.stream().sorted().collect(Collectors.toList()),
            sorted
        );
    }

    @Property
    void sortingIsIdempotent(@ForAll List<Integer> input) {
        List<Integer> sortedOnce = Sorter.sort(input);
        List<Integer> sortedTwice = Sorter.sort(sortedOnce);
        assertEquals(sortedOnce, sortedTwice);
    }
}

When Property-Based Testing Shines

// Serialization round-trip: serialize then deserialize = original
@Property
void jsonRoundTrip(@ForAll("users") User user) {
    String json = serializer.toJson(user);
    User deserialized = serializer.fromJson(json, User.class);
    assertEquals(user, deserialized);
}

@Provide
Arbitrary<User> users() {
    Arbitrary<String> names = Arbitraries.strings().alpha().ofMinLength(1).ofMaxLength(50);
    Arbitrary<String> emails = Arbitraries.strings().alpha().ofMinLength(3)
        .map(s -> s + "@example.com");
    Arbitrary<Integer> ages = Arbitraries.integers().between(0, 150);
    return Combinators.combine(names, emails, ages).as(User::new);
}

// Encoding round-trip
@Property
void base64RoundTrip(@ForAll byte[] data) {
    String encoded = Base64.getEncoder().encodeToString(data);
    byte[] decoded = Base64.getDecoder().decode(encoded);
    assertArrayEquals(data, decoded);
}
When NOT to use property-based testing:
  • CRUD operations — hard to express meaningful properties beyond "it doesn't crash"
  • Complex business rules where the property would just restate the implementation
  • UI tests — visual correctness is not easily expressed as a property
  • When example-based tests are clearer — don't use it just because you can

Best for: parsers, serializers, encoders, mathematical functions, data structure invariants.

Interview Tip: You probably won't write property-based tests in an interview, but mentioning "I'd also add property-based tests for the serializer round-trip" shows depth beyond the standard toolbox.

7. Testability by Design

Designing Code That Is Easy to Test ▼

Why this matters most: The number one reason code is hard to test is not the testing tools — it is the code's design. Testable code is loosely coupled, has clear interfaces, and avoids hidden dependencies. These are the exact same properties that make code maintainable.

Dependency Injection — Pass, Don't Construct

// HARD TO TEST: creates its own dependency
public class OrderProcessor {
    public void process(Order order) {
        PaymentGateway gateway = new StripePaymentGateway(); // hidden dependency!
        gateway.charge(order.getTotal());
        // How do you test this without hitting Stripe?
    }
}

// EASY TO TEST: dependency injected
public class OrderProcessor {
    private final PaymentGateway gateway;

    public OrderProcessor(PaymentGateway gateway) {  // inject it
        this.gateway = gateway;
    }

    public void process(Order order) {
        gateway.charge(order.getTotal());
    }
}

// Test
@Test
void shouldChargeCorrectAmount() {
    PaymentGateway mockGateway = mock(PaymentGateway.class);
    OrderProcessor processor = new OrderProcessor(mockGateway);

    processor.process(orderOf(new BigDecimal("49.99")));

    verify(mockGateway).charge(new BigDecimal("49.99"));
}

Interface Extraction — Program to Abstractions

// HARD TO TEST: depends on concrete class with network calls
public class ReportGenerator {
    private final AwsS3Client s3Client; // concrete AWS SDK class

    public Report generate(String dataKey) {
        byte[] data = s3Client.getObject("my-bucket", dataKey); // real AWS call
        return parseAndGenerate(data);
    }
}

// EASY TO TEST: extract an interface
public interface FileStorage {
    byte[] read(String key);
    void write(String key, byte[] data);
}

public class S3FileStorage implements FileStorage {
    private final AwsS3Client s3Client;
    // ... real implementation
}

public class ReportGenerator {
    private final FileStorage storage;  // depends on interface

    public Report generate(String dataKey) {
        byte[] data = storage.read(dataKey);
        return parseAndGenerate(data);
    }
}

// Test with in-memory implementation
public class InMemoryFileStorage implements FileStorage {
    private final Map<String, byte[]> files = new HashMap<>();

    @Override
    public byte[] read(String key) {
        return files.get(key);
    }

    @Override
    public void write(String key, byte[] data) {
        files.put(key, data);
    }
}

@Test
void shouldGenerateReportFromStoredData() {
    InMemoryFileStorage storage = new InMemoryFileStorage();
    storage.write("data.csv", "name,score\nAlice,95\nBob,87".getBytes());

    ReportGenerator generator = new ReportGenerator(storage);
    Report report = generator.generate("data.csv");

    assertEquals(2, report.getRowCount());
}

Clock Injection — Taming Time

Code that calls Instant.now() or LocalDate.now() directly is impossible to test deterministically.

// HARD TO TEST: time is not controllable
public class SubscriptionService {
    public boolean isExpired(Subscription sub) {
        return sub.getExpiryDate().isBefore(LocalDate.now()); // untestable!
    }
}

// EASY TO TEST: inject a Clock
public class SubscriptionService {
    private final Clock clock;

    public SubscriptionService(Clock clock) {
        this.clock = clock;
    }

    public boolean isExpired(Subscription sub) {
        return sub.getExpiryDate().isBefore(LocalDate.now(clock));
    }
}

// Production: new SubscriptionService(Clock.systemDefaultZone())
// Test:
@Test
void shouldDetectExpiredSubscription() {
    Clock fixedClock = Clock.fixed(
        LocalDate.of(2025, 6, 15).atStartOfDay(ZoneId.systemDefault()).toInstant(),
        ZoneId.systemDefault()
    );
    SubscriptionService service = new SubscriptionService(fixedClock);

    Subscription sub = new Subscription(LocalDate.of(2025, 6, 1)); // expired

    assertTrue(service.isExpired(sub));
}

@Test
void shouldDetectActiveSubscription() {
    Clock fixedClock = Clock.fixed(
        LocalDate.of(2025, 6, 15).atStartOfDay(ZoneId.systemDefault()).toInstant(),
        ZoneId.systemDefault()
    );
    SubscriptionService service = new SubscriptionService(fixedClock);

    Subscription sub = new Subscription(LocalDate.of(2025, 12, 31)); // still active

    assertFalse(service.isExpired(sub));
}

Avoiding Static Methods

// HARD TO TEST: static methods cannot be mocked (without PowerMock hacks)
public class NotificationSender {
    public void send(String userId, String message) {
        String email = UserDirectory.lookupEmail(userId);  // static call!
        SmtpClient.send(email, message);                   // static call!
    }
}

// EASY TO TEST: wrap static calls behind injectable interfaces
public class NotificationSender {
    private final UserDirectory userDirectory;
    private final MailClient mailClient;

    public NotificationSender(UserDirectory userDirectory, MailClient mailClient) {
        this.userDirectory = userDirectory;
        this.mailClient = mailClient;
    }

    public void send(String userId, String message) {
        String email = userDirectory.lookupEmail(userId);
        mailClient.send(email, message);
    }
}
When NOT to over-engineer for testability:
  • Simple utility/math functions: Math.max(a, b) does not need an interface
  • One implementation forever: If there will never be a second implementation, an interface adds indirection for no gain. Use a class and mock it if needed
  • Prototypes: Adding DI and interfaces slows exploration. Refactor for testability when the design stabilizes
Interview Tip: When presenting your LLD solution, say "I'm injecting the repository through the constructor so we can unit-test the service with a mock." This demonstrates design-for-testability naturally.

5 Q&A Pairs for Spaced Repetition ▼
Front (Question) Back (Answer)
What is the difference between a Mockito mock and a spy? A mock is a complete substitute (all methods return defaults unless stubbed). A spy wraps a real object — unstubbed methods call the real implementation. Use spies for legacy code you can't refactor.
What are the three steps of the TDD cycle? RED (write a failing test), GREEN (write minimum code to pass), REFACTOR (clean up without adding behavior). Then repeat.
How do you make time-dependent code testable in Java? Inject a java.time.Clock via the constructor. In production, pass Clock.systemDefaultZone(). In tests, pass Clock.fixed(...) to control the current time.
What is property-based testing and when is it most useful? Instead of specific examples, you specify invariants that must hold for all generated inputs (e.g., "encoding then decoding produces the original"). Best for: serialization round-trips, parsers, mathematical functions, data structure invariants.
What pattern lets you coordinate many threads to start simultaneously in a test? Use a CountDownLatch(1) as a start gate. All threads call await(), then the test calls countDown() to release them at once. A second latch counts completion.

Practice Exercise: TDD a Stack

Exercise — Build a Generic Stack Using TDD ▼

Task: Implement a generic BoundedStack<T> class using strict TDD (write the test first, then the code). The stack has a maximum capacity.

Requirements

  • push(T item) — adds to the top; throws StackOverflowException if full
  • pop() — removes and returns the top; throws EmptyStackException if empty
  • peek() — returns the top without removing; throws EmptyStackException if empty
  • size() — returns the number of elements
  • isEmpty() and isFull() — boolean queries
  • Rejects null items
  • Capacity must be a positive integer (validated in constructor)

Suggested TDD Order

  1. New stack is empty and has size 0
  2. Push one item, size is 1, isEmpty is false
  3. Push and pop returns the same item (LIFO)
  4. Push two, pop returns the second (LIFO order)
  5. Pop on empty stack throws EmptyStackException
  6. Peek returns top without removing
  7. Push to capacity, isFull is true
  8. Push beyond capacity throws StackOverflowException
  9. Push null throws NullPointerException
  10. Constructor rejects zero or negative capacity

Sample Solution

// Tests
class BoundedStackTest {

    @Test
    void newStackIsEmpty() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        assertTrue(stack.isEmpty());
        assertEquals(0, stack.size());
    }

    @Test
    void pushIncreasesSize() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        stack.push("A");
        assertFalse(stack.isEmpty());
        assertEquals(1, stack.size());
    }

    @Test
    void popReturnsLastPushedItem() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        stack.push("A");
        assertEquals("A", stack.pop());
        assertTrue(stack.isEmpty());
    }

    @Test
    void stackIsLifo() {
        BoundedStack<Integer> stack = new BoundedStack<>(5);
        stack.push(1);
        stack.push(2);
        stack.push(3);
        assertEquals(3, stack.pop());
        assertEquals(2, stack.pop());
        assertEquals(1, stack.pop());
    }

    @Test
    void popOnEmptyThrows() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        assertThrows(EmptyStackException.class, stack::pop);
    }

    @Test
    void peekReturnsTopWithoutRemoving() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        stack.push("A");
        assertEquals("A", stack.peek());
        assertEquals(1, stack.size()); // still there
    }

    @Test
    void peekOnEmptyThrows() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        assertThrows(EmptyStackException.class, stack::peek);
    }

    @Test
    void pushToCapacityMakesItFull() {
        BoundedStack<String> stack = new BoundedStack<>(2);
        stack.push("A");
        stack.push("B");
        assertTrue(stack.isFull());
    }

    @Test
    void pushBeyondCapacityThrows() {
        BoundedStack<String> stack = new BoundedStack<>(1);
        stack.push("A");
        assertThrows(StackOverflowException.class, () -> stack.push("B"));
    }

    @Test
    void pushNullThrows() {
        BoundedStack<String> stack = new BoundedStack<>(5);
        assertThrows(NullPointerException.class, () -> stack.push(null));
    }

    @Test
    void constructorRejectsInvalidCapacity() {
        assertThrows(IllegalArgumentException.class, () -> new BoundedStack<>(0));
        assertThrows(IllegalArgumentException.class, () -> new BoundedStack<>(-3));
    }
}

// Implementation
public class BoundedStack<T> {
    private final Object[] elements;
    private int top = -1;

    public BoundedStack(int capacity) {
        if (capacity <= 0) {
            throw new IllegalArgumentException(
                "Capacity must be positive, got: " + capacity);
        }
        this.elements = new Object[capacity];
    }

    public void push(T item) {
        Objects.requireNonNull(item, "Cannot push null onto stack");
        if (isFull()) {
            throw new StackOverflowException(
                "Stack is full (capacity: " + elements.length + ")");
        }
        elements[++top] = item;
    }

    @SuppressWarnings("unchecked")
    public T pop() {
        if (isEmpty()) {
            throw new EmptyStackException();
        }
        T item = (T) elements[top];
        elements[top--] = null; // help GC
        return item;
    }

    @SuppressWarnings("unchecked")
    public T peek() {
        if (isEmpty()) {
            throw new EmptyStackException();
        }
        return (T) elements[top];
    }

    public int size() {
        return top + 1;
    }

    public boolean isEmpty() {
        return top == -1;
    }

    public boolean isFull() {
        return top == elements.length - 1;
    }
}
