Bridging LLD and System Design

Why This Module Matters for Staff Engineers ▼

In staff-level interviews, the line between Low-Level Design and System Design blurs. You are expected to zoom in from a high-level architecture to the class-level design of a critical component, and zoom back out to explain how that component fits into the broader system. This module covers the architectural patterns that live at that boundary.

Why it matters: A senior engineer might design a clean class hierarchy. A staff engineer explains why the aggregate boundary sits where it does, how the domain events propagate across services, and what happens when the circuit breaker trips at 3 AM.

1. Domain-Driven Design in Java

DDD - Aggregates, Repositories, Domain Events & Bounded Contexts ▼

Why DDD? Domain-Driven Design gives you a principled way to organize complex business logic into cohesive units. Without DDD, services become "CRUD wrappers" where business rules leak into controllers, services, and even the database layer. DDD forces you to model the domain first and let the infrastructure serve it.

Core Concepts:

  • Entity: Has a unique identity that persists across state changes (e.g., an Order with an orderId)
  • Value Object: Defined by its attributes, not identity. Immutable. Two Value Objects with the same fields are equal (e.g., Money, Address)
  • Aggregate: A cluster of entities and value objects with a single root entity (the Aggregate Root). All external access goes through the root. The aggregate enforces invariants.
  • Repository: Abstraction for persisting and retrieving aggregates. One repository per aggregate root.
  • Domain Event: Something meaningful that happened in the domain. Used to decouple bounded contexts.
  • Bounded Context: A boundary within which a particular domain model applies. The same real-world concept (e.g., "Customer") may have different models in different contexts.

Value Object vs Entity

// VALUE OBJECT - Immutable, equality by attributes
// WHY: Money should never be mutated. $10 is $10 regardless of "which" $10 it is.
public record Money(BigDecimal amount, Currency currency) {

    public Money {
        Objects.requireNonNull(amount, "Amount must not be null");
        Objects.requireNonNull(currency, "Currency must not be null");
        if (amount.scale() > currency.getDefaultFractionDigits()) {
            throw new IllegalArgumentException("Scale exceeds currency precision");
        }
    }

    public static Money of(double amount, Currency currency) {
        return new Money(BigDecimal.valueOf(amount), currency);
    }

    public static Money usd(double amount) {
        return of(amount, Currency.getInstance("USD"));
    }

    public Money add(Money other) {
        assertSameCurrency(other);
        return new Money(this.amount.add(other.amount), this.currency);
    }

    public Money subtract(Money other) {
        assertSameCurrency(other);
        return new Money(this.amount.subtract(other.amount), this.currency);
    }

    public Money multiply(int quantity) {
        return new Money(this.amount.multiply(BigDecimal.valueOf(quantity)), this.currency);
    }

    public boolean isGreaterThan(Money other) {
        assertSameCurrency(other);
        return this.amount.compareTo(other.amount) > 0;
    }

    private void assertSameCurrency(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException(
                "Cannot operate on different currencies: " + currency + " vs " + other.currency);
        }
    }
}

// ENTITY - Has identity, mutable state
// WHY: Two orders with the same items are NOT the same order.
public class Order {
    private final OrderId id;             // Identity
    private final CustomerId customerId;
    private OrderStatus status;
    private final List<OrderLine> lines;
    private Money totalAmount;
    private final List<DomainEvent> domainEvents = new ArrayList<>();

    // Entities are equal by identity, not by attributes
    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Order order)) return false;
        return id.equals(order.id);
    }

    @Override
    public int hashCode() {
        return id.hashCode();
    }
}

Aggregate Root with Invariant Enforcement

// The Order aggregate root enforces ALL business invariants.
// External code NEVER modifies OrderLine directly.
public class Order {
    private final OrderId id;
    private final CustomerId customerId;
    private OrderStatus status;
    private final List<OrderLine> lines = new ArrayList<>();
    private Money totalAmount;
    private final Instant createdAt;
    private Instant updatedAt;
    private final List<DomainEvent> domainEvents = new ArrayList<>();

    // Factory method - not a public constructor
    public static Order create(CustomerId customerId) {
        var order = new Order(OrderId.generate(), customerId);
        order.status = OrderStatus.DRAFT;
        order.totalAmount = Money.usd(0);
        order.createdAt = Instant.now();
        order.registerEvent(new OrderCreatedEvent(order.id, customerId));
        return order;
    }

    // Business method with invariant enforcement
    public void addItem(ProductId productId, String name, Money unitPrice, int quantity) {
        // INVARIANT: Cannot modify a confirmed or shipped order
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException(
                "Cannot add items to order in status: " + status);
        }
        // INVARIANT: Max 50 line items per order
        if (lines.size() >= 50) {
            throw new BusinessRuleViolation("Order cannot exceed 50 line items");
        }
        // INVARIANT: Quantity must be positive
        if (quantity <= 0) {
            throw new IllegalArgumentException("Quantity must be positive");
        }

        var line = new OrderLine(productId, name, unitPrice, quantity);
        lines.add(line);
        recalculateTotal();
        updatedAt = Instant.now();

        registerEvent(new OrderItemAddedEvent(id, productId, quantity));
    }

    public void confirm() {
        // INVARIANT: Order must have at least one item
        if (lines.isEmpty()) {
            throw new BusinessRuleViolation("Cannot confirm an empty order");
        }
        // INVARIANT: Order must be in DRAFT status
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Can only confirm DRAFT orders");
        }

        this.status = OrderStatus.CONFIRMED;
        this.updatedAt = Instant.now();
        registerEvent(new OrderConfirmedEvent(id, customerId, totalAmount));
    }

    public void cancel(String reason) {
        if (status == OrderStatus.SHIPPED || status == OrderStatus.DELIVERED) {
            throw new IllegalStateException("Cannot cancel " + status + " order");
        }
        this.status = OrderStatus.CANCELLED;
        this.updatedAt = Instant.now();
        registerEvent(new OrderCancelledEvent(id, reason));
    }

    // Domain events are collected and dispatched after persistence
    public List<DomainEvent> getDomainEvents() {
        return Collections.unmodifiableList(domainEvents);
    }

    public void clearDomainEvents() {
        domainEvents.clear();
    }

    private void registerEvent(DomainEvent event) {
        domainEvents.add(event);
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(OrderLine::lineTotal)
            .reduce(Money.usd(0), Money::add);
    }

    // Expose read-only view of lines
    public List<OrderLine> getLines() {
        return Collections.unmodifiableList(lines);
    }
}

// Value Object nested within the aggregate
public record OrderLine(
    ProductId productId,
    String productName,
    Money unitPrice,
    int quantity
) {
    public OrderLine {
        Objects.requireNonNull(productId);
        if (quantity <= 0) throw new IllegalArgumentException("Quantity must be positive");
    }

    public Money lineTotal() {
        return unitPrice.multiply(quantity);
    }
}

Repository Pattern

// Repository interface - lives in the DOMAIN layer
// WHY: The domain should not know about JPA, SQL, or any persistence tech
public interface OrderRepository {
    Optional<Order> findById(OrderId id);
    List<Order> findByCustomer(CustomerId customerId);
    Order save(Order order);
    void delete(OrderId id);
}

// Infrastructure implementation - lives in the INFRASTRUCTURE layer
public class JpaOrderRepository implements OrderRepository {
    private final EntityManager em;
    private final ApplicationEventPublisher eventPublisher;

    public JpaOrderRepository(EntityManager em, ApplicationEventPublisher eventPublisher) {
        this.em = em;
        this.eventPublisher = eventPublisher;
    }

    @Override
    @Transactional
    public Order save(Order order) {
        Order saved = em.merge(order);

        // Dispatch domain events after successful persistence
        for (DomainEvent event : saved.getDomainEvents()) {
            eventPublisher.publishEvent(event);
        }
        saved.clearDomainEvents();

        return saved;
    }

    @Override
    public Optional<Order> findById(OrderId id) {
        return Optional.ofNullable(em.find(Order.class, id));
    }
}

Domain Events and Bounded Contexts

// Domain Event - immutable record of something that happened
public sealed interface DomainEvent permits
        OrderCreatedEvent, OrderConfirmedEvent, OrderCancelledEvent, OrderItemAddedEvent {
    Instant occurredAt();
    String aggregateId();
}

public record OrderConfirmedEvent(
    OrderId orderId,
    CustomerId customerId,
    Money totalAmount,
    Instant occurredAt
) implements DomainEvent {
    public OrderConfirmedEvent(OrderId orderId, CustomerId customerId, Money totalAmount) {
        this(orderId, customerId, totalAmount, Instant.now());
    }

    @Override
    public String aggregateId() { return orderId.value(); }
}

// In ANOTHER bounded context (Shipping), we listen for Order events
// Note: the Shipping context has its OWN model of what an "order" means
@Component
public class ShippingOrderEventHandler {

    private final ShipmentRepository shipmentRepository;

    @EventListener
    public void onOrderConfirmed(OrderConfirmedEvent event) {
        // Shipping context creates its own entity, NOT reusing Order
        var shipment = Shipment.createFor(
            new ShipmentOrderRef(event.orderId().value()),
            event.customerId()
        );
        shipmentRepository.save(shipment);
    }
}
When NOT to Use DDD:
  • Simple CRUD applications with minimal business logic
  • Small teams or early prototypes where speed outweighs rigor
  • Read-heavy reporting services (consider CQRS read models instead)
  • When the domain experts are not available for modeling sessions

Amazon's early services started as simple CRUD and evolved DDD patterns only when business complexity demanded it.

Interview Tip: When designing an aggregate, ask: "What data MUST be consistent within a single transaction?" That boundary IS your aggregate. Everything else can be eventually consistent via domain events. At Netflix, the Subscription aggregate enforces billing invariants, but the recommendation engine updates asynchronously.

2. CQRS Pattern

CQRS - Command Query Responsibility Segregation ▼

Why CQRS? Most applications have fundamentally different read and write characteristics. Writes need strong consistency, validation, and invariant enforcement. Reads need speed, denormalization, and flexible projections. CQRS separates these concerns into distinct models, allowing each to be optimized independently.

Real-world example: Amazon's product catalog. Writes (seller updates a listing) go through a complex validation pipeline. Reads (customer browses products) hit a heavily denormalized, cached read model optimized for search and display. These are completely separate codepaths.

Command Side

// Commands are imperative - they express INTENT
public sealed interface Command permits
        PlaceOrderCommand, CancelOrderCommand, AddItemCommand {
}

public record PlaceOrderCommand(
    CustomerId customerId,
    List<OrderItemRequest> items,
    ShippingAddress shippingAddress
) implements Command {}

public record CancelOrderCommand(
    OrderId orderId,
    String reason
) implements Command {}

// Command Handler - orchestrates the write side
@Service
public class OrderCommandHandler {
    private final OrderRepository orderRepository;
    private final ProductCatalogService catalogService;
    private final DomainEventPublisher eventPublisher;

    @Transactional
    public OrderId handle(PlaceOrderCommand cmd) {
        // 1. Validate the command
        var customer = validateCustomer(cmd.customerId());

        // 2. Create the aggregate
        var order = Order.create(cmd.customerId());

        // 3. Execute business logic through the aggregate
        for (var item : cmd.items()) {
            var product = catalogService.getProduct(item.productId());
            order.addItem(product.id(), product.name(), product.price(), item.quantity());
        }

        order.confirm();

        // 4. Persist the aggregate (dispatches domain events)
        orderRepository.save(order);

        return order.getId();
    }

    @Transactional
    public void handle(CancelOrderCommand cmd) {
        var order = orderRepository.findById(cmd.orderId())
            .orElseThrow(() -> new OrderNotFoundException(cmd.orderId()));

        order.cancel(cmd.reason());
        orderRepository.save(order);
    }
}

public record OrderDetailsView(

// Query Handler - reads from the denormalized read store
@Service
public class OrderQueryHandler {
    private final OrderReadRepository readRepository;

    public OrderDetailsView handle(GetOrderDetailsQuery query) {
        return readRepository.findOrderDetails(query.orderId().value())
            .orElseThrow(() -> new OrderNotFoundException(query.orderId()));
    }

    public Page<OrderSummaryView> handle(ListCustomerOrdersQuery query) {
        return readRepository.findByCustomer(
            query.customerId().value(),
            PageRequest.of(query.page(), query.pageSize())
        );

// Read-side projection updater - builds the read model from events
@Component
public class OrderProjectionUpdater {
    private final OrderReadRepository readRepository;

    @EventListener
    public void on(OrderConfirmedEvent event) {
        // Build/update the denormalized read model
        var view = OrderDetailsView.builder()
            .orderId(event.orderId().value())
            .status("CONFIRMED")
            .totalAmount(event.totalAmount().toString())
            .placedAt(event.occurredAt())
            .build();

        readRepository.save(view);
    }

    @EventListener
    public void on(OrderCancelledEvent event) {
        readRepository.updateStatus(event.orderId().value(), "CANCELLED");
    }
}
When NOT to Use CQRS:
  • Simple domains where reads and writes use the same model
  • When eventual consistency is unacceptable (the read model lags behind writes)
  • Small teams that cannot maintain two separate models
  • Applications where write volume and read volume are similar and simple
Interview Tip: CQRS is not all-or-nothing. You can apply it to a single bounded context that has complex read/write asymmetry while keeping other contexts as simple CRUD. Always state the trade-off: "We gain independent scaling and optimized queries at the cost of eventual consistency and operational complexity."

3. Event Sourcing

Event Sourcing - The Event Store as Source of Truth ▼

Why Event Sourcing? Instead of storing just the current state, you store the complete sequence of events that led to the current state. This gives you a full audit trail, the ability to replay events to rebuild state, and temporal queries ("What was the order state at 2pm yesterday?").

Real-world example: Banking ledgers are the original event store. You never "update" a balance -- you append credits and debits. The balance is a projection derived from replaying all events. Amazon's order processing similarly stores every state transition as an immutable event.

Event Sourced Aggregate

// Base class for event-sourced aggregates
public abstract class EventSourcedAggregate {
    private final List<DomainEvent> uncommittedEvents = new ArrayList<>();
    private int version = 0;

    // Rebuild state by replaying events
    public void loadFromHistory(List<DomainEvent> eventStream) {
        for (DomainEvent event : eventStream) {
            applyEvent(event, false);
            version++;
        }
    }

    // Apply a new event (mutates state + records for persistence)
    protected void raiseEvent(DomainEvent event) {
        applyEvent(event, true);
    }

    private void applyEvent(DomainEvent event, boolean isNew) {
        // Dispatch to the correct handler via pattern matching
        apply(event);
        if (isNew) {
            uncommittedEvents.add(event);
        }
    }

    protected abstract void apply(DomainEvent event);

    public List<DomainEvent> getUncommittedEvents() {
        return Collections.unmodifiableList(uncommittedEvents);
    }

    public void markEventsAsCommitted() {
        uncommittedEvents.clear();
    }

    public int getVersion() { return version; }
}

// Concrete event-sourced Order
public class EventSourcedOrder extends EventSourcedAggregate {
    private OrderId id;
    private CustomerId customerId;
    private OrderStatus status;
    private final List<OrderLine> lines = new ArrayList<>();
    private Money totalAmount = Money.usd(0);

    // Commands produce events (never mutate state directly)
    public static EventSourcedOrder create(CustomerId customerId) {
        var order = new EventSourcedOrder();
        order.raiseEvent(new OrderCreatedEvent(OrderId.generate(), customerId));
        return order;
    }

    public void addItem(ProductId productId, String name, Money price, int qty) {
        if (status != OrderStatus.DRAFT) {
            throw new IllegalStateException("Cannot modify non-draft order");
        }
        raiseEvent(new OrderItemAddedEvent(id, productId, name, price, qty));
    }

    public void confirm() {
        if (lines.isEmpty()) throw new BusinessRuleViolation("Order is empty");
        if (status != OrderStatus.DRAFT) throw new IllegalStateException("Not in draft");
        raiseEvent(new OrderConfirmedEvent(id, customerId, totalAmount));
    }

    // State mutation happens ONLY in apply methods
    @Override
    protected void apply(DomainEvent event) {
        switch (event) {
            case OrderCreatedEvent e -> {
                this.id = e.orderId();
                this.customerId = e.customerId();
                this.status = OrderStatus.DRAFT;
            }
            case OrderItemAddedEvent e -> {
                lines.add(new OrderLine(e.productId(), e.name(), e.price(), e.quantity()));
                recalculateTotal();
            }
            case OrderConfirmedEvent e -> {
                this.status = OrderStatus.CONFIRMED;
            }
            case OrderCancelledEvent e -> {
                this.status = OrderStatus.CANCELLED;
            }
            default -> throw new IllegalArgumentException("Unknown event: " + event);
        }
    }

    private void recalculateTotal() {
        this.totalAmount = lines.stream()
            .map(OrderLine::lineTotal)
            .reduce(Money.usd(0), Money::add);
    }
}

// Event Store interface
public interface EventStore {
    void saveEvents(String aggregateId, List<DomainEvent> events, int expectedVersion);
    List<DomainEvent> getEvents(String aggregateId);
    List<DomainEvent> getEventsSince(String aggregateId, int sinceVersion);
}

// Repository using the event store
public class EventSourcedOrderRepository {
    private final EventStore eventStore;

    public EventSourcedOrder findById(OrderId id) {
        List<DomainEvent> events = eventStore.getEvents(id.value());
        if (events.isEmpty()) {
            throw new OrderNotFoundException(id);
        }
        var order = new EventSourcedOrder();
        order.loadFromHistory(events);
        return order;
    }

    public void save(EventSourcedOrder order) {
        eventStore.saveEvents(
            order.getId().value(),
            order.getUncommittedEvents(),
            order.getVersion()
        );
        order.markEventsAsCommitted();
    }
}
When NOT to Use Event Sourcing:
  • Simple CRUD where you never need to ask "how did we get here?"
  • When event schema evolution is a bigger burden than the benefit
  • If you need strong consistency across aggregates (ES is inherently eventually consistent)
  • When the team lacks experience with this pattern (the learning curve is steep)

Benefits: Full audit trail, temporal queries, event replay for debugging, natural fit with CQRS. Costs: Event schema versioning, eventual consistency, more complex infrastructure, snapshot management for long-lived aggregates.

Interview Tip: If you mention Event Sourcing, be prepared to discuss event versioning (what happens when you add a field to OrderCreatedEvent v2?), snapshotting (how to avoid replaying 10M events), and idempotency (what if the same event is delivered twice?).

4. Saga Pattern

Saga - Distributed Transactions via Compensation ▼

Why Sagas? In a microservice architecture, a single business operation (e.g., "place an order") spans multiple services (Order, Payment, Inventory, Shipping). You cannot use a distributed transaction (2PC) because it couples services and kills availability. Sagas replace distributed transactions with a sequence of local transactions, each with a compensating action for rollback.

Real-world example: When you book a trip on Expedia, the system reserves a flight, then a hotel, then a car. If the car reservation fails, it must cancel the hotel and flight. Each step is a local transaction in a separate service, with compensating actions defined for each.

Orchestration-Based Saga (State Machine)

// Saga state - persisted for crash recovery
public enum OrderSagaState {
    STARTED,
    PAYMENT_PENDING,
    PAYMENT_COMPLETED,
    INVENTORY_RESERVED,
    COMPLETED,
    // Compensation states
    COMPENSATING_INVENTORY,
    COMPENSATING_PAYMENT,
    COMPENSATING_ORDER,
    FAILED
}

// Saga orchestrator - the "brain" that drives the workflow
@Service
public class OrderSagaOrchestrator {
    private final SagaRepository sagaRepository;
    private final PaymentService paymentService;
    private final InventoryService inventoryService;
    private final ShippingService shippingService;
    private final OrderService orderService;

    @Transactional
    public String startSaga(PlaceOrderCommand cmd) {
        // 1. Create the order (local transaction)
        OrderId orderId = orderService.createOrder(cmd);

        // 2. Initialize saga state
        var saga = new OrderSagaState(orderId, OrderSagaState.STARTED);
        sagaRepository.save(saga);

        // 3. Advance the saga
        advance(saga);

        return saga.getId();
    }

    public void advance(OrderSaga saga) {
        try {
            switch (saga.getState()) {
                case STARTED -> {
                    paymentService.processPayment(saga.getOrderId(), saga.getAmount());
                    saga.setState(OrderSagaState.PAYMENT_PENDING);
                }
                case PAYMENT_COMPLETED -> {
                    inventoryService.reserve(saga.getOrderId(), saga.getItems());
                    saga.setState(OrderSagaState.INVENTORY_RESERVED);
                }
                case INVENTORY_RESERVED -> {
                    shippingService.createShipment(saga.getOrderId());
                    saga.setState(OrderSagaState.COMPLETED);
                }
                // Compensation steps
                case COMPENSATING_INVENTORY -> {
                    inventoryService.release(saga.getOrderId());
                    saga.setState(OrderSagaState.COMPENSATING_PAYMENT);
                    advance(saga); // Continue compensating
                }
                case COMPENSATING_PAYMENT -> {
                    paymentService.refund(saga.getOrderId());
                    saga.setState(OrderSagaState.COMPENSATING_ORDER);
                    advance(saga);
                }
                case COMPENSATING_ORDER -> {
                    orderService.cancelOrder(saga.getOrderId());
                    saga.setState(OrderSagaState.FAILED);
                }
                default -> {} // Terminal states
            }
        } catch (Exception e) {
            compensate(saga, e);
        }

        sagaRepository.save(saga);
    }

    private void compensate(OrderSaga saga, Exception cause) {
        // Determine which compensation step to start based on current state
        OrderSagaState compensationStart = switch (saga.getState()) {
            case INVENTORY_RESERVED -> OrderSagaState.COMPENSATING_INVENTORY;
            case PAYMENT_COMPLETED -> OrderSagaState.COMPENSATING_PAYMENT;
            case PAYMENT_PENDING -> OrderSagaState.COMPENSATING_ORDER;
            default -> OrderSagaState.FAILED;
        };

        saga.setState(compensationStart);
        saga.setFailureReason(cause.getMessage());
        advance(saga);
    }

    // Called by event listeners when async steps complete
    @EventListener
    public void onPaymentCompleted(PaymentCompletedEvent event) {
        var saga = sagaRepository.findByOrderId(event.orderId());
        saga.setState(OrderSagaState.PAYMENT_COMPLETED);
        advance(saga);
    }

    @EventListener
    public void onPaymentFailed(PaymentFailedEvent event) {
        var saga = sagaRepository.findByOrderId(event.orderId());
        compensate(saga, new PaymentException(event.reason()));
    }
}

Choreography vs Orchestration

Aspect Orchestration Choreography
Coordination Central orchestrator drives the flow Services react to events independently
Coupling Orchestrator knows all participants Services only know about events
Visibility Easy to see the full workflow in one place Workflow is implicit, harder to trace
Single Point of Failure Yes (the orchestrator) No central coordinator
Best For Complex workflows with many steps Simple 2-3 step flows between services
When NOT to Use Sagas:
  • When a single database transaction suffices (do not over-engineer)
  • If the business can tolerate occasional inconsistency handled by manual reconciliation
  • For read-only operations (sagas are for write coordination)
Interview Tip: Always mention idempotency when discussing sagas. Compensating actions may be retried after failures, so each step must be safe to execute multiple times. Use idempotency keys and check-before-act patterns.

5. Circuit Breakers & Resilience

Resilience4j Patterns - CircuitBreaker, Retry, RateLimiter, Bulkhead ▼

Why Resilience Patterns? In a distributed system, failures are not exceptional -- they are inevitable. Without resilience patterns, a single slow or failing dependency can cascade through the entire system (the "cascading failure" problem). Netflix famously learned this lesson and built Hystrix (now superseded by Resilience4j).

Real-world example: In 2012, a single Netflix microservice became slow, causing thread pools to fill across dozens of downstream services, leading to a full site outage. Circuit breakers would have isolated the failure to that one dependency.

Circuit Breaker with Resilience4j

// Configuration
CircuitBreakerConfig config = CircuitBreakerConfig.custom()
    .failureRateThreshold(50)           // Open when 50% of calls fail
    .waitDurationInOpenState(Duration.ofSeconds(30)) // Stay open for 30s
    .slidingWindowSize(10)              // Evaluate last 10 calls
    .slidingWindowType(SlidingWindowType.COUNT_BASED)
    .permittedNumberOfCallsInHalfOpenState(3)  // Allow 3 trial calls
    .slowCallDurationThreshold(Duration.ofSeconds(2)) // 2s = slow
    .slowCallRateThreshold(80)          // Open if 80% of calls are slow
    .recordExceptions(IOException.class, TimeoutException.class)
    .ignoreExceptions(BusinessValidationException.class)
    .build();

CircuitBreaker circuitBreaker = CircuitBreaker.of("paymentService", config);

// Using the circuit breaker
@Service
public class PaymentServiceClient {
    private final CircuitBreaker circuitBreaker;
    private final RestTemplate restTemplate;

    public PaymentResult processPayment(PaymentRequest request) {
        // Wrap the remote call with the circuit breaker
        Supplier<PaymentResult> decoratedCall = CircuitBreaker
            .decorateSupplier(circuitBreaker, () -> {
                return restTemplate.postForObject(
                    "http://payment-service/api/payments",
                    request,
                    PaymentResult.class
                );
            });

        // Execute with fallback
        return Try.ofSupplier(decoratedCall)
            .recover(CallNotPermittedException.class, e -> {
                // Circuit is OPEN - fail fast with fallback
                return PaymentResult.queued("Circuit breaker open, payment queued for retry");
            })
            .recover(Exception.class, e -> {
                // Other failure - return degraded result
                return PaymentResult.failed(e.getMessage());
            })
            .get();
    }
}

// Listening to circuit breaker state changes
circuitBreaker.getEventPublisher()
    .onStateTransition(event ->
        log.warn("Circuit breaker '{}' transitioned: {} -> {}",
            event.getCircuitBreakerName(),
            event.getStateTransition().getFromState(),
            event.getStateTransition().getToState()))
    .onFailureRateExceeded(event ->
        log.error("Failure rate exceeded: {}%", event.getFailureRate()));

Retry with Exponential Backoff

RetryConfig retryConfig = RetryConfig.custom()
    .maxAttempts(3)
    .waitDuration(Duration.ofMillis(500))
    .intervalFunction(IntervalFunction.ofExponentialBackoff(500, 2))
    .retryOnException(e -> e instanceof IOException)
    .retryExceptions(TimeoutException.class, ConnectException.class)
    .ignoreExceptions(BusinessValidationException.class)
    .build();

Retry retry = Retry.of("paymentRetry", retryConfig);

// Combine circuit breaker + retry
Supplier<PaymentResult> resilientCall = Decorators
    .ofSupplier(() -> callPaymentService(request))
    .withRetry(retry)                    // Retry first
    .withCircuitBreaker(circuitBreaker)  // Then circuit breaker
    .decorate();

// Order matters: Retry wraps CircuitBreaker
// So retries happen INSIDE the circuit breaker's tracking

Bulkhead - Isolating Thread Pools

// WHY: Prevent one slow service from consuming all threads
// Each downstream service gets its own limited thread pool

BulkheadConfig bulkheadConfig = BulkheadConfig.custom()
    .maxConcurrentCalls(10)           // Max 10 concurrent calls
    .maxWaitDuration(Duration.ofMillis(500)) // Wait up to 500ms for a slot
    .build();

Bulkhead bulkhead = Bulkhead.of("paymentBulkhead", bulkheadConfig);

// Thread pool bulkhead (async)
ThreadPoolBulkheadConfig threadPoolConfig = ThreadPoolBulkheadConfig.custom()
    .maxThreadPoolSize(10)
    .coreThreadPoolSize(5)
    .queueCapacity(20)
    .build();

ThreadPoolBulkhead threadPoolBulkhead =
    ThreadPoolBulkhead.of("paymentPool", threadPoolConfig);

Rate Limiter

RateLimiterConfig rateLimiterConfig = RateLimiterConfig.custom()
    .limitForPeriod(100)                       // 100 calls per period
    .limitRefreshPeriod(Duration.ofSeconds(1)) // Period = 1 second
    .timeoutDuration(Duration.ofMillis(500))   // Wait up to 500ms for permit
    .build();

RateLimiter rateLimiter = RateLimiter.of("paymentRateLimit", rateLimiterConfig);

// Composing all patterns together
Supplier<PaymentResult> fullyResilientCall = Decorators
    .ofSupplier(() -> callPaymentService(request))
    .withRateLimiter(rateLimiter)       // 1. Rate limit outgoing calls
    .withBulkhead(bulkhead)             // 2. Limit concurrency
    .withRetry(retry)                   // 3. Retry transient failures
    .withCircuitBreaker(circuitBreaker) // 4. Fail fast if dependency down
    .decorate();
When NOT to Use Circuit Breakers:
  • For in-process calls (the overhead is not worth it)
  • When the called resource has no meaningful fallback
  • For database connections (use connection pool timeouts instead)
  • When you want calls to always go through regardless of failures
Interview Tip: When discussing resilience, always mention the composition order. RateLimiter → Bulkhead → Retry → CircuitBreaker is the standard order. The outermost decorator executes first. Getting this wrong in an interview signals surface-level knowledge.

6. Microservice Boundaries

Service Boundaries - Communication, Data Ownership & API Design ▼

Why Boundaries Matter: The most common mistake in microservice architecture is drawing boundaries wrong. Too granular creates a "distributed monolith" with chatty inter-service calls. Too coarse gives you a monolith you renamed. The key insight: service boundaries should align with bounded contexts from DDD.

Amazon's Rule: A service is owned by a "two-pizza team" (~6-8 people). If the team cannot understand the full service, it is too big. If the team has idle members, it is too small or split too finely.

Communication Patterns

Pattern When to Use Trade-offs
Sync REST/gRPC Queries, real-time reads, simple request-response Tight coupling, cascading failures, harder to scale independently
Async Events (Kafka/SQS) State changes, notifications, data replication Eventually consistent, harder to debug, message ordering concerns
Async Commands (SQS/RabbitMQ) Fire-and-forget operations, workload offloading No immediate response, need status polling or callbacks

Data Ownership and Anti-Corruption Layer

// ANTI-PATTERN: Service A directly reads Service B's database
// This creates hidden coupling and breaks independent deployability

// CORRECT: Each service owns its data and exposes it via API
// Use an Anti-Corruption Layer (ACL) to translate between contexts

// Order Service has its own view of Product
public record OrderProduct(
    String productId,
    String name,
    Money price,
    int availableQuantity
) {}

// Anti-Corruption Layer translates from Catalog Service's model
@Service
public class ProductCatalogACL {
    private final CatalogServiceClient catalogClient;

    public OrderProduct lookupProduct(String productId) {
        // Call the Catalog Service API
        CatalogProductDto dto = catalogClient.getProduct(productId);

        // Translate to OUR domain model
        // Catalog's "listPrice" becomes our "price"
        // Catalog's "stockCount" becomes our "availableQuantity"
        return new OrderProduct(
            dto.sku(),
            dto.displayName(),
            Money.of(dto.listPrice(), Currency.getInstance(dto.currency())),
            dto.stockCount()
        );
    }
}

// API at the boundary - versioned and stable
@RestController
@RequestMapping("/api/v1/orders")
public class OrderApiController {

    private final OrderQueryHandler queryHandler;

    // Commands use POST with a clear action
    @PostMapping
    public ResponseEntity<OrderCreatedResponse> placeOrder(
            @Valid @RequestBody PlaceOrderRequest request) {
        var cmd = toCommand(request);
        OrderId orderId = commandHandler.handle(cmd);
        return ResponseEntity
            .created(URI.create("/api/v1/orders/" + orderId.value()))

Interview Tip: When asked "How would you split this monolith?", start with: (1) Identify bounded contexts through the domain language. (2) Look for data that changes together -- that stays in one service. (3) Look for data that is only read together -- that can be denormalized. (4) Prefer async events over sync calls for cross-service coordination.

7. Observability

Structured Logging, Metrics & Distributed Tracing ▼

Why Observability? In a monolith, you can attach a debugger. In a distributed system, a single request may touch 10 services. Without observability, debugging production issues means guessing. Observability is the three pillars: logs (what happened), metrics (how much), traces (where).

Real-world example: At Amazon, every request carries a correlation ID (X-Amzn-Trace-Id). When a customer reports "my order page was slow", engineers can search for that trace ID and see exactly which service introduced latency, what database query was slow, and what the retry count was.

Structured Logging with SLF4J + MDC

// WHY structured logging: "Order 123 failed" is unsearchable.
// {"orderId":"123","event":"payment_failed","reason":"timeout"} is queryable.

@Service
public class OrderService {
    private static final Logger log = LoggerFactory.getLogger(OrderService.class);

    public OrderResult processOrder(OrderRequest request) {
        // MDC (Mapped Diagnostic Context) adds context to ALL log lines
        // in this thread, including logs from called methods and libraries
        MDC.put("orderId", request.orderId());
        MDC.put("customerId", request.customerId());
        MDC.put("correlationId", request.correlationId());

        try {
            log.info("Processing order started",
                kv("itemCount", request.items().size()),
                kv("totalAmount", request.totalAmount()));

            var result = doProcess(request);

            log.info("Order processed successfully",
                kv("status", result.status()),
                kv("processingTimeMs", result.durationMs()));

            return result;
        } catch (PaymentException e) {
            log.error("Payment failed for order",
                kv("errorCode", e.getCode()),
                kv("paymentProvider", e.getProvider()),
                e);  // Stack trace included
            throw e;
        } finally {
            MDC.clear();  // CRITICAL: clean up to avoid context leaking
        }
    }
}

// Correlation ID filter - propagates trace context across services
@Component
public class CorrelationIdFilter extends OncePerRequestFilter {
    private static final String CORRELATION_HEADER = "X-Correlation-Id";

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {

        String correlationId = request.getHeader(CORRELATION_HEADER);
        if (correlationId == null || correlationId.isBlank()) {
            correlationId = UUID.randomUUID().toString();
        }

        MDC.put("correlationId", correlationId);
        response.setHeader(CORRELATION_HEADER, correlationId);

        try {
            chain.doFilter(request, response);
        } finally {
            MDC.clear();
        }
    }
}

Metrics with Micrometer

@Service
public class OrderMetrics {
    private final Counter ordersPlaced;
    private final Counter ordersFailed;
    private final Timer orderProcessingTime;
    private final DistributionSummary orderValues;
    private final AtomicInteger activeOrders;

    public OrderMetrics(MeterRegistry registry) {
        // Counters: monotonically increasing values
        this.ordersPlaced = Counter.builder("orders.placed")
            .description("Total orders placed")
            .tag("service", "order-service")
            .register(registry);

        this.ordersFailed = Counter.builder("orders.failed")
            .description("Total failed orders")
            .tag("service", "order-service")
            .register(registry);

        // Timer: measures duration and count
        this.orderProcessingTime = Timer.builder("orders.processing.time")
            .description("Order processing duration")
            .publishPercentiles(0.5, 0.95, 0.99) // p50, p95, p99
            .sla(Duration.ofMillis(100), Duration.ofMillis(500), Duration.ofSeconds(1))
            .register(registry);

        // Distribution summary: measures value distributions
        this.orderValues = DistributionSummary.builder("orders.value")
            .description("Distribution of order values")
            .baseUnit("usd")
            .publishPercentiles(0.5, 0.95)
            .register(registry);

        // Gauge: current value (e.g., in-flight requests)
        this.activeOrders = registry.gauge("orders.active",
            new AtomicInteger(0));
    }

    public void recordOrderPlaced(Money amount) {
        ordersPlaced.increment();
        orderValues.record(amount.amount().doubleValue());
    }

    public Timer.Sample startTimer() {
        return Timer.start();
    }

    public void stopTimer(Timer.Sample sample) {
        sample.stop(orderProcessingTime);
    }
}

Distributed Tracing with OpenTelemetry

// OpenTelemetry auto-instruments most frameworks (Spring, gRPC, JDBC)
// Manual instrumentation for custom spans:

@Service
public class OrderService {
    private final Tracer tracer;

    public OrderResult processOrder(OrderRequest request) {
        Span span = tracer.spanBuilder("processOrder")
            .setAttribute("order.id", request.orderId())
            .setAttribute("order.item_count", request.items().size())
            .startSpan();

        try (Scope scope = span.makeCurrent()) {
            // Child spans are automatically linked
            var paymentResult = paymentService.charge(request); // traced
            var inventoryResult = inventoryService.reserve(request); // traced

            span.setAttribute("order.status", "completed");
            span.setStatus(StatusCode.OK);
            return new OrderResult(paymentResult, inventoryResult);

        } catch (Exception e) {
            span.setStatus(StatusCode.ERROR, e.getMessage());
            span.recordException(e);
            throw e;
        } finally {
            span.end();
        }
    }
}

// Propagating context across async boundaries
@Service
public class AsyncOrderProcessor {
    private final Tracer tracer;
    private final ExecutorService executor;

    public CompletableFuture<Void> processAsync(OrderRequest request) {
        // Capture current context
        Context currentContext = Context.current();

        return CompletableFuture.runAsync(() -> {
            // Restore context in the new thread
            try (Scope scope = currentContext.makeCurrent()) {
                Span childSpan = tracer.spanBuilder("asyncProcess")
                    .startSpan();
                try {
                    // Process...
                } finally {
                    childSpan.end();
                }
            }
        }, executor);
    }
}
When NOT to Over-Invest in Observability:
  • Early prototypes: start with structured logging, add metrics and tracing as you scale
  • Single-service applications: a monolith does not need distributed tracing
  • Cost-sensitive environments: tracing every request at 100% sampling rate is expensive; sample intelligently
Interview Tip: When discussing observability, mention the RED method for services (Rate, Errors, Duration) and the USE method for resources (Utilization, Saturation, Errors). These frameworks show you think about monitoring systematically, not ad hoc.

5 Q&A Pairs for Spaced Repetition ▼
Question Answer
What is the key rule for Aggregate design? All access to entities within the aggregate goes through the Aggregate Root. The aggregate boundary defines the transactional consistency boundary.
What problem does CQRS solve? It separates the read model (optimized for queries, denormalized) from the write model (enforces invariants, normalized), allowing each to scale and evolve independently.
How does Event Sourcing differ from traditional state persistence? Instead of storing current state, you store the sequence of events. Current state is derived by replaying events. Benefits: full audit trail, temporal queries, event replay.
What are the three states of a Circuit Breaker? CLOSED (requests pass through, failures counted), OPEN (requests fail fast), HALF_OPEN (limited trial calls to test recovery).
Orchestration vs Choreography sagas: when to use which? Orchestration: complex multi-step workflows (centralized control, easier to reason about). Choreography: simple 2-3 step flows (decoupled, no single point of failure).

Practice Exercise

Design Aggregate Boundaries for an E-Commerce Domain ▼

The Challenge

You are designing an e-commerce platform. The domain includes: Customers, Orders, Products, Inventory, Payments, Shipping, Reviews, and Promotions.

Your tasks:

  1. Identify the bounded contexts
  2. Define the aggregate roots and their boundaries for the Order bounded context
  3. Determine which data is an Entity vs a Value Object
  4. Design the domain events that flow between bounded contexts
  5. Decide which cross-context communication is sync vs async

Guiding Questions

  • Should OrderLine be an entity or value object? (Hint: does it have an identity that matters outside the order?)
  • Should Product be part of the Order aggregate? (Hint: can a product change price without affecting existing orders?)
  • When an order is placed, should inventory be reserved synchronously or asynchronously?
  • Where does the ShippingAddress live? Is it a value object on the Order or a reference to the Customer?

Sample Solution Sketch

flowchart TB subgraph "Order Context" OA[Order Aggregate Root] OL[OrderLine - Value Object] SA[ShippingAddress - Value Object] OA --> OL OA --> SA end subgraph "Catalog Context" PA[Product Aggregate Root] PR[Price - Value Object] PA --> PR end subgraph "Inventory Context" IA[InventoryItem Aggregate Root] RS[Reservation - Entity] IA --> RS end subgraph "Payment Context" PY[Payment Aggregate Root] TX[Transaction - Entity] PY --> TX end subgraph "Shipping Context" SH[Shipment Aggregate Root] TK[Tracking - Value Object] SH --> TK end OA -->|"OrderConfirmed event (async)"| IA OA -->|"OrderConfirmed event (async)"| PY PY -->|"PaymentCompleted event (async)"| SH OA -.->|"GET product price (sync)"| PA

Key Design Decisions:

  • OrderLine is a Value Object: It has no identity outside the order. Two order lines with the same product and quantity are interchangeable within an order.
  • ShippingAddress is a Value Object on Order: We snapshot the address at order time. If the customer later changes their address, existing orders are unaffected.
  • Product is NOT in the Order aggregate: The Order stores a snapshot of price at order time (unitPrice in OrderLine). Product price changes do not affect existing orders.
  • Inventory reservation is async: The order is confirmed immediately, and an OrderConfirmed event triggers inventory reservation. If inventory is unavailable, a compensating event cancels the order.
  • Product lookup is sync: When adding items to a cart, we need the current price immediately, so we call the Catalog service synchronously.
