1. Singly vs Doubly Linked Lists

Structure & Memory Trade-offs ▼

A singly linked list stores a value and a pointer to the next node. A doubly linked list adds a pointer to the previous node. This extra pointer doubles the overhead per node but enables O(1) removal when you already hold a reference to the node.

When to Use Each

  • Singly linked list: Stack implementation, simple chaining in hash tables, adjacency lists for graphs. Use when you only traverse forward and memory matters.
  • Doubly linked list: LRU cache (O(1) eviction), browser back/forward history, undo/redo systems, text editor cursor movement. Use when you need O(1) deletion of arbitrary nodes or bidirectional traversal.

Real-World Usage

Linux kernel: The kernel's struct list_head is a circular doubly linked list embedded inside other structs, enabling O(1) insertion and removal without separate allocation. It is used in the process scheduler, file system caches, and device driver queues.

Java LinkedHashMap: Internally uses a doubly linked list threaded through a hash map to maintain insertion order. This is the backbone of many LRU cache implementations in production Java services.

Redis: Uses doubly linked lists (quicklist, a linked list of ziplists) for its List data type, balancing memory efficiency with O(1) push/pop at both ends.

2. Core Operations & Edge Cases

Insert, Delete, Search — The Sentinel Node Trick ▼

The biggest source of bugs in linked list code is forgetting edge cases: empty list, single-node list, operations on the head, and operations on the tail. The sentinel (dummy) node technique eliminates most of these by ensuring there is always a node before the first real element.

(sentinel)"] --> A["3"] --> B["7"] --> C["1"] --> N["null"] style D fill:#555,stroke:#888,color:#fff

Python — Complete Operations with Sentinel

class SinglyLinkedList:
    def __init__(self):
        self.dummy = ListNode(-1)  # sentinel
        self.size = 0

    def insert_at_head(self, val):
        """O(1) - Always works, even on empty list."""
        node = ListNode(val, self.dummy.next)
        self.dummy.next = node
        self.size += 1

    def insert_at_index(self, index, val):
        """O(n) - Insert before the index-th node (0-based)."""
        if index < 0 or index > self.size:
            return  # invalid
        prev = self.dummy
        for _ in range(index):
            prev = prev.next
        node = ListNode(val, prev.next)
        prev.next = node
        self.size += 1

    def delete_at_index(self, index):
        """O(n) - Delete the index-th node."""
        if index < 0 or index >= self.size:
            return  # invalid
        prev = self.dummy
        for _ in range(index):
            prev = prev.next
        prev.next = prev.next.next
        self.size -= 1

    def search(self, val):
        """O(n) - Return index or -1."""
        curr = self.dummy.next
        idx = 0
        while curr:
            if curr.val == val:
                return idx
            curr = curr.next
            idx += 1
        return -1

    def to_list(self):
        """Helper for debugging."""
        result, curr = [], self.dummy.next
        while curr:
            result.append(curr.val)
            curr = curr.next
        return result

Java

class SinglyLinkedList {
    private ListNode dummy = new ListNode(-1);
    private int size = 0;

    /** O(1) - insert at head. */
    public void insertAtHead(int val) {
        ListNode node = new ListNode(val, dummy.next);
        dummy.next = node;
        size++;
    }

    /** O(n) - delete the index-th node (0-based). */
    public void deleteAtIndex(int index) {
        if (index < 0 || index >= size) return;
        ListNode prev = dummy;
        for (int i = 0; i < index; i++) prev = prev.next;
        prev.next = prev.next.next;
        size--;
    }

    /** O(n) - search, return index or -1. */
    public int search(int val) {
        ListNode curr = dummy.next;
        for (int i = 0; curr != null; i++, curr = curr.next) {
            if (curr.val == val) return i;
        }
        return -1;
    }
}

3. Fast & Slow Pointers (Floyd's Algorithm)

Cycle Detection & Finding the Cycle Start ▼

Floyd's tortoise-and-hare algorithm uses two pointers moving at different speeds. The slow pointer moves one step at a time; the fast pointer moves two. If there is a cycle, they will meet inside the cycle. If there is no cycle, the fast pointer reaches null.

graph LR A["1"] --> B["2"] --> C["3"] --> D["4"] --> E["5"] --> F["6"] F --> C style C fill:#f59e0b,stroke:#d97706,color:#000 style F fill:#f59e0b,stroke:#d97706,color:#000

Cycle: node 6 points back to node 3. The cycle length is 4 (3 → 4 → 5 → 6 → 3).

Python

def has_cycle(head: ListNode) -> bool:
    """Detect if linked list has a cycle. O(n) time, O(1) space."""
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            return True
    return False


def find_cycle_start(head: ListNode) -> ListNode:
    """
    Find the node where the cycle begins, or None if no cycle.
    O(n) time, O(1) space.
    """
    slow = fast = head
    # Phase 1: detect meeting point inside cycle
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
        if slow is fast:
            break
    else:
        return None  # no cycle

    # Phase 2: find cycle entry
    # Move one pointer back to head; advance both at speed 1
    slow = head
    while slow is not fast:
        slow = slow.next
        fast = fast.next
    return slow  # this is the cycle start

Java

public ListNode detectCycle(ListNode head) {
    ListNode slow = head, fast = head;
    // Phase 1: find meeting point
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
        if (slow == fast) {
            // Phase 2: find entry point
            slow = head;
            while (slow != fast) {
                slow = slow.next;
                fast = fast.next;
            }
            return slow;
        }
    }
    return null; // no cycle
}

Proof: Why Does Phase 2 Work?

Let F = distance from head to cycle start, C = cycle length, and a = distance from cycle start to the meeting point (measured along the cycle).

When slow and fast meet:

  • Slow has traveled: F + a steps
  • Fast has traveled: F + a + kC steps (for some integer k ≥ 1, meaning fast lapped slow k times)
  • Since fast moves twice as fast: 2(F + a) = F + a + kC
  • Simplifying: F + a = kC, so F = kC - a

Now if we place one pointer at head and keep the other at the meeting point, both moving at speed 1:

  • The head pointer walks F steps to reach the cycle start.
  • The meeting-point pointer walks F = kC - a steps. Starting from position a into the cycle, it wraps around k full times minus a, ending exactly at the cycle start.

Therefore, both pointers arrive at the cycle start simultaneously.

Finding the Middle Node ▼

When slow reaches the middle, fast has reached the end. This is used as a building block in merge sort on linked lists, palindrome checking, and many other problems.

def find_middle(head: ListNode) -> ListNode:
    """
    Return the middle node. For even-length lists, returns the
    second middle node (useful for splitting in merge sort).
    1 -> 2 -> 3 -> 4 -> 5  => returns node 3
    1 -> 2 -> 3 -> 4       => returns node 3  (second middle)
    """
    slow = fast = head
    while fast and fast.next:
        slow = slow.next
        fast = fast.next.next
    return slow


def find_first_middle(head: ListNode) -> ListNode:
    """
    For even-length lists, returns the FIRST middle node.
    Useful when you need to split: first half ends here.
    1 -> 2 -> 3 -> 4       => returns node 2  (first middle)
    """
    slow = fast = head
    while fast.next and fast.next.next:
        slow = slow.next
        fast = fast.next.next
    return slow

Java

public ListNode findMiddle(ListNode head) {
    ListNode slow = head, fast = head;
    while (fast != null && fast.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    return slow;
}

4. In-Place Reversal Patterns

Reverse Entire List (Iterative & Recursive) ▼

Reversing a linked list is the single most fundamental linked list operation. Master both approaches — iterative (O(1) space) and recursive (O(n) stack space) — because interviewers will ask for both.

Iterative (O(1) space)

def reverse_list(head: ListNode) -> ListNode:
    """
    Reverse a singly linked list in-place.
    Time: O(n), Space: O(1)
    """
    prev = None
    curr = head
    while curr:
        nxt = curr.next      # save next
        curr.next = prev     # reverse pointer
        prev = curr          # advance prev
        curr = nxt           # advance curr
    return prev              # prev is the new head

Recursive (O(n) stack space)

def reverse_list_recursive(head: ListNode) -> ListNode:
    """
    Recursive reversal. Each call reverses the rest, then
    makes the next node point back to current.
    Time: O(n), Space: O(n) call stack
    """
    # Base case: empty list or single node
    if not head or not head.next:
        return head

    # Recursively reverse the rest
    new_head = reverse_list_recursive(head.next)

    # Make the next node point back to us
    head.next.next = head
    head.next = None  # current node becomes tail

    return new_head

Java (Iterative)

public ListNode reverseList(ListNode head) {
    ListNode prev = null, curr = head;
    while (curr != null) {
        ListNode nxt = curr.next;
        curr.next = prev;
        prev = curr;
        curr = nxt;
    }
    return prev;
}
Reverse Sublist (LeetCode #92) ▼

Problem: Given a linked list and positions left and right (1-indexed), reverse the nodes from position left to right. Do it in one pass.

graph LR subgraph "Before: reverse positions 2 to 4" A1["1"] --> B1["2"] --> C1["3"] --> D1["4"] --> E1["5"] end subgraph "After" A2["1"] --> D2["4"] --> C2["3"] --> B2["2"] --> E2["5"] end 
def reverse_between(head: ListNode, left: int, right: int) -> ListNode:
    """
    Reverse nodes from position left to right (1-indexed).
    Time: O(n), Space: O(1)

    Key insight: use a dummy node, find the node just before 'left',
    then repeatedly move the next node to the front of the reversed section.
    """
    if left == right:
        return head

    dummy = ListNode(0, head)
    prev = dummy

    # Move prev to the node just before position 'left'
    for _ in range(left - 1):
        prev = prev.next

    # curr is the first node to reverse
    curr = prev.next

    # Repeatedly take curr.next and insert it after prev
    for _ in range(right - left):
        nxt = curr.next        # node to move
        curr.next = nxt.next   # skip over nxt
        nxt.next = prev.next   # nxt points to front of reversed section
        prev.next = nxt        # prev points to nxt (new front)

    return dummy.next

Java

public ListNode reverseBetween(ListNode head, int left, int right) {
    ListNode dummy = new ListNode(0, head);
    ListNode prev = dummy;
    for (int i = 0; i < left - 1; i++) prev = prev.next;

    ListNode curr = prev.next;
    for (int i = 0; i < right - left; i++) {
        ListNode nxt = curr.next;
        curr.next = nxt.next;
        nxt.next = prev.next;
        prev.next = nxt;
    }
    return dummy.next;
}
Reverse Nodes in k-Group (LeetCode #25) — Hard ▼

Problem: Given a linked list, reverse the nodes k at a time. If the remaining nodes are fewer than k, leave them as-is.

graph LR subgraph "Before (k=3)" A1["1"] --> B1["2"] --> C1["3"] --> D1["4"] --> E1["5"] end subgraph After C2["3"] --> B2["2"] --> A2["1"] --> D2["4"] --> E2["5"] end

Iterative Approach

def reverse_k_group(head: ListNode, k: int) -> ListNode:
    """
    Reverse nodes in k-group. O(n) time, O(1) space.

    Strategy:
    1. Check if there are k nodes remaining.
    2. If yes, reverse those k nodes.
    3. Connect the reversed group to the previous group.
    4. Move to the next group.
    """
    dummy = ListNode(0, head)
    group_prev = dummy  # node before current group

    while True:
        # Check if k nodes remain
        kth = group_prev
        for _ in range(k):
            kth = kth.next
            if not kth:
                return dummy.next  # fewer than k nodes left

        group_next = kth.next  # node after current group

        # Reverse the k nodes
        prev = group_next  # will become the tail's next
        curr = group_prev.next
        for _ in range(k):
            nxt = curr.next
            curr.next = prev
            prev = curr
            curr = nxt

        # Connect: group_prev -> reversed group start (kth)
        # The original first node is now the tail of the group
        tail = group_prev.next  # was first, now last in group
        group_prev.next = kth   # kth is now the first in reversed group

        # Move group_prev to the tail for next iteration
        group_prev = tail

    return dummy.next

Recursive Approach

def reverse_k_group_recursive(head: ListNode, k: int) -> ListNode:
    """Recursive: reverse first k, then recurse on the rest."""
    # Check if k nodes exist
    node = head
    for _ in range(k):
        if not node:
            return head  # fewer than k, don't reverse
        node = node.next

    # Reverse first k nodes
    prev = None
    curr = head
    for _ in range(k):
        nxt = curr.next
        curr.next = prev
        prev = curr
        curr = nxt

    # head is now the tail of the reversed group
    # curr is the start of the next group
    head.next = reverse_k_group_recursive(curr, k)
    return prev  # prev is the new head of this group

Java (Iterative)

public ListNode reverseKGroup(ListNode head, int k) {
    ListNode dummy = new ListNode(0, head);
    ListNode groupPrev = dummy;

    while (true) {
        ListNode kth = groupPrev;
        for (int i = 0; i < k; i++) {
            kth = kth.next;
            if (kth == null) return dummy.next;
        }
        ListNode groupNext = kth.next;
        ListNode prev = groupNext, curr = groupPrev.next;
        for (int i = 0; i < k; i++) {
            ListNode nxt = curr.next;
            curr.next = prev;
            prev = curr;
            curr = nxt;
        }
        ListNode tail = groupPrev.next;
        groupPrev.next = kth;
        groupPrev = tail;
    }
}

5. Merge Patterns

Merge Two Sorted Lists (LeetCode #21) ▼

This is a building block for merge sort, merge-k-lists, and many other problems. The sentinel node technique makes the code much cleaner.

def merge_two_lists(l1: ListNode, l2: ListNode) -> ListNode:
    """
    Merge two sorted linked lists into one sorted list.
    Time: O(n + m), Space: O(1) (reuses existing nodes)
    """
    dummy = ListNode(0)
    tail = dummy

    while l1 and l2:
        if l1.val <= l2.val:
            tail.next = l1
            l1 = l1.next
        else:
            tail.next = l2
            l2 = l2.next
        tail = tail.next

    # Attach remaining nodes (one list may be non-empty)
    tail.next = l1 if l1 else l2
    return dummy.next

Java

public ListNode mergeTwoLists(ListNode l1, ListNode l2) {
    ListNode dummy = new ListNode(0), tail = dummy;
    while (l1 != null && l2 != null) {
        if (l1.val <= l2.val) { tail.next = l1; l1 = l1.next; }
        else                   { tail.next = l2; l2 = l2.next; }
        tail = tail.next;
    }
    tail.next = (l1 != null) ? l1 : l2;
    return dummy.next;
}
Merge K Sorted Lists (LeetCode #23) — Hard ▼

Two approaches, both important for interviews. Let N = total nodes, k = number of lists.

Approach Time Space When to Use
Min-heapO(N log k)O(k)Streaming data, unknown total size
Divide & conquerO(N log k)O(log k)All lists available upfront, lower constant factor

Approach 1: Min-Heap

import heapq

def merge_k_lists_heap(lists: list) -> ListNode:
    """
    Use a min-heap of size k. Always extract the smallest node,
    then push its successor.
    Time: O(N log k), Space: O(k)
    """
    heap = []
    for i, head in enumerate(lists):
        if head:
            # (value, index_tiebreaker, node)
            heapq.heappush(heap, (head.val, i, head))

    dummy = ListNode(0)
    tail = dummy
    idx = len(lists)  # monotonic tiebreaker

    while heap:
        val, _, node = heapq.heappop(heap)
        tail.next = node
        tail = tail.next
        if node.next:
            heapq.heappush(heap, (node.next.val, idx, node.next))
            idx += 1

    return dummy.next

Approach 2: Divide & Conquer

def merge_k_lists_dc(lists: list) -> ListNode:
    """
    Pair up lists and merge, reducing k by half each round.
    Like merge sort's merge phase.
    Time: O(N log k), Space: O(log k) recursion stack
    """
    if not lists:
        return None
    if len(lists) == 1:
        return lists[0]

    mid = len(lists) // 2
    left = merge_k_lists_dc(lists[:mid])
    right = merge_k_lists_dc(lists[mid:])
    return merge_two_lists(left, right)

Java (Min-Heap)

public ListNode mergeKLists(ListNode[] lists) {
    PriorityQueue<ListNode> heap = new PriorityQueue<>(
        (a, b) -> a.val - b.val
    );
    for (ListNode head : lists) {
        if (head != null) heap.offer(head);
    }
    ListNode dummy = new ListNode(0), tail = dummy;
    while (!heap.isEmpty()) {
        ListNode node = heap.poll();
        tail.next = node;
        tail = tail.next;
        if (node.next != null) heap.offer(node.next);
    }
    return dummy.next;
}
Intersection of Two Linked Lists (LeetCode #160) ▼

Key Insight: If list A has length a and list B has length b, and they merge at some node, then a pointer that walks A then B travels a + b steps, and a pointer that walks B then A also travels b + a steps. They will meet at the intersection (or both reach null).

def get_intersection_node(headA: ListNode, headB: ListNode) -> ListNode:
    """
    Find intersection node of two singly linked lists.
    Time: O(n + m), Space: O(1)

    Elegant trick: when pointer A reaches end of list A,
    redirect it to head of list B (and vice versa).
    Both pointers traverse exactly a + b nodes.
    """
    if not headA or not headB:
        return None

    a, b = headA, headB
    while a is not b:
        a = a.next if a else headB
        b = b.next if b else headA
    return a  # either intersection node or None

Java

public ListNode getIntersectionNode(ListNode headA, ListNode headB) {
    ListNode a = headA, b = headB;
    while (a != b) {
        a = (a != null) ? a.next : headB;
        b = (b != null) ? b.next : headA;
    }
    return a;
}

6. Advanced Problems

Palindrome Linked List in O(1) Space (LeetCode #234) ▼

Strategy: Find the middle, reverse the second half, compare both halves, then restore the list (good practice, though interviewers rarely require it).

def is_palindrome(head: ListNode) -> bool:
    """
    Check if linked list is a palindrome.
    Time: O(n), Space: O(1)

    Steps:
    1. Find middle using slow/fast pointers
    2. Reverse second half
    3. Compare first half with reversed second half
    4. (Optional) Restore the list
    """
    if not head or not head.next:
        return True

    # Step 1: find first middle (for even-length lists)
    slow = fast = head
    while fast.next and fast.next.next:
        slow = slow.next
        fast = fast.next.next

    # Step 2: reverse second half (starting after slow)
    second_half = reverse_list(slow.next)

    # Step 3: compare
    p1, p2 = head, second_half
    result = True
    while p2:  # second half is shorter or equal
        if p1.val != p2.val:
            result = False
            break
        p1 = p1.next
        p2 = p2.next

    # Step 4: restore (optional but polite)
    slow.next = reverse_list(second_half)

    return result

Java

public boolean isPalindrome(ListNode head) {
    if (head == null || head.next == null) return true;
    // Find first middle
    ListNode slow = head, fast = head;
    while (fast.next != null && fast.next.next != null) {
        slow = slow.next;
        fast = fast.next.next;
    }
    // Reverse second half
    ListNode secondHalf = reverseList(slow.next);
    // Compare
    ListNode p1 = head, p2 = secondHalf;
    boolean result = true;
    while (p2 != null) {
        if (p1.val != p2.val) { result = false; break; }
        p1 = p1.next;
        p2 = p2.next;
    }
    slow.next = reverseList(secondHalf); // restore
    return result;
}
Copy List with Random Pointer (LeetCode #138) ▼

Problem: Deep copy a linked list where each node has a next pointer and a random pointer to any node in the list (or null).

Approach 1: HashMap (O(n) space)

def copy_random_list(head):
    """
    Deep copy with random pointer using a hash map.
    Time: O(n), Space: O(n)
    """
    if not head:
        return None

    # Map original node -> copy node
    old_to_new = {}

    # First pass: create all copy nodes
    curr = head
    while curr:
        old_to_new[curr] = ListNode(curr.val)
        curr = curr.next

    # Second pass: wire up next and random pointers
    curr = head
    while curr:
        copy = old_to_new[curr]
        copy.next = old_to_new.get(curr.next)
        copy.random = old_to_new.get(curr.random)
        curr = curr.next

    return old_to_new[head]

Approach 2: Interleaving (O(1) space) — The Clever Way

def copy_random_list_o1(head):
    """
    O(1) space approach: interleave copies with originals.

    Step 1: A -> A' -> B -> B' -> C -> C'
    Step 2: Set random pointers using A'.random = A.random.next
    Step 3: Unweave the two lists
    """
    if not head:
        return None

    # Step 1: Interleave - insert copy after each original
    curr = head
    while curr:
        copy = ListNode(curr.val)
        copy.next = curr.next
        curr.next = copy
        curr = copy.next

    # Step 2: Set random pointers
    curr = head
    while curr:
        if curr.random:
            curr.next.random = curr.random.next
        curr = curr.next.next

    # Step 3: Unweave
    new_head = head.next
    curr = head
    while curr:
        copy = curr.next
        curr.next = copy.next
        copy.next = copy.next.next if copy.next else None
        curr = curr.next

    return new_head

Doubly linked list: most recently used near head, least recently used near tail. Evict from tail.

Python

class LRUCache:
    """
    LRU Cache with O(1) get and put.

    Data structures:
    - Hash map: key -> DListNode for O(1) lookup
    - Doubly linked list: maintains access order
      Head side = most recently used
      Tail side = least recently used (eviction candidate)

    Why doubly linked list?
    - O(1) removal of any node (need prev pointer)
    - O(1) insertion at head
    - Maintains ordering without shifting elements

    Why not just a list/array?
    - Removing from middle of array is O(n)
    - Linked list gives O(1) with direct node reference
    """
    def __init__(self, capacity: int):
        self.cap = capacity
        self.cache = {}  # key -> DListNode

        # Sentinel nodes to avoid null checks
        self.head = DListNode(0)  # dummy head
        self.tail = DListNode(0)  # dummy tail
        self.head.next = self.tail
        self.tail.prev = self.head

    def _remove(self, node: DListNode):
        """Remove node from its current position. O(1)."""
        node.prev.next = node.next
        node.next.prev = node.prev

    def _add_to_front(self, node: DListNode):
        """Add node right after head (most recently used). O(1)."""
        node.next = self.head.next
        node.prev = self.head
        self.head.next.prev = node
        self.head.next = node

    def get(self, key: int) -> int:
        """O(1) lookup. Move accessed node to front."""
        if key not in self.cache:
            return -1
        node = self.cache[key]
        self._remove(node)
        self._add_to_front(node)
        return node.val

    def put(self, key: int, value: int):
        """O(1) insert/update. Evict LRU if at capacity."""
        if key in self.cache:
            self._remove(self.cache[key])

        node = DListNode(value)
        node.key = key  # store key for eviction lookup
        self.cache[key] = node
        self._add_to_front(node)

        if len(self.cache) > self.cap:
            # Evict least recently used (node before tail)
            lru = self.tail.prev
            self._remove(lru)
            del self.cache[lru.key]

Java

class LRUCache {
    private int capacity;
    private Map<Integer, DListNode> cache = new HashMap<>();
    private DListNode head = new DListNode(0), tail = new DListNode(0);

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    private void remove(DListNode node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void addToFront(DListNode node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }

    public int get(int key) {
        if (!cache.containsKey(key)) return -1;
        DListNode node = cache.get(key);
        remove(node);
        addToFront(node);
        return node.val;
    }

    public void put(int key, int value) {
        if (cache.containsKey(key)) remove(cache.get(key));
        DListNode node = new DListNode(value);
        node.key = key;
        cache.put(key, node);
        addToFront(node);
        if (cache.size() > capacity) {
            DListNode lru = tail.prev;
            remove(lru);
            cache.remove(lru.key);
        }
    }
}

class DListNode {
    int key, val;
    DListNode prev, next;
    DListNode(int val) { this.val = val; }
}

Production usage: This exact pattern powers caching in Memcached, CPU TLB caches, and database buffer pools. Redis uses an approximation (random sampling + eviction) because maintaining a strict LRU list is too expensive at Redis's throughput (100K+ ops/sec per core).

Flatten a Multilevel Doubly Linked List (LeetCode #430) ▼

Problem: You have a doubly linked list where some nodes have a child pointer to another doubly linked list. Flatten all levels into a single list.

def flatten(head):
    """
    Flatten multilevel doubly linked list.
    Time: O(n), Space: O(1) iterative approach

    Strategy: when we encounter a child, splice the child list
    in between current and current.next. Use a stack-like approach
    with the 'next' pointers to remember where to resume.
    """
    if not head:
        return head

    curr = head
    while curr:
        if curr.child:
            # Find the tail of the child list
            child_tail = curr.child
            while child_tail.next:
                child_tail = child_tail.next

            # Splice: curr <-> child ... child_tail <-> curr.next
            child_tail.next = curr.next
            if curr.next:
                curr.next.prev = child_tail

            curr.next = curr.child
            curr.child.prev = curr
            curr.child = None  # remove child pointer

        curr = curr.next

    return head

7. Staff-Level Discussion: When Linked Lists Are Wrong

Memory, Cache-Friendliness & Practical Alternatives ▼

At the staff level, interviewers expect you to know not just how to implement linked lists but when they are the wrong choice (which is most of the time in modern systems).

Why Arrays Almost Always Win

Bjarne Stroustrup's benchmark: Inserting into the middle of a sorted sequence is faster with a std::vector than a std::list for sequences up to ~500K elements, even though insert is theoretically O(n) for vector vs O(1) for list. The reason: cache misses from pointer chasing dominate the O(1) "constant."

When Linked Lists ARE the Right Choice

  • O(1) splice operations: Moving a contiguous chunk of nodes from one list to another (Linux kernel scheduler).
  • Intrusive lists: Where nodes are embedded inside other structs (no separate allocation). Used in OS kernels and game engines.
  • LRU / LFU caches: Need O(1) arbitrary removal by reference. The doubly-linked-list + hash map combo is optimal.
  • Memory allocators: Free lists track available memory blocks. No allocation overhead since the free block IS the node.
  • Real-time systems: Worst-case O(1) insert/delete matters more than average throughput. Dynamic arrays have amortized O(1) but worst-case O(n) for resize.

Skip Lists: The Linked List That Scales

A skip list adds multiple levels of forward pointers to a linked list, achieving O(log n) search, insert, and delete — comparable to balanced BSTs but simpler to implement and more cache-friendly for concurrent access.

Interview Tips

Common Follow-ups & What Interviewers Look For ▼

1. "Why use a dummy/sentinel node?"

What they look for: You understand it eliminates edge cases for head insertion/deletion without conditional checks. Mention that it adds a constant 8-16 bytes — negligible cost for cleaner, less buggy code.

Red flag: Writing separate if-statements for head operations instead of using a dummy.

2. "Can you do this without extra space?"

What they look for: Knowledge of in-place techniques: pointer reversal, slow/fast pointers, Morris traversal (for trees). For palindrome check: reverse second half in-place rather than copying to an array.

3. "What's the time complexity of your approach on a list with a cycle?"

What they look for: Understanding that naive traversal loops forever. Floyd's algorithm terminates in O(n) because the fast pointer closes the gap by 1 each step. The maximum distance to close is C (cycle length), and C ≤ n.

4. "In production, would you use a linked list here?"

What they look for: Staff candidates should push back. Arrays/vectors are almost always better due to cache locality. Mention specific exceptions: LRU cache, memory allocator free lists, OS kernel task scheduling. Show you think beyond theoretical big-O.

5. "How would you test your linked list implementation?"

What they look for:

  • Empty list, single node, two nodes
  • Operations on head and tail
  • Cycle detection on acyclic lists (should return false)
  • Large lists (performance regression)
  • Concurrent access (thread safety for LRU cache)

Self-Check Quiz

Flashcards

Key Concept Flashcards (Click to Reveal Answers) ▼
Q: In Floyd's cycle detection, after slow and fast meet, how do you find the cycle start?
A: Move one pointer back to the head. Advance both pointers at speed 1. They meet at the cycle start. This works because the distance from head to cycle start equals kC - a (where C is cycle length and a is the meeting offset).
Q: Why does an LRU cache need a doubly linked list instead of a singly linked list?
A: To remove a node in O(1), you need access to its predecessor. A doubly linked list stores a prev pointer, so removal is O(1). With a singly linked list, you must traverse from head to find the predecessor, making removal O(n).
Q: What is the time complexity of merging k sorted linked lists using a min-heap?
A: O(N log k) where N is the total number of nodes and k is the number of lists. Each of the N nodes is pushed/popped from the heap once, and each heap operation costs O(log k).
Q: Why are linked lists cache-unfriendly?
A: Nodes are allocated separately on the heap, scattered across memory. Traversing the list causes a cache miss at nearly every node because the next node is unlikely to be in the same cache line (64 bytes on x86). Arrays store elements contiguously, so the CPU prefetcher loads upcoming elements automatically.
Q: How do you check if a linked list is a palindrome in O(1) extra space?
A: (1) Find the middle using slow/fast pointers. (2) Reverse the second half in-place. (3) Compare first half with reversed second half. (4) Optionally restore the list by reversing again.
Q: When is a linked list actually the right data structure?
A: When you need O(1) worst-case insertion/deletion by reference (LRU cache, OS task schedulers), O(1) splice of contiguous segments (kernel lists), or when nodes are embedded in other structs (intrusive lists). For most application-level code, dynamic arrays win.
