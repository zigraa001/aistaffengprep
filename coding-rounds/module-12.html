Module 12: Greedy Algorithms & Intervals

Learning Objectives

  • Understand the greedy choice property and optimal substructure
  • Distinguish when greedy works vs when you need DP
  • Master interval scheduling patterns: merge, remove overlap, meeting rooms
  • Solve classic greedy problems: jump game, gas station, task scheduler
  • Apply the exchange argument to prove greedy correctness
  • Recognize greedy in graph algorithms (MST overview)

1. The Greedy Strategy

When Does Greedy Work? ▼

A greedy algorithm makes the locally optimal choice at each step, hoping it leads to a globally optimal solution. This works when the problem has two properties:

  • Greedy Choice Property: A locally optimal choice is part of some globally optimal solution. You never need to reconsider past choices.
  • Optimal Substructure: An optimal solution to the problem contains optimal solutions to its subproblems (shared with DP).

Why greedy fails on some problems: Consider the coin change problem with denominations [1, 3, 4] and target 6. Greedy picks 4+1+1=3 coins, but optimal is 3+3=2 coins. The greedy choice (pick the largest coin) is NOT part of the optimal solution.

The Exchange Argument (Proof Technique)

To prove a greedy algorithm is correct, use the exchange argument:

  1. Assume an optimal solution O* that differs from greedy solution G
  2. Find the first place where they differ
  3. Show you can "exchange" O*'s choice for G's choice without worsening the solution
  4. Repeat until O* = G, proving greedy is also optimal

Example: For activity selection (pick max non-overlapping intervals), greedy picks the earliest-ending activity. If O* picks a different first activity a*, we can swap a* with greedy's pick g (which ends no later) without creating conflicts, since g frees up at least as much room for future activities.

2. Interval Problems

Merge Intervals (LC #56) ▼

Problem: Given an array of intervals, merge all overlapping intervals.

Key Insight: Sort by start time. Then scan left to right: if the current interval overlaps with the last merged one, extend; otherwise start a new merged interval.

def merge(intervals: list[list[int]]) -> list[list[int]]:
    intervals.sort(key=lambda x: x[0])
    merged = [intervals[0]]
    for start, end in intervals[1:]:
        if start <= merged[-1][1]:   # overlaps
            merged[-1][1] = max(merged[-1][1], end)
        else:
            merged.append([start, end])
    return merged
# Time: O(n log n)  Space: O(n) for output

Why sort by start? After sorting, we only need to check overlap with the previous interval. Without sorting, every pair might need checking (O(n^2)).

Non-Overlapping Intervals (LC #435) ▼

Problem: Find the minimum number of intervals to remove so that the rest are non-overlapping.

Key Insight: This is equivalent to finding the maximum number of non-overlapping intervals (classic activity selection). Sort by end time and greedily keep intervals that don't overlap with the last kept one.

def eraseOverlapIntervals(intervals: list[list[int]]) -> int:
    intervals.sort(key=lambda x: x[1])  # sort by END time
    kept = 0
    prev_end = float('-inf')
    for start, end in intervals:
        if start >= prev_end:  # no overlap
            kept += 1
            prev_end = end
    return len(intervals) - kept
# Time: O(n log n)  Space: O(1)

Why sort by end time, not start? Sorting by end time ensures we always pick the interval that leaves the most room for future intervals. This is the greedy choice property: finishing earliest is always at least as good as finishing later.

Meeting Rooms II (LC #253) ▼

Problem: Find the minimum number of conference rooms needed to host all meetings.

Key Insight: The answer equals the maximum number of overlapping meetings at any point. Use a sweep line: separate starts and ends, sort them, and track concurrent meetings.

import heapq

def minMeetingRooms(intervals: list[list[int]]) -> int:
    if not intervals:
        return 0
    # Approach 1: Min-heap of end times
    intervals.sort(key=lambda x: x[0])
    heap = [intervals[0][1]]  # end time of first meeting
    for start, end in intervals[1:]:
        if start >= heap[0]:   # room freed up
            heapq.heapreplace(heap, end)
        else:
            heapq.heappush(heap, end)
    return len(heap)

def minMeetingRooms_sweep(intervals: list[list[int]]) -> int:
    # Approach 2: Sweep line (often cleaner)
    events = []
    for start, end in intervals:
        events.append((start, 1))   # meeting starts
        events.append((end, -1))    # meeting ends
    events.sort()
    rooms = max_rooms = 0
    for _, delta in events:
        rooms += delta
        max_rooms = max(max_rooms, rooms)
    return max_rooms
# Both: Time O(n log n), Space O(n)

Real-world: Same as connection pooling, thread scheduling, and cloud VM allocation.

3. Classic Greedy Problems

Jump Game I & II (LC #55, #45) ▼

Jump Game I (#55): Can you reach the last index? Track the farthest reachable index.

def canJump(nums: list[int]) -> bool:
    farthest = 0
    for i in range(len(nums)):
        if i > farthest:
            return False
        farthest = max(farthest, i + nums[i])
    return True

Jump Game II (#45): Minimum jumps to reach end. Use a BFS-like greedy: track the current level's boundary and the farthest you can reach.

def jump(nums: list[int]) -> int:
    jumps = 0
    cur_end = 0       # end of current jump range
    farthest = 0      # farthest we can reach
    for i in range(len(nums) - 1):
        farthest = max(farthest, i + nums[i])
        if i == cur_end:       # must jump
            jumps += 1
            cur_end = farthest
    return jumps
# Time: O(n)  Space: O(1)

Why greedy works here: Within each "jump level," we explore all reachable positions and pick the one that gets us farthest. This is optimal because jumping further never hurts.

Gas Station (LC #134) ▼

Problem: N gas stations in a circle with gas[i] fuel and cost[i] to reach the next station. Find the starting station to complete the circuit, or -1 if impossible.

Key Insight: If total gas ≥ total cost, a solution exists. To find it: if running out of fuel at station j when starting from i, then no station between i and j works either. Start fresh from j+1.

def canCompleteCircuit(gas: list[int], cost: list[int]) -> int:
    if sum(gas) < sum(cost):
        return -1
    tank = 0
    start = 0
    for i in range(len(gas)):
        tank += gas[i] - cost[i]
        if tank < 0:
            start = i + 1  # everything before i+1 fails
            tank = 0
    return start
# Time: O(n)  Space: O(1)

Why the skip is valid: If you run out at j starting from i, it means cumulative surplus was positive from i to some point but went negative by j. Any start between i+1..j would have less cumulative surplus (since station i had positive surplus), so they all fail too.

Task Scheduler (LC #621) ▼

Problem: Schedule tasks with cooldown n between same tasks. Minimize total time (including idles).

Key Insight: The most frequent task determines the structure. Place it first and fill gaps with other tasks.

from collections import Counter

def leastInterval(tasks: list[str], n: int) -> int:
    freq = Counter(tasks)
    max_freq = max(freq.values())
    # count how many tasks have the max frequency
    max_count = sum(1 for v in freq.values() if v == max_freq)
    # Formula: (max_freq - 1) frames of size (n+1), plus final group
    result = (max_freq - 1) * (n + 1) + max_count
    # If tasks fill all gaps, answer is just len(tasks)
    return max(result, len(tasks))
# Time: O(n)  Space: O(1) (26 letters max)

Visualization (A=6, B=2, C=1, n=2): A B C | A B _ | A _ _ | A _ _ | A _ _ | A -- most frequent task creates the frame, others fill gaps.

Partition Labels (LC #763) ▼

Problem: Partition a string so each letter appears in at most one part. Maximize partition count.

def partitionLabels(s: str) -> list[int]:
    last = {c: i for i, c in enumerate(s)}  # last occurrence
    start = end = 0
    result = []
    for i, c in enumerate(s):
        end = max(end, last[c])
        if i == end:             # all chars in partition accounted for
            result.append(end - start + 1)
            start = i + 1
    return result
# Time: O(n)  Space: O(1) (26 letters)

Why greedy: We must extend each partition to include the last occurrence of every character we've seen. Once we reach that boundary, we know it's safe to cut.

4. Greedy in Graph Algorithms (MST)

Kruskal's & Prim's MST Overview ▼

Minimum Spanning Tree algorithms are textbook greedy. Both produce optimal results because of the cut property: the lightest edge crossing any cut must be in some MST.

AlgorithmStrategyTimeBest For
Kruskal'sSort edges, add smallest that doesn't form cycle (Union-Find)O(E log E)Sparse
Prim'sGrow tree, always add cheapest edge to new vertex (min-heap)O(E log V)Dense
 
# Kruskal's MST (simplified)
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
    def find(self, x):
        if self.parent[x] != x:
            self.parent[x] = self.find(self.parent[x])
        return self.parent[x]
    def union(self, x, y):
        px, py = self.find(x), self.find(y)
        if px == py: return False
        if self.rank[px] < self.rank[py]: px, py = py, px
        self.parent[py] = px
        if self.rank[px] == self.rank[py]: self.rank[px] += 1
        return True

def kruskal(n, edges):  # edges = [(weight, u, v)]
    edges.sort()
    uf = UnionFind(n)
    mst_weight = 0
    for w, u, v in edges:
        if uf.union(u, v):
            mst_weight += w
    return mst_weight

5. Greedy vs DP: When to Use Which

Decision Framework & Comparison Table ▼

Rule of thumb: If you can prove the greedy choice property (locally optimal = globally optimal), use greedy. If choices interact or you need to try combinations, use DP.

ProblemGreedy?DP?Why
Activity SelectionYesOverkillEarliest-end choice never blocks better options
Coin Change (arbitrary)NoYesLargest coin may skip optimal combos (e.g., [1,3,4] target 6)
Fractional KnapsackYesOverkillCan take fractions; best value/weight ratio first is optimal
0/1 KnapsackNoYesMust take whole items; greedy by ratio fails
Jump Game IIYesSlowerJumping farthest each level is provably optimal

Interview tip: If you suspect greedy, try to find a counterexample. If you can't find one in 30 seconds, code the greedy solution and mention you'd prove correctness via exchange argument.
