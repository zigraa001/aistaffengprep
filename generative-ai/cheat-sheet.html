1. Transformers & Attention

Attention Formula

Attention(Q,K,V) = softmax(QKᵀ / √d_k) · V
  • Q, K, V — Query, Key, Value matrices from linear projections
  • √d_k — scaling prevents softmax saturation (d_k = head_dim)
  • Multi-head — h heads run in parallel, concat + project: h × d_k = d_model

Attention Variants

  • MHA — Multi-Head Attention (original); KV heads = Q heads
  • MQA — Multi-Query: 1 KV head, all Q heads share it; fastest
  • GQA — Grouped-Query: g KV groups (Llama 3, Mistral); 4–8× smaller KV cache, <0.5% quality loss
  • FlashAttention — IO-aware tiling; same result, 2–4× faster, O(N) memory vs O(N²)

Transformer Block (decoder)

  • Masked Self-Attention → Add & Norm
  • Feed-Forward (2 linear layers, GELU) → Add & Norm
  • FFN hidden dim ≈ 4× d_model

Positional Encoding

  • Sinusoidal — original; fixed, extrapolates poorly
  • RoPE — Rotary PE (GPT-NeoX, Llama); rotates Q/K, extrapolates well
  • ALiBi — attention bias by distance; no learned PE

Key Numbers (GPT-4 class)

  • d_model: 8192–12288; n_heads: 64–96; n_layers: 80–120
  • Context: 8K–128K tokens; vocab: ~50K–150K

KV Cache Memory

bytes/token = 2 × 2 × n_layers × d_model
(2 for K+V, 2 for fp16)
  • Llama-2-7B (32 layers, d=4096): 524 KB/token
  • Llama-2-70B (80 layers, d=8192): 2.6 MB/token
  • GQA with g=8 groups: divide by (n_heads / g)

Speculative Decoding

Speedup ≈ K×α / (K×t_ratio + 1)
  • K = draft tokens per step, α = acceptance rate (~0.7–0.85)
  • t_ratio = draft latency / target latency (<0.1 ideal)
  • Expected speedup: 1.5–3×

Tensor Parallelism Overhead

  • 4 all-reduces/layer × NVLink latency ~80µs = ~2.56ms/layer for 70B on 8×A100
  • Use NVLink for TP within node, InfiniBand across nodes

Repetition Controls

  • presence_penalty — flat penalty for any prior token (−2 to 2)
  • frequency_penalty — penalty scales with token count (−2 to 2)
  • repetition_penalty — multiplier on logit (HF; 1.0 = off, 1.2 typical)
  • no_repeat_ngram_size — forbid repeating n-gram (n=3 common)

Advanced / Local Models

  • mirostat (mode 2) — targets perplexity τ; auto-adjusts temp; great for long generation
  • tfs_z — Tail Free Sampling; removes low-curvature tail tokens
  • grammar / GBNF — constrained decoding to JSON schema, regex, CFG

Output Control

  • max_tokens — hard cap on output length
  • stop — stop sequences (e.g., ["\n\n", "###"])
  • logprobs — return per-token log probabilities; use for calibration
  • logit_bias — add constant to token logits; ban tokens with −100
  • response_format: {type: "json_object"} — force JSON output
  • seed — deterministic sampling (best-effort)

4. RAG & Retrieval

RAG Pipeline

  • Chunking — fixed (512 tokens), sentence, semantic, or hierarchical
  • Embedding — BGE-large, E5-mistral, OpenAI text-embedding-3-large
  • Index — HNSW (ANN, fast) or flat (exact, small corpus)
  • Hybrid search — dense + BM25 (sparse); RRF fusion
  • Reranking — cross-encoder (Cohere, BGE-reranker) after top-50 retrieval
  • Generation — inject top-k (3–5) chunks into prompt

Semantic Collapse (Stanford)

  • At 1K docs: 95% precision → at 100K docs: 12% precision
  • Cause: embedding space saturates, similar vectors crowd each other
  • Fix: Hierarchical RAG — cluster docs (HDBSCAN), route query to right cluster first

Hierarchical RAG

  • Document routing: lightweight classifier → select relevant corpus subset
  • HDBSCAN clustering: density-based, handles noise; min_cluster_size=5
  • Graph-based retrieval: entity graph → multi-hop reasoning
  • Result: 10× faster, 2× more accurate vs flat RAG at scale

5. AI Agents

ReAct Loop

  • Thought — LLM reasons about what to do
  • Action — calls a tool (function/API)
  • Observation — tool result injected into context
  • Repeat until final answer; max steps to prevent loops

Agent Architectures

  • Single-agent — one LLM + tools; simple, fragile at scale
  • Multi-agent — orchestrator + specialist agents; LangGraph, AutoGen
  • Plan-and-Execute — planner creates task list; executor runs each
  • Reflection — agent critiques own output; improves iteratively

Tool Design Best Practices

  • Idempotent tools — safe to retry without side effects
  • Precise docstrings — LLM decides when to call based on description
  • Return structured JSON, not prose
  • Include error types in schema — agent can self-correct

  • Context overflow — summarize old observations; sliding window
  • Hallucinated tool calls — validate args against schema before execution

10. AI Tradeoffs Framework

Accuracy vs Latency

  • Larger model → better accuracy, 2–10× more latency
  • Cascade — cheap model handles easy cases, expensive for hard ones; best of both
  • Quantization — INT4: 4× smaller, 3–5% quality loss, 2× faster
  • Speculative decoding — same quality, 1.5–3× faster at decode
  • Target: <200ms for interactive, <2s for batch

Explainability vs Performance

  • Logistic Regression = fully interpretable, coefficients = feature weights
  • Tree ensembles = partial via SHAP/LIME
  • LLMs = chain-of-thought = soft explainability (not guaranteed faithful)
  • High-stakes domains (credit, healthcare) often require interpretable models by regulation

Data Flywheel

  • More users → more data → better model → more users
  • Requires intentional data collection (feedback loops, HITL labels)
  • Log all model inputs/outputs from day 1
  • Weekly eval on production samples; don't wait for complaints

Training-Serving Skew

  • Cause: feature computed differently in training pipeline vs serving
  • Fix: shared feature store (Feast/Tecton); point-in-time correct joins
  • Monitor: input distribution drift (KL divergence alert >0.1)

11. AI-Native Application Patterns

Model Cascade

  • Route easy queries to cheap model, hard ones to expensive model
  • Confidence threshold: if max_prob < 0.8 → escalate
  • Typical setup: GPT-4o-mini (95% traffic) → GPT-4o (5%)
  • Can cut cost by 10–50× with <1% quality loss

Human-in-the-Loop (HITL)

  • Flag low-confidence predictions for human review
  • Use human feedback to create fine-tune dataset
  • Track human correction rate; if >10% → model needs retraining

Model-as-Feature

  • LLM generates embedding or classification label as a feature for downstream ML
  • Example: sentiment score from LLM → input to recommendation model
  • Decouples LLM latency (async pre-compute) from serving latency

Guardrail Layer

  • Input guardrails: detect PII, prompt injection, off-topic
  • Output guardrails: factuality check, toxicity, hallucination score
  • Tools: Guardrails AI, Llama Guard, Azure Content Safety
  • Add latency: typically 50–150ms; parallelize where possible

Async Processing Pattern

  • Accept request → return job_id immediately (202 Accepted)
  • Run LLM call async (Celery, SQS, Kafka consumer)
  • Client polls or receives webhook on completion
  • Use for: document processing, batch embeddings, long generation

Context Window Management

  • Summarize — compress old turns into summary when >50% window used
  • Sliding window — keep last N turns; drop oldest
  • Retrieval-augmented memory — embed old turns, retrieve relevant ones
  • Prefix caching — system prompt KV reuse; 50–70% cost reduction
Interview tip: When asked to design an AI-native system, always address: (1) fallback strategy when model fails, (2) eval pipeline, (3) cost at scale, (4) how the data flywheel works.
