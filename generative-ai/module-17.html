Module 17: Traditional ML, DS Collaboration & AI Tradeoffs

What every staff engineer must know about classical ML models, partnering with Data Science teams, and making principled AI technology decisions.

Why This Module Exists

Staff engineers at AI-native companies are expected to do more than ship LLM features. You will be asked: "Why not a simpler model?", "How do you work with Data Science?", and "What are the tradeoffs of this AI approach?" This module gives you the vocabulary and frameworks to answer those questions cold.

What You'll Learn

  • Key ML metrics: classification, regression, ranking, clustering
  • The ML vs DL vs LLM decision framework with concrete criteria
  • How to partner effectively with Data Science and ML teams

  • AI tradeoffs that interviewers ask about: accuracy vs latency, explainability vs performance, build vs buy
  • AI-native application design patterns

1. Traditional ML Model Landscape

Supervised Learning: The Workhorse Models ▼

Linear & Logistic Regression

Classification Regression Interpretable Fast

  • How it works: Fits a weighted sum of features (+ bias) to the target. Logistic adds a sigmoid to produce probabilities.
  • Key hyperparameters: regularization type (L1=Lasso → sparse features, L2=Ridge → small weights), C (inverse of regularization strength)
  • When to use: Baseline model, interpretability required, features are mostly linear, low latency needed, small datasets (<100K rows)
  • When to avoid: Non-linear relationships, high-cardinality categoricals without feature engineering, image/text/audio input
  • Metrics to monitor: Classification: AUC-ROC, log loss, F1. Regression: RMSE, MAE, R²
from sklearn.linear_model import LogisticRegression, Ridge from sklearn.pipeline import Pipeline from sklearn.preprocessing import StandardScaler # Classification clf = Pipeline([ ("scaler", StandardScaler()), ("model", LogisticRegression(C=1.0, penalty="l2", max_iter=1000, solver="lbfgs")) ]) clf.fit(X_train, y_train) # Coefficients tell you feature importance (after scaling) coef = dict(zip(feature_names, clf["model"].coef_[0])) print(sorted(coef.items(), key=lambda x: abs(x[1]), reverse=True)[:5])

Decision Trees

Classification Regression Interpretable Prone to overfit

  • How it works: Recursively splits data by the feature+threshold that maximises information gain (Gini impurity or entropy for classification; variance reduction for regression).
  • Key hyperparameters: max_depth (most important — controls overfit), min_samples_leaf, min_samples_split
  • When to use: Exploring feature interactions, explainability for stakeholders, handling mixed feature types without preprocessing
  • When to avoid: Standalone for production — almost always outperformed by ensembles. High-cardinality categoricals cause issues.
  • Staff insight: Decision trees are the building block of random forests and gradient boosting. Understand them deeply — interviewers test this.

Random Forest

Classification Regression Robust Handles missing values

  • How it works: Trains N decision trees on bootstrapped data subsets (bagging) + random feature subsets. Averages predictions (regression) or votes (classification). Decorrelated trees reduce variance.
  • Key hyperparameters: n_estimators (100–500), max_features (sqrt for classification, 1/3 for regression), max_depth
  • When to use: Tabular data, feature importance needed, robust baseline, missing values (set max_features appropriately)
  • Feature importance: Mean decrease in impurity (MDI) — but biased toward high-cardinality features. Use permutation importance for unbiased estimates.
  • Metrics to watch: OOB (out-of-bag) error — free validation without a separate set.

Gradient Boosting: XGBoost / LightGBM / CatBoost

Classification Regression Needs tuning SOTA tabular

  • How it works: Builds trees sequentially. Each new tree corrects the residual error of all previous trees. Uses gradient descent in function space. Much more powerful than random forest for structured data.
  • XGBoost vs LightGBM vs CatBoost:
    • XGBoost: Level-wise tree growth, column subsampling, most widely deployed
    • LightGBM: Leaf-wise growth → faster training on large datasets, better for high-cardinality features
    • CatBoost: Native categorical support (no one-hot needed), ordered boosting to prevent target leakage
  • Key hyperparameters: learning_rate (0.01–0.3), n_estimators (100–2000), max_depth (3–8), subsample, reg_alpha/lambda (L1/L2)
  • Production tip: Always use early stopping with a validation set. SHAP values for explainability.
import lightgbm as lgb import shap model = lgb.LGBMClassifier( n_estimators=500, learning_rate=0.05, max_depth=6, num_leaves=63, # LightGBM: use num_leaves, not max_depth alone subsample=0.8, colsample_bytree=0.8, reg_alpha=0.1, # L1 regularization reg_lambda=1.0, # L2 regularization early_stopping_rounds=50, ) model.fit(X_train, y_train, eval_set=[(X_val, y_val)], callbacks=[lgb.log_evaluation(50)]) # SHAP values for explainability — critical for enterprise / regulated industries explainer = shap.TreeExplainer(model) shap_values = explainer.shap_values(X_val) shap.summary_plot(shap_values, X_val, feature_names=feature_names)

Support Vector Machines (SVM)

Classification Regression (SVR) Slow on large data

  • How it works: Finds the hyperplane with maximum margin between classes. Kernel trick (RBF, polynomial) maps to high-dimensional space for non-linear boundaries.
  • When to use: Small-to-medium datasets (<100K), high-dimensional sparse data (text with TF-IDF), clear margin of separation
  • When to avoid: Large datasets (O(n²–n³) training time), noisy data with overlapping classes, need for probability outputs (platt scaling required)
  • Key hyperparameters: C (margin hardness), kernel (rbf is default), gamma (RBF width)

k-Nearest Neighbours (kNN)

Classification Regression No training Slow inference

  • How it works: At inference time, finds the k closest training examples (by distance) and returns their majority class / average value.
  • When to use: Recommendation systems (approximate kNN), anomaly detection, when you don't have labeled data but have similar items
  • Production note: Exact kNN is O(n) per query. In production, use HNSW (Faiss, Qdrant) for approximate nearest-neighbour search — this is the engine behind vector databases.

Naive Bayes

Classification Very fast Works with small data

  • How it works: Applies Bayes' theorem assuming feature independence (naive). P(class|features) ∝ P(features|class) × P(class).
  • Variants: Multinomial NB (text/counts), Gaussian NB (continuous features), Bernoulli NB (binary features)
  • When to use: Text classification baseline, spam filtering, very limited compute, streaming/online learning, real-time classification at microsecond latency
  • Classic example: Gmail spam filter used NB for years. Still the fastest option when accuracy vs speed tradeoff matters.
Unsupervised Learning: Clustering, Dimensionality Reduction ▼

K-Means Clustering

Clustering Fast Requires k upfront

  • How it works: Initializes k centroids, assigns each point to nearest centroid, updates centroids to cluster means. Repeats until convergence. Minimizes within-cluster sum of squares (inertia).
  • Choose k with: Elbow method (inertia vs k), Silhouette score (max = best), Gap statistic
  • Metrics: Inertia (within-cluster SS), Silhouette score (-1 to 1, target >0.5), Davies-Bouldin index (lower = better)
  • Limitations: Assumes spherical clusters, sensitive to outliers, scale-dependent (must standardize). Use DBSCAN for arbitrary shapes, HDBSCAN for varying densities.

DBSCAN / HDBSCAN

Clustering No k needed Finds outliers

  • DBSCAN: Core points (≥min_samples within eps radius) form clusters. Border points attach to nearest core. Noise points (outliers) get label -1.
  • HDBSCAN: Hierarchical extension. Handles varying densities. More robust than DBSCAN — preferred for production. Used in the Hierarchical RAG section of module-08.
  • When to use: Unknown number of clusters, expect outliers, non-spherical cluster shapes, document clustering in RAG pipelines

PCA — Principal Component Analysis

Dimensionality Reduction Linear Interpretable components

  • How it works: Finds orthogonal directions (principal components) of maximum variance in the data. Projects data onto top-k components. SVD under the hood.
  • When to use: Remove correlated features, speed up downstream models, visualize high-dimensional data (with 2 components), noise reduction
  • Key metric: Explained variance ratio — keep enough components to explain 90–95% of variance
  • Limitation: Linear only. Use UMAP/t-SNE for non-linear structure.

t-SNE / UMAP — Non-linear Visualization

Visualization Not for new points (t-SNE)

  • t-SNE: Preserves local structure. Great for visualization. Cannot project new points without refitting. Perplexity parameter (5–50) controls neighborhood size.
  • UMAP: Faster, preserves both local and global structure, can project new points. Preferred for production and embedding visualization.
  • Use case: Visualize LLM embedding clusters, debug RAG retrieval quality, explore customer segmentation

The Precision-Recall Tradeoff (Staff Must Know)

Adjusting the classification threshold shifts precision and recall in opposite directions. Lower threshold → more positives predicted → higher recall, lower precision. Higher threshold → fewer positives → higher precision, lower recall. The right threshold depends on your cost asymmetry: how much does a false positive cost vs a false negative? In fraud detection, a missed fraud (FN) is far costlier than a false alarm (FP), so optimize recall.

Staff-Level Interview Answer: "My default is to start with the simplest model that could work. If the data is tabular and labeled, I reach for LightGBM first — it trains in seconds, gives feature importance, and beats neural nets on most tabular benchmarks. I only move to LLMs when the task involves language understanding, zero-shot generalization, or when the business needs natural language I/O. LLMs are a last resort on cost and latency grounds, not a first reach."

4. Partnering with Data Science & ML Teams

The Staff Engineer's Role in ML Projects ▼
The Core Tension: Data Scientists optimize for model accuracy. Engineers optimize for reliability, scalability, and maintainability. Staff engineers bridge this gap — they need enough ML literacy to ask the right questions, and enough systems experience to build the right infrastructure.

What Data Scientists Expect from a Staff Engineer

🏗️
Build reliable ML infrastructure

Feature stores (Feast, Tecton, Hopsworks), model training pipelines (Kubeflow, Vertex AI Pipelines, SageMaker Pipelines), experiment tracking (MLflow, W&B), model registries.

🚀
Own model deployment and serving

Serve models via REST/gRPC (TorchServe, Triton Inference Server, BentoML). Manage A/B testing infrastructure. Handle rollbacks. Maintain SLOs for model latency and throughput.

📊
Build monitoring and observability

Track prediction drift, feature drift, data quality. Alert on performance degradation. Build dashboards for model performance metrics. Handle retraining triggers.

🔒
Enforce data quality and contracts

Define schema contracts for features. Validate data at ingestion (Great Expectations, Soda). Prevent training-serving skew — the #1 cause of silent model failures in production.

💬
Translate between ML and product

Help DS frame the problem correctly (prediction vs recommendation vs anomaly). Ensure metrics align with business outcomes. Push back when the ML solution is over-engineered for the problem.

Questions a Staff Engineer Should Ask When Reviewing an ML Project

  • What is the offline-to-online gap? Does the eval set reflect production data distribution?
  • What happens when the model is wrong? Is there a fallback? Can the user override?
  • What is the retraining strategy? Scheduled (daily/weekly), trigger-based (drift threshold), or continuous?
  • How are features computed at serving time? Do they match how they were computed at training time?
  • Does the model need to be explainable to regulators, users, or support teams?
  • What is the label latency? If ground truth arrives hours later, how do you handle it?
  • What is the data lineage? Can you trace any prediction back to the exact features used?

Training-Serving Skew: The Silent Killer

Training-serving skew occurs when features are computed differently at training time vs serving time. Example: at training, "user's total purchases" is computed from the full history. At serving, it's computed from the last 30 days due to a caching optimization. The model was never trained on that distribution. The model appears to work but silently underperforms.

Fix: Use the same feature computation code for training and serving. Feature stores enforce this by having one feature definition used for both offline and online retrieval.

Experiment Tracking with MLflow

import mlflow import mlflow.sklearn from sklearn.ensemble import GradientBoostingClassifier from sklearn.metrics import roc_auc_score mlflow.set_experiment("churn-prediction-v2") with mlflow.start_run(run_name="lgbm-baseline"): # Log hyperparameters params = {"n_estimators": 300, "learning_rate": 0.05, "max_depth": 6} mlflow.log_params(params) model = GradientBoostingClassifier(**params) model.fit(X_train, y_train) # Log metrics train_auc = roc_auc_score(y_train, model.predict_proba(X_train)[:, 1]) val_auc = roc_auc_score(y_val, model.predict_proba(X_val)[:, 1]) mlflow.log_metric("train_auc", train_auc) mlflow.log_metric("val_auc", val_auc) mlflow.log_metric("overfit_gap", train_auc - val_auc) # ← flag if > 0.05 # Log model artifact + schema signature = mlflow.models.infer_signature(X_train, model.predict(X_train)) mlflow.sklearn.log_model(model, "model", signature=signature) # Register in model registry for deployment mlflow.register_model(f"runs:/{mlflow.active_run().info.run_id}/model", "churn-prediction-prod")

6. AI-Native Application Design Patterns

What Makes an Application "AI-Native" ▼

An AI-native app is one where AI is not a feature but the core product mechanism. The UX, data model, and architecture are all shaped around the model. Staff engineers at AI-native companies must design for:

graph TD subgraph "AI-Native App Architecture" U[User Input] --> GW[API Gateway] GW --> S[Safety/Guardrail Layer] S --> R[Router\nSimple vs Complex] R -->|Simple| FM[Fast Model\nsmall/cached] R -->|Complex| LM[Large Model\nRAG + Tools] FM --> C[Context Manager] LM --> C C --> FB[Feedback Collection] FB --> FT[Fine-tune Pipeline\ndata flywheel] FT --> LM end

The 6 Core AI-Native Patterns

1. Model Cascade (Cheap → Expensive)

Route easy requests to a small cheap model; escalate hard ones to a large expensive model. The fast model classifies difficulty first.

Example: Tier-1 support: GPT-4o-mini handles 70% of tickets (<$0.001 each). Complex edge cases go to GPT-4o ($0.01 each). 3× cost reduction, <5% quality degradation.

def smart_route(query: str, threshold: float = 0.8) -> str: # Fast classifier to assess query complexity complexity_score = complexity_classifier.predict([query])[0] if complexity_score < threshold: return call_model("gpt-4o-mini", query) # cheap, fast else: return call_model("gpt-4o", query, # expensive, accurate tools=all_tools, rag=True)

2. Human-in-the-Loop (HITL)

For high-stakes or low-confidence decisions, route to a human reviewer instead of auto-accepting the model output.

Triggers for HITL: Confidence below threshold, novel input distribution, high-value transaction, regulated domain (medical, legal, financial).

Key metric: Auto-accept rate (target >85% for cost efficiency). Track FP rate of auto-accepts — if it creeps up, lower the threshold.

3. Data Flywheel

User interactions generate labeled data → label data is used to fine-tune → fine-tuned model is better → attracts more users → more data. Compounding advantage over time.

Staff engineer's role: Build the data collection pipeline, define quality filters, automate the fine-tuning trigger, A/B test the new model before full rollout.

4. Model-as-Feature (Embedding Features)

Use a pre-trained model to generate embedding features, then feed those embeddings into a lightweight classical ML model (logistic regression, XGBoost). Best of both worlds: rich representation from the foundation model + fast, interpretable downstream model.

Example: Embed product descriptions with text-embedding-3-small → feed 1536-dim vectors to LightGBM for price prediction. Trains in minutes, inference in <5ms.

5. Guardrail Layer (Input + Output)

Input guardrails: PII detection, prompt injection detection, topic classification (refuse off-topic requests).
Output guardrails: Hallucination detection (faithfulness check via RAGAS), toxicity filtering (Perspective API), PII scrubbing in outputs, schema validation for structured outputs.

Latency budget: Guardrails add 50–200ms. Budget this into your TTFT SLO. Run input guardrails in parallel with model prefill to hide latency.

6. Async / Background AI Processing

Not every AI task needs to be synchronous. Move expensive AI operations to background queues (Celery, SQS + Lambda, Pub/Sub). Return a job ID immediately; stream results when ready or notify via webhook.

Use cases: Document analysis, report generation, batch embedding, retraining pipelines, nightly recommendation pre-computation.

Pattern: POST /analyze → 202 Accepted + job_id → GET /results/{job_id} → 200 with results when complete.

Active Recall Questions

1. A product team wants to build a churn prediction model. Walk me through your model selection process.

Start with the simplest model that could work. For churn (tabular data, binary classification): (1) EDA to understand class imbalance and feature types. (2) Logistic regression as interpretable baseline. (3) LightGBM with early stopping as the likely production model. (4) Use SHAP values to explain predictions to stakeholders. Would only consider an LLM if churn signals are embedded in free-text customer support tickets, in which case I'd embed the tickets and use them as features for LightGBM rather than calling an LLM at inference time.

2. What is training-serving skew and how do you prevent it?

Training-serving skew is when features are computed differently at training time vs serving time, causing silent model degradation. Prevent it by: (1) Using a feature store where the same feature definition is used for both offline training data retrieval and online serving lookups. (2) Integration tests that compute features on a sample request and compare training vs serving values. (3) Adding feature distribution monitoring in production and alerting on drift. (4) Logging all feature values used for each prediction so you can reproduce and debug.

3. When would you choose XGBoost over an LLM for a classification problem?

Almost always for structured tabular data. XGBoost: (1) Trains in seconds/minutes vs hours/days. (2) Inference in microseconds vs 100ms–seconds. (3) 10–100× cheaper. (4) More interpretable via SHAP. (5) No hallucination risk. I'd only consider an LLM if: the input is unstructured text, I need zero-shot generalization to unseen categories, or the task requires reasoning steps that can't be captured in features. Even then I might embed the text and use XGBoost downstream rather than calling an LLM at inference time.

4. You have AUC-ROC = 0.95 but your product team is unhappy with the model. What questions do you ask?

(1) Is AUC-ROC the right metric? For highly imbalanced data, a model with 0.95 AUC can still have terrible precision — check AUC-PR. (2) What threshold is being used? AUC is threshold-agnostic but the business is experiencing a specific threshold. (3) Is the eval set representative? 0.95 on a clean held-out set but 0.70 on production data suggests distribution shift. (4) Are there subgroup failures? Overall AUC can mask poor performance on important customer segments. (5) What does the confusion matrix look like? Understand where errors are — FP or FN — and which error type costs more.

5. How do you explain to a stakeholder why your LLM-based feature sometimes gives wrong answers?

Frame it in business terms: "LLMs are probabilistic, not deterministic — they predict the most likely correct answer based on patterns learned from billions of documents, not a lookup table. For well-supported facts, they're >95% reliable. For edge cases or very recent events, confidence is lower. This is why we have: (1) RAG to ground answers in authoritative sources, (2) faithfulness scoring to detect when the model makes claims not supported by the context, and (3) a human review queue for answers below our confidence threshold. We target <2% error rate on supported queries, similar to a well-trained customer support agent."

6. What's the difference between presence_penalty and repetition_penalty?

presence_penalty (OpenAI): additive flat penalty subtracted from any token that appeared at least once. Binary — doesn't grow with frequency. Used to encourage topic variety. frequency_penalty (OpenAI): additive penalty proportional to count — the more times a token appears, the larger the penalty. repetition_penalty (HuggingFace/llama.cpp): multiplicative — divides the logit of previously seen tokens by the penalty value. 1.0 = off, 1.3 = strong. The additive vs multiplicative difference means they interact differently with temperature scaling.
